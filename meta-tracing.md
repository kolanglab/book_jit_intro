# メタトレーシング：インタプリタを書けば JIT がついてくる

前章のトレーシング JIT には、見過ごせない実装コストがあります。
**言語ごとに、その言語専用のトレーシング JIT を書かねばならない**ことです。
JavaScript 用、Python 用、Ruby 用…と、似た仕組みを一から作り直す。
言語処理系を作る人にとって、これは大きな負担です。

**メタトレーシング**（meta-tracing）は、この負担を魔法のように消し去る
アイデアです。一言でいえば、

> **ユーザのプログラムをトレースするのではなく、ユーザのプログラムを
> 実行しているインタプリタをトレースする。**

これだけです。しかし、この一段のずらしが、信じがたい帰結を生みます。
**あなたはただインタプリタを書くだけでよい。トレーシング JIT は、
フレームワークがインタプリタから自動で生成してくれる。** この章では、
その仕掛けを解きほぐします。

## 二つのレベルを区別する

混乱を避けるため、まず登場人物をはっきりさせます。メタトレーシングには
**二つのレベルのプログラム**があります。

- **アプリケーションレベル**：ユーザが書いた、たとえば Python のプログラム
  （`for i in range(n): sum += i` など）。
- **インタプリタレベル**：その Python を実行するインタプリタ自身のコード
  （`while` で命令を取り出し、`case` でディスパッチする、前章で見たような
  ループ）。

ふつうのトレーシング JIT（前章）は、**アプリケーションレベルのループ**を
トレースしました。`for i in range(n)` を 1 周ぶん記録した。

メタトレーシングは違います。トレースするのは**インタプリタレベルの実行**
です。ただし、ディスパッチループの 1 周ではありません —— それはバイト
コード 1 命令ぶんにしかなりません。アプリケーションのループが 1 周する
あいだに、インタプリタは「命令を取り出してディスパッチ」を何度も繰り返し
ます。メタトレーサは、**アプリのループが 1 周して同じ場所（同じ `pc`）に
戻ってくるまで**、ディスパッチループを何周も**展開**（unrolling）しながら、
その**インタプリタの動き**をまとめて記録するのです。

## なぜそれでうまくいくのか

「インタプリタをトレースして何が嬉しいのか」と思うかもしれません。鍵は、
トレース中に**インタプリタの内部状態が定数として畳み込まれる**ことです。

アプリケーションのループ `for i in range(n): sum += i` を回すあいだ、
インタプリタの `pc`（どの命令を実行中か）は、`sum += i` を表すバイト
コードの上を**同じ順番でぐるぐる回ります**。メタトレーサが、アプリの
ループ 1 周ぶん —— ディスパッチループの数周ぶん —— を展開しながら
記録すると、その間 `pc` は「2, 3, 4, 5, 2」と決まった値をたどって
出発点に戻ります。つまり、**インタプリタのディスパッチ（`case insn[0]`）の
分岐結果が、トレース上では定数に畳める**のです。

```
（メタトレースのイメージ：記録されたトレースのうち、:add 1 命令ぶんの断片。
 これが pc = 2, 3, 4, 5 と、アプリのループ 1 周ぶん続く）
  insn = code[pc]            ; pc は定数 → どの命令かコンパイル時に確定
  case insn[0]               ; → :add に確定、ディスパッチが消える
    vars[:sum] = vars[:sum] + vars[:i]   ; 残るのは「本当の計算」だけ
  pc += 1                    ; pc も定数として進む
```

ディスパッチ（`case`）も `pc` の管理も、トレースの上では定数畳み込みで
消えてしまい、**残るのはユーザがやりたかった計算だけ**になります。
[なぜ JIT か](why-jit.md)で見た「ディスパッチを消す」が、インタプリタを
トレースするだけで自動的に達成されるのです。これは第 II 部で扱う
[部分評価と Futamura 射影](research-partial-eval.md)の、トレーシング版の
現れだと言えます。

> [!IMPORTANT]
> メタトレーシングの利得を一言でいえば、**「インタプリタのオーバー
> ヘッド（ディスパッチ・pc 管理）はインタプリタレベルの定数なので、
> トレースすると畳み込まれて消える」**。だから言語ごとに最適化器を
> 書かなくても、インタプリタさえ書けば、その言語に特化した高速トレースが
> 自動で得られる。これがメタトレーシングの一文要約です。

## RPython と PyPy

このアイデアを実用化したのが **PyPy** プロジェクトの **RPython** です
[](#cite:bolz2009)[](#cite:rigo2006)。RPython は "Restricted Python"、つまり Python の
**静的に型が決まる範囲に限定した部分集合**で、インタプリタを書くための
言語です。流れはこうです。

1. 言語実装者は、対象言語（Python など）のインタプリタを **RPython で
   書く**。これはふつうのインタプリタを書く作業とほとんど同じです。
2. RPython ツールチェインが、そのインタプリタを C へ変換してコンパイル
   する。**同時に、メタトレーシング JIT を自動で組み込む**。
3. できあがった処理系は、ユーザプログラムを実行しながら、ホットな
   ループのインタプリタトレースを記録・最適化し、機械語にする。

実装者は JIT を一行も書いていないのに、対象言語の JIT が手に入る。
PyPy（Python 処理系）がこれで作られ、CPython より数倍速い性能を
達成しました。同じ仕組みで、Ruby・PHP・Prolog・Smalltalk など
さまざまな言語の処理系も作られています（多くは研究プロトタイプの
段階にとどまりますが、「インタプリタを書けば JIT がついてくる」ことの
実証になっています）。

## ヒント：実装者がトレーサに教えること

完全自動とはいえ、良いトレースを得るには、実装者がメタトレーサに
**ヒント**を与える必要があります。インタプリタのコードに数か所、
「ここを見てほしい」という印をつけるのです。代表的なヒントを挙げます。

- **`jit_merge_point`**：「ここがアプリケーションレベルのループの
  合流点（バイトコードのディスパッチの先頭）だ」と教える印。メタトレーサは
  ここを基準にトレースの始まりと終わりを判断します。
- **`can_enter_jit`**：「ここがバックエッジ（アプリのループが戻る地点）だ」。
  ホットさを数える場所。
- **`promote(x)`**：「`x` はめったに変わらないから、トレース中は**定数
  として扱ってよい**（念のためガードは置く）」。型やクラスを定数化して
  特殊化を促す投機のヒント。
- **`elidable`（純粋）**：「この関数は同じ引数なら同じ結果を返す副作用
  なしの関数だ」。トレース中に呼び出しごと畳み込んでよい、と教える。

これらは、前章までに見た**プロファイリングのカウンタ設置点**や
**投機の前提**を、インタプリタのソースの上に手で配置したものです。
言い換えると、メタトレーシングは「JIT の判断ポイントを、汎用フレーム
ワークと実装者のヒントで分担する」設計だと言えます。

```ruby
# RPython 風のヒントを Ruby の疑似コードで（実際の API ではない）
def interpret(prog, vars)
  pc = 0
  while (insn = prog[pc])
    jit_merge_point(pc: pc, prog: prog)   # ★ ディスパッチの先頭
    case insn[0]
    when :add then vars[insn[1]] = vars[insn[2]] + vars[insn[3]]; pc += 1
    when :lt  then vars[insn[1]] = vars[insn[2]] < vars[insn[3]]; pc += 1
    when :goto
      target = insn[1]
      can_enter_jit(pc: target) if target < pc   # ★ バックエッジ（ループ）
      pc = target
    when :halt then return vars
    end
  end
end
```

`jit_merge_point` で `pc` と `prog` を渡しているのが肝です。これにより
メタトレーサは「`pc` はトレース中ほぼ定数として畳んでよい量だ」と知り、
前述のディスパッチ消去が起こせます。次の節からは、この一つひとつを
**具体的にどう使い、何を間違えると壊れるのか**を、コードとトレード
オフで掘り下げていきます。

### green と red：どの変数を「定数の鍵」にするか

`jit_merge_point` に渡す変数は、ただ並べればよいのではありません。
**green** と **red** という二種類に分けて指定します。これがメタ
トレーシングを使いこなす上で**最初にして最重要の概念**です。

- **green 変数**：トレース中に**定数として畳み込む鍵**となる量。
  典型は `pc`（プログラムカウンタ）、バイトコード配列 `prog`、現在の
  メソッドオブジェクトなど。**「アプリケーションのループを 1 周する
  あいだ、同じ値に戻ってくる」量**がこれにあたります。
- **red 変数**：ふつうに値が流れていく量。典型は変数の中身 `vars`、
  スタックの値、計算の途中結果。**毎回違ってよい**値です。

なぜこの区別が決定的なのか。メタトレーサは、**green 変数の組み合わせが
同じなら同じトレースに合流する**と判断します。`pc` を green にすると、
`pc = 2` のときのトレース、`pc = 5` のときのトレース…と、**命令位置
ごとに特殊化された**コードができます。トレース内では `pc` が定数なので
`insn = prog[pc]` が定数畳み込みされ、`case insn[0]` のディスパッチが
丸ごと消える —— [なぜ JIT か](why-jit.md)で見た「ディスパッチを消す」が、
ここで自動的に起こるわけです。

```ruby
# green と red を明示する（RPython 風の疑似コード）
JITDRIVER = JitDriver.new(
  greens: [:pc, :prog],   # ★ 定数の鍵：命令位置とバイトコード列
  reds:   [:vars]         # 普通に流れる値：変数テーブル
)

def interpret(prog, vars)
  pc = 0
  while (insn = prog[pc])
    JITDRIVER.jit_merge_point(pc: pc, prog: prog, vars: vars)
    # この点に「pc, prog が同じ値」で来たトレースは合流・再利用される
    ...
  end
end
```

ここで **`vars` を誤って green にしたらどうなる**でしょうか。`vars` は
ループのたびに変化するので、「green の組み合わせ」が毎周ことごとく
変わってしまいます。メタトレーサは「これは別のループ地点だ」と判断し、
**合流できずトレースが延々と量産される**——ひどい場合はメモリを食い
つぶしてコンパイルが終わりません。逆に **`pc` を red にしてしまう**と、
`pc` が定数畳み込みされないため `insn = prog[pc]` が残り、**ディスパッチ
が消えません**。せっかくトレースしても、インタプリタのオーバーヘッドが
そのまま機械語に焼き付いてしまいます。

> [!IMPORTANT]
> green/red の分類は、メタトレーシングの**性能を左右する一番の急所**
> です。原則は「**アプリケーションのループが 1 周して `jit_merge_point`
> に戻ってきたとき、同じ値に戻っている量＝green、毎回変わってよい量＝
> red**」。green は「これは定数のはずだ」
> という実装者の宣言であり、外れる（つまり green にした量がしょっちゅう
> 変わる）と、トレースが爆発するか、まったく速くなりません。

> [!TIP]
> 迷ったら **`pc` 系（命令位置・呼び出し中のメソッド）は green、
> データ系（変数・スタック・オブジェクト）は red** と覚えてください。
> 「コードの形を決めるもの」が green、「コードに流れるもの」が red、
> という直感です。これは前章[トレーシング](tracing.md)で「ループ本体は
> 一直線に焼く」と言ったときの、何を定数とみなすかの選択そのものです。

### `promote(x)`：実行時の値を「いま定数」にして特殊化する

green/red はループ全体の構造に関する分類でした。これに対し `promote`
は、**トレースのその一点で、red の値を一時的に定数として扱わせる**ヒント
です。典型的な使いどころは、**オブジェクトのクラス（型）**です。

> [!TIP]
> **green と promote はどちらも「定数化」だが、性質が対照的**です。
> - **green**：ループ全体を通じて変わらない**恒久的な定数の鍵**（`pc`・
>   バイトコード列など）。トレースの合流条件そのものを決め、ガードを
>   伴いません。
> - **promote**：本来は red（流れる値）を、**そのトレース局所で一時的に
>   定数化する投機**。前提が崩れたときに備えてガードが自動で挿入され、
>   外れれば**脱最適化**します。
>
> ひとことで言えば、green は「最初から定数だと分かっているもの」、
> promote は「定数だと**賭ける**もの」です。

`vars[:x] + vars[:y]` を考えます。`+` の意味は `x` の型で変わります
（整数なら加算、文字列なら連結…）。型は red な値（実行時に決まる）
ですが、**実際にはほとんど変わりません**。そこで型を `promote` して
「いまは整数だと決め打つ」と、整数加算へ特殊化できます。

```ruby
# promote で型を定数化し、特殊化を促す（疑似コード）
def add(x, y)
  cls = promote(x.class)        # ★ x.class を「いまの定数」として扱う
  guard(x.class == cls)         # ★ 外れたら脱最適化する保険（自動で挿入）
  case cls
  when IntBox    then IntBox.new(x.value + y.value)   # 整数加算へ特殊化
  when FloatBox  then FloatBox.new(x.value + y.value)
  when StringBox then StringBox.new(x.value + y.value)
  end
end
```

`promote` がやっているのは、**「投機 ＋ ガード」**の宣言です
[](#cite:bolz2009)。トレーサは `cls` をその場の定数として扱い、`case` を
畳み込んで整数加算 1 命令にまで削ります。その代わり、本当に整数で
来続けるという**前提が崩れたとき**に備えて、`guard(x.class == cls)`
を自動で置きます。これは前章までの**投機**そのものです。

トレードオフは明快です。**当たれば**ディスパッチが消えて劇的に速い。
**外れる**（たとえば整数と文字列が交互に来る）と、毎回ガードが失敗して
[脱最適化](deopt-osr.md)が連発し、かえって遅くなります。

> [!WARNING]
> `promote` は「ほとんど変わらない」値にだけ使ってください。
> ループのたびに変わる値（ループ変数 `i` そのものなど）を `promote`
> すると、`i` の値ごとに別トレースが量産され、ガード失敗による
> **脱最適化**が頻発します。`promote` は「型・クラス・配列長・
> メソッドの同一性」のような、**値域が事実上少数に限られる量**に
> 当てるのが定石です。

### `elidable`：純粋関数を「結果ごと」畳み込む

`promote` が「値を定数化する」なら、`elidable` は**関数呼び出しごと
結果を畳み込む**ヒントです。`elidable` を付けた関数は、トレーサに
「**同じ引数なら必ず同じ結果を返す、副作用のない関数だ**」と宣言します。
すると、引数がトレース上で定数に畳まれていれば、**呼び出し自体を消して
結果の定数に置き換え**られます。

代表例は**メソッド探索**です。動的言語では `obj.foo` のたびにクラスの
メソッド表を辿りますが、これは「クラスとメソッド名が同じなら同じ
メソッドを返す」純粋関数です。

```ruby
# メソッド探索を elidable にして、探索結果を畳み込む（疑似コード）
elidable def lookup_method(cls, name)
  # cls の継承チェインを辿って name のメソッドを探す（読み取りのみ）
  c = cls
  while c
    m = c.methods[name]
    return m if m
    c = c.superclass
  end
  nil
end

def call_method(recv, name, args)
  cls = promote(recv.class)            # ★ クラスを定数化（前節）
  meth = lookup_method(cls, name)      # ★ elidable なので結果を畳める
  guard(recv.class == cls)             # クラスが変わっていないか保険
  meth.invoke(recv, args)
end
```

`promote` で `cls` が定数になり、`lookup_method` は `elidable` なので、
トレーサは継承チェインを辿るループ全体を実行時に 1 度だけ計算し、
**結果のメソッドオブジェクトを定数として焼き込みます**。トレース後の
コードには探索ループは一切残らず、`meth.invoke` への直接呼び出し（さらに
インライン展開）だけが残ります。これが**インラインキャッシュ**
[](#cite:deutsch1984)の、メタトレーシングによる自動的な実現です。

> [!WARNING]
> `elidable` を**副作用のある関数に誤って付ける**と、処理系が壊れます。
> たとえばカウンタを増やす・ログを出す・キャッシュを書き換える関数に
> 付けると、トレーサは「結果が同じだから呼ばなくてよい」と判断して
> **呼び出しを丸ごと消して**しまい、副作用が起きなくなります。これは
> 再現の難しい厄介なバグになります。`elidable` は「**読むだけ・引数の
> みに依存・例外も決定的**」を満たす関数にだけ付けてください。
> メソッド表の更新（クラス定義の変更）があったときに探索結果が変わる
> 場合は、**バージョン番号を引数に含める**などして「同じ引数」の条件を
> 正しく保つ工夫も要ります。

### virtualizable：フレームオブジェクトの確保をトレースから消す

もう一つ、実用上きわめて重要なのが**インタプリタのフレーム**の扱いです。
多くのインタプリタは、メソッド呼び出しのたびに「フレームオブジェクト」を
ヒープに確保し、そこにローカル変数やスタックを置きます。素朴にトレース
すると、**この確保がそのままトレースに残ってしまい**、ループのたびに
オブジェクトを作る無駄が焼き付きます。

これを消すのが**仮想化**（virtualization、エスケープ解析による
**スカラ置換**）です。トレース内で作られてトレース内で死ぬだけの
オブジェクトなら、トレーサはヒント無しでも自動で確保を消せます。厄介
なのはインタプリタのフレームで、`eval` やデバッガ、例外のバックトレース
といった**外から覗かれる経路が常に存在する**ため、素朴な解析では
「エスケープしない」と言い切れません。そこで実装者がフレームに
`virtualizable` の印を付けます。これは「**実際には漏れうるが、漏れない
前提で最適化してよい。本当に覗かれたら、そのとき実体化（force）して
つじつまを合わせる**」という宣言です。印が付いたフレームについて、
トレーサは**オブジェクトの確保をやめて、その中身を個別のレジスタ
や仮想の値としてばらして持つ**ようにします。

```ruby
# フレームを virtualizable にして、確保を消す（疑似コード）
class Frame
  virtualizable :locals, :stack   # ★ 中身はトレース内でばらしてよい
  def initialize(size)
    @locals = Array.new(size)
    @stack  = []
  end
end

def interpret(code, frame)        # frame は呼び出しごとに新規確保される
  pc = 0
  while (insn = code[pc])
    JITDRIVER.jit_merge_point(pc: pc, code: code, frame: frame)
    case insn[0]
    when :load  then frame.stack.push(frame.locals[insn[1]])
    when :store then frame.locals[insn[1]] = frame.stack.pop
    ...
    end
  end
end
```

`virtualizable` が効くと、`frame.locals[i]` や `frame.stack` への読み書きは
**ヒープ上のオブジェクト操作ではなく、トレース内の仮想の値の付け替え**
になります。`push`/`pop` の対が打ち消し合えば、スタック操作すら消えて
**値が直接レジスタ間で受け渡される**コードになります。インタプリタの
「フレームを作って値を出し入れする」という骨組みが、トレース後には
跡形もなく消えるわけです。

> [!TIP]
> ここから「**インタプリタをトレースしやすい形に書く**」コツが見えて
> きます。具体的には、(1) フレームやスタックは `virtualizable` にできる
> 単純な構造（固定長配列）にする、(2) 1 命令の処理を短く・分岐を浅く
> 保つ（長い `case` は green な `pc` で畳めるが、命令内の深い動的分岐は
> 残る）、(3) 副作用のない補助関数は積極的に `elidable` にする、
> (4) 型やクラスは要所で `promote` する。**インタプリタの書き方が、
> そのまま生成される JIT の質を決める**のがメタトレーシングです。

> [!NOTE]
> では、フレームが**本当に覗かれた**らどうなるか。そのときは、ばらして
> 持っていた中身から本物のフレームオブジェクトを**強制的に実体化**
> （force）し、覗いた側につじつまの合った姿を見せます。正しさは保たれ
> ますが、force が頻発すれば仮想化の利得は消えます。だから「速いパスでは
> フレームを覗かない」ようインタプリタを設計しておくことが、仮想化を
> 効かせるコツになります。

## メタトレーシングが処理系設計に与えた影響

メタトレーシングは、単に「楽に JIT を作れる」だけの技術ではありません。
**処理系の設計思想そのもの**を変えました[](#cite:bolz2015)。

従来は、「インタプリタ（実行用）」と「JIT コンパイラ（最適化用）」を
**別々に**実装し、両者の意味が食い違わないよう苦労して整合させていました。
[脱最適化](deopt-osr.md)が難しいのも、この二重実装のためです。
メタトレーシングでは、**インタプリタが唯一の正典**になります。JIT は
そのインタプリタのトレースなので、定義上インタプリタと意味がずれません。
「実装が一つで済む」ことは、正しさの面でも大きな利点です。

> [!NOTE]
> ただし無料の昼食ではありません。メタトレーシングで良い性能を出すには、
> インタプリタを「トレースしやすい形」に書く技術（ヒントの配置、データ
> 構造の選び方）が要ります。また、トレーシング JIT の弱点（分岐が
> ばらつくコードへの弱さ、[トレースの木](tracing.md)の管理）は、
> メタレベルでもそのまま受け継がれます。

もう少し踏み込むと、**トレーシングの長所も短所も、メタレベルにそのまま
継承されます**。メタトレーシングは「インタプリタをトレースする」だけなので、前章
[トレーシング](tracing.md)で見た性質が、ほぼそっくりメタレベルに
持ち上がるのです。これは長所でもあり短所でもあります。

まず**ガード**です。前章でトレースの途中に「前提が崩れたら抜ける」
[脱最適化](deopt-osr.md)点を置いたのと同じことが、メタレベルでも起こり
ます。上で見た `promote` のクラスガードや `elidable` のバージョンガードが、
そのまま脱最適化点になります。ガードが失敗しがちなコード（型が乱れる、
分岐が五分五分）に弱いという**トレーシング固有の弱点**も、当然受け
継がれます。

次に**トレースの木と再帰**です。前章では、トレースの途中で分岐が
ばらつくと**サイドトレース**を生やして木に育てる、と説明しました。
メタレベルでも同じで、アプリのループが `if` で割れれば、対応する
インタプリタのトレースから側枝が伸びます。**再帰関数**も同様に厄介です。
アプリ側の再帰呼び出しは、インタプリタ側では「`call` 命令を実行して
新しいフレームを積み、ディスパッチループに再突入する」という形に
なります。素朴にトレースすると再帰の各段が一直線に展開され、トレースが
無限に伸びかねません。

```ruby
# 再帰呼び出しのトレースを止める：呼び出し先を別トレースに切る（疑似コード）
when :call
  callee = lookup_method(promote(recv.class), insn[1])
  # ★ 呼び出し先のループ先頭が既知のトレースなら、そこへ飛ぶだけにする
  #   （インライン展開を打ち切り、別トレースへの遷移として扱う）
  if trace_exists_for?(callee)
    enter_trace(callee, recv, args)   # 既存トレースへジャンプ
  else
    # まだ無ければインライン展開を打ち切り、呼び出し先入口の
    # jit_merge_point に別トレースの記録を委ねる
    dispatch_at(callee.entry)
  end
```

> [!NOTE]
> ここでの `dispatch_at` まわりは**実装の詳細を単純化**しています。
> 実際の RPython では、トレースを始める/別トレースへ切り出す判断は
> 呼び出し先入口の `jit_merge_point` が担い、`can_enter_jit`（前述の
> バックエッジのホットさ計数）とは役割が分かれています。

ここで肝になるのが、`callee.entry`（呼び出し先メソッドの命令位置）を
**green として扱う**ことです。これにより「同じメソッドのトレースには
合流する」が成り立ち、再帰の各段が同じ 1 つのトレースへ収束します。
前章[トレーシング](tracing.md)で「ループ先頭で合流させる」と言った
仕掛けが、メタレベルでは「**インタプリタが管理するメソッド/ループの
入口を green にする**」という形で再現されているのです。

> [!IMPORTANT]
> メタトレーシングは魔法ではなく、**トレーシング JIT の性質を一段
> 持ち上げた**ものです。前章で学んだガード・サイドトレース・トレース木・
> 再帰の打ち切りは、すべて「インタプリタをトレースする」文脈で再び
> 現れます。逆にいえば、**前章を理解していれば、メタトレーシングの
> 振る舞いはほぼ予測できる**ということです。

## メソッド版もある：部分評価という近縁

メタトレーシングは「インタプリタを**トレース**して JIT を得る」方式
でした。実は、もう一つの道があります。「インタプリタを**部分評価**して
JIT を得る」方式 —— こちらはメソッド JIT に対応し、**Truffle/GraalVM**が
代表です[](#cite:wurthinger2017)。トレースするか部分評価するかの違いは
ありますが、**「言語実装者はインタプリタを書くだけ、最適化はフレーム
ワークが担う」**という核心思想は共通です。

両者の違いを、いままで見てきたヒントを軸に一段だけ具体化しておきます。
**メタトレーシング**は、実際にインタプリタを走らせ、**通った 1 本の
経路を記録**してから最適化します。だから `promote` で「**いま**通った
クラス」を定数化し、通らなかった `case` の枝は最初から記録されません。
これに対し **Truffle の部分評価**は、しばらく実行して観測で温まった
**インタプリタのコードそのもの**（正確には、特殊化済みの AST ノード木）を、
実行時にまとめて部分評価します。経路を 1 本に絞って記録するのではなく、
**ノードを跨いで定数畳み込み・インライン展開を行き渡らせて**コンパイル
単位を作るのです。

ヒントの対応も見ておくと理解が早いです。Truffle では `promote` に
あたる役割を**ノードの自己書き換え**（観測した型に特殊化したノードへ
差し替える）や `@CompilationFinal`（一度決まれば定数とみなせるフィールドの印）
が担います。`elidable` のような純粋な補助関数の畳み込みは、部分評価が
副作用のない処理をインライン化・定数畳み込みできる構造にしておくことで
実現します。`virtualizable` には**フレームのエスケープ解析**が対応します。
**「同じ着想を、トレースで実現するか、部分評価器で実現するか」**という対比です。

| 観点 | メタトレーシング (RPython/PyPy) | 部分評価 (Truffle/Graal) |
| --- | --- | --- |
| 単位 | 通った 1 経路（トレース） | メソッド/ノード木 |
| 特殊化の鍵 | green 変数・`promote` | 自己書き換えノード・`@CompilationFinal`・Assumption（後述） |
| 分岐 | サイドトレースで木に育てる | 枝を含めてまとめてコンパイル |
| 弱点 | 分岐がばらつくコードに弱い | 投機の前提作りが手間 |

どちらが優れているかは一概に言えません。**分岐の少ないホットループ**は
トレーシングが素直に強く、**分岐が入り組んだコード**は経路を 1 本に
絞らない部分評価が有利、という傾向があります。この部分評価による道は、
理論的なルーツ（Futamura 射影）も含めて、次の第 II 部
[部分評価と Futamura 射影](research-partial-eval.md)でじっくり扱います。
そこでは「**なぜインタプリタを部分評価するとコンパイラになるのか**」を、
[](#cite:futamura1971)の射影に立ち返って説明します。

## まとめ

- メタトレーシングは、**ユーザプログラムでなくインタプリタをトレース**
  する。一段ずらすことで、言語ごとの JIT を書かずに済ませる。
- トレース中、インタプリタの `pc` やディスパッチは**定数畳み込みで消え**、
  ユーザの計算だけが残る。これが速さの源。
- **RPython/PyPy** が実用化。実装者はインタプリタを RPython で書くだけで、
  メタトレーシング JIT が自動で付く。
- 良いトレースには**ヒント**を実装者が配置する。`jit_merge_point` に渡す
  変数は **green（`pc` など定数の鍵）／red（流れる値）**に分け、green の
  選択が性能を左右する。
- **`promote`** は値を「いま定数」にして特殊化（外れると**脱最適化**）、
  **`elidable`** は純粋関数を結果ごと畳み込む（副作用に付けると壊れる）、
  **`virtualizable`** はフレーム確保をスカラ置換で消す。
- トレーシングの長所・短所（ガード・サイドトレース・再帰の打ち切り）は
  **メタレベルにそのまま継承**される。
- インタプリタが唯一の正典になるので、JIT との意味のずれが起きにくい。
- 近縁に、部分評価でメソッド JIT を得る **Truffle/Graal** がある（次部）。

ここまでで「何を・どう」コンパイルするかが揃いました。次は、これらを
**速い段と遅い段の組み合わせ**として運用する仕組み —— 階層的 JIT に
進みます。
