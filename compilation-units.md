# コンパイルの単位：メソッド・トレース・基本ブロック

前章では「式 1 本」を機械語にしました。しかし本物のプログラムは、
メソッドが呼び合い、ループが回り、分岐が枝分かれします。JIT を設計する
とき、最初に決めるべき大きな問いは **「どのかたまりを 1 回のコンパイルの
単位にするか」** です。この選択が、JIT 全体の性格を決めます。

主な選択肢は三つです。**メソッド単位**（メソッド JIT）、**実行経路単位**
（トレーシング JIT）、そして**基本ブロック単位**（基本ブロックバージョニング）。
順に見ていきましょう。

## 共通の前提：基本ブロックと制御フロー

どの方式を理解するにも、**基本ブロック**（basic block）という言葉が
必要です。基本ブロックとは、**途中に分岐の出入りがない、一直線に実行
される命令の並び**のことです。先頭から入ったら、必ず最後まで実行されて
末尾から出ます。分岐（`if` やループ）が来ると、そこでブロックが切れます。

```ruby
def abs_diff(a, b)
  if a > b          # ← ここで分岐。ブロックが切れる
    a - b           # ブロック B1
  else
    b - a           # ブロック B2
  end               # B1, B2 はここ（ブロック B3）へ合流する
end
```

このメソッドは、`a > b` を判定するブロック、真のときのブロック、偽の
ときのブロック、合流後のブロック…という小さな直線の集まりです。
ブロックどうしを「どちらへ進むか」の矢印でつないだ図を**制御フローグラフ**
（control flow graph、CFG）と呼びます。JIT の各方式は、この CFG の
**どの部分をまとめてコンパイルするか**が違うのだ、と捉えると見通しが
よくなります。

## メソッド JIT：メソッドまるごとを単位にする

いちばん素直なのが、**メソッド（関数）を 1 つの単位としてコンパイルする**
方式です。「このメソッドは熱い」と判定したら、そのメソッドの CFG 全体を
機械語にします。JVM の HotSpot、V8 の TurboFan、Ruby の YJIT など、
多くの主要 JIT がこの系統です。

メソッド JIT の利点は、境界が分かりやすいことです。メソッドは元から
プログラムの自然な単位で、入口と出口がはっきりしています。プロファイル
（呼び出し回数）も取りやすい。

弱点は、**実際に通らない経路まで含んでしまう**ことです。メソッドの中に
めったに起きないエラー処理の分岐があっても、メソッドをまるごとコンパイル
するなら、その死んだ枝も機械語にしなければなりません。また、ループが
別のメソッドの呼び出しをまたいでいると、ループという「本当に熱い単位」が
メソッドの壁で分断されてしまいます。

```ruby
# 熱いループが2つのメソッドにまたがる例
def hot_loop(n)
  sum = 0
  n.times { |i| sum = step(sum, i) }   # ループ本体は step を呼ぶ
  sum
end

def step(acc, i) = acc + i * i          # 本当に熱いのは step を含む経路全体
```

メソッド JIT はこれを「`hot_loop` をコンパイルし、`step` も別途
コンパイルし、ホットなら[インライン化](why-jit.md)で `step` を `hot_loop` に
取り込む」と解きます。インライン化がメソッド JIT の生命線である理由が
ここにあります。

## トレーシング JIT：実際に通った経路を単位にする

発想を変えてみましょう。プログラムの構造（メソッドやループ）ではなく、
**実行が実際にたどった 1 本の道**をコンパイルの単位にしたらどうか。
これが**トレーシング JIT**です。

熱いループを見つけたら、その**ループを 1 周する間に実行された命令を、
分岐をまたいでも、メソッド呼び出しをまたいでも、ぜんぶ順番に記録**します。
この記録された一直線の命令列を**トレース**（trace）と呼びます。トレースは
「今回たまたま通った道」なので、分岐のところでは「**今回はこっちへ進んだ**」
という事実が含まれます。

```
（記録されたトレースのイメージ：ループ1周ぶん）
  i < n が真だった           → ガード: i < n でなければ脱出
  sum を読む
  step へ入った（呼び出しを越えて記録が続く）
  acc + i*i を計算
  step から戻った
  sum へ書き戻す
  i += 1
  ループ先頭へ戻る
```

ここで重要なのが**ガード**です。「今回 `i < n` が真だった」という前提で
一直線のコードを作るので、もし将来 `i < n` が偽になったら、この一直線の
トレースから抜け出さなければなりません。だから分岐のあった場所には
「前提どおりか確かめ、違えばトレースを脱出する」ガードを置きます。

トレーシング JIT の強みは明快です。

- **実際に通った道だけ**をコンパイルするので、死んだ枝に手間をかけない。
- メソッドの壁を越えてトレースが伸びるので、**インライン化が自動的に
  起こる**。`step` の中身もトレースに展開済みです。
- 一直線なので最適化が単純（直線コードの最適化は分岐のある CFG より
  ずっと簡単）。

弱点もあります。分岐の多いコード（同じループでも毎回違う枝を通る）では、
トレースが枝分かれして**トレースの木**が膨らみ、管理が難しくなります。
また「どこからトレースを始め、どこで止めるか」の判断が繊細です。
この難しさこそがトレーシング JIT の肝で、[トレーシングJITの仕組み](tracing.md)で
じっくり扱います。歴史的には、組み込み向けの HotpathVM[](#cite:gal2006) や、
JavaScript 向けの TraceMonkey[](#cite:gal2009) が先駆けで、
[LuaJIT](case-luajit.md) と [PyPy](case-python.md) が到達点です。

> [!NOTE]
> トレーシングの源流は、JIT ですらない **Dynamo**[](#cite:bala2000) という
> 研究にあります。Dynamo は、すでに機械語になっているプログラムを
> 実行しながら、熱い経路（トレース）を見つけて最適化し直す
> 「動的最適化器」でした。「実行経路を単位に最適化する」というトレーシングの
> 核心は、ここで生まれました。

## メソッド JIT 対トレーシング JIT

両者の違いを整理します。

| 観点 | メソッド JIT | トレーシング JIT |
|------|-------------|------------------|
| コンパイル単位 | メソッドの CFG 全体 | 実際に通った 1 本の経路 |
| 分岐の扱い | 全枝をコンパイル | 通った枝のみ＋ガード |
| インライン化 | 明示的に行う（要判断） | 経路をたどる過程で自動的に |
| 得意なコード | 分岐が散らばる一般のコード | 素直なループ |
| 苦手なコード | 巨大メソッド、深い呼び出し | 分岐が毎回変わるループ |
| 代表例 | HotSpot, V8, YJIT | LuaJIT, PyPy, TraceMonkey |

どちらが優れているかは、対象言語とワークロード次第です。2000 年代後半は
トレーシングが大いに期待されましたが、その後 JavaScript の主要エンジンは
メソッド JIT へ回帰しました。一方で PyPy と LuaJIT はトレーシングで
高い成果を上げ続けています。「勝者」はおらず、**設計上のトレードオフ**だと
理解するのが正確です。この歴史的な揺り戻しは[ケーススタディ](part-cases.md)の
各章で具体的に見えてきます。

## 基本ブロックバージョニング：その中間

第三の道があります。メソッドより細かく、トレースより構造的な、**基本
ブロック**を単位にする方式 —— **基本ブロックバージョニング**（basic
block versioning、BBV）です[](#cite:chevalier2015)。Ruby の
[YJIT](case-ruby.md) が採用して一躍有名になりました。

BBV のアイデアはこうです。基本ブロックをコンパイルするとき、**そのブロックに
入ってくる時点で分かっている型の情報ごとに、別バージョンの機械語を生成
する**。同じソース上のブロックでも、「`a` が整数として入ってくる版」と
「`a` がオブジェクトとして入ってくる版」を別々に作るのです。

なぜそれが嬉しいか。型が分かっている版では、その型のガードを**もう置か
なくてよい**からです。一度上流で「`a` は整数だ」と確かめたら、その先の
ブロックは整数版を使うので、`a` の型チェックを繰り返さずに済みます。
型チェックの除去が、特別な解析なしに、ブロックを版分けするだけで自然に
起こる —— これが BBV の鮮やかなところです。

```ruby
# 概念図：同じブロックを「入ってくる型」で版分けする
#   x = read_some_value()
#   ─ ここで x の型をガード ─→ x は Integer と判明
#
#   以降のブロックは「x: Integer 版」を使う
#     y = x + 1     ← x は整数と分かっているので型チェック不要
#     z = y * 2     ← y も整数と伝播するので、ここも不要
#
# もし別の経路から x が String として入ってきたら、
# 「x: String 版」のブロックが（必要になった時点で）別に作られる
```

BBV はメソッド JIT の一種ですが、コンパイルの粒度をブロックまで下げ、
**型情報の流れに沿って版を増やす**ことで、トレーシングに近い「実際に
起きた型に特化する」効果を、構造を保ったまま得ます。詳しい挙動は
[Ruby のケーススタディ](case-ruby.md)で YJIT として再訪します。

## そのほかの単位

実用の処理系は、これらをきれいに分けず混ぜて使います。

- **領域（region）ベース**：複数の基本ブロックをまとめた、メソッドより
  小さくトレースより大きい「領域」を単位にする折衷案。IBM の JIT
  [](#cite:suganuma2000)などが採った方式です。
- **ループ単位**：メソッドの中でも特にループ本体だけを切り出して
  コンパイルする。後述の **OSR**（[On-Stack Replacement](deopt-osr.md)）と
  組み合わせ、「すでに回り始めた長いループ」を途中から機械語に乗せ換える
  ために使われます。
- **命令単位（特殊化インタプリタ）**：そもそも機械語まで行かず、
  バイトコードの 1 命令を、観測した型に特化した別バイトコードへ
  差し替えるだけ、という最も軽い「単位」もあります。CPython 3.11 の
  特殊化適応型インタプリタ[](#cite:pep659)がこれで、JIT の一歩手前の
  軽量な特殊化として[プロファイリング](profiling.md)の章で扱います。

> [!IMPORTANT]
> 「コンパイルの単位」は、粒度のスペクトルとして眺めると整理できます。
> **命令 → 基本ブロック → トレース → 領域 → メソッド** と粒度が上がる
> ほど、1 回のコンパイルで取れる最適化の見通しは広がりますが、その分
> コンパイルは重く、投機が外れたときの巻き戻しも大がかりになります。
> どの粒度を選ぶかは、結局[なぜ JIT か](why-jit.md)で見た損得勘定の
> 別の顔です。

## まとめ

- JIT 設計の最初の分岐点は「**何を単位にコンパイルするか**」。
- **メソッド JIT**：メソッドの CFG 全体。境界が明快で、インライン化が
  生命線。主流派。
- **トレーシング JIT**：実際に通った 1 本の経路＋ガード。インライン化が
  自動で起こり、直線最適化が効く。LuaJIT・PyPy が代表。
- **基本ブロックバージョニング**：ブロックを入口の型ごとに版分けし、
  型チェック除去を自然に得る。YJIT が採用。
- これらは粒度のスペクトル（命令→ブロック→トレース→領域→メソッド）の
  上の選択であり、優劣でなくトレードオフ。

単位が決まったら、その単位を**どう表現して最適化するか**が次の問題です。
機械語を直接いじるのは大変なので、JIT は中間表現を経由します。次章へ。
