# コンパイルの単位：メソッド・トレース・基本ブロック

前章では「式 1 本」を機械語にしました。しかし本物のプログラムは、
メソッドが呼び合い、ループが回り、分岐が枝分かれします。JIT を設計する
とき、最初に決めるべき大きな問いは **「どのかたまりを 1 回のコンパイルの
単位にするか」** です。この選択が、JIT 全体の性格を決めます。

主な選択肢は三つです。**メソッド単位**（メソッド JIT）、**実行経路単位**
（トレーシング JIT）、そして**基本ブロック単位**（基本ブロックバージョニング）。
順に見ていきましょう。

## 共通の前提：基本ブロックと制御フロー

どの方式を理解するにも、**基本ブロック**（basic block）という言葉が
必要です。基本ブロックとは、**途中に分岐の出入りがない、一直線に実行
される命令の並び**のことです。先頭から入ったら、必ず最後まで実行されて
末尾から出ます。分岐（`if` やループ）が来ると、そこでブロックが切れます。

```ruby
def abs_diff(a, b)
  if a > b          # ← ここで分岐。ブロックが切れる
    a - b           # ブロック B1
  else
    b - a           # ブロック B2
  end               # B1, B2 はここ（ブロック B3）へ合流する
end
```

このメソッドは、`a > b` を判定するブロック、真のときのブロック、偽の
ときのブロック、合流後のブロック…という小さな直線の集まりです。
ブロックどうしを「どちらへ進むか」の矢印でつないだ図を**制御フローグラフ**
（control flow graph、CFG）と呼びます。JIT の各方式は、この CFG の
**どの部分をまとめてコンパイルするか**が違うのだ、と捉えると見通しが
よくなります[](#cite:aho2006)。

本物のバイトコードから基本ブロックを切り出す手続きは、実はとても機械的です。
「ここから新しいブロックが始まる」という命令（**リーダー**、leader）を
拾い、次のリーダーの直前までを 1 ブロックにするだけです。リーダーになる
のは、(1) 先頭命令、(2) 分岐命令の飛び先、(3) 分岐命令の直後の命令、の
三種類です。小さなおもちゃで切り分けてみましょう。

```ruby
# スタックマシン風の架空バイトコード列を、基本ブロックに分割する
bytecode = [
  [0, :load,   :a],
  [1, :load,   :b],
  [2, :gt],            # a > b
  [3, :branch, 7],     # 偽なら 7 へ（真なら 4 へ落ちる）
  [4, :load,   :a],
  [5, :load,   :b],
  [6, :sub],           # a - b
  [7, :ret],
]

def split_blocks(bytecode)
  jump_targets = bytecode.each_with_object([]) do |(pc, op, arg), acc|
    acc << arg if op == :branch       # (2) 飛び先
    acc << pc + 1 if op == :branch     # (3) 分岐の直後
  end
  leaders = ([0] + jump_targets).uniq.sort   # (1) 先頭 + (2) + (3)

  blocks, current = [], []
  bytecode.each do |insn|
    pc = insn[0]
    if leaders.include?(pc) && !current.empty?
      blocks << current
      current = []
    end
    current << insn
  end
  blocks << current unless current.empty?
  blocks
end

split_blocks(bytecode).each_with_index do |b, i|
  puts "B#{i}: pc #{b.first[0]}..#{b.last[0]}"
end
# => B0: pc 0..3   （比較して分岐するブロック）
#    B1: pc 4..6   （a - b を計算するブロック）
#    B2: pc 7..7   （戻るブロック。B0 と B1 の合流先）
```

このように、リーダーを見つけて区切るだけで CFG の骨格ができます。
JIT の各方式は、こうして得たブロック群を **何個ずつ束ねて 1 回のコンパイル
にかけるか** を選んでいるにすぎません。1 ブロックなら基本ブロック単位、
通った道に沿って数珠つなぎにすればトレース、メソッド内の全ブロックを
束ねればメソッド単位、というわけです。

## メソッド JIT：メソッドまるごとを単位にする

いちばん素直なのが、**メソッド（関数）を 1 つの単位としてコンパイルする**
方式です。「このメソッドは熱い」と判定したら、そのメソッドの CFG 全体を
機械語にします。JVM の HotSpot、V8 の TurboFan、Ruby の YJIT など、
多くの主要 JIT がこの系統です。

メソッド JIT の利点は、境界が分かりやすいことです。メソッドは元から
プログラムの自然な単位で、入口と出口がはっきりしています。プロファイル
（呼び出し回数）も取りやすい。

弱点は、**実際に通らない経路まで含んでしまう**ことです。メソッドの中に
めったに起きないエラー処理の分岐があっても、メソッドをまるごとコンパイル
するなら、その死んだ枝も機械語にしなければなりません。また、ループが
別のメソッドの呼び出しをまたいでいると、ループという「本当に熱い単位」が
メソッドの壁で分断されてしまいます。

```ruby
# 熱いループが2つのメソッドにまたがる例
def hot_loop(n)
  sum = 0
  n.times { |i| sum = step(sum, i) }   # ループ本体は step を呼ぶ
  sum
end

def step(acc, i) = acc + i * i          # 本当に熱いのは step を含む経路全体
```

メソッド JIT はこれを「`hot_loop` をコンパイルし、`step` も別途
コンパイルし、ホットなら[インライン化](why-jit.md)で `step` を `hot_loop` に
取り込む」と解きます。インライン化がメソッド JIT の生命線である理由が
ここにあります。

### インライン化を「するか・しないか」を例えばこう決める

インライン化は[最適化の母](why-jit.md)と呼ばれます。呼び出し先を呼び出し元へ
展開すると、呼び出しの手間（引数の積み込み・戻り先の保存）が消えるだけで
なく、その先の定数畳み込み・冗長除去・型特殊化がいっせいに効くからです。
とはいえ何でも展開すると、コードが膨れてかえって遅くなります。**「例えば
こう決める」** という具体的な判断材料を、おもちゃで形にしてみましょう。

```ruby
# 呼び出し先をインライン化すべきか判断するヒューリスティック（概念実装）
InlineDecision = Struct.new(:inline?, :reason)

def should_inline?(callee, ctx)
  # ctx: 呼び出し地点の状況（呼び出し頻度、再帰深さ、現在のコード膨張量など）

  # (1) 小さすぎるものは「ほぼ無条件で」展開する。
  #     アクセサ（@x を返すだけ）は本体より呼び出しの方が高くつく。
  return InlineDecision.new(true, :tiny) if callee.bytecode_size <= 5

  # (2) 大きすぎるものは展開しない。閾値はバイトコード長で例えば 100 程度。
  return InlineDecision.new(false, :too_big) if callee.bytecode_size > 100

  # (3) めったに呼ばれない呼び出し地点は展開しない。
  #     「この呼び出しがメソッド全体の何%を占めるか」で見るのが定石。
  return InlineDecision.new(false, :cold) if ctx.call_share < 0.01

  # (4) 再帰は深さに上限を設ける。例えば 2 段まで展開して、それ以上は呼び出しに残す。
  return InlineDecision.new(false, :recursion_cap) if ctx.recursion_depth >= 2

  # (5) すでに展開しすぎて膨張予算を使い切ったら止める。
  return InlineDecision.new(false, :budget) if ctx.inlined_bytes + callee.bytecode_size > 1000

  # (6) 多態な呼び出しは、観測した上位クラスだけ展開し、残りはガード経由で呼び出しに落とす。
  if ctx.observed_classes.size > 3
    return InlineDecision.new(false, :too_polymorphic)
  end

  InlineDecision.new(true, :ok)
end
```

数値感をつかむための目安です。アクセサ級（数命令）は呼び出しコストの方が
大きいので**ほぼ常に展開**、中くらい（数十命令）は**頻度しだい**、巨大な
ものは**まず展開しない**。再帰は 1〜2 段だけ展開して残りはループや呼び出しに
残す、というのが実用的な落とし所です。「メソッド全体で膨張は何倍まで許すか」
という**インライン化予算**（例：元のコードの数倍まで）を持ち、予算を使い
切ったら以降は展開を諦める、という設計が一般的です。

多態な呼び出しの扱いはとくに重要です。同じ `obj.area` でも、観測上
`Circle` が 90%、`Square` が 9%、それ以外が 1% なら、**上位 2 クラスだけ
インライン化**し、ガードに外れた残り 1% は通常の呼び出しへ落とします。
これを**多態インラインキャッシュ**を使ったインライン化と呼び、SELF
[](#cite:holzle1994)以来の古典的な手法です[](#cite:detlefs1999)。

```ruby
# 多態な obj.area を「上位クラスだけ展開」したイメージ（疑似コード）
case obj.class
when Circle   # 観測 90% → ここはインライン展開済み
  r = obj.instance_variable_get(:@r); 3.14159 * r * r
when Square   # 観測 9%  → ここもインライン展開済み
  s = obj.instance_variable_get(:@s); s * s
else          # 観測 1%  → 展開せず、ふつうの動的呼び出しに落とす
  obj.area
end
```

> [!TIP]
> インライン化の判断は「呼び出し先の素性」だけでなく「呼び出し**地点**の
> 熱さ」で決めるのがコツです。同じ小さなメソッドでも、毎周回呼ばれる
> ループ内なら展開する価値が大きく、初期化で一度きりなら展開しても
> リターンがありません。だから判断には呼び出し地点の頻度（プロファイル）が
> 要ります。詳しくは[プロファイリング](profiling.md)で扱います。

## トレーシング JIT：実際に通った経路を単位にする

発想を変えてみましょう。プログラムの構造（メソッドやループ）ではなく、
**実行が実際にたどった 1 本の道**をコンパイルの単位にしたらどうか。
これが**トレーシング JIT**です。

熱いループを見つけたら、その**ループを 1 周する間に実行された命令を、
分岐をまたいでも、メソッド呼び出しをまたいでも、ぜんぶ順番に記録**します。
この記録された一直線の命令列を**トレース**（trace）と呼びます。トレースは
「今回たまたま通った道」なので、分岐のところでは「**今回はこっちへ進んだ**」
という事実が含まれます。

```
（記録されたトレースのイメージ：ループ1周ぶん）
  i < n が真だった           → ガード: i < n でなければ脱出
  sum を読む
  step へ入った（呼び出しを越えて記録が続く）
  acc + i*i を計算
  step から戻った
  sum へ書き戻す
  i += 1
  ループ先頭へ戻る
```

ここで重要なのが**ガード**です。「今回 `i < n` が真だった」という前提で
一直線のコードを作るので、もし将来 `i < n` が偽になったら、この一直線の
トレースから抜け出さなければなりません。だから分岐のあった場所には
「前提どおりか確かめ、違えばトレースを脱出する」ガードを置きます。

トレーシング JIT の強みは明快です。

- **実際に通った道だけ**をコンパイルするので、死んだ枝に手間をかけない。
- メソッドの壁を越えてトレースが伸びるので、**インライン化が自動的に
  起こる**。`step` の中身もトレースに展開済みです。
- 一直線なので最適化が単純（直線コードの最適化は分岐のある CFG より
  ずっと簡単）。

弱点もあります。分岐の多いコード（同じループでも毎回違う枝を通る）では、
トレースが枝分かれして**トレースの木**が膨らみ、管理が難しくなります。
また「どこからトレースを始め、どこで止めるか」の判断が繊細です。

「どこから始め、どこで止めるか」を、もう一段具体に書いてみましょう。
トレースを**始める**起点は、典型的には次の三つです。

- **ループヘッダ**：あるループの先頭を何回も通ったら（例：1000 回など、
  しきい値を超えたら）熱いとみなし、そこを起点に記録を始めます。
  もっとも一般的な起点です。
- **関数（メソッド）入口**：ループを持たないが何度も呼ばれるメソッドの
  入口を起点にする方式です。
- **サイドイグジット**：既存のトレースのガードが頻繁に外れる地点
  （**サイドイグジット**、side exit）を新たな起点にし、外れた先の道を
  別トレースとして足す。これでトレースの木を育てます。

トレースを**止める**条件も、暴走を防ぐために具体的な上限で縛ります。

- **起点に戻ってきた**：ループ 1 周を記録し終え、出発点のループヘッダに
  戻ってきたら、そこで閉じて「ループの環」を作ります（理想的な停止）。
- **長さ上限**：記録した命令が長くなりすぎたら（例：数千命令で）打ち切る。
  際限なく伸びるとコンパイルも巻き戻しも重くなるためです。
- **呼び出し深さ上限**：トレースは呼び出しをまたいで伸びますが、深い再帰や
  深い呼び出しで無限に潜らないよう、深さに上限（例：十数段）を設けます。

```ruby
# トレースを記録するレコーダの「開始・停止」だけを取り出したおもちゃ
class TraceRecorder
  MAX_LEN = 3000      # 長さ上限
  MAX_DEPTH = 16      # 呼び出し深さ上限

  def initialize(anchor)  # anchor: 起点（このループヘッダに戻れば閉じる）
    @anchor = anchor
    @insns = []
    @depth = 0
  end

  def record(insn)
    return :abort_too_long if @insns.size >= MAX_LEN
    @depth += 1 if insn.op == :call
    @depth -= 1 if insn.op == :ret
    return :abort_too_deep if @depth > TraceRecorder::MAX_DEPTH

    @insns << insn
    # 起点のループヘッダに戻ってきたら、その時点で環として閉じる
    if insn.op == :loop_header && insn.target == @anchor && @insns.size > 1
      return :complete
    end
    :continue
  end
end
```

止め方を誤ると、トレースが長くなりすぎてコンパイル費用が膨らんだり、
逆に短すぎてループ全体を覆えなかったりします。この匙加減こそが
トレーシング JIT の肝で、[トレーシングJITの仕組み](tracing.md)で
じっくり扱います。歴史的には、組み込み向けの HotpathVM[](#cite:gal2006) や、
JavaScript 向けの TraceMonkey[](#cite:gal2009) が先駆けで、
[LuaJIT](case-luajit.md) と [PyPy](case-python.md) が到達点です。

> [!NOTE]
> トレーシングの源流は、JIT ですらない **Dynamo**[](#cite:bala2000) という
> 研究にあります。Dynamo は、すでに機械語になっているプログラムを
> 実行しながら、熱い経路（トレース）を見つけて最適化し直す
> 「動的最適化器」でした。「実行経路を単位に最適化する」というトレーシングの
> 核心は、ここで生まれました。

## メソッド JIT 対トレーシング JIT

両者の違いを整理します。

| 観点 | メソッド JIT | トレーシング JIT |
|------|-------------|------------------|
| コンパイル単位 | メソッドの CFG 全体 | 実際に通った 1 本の経路 |
| 分岐の扱い | 全枝をコンパイル | 通った枝のみ＋ガード |
| インライン化 | 明示的に行う（要判断） | 経路をたどる過程で自動的に |
| 得意なコード | 分岐が散らばる一般のコード | 素直なループ |
| 苦手なコード | 巨大メソッド、深い呼び出し | 分岐が毎回変わるループ |
| 代表例 | HotSpot, V8, YJIT | LuaJIT, PyPy, TraceMonkey |

どちらが優れているかは、対象言語とワークロード次第です。2000 年代後半は
トレーシングが大いに期待されましたが、その後 JavaScript の主要エンジンは
メソッド JIT へ回帰しました。一方で PyPy と LuaJIT はトレーシングで
高い成果を上げ続けています。「勝者」はおらず、**設計上のトレードオフ**だと
理解するのが正確です。この歴史的な揺り戻しは[ケーススタディ](part-cases.md)の
各章で具体的に見えてきます。

## 基本ブロックバージョニング：その中間

第三の道があります。メソッドより細かく、トレースより構造的な、**基本
ブロック**を単位にする方式 —— **基本ブロックバージョニング**（basic
block versioning、BBV）です[](#cite:chevalier2015)。Ruby の
[YJIT](case-ruby.md) が採用して一躍有名になりました。

BBV のアイデアはこうです。基本ブロックをコンパイルするとき、**そのブロックに
入ってくる時点で分かっている型の情報ごとに、別バージョンの機械語を生成
する**。同じソース上のブロックでも、「`a` が整数として入ってくる版」と
「`a` がオブジェクトとして入ってくる版」を別々に作るのです。

なぜそれが嬉しいか。型が分かっている版では、その型のガードを**もう置か
なくてよい**からです。一度上流で「`a` は整数だ」と確かめたら、その先の
ブロックは整数版を使うので、`a` の型チェックを繰り返さずに済みます。
型チェックの除去が、特別な解析なしに、ブロックを版分けするだけで自然に
起こる —— これが BBV の鮮やかなところです。

```ruby
# 概念図：同じブロックを「入ってくる型」で版分けする
#   x = read_some_value()
#   ─ ここで x の型をガード ─→ x は Integer と判明
#
#   以降のブロックは「x: Integer 版」を使う
#     y = x + 1     ← x は整数と分かっているので型チェック不要
#     z = y * 2     ← y も整数と伝播するので、ここも不要
#
# もし別の経路から x が String として入ってきたら、
# 「x: String 版」のブロックが（必要になった時点で）別に作られる
```

### BBV の動くおもちゃ：型コンテキストで版をキャッシュする

言葉だけでは腑に落ちないので、BBV の核心を小さな Ruby で動かしてみましょう。
キモは三つです。**(1)** ブロックの版を「入口で分かっている型の集合
（**型コンテキスト**、type context）」をキーにキャッシュする。**(2)** 初めて
見る型コンテキストなら新しい版を生成する。**(3)** すでに作った版があれば
それを使い回す。型が分かっている版では**ガードを置かない**＝型チェックが
自然に消える、という様子を出力で確認します。

```ruby
# x + 1 を計算する 1 ブロックを、入口の型コンテキストごとに版分けする
class BBVCompiler
  VERSION_CAP = 4   # 1 ブロックあたりの版の上限（版の爆発を防ぐ）

  def initialize
    @versions = Hash.new { |h, k| h[k] = {} }  # block_id => { type_ctx => code }
  end

  # block_id のブロックを、入口の型コンテキスト type_ctx 向けに取得する
  def code_for(block_id, type_ctx)
    table = @versions[block_id]

    # (3) すでにこの型コンテキストの版があれば使い回す（コンパイル不要）
    return table[type_ctx] if table.key?(type_ctx)

    # 版の上限を超えそうなら、型を限定しない「一般版」へ落とす
    if table.size >= VERSION_CAP
      return (table[:generic] ||= compile(block_id, :generic))
    end

    # (2) 初出の型コンテキスト → 新しい特殊化版を生成してキャッシュ
    table[type_ctx] = compile(block_id, type_ctx)
  end

  private

  def compile(block_id, type_ctx)
    if type_ctx == :generic
      # 型が不明 → 実行時に毎回ガード（型チェック）が要る一般版
      "GENERIC: guard(x.is_a? Integer); push(x + 1)"
    elsif type_ctx[:x] == :Integer
      # x は整数と判明 → ガードなしの特化版。型チェックが消える！
      "SPECIAL[x:Integer]: add_int(x, 1)   # no guard"
    else
      "SPECIAL[#{type_ctx.inspect}]: generic_add(x, 1); guard?"
    end
  end
end

c = BBVCompiler.new
puts c.code_for(:blkA, { x: :Integer })  # 初出 → 特化版を生成
puts c.code_for(:blkA, { x: :Integer })  # 既出 → 同じ版を使い回す
puts c.code_for(:blkA, { x: :Float })    # 別の型 → 別の版を生成
# => SPECIAL[x:Integer]: add_int(x, 1)   # no guard
#    SPECIAL[x:Integer]: add_int(x, 1)   # no guard   ← 再コンパイルなし
#    SPECIAL[{:x=>:Float}]: generic_add(x, 1); guard?
```

注目すべきは `x: Integer` 版に **ガードが現れない** ことです。上流のブロックで
一度 `x` が整数だと確かめてあれば、この版に入ってきた時点で型は保証済み
なので、ブロックの中で型チェックを繰り返す必要がありません。特別な型推論を
回さなくても、**版を分けるだけで型チェックが自然消滅する**のが BBV の妙です
[](#cite:chevalier2015)。

### 型が下流へ伝播する様子

もう一歩進めて、ガードを通った後に型情報が**下流のブロックへ伝播**する
様子を見ましょう。ガードが「ここで `x` は整数だ」と確定させると、その先の
ブロックは整数版で生成され、さらにその先も整数版になります。

```ruby
# x = read();  y = x + 1;  z = y * 2  を 3 ブロックに分け、型を伝播させる
ctx0 = {}                                   # 入口：x の型は未知
# B0: read() の結果にガード → x は Integer と判明
ctx1 = ctx0.merge(x: :Integer)              # B1 へ渡る型コンテキスト
# B1: y = x + 1。x:Integer なのでガード不要。+ の結果 y も Integer と分かる
ctx2 = ctx1.merge(y: :Integer)              # B2 へ渡る型コンテキスト
# B2: z = y * 2。y:Integer なので、ここもガード不要

[ctx1, ctx2].each_with_index do |ctx, i|
  puts "B#{i+1} 版の入口型コンテキスト: #{ctx.inspect}  → ガード: なし"
end
# => B1 版の入口型コンテキスト: {:x=>:Integer}            → ガード: なし
#    B2 版の入口型コンテキスト: {:x=>:Integer, :y=>:Integer} → ガード: なし
```

最初の 1 回のガードだけで、後続のブロック群から型チェックがまとめて
消えます。これが「型チェックの除去が、特別な解析なしに、ブロックを版分け
するだけで起こる」の正体です。

### 版の爆発と上限（version cap）

うまい話には代償があります。型コンテキストの組み合わせは原理上いくらでも
増えるので、放っておくと**版が爆発**します。多態なコード（同じブロックに
何十種類もの型の組が流れ込む）では、版の数が制御不能になりかねません。

だから実装は **1 ブロックあたりの版の上限**（version cap）を設けます。
上限に達したら、それ以上は型で特化せず、**一般版**（generic、型を仮定
しない＝ガードを置く版）へ落とします。上の `BBVCompiler` で
`VERSION_CAP = 4` を超えたら `:generic` を返していたのが、まさにこの安全弁です。

```ruby
c = BBVCompiler.new
# 5 種類の型コンテキストを流し込む（上限 4 を超える）
[:Integer, :Float, :String, :Array, :Hash].each do |t|
  puts c.code_for(:blkB, { x: t })
end
# => SPECIAL[...Integer...]      ← 1 版目
#    SPECIAL[...Float...]        ← 2 版目
#    SPECIAL[...String...]       ← 3 版目
#    SPECIAL[...Array...]        ← 4 版目（上限ぴったり）
#    GENERIC: guard(...); ...    ← 5 種類目は一般版に落ちる
```

数値感としては、版の上限を **数個（例：4〜5）** に抑えるのが定石です。
これでも単型・少数の多態（型が 1〜数種類）のホットコードは十分特化でき、
かつ過度に多態なコードでも版が膨れません。上限を上げれば特化の機会は
増えますが、コード量とコンパイル費用がかさみます。この **「特化の利得 vs
コード膨張」** の綱引きを、たった一つの数（version cap）で調整できるのが
BBV の運用上の利点です。

> [!WARNING]
> 版の上限は[脱最適化](deopt-osr.md)とも絡みます。一般版へ落とした
> ブロックは型を仮定しないため遅くなりますが、少なくとも**正しく動き続け**、
> 想定外の型が来てもクラッシュしません。「速い特化版」と「遅いが万能な
> 一般版」を併存させ、上限で前者の数を抑える —— これが BBV の安全設計の
> 要です。

BBV はメソッド JIT の一種ですが、コンパイルの粒度をブロックまで下げ、
**型情報の流れに沿って版を増やす**ことで、トレーシングに近い「実際に
起きた型に特化する」効果を、構造を保ったまま得ます。版を必要になった
ときだけ遅延生成するので、通らないブロックの版は作られない —— トレーシングの
「死んだ枝に手間をかけない」性質も部分的に共有します。詳しい挙動と、
基本ブロックバージョニングを型解析の枠組みとして洗練させた後続研究
[](#cite:chevalier2021)については、[Ruby のケーススタディ](case-ruby.md)で
YJIT として再訪します。

## そのほかの単位

実用の処理系は、これらをきれいに分けず混ぜて使います。

- **領域（region）ベース**：複数の基本ブロックをまとめた、メソッドより
  小さくトレースより大きい「領域」を単位にする折衷案。IBM の JIT
  [](#cite:suganuma2000)などが採った方式です。領域の選び方を一段具体に
  すると、**ホットなブロックを起点に、そこから到達できる範囲を貪欲に
  取り込み、めったに来ないブロックは除外する**、という形になります。

  ```ruby
  # ホットなブロックから到達できる範囲を貪欲に集めて「領域」を作るおまけ実装
  # cfg: block_id => 後続ブロックの配列、freq: block_id => 実行頻度
  def select_region(cfg, freq, start, threshold)
    region, queue = [], [start]
    until queue.empty?
      b = queue.shift
      next if region.include?(b)
      next if freq[b] < threshold        # めったに来ないブロックは除外
      region << b
      cfg[b].each { |succ| queue << succ } # 到達できる後続を貪欲にたどる
    end
    region
  end

  cfg  = { e: [:l], l: [:body, :exit], body: [:l], exit: [], err: [] }
  freq = { e: 1, l: 1000, body: 999, exit: 1, err: 0 }
  p select_region(cfg, freq, :l, 10)
  # => [:l, :body]   ← 熱いループだけを領域化。exit(1) や err(0) は除外される
  ```

  メソッド全体より無駄が少なく（めったに通らない枝を切れる）、トレースより構造的
  （分岐を含む領域をそのまま扱える）という、ちょうど中間の性質になります。
- **ループ単位**：メソッドの中でも特にループ本体だけを切り出して
  コンパイルする。後述の **OSR**（[On-Stack Replacement](deopt-osr.md)）と
  組み合わせ、「すでに回り始めた長いループ」を途中から機械語に乗せ換える
  ために使われます。
- **命令単位（特殊化インタプリタ）**：そもそも機械語まで行かず、
  バイトコードの 1 命令を、観測した型に特化した別バイトコードへ
  差し替えるだけ、という最も軽い「単位」もあります。CPython 3.11 の
  特殊化適応型インタプリタ[](#cite:pep659)がこれで、JIT の一歩手前の
  軽量な特殊化として[プロファイリング](profiling.md)の章で扱います。

> [!IMPORTANT]
> 「コンパイルの単位」は、粒度のスペクトルとして眺めると整理できます。
> **命令 → 基本ブロック → トレース → 領域 → メソッド** と粒度が上がる
> ほど、1 回のコンパイルで取れる最適化の見通しは広がりますが、その分
> コンパイルは重く、投機が外れたときの巻き戻しも大がかりになります。
> どの粒度を選ぶかは、結局[なぜ JIT か](why-jit.md)で見た損得勘定の
> 別の顔です。

この損得を、三つの軸 ——**コンパイル費用**（1 単位を機械語にする重さ）、
**最適化の見通し**（一度にどれだけ広く最適化できるか）、**巻き戻しコスト**
（投機が外れて[脱最適化](deopt-osr.md)するときの大変さ）—— で並べると、
粒度のスペクトルがどう変化するかが具体的に見えます。

| 粒度 | コンパイル費用 | 最適化の見通し | 巻き戻しコスト |
|------|---------------|---------------|---------------|
| 命令（特殊化） | 極小（命令 1 個を差し替えるだけ） | ほぼ局所のみ | 極小（命令を戻すだけ） |
| 基本ブロック | 小（数命令〜十数命令） | ブロック内＋型コンテキストの伝播 | 小（ブロック境界で戻れる） |
| トレース | 中（一直線の数百〜数千命令） | 直線全体に強く効く | 中（ガードからサイドイグジット） |
| 領域 | 中〜大（分岐を含む複数ブロック） | 領域内の分岐をまたいで効く | 中（領域の入口へ） |
| メソッド | 大（CFG 全体＋インライン展開） | メソッド全体・呼び出し越え | 大（[OSR](deopt-osr.md)等で慎重に） |

数値の桁感をつかむための例えをひとつ。命令単位の特殊化（CPython 3.11 の
適応型インタプリタ[](#cite:pep659)など）は、1 命令を観測した型に応じて
差し替えるだけなので、費用はほぼゼロ・巻き戻しも一瞬です。代わりに最適化は
その命令の周りだけ。対極のメソッド単位（HotSpot など）は、CFG 全体に
インライン化された呼び出し先まで含めて最適化できますが、1 回のコンパイルが
重く、想定が外れたときは実行中のスタックを安全に作り直す（OSR や
脱最適化）大仕事が要ります。基本ブロック・トレース・領域はその中間に
散らばり、**「どこまでの広さを、どれだけの費用で、どれだけ気軽に捨てられる
速度で買うか」** を選んでいるのだ、と捉えると全体が一望できます。

## まとめ

- JIT 設計の最初の分岐点は「**何を単位にコンパイルするか**」。
- **メソッド JIT**：メソッドの CFG 全体。境界が明快で、インライン化が
  生命線。主流派。
- **トレーシング JIT**：実際に通った 1 本の経路＋ガード。インライン化が
  自動で起こり、直線最適化が効く。LuaJIT・PyPy が代表。
- **基本ブロックバージョニング**：ブロックを入口の型ごとに版分けし、
  型チェック除去を自然に得る。YJIT が採用。
- これらは粒度のスペクトル（命令→ブロック→トレース→領域→メソッド）の
  上の選択であり、優劣でなくトレードオフ。

単位が決まったら、その単位を**どう表現して最適化するか**が次の問題です。
機械語を直接いじるのは大変なので、JIT は中間表現を経由します。次章へ。
