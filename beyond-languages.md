# 言語処理系の外の JIT

ここまで本書は、JavaScript や Ruby や Java といったプログラミング言語の
処理系を主役にしてきました。けれど、ある日ふと気づくのです。普段使って
いる正規表現の検索が妙に速い。SQL の集計が、行数のわりに一瞬で返る。
Linux の `tcpdump` が捌けないほどのパケットを楽々とフィルタしている。
古い x86 用のアプリが、ARM の Mac でなぜか普通に動く。これらの裏側にも、
本書で見てきたのと**同じ JIT の発想**が、ひっそりと息づいています。

この最終章では、言語処理系の外へ足を伸ばします。主張は一つです。

> **JIT とは「実行時に分かる情報を使って、汎用の解釈系を、その場限りの
> 特化した機械語に変える」という一つの発想であり、その姿は言語処理系の
> 内外、いたる所に現れる。**

ここまで身につけた語彙（[ディスパッチの消去](why-jit.md)、[特殊化](why-jit.md)、
[投機と脱最適化](deopt-osr.md)、[テンプレート/Copy-and-Patch](research-copy-and-patch.md)、
[トレーシング](tracing.md)、[検証＋JIT](research-wasm.md)）だけを携えて、
言語の外の JIT を巡りましょう。それぞれの応用について、何を実行時に
コンパイルするのかと、本書のどの技法が現れるのかを、必ず結びつけて
読んでいきます。

## パターン専用の機械語を吐く正規表現エンジン

身近な例から始めましょう。`/(\d{4})-(\d{2})-(\d{2})/` のような正規表現で
日付を探すとき、エンジンは内部で何をしているのでしょうか。

素朴な実装は、正規表現を **NFA（非決定性有限オートマトン）** や
バックトラック式の状態機械に変換し、対象文字列を 1 文字ずつ送りながら、
「いま自分は状態いくつにいるか」を見て次に進む先を毎回判断します。
これはまさに、[なぜ JIT か](why-jit.md)で見たインタプリタのディスパッチ
ループそのものです。

```
# 解釈系としての正規表現マッチャ（概念）
pc = START                     # 「いまどこにいるか」＝状態
for ch in input:
    op = pattern_program[pc]   # いまの命令を読む（フェッチ）
    switch op:                 # 何の命令か分岐する（ディスパッチ）
        case CHAR:  if ch != op.c: backtrack()
        case RANGE: if ch not in op.set: backtrack()
        case SPLIT: push_alternative(op.target)
        ...
    pc = next_pc
```

この `switch op` こそ、本書で繰り返し「消したい」と言ってきた**ディスパッチ
オーバーヘッド**です。パターンは検索のあいだずっと固定なのに、文字を
読むたびに「次は CHAR か、RANGE か、SPLIT か」を毎回問い直しています。
パターンが分かっているなら、その分岐は前もって畳めるはずです。

パターンが分かれば解釈ループを畳める。正規表現エンジンの JIT が
やることは、まさにこれです。正規表現という
「プログラム」を、その正規表現専用の機械語へコンパイルする。`\d` という
命令を解釈する代わりに、`\d` のチェックを行う機械語片を直接並べてしまう。

```
# /\d\d/ を解釈する代わりに、専用の機械語へ畳んだもの（疑似アセンブリ）
match_two_digits:
    mov   al, [rsi]        ; 1 文字目をロード
    sub   al, '0'
    cmp   al, 9
    ja    .fail            ; '0'..'9' でなければ失敗
    mov   al, [rsi+1]      ; 2 文字目をロード
    sub   al, '0'
    cmp   al, 9
    ja    .fail
    mov   rax, 1           ; マッチ成功
    ret
.fail:
    xor   rax, rax
    ret
```

命令の種類を見て飛ぶ `switch` が、一直線に並んだ比較と分岐に置き換わって
います。これは[なぜ JIT か](why-jit.md)で論じた**ディスパッチの消去**と、
[ベースライン JIT](tiered.md) が「バイトコードをほぼ一対一で機械語へ落とす」
やり方の、正規表現版です。正規表現の各オペコードに対応する機械語の
断片（テンプレート）を順に貼っていくだけなら、これは
[Copy-and-Patch](research-copy-and-patch.md)[](#cite:xu2021) の発想とも
重なります。テンプレートを並べ、文字クラスの集合や分岐先という「穴」だけを
パッチで埋めるのです。

実際の処理系を挙げましょう。

- **V8 の irregexp**：[V8](case-v8.md) の正規表現エンジン。よく使われる
  パターンを実行時にネイティブコードへコンパイルします。
- **PCRE2 の JIT**（Sljit ベース）：Perl 互換正規表現ライブラリの JIT
  モード。`pcre2_jit_compile` を呼ぶと、パターンを機械語化します。多くの
  言語処理系の正規表現の裏で使われています。
- **Rust の `regex` クレート**：こちらは少し趣が異なり、安全性を優先して
  ネイティブコード生成は避けつつ、内部で **DFA（決定性有限オートマトン）**を
  実行時に遅延構築します。状態遷移表をその場で作るのも、「実行時に対象に
  特化した表現を組み立てる」という JIT 的発想の親戚です。

> [!NOTE]
> 正規表現 JIT には、本書で論じた**コンパイルコストの損益分岐**が
> はっきり現れます。短い文字列に一度マッチさせるだけなら、パターンを
> 機械語化する手間のほうが高くつき、解釈実行のほうが速い。逆に、同じ
> パターンを長大なログに何百万回も当てるなら、ディスパッチ消去の利得が
> 圧倒的に効く。[ウォームアップ](research-warmup.md)で見た「コンパイルの
> 元を取れるか」という問いが、ここでも同じ形で現れるのです。だから多くの
> エンジンは、解釈実行を既定にし、ホットなパターンだけを JIT します。

> [!WARNING]
> 正規表現を機械語にすると、**正規表現が攻撃面になる**点には注意が要ります。
> ユーザ入力をそのままパターンとしてコンパイルする経路があると、悪意ある
> 正規表現が病的なコード（あるいは病的なバックトラック）を生む恐れがある。
> [WebAssembly](research-wasm.md) の節で見る「**検証してから JIT する**」
> という規律が、ここでも効いてきます。信頼できないものを機械語にする前には、
> 必ず素性を確かめる。これは言語処理系の外でも変わらない鉄則です。

## クエリを「プログラム」とみなしてコンパイルするデータベース

次は、もっと意外な場所、**データベース**です。`SELECT SUM(price) FROM
orders WHERE region = 'JP'` という SQL を実行するとき、何が起きているか。

古典的な実行モデルは **Volcano（火山）モデル**、別名イテレータモデルです。
クエリは「スキャン → フィルタ → 集計」といったオペレータの木に変換され、
各オペレータは `next()` という関数を持ち、上から `next()` を呼ぶと
1 行（1 タプル）が返ってくる、という仕組みで動きます。

```
# Volcano モデルの集計オペレータ（概念）
def next(self):
    while True:
        row = self.child.next()      # 下のオペレータから 1 行もらう（仮想呼び出し）
        if row is None: return self.result
        if self.predicate(row):      # 述語を「解釈」して評価
            self.acc += row[self.col]
```

ここでも[なぜ JIT か](why-jit.md)の構図がそっくり現れます。`self.child.next()`
は仮想呼び出しで、行を 1 つ取り出すたびに関数呼び出しの壁を越えます。
`self.predicate(row)` は述語（`region = 'JP'`）を実行時に解釈して
評価し、`row[self.col]` は列のオフセットを実行時に索き直します。
クエリの形は実行のあいだずっと固定なのに、行ごとに同じ解釈を繰り返している。
**これは正規表現エンジンとまったく同じ病です。**

ここで視点を変えます。SQL クエリを「プログラム」、
テーブルの中身をその「入力データ」だと見るのです。すると、毎回クエリを解釈するのは、
「プログラムを実行するたびにソースコードを読み直す」ようなもの。一度
そのクエリ専用の機械語にコンパイルしてしまえば、あとは入力データを
流し込むだけで済みます。

これはまさに[部分評価](research-partial-eval.md)の見方です。汎用の
クエリ実行器 `exec(query, data)` を、特定の `query` で部分評価して、
`data` だけを引数に取る特化済みの実行器 `exec_for_this_query(data)` を
生成する。理屈の上では、[Futamura 射影](research-partial-eval.md)が言う
「インタプリタ＋プログラム → コンパイル済みプログラム」を、データベースが
行っているわけです。生成されるコードはこうなります。

```
# 上のクエリを特化・コンパイルしたもの（概念）
def run_compiled(orders):
    acc = 0
    for row in orders:               # 仮想呼び出しが消え、直接ループ
        if row.region == 'JP':       # 述語が機械語に焼き込まれた
            acc += row.price         # 列オフセットが定数に畳まれた
    return acc
```

仮想呼び出しの連鎖が一本のタイトなループに潰れ、述語と列オフセットは
定数として焼き込まれています。具体的な実装を見ていきましょう。

- **PostgreSQL の LLVM JIT**：PostgreSQL は、式評価とタプル分解という
  特に重い部分を、実行時に [LLVM](frameworks.md)[](#cite:lattner2004) で
  ネイティブコードへコンパイルします。コスト見積もりが一定の閾値を超える
  重いクエリだけを JIT する設計で、ここにも「コンパイルの元を取れるか」の
  [損益分岐](research-warmup.md)の判断が効いています。
- **Apache Spark の whole-stage codegen**：複数のオペレータを一段に
  融合し、一つの巨大なループ関数を実行時に Java のソースとして生成・
  コンパイルします。オペレータ境界の仮想呼び出しをまとめて消すのが狙いで、
  「**オペレータ融合**」と呼ばれる発想です。
- **HyPer に代表される関係代数の特化**：研究系のデータベースは、関係代数の
  プランを LLVM IR へ直接落とし、データを CPU レジスタに載せたまま
  パイプラインを駆け抜ける、極限まで特化した機械語を生成します。この潮流を
  決定づけたのが Neumann の論文[](#cite:neumann2011)で、クエリコンパイルの
  古典になっています。

> [!TIP]
> データベースの JIT は、本書の**特化（specialization）**という言葉の
> 意味を、もっとも純粋な形で見せてくれます。言語処理系では「動的な型を
> 投機して特化する」という[投機](deopt-osr.md)が主役でしたが、SQL の型は
> スキーマで静的に確定しています。だから投機も[脱最適化](deopt-osr.md)も
> ほぼ要りません。残るのは純粋な「**定数畳み込み＋仮想呼び出しの除去**」、
> つまり[部分評価](research-partial-eval.md)そのものです。投機の機械を取り
> 払っても、なお JIT が効く。「特化」と「投機」は別の道具だと、ここで
> はっきり切り分けられます。

> [!IMPORTANT]
> 「クエリをコンパイルすると速い」ことの理由は、本書の知識で完全に説明
> できます。Volcano モデルの遅さは、行ごとの**ディスパッチ**[](#cite:aho2006)と、
> CPU から見たときの分岐予測の外れにあります。専用機械語はディスパッチを
> 消し、述語を直線コードに畳むので、CPU のパイプラインがよく流れる。
> [なぜ JIT か](why-jit.md)で論じた「解釈の間接費を消して、特化した直線
> コードにする」が、データの世界でもそのまま成り立つのです。

## カーネルの中で安全に走らせる eBPF の JIT

OS のカーネルの中にも JIT があります。**eBPF**（extended Berkeley Packet
Filter）です。もともとはパケットフィルタの仕組みでしたが、いまや Linux
カーネルの中で、ネットワーク監視・セキュリティ・トレーシングなど多様な
小さなプログラムをユーザが差し込める汎用の仕組みになっています。

ここで難題が立ちはだかります。カーネル空間で他人のコードを動かすのは
極めて危険です。無限ループ一つでシステムが固まり、不正なメモリアクセス一つで
カーネルが吹き飛ぶ。かといって、毎回ユーザ空間とカーネル空間を往復していては
遅すぎる。この「**安全に、かつ速く**」という相反する要求を、eBPF は
二段構えで解きます。

1. **検証（verifier）**：差し込まれた eBPF バイトコードを、カーネルが
   ロード時に静的に検証します。ループが必ず止まるか、メモリアクセスが
   範囲内か、未初期化の値を使っていないか。これらをすべての実行経路に
   ついて確かめ、一つでも危ういものがあればロードを拒否します。
2. **JIT コンパイル**：検証を通った安全なバイトコードを、カーネルが
   その場でネイティブ機械語へ JIT します。検証で安全が保証されている
   ので、実行時に重いチェックを入れる必要がなく、機械語は速く走れます。

お気づきでしょうか。この「**検証してから JIT する**」二段構えは、
[WebAssembly](research-wasm.md)[](#cite:haas2017) とまったく同じ思想です。
WebAssembly も、信頼できないコードを「型・スタック・メモリ境界が安全だ」と
検証してからネイティブへコンパイルすることで、「サンドボックスの安全」と
「ネイティブの速さ」を両立しました。eBPF はそれを、ブラウザの代わりに
OS カーネルという最も保護したい場所で実現しているのです。

> [!NOTE]
> 何を実行時にコンパイルしているか、を本書の言葉で言い直してみましょう。eBPF が
> 特化しているのは「安全だと検証済みの、ホットパスに置かれる小さな
> プログラム」です。検証という重い前処理をロード時に一度だけ済ませて
> しまえば、あとは投機も[脱最適化](deopt-osr.md)も要りません。SQL JIT が
> 「スキーマで型が確定しているから投機が要らない」のと同じく、eBPF は
> 「検証で安全が確定しているから実行時チェックが要らない」。**前提を前もって
> 確定させれば、その分だけ生成コードを単純で速くできる**。この JIT 設計の
> 普遍的な綱引きが、ここにも現れます。
>
> もう一つ、[セキュリティの章](security.md)との接点もあります。カーネル内に
> 機械語を生成する eBPF JIT は JIT スプレーの標的にもなりうるため、Linux には
> [定数ブラインディング](security.md)を eBPF JIT に施す `bpf_jit_harden`
> という設定があります。あの章で学んだ技法の、まさに実戦の現場です。

## 形と型が決まってからカーネルを生む GPU と数値計算

数値計算と GPU の世界にも、JIT は深く食い込んでいます。理由はいつもと同じ。
**実行時にならないと分からない情報がある**からです。

GPU で動かす計算カーネルは、`for` ループの回数、配列の形（行列の縦横）、
要素の型（`float32` か `float64` か）といった情報で性能が大きく変わります。
ところがこれらは、しばしばプログラムを動かしてみるまで確定しません。
画像の解像度はユーザが選んだファイルで決まり、行列のサイズは入力データで
決まる。そこで、形や型が実行時に確定してから、その値に特化した GPU カーネルを
コンパイルするわけです。

```python
# 形・型が実行時に決まってから特化する（NumPy/Numba 系の概念）
def saxpy(a, x, y):        # a*x + y。x,y の形と型は呼ばれるまで未知
    return a * x + y

saxpy(2.0, vec_f32, vec_f32)   # float32・長さ N 用のカーネルを生成・キャッシュ
saxpy(2.0, mat_f64, mat_f64)   # float64・M×N 用に別のカーネルを生成
```

呼ばれた瞬間に「`float32` の長さ N」という具体が分かるので、その形・型に
完全特化したカーネルを一度だけ生成し、以後はキャッシュして使い回します。
これは[ケーススタディ](case-others.md)で見た Julia の型ごとの特殊化や、
[なぜ JIT か](why-jit.md)の型による特化と、まったく同じ構図です。
実装を挙げましょう。

- **CUDA の NVRTC**：CUDA C++ のソースを実行時にコンパイルするライブラリ。
  プログラムが走り出してから、確定した定数（ループ回数やタイルサイズ）を
  ソースに埋め込んで、その値専用のカーネルを生成できます。実行時に
  分かった定数を焼き込むのは、[部分評価](research-partial-eval.md)の
  一手です。
- **OpenCL / シェーダコンパイル**：OpenCL カーネルや、ゲームの
  GLSL/HLSL シェーダは、対象 GPU が実行時まで分からないため、その GPU 向けの
  機械語へドライバが JIT します。「同じソースを、走らせるハードに合わせて
  実行時に特化する」のは、後で見る動的バイナリ変換とも通じる発想です。
  この方式の代償が、ゲーマーにはおなじみの**シェーダコンパイルの
  カクつき**（stutter）です。初見の場面でシェーダ（DirectX 12 や Vulkan では
  パイプライン状態オブジェクト、PSO）のコンパイルが走り、フレームが落ちる。
  これは[ウォームアップ問題](research-warmup.md)の、もっとも体感されやすい実例です。
  対策も本書で見た形そのもので、ゲームは起動時の事前コンパイルや PSO
  キャッシュ（コンパイル結果の保存・再利用）を使い、Steam はユーザの
  GPU・ドライバの組に合うコンパイル済みシェーダキャッシュを事前配信まで
  します（[コンパイルキャッシュ](research-warmup.md)の発想です）。
- **NumPy 系の Numba / JAX**：Python の数値コードにデコレータを付ける
  だけで、配列の形・型（dtype）が確定した時点で [LLVM](frameworks.md) や
  XLA（Google の機械学習向け線形代数コンパイラ）を介してネイティブ/GPU
  カーネルへコンパイルします。

> [!TIP]
> ここで効いている特化の軸は「型」だけではありません。形（shape）と
> サイズ（定数）も特化の対象です。ループ回数が定数だと分かれば
> [ループ展開](compilation-units.md)ができ、配列が連続だと分かれば
> ベクトル化（SIMD）が効き、行列サイズが固定だと分かれば添字計算が定数に
> 畳めます。言語処理系の JIT が「型」を投機・特化したのに対し、数値計算の
> JIT は「形とサイズ」を実行時に確定させて特化する。特化する対象は
> 違っても、「**実行時に分かった具体を機械語に焼き込む**」という核は同じです。

## 機械語から機械語への JIT、動的バイナリ変換

最後に、もっとも極端な応用を見ます。これまでは「バイトコード → 機械語」
「クエリ → 機械語」でしたが、ここでは入力そのものが機械語です。
**動的バイナリ変換**（dynamic binary translation, DBT）は、ある CPU 向けの
機械語を、別の（あるいは同じ）CPU 向けの機械語へ、実行時に翻訳する
技術です。

### あるアーキを別のアーキへ翻訳する QEMU の TCG

エミュレータ **QEMU** は、たとえば ARM 用にコンパイルされたバイナリを、
x86 のマシン上で動かせます。素朴なエミュレータは、ゲスト CPU の命令を
1 つずつ取り出して「この ARM 命令は何をする命令か」を解釈実行します。
もうこの構図には見覚えがあるはずです。ゲスト命令のフェッチと
ディスパッチ、つまり[インタプリタのループ](why-jit.md)そのものです。

QEMU の **TCG**（Tiny Code Generator）は、ここに JIT を持ち込みます。
ゲストの機械語を一命令ずつ解釈する代わりに、ゲスト機械語のひとかたまり
（基本ブロック）を、ホスト機械語へ翻訳してしまう。一度翻訳したブロックは
キャッシュ（translation block cache）に保存し、同じ場所をまた通ったら
翻訳済みのホスト機械語を直接走らせます。

TCG の作りは、本書の道具立てで読み解けます。ゲスト命令を、まず
TCG オペという共通の中間表現（[IR](ir.md)）へ落とし、それをホストの
機械語へ。各ゲスト命令に対応するホストコードの断片を貼り合わせていく
作りは、[テンプレート方式](research-copy-and-patch.md)の JIT に近い発想です。
翻訳済みブロックを繋いで解釈ループへの復帰を省く **block chaining** は、
言語処理系で言うインタプリタのディスパッチ消去に当たります。

> [!NOTE]
> 「翻訳の単位を基本ブロックにし、繋いでキャッシュする」という QEMU の
> 作りは、[コンパイルの単位](compilation-units.md)で論じた話と地続きです。
> 単位を大きく取れば翻訳の手間は減るが投機が難しくなり、小さく取れば
> 柔軟だが繋ぎのオーバーヘッドが増える。この綱引きは、ゲスト機械語を
> 相手にしても同じ形で現れます。

### AOT と JIT のハイブリッドである Rosetta 2

Apple の **Rosetta 2** は、x86 用にビルドされた macOS アプリを、
Apple Silicon（ARM）の Mac で動かす仕組みです。ここに、本書で何度も見た
[AOT と JIT の使い分け](case-others.md)が、鮮やかに現れます。

- **基本は AOT**：アプリを初めて起動する前に、x86 のバイナリ全体を
  一括で ARM 機械語へ翻訳し、結果を保存します。起動のたびに翻訳する
  [ウォームアップ](research-warmup.md)のコストを避け、二度目以降は翻訳済み
  コードで即座に立ち上がる、という判断です。これは [Android ART](case-others.md)が
  アイドル時間に AOT するのと同じ「コンパイルを前もって済ませる」発想です。
- **動的なコードには JIT**：ところが、AOT だけでは足りない場面があります。
  アプリ自身が実行時に x86 機械語を生成する場合（たとえば中に JIT を
  持つアプリ、ブラウザの JavaScript エンジンなど）は、起動前には存在
  しないコードなので AOT できません。Rosetta 2 は、こうした実行時に
  現れた x86 コードを JIT で翻訳するフォールバックを併せ持ちます。

つまり Rosetta 2 は、「**事前に分かるものは AOT、実行時にしか現れない
ものは JIT**」という、本書を貫く使い分けを、機械語から機械語への翻訳という
舞台で実践しているのです。さらに Apple Silicon は、x86 の強いメモリ順序
（TSO、Total Store Ordering）をハードウェアのモード切替として備えて
おり、Rosetta 2 の生成コードはメモリフェンス命令を大量に挟まずに済みます。
ソフトの JIT をハードの設計が下支えする、徹底したハイブリッドです。

### 同じアーキを最適化する源流の Dynamo

そして、本書をここまで読んだ読者には外せない名前が、動的バイナリ変換にも
あります。**Dynamo**[](#cite:bala2000) です。[トレーシング](tracing.md)の
章で「トレーシング JIT の源流」として登場した、あの Dynamo は、もともと
**動的バイナリ最適化器**でした。

Dynamo が翻訳するのは、同じアーキの機械語から、同じアーキの機械語へです。
アーキを乗り換えるのが目的ではなく、すでにネイティブで動いている
プログラムを実行しながら観測し、よく通る経路（ホットトレース）を見つけて、
そのトレースだけをさらに最適化した機械語に置き換えて速くする。実際、
原論文は「`-O` 最適化済みの SPECint95 バイナリを Dynamo の上で走らせると、
平均で `-O4` 最適化版に匹敵する性能に達した」と報告しています
[](#cite:bala2000)。**コンパイル済みのバイナリですら、実行時の観測で
まだ速くできる**のです。
[トレーシング](tracing.md)で学んだ「実際に通った 1 本の経路を最適化の単位に
する」という発想は、もともと言語処理系ですらない、機械語の動的最適化から
生まれたのです。

> [!IMPORTANT]
> Dynamo は、本章の主張を象徴しています。バイトコードもソースコードもなく、
> 入力はただのネイティブ機械語。それでも、「実行時に観測して、ホットな
> 部分を特化したコードへ置き換える」という JIT の核は、そのまま成立します。
> [トレーシング](tracing.md)という本書の主要技法が、言語処理系の外で
> 生まれて中へ入ってきた。この事実こそ、「JIT は言語処理系だけのもの
> ではない」という本章の主張の、何よりの証拠です。

## 実行時情報による特化という一つの発想

言語の外を一巡しました。並べてみると、驚くほど同じパターンの変奏です。

| 応用 | 何を特化するか | 本書のどの技法 |
|------|----------------|----------------|
| 正規表現エンジン | パターン → 専用機械語 | [ディスパッチ消去](why-jit.md)・[テンプレート](research-copy-and-patch.md) |
| データベース | クエリ → 専用機械語 | [部分評価](research-partial-eval.md)・[特殊化](why-jit.md) |
| eBPF | 検証済みバイトコード → 機械語 | [検証＋JIT](research-wasm.md) |
| GPU/数値計算 | 形・型・サイズ → 専用カーネル | [型による特化](why-jit.md) |
| 動的バイナリ変換 | 機械語 → 機械語 | [トレーシング](tracing.md)・[テンプレート](research-copy-and-patch.md)・AOT/JIT |

どれも、根っこは一つの発想に行き着きます。

> **どこかに「汎用の解釈系」がある。正規表現マッチャ、Volcano モデルの
> クエリ実行器、バイトコードインタプリタ、エミュレータ。これらは
> いずれも「入力を 1 つずつ見て、何をすべきか毎回判断する」ループだ。
> そして、その「毎回判断している対象」が実行時に固定だと分かれば、判断を
> 前もって畳んで、その場限りの特化した機械語に変えられる。これが JIT だ。**

本書の第 I 部で学んだ技法（[投機](deopt-osr.md)、[特殊化](why-jit.md)、
[テンプレート](research-copy-and-patch.md)、[トレーシング](tracing.md)、
[検証](research-wasm.md)）は、JavaScript や Ruby のためだけの道具では
ありませんでした。それらは「**実行時情報による特化**」という、もっと普遍的な
発想を実現するための部品だったのです。だからこそ、同じ部品が正規表現にも、
データベースにも、カーネルにも、GPU にも、エミュレータにも、形を変えて
現れます。

第 III 部・ケーススタディを、これで終えます。本書を通じて手にしたのは、
個別の処理系の知識そのものよりも、**新しい JIT に出会ったときに「これは
あの技法の、この変奏だ」と読み解く地図**です。次に未知の高速化技術に
出会ったら、まずこう問うてください。「ここにある汎用の解釈系は何か。実行時に
何が固定だと分かるのか。それで何を特化できるのか」。きっと、本書で見た
顔ぶれが、そこにも見つかるはずです。

旅の締めくくりに、巻末の[用語集](glossary.md)を用意しました。ここまでに
出会った言葉を、いつでも振り返れるように。よい JIT の航海を。
