# 階層的 JIT：速い段と賢い段を使い分ける

ここまで「コンパイルする」と一括りにしてきましたが、[なぜ JIT か](why-jit.md)で
ふれたように、コンパイルには**重さの段階**があります。ごく軽く機械語を
吐くこともできれば、時間をかけて徹底的に最適化することもできる。
問題は、**どちらか一方では困る**ことです。

- いきなり全力で最適化すると、コンパイルに時間がかかりすぎ、起動直後の
  プログラムが「コンパイル待ち」で固まる。
- かといって軽いコンパイルだけだと、長時間回り続ける本当に熱い場所で、
  得られたはずの速さを取り逃す。

答えは「**両方持って、状況に応じて使い分ける**」こと。これが**階層的 JIT**
（tiered JIT compilation）です。現代の主要処理系はほぼすべてこの構成です。

## ティアという考え方

階層的 JIT では、実行形式を**ティア**（tier、段）として並べます。下の段
ほど「起動が速いが実行は遅い」、上の段ほど「起動は遅いが実行が速い」。
コードは、熱くなるにつれて段を駆け上がります。典型的な四段はこうです。

| ティア | 形態 | 起動 | 実行速度 | 役割 |
|--------|------|------|----------|------|
| 0 | インタプリタ | 即時 | 最遅 | まず動かす。プロファイルを集める |
| 1 | ベースライン JIT | 速い | 中 | ディスパッチを消した素朴な機械語 |
| 2 | 軽量最適化 JIT | 中 | 速い | 型特殊化など、ほどほどの最適化 |
| 3 | 全力最適化 JIT | 遅い | 最速 | 本当に熱い場所だけ、徹底最適化 |

[V8](case-v8.md) なら Ignition（インタプリタ）→ Sparkplug（ベースライン）→
Maglev（軽量最適化）→ TurboFan（最上位の全力最適化）[^v8gen]、[JVM](case-jvm.md) なら
インタプリタ → C1（ベースライン）→ C2（全力最適化）といった具合に、実在の
処理系がこの段構成を持っています。ただし**概念上の段番号と、実装が内部で使う
ティア番号は一致するとは限りません**。たとえば HotSpot は概念上は三段
（インタプリタ＋C1＋C2）ですが、実装上のティアはレベル 0〜4 で表され、
C1 がプロファイルの有無で 1・2・3 に分かれるため、最上位の C2 は
**Tier 4** です（詳しくは[JVM のケース](case-jvm.md)）。ログや文献で
「Tier 4」「レベル 4」という表記を見たら、それはこの実装上の番号です。
本書では混乱を避け、
概念上の役割（インタプリタ／ベースライン／軽量最適化／全力最適化）で呼びます。

[^v8gen]: 近年の V8 では、TurboFan 内部のコンパイルパイプラインと中間表現を
    刷新する **Turboshaft** への移行が進んでいます。これは別のティアが
    増えたのではなく、最上位段の「中身」の作り直しです（詳しくは
    [V8 のケーススタディ](case-v8.md)）。本書では最上位の全力最適化段の
    代表として TurboFan の名を使います。

## なぜ段を分けるのか：ウォームアップ

段を分ける最大の理由は、**ウォームアップ**（warmup、暖機）を滑らかに
することです。プログラムを起動した直後は、まだどこが熱いか分かりません。
ここで全力最適化を始めると、コンパイルのコストばかりかかって、肝心の
実行が進みません。

だから最初は**ティア 0（インタプリタ）でとにかく動かし**、その間に
[プロファイリング](profiling.md)でデータを集めます。少し熱くなったら
ティア 1（ベースライン JIT）へ。ベースライン JIT は最適化をほとんど
しない代わりに、**ごく短時間で機械語を出します**。典型は**テンプレート方式**
（template JIT）です。バイトコードの各命令に対応する機械語片（テンプレート）を
あらかじめ用意しておき、コンパイル時にはバイトコード列をなぞって、対応する
機械語片を**順に並べて貼り合わせるだけ**。コード生成が実質コピーの繰り返し
なので一瞬で終わり、それでいてディスパッチを消すだけでもインタプリタより
大きく速くなります。

そして本当に熱い、ごく一部の場所だけが、時間をかけて上位ティアへ昇格
します。コストの高い最適化を、元が確実に取れる場所だけに集中させるのです。

> [!NOTE]
> 「ベースライン JIT」は、最適化をしない（あるいはごく軽い）高速な
> コンパイラを指す言葉です。V8 の Sparkplug[](#cite:v8sparkplug) が
> その典型です。[Ruby の YJIT](case-ruby.md) は、テンプレート型の素朴な
> ベースラインよりは一歩踏み込んだ**軽量・高速側の JIT**（基本ブロック
> バージョニングで型を特殊化する）ですが、TurboFan 級の重い最適化までは
> しません。「JIT＝重い最適化コンパイラ」という思い込みを捨てるのが
> 大事で、**速く雑に出す JIT**も立派な JIT です。

## 昇格をシミュレートしてみる

ティアの昇格は、要するに「[プロファイリング](profiling.md)のカウンタが
閾値を超えたら次の段へ移す」ことです。小さなシミュレータで動きを
見てみましょう。

```ruby
# ティア昇格のシミュレーション
class TieredVM
  TIERS = [:interp, :baseline, :light_opt, :full_opt]
  # 各ティアへ上がるのに必要な実行回数（累積）
  THRESHOLDS = { baseline: 10, light_opt: 100, full_opt: 1000 }
  # 各ティアの「1 回あたりの相対実行コスト」（小さいほど速い）
  COST = { interp: 10.0, baseline: 3.0, light_opt: 1.5, full_opt: 1.0 }

  def initialize
    @count = Hash.new(0)
    @tier  = Hash.new(:interp)
  end

  def execute(method)
    @count[method] += 1
    n = @count[method]
    # 閾値を越えたら昇格（高い段から順にチェック）
    if    n == THRESHOLDS[:full_opt]  then @tier[method] = :full_opt
    elsif n == THRESHOLDS[:light_opt] then @tier[method] = :light_opt
    elsif n == THRESHOLDS[:baseline]  then @tier[method] = :baseline
    end
    COST[@tier[method]]   # この実行にかかったコストを返す
  end

  def tier(method) = @tier[method]
end

vm = TieredVM.new
total = 0.0
2000.times { total += vm.execute(:hot_method) }
puts "最終ティア: #{vm.tier(:hot_method)}"
puts "総コスト: #{total.round(1)}"

# 比較：ずっとインタプリタのままだった場合
puts "もしインタプリタのままなら: #{(2000 * 10.0).round(1)}"
```

```
最終ティア: full_opt
総コスト: 2711.0
もしインタプリタのままなら: 20000.0
```

最初の 9 回はインタプリタ（コスト 10）で動き、10 回でベースライン
（3）、100 回で軽量最適化（1.5）、1000 回で全力最適化（1）へ昇格。
2000 回の総コストは、ずっとインタプリタのままなら 20000 だったのが
2711 に下がりました。**熱い場所ほど良い段で回る時間が長くなる**ので、
段を分けるだけでこれだけ効くのです。

> [!TIP]
> このシミュレータには、本物にある大事な要素が一つ抜けています。
> **コンパイル自体のコスト**です。本物では、上位ティアへ上がるたびに
> 「そのコンパイルに何ミリ秒」という出費が発生します。だから閾値は、
> 「昇格のコンパイル費用 < 昇格後に取り返せる高速化」となるように
> 設計されます。閾値を上の例のように固定値にするか、もっと賢く動的に
> 決めるかは、処理系のチューニングどころです。

### コンパイル費用を組み込む

さきほどのシミュレータでは、昇格は「タダ」でした。実際には、上位
ティアへ上がるたびに**一時金（コンパイル費用）を払って**います。
しかも上の段ほどコンパイラは賢く、賢いぶん時間がかかる。ベースラインが
バイトコード命令ごとに機械語片を並べるだけ（[前述](tiered.md#なぜ段を分けるのか：ウォームアップ)の
テンプレート方式）なのに対し、全力最適化は[中間表現](ir.md)を組み、
何度も最適化パスを回します。ここに「一時金」を足してみましょう。

```ruby
# コンパイル費用つきティア昇格シミュレータ
class TieredVMWithCost
  COST = { interp: 10.0, baseline: 3.0, light_opt: 1.5, full_opt: 1.0 }
  # 各ティアへ昇格するときに一度だけ払う「コンパイル費用」（相対値）。
  # 上の段ほど賢く、賢いぶん高い。
  COMPILE_COST = { baseline: 50.0, light_opt: 400.0, full_opt: 3000.0 }

  def initialize(thresholds)
    @thresholds = thresholds            # 閾値を外から差し替え可能にする
    @count   = Hash.new(0)
    @tier    = Hash.new(:interp)
    @compile = 0.0                      # 払ったコンパイル費用の累計
  end

  def execute(method)
    @count[method] += 1
    n = @count[method]
    if    n == @thresholds[:full_opt]  then promote(method, :full_opt)
    elsif n == @thresholds[:light_opt] then promote(method, :light_opt)
    elsif n == @thresholds[:baseline]  then promote(method, :baseline)
    end
    COST[@tier[method]]
  end

  def promote(method, to)
    @tier[method] = to
    @compile += COMPILE_COST[to]        # 昇格の瞬間に一時金を払う
  end

  attr_reader :compile
  def tier(method) = @tier[method]
end

# 閾値を変えて「総コスト」を比べる。総コスト = 実行コスト + コンパイル費用
def simulate(thresholds, runs)
  vm = TieredVMWithCost.new(thresholds)
  exec = 0.0
  runs.times { exec += vm.execute(:m) }
  [exec, vm.compile, exec + vm.compile, vm.tier(:m)]
end

defaults = { baseline: 10, light_opt: 100, full_opt: 1000 }
[200, 2000, 50000].each do |runs|
  e, c, t, tier = simulate(defaults, runs)
  puts "実行 #{runs} 回: 実行=#{e.round} + コンパイル=#{c.round} " \
       "= 総 #{t.round}（最終 #{tier}）"
end
```

```
実行 200 回: 実行=512 + コンパイル=450 = 総 962（最終 light_opt）
実行 2000 回: 実行=2711 + コンパイル=3450 = 総 6161（最終 full_opt）
実行 50000 回: 実行=50711 + コンパイル=3450 = 総 54161（最終 full_opt）
```

200 回では閾値 1000 に届かず全力最適化（一時金 3000）まで上がらず、
軽量最適化止まりで一時金は 450 で済んでいます。ここで効いてくるのが、
**コンパイル費用は一度きり**だが**実行コストは回数に比例して効いてくる**
という非対称性です。たった 200 回のメソッドを全力最適化まで上げて
しまえば、3000 の一時金を実行の利得（高々数百）で取り返せず、明らかに
損です。逆に 50000 回回るなら、3450 の一時金は誤差で、上げれば上げる
ほど得をします。**「あと何回呼ばれそうか」を見積もれれば、最適な到達
ティアが決まる**わけです。

### 閾値が低すぎ・高すぎの損得を測る

では閾値をずらすと総コストはどう動くか。**低すぎる閾値**（すぐ昇格）と
**高すぎる閾値**（なかなか昇格しない）を、同じ実行回数で比べます。

```ruby
runs = 3000
scenarios = {
  "攻め（低い閾値）"     => { baseline: 2,   light_opt: 5,    full_opt: 20 },
  "標準"                 => { baseline: 10,  light_opt: 100,  full_opt: 1000 },
  "守り（高い閾値）"     => { baseline: 100, light_opt: 1000, full_opt: 10000 },
}
scenarios.each do |name, th|
  e, c, t, tier = simulate(th, runs)
  puts "#{name}: 総 #{t.round}（実行=#{e.round}, コンパイル=#{c.round}, 最終 #{tier}）"
end
```

```
攻め（低い閾値）: 総 6473（実行=3023, コンパイル=3450, 最終 full_opt）
標準: 総 7161（実行=3711, コンパイル=3450, 最終 full_opt）
守り（高い閾値）: 総 7142（実行=6692, コンパイル=450, 最終 light_opt）
```

3000 回という「そこそこ熱い」負荷では、**攻めの閾値が一番得**でした。
早く上位へ上がるぶん、良い段で回る時間が長く、実行コストが小さくなる
からです。一方「守り」は閾値が高すぎて、3000 回でも全力最適化に届かず
（一時金は 450 で安いものの）、遅い段で回り続けた実行コストがかさんで
います。つまりこの負荷では、**昇格をけちったぶんがそのまま損**になった
わけです。ところが同じ攻めの閾値を、**ほとんど回らない**メソッド
（数十回で終わる短命なもの）にかけると、話は逆転します。

```ruby
[5, 30, 300].each do |runs|
  results = scenarios.map { |name, th| [name, simulate(th, runs)[2]] }
  best = results.min_by { |_, t| t }
  puts "実行 #{runs} 回 → #{results.map { |n, t| "#{n}:#{t.round}" }.join("  ")}" \
       "  ★勝者=#{best[0]}"
end
```

```
実行 5 回 → 攻め（低い閾値）:471  標準:50  守り（高い閾値）:50  ★勝者=標準
実行 30 回 → 攻め（低い閾値）:3503  標準:203  守り（高い閾値）:300  ★勝者=標準
実行 300 回 → 攻め（低い閾値）:3773  標準:1112  守り（高い閾値）:1643  ★勝者=標準
```

5 回しか回らないメソッドでも、攻めの閾値は 2 回目でベースライン、
5 回目で軽量最適化へと昇格させ、一時金 450 を払います。実行はそこで
終わるので利得はほぼゼロ、総コストは昇格しない標準（50）の 9 倍以上に
膨らみました（なおこの行で標準と守りは同コストの同着で、勝者表示は先に
並んだほうを出しているだけです）。30 回なら閾値 20 で全力最適化
（一時金 3000）まで上がってしまい、計 3450 の一時金をほぼまるごと
無駄打ちして標準の 17 倍です。**コンパイル費用を
回収する前に実行が終わってしまう**のがその正体で、これが「閾値が
低すぎる」ことの損。起動直後に大量の一度きりメソッドが走るアプリ
（短命なコマンドラインツールなど）でとくに痛い。逆に「閾値が高すぎる」と、
先ほどの 3000 回の「守り」のように、本当に熱い場所がいつまでも遅い段に
留まり、取れたはずの速さを取り逃します。**同じ閾値でも、負荷が違えば
最適解が裏返る** —— これが固定閾値の限界であり、次節の動的調整の動機です。

> [!IMPORTANT]
> 閾値設計の本質は、**一度きりのコンパイル費用と、回数に比例する実行
> 利得の損益分岐点**を当てにいくことです。固定の閾値では、この分岐点が
> 負荷によって動くのに追従できません。だから現実の処理系は、後述する
> ように閾値を**動的に**調整します。なお「上げたコードを途中で下げる」
> 降格＝[脱最適化](deopt-osr.md)も、見方を変えれば「払った一時金が
> 無駄になる」イベントです。何度も降格する場所を上げ続けるのは、
> 一時金を捨て続けるのに等しい —— この観点は後の[降格の運用](tiered.md#降格を運用する：tier down と再投機の抑制)で扱います。

## 上の段だけでなく、下の段へも：脱最適化

ティアは一方通行ではありません。上位ティアのコードは**投機**の上に
建っています。「この変数はいつも整数」という前提でガードを置き、外れたら
インタプリタへ戻る —— [脱最適化](deopt-osr.md)です。階層的 JIT に
おいては、脱最適化は「**上位ティアから下位ティアへ落ちる**」
こととして現れます。

つまりコードは、熱くなれば段を上がり、投機が外れれば段を下がる。場合に
よっては、何度も脱最適化を繰り返す場所を「ここは投機しないほうが
よい」と学習し、より保守的なコードで作り直します。ティアは、こうした
**昇格と降格のあいだを行き来する動的な状態**なのです。

## 同時に動かす：バックグラウンドコンパイル

階層的 JIT のもう一つの肝は、**コンパイルを実行と並行して行う**ことです。
ティア 2 や 3 のコンパイルは時間がかかるので、それを実行スレッドの中で
待っていては本末転倒です。そこで、

1. 実行スレッドは**現在のティアで走り続ける**（例：ベースライン）。
2. 別のコンパイルスレッドが、裏で上位ティアのコードを作る。
3. できあがったら、次にその関数に入るとき**こっそり差し替える**。

すでに**実行中の**長いループを途中で上位ティアへ乗り換えるには、
スタック上の状態を作り替える特別な技術 —— **OSR**（On-Stack Replacement、
[脱最適化と OSR](deopt-osr.md)で扱います）—— が必要になります。
逆に、まだ呼ばれていない関数の差し替えは、入口のポインタを書き換える
だけなので簡単です。

## 下の段が計装し、上の段が読む：プロファイルの受け渡し

段を分けるもう一つの旨味は、**下位ティアが上位ティアのための情報を
ためてくれる**ことです。インタプリタやベースラインは、どうせ遅いの
だから、ついでに**計装**（instrumentation、観測コードの埋め込み）を
仕込んでおく。上位コンパイラは、その記録を読んで初めて「攻めの投機」が
できます。これは[プロファイリング](profiling.md)で見たカウンタや型
フィードバックを、ティア間の**分業**として位置づけ直したものです。

ベースライン段が具体的に何を記録するかを、おもちゃで見てみましょう。
ここでは（1）呼び出し回数、（2）各引数で受け取った型、（3）分岐の
どちらへ進んだかの確率、を貯めます。

```ruby
# ベースライン段が貯めるプロファイル
class Profile
  def initialize
    @calls = 0
    @types = Hash.new { |h, k| h[k] = Hash.new(0) }  # 引数位置 => {型 => 回数}
    @branch = Hash.new { |h, k| h[k] = [0, 0] }      # 分岐ID => [真の回数, 偽の回数]
  end

  # ベースラインの各命令実行時に呼ばれる「観測フック」
  def record_call(args)
    @calls += 1
    args.each_with_index { |a, i| @types[i][a.class] += 1 }
  end

  def record_branch(id, taken)
    @branch[id][taken ? 0 : 1] += 1
  end

  # --- 上位コンパイラ向けの「読み出し API」---

  # 引数 i の型が十分に偏っているか（単態 monomorphic か）を答える
  def dominant_type(i, ratio: 0.95)
    counts = @types[i]
    return nil if counts.empty?
    type, n = counts.max_by { |_, c| c }
    n.fdiv(@calls) >= ratio ? type : nil   # 95% 以上なら投機の根拠にする
  end

  # 分岐 id の偏り（0.0=常に偽, 1.0=常に真）
  def branch_probability(id)
    t, f = @branch[id]
    total = t + f
    total.zero? ? nil : t.fdiv(total)
  end

  attr_reader :calls
end
```

ベースラインはこれを淡々と更新するだけ。やがてこのメソッドが熱く
なって上位ティアのコンパイルが起動すると、コンパイラは読み出し API を
叩いて**投機の方針**を決めます。

```ruby
# 上位コンパイラが、ためられたプロファイルを読んで方針を立てる
def plan_speculation(profile)
  plans = []
  if (t = profile.dominant_type(0))
    plans << "引数0は #{t} と仮定して型ガードを1枚だけ置く（外れたら脱最適化）"
  else
    plans << "引数0は型が乱れている（メガモーフィック）→ 投機せず汎用コードのまま"
  end
  if (p = profile.branch_probability(:loop_exit))
    if p < 0.01
      plans << "loop_exit はほぼ偽 → ループ本体を主経路として最適化"
    elsif p > 0.99
      plans << "loop_exit はほぼ真 → ループはほぼ回らない前提で軽く扱う"
    end
  end
  plans
end

# ベースラインで 1000 回回った様子を再現
prof = Profile.new
1000.times { |i| prof.record_call([42]); prof.record_branch(:loop_exit, i == 999) }
puts plan_speculation(prof)
```

```
引数0は Integer と仮定して型ガードを1枚だけ置く（外れたら脱最適化）
loop_exit はほぼ偽 → ループ本体を主経路として最適化
```

これがティア間分業の核心です。ベースラインは**速く動くこと**に専念し、
記録は安いカウンタ更新だけ。賢い判断は、元が取れると分かった上位
コンパイル時にまとめて行う。投機が当たればガード一枚で速いコードに
化け、外れれば[脱最適化](deopt-osr.md)で下位へ戻ります。

> [!NOTE]
> [JVM](case-jvm.md) の HotSpot はこの分業のお手本です。C1（本書の概念上の
> ティア 1）は軽く最適化しつつ**プロファイルを取るための計装を埋め込んだ版**を
> 持ち、十分な統計がたまると C2（概念上のティア 3、全力最適化段）がそれを
> 読んで攻めの最適化をかけます。C2 はもともとサーバ用の単体コンパイラと
> して生まれたもので[](#cite:paleczny2001)、C1 と常時組み合わせる階層的
> コンパイルが標準になったのは JDK 7/8 以降です。型フィードバックの考え方
> そのものは、仮想呼び出しの宛先を観測して特殊化する
> 技法[](#cite:holzle1994)にさかのぼります。

> [!WARNING]
> 計装はタダではありません。カウンタ更新一つでも、ホットなループでは
> 塵も積もれば山。だから「ベースラインは観測する／全力最適化版は観測を
> 外す」のが定石です。上位ティアまで上がった熱いコードに観測コードを
> 残し続けると、せっかくの最適化を観測のオーバーヘッドで食いつぶします。
> プロファイルは**下位ティアで集め、上位ティアでは捨てる**。

## 閾値を動かす：tier-up 閾値の動的調整

ここまでの閾値は固定値でしたが、現実の処理系は閾値を**動的に補正**
します。理由は前節で見たとおり、損益分岐点が負荷で動くからです。
代表的な補正の軸を、判断ロジックのおもちゃで示します。

```ruby
# 動的に tier-up 閾値を決める
class AdaptiveThreshold
  BASE = 1000  # 基準の閾値

  def initialize
    @queue_len = 0   # いまコンパイルキューに何件たまっているか
  end
  attr_accessor :queue_len

  # method_size: バイトコード命令数, loopy: ループを含むか
  def threshold_for(method_size:, loopy:)
    t = BASE.to_f

    # (1) 関数が大きいほどコンパイルは高い → 上げる元が取りにくいので閾値↑
    t *= 1.0 + (method_size / 200.0)

    # (2) ループを含むコードは1回の呼び出しで大量に回る → 早めに上げたい → 閾値↓
    t *= 0.5 if loopy

    # (3) キューが混んでいるほど、上げても順番待ちで効果が遅れる → 閾値↑
    t *= 1.0 + (@queue_len / 10.0)

    t.round
  end
end

at = AdaptiveThreshold.new
puts "小さい単純関数:        #{at.threshold_for(method_size: 20,  loopy: false)}"
puts "小さいループ関数:      #{at.threshold_for(method_size: 20,  loopy: true)}"
puts "巨大な関数:            #{at.threshold_for(method_size: 2000, loopy: false)}"
at.queue_len = 50
puts "キュー混雑時（小関数）: #{at.threshold_for(method_size: 20,  loopy: false)}"
```

```
小さい単純関数:        1100
小さいループ関数:      550
巨大な関数:            11000
キュー混雑時（小関数）: 6600
```

それぞれの補正には、はっきりした理屈があります。

- **関数サイズで補正**：大きい関数はコンパイル費用（前節の一時金）が
  高いので、より確実に熱いと分かるまで待つ。逆に小さい関数は安く上げ
  られるので気軽に昇格させてよい。
- **ループの有無で補正**：ループを含むコードは一度の呼び出しで内部を
  何千回も回りうる。呼び出し回数カウンタだけでは熱さを過小評価する
  ので、閾値を下げる。あるいはループ専用に**バックエッジカウンタ**
  （ループの戻り回数を数える）を別に持ち、それが閾値を超えたら[OSR](deopt-osr.md)で
  途中から上位ティアへ乗り換えます。
- **キュー混雑で補正**：コンパイラスレッドが詰まっているとき、さらに
  注文を増やしても列が伸びるだけ。混雑時は閾値を上げて注文をしぼり、
  本当に熱いものだけを通します。

なお、[プロファイリング](profiling.md)で見たカウンタの**エイジング**
（定期的な減衰）も、この仲間です。閾値の側を動かす代わりにカウンタの側を
冷ますことで、「最近の熱さ」だけで昇格を判定する動的補正として働きます。

> [!TIP]
> 実際の HotSpot は、もっと素朴ながら堅実な式を使います。呼び出し回数と
> バックエッジ回数の重み付き和を、システム全体のコンパイラ負荷を見ながら
> スケールさせ、「最近の負荷に応じて閾値を上下させる」のが現代版の流儀です。
> こうした**適応的最適化** —— 費用対効果や負荷を見て昇格を決める考え方 —— は、
> Jikes RVM（旧 Jalapeño）の Arnold らの研究にさかのぼります
> [](#cite:arnold2000)。固定値より、負荷追従型の
> ほうが、起動時の洪水にも定常運転にもうまく適応します。

## コンパイルキューと並行コンパイル

上位ティアのコンパイルは時間がかかるので、[前述](tiered.md#同時に動かす：バックグラウンドコンパイル)の
とおり裏で走らせます。ここで肝になるのが**コンパイルキュー**の運用です。
昇格判定が下るたびに「この関数を上位ティアでコンパイルせよ」という
**注文**を、優先度つきのキューに積みます。コンパイラスレッドはそこから
取り出して順に処理する。実行スレッドは**待たずに現ティアで走り続け**、
できあがったら差し替える —— この「待たない」が決定的です。

```ruby
# 優先度つきコンパイルキューのおもちゃ
class CompileQueue
  Job = Struct.new(:method, :tier, :hotness)

  def initialize
    @jobs = []
    @enqueued = {}   # 同じ (method, tier) の二重注文を防ぐ
  end

  def enqueue(method, tier, hotness)
    key = [method, tier]
    return if @enqueued[key]    # すでに注文済みなら無視（重複コンパイル防止）
    @enqueued[key] = true
    @jobs << Job.new(method, tier, hotness)
  end

  # 熱い順に取り出す。実行スレッドはこれを待たない
  def take_hottest
    return nil if @jobs.empty?
    job = @jobs.max_by(&:hotness)
    @jobs.delete(job)
    @enqueued.delete([job.method, job.tier])
    job
  end

  def size = @jobs.size
end

q = CompileQueue.new
q.enqueue(:render,    :full_opt, 5000)
q.enqueue(:parse_csv, :full_opt, 80000)   # こっちの方が熱い
q.enqueue(:render,    :full_opt, 6000)    # render は注文済み → 無視
puts "キュー長: #{q.size}"
puts "最初に処理: #{q.take_hottest.method}"       # 熱い parse_csv が先
```

```
キュー長: 2
最初に処理: parse_csv
```

ポイントは三つです。

- **優先度（熱さ）順**：列の先頭から処理すると、ぬるい関数が熱い関数を
  待たせてしまう。熱い順に取り出せば、効果の大きいものから片付きます。
- **重複注文の排除**：同じ関数が短時間に何度も閾値を跨ぐと、同じ
  コンパイル注文が何度も積まれかねない。`@enqueued` で「注文済み」を
  覚えておき、二重コンパイルを防ぎます。
- **取り消し（本書では割愛）**：注文が処理される前にその関数が
  [脱最適化](deopt-osr.md)で冷えたら、注文ごと捨てるのが理想です。

並行コンパイルには罠もあります。コンパイル中に元のコードが脱最適化
されたり、依存していた前提（クラス階層など）が崩れたりすると、せっかく
作ったコードが**完成と同時に無効**になりえます。だから差し替えの瞬間に
「いま作った版の前提はまだ生きているか」を確認し、崩れていれば破棄する
安全弁が要ります。この「前提が崩れたら無効化」の仕組みは、降格＝
[脱最適化](deopt-osr.md)と地続きです。

## コードキャッシュの管理：置き場は有限

生成した機械語は、どこかに置かねばなりません。この置き場を**コード
キャッシュ**（code cache）と呼びます。容量は有限なので、長く動かして
昇格・降格を繰り返すうちに、**断片化**したり**埋まりきったり**します。
すると新しいコンパイル結果が置けず、JIT が事実上止まってしまう。
対策の骨格を、ごく単純な退去つきキャッシュで示します。

```ruby
# 容量つきコードキャッシュ（LRU 退去）
class CodeCache
  Entry = Struct.new(:method, :tier, :size, :last_used)

  def initialize(capacity)
    @capacity = capacity
    @used = 0
    @entries = {}      # method => Entry
    @clock = 0
  end

  def install(method, tier, size)
    drop(method) if @entries[method]      # 同じ関数の古い版は捨てる
    evict_until(size)                     # 空きを作る
    @entries[method] = Entry.new(method, tier, size, tick)
    @used += size
  end

  def touch(method)                       # 実行されたら使用時刻を更新
    @entries[method]&.last_used = tick
  end

  private

  def tick = (@clock += 1)

  def drop(method)
    e = @entries.delete(method)
    @used -= e.size if e
  end

  # 空きが足りない間、一番長く使われていない版を退去させる
  def evict_until(need)
    while @used + need > @capacity && !@entries.empty?
      victim = @entries.values.min_by(&:last_used)
      puts "  退去: #{victim.method}(#{victim.tier}, #{victim.size})"
      drop(victim.method)
    end
  end
end

cache = CodeCache.new(100)
cache.install(:a, :full_opt, 40)
cache.install(:b, :full_opt, 40)
cache.touch(:a)                 # a を使ったので新しくする
cache.install(:c, :full_opt, 40)   # 容量100に収まらない → b が退去される
```

```
  退去: b(full_opt, 40)
```

実際のコードキャッシュ管理では、これにいくつもの現実的配慮が乗ります。

- **古い版の破棄**：同じ関数を上位ティアで作り直したら、下位ティアの
  版はもう要りません（`drop` で捨てた部分）。降格が起きれば逆に上位版を
  捨てます。版が増えすぎないよう、こまめに掃除します。
- **退去の選定**：上の例は単純な LRU（最近最も使われていない版を退去）
  ですが、本物は「熱さ」「ティア」「再生成のしやすさ」も加味します。
  ベースライン版は安く作り直せるので退去候補にしやすい、など。
- **断片化対策**：固定長スロットに区切る、ときどき詰め直す（コンパクション）、
  ティアごとに別領域を持つ、といった工夫で空き地の細切れを防ぎます。
- **退去後の挙動**：退去された関数を次に呼ぶと、コードが無いので下位
  ティアに落ちて再実行 —— つまりこれも**降格の一種**です。容量不足が
  続けば「上げては退去され、また上げる」という無駄が起き、これは「閾値が
  低すぎる」のとよく似た損につながります。

> [!WARNING]
> コードキャッシュ枯渇は、長時間動くサーバで実際に起きる障害です。
> JVM では code cache が満杯になると `CodeCache is full. Compiler has
> been disabled.` という警告とともに、**新規のコンパイルが止まります**。
> コンパイル済みのコードはそのまま走り続けますが、以後に熱くなった場所は
> 上位ティアへ上がれず、じわじわ性能が落ちる。既定では古いコードを捨てて
> 空きを作る掃除（UseCodeCacheFlushing）が有効になっており、それでも
> 追いつかなければ容量（ReservedCodeCacheSize、既定 240MB）を広げるのが
> 定番の対処です。容量はチューニングのつまみであり、
> 退去戦略は「どの版を生かすか」という[脱最適化](deopt-osr.md)・降格の
> 判断と密接に絡みます。

## 降格を運用する：tier down と再投機の抑制

[前述](tiered.md#上の段だけでなく、下の段へも：脱最適化)のとおり、降格＝[脱最適化](deopt-osr.md)は
「上位から下位へ落ちる」こととして現れます。ここまでに見たとおり、コードが
下位ティアへ落ちる**降格には三つの顔**があります。

| 降格の顔 | きっかけ | 範囲 |
|----------|----------|------|
| ガード破れ＝[脱最適化](deopt-osr.md) | 投機の前提が個別の実行で外れた | そのフレーム一つ |
| 大域前提の崩壊による一括無効化 | クラスのロードなどで「上書きされていない」等の前提が覆った | 依存していた全コード |
| コードキャッシュからの退去 | 置き場が足りず、生成済み機械語が捨てられた | 退去された版 |

一回の降格は、投機が外れた
だけのこと。問題は、**同じ場所が何度も降格を繰り返す**ときです。上げる
たびにコンパイル費用（一時金）を払い、すぐ落ちてまた払う —— 一時金を
捨て続けるのに等しい。運用上の対策は「**何度も落ちる場所は無理に上げ
ない**」と学習することです。

```ruby
# 降格回数を学習して再投機を抑制する
class DeoptPolicy
  RESPECULATE_LIMIT = 3   # この回数を超えて落ちたら、もう上げない

  def initialize
    @deopt_count = Hash.new(0)
  end

  def on_deopt(method)
    @deopt_count[method] += 1
  end

  # 上位ティアへの昇格を許すか？
  def may_promote?(method)
    @deopt_count[method] < RESPECULATE_LIMIT
  end

  # 上げてよいが、何度か落ちたなら「投機控えめ」で作るよう指示する
  def speculation_level(method)
    case @deopt_count[method]
    when 0    then :aggressive   # 攻めの投機（型ガード1枚、など）
    when 1..2 then :conservative # 控えめ（複数型を許す多態版）
    else           :none         # もう投機しない。汎用コードに留め置く
    end
  end
end

pol = DeoptPolicy.new
m = :flaky
puts "初回:    昇格可=#{pol.may_promote?(m)}, 投機=#{pol.speculation_level(m)}"
2.times { pol.on_deopt(m) }
puts "2回降格後: 昇格可=#{pol.may_promote?(m)}, 投機=#{pol.speculation_level(m)}"
3.times { pol.on_deopt(m) }   # 計5回
puts "5回降格後: 昇格可=#{pol.may_promote?(m)}, 投機=#{pol.speculation_level(m)}"
```

```
初回:    昇格可=true, 投機=aggressive
2回降格後: 昇格可=true, 投機=conservative
5回降格後: 昇格可=false, 投機=none
```

ここには段階的な運用判断が現れています。

- **初回は攻める**：まだ外れた実績がないので、型ガード一枚の攻めの
  投機で速いコードを作る。当たれば大勝ち。
- **数回落ちたら控えめに**：一回外れただけで諦めるのは早計（たまたま
  かもしれない）。そこで投機を弱め、複数の型を許す多態版で作り直す。
  攻めの速さは落ちるが、降格しにくくなります。
- **しつこく落ちるなら上げない**：何度作っても落ちる場所は、本質的に
  予測できないのです。ここはもう下位ティアに**留め置く**ほうが、一時金を
  捨て続けるより安い。降格回数を数えて再投機を抑制する、この「頻度
  学習」は、ガード単位の視点から[脱最適化と OSR](deopt-osr.md)でも
  ふれます。

> [!NOTE]
> 攻めの投機と降格の頻度学習は、攻守一体で初めて性能が出ます。そして
> そもそも「投機して、外れたら戻る」こと自体が常に正しい結果を保つのか ——
> この問いは、数学的に定式化して証明する研究の対象にも
> なっています[](#cite:fluckiger2018)。詳しくは第 II 部の
> [投機的最適化の正しさ](research-deopt-correctness.md)で扱います。

## 段はいくつ必要か

「段は多いほどよい」わけではありません。段が増えると、それぞれの
コンパイラを実装・保守するコストがかかり、段の間の遷移（プロファイルの
受け渡し、コードの差し替え）も複雑になります。

- **二段**（インタプリタ＋一つの JIT）：実装が単純。YJIT をはじめ多くの
  処理系の出発点。Ruby は現在もこの構成で、Ruby 4.0（2025）では YJIT に
  加えて新世代の最適化志向 JIT である **ZJIT** が実験導入されました。
  ただし現状は YJIT と ZJIT の**どちらか一方しか有効にできず**、二つが
  階層として重なるわけではありません。将来 YJIT を置き換えることを
  見据えた動きです。
- **三段**（インタプリタ＋ベースライン＋最適化）：JVM の標準的な構成
  （インタプリタ＋C1＋C2）。なお HotSpot の実装ティアは前述のとおり
  レベル 0〜4 で、概念上の三段とは番号がずれます。
- **四段**：V8 のように、起動の速さと最高速度の両立を極限まで突き詰めた
  構成。

どこまで段を増やすかは、その処理系が**どんな実行プロファイルを重視するか**
で決まります。短命なコマンドラインツール（起動の速さが命）と、何日も
動き続けるサーバ（最高速度が命）では、最適な段の数が違います。この
「起動 vs 最高速度」のトレードオフを正面から測り直したのが、第 II 部の
[ウォームアップ問題](research-warmup.md)です。

> [!IMPORTANT]
> 階層的 JIT は、本書のここまでの技法を**運用の枠組み**としてまとめる
> 章でした。プロファイリングで熱さを測り、軽い JIT で素早く立ち上げ、
> 投機で攻めの最適化をし、外れたら脱最適化で戻る —— これらが
> 「ティアの昇格と降格」という一つの運動として統合されます。
> どの処理系の JIT を見るときも、「この段は何ティア構成で、各段は
> 何をしているか」を最初に問うと、全体像が早くつかめます。

## まとめ

- 階層的 JIT は、**起動の速い軽い段**と**実行の速い重い段**を並べ、
  コードを熱さに応じて段を上げ下げする。
- 起動直後はインタプリタで動かし、プロファイルを集めつつ、熱い場所だけを
  上位ティアへ昇格させる（**ウォームアップ**）。
- **ベースライン JIT** は最適化しない高速コンパイラ。「速く雑に出す」のも
  立派な JIT。
- 昇格には**一度きりのコンパイル費用**がかかり、実行利得は回数に比例する。
  閾値設計はこの**損益分岐点**を当てにいく作業で、低すぎても高すぎても損。
- 下位ティアは**計装してプロファイルを貯め**、上位コンパイラがそれを読んで
  投機する。観測は下位で行い、上位では捨てるのが定石。
- 閾値は固定でなく、**関数サイズ・ループの有無・キュー混雑**で動的に補正する。
- コンパイルは**優先度つきキュー**で熱い順に裏で処理し、待たずに差し替える。
  生成コードは有限の**コードキャッシュ**に置き、断片化・退去・古い版の破棄を管理する。
- 投機が外れると上位から下位へ落ちる（**脱最適化＝降格**）。
  何度も落ちる場所は**再投機を抑制**して下位に留め置く。
  ティアは昇格と降格を行き来する動的な状態。
- 実行中ループの差し替えには **OSR** が要る。
- 段の数は「起動の速さ vs 最高速度」のトレードオフで決まる。

次は、この章で何度も顔を出した「投機が外れたら戻る」の正体 ——
脱最適化と OSR を、ようやく正面から扱います。
