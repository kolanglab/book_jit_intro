# WebAssembly の JIT

第 II 部の締めくくりに、いちばん新しい実行基盤 —— **WebAssembly**
（略して Wasm）の JIT を見ます。WebAssembly は 2017 年に公開された
[](#cite:haas2017)、ブラウザやサーバで動く**ポータブルなバイナリ命令
形式**です。C・C++・Rust などをコンパイルした結果を、安全かつ高速に
実行することを狙って設計されました。

WebAssembly が本書にとって興味深いのは、それが**「JIT しやすいように
最初から設計された」中間表現**だからです。本書でここまで見てきた JIT の
苦労 —— 型が分からない、投機が要る、デオプトが要る —— の多くを、
WebAssembly は**言語の設計で回避**しています。「JIT の都合に合わせて
バイトコードを設計すると、どうなるか」の答えがここにあります。

## なぜ JavaScript でなく WebAssembly なのか

ブラウザにはすでに [V8](case-v8.md) のような高度な JavaScript JIT が
あります。なぜ別の実行基盤が要ったのでしょうか。

理由は、JavaScript が**JIT 泣かせ**だからです。[なぜ JIT か](why-jit.md)で
見たとおり、JavaScript は動的型付けで、`a + b` の意味が実行時まで
決まりません。だから JavaScript の JIT は、型を観測し、投機し、ガードを
置き、外れたらデオプトする —— という重い機械を必要とします。しかも、
同じ計算をしても**プロファイルが暖まるまで遅い**（[ウォームアップ問題](research-warmup.md)）。
数値計算やゲームのような重い処理を、JavaScript で安定して速く動かすのは
難しい。

WebAssembly は、この問題を**入口で断ち切り**ます。Wasm のバイナリは、
**型がすべて静的に決まっている**のです。

## WebAssembly が JIT を楽にする設計

WebAssembly のテキスト表現（WAT）で、二つの整数を足す関数を見てみましょう。

```wasm
(func $add (param $a i32) (param $b i32) (result i32)
  local.get $a      ;; a をスタックに積む（a は i32 と宣言済み）
  local.get $b      ;; b をスタックに積む（b も i32）
  i32.add)          ;; i32 同士の加算と最初から分かっている
```

注目すべきは `i32.add` です。これは「整数 32 ビットの加算」と**命令の
時点で確定**しています。JavaScript の `a + b` のように「型を見てから
何の加算か決める」必要がありません。WebAssembly の設計が JIT を楽にする
ポイントを整理します。

- **型が静的に決まっている**：`i32`・`i64`・`f32`・`f64` などの型が
  命令とローカル変数に明記されている。[なぜ JIT か](why-jit.md)で見た
  「型による分岐」のコストが、そもそも無い。
- **投機がほぼ要らない**：型が確定しているので、JavaScript のような
  「型を賭けてガードを置く」投機が原則不要。**デオプティマイズの機械が
  いらない**（あるいはごく限定的）。
- **構造化された制御フロー**：Wasm の分岐やループは、`goto` のような
  無秩序なジャンプでなく、`block`・`loop`・`if` という入れ子の構造を持つ。
  [基本ブロック](compilation-units.md)や[ループ](deopt-osr.md)が
  きれいに識別でき、解析が楽。
- **事前検証できる**：Wasm のバイナリは、実行前に「型が整合しているか・
  スタックが破綻しないか」を**高速に検証**できるよう設計されている。
  安全性のチェックを実行時に散らさなくてよい。

> [!IMPORTANT]
> WebAssembly は、本書で苦労して身につけた JIT 技法の多くを「**言語側で
> 不要にした**」中間表現です。型推論も投機もデオプトも、ほとんど要らない。
> その代わり、動的言語の柔軟さ（型が走りながら決まる）は手放しています。
> 「JIT を楽にするには、コンパイル前の言語をどう設計すればよいか」の
> 一つの極端な答えが WebAssembly だ、と捉えると、本書全体の中での
> 位置づけが見えます。

## それでも階層的：ベースラインと最適化の二段

型が決まっていて投機が要らないなら、JIT は簡単…かというと、まだ
[ウォームアップ](research-warmup.md)の問題が残ります。大きな Wasm
モジュールを、いきなり最適化コンパイラで全部コンパイルすると、起動が
遅れます。そこで Wasm エンジンも、[階層的 JIT](tiered.md)を採ります。

代表が V8 の Wasm 実装です。

- **Liftoff（ベースライン）**：Wasm の命令を**一回なめるだけ**で、
  最適化せずに素早く機械語を出すベースラインコンパイラ。起動を速くする。
  [コンパイルの単位](compilation-units.md)で見たテンプレート方式に近い
  作りです。
- **TurboFan（最適化）**：[V8 の最適化コンパイラ](case-v8.md)を Wasm にも
  使い、熱い関数を時間をかけて最適化する。

起動直後は Liftoff で素早く動かし、裏で TurboFan が熱い関数を最適化して
差し替える。JavaScript で見た[階層的](tiered.md)な構成が、Wasm でも
そのまま効くわけです。Firefox の SpiderMonkey も、ベースライン Wasm
コンパイラと最適化コンパイラ（Ion）の二段構成を採っています。

## ブラウザの外へ：Wasmtime と Cranelift

WebAssembly は、いまやブラウザの外でも広く使われています。サーバサイドや
プラグイン実行基盤として動かす**スタンドアロンランタイム**が登場しました。
代表が **Wasmtime** で、その JIT バックエンドが[フレームワーク](frameworks.md)の
章で紹介した **Cranelift**[](#cite:cranelift)です。

ここでも設計の力点は「**速く生成すること**」です。[ウォームアップ問題](research-warmup.md)で
見たように、サーバレス関数のような短命なワークロードでは、コンパイルの
速さが起動時間に直結します。Cranelift は LLVM より生成が速いことを
重視して設計されており、「起動の速い Wasm 実行」を支えています。
WebAssembly が、本書でたどってきた JIT 設計の論点 —— 生成の速さ対最適化の
品質、ベースライン対最適化、起動対最高速度 —— を、新しい舞台で
そっくり引き継いでいることが分かります。

## 次の課題：動的言語を Wasm に乗せる

最後に、現在進行形の研究テーマにふれます。WebAssembly は当初、C/C++/Rust の
ような**静的型の言語**の行き先として設計されました。では、Ruby や
Python のような**動的言語**を WebAssembly で動かしたいときはどうするか。

素朴には、「動的言語の処理系（インタプリタ）ごと WebAssembly にコンパイル
して、その上で動的言語を実行する」ことになります。実際 CRuby も
`wasm32-wasi` 向けにビルドでき、ブラウザ上で Ruby が動きます。しかし
これだと、Wasm の上でインタプリタが動き、その上で動的言語が動く、という
**二重の実行**になり、せっかくの Wasm の速さを動的言語側で活かしきれません。

そこで、WebAssembly 自体を拡張する動きが進んでいます。オブジェクトや配列を
Wasm が直接扱える **WasmGC**、関数を値として柔軟に扱う仕組みなど、
**動的言語の処理系を Wasm の上でうまく JIT 的に動かす**ための機能拡張です。
「Wasm の上で、動的言語をどう速く動かすか」は、[メタトレーシング](meta-tracing.md)や
[部分評価](research-partial-eval.md)で見た「インタプリタを速くする」技法の、
新しい応用先になりつつあります。

> [!NOTE]
> ここに、本書の主題がぐるりと一周する面白さがあります。WebAssembly は
> 「JIT を不要にするほど素直な中間表現」として生まれましたが、その上で
> 動的言語を速く動かそうとすると、また[投機](deopt-osr.md)や
> [トレーシング](tracing.md)といった本書の技法が必要になる。JIT の
> 知恵は、実行基盤が新しくなっても、形を変えて生き続けるのです。

## まとめ

- **WebAssembly**[](#cite:haas2017)は、JIT しやすいよう設計された
  ポータブルなバイナリ命令形式。型が静的に決まり、制御フローが構造化され、
  事前検証できる。
- そのため、JavaScript の JIT が必要とした**型推論・投機・デオプト**が
  ほぼ要らない。JIT の苦労の多くを**言語設計で回避**している。
- それでも起動の速さのため**階層的**。V8 は Liftoff（ベースライン）＋
  TurboFan（最適化）の二段。SpiderMonkey も同様。
- ブラウザ外では **Wasmtime**＋**Cranelift** が、「速く生成する」JIT で
  短命ワークロードの起動を支える。
- **動的言語を Wasm に乗せる**のは現在の課題。WasmGC などの拡張と、本書の
  JIT 技法の応用が交差する最前線。

これで第 II 部・最新研究を終えます。理論と技法の地図がそろいました。
第 III 部では、これらが実在の処理系でどう組み合わされているかを、
具体的なコードとともに見ていきます。
