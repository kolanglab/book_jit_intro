# WebAssembly の JIT

第 II 部の締めくくりに、いちばん新しい実行基盤 —— **WebAssembly**
（略して Wasm）の JIT を見ます。WebAssembly は 2017 年に公開された
[](#cite:haas2017)、ブラウザやサーバで動く**ポータブルなバイナリ命令
形式**です。C・C++・Rust などをコンパイルした結果を、安全かつ高速に
実行することを狙って設計されました。原論文は、低水準なコンパイル
ターゲットが備えるべき性質として **safe（安全に実行できる）・fast
（高速に実行できる）・portable（言語・ハードウェア・プラットフォーム
非依存）** な意味論と、**compact（コンパクト）で効率よく検証・コンパイル
できる表現**を掲げています[](#cite:haas2017)。

WebAssembly が本書にとって興味深いのは、それが**「JIT しやすいように
最初から設計された」中間表現**だからです。本書でここまで見てきた JIT の
苦労 —— 型が分からない、投機が要る、脱最適化が要る —— の多くを、
WebAssembly は**言語の設計で回避**しています。「JIT の都合に合わせて
バイトコードを設計すると、どうなるか」の答えがここにあります。

## なぜ JavaScript でなく WebAssembly なのか

ブラウザにはすでに [V8](case-v8.md) のような高度な JavaScript JIT が
あります。なぜ別の実行基盤が要ったのでしょうか。

理由は、JavaScript が**JIT 泣かせ**だからです。[なぜ JIT か](why-jit.md)で
見たとおり、JavaScript は動的型付けで、`a + b` の意味が実行時まで
決まりません。だから JavaScript の JIT は、型を観測し、投機し、ガードを
置き、外れたら脱最適化する —— という重い機械を必要とします。しかも、
同じ計算をしても**プロファイルが暖まるまで遅い**（[ウォームアップ問題](research-warmup.md)）。
数値計算やゲームのような重い処理を、JavaScript で安定して速く動かすのは
難しい。

WebAssembly は、この問題を**入口で断ち切り**ます。Wasm のバイナリは、
**型がすべて静的に決まっている**のです。その効き目は数値にも出ています。
原論文は PolyBenchC ベンチマーク（24 本）で Wasm の実行時間をネイティブ
コードと比べ、**7 本がネイティブの 10% 以内、ほぼすべてがネイティブの
2 倍以内**に収まったと報告しています[](#cite:haas2017)。同じベンチマークを
asm.js と比べると、Wasm は**平均 33.7% 高速**でした[](#cite:haas2017)。
「JavaScript で重い処理を安定して速く」が難しかったのに対し、Wasm は
ネイティブに肉薄する速度を、移植性を保ったまま出せるわけです。

## WebAssembly が JIT を楽にする設計

WebAssembly のテキスト表現（WAT）で、二つの整数を足す関数を見てみましょう。

```wasm
(func $add (param $a i32) (param $b i32) (result i32)
  local.get $a      ;; a をスタックに積む（a は i32 と宣言済み）
  local.get $b      ;; b をスタックに積む（b も i32）
  i32.add)          ;; i32 同士の加算と最初から分かっている
```

注目すべきは `i32.add` です。これは「整数 32 ビットの加算」と**命令の
時点で確定**しています。JavaScript の `a + b` のように「型を見てから
何の加算か決める」必要がありません。WebAssembly の設計が JIT を楽にする
ポイントを整理します。

- **型が静的に決まっている**：基本の値型は `i32`・`i64`・`f32`・`f64`
  の四つ（haas2017 の MVP 時点。のちに SIMD の `v128` や参照型が加わった）で
  [](#cite:haas2017)、命令とローカル変数に明記されている。
  [なぜ JIT か](why-jit.md)で見た「型による分岐」のコストが、そもそも無い。
- **投機がほぼ要らない**：型が確定しているので、JavaScript のような
  「型を賭けてガードを置く」投機が原則不要。**脱最適化の機械が
  いらない**（あるいはごく限定的）。
- **構造化された制御フロー**：Wasm の分岐やループは、`goto` のような
  無秩序なジャンプでなく、`block`・`loop`・`if` という入れ子の構造を持つ。
  原論文によれば、この設計は**既約（irreducible）なループや、スタック高さの
  ずれた分岐が構造上できない**ことを保証し、その結果 Wasm コードは
  **1 パスで検証でき、1 パスでコンパイルでき、さらに 1 パスで SSA 形式の
  中間表現に変換さえできる**[](#cite:haas2017)。[基本ブロック](compilation-units.md)や
  [ループ](deopt-osr.md)がきれいに識別でき、解析が楽。
- **事前検証できる**：Wasm のバイナリは、実行前に「型が整合しているか・
  スタックが破綻しないか」を**高速に検証**できるよう設計されている。
  検証は本質的に型検査であり、原論文は型システムの**健全性（soundness）を
  証明**しています[](#cite:haas2017)。安全性のチェックを実行時に散らさなくてよい。

### なぜ投機・脱最適化がほぼ要らないのか、一段具体に

「型が静的に決まっているから楽」というのを、もう一段ほぐします。
JavaScript の JIT が `a + b` をコンパイルするとき、内部では次のような
**投機**をしていました（[なぜ JIT か](why-jit.md)・[脱最適化と OSR](deopt-osr.md)）。

```ruby
# JavaScript の a + b を JIT する際の擬似コード（投機あり）
# 「両方とも i32 だろう」と賭けて、外れたらインタプリタへ戻る
def emit_add(a, b)
  emit "guard_is_smi #{a}, deopt_label"   # a が整数でなければ脱最適化
  emit "guard_is_smi #{b}, deopt_label"   # b が整数でなければ脱最適化
  emit "add_i32 #{a}, #{b}"               # ここでようやく整数加算
  # deopt_label: では観測した型を捨て、インタプリタの状態を復元する
end
```

賭けが外れれば、積み上げた最適化コードを捨ててインタプリタの状態へ
**脱最適化**する必要がありました。スタック上の値・ローカル変数を
すべて復元するこの仕組みは、本書で最も重い機械の一つです。

一方 WebAssembly の `i32.add` には、この `guard_is_smi` 二行も
`deopt_label` も**まるごと不要**です。検証器がモジュールの読み込み時に
「`i32.add` の二つのオペランドはスタック上で必ず `i32` である」ことを
保証しているので、JIT は無条件に整数加算一命令を吐けます。

```ruby
# WebAssembly の i32.add を JIT する際の擬似コード（投機なし）
def emit_add
  rhs = stack.pop   # 検証済み：必ず i32
  lhs = stack.pop   # 検証済み：必ず i32
  reg = alloc_reg
  emit "add_i32 #{reg}, #{lhs}, #{rhs}"   # ガードも脱最適化もなし
  stack.push(reg)
end
```

> [!NOTE]
> 効果はコード量と分岐の両方に効きます。投機ありの経路では
> 加算一回につき**機械語が 3 命令前後**になり、しかも `guard` 由来の
> 条件分岐が[分岐予測](why-jit.md)を乱します。投機なしなら**1 命令**で
> 済み、分岐も増えません。脱最適化のためのメタデータ（どのレジスタが
> どのローカル変数か、という対応表）を関数ごとに持つ必要もなく、
> その分だけ生成コードと付随データが小さくなります。

構造化制御フローも、同じ理屈で効きます。Wasm の `block`・`loop`・`if` は
**入れ子**になっていて、`br`（分岐）は「現在の入れ子から N 段外へ抜ける」
としか書けません。任意番地への `goto` が存在しないので、バイトコードを
読みながら[基本ブロック](compilation-units.md)と[制御フローグラフ](ir.md)を
**一度のなめ（1 パス）で**素直に組み立てられます。JavaScript の
バイトコードのように「飛び先を後で解決し、合流点を探す」手間が要りません。

```ruby
# 構造化制御フローのコンパイルは、入れ子をたどる素直な再帰になる
def compile(instr)
  case instr.op
  when :block then with_label(:end) { compile_body(instr.body) }
  when :loop  then with_label(:top) { compile_body(instr.body) }  # 先頭にラベル
  when :br    then emit "jmp #{label_at_depth(instr.depth)}"      # N 段外へ
  when :if    then emit_branch_then_else(instr)
  else emit_normal(instr)
  end
end
```

> [!IMPORTANT]
> WebAssembly は、本書で苦労して身につけた JIT 技法の多くを「**言語側で
> 不要にした**」中間表現です。型推論も投機も脱最適化も、ほとんど要らない。
> その代わり、動的言語の柔軟さ（型が走りながら決まる）は手放しています。
> 「JIT を楽にするには、コンパイル前の言語をどう設計すればよいか」の
> 一つの極端な答えが WebAssembly だ、と捉えると、本書全体の中での
> 位置づけが見えます。

## それでも階層的：ベースラインと最適化の二段

型が決まっていて投機が要らないなら、JIT は簡単…かというと、まだ
[ウォームアップ](research-warmup.md)の問題が残ります。大きな Wasm
モジュールを、いきなり最適化コンパイラで全部コンパイルすると、起動が
遅れます。そこで Wasm エンジンも、[階層的 JIT](tiered.md)を採ります。

代表が V8 の Wasm 実装です。

- **Liftoff（ベースライン）**：Wasm の命令を**一回なめるだけ**で、
  最適化せずに素早く機械語を出すベースラインコンパイラ。起動を速くする。
  [コンパイルの単位](compilation-units.md)で見たテンプレート方式に近い
  作りです。
- **TurboFan（最適化）**：[V8 の最適化コンパイラ](case-v8.md)を Wasm にも
  使い、熱い関数を時間をかけて最適化する。

起動直後は Liftoff で素早く動かし、裏で TurboFan が熱い関数を最適化して
差し替える。JavaScript で見た[階層的](tiered.md)な構成が、Wasm でも
そのまま効くわけです。Firefox の SpiderMonkey も、ベースライン Wasm
コンパイラと最適化コンパイラ（Ion）の二段構成を採っています。実際、
原論文の時点ですでに SpiderMonkey は**二つの Wasm コンパイル段**を持ち、
第一段は「検証と一体で、機械語を 1 パスで吐く Wasm 専用の高速ベースライン
JIT」で、内部 IR を作らず、前向きの 1 パスで素朴な貪欲レジスタ割り当てを
する作りだと報告されています[](#cite:haas2017)。なお V8 の Liftoff
（後述）はこの原論文より後に登場したもので、上記の SpiderMonkey の
ベースラインと同じ「検証と一体・1 パス・IR なし」の発想を、V8 でも
体現したものです。

### ベースラインの具体：バイトコード 1 命令 → 機械語テンプレート

Liftoff のようなベースラインコンパイラがなぜ速いのかは、その作りを見ると
分かります。最適化コンパイラが[中間表現](ir.md)を組み立てて何度も
パスをかけるのに対し、ベースラインは**Wasm 命令を一つ読むたびに、
対応する機械語の定型（テンプレート）をそのまま吐く**だけです。
[コンパイルの単位](compilation-units.md)で見たテンプレート方式そのものです。

```ruby
# Liftoff 風ベースラインコンパイラの中核：命令 → 機械語テンプレート
# スタックマシンの「論理スタック」を、レジスタと実機スタックに割り当てる
class Liftoff
  def compile_instr(instr)
    case instr.op
    when :i32_const
      # 定数はレジスタへ即値ロード（または論理スタックに定数として記録）
      reg = @stack.alloc_reg
      emit "mov #{reg}, #{instr.value}"
    when :local_get
      reg = @stack.alloc_reg
      emit "mov #{reg}, [rbp - #{@locals.offset(instr.idx)}]"
    when :i32_add
      b = @stack.pop_to_reg   # 検証済みなので型チェック不要
      a = @stack.pop_to_reg
      emit "add #{a}, #{b}"   # その場で 1 命令
      @stack.push(a)
    when :call
      spill_volatile_regs     # 呼び出し規約に合わせて積む
      emit "call #{instr.target}"
      @stack.push(RET_REG)
    # ...各 Wasm 命令にテンプレートが一つずつ対応する
    end
  end
end
```

ポイントは**最適化をしないこと自体が速さの源**だという点です。
レジスタ割り当ても「空いているレジスタを順に使い、足りなければ
実機スタックへ退避（spill）する」程度の素朴な方式で済ませます。
[線形走査](ir.md)[](#cite:poletto1999)のような本格的な割り当ても、共通部分式の
除去も、ループ最適化もしません。そのぶん、コンパイルは命令数に**ほぼ
比例（O(n) の 1 パス）**で終わります。

> [!TIP]
> 概算で感覚をつかんでおきましょう。ベースライン段は最適化段に比べて
> **コンパイルが一桁ほど速い**かわりに、生成コードの実行は数割から
> 数倍遅い、というのが各エンジンに共通する大まかな傾向です。だから
> 「起動はベースライン、定常はベースラインを最適化版で置き換える」という
> [階層的](tiered.md)な役割分担が成立します。[ウォームアップ](research-warmup.md)で
> 見たコスト構造が、そのまま Wasm にも当てはまります。

### コンパイル戦略：遅延コンパイル・ティアアップ・コードキャッシュ

二段構成に加えて、Wasm エンジンは起動を速くするための戦略をいくつも
重ねています。いずれも「**いま要らない仕事は後回しにする**」発想です。

- **遅延コンパイル（lazy compilation）**：モジュールに 1 万個の関数が
  あっても、起動時に全部はコンパイルしません。各関数を**最初に呼ばれた
  瞬間まで**コンパイルを遅らせます。一度も呼ばれない関数（エラー処理や
  まれな経路）には一切コストを払わずに済みます。
- **ティアアップ（tier-up）**：[プロファイリング](profiling.md)で
  関数の呼び出し回数を数え、閾値を超えたらベースラインから最適化段へ
  昇格させます。下の擬似コードのとおり、判断自体はカウンタ比較だけです。
- **コードキャッシュ**：コンパイル結果を**ディスクに保存**しておき、
  次回同じモジュールを読むときはコンパイルを丸ごと省きます。ブラウザは
  同じ `.wasm` への再訪で、サーバランタイムは同じ関数イメージの再起動で
  効きます。起動を「コンパイル時間ゼロ」に近づける切り札です。

```ruby
# 遅延コンパイル＋ティアアップの骨格
def call_wasm_func(f)
  unless f.compiled?
    f.code = baseline_compile(f)   # 初回呼び出しで初めてベースライン生成
  end
  f.hit_count += 1
  if f.hit_count == TIER_UP_THRESHOLD     # 例：10 回など
    enqueue_background_optimize(f)         # 裏で最適化段がコンパイル
    # 完成したら f.code を差し替える（実行中の呼び出しはそのまま）
  end
  f.code.call
end
```

> [!WARNING]
> 遅延コンパイルは起動を速くしますが、**初回呼び出しに遅延の山**を作ります。
> めったに呼ばれない関数が、ある瞬間に初めて呼ばれてベースライン
> コンパイルが走ると、その 1 回だけ目立って遅くなります。応答時間の
> 安定性が重要な用途では、起動時に**先回りでコンパイルしておく**
> （eager / プリコンパイル）選択肢とのトレードオフになります。

## ブラウザの外へ：Wasmtime と Cranelift

WebAssembly は、いまやブラウザの外でも広く使われています。サーバサイドや
プラグイン実行基盤として動かす**スタンドアロンランタイム**が登場しました。
代表が **Wasmtime** で、その JIT バックエンドが[フレームワーク](frameworks.md)の
章で紹介した **Cranelift**[](#cite:cranelift)です。

ここでも設計の力点は「**速く生成すること**」です。[ウォームアップ問題](research-warmup.md)で
見たように、サーバレス関数のような短命なワークロードでは、コンパイルの
速さが起動時間に直結します。Cranelift は LLVM[](#cite:lattner2004)より
生成が速いことを重視して設計されており、「起動の速い Wasm 実行」を
支えています。

なぜ Cranelift は速いのか、を具体に落とすと、設計の選択が見えてきます。

- **最適化パスを欲張らない**：LLVM は何十ものパスを段階的にかけて
  最高品質のコードを狙いますが、Cranelift はパス数を絞ります。生成品質を
  数割譲ってでも、コンパイル時間を縮める方を選んでいます。
- **e-graph による書き換え**：式の等価な書き換え候補を一つのグラフに
  まとめて持ち、選択を後回しにすることで、何度も[中間表現](ir.md)を
  作り直す手間を避けています。
- **アーキテクチャ非依存の下ごしらえ＋機械ごとの仕上げ**を分け、
  移植性と生成速度を両立させています。

短命ワークロードでの効き目は、ごく単純なモデルでも見えます。

```ruby
# 短命ワークロードでは「コンパイル+実行」の合計が効く（相対値の概算）
def total_time(compile_ms, run_per_call_ms, calls)
  compile_ms + run_per_call_ms * calls
end

# 高品質だが生成が遅い最適化コンパイラ
opt      = total_time(100.0, 1.0, calls)   # 生成 100、1 回 1.0
# 生成が速いベースライン／Cranelift 寄り
fast_gen = total_time( 10.0, 1.4, calls)   # 生成 10、1 回 1.4（実行はやや遅い）

# calls = 50（短命）なら   fast_gen=80   < opt=150   → 速い生成が勝つ
# calls = 1000（長命）なら fast_gen=1410 > opt=1100  → 高品質が勝つ
```

呼び出し回数が少ない短命なワークロードでは**生成の速さがそのまま勝敗を
決め**、長く動き続けるワークロードでは最適化品質が効いてくる。だから
[階層的](tiered.md)に「まず速く出し、回数が伸びたら作り直す」のが合理的
だと、改めて確認できます。WebAssembly が、本書でたどってきた JIT 設計の
論点 —— 生成の速さ対最適化の品質、ベースライン対最適化、起動対最高速度 ——
を、新しい舞台でそっくり引き継いでいることが分かります。

> [!NOTE]
> ここに、これまでの章との**設計思想の違い**がはっきり出ます。動的言語の
> JIT は、実行時に観測した型やプロファイルを使って「いまこの場では
> 整数しか来ない」と**攻めの特殊化**をし、外れたら[脱最適化](deopt-osr.md)で
> 取り返しました。WebAssembly はそれをほとんどしません。移植性・安全性・
> 予測可能性（同じモジュールはどこでも同じく動く）を取るかわりに、実行時
> 情報を使った攻めの特殊化という最後のひと押しは、設計として手放して
> いるのです。「賭けて勝つ」より「賭けずに安定して速い」を選んだ、と
> 言えます。

## 次の課題：動的言語を Wasm に乗せる

最後に、現在進行形の研究テーマにふれます。WebAssembly は当初、C/C++/Rust の
ような**静的型の言語**の行き先として設計されました。では、Ruby や
Python のような**動的言語**を WebAssembly で動かしたいときはどうするか。

素朴には、「動的言語の処理系（インタプリタ）ごと WebAssembly にコンパイル
して、その上で動的言語を実行する」ことになります。実際 CRuby も
`wasm32-wasi` 向けにビルドでき、ブラウザ上で Ruby が動きます。しかし
これだと、Wasm の上でインタプリタが動き、その上で動的言語が動く、という
**二重の実行**になり、せっかくの Wasm の速さを動的言語側で活かしきれません。

そこで、WebAssembly 自体を拡張する動きが進んでいます。オブジェクトや配列を
Wasm が直接扱える **WasmGC**、関数を値として柔軟に扱う仕組みなど、
**動的言語の処理系を Wasm の上でうまく JIT 的に動かす**ための機能拡張です。
「Wasm の上で、動的言語をどう速く動かすか」は、[メタトレーシング](meta-tracing.md)や
[部分評価](research-partial-eval.md)で見た「インタプリタを速くする」技法の、
新しい応用先になりつつあります。

ただし、ここで**話が一巡して戻ってきます**。参照型や WasmGC が入ると、
「ある参照が指す先の具体的な型は実行時まで分からない」という状況が
Wasm の世界にも生まれます。たとえば WasmGC のオブジェクトに対する
仮想呼び出しは、こんな姿になります。

```ruby
# 参照型・WasmGC が入ると、多態（実行時に型が決まる）が戻ってくる
# obj.method() は、obj の実際の型を見るまで飛び先が決まらない
def emit_virtual_call(obj, method)
  emit "load_type_id #{obj}"          # 実行時に型 ID を読む
  emit "lookup_vtable method, type"   # 表を引いて飛び先を決める
  emit "call_indirect"                # 間接呼び出し（飛び先が動的）
end
```

間接呼び出しが出てきた瞬間、本書でおなじみの最適化余地が**再び現れます**。
「この呼び出し地点では、ほとんど同じ型しか来ない」と[プロファイリング](profiling.md)で
観測できれば、[なぜ JIT か](why-jit.md)で見た**インラインキャッシュ**や
[投機](deopt-osr.md)的なインライン展開が、また有効になるのです。つまり
WasmGC は、静的型一辺倒だった Wasm に**多態を持ち込み、それと引き換えに
投機と脱最適化の余地を再び開く**ことになります。

> [!NOTE]
> ここに、本書の主題がぐるりと一周する面白さがあります。WebAssembly は
> 「JIT を不要にするほど素直な中間表現」として生まれましたが、その上で
> 動的言語を速く動かそうとすると、また[投機](deopt-osr.md)や
> [トレーシング](tracing.md)といった本書の技法が必要になる。JIT の
> 知恵は、実行基盤が新しくなっても、形を変えて生き続けるのです。

## まとめ

- **WebAssembly**[](#cite:haas2017)は、JIT しやすいよう設計された
  ポータブルなバイナリ命令形式。設計目標は safe・fast・portable・compact。
  MVP 時点の基本値型は `i32`・`i64`・`f32`・`f64` で、現在は `v128` や参照型も
  加わったが、命令に型が明示され事前検証される軸は同じ。制御フローが
  構造化され、健全性が証明された型検査で事前検証できる。PolyBenchC では
  7 本がネイティブの 10% 以内、ほぼすべてが 2 倍以内に収まった[](#cite:haas2017)。
- そのため、JavaScript の JIT が必要とした**型推論・投機・脱最適化**が
  ほぼ要らない。`i32.add` は `guard` も `deopt` も無しに 1 命令で吐ける。
  構造化制御フローのおかげで CFG も 1 パスで素直に組める。JIT の苦労の
  多くを**言語設計で回避**している。
- それでも起動の速さのため**階層的**。V8 は Liftoff（ベースライン）＋
  TurboFan（最適化）の二段。ベースラインは「命令 1 つ → 機械語テンプレート」
  の 1 パスで速く、最適化段が裏で差し替える。SpiderMonkey も同様。
- **遅延コンパイル・ティアアップ・コードキャッシュ**で「いま要らない仕事を
  後回し」にし、起動を縮める。遅延コンパイルは初回呼び出しの遅延と
  トレードオフ。
- ブラウザ外では **Wasmtime**＋**Cranelift** が、「速く生成する」JIT で
  短命ワークロードの起動を支える。短命なら生成速度、長命なら最適化品質が
  効くという構図はこれまでの章と同じ。
- WebAssembly は移植性・安全・予測可能性を取り、**実行時情報による攻めの
  特殊化はしない**という設計思想。動的言語の JIT と対照的。
- **動的言語を Wasm に乗せる**のは現在の課題。WasmGC や参照型が**多態を
  持ち込む**と、インラインキャッシュや投機の余地が再び開く。本書の JIT
  技法の応用が交差する最前線。

これで第 II 部・最新研究を終えます。理論と技法の地図がそろいました。
第 III 部では、これらが実在の処理系でどう組み合わされているかを、
具体的なコードとともに見ていきます。
