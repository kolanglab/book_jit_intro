# JIT とセキュリティ

ここまでの章で、JIT が何をする装置なのかは骨身にしみたはずです。
[最小の JIT](hello-jit.md)で見たとおり、JIT の正体は身も蓋もなく言えば
**「データ（ただのバイト列）を、実行可能なコードに変える」**装置です。
Ruby の配列に詰めた整数の列を `mmap` でもらった領域へ書き込み、その先頭
アドレスを関数ポインタとみなして呼ぶ —— それだけのことでした。

これは恐ろしく強力です。そして、強力なものは**攻撃者にとっても魅力的**
です。「実行中にメモリの中身を実行可能コードに変えられる」という性質は、
裏を返せば「攻撃者がそのメモリに自分の都合のいいバイト列を紛れ込ませ
られたら、それがそのまま CPU に実行される」かもしれない、ということです。
ふつうのプログラムなら「データ領域は実行できない」という OS の壁に守られ
ますが、JIT はその壁に自分から穴を開ける商売をしています。

> [!NOTE]
> この章で扱う「安全」は、[脱最適化と OSR](deopt-osr.md)で扱った「安全」
> とは**別種**のものです。あちらは「投機（型の決め打ち）が外れても、
> 正しい答えに戻れる」という**正しさ**の安全網でした。こちらは「攻撃者に
> JIT の仕組みを悪用されない」という**セキュリティ**の安全です。同じ
> 「安全」でも守る相手が違います。前者の相手は自分の楽観的な賭けで、
> 後者の相手は悪意ある第三者です。

本章は防御の章です。攻撃の「考え方」と「なぜ対策が要るのか」を説明し、
重心は徹底して**対策**に置きます。すぐ動く攻撃コードは書きません
（書く必要もありません。原理が分かれば、なぜ対策が効くかは分かります）。
題材は引き続き Linux の x86-64、おもちゃは [最小の JIT](hello-jit.md) の
延長です。

## なぜ JIT はとりわけ狙われるのか

ふつうの実行ファイルは、コンパイル時に機械語が固まり、実行時には
**読めて実行できるが書き換えられない（RX）**領域に置かれます。攻撃者が
任意のコードを送り込もうとしても、書き込める領域（スタックやヒープ）は
**実行できない（RW）**ので、書いたものは命令として走りません。この
「データは実行できない」というルールを **DEP**（Data Execution
Prevention）や **NX**（No-eXecute）ビットと呼びます。攻撃者はこの壁を
越えるために、既存のコード片を継ぎはぎして悪さをする ROP（Return-Oriented
Programming）のような遠回りを強いられます。

ところが JIT は、この壁に正面から穴を開けます。設計上**どうしても**、
「いま書き込んだバイト列を、このあと実行可能にする」瞬間が要るからです。
ここに二つの攻撃面が生まれます。

- **書き込み可能で実行可能なメモリ（RWX）そのもの**：もし JIT が手を
  抜いて「読み書き実行すべて可」の領域を持ち続けると、攻撃者が別のバグ
  （バッファオーバーフローなど）でその領域に書き込めたとき、書いた
  バイト列が**そのまま命令として走る**。NX の壁が無効化された状態です。
- **JIT に生成させる中身**：攻撃者が JIT へ流し込む入力（JavaScript の
  ソースなど）を通じて、**生成されるバイト列の一部を間接的に制御**できる
  場合がある。出力された機械語の中に、攻撃者の都合のいいバイトを忍び込ま
  せられる余地が生まれます（後述の JIT スプレー）。

> [!IMPORTANT]
> JIT のセキュリティ対策は、ほぼすべて「**この穴をできるだけ小さく、
> 短時間に、予測しにくくする**」ための工夫だと捉えると見通しがよく
> なります。穴を完全に塞ぐことはできません（塞いだら JIT が成り立たない）。
> だから「書ける時間と実行できる時間を重ねない」「攻撃者に中身を予測
> させない」「場所を予測させない」という三方向で、穴を実用上踏みにくく
> していきます。

## W^X：書けるなら実行できない、実行できるなら書けない

[最小の JIT](hello-jit.md) で予告した原則が **W^X**（Write XOR eXecute、
書き込みと実行を同時には許さない）です。一つのメモリ領域について、
**「書き込み可」と「実行可」を同時には立てない**。書いている最中は実行
できず、実行できるようになったら書けない。こうすれば、攻撃者が「実行
可能な領域へ書き込む」隙が原理的に消えます。

### 二段構え：RW で書いてから RX へ落とす

いちばん素直な実装が、[最小の JIT](hello-jit.md) で見た二段構えです。

1. まず **RW（読み書きのみ、実行不可）** で領域をもらう。
2. そこへ機械語を**書き込む**。
3. 書き終わったら `mprotect` で **RX（読み実行のみ、書き込み不可）** へ
   落とす。

```ruby
# W^X 版（再掲）：RW で確保 → 書き込み → RX へ落とす
def jit(bytes)
  size = 4096
  mem  = Mmap.call(nil, size,
                   PROT_READ | PROT_WRITE,          # EXEC は付けない
                   MAP_PRIVATE | MAP_ANON, -1, 0)
  code = bytes.pack('C*')
  Fiddle::Pointer.new(mem.to_i)[0, code.bytesize] = code
  Mprotect.call(mem, size, PROT_READ | PROT_EXEC)   # ここで実行可へ、書き込み不可に
  mem
end
```

このとき大事なのは、**RWX という状態を一瞬たりとも作らない**ことです。
「とりあえず RWX でもらって、あとで書き込み権限を外す」のは W^X 違反で、
外す前のわずかな窓を狙われます。最初から RW でもらい、書き終えてから
**RW を捨てて RX に切り替える**のが正しい順序です。

> [!WARNING]
> 「RWX を一瞬作って、すぐ RX に落とすから大丈夫」という油断が事故の
> もとです。マルチスレッドの JIT では、その「一瞬」のあいだに別スレッドが
> その領域に触れられます。攻撃者は競合（レース）を起こして、書き込み可能な
> 窓に滑り込もうとします。**RWX は窓を「狭くする」のではなく「作らない」**
> のが鉄則です。

### 二重マッピング：同じ物理ページを RW と RX の二つの窓で見る

二段構えには弱点があります。`mprotect` で権限を切り替えるたびに**システム
コール**が要り、しかもページ単位でしか効きません。コードを少し書き足す
たびに RW へ戻して、また RX へ落として…とやると、切り替えコストが効いて
きます。

そこで使われるのが**二重マッピング**（dual mapping、別名 split mapping）
です。考え方はこうです。

- 同じ**物理ページ**を、**二つの異なる仮想アドレス**から見えるように
  マップする。
- 一方の仮想アドレス窓は **RW**（書き込み専用の窓）、もう一方は **RX**
  （実行専用の窓）にする。
- JIT は **RW 窓のアドレスへ書き込み**、CPU は **RX 窓のアドレスを実行**
  する。書いた内容は同じ物理ページなので、RX 窓から見える命令にちゃんと
  反映される。

```
        物理ページ（実体は一つ）
        ┌───────────────┐
        │  機械語のバイト列   │
        └───────────────┘
          ▲              ▲
   書き込む │              │ 実行する
     RW 窓 │              │ RX 窓
 0x7f...a000           0x7f...b000   ← 別々の仮想アドレス
```

これなら、**いちいち `mprotect` を呼ばずに**「書く」と「実行する」を
両立できます。しかも、それぞれの窓は片方の権限しか持たないので、W^X は
保たれています（RW 窓は実行不可、RX 窓は書き込み不可）。Linux なら
`memfd_create` で無名のメモリファイルを作り、それを二か所に `mmap` する、
といった実装になります。

> [!NOTE]
> 二重マッピングは「速いが、攻撃面がやや広い」トレードオフを抱えます。
> RW 窓のアドレスを攻撃者に知られると、そこへの書き込みが RX 窓に反映
> される —— つまり実行コードを書き換えられてしまう。だから RW 窓の
> アドレスは**秘密**に保つ（後述のランダム化と相性がよい）必要があり
> ます。二段構えの `mprotect` 方式は遅いぶん、書き込み可能な窓が
> ふだん存在しないので攻撃面は小さい。速さと攻撃面のトレードオフです。

### スレッド単位の切り替え：Apple Silicon の場合

[最小の JIT](hello-jit.md) でも触れたとおり、Apple Silicon（ARM の Mac）や
iOS では W^X が**ハードウェアとカーネルで強制**され、同じページを RW と RX
で同時に持てません。代わりに、**実行している今のスレッドだけ**書き込み可否を
切り替える仕組みが用意されています。

```c
// Apple のプラットフォームでの典型的な書き換え（概念）
pthread_jit_write_protect_np(false);  // このスレッドだけ「JIT 領域へ書ける」状態に
memcpy(jit_area, code, len);          // 機械語を書き込む
pthread_jit_write_protect_np(true);   // 書き込み禁止へ戻す（＝実行可へ）
// 以後この領域を関数として呼べる
```

ポイントは、これが**スレッドローカル**だということです。別スレッドから
見れば、その JIT 領域はずっと実行専用（書き込み不可）のまま。だから、
書き込み中の窓に他スレッドが滑り込む前述のレースが、原理的に起きにくく
なります。`pthread_jit_write_protect_np` 自体はシステムコールより軽い
（プロセッサのレジスタ一発で切り替わる）ので、二段構えの `mprotect` より
ずっと安い。

> [!IMPORTANT]
> どの方式でも共通する勘どころは、**切り替えの粒度とバッチ化**です。
> コード片を一つ書くたびに切り替えると、切り替えコスト（システムコールや
> レジスタ操作）が積み上がります。だから本物の JIT は、コードを**大きな
> アリーナ**にまとめて書き、「書き込みモードへ切り替え → まとめて全部書く
> → 実行モードへ切り替え」を**一回**で済ませる、という工夫をします。
> 「正しさ（W^X を一度も破らない）」と「速さ（切り替え回数を減らす）」の
> 綱引きで、ここはほぼ常に**バッチ化**が選ばれます。

## JIT スプレー：生成されたコードに毒を仕込む

W^X を守れば「攻撃者が書いた生のバイト列を実行する」道は塞げます。
ところが、もっと巧妙な攻撃があります。**攻撃者自身は一バイトも書かず、
JIT に毒を生成させる**のです。これが **JIT スプレー**（JIT spraying）です。

### 命令境界をずらす、という発想

x86-64 の機械語は**可変長**で、しかも**どこからでもデコードを始められて
しまう**という、攻撃者にとって都合のいい性質を持ちます。同じバイト列でも、
読み始める位置が 1 バイトずれると、まったく別の命令列に化けます。

[最小の JIT](hello-jit.md) で見た「即値を埋め込む」命令を思い出して
ください。整数 `0x90909090` を `eax` に入れる命令は、こうでした。

```
B8 90 90 90 90    mov eax, 0x90909090
```

`B8` のあとに、即値 `90 90 90 90` が**そのままバイト列として**並びます。
ところで `90` という 1 バイトは、x86 では `nop`（何もしない命令）です。
だから、もし CPU が `B8` を飛ばして**即値の途中（2 バイト目）から**
デコードを始めると、こう見えます。

```
90    nop
90    nop
90    nop
...   （攻撃者が即値として制御したバイト列が、命令列として実行される）
```

攻撃者は、JavaScript なら `var x = 0x90909090; var y = 0x...; ...` のように
**定数を大量に並べた式**を JIT に食わせます。JIT は素直に、それらの定数を
`mov` の即値として機械語へ埋め込みます。すると、生成された RX 領域の中に、
**攻撃者が中身を制御できるバイト列**が点々と埋まる。あとは別のバグで実行を
**即値の途中の番地へジャンプ**させられれば、攻撃者の仕込んだバイト列が
命令として走り出す —— これが JIT スプレーの骨格です。

> [!NOTE]
> 「スプレー（spray、撒く）」の名のとおり、攻撃者は同じパターンを**大量に**
> 生成させてメモリ中にばらまきます。ジャンプ先の番地を正確に当てられなくても、
> 広い範囲に同じ毒を撒いておけば「どこかには当たる」確率を上げられる。
> しかも先頭を `nop` 相当のバイト（NOP スレッド）で埋めておけば、多少
> ずれた番地に飛んでも、`nop` を滑って目的のガジェットに到達できます。

この攻撃の怖さは、**W^X を完全に守っていても成立しうる**点です。生成された
コードは正規の手順で RX になっており、攻撃者は「実行可能領域へ書き込んだ」
わけではありません。**JIT が攻撃者の制御する値を、正直に機械語へ埋め込んだ**
ことが穴になっています。だから対策も、W^X とは別の角度から要ります。

## 定数ブラインディング：狙った即値をコードに出さない

JIT スプレーの肝は「攻撃者が**狙った即値**を、機械語の中に**そのままの
バイトで**出させる」ことでした。ならば対策は単純です。**狙った即値を、
そのままの形では機械語に出さない**。これが**定数ブラインディング**
（constant blinding、定数の目隠し）です。

やり方はこうです。即値 `k` を機械語へ埋め込む代わりに、

1. 実行時に乱数 `r` を引く。
2. 機械語には `k ^ r`（k と r の排他的論理和）という**別の値**を埋め込む。
3. 実行時に `xor` 命令で `r` を打ち消し、`k` を復元する。

`k ^ r` は `r` がランダムなので、攻撃者から見れば**予測できないバイト列**に
なります。狙った `k` は機械語のどこにも現れません。

### おもちゃで体感する

[最小の JIT](hello-jit.md) の「即値を `eax` に入れて返す」を、ブラインド版に
してみます。素朴版は `mov eax, k; ret` でしたが、ブラインド版は
`mov eax, (k^r); xor eax, r; ret` にします。

```ruby
# 素朴版: 即値 k をそのまま機械語に埋める  (mov eax, k; ret)
def emit_naive(k)
  [0xB8] + [k].pack('L<').bytes + [0xC3]
end

# 定数ブラインディング版:
#   mov eax, (k ^ r)   ; ブラインドした値を置く（k は機械語に現れない）
#   xor eax, r         ; 実行時に r で打ち消す → eax = k
#   ret
def emit_blinded(k)
  r       = rand(0xFFFFFFFF)
  blinded = k ^ r
  code  = [0xB8] + [blinded].pack('L<').bytes   # mov eax, k^r
  code += [0x35] + [r].pack('L<').bytes         # xor eax, r
  code += [0xC3]                                # ret
  [code, r, blinded]
end

K = 0x90909090   # 攻撃者が「埋めたい」狙いの即値（nop スレッドの例）

naive = emit_naive(K)
puts "素朴版:       " + naive.map { |b| '%02X' % b }.join(' ')

code, r, blinded = emit_blinded(K)
puts "ブラインド版: " + code.map { |b| '%02X' % b }.join(' ')
puts "  r=0x%08X, k^r=0x%08X — 狙った 0x%08X は機械語に出ない" % [r, blinded, K]
```

実行するとこうなります（`r` は乱数なので毎回変わります）。

```
素朴版:       B8 90 90 90 90 C3
ブラインド版: B8 48 83 E7 16 35 D8 13 77 86 C3
  r=0x867713D8, k^r=0x16E78348 — 狙った 0x90909090 は機械語に出ない
```

素朴版のバイト列には、攻撃者が狙った `90 90 90 90` が**そっくりそのまま**
並んでいます。これを命令境界からずらして読まれると、`nop` の列として
実行されてしまう。一方ブラインド版には `90 90 90 90` がどこにもなく、
代わりに予測不能な `48 83 E7 16`（＝`k^r`）が並んでいます。攻撃者は乱数
`r` を知らないので、機械語に出るバイト列を予測できません。もちろん、
両方とも関数として呼べば結果は同じ `0x90909090` を返します（ブラインド版は
実行時に `xor` で `r` を打ち消すからです）。

> [!IMPORTANT]
> ブラインディングはタダではありません。即値 1 つにつき `xor`（と乱数の
> 取得）が増え、**コードが膨らみ・実行が少し遅くなる**。だから本物の JIT は、
> **全部の定数をブラインドしません**。攻撃に使える危険な定数 ——
> ジャンプ先になりうる大きな即値や、ユーザー入力由来でビット幅の広い定数
> —— に**絞って**ブラインドし、`0` や `1` のような小さな・無害な定数は
> 素通しします。「どの定数が危ないか」の見極めが効きどころで、ここでも
> **安全性とコストのトレードオフ**が顔を出します。

> [!TIP]
> ブラインディングは「攻撃者に**中身**を予測させない」対策でした。これに
> 「攻撃者に**位置**を予測させない」対策（次節）を重ねると効きます。中身も
> 場所も読めなければ、JIT スプレーで「狙った番地に、狙ったバイト列」を
> 仕込むのは一気に難しくなります。多層防御（defense in depth）の発想です。

## コードのランダム化：場所を読ませない

通常の実行ファイルは、ロード時に配置アドレスをランダムにずらす **ASLR**
（Address Space Layout Randomization）で守られています。攻撃者が「あの
関数はこの番地」と決め打ちできないようにする仕組みです。**JIT 版の ASLR**
は、生成コードの配置にこれを適用します。

- **基底アドレスのランダム化**：JIT コード用のアリーナを、起動ごとに
  ランダムな番地へ確保する。攻撃者は生成コードが**どこにあるか**を
  推測しにくくなる。
- **アリーナ内のランダムなパディング**：個々の生成コードの前後に、長さ
  ランダムな詰め物（`nop` や未使用領域）を入れ、関数の**先頭オフセットを
  ばらす**。同じ入力でも、毎回コードの位置が変わる。
- **命令選択・順序のゆらぎ**：意味が同じなら、わざと別の命令列や別の
  レジスタ割り当てを選ぶ。生成コードの**バイトパターン自体**を実行ごとに
  変え、攻撃者が「このバイト列がガジェットになる」と当てにくくする。

狙いは、JIT スプレーの「広い範囲に毒を撒いて、どこかに当てる」という確率
ゲームを**割に合わなくする**ことです。撒いた毒の位置も中身も毎回変われば、
攻撃者は当て推量するしかなくなります。

> [!NOTE]
> ランダム化は確率的な防御で、「絶対に防げる」とは言いません。試行回数を
> 稼げる攻撃者（同じプロセスへ何度もアクセスできる、メモリ開示のバグと
> 組み合わせられる、など）には突破されうる。だから単独で頼るのではなく、
> W^X・定数ブラインディング・後述のサイト隔離と**重ねて**使います。
> どれか一つで完璧を狙うのでなく、層を重ねて「割に合わなくする」のが
> 現実的な構えです。

## 投機実行サイドチャネル：CPU の投機が漏らすもの

ここまでは「攻撃者に書かせない・読ませない・予測させない」という、
**メモリの中身**をめぐる攻防でした。最後に、まったく毛色の違う ——
そして JIT、とりわけ JavaScript エンジンを深く悩ませた —— 攻撃の系統を
扱います。**投機実行サイドチャネル**、いわゆる Spectre 系の問題です。

### JIT の投機と CPU の投機は別物である

まず混同を避けます。本書はここまで「投機」を**型の決め打ち**の意味で
使ってきました（[脱最適化と OSR](deopt-osr.md)）。「どうせ整数だろう」と
賭けて、外れたらガードで脱最適化する、あれです。一方ここで問題になるのは
**CPU 自身の投機実行**です。

> [!IMPORTANT]
> **JIT の投機（型の決め打ち）と、CPU の投機実行は、別の層の別の話**です。
> JIT の投機は「ソフトウェアが型を賭ける」こと。CPU の投機実行は
> 「ハードウェアが分岐の結果を待たずに先走って命令を実行し、予測が外れたら
> なかったことにする」こと。この章で問題になるのは後者です。ただし、両者は
> **絡みます**。JIT が置いた[ガード](deopt-osr.md)（境界チェックや型
> チェックの分岐）の**後ろ**でも、CPU は予測に従って勝手に先走るからです。

現代の CPU は、分岐命令に出会うと結果を待たずに「たぶんこっち」と予測して
先の命令を**投機的に実行**します。予測が当たれば速い。外れたら、投機実行の
結果は**アーキテクチャ状態（レジスタやメモリ）には反映せず**捨てます。
ここまでなら正しさは保たれています。

問題は、捨てたはずの投機実行が、**キャッシュには痕跡を残す**ことです。
投機的に読んだメモリはキャッシュに乗り、それは「巻き戻し」では消えません。
攻撃者は、あるメモリ番地へのアクセスが**速いか遅いか**（キャッシュに乗って
いるか）を計ることで、投機実行が「何を読んだか」を**間接的に**読み取れます。
これが**サイドチャネル**（横道からの情報漏れ）です。

### なぜ JIT・JS エンジンが標的になったのか

配列アクセスの境界チェックを考えます。JIT は、こんなコードを出します。

```
   ; rdi = 添字 i、rsi = 配列の長さ len
   cmp  rdi, rsi
   jae  out_of_bounds      ; i >= len なら範囲外処理へ（ガード）
   ... arr[i] を読む（範囲内だと信じて） ...
```

論理的には完璧です。`i >= len` なら範囲外処理へ飛ぶので、`arr[i]` の読み
込みは「`i` が範囲内のとき」しか起きない。**はずです**。

ところが CPU は、`jae` の結果を待たずに先走ります。`i` が範囲内だろうと
**予測**して、`arr[i]` の読み込みを**投機的に実行**してしまう。たとえ実際の
`i` が範囲外でも、です。投機実行で読んだ範囲外のメモリ（本来は読めないはずの
秘密データ）は、すぐ「巻き戻し」で捨てられますが、**その値に基づくキャッシュ
の痕跡が残る**。攻撃者は `i` をわざと範囲外にした上で、キャッシュの痕跡から
秘密データを 1 ビットずつ盗み出せます。

JavaScript エンジンが格好の標的になったのは、攻撃者が**ブラウザ上で任意の
JavaScript を走らせられる**からです。JIT は配列アクセスを上のような「境界
チェック＋読み込み」に素直にコンパイルし、攻撃者はそれを投機実行させる
JavaScript を書ける。「サンドボックスの中で安全に動かしているつもりの
JavaScript」から、**同じプロセス内の他のデータ**（別タブの情報など）が漏れ
うる、という深刻な事態が明らかになりました。

> [!NOTE]
> ここで効いているのは、本書が何度も言ってきた「ガードは数命令で、当たって
> いるかぎりほぼ無視できる」という JIT の強みの、皮肉な裏面です
> （[なぜ JIT か](why-jit.md)・[脱最適化と OSR](deopt-osr.md)）。ガードが
> 軽い分岐だからこそ、CPU はそれを軽々と投機的に踏み越える。**速くする
> ための分岐が、漏らすための分岐になった**わけです。

### 対策その一：投機バリアで先走りを止める

いちばん直接的な対策は、「ここから先は投機で先走るな」と CPU に命じる
**投機バリア**を、危ないアクセスの前に置くことです。x86 では `lfence`、
ARM では `csdb`（Consumption of Speculative Data Barrier）といった命令が
これにあたります。

```
   cmp  rdi, rsi
   jae  out_of_bounds
   lfence                  ; ここで投機実行を堰き止める（先走り禁止）
   ... arr[i] を読む ...
```

`lfence` を挟むと、CPU は分岐の結果が確定するまで先の読み込みを始めません。
範囲外なら、その読み込みは**そもそも投機的にも実行されない**ので、痕跡も
残りません。

> [!WARNING]
> 投機バリアは効きますが、**遅い**。`lfence` は CPU の命令パイプラインを
> 実質的に堰き止めるので、本書がこれまで積み上げてきた「投機で先走って
> 速くする」という現代 CPU の旨味を、その場所だけ捨てることになります。
> 配列アクセスのたびに `lfence` を入れたら、JIT で速くした意味が大きく
> 削がれます。だから無差別には使えません。

### 対策その二：インデックスマスキング

もっと安い対策が、配列の添字を**必ず範囲内に丸め込む**やり方です。分岐で
「範囲外なら飛ぶ」のではなく、添字そのものをビット演算で範囲内に畳む。
投機実行で分岐を踏み越えられても、**読みに行く番地が範囲内に丸まっている**
ので、範囲外メモリには触れません。

配列の長さが 2 のべき乗 `len` なら、`mask = len - 1` との AND 一発で
丸められます。Ruby のおもちゃで考え方を見ます（実際の JIT は機械語で
やりますが、発想は同じです）。

```ruby
arr = (0..7).to_a   # 長さ 8（2 のべき乗）

# マスク版: 添字を必ず配列内に丸める。
# 投機実行で境界分岐を越えても、添字自体が範囲内に畳まれているので漏れない。
def load_masked(arr, i)
  return nil if i >= arr.size   # 通常の境界チェック（論理的な正しさ用）
  mask = arr.size - 1           # size=8 → mask=0b111
  safe = i & mask               # 範囲外の i も 0..7 に丸め込まれる
  arr[safe]
end

p load_masked(arr, 3)      # => 3
p (10 & (arr.size - 1))    # => 2   （範囲外 i=10=0b1010 が 0b010=2 に丸まる）
```

`i = 10`（範囲外）でも、`10 & 7 = 2` となって**配列内の番地**に丸め込まれ
ます。仮に CPU が境界分岐を投機的に踏み越えても、実際に読みに行く添字は
すでに `0..7` に畳まれているので、範囲外の秘密データには手が届きません。
マスクは `and` 一命令で済み、`lfence` のようにパイプラインを止めないので、
投機バリアよりずっと安い。これが多くの JIT が配列境界に採った主力対策です。

> [!TIP]
> 長さが 2 のべき乗でないときは、別の丸め方が要ります。たとえば
> 「`i >= len` なら `i` を `0` に潰すマスクを、分岐ではなく**条件付き
> 演算**（`cmov` など、分岐を伴わない命令）で作る」やり方です。肝は
> **分岐を使わずに添字を範囲内へ畳む**こと。分岐があると、そこを CPU が
> 投機的に踏み越えてしまうからです。「分岐をデータ依存の演算に置き換える」
> のが、この種の対策に共通する型です。

### 対策その三：サイト隔離（プロセスを分ける）

ソフトウェア側でいくら投機を抑えても、CPU のミクロアーキテクチャに根ざす
漏れを完全に塞ぐのは困難です。そこでブラウザが採った構えが**サイト隔離**
（site isolation）です。発想を変えて、「**漏れても困らないように、秘密を
別プロセスへ追い出す**」。

投機実行サイドチャネルが読めるのは、原則として**同じプロセスのアドレス
空間**にあるデータでした。ならば、異なるサイト（オリジン）の内容を**別々の
OS プロセス**で動かせば、攻撃者の JavaScript が走るプロセスには、盗みたい
他サイトのデータが**そもそも存在しない**。投機実行で先走っても、読むべき
秘密がアドレス空間にいなければ漏れようがありません。

> [!IMPORTANT]
> サイト隔離は、JIT 単体の対策ではなく**システム全体の構え**です。
> ここに、セキュリティの大事な一般則があらわれています。**ある層
> （JIT のコード生成）で塞ぎきれない脅威は、別の層（プロセス分離）で
> 受け止める**。投機バリア・マスキング（コード生成の層）と、サイト隔離
> （プロセスの層）を**重ねる**ことで、はじめて実用的な防御になります。
> 単一の銀の弾丸はなく、多層防御で「割に合わなくする」のがここでも基本
> です。

## メモリ安全とサンドボックス：そもそも危険なコードを生ませない

ここまでの対策の多くは、「**危ういコードを生成してしまったあとで**、
被害を抑える」性格のものでした。視点を変えて、「**そもそも危ういコードや
未定義動作を生まない**ように、入口で縛る」という方向もあります。その代表が
[WebAssembly](research-wasm.md) です。

[WebAssembly の JIT](research-wasm.md) で見たとおり、Wasm のバイナリは
実行前に**検証**できます[](#cite:haas2017)。型が整合しているか、スタックが
破綻しないか、ジャンプ先が妥当か —— を読み込み時に高速に確かめられる。
さらに Wasm のメモリアクセスは、専用の**線形メモリ**という箱の中に
閉じ込められ、その外には**手が届かない**ように設計されています。

これがセキュリティに効きます。Wasm をコンパイルする JIT は、

- **型を投機しなくてよい**（[脱最適化と OSR](deopt-osr.md) の重い機械が
  要らない）ぶん、コード生成が単純で、危ういパターンを生む余地が減る。
- メモリアクセスが線形メモリの内側に**構造的に閉じ込められている**ので、
  「うっかり範囲外を読む（書く）コードを生成してしまう」事故が起きにくい。
- 未定義動作が（言語仕様の上で）ほぼ無いので、「処理系ごとに振る舞いが
  違って、その差を突かれる」という攻撃面が小さい。

つまり Wasm は、「**検証済みで、未定義動作がなく、メモリが箱に閉じている**」
という性質によって、JIT が吐くコードの安全性を**底上げ**しています
[](#cite:haas2017)。「JIT の都合に合わせてバイトコードを設計すると、性能
だけでなく安全も得られる」例だと言えます。

> [!NOTE]
> もう一つ、ハードウェアの助けを借りる方向もあります。Apple Silicon
> などが備える**ポインタ認証**（PAC, Pointer Authentication）は、関数の
> 戻りアドレスなどのポインタに、秘密鍵から作った**署名**を埋め込み、使う
> 前に検証します。攻撃者がポインタを書き換えても、正しい署名を作れないので
> 検知できる。これは ROP のような「ポインタを差し替えて制御を奪う」攻撃を
> 難しくします。JIT は生成コードでこの署名・検証命令を吐くことで、自分の
> 作ったコードの制御フローを守れます。ソフト（W^X・ブラインディング）と
> ハード（PAC・投機バリア命令）の両輪で守る、という構図です。

## まとめ

- JIT の正体は「**データを実行可能コードに変える**」装置で、それゆえ
  攻撃者にとって魅力的な標的になる。本章の「安全」は、[脱最適化](deopt-osr.md)
  の「投機が外れても正しい答えに戻る」安全とは**別種**の、
  **悪意ある第三者への防御**である。
- **W^X**：書き込みと実行を同時に許さない。**RWX を一瞬も作らない**のが
  鉄則。実装は、(1) RW で書いて `mprotect` で RX へ落とす**二段構え**、
  (2) 同じ物理ページを RW 窓と RX 窓で見る**二重マッピング**、(3) Apple
  Silicon の `pthread_jit_write_protect_np` のような**スレッド単位の
  切り替え**。いずれも切り替えの**バッチ化**で速さと両立させる。
- **JIT スプレー**：攻撃者が JIT に「制御できる定数」を大量に生成させ、
  **命令境界をずらして**埋め込んだバイト列を実行させる古典的攻撃。W^X を
  守っていても成立しうる。
- **定数ブラインディング**：即値 `k` を `k ^ r` の形で機械語に出し、実行時に
  `xor` で `r` を打ち消して復元する。狙った即値が機械語に**出ない**。命令が
  増えるので、**危険な定数だけに絞って**かける。
- **コードのランダム化**（JIT 版 ASLR）：配置・パディング・命令選択をゆらし、
  攻撃者に**位置を読ませない**。確率的防御なので単独では頼らない。
- **投機実行サイドチャネル（Spectre 系）**：**JIT の投機（型の決め打ち）と
  CPU の投機実行は別物**だが絡む。[ガード](deopt-osr.md)の軽い分岐を CPU が
  投機的に踏み越え、範囲外読み込みの痕跡がキャッシュ経由で漏れる。対策は
  **投機バリア**（`lfence`/`csdb`、ただし遅い）、**インデックスマスキング**
  （`index & mask`、安い主力）、**サイト隔離**（プロセスを分ける）。
- **メモリ安全とサンドボックス**：[WebAssembly](research-wasm.md) は検証済み・
  未定義動作なし・線形メモリで安全を**底上げ**する[](#cite:haas2017)。
  ポインタ認証（PAC）などハードウェアの助けも借りる。
- 単一の銀の弾丸はない。**多層防御**で「攻撃を割に合わなくする」のが基本。

セキュリティ対策は、W^X の切り替えにせよ定数ブラインディングにせよ、
「**生成のたびに正しく・もれなく行う**」必要があり、手で一つずつ面倒を
見るのは現実的ではありません。これらの安全策も含め、コードメモリの管理・
アセンブラ・ラベル解決・そして安全なコード生成の作法までを**肩代わり**して
くれる道具立て —— 次章の [JIT フレームワーク](frameworks.md) へ進みます。
