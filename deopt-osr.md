# デオプティマイズと OSR：賭けに負けても安全に戻る

本書はここまで、何度も「投機」という言葉を使ってきました。[なぜ JIT か](why-jit.md)で
導入し、[トレーシング](tracing.md)のガード、[階層的 JIT](tiered.md)の
降格として現れてきた、JIT の中心戦略です。この章では、その投機の
**安全網** —— 賭けに負けたときに、何事もなかったかのように元の正しい
実行へ戻る仕組み —— を正面から扱います。

二つの技術が主役です。**デオプティマイズ**（deoptimization、最適化解除）と、
その親戚の **OSR**（On-Stack Replacement、実行中スタックの差し替え）。
どちらも「すでに走り出した実行の状態を、別の実行形式の状態へ作り替える」
という、JIT ならではの離れ業です。

## なぜ戻れなければならないのか

JIT が攻めの最適化をできるのは、**外れたら戻れる**という保証があるから
です。これは強調してもしすぎることはありません。

たとえば「この変数 `x` はいつも整数だ」と観測したとき、JIT は二つの
態度を取れます。

- **保守的**：念のため、整数・浮動小数点数・文字列…すべての場合に
  対応したコードを出す。安全だが、ちっとも速くならない。
- **投機的**：整数だと**決め打ち**し、生の整数演算で一直線のコードを出す。
  ただし入口に「本当に整数か」のガードを置き、**外れたらここから抜ける**。

投機が成り立つのは、「外れたときに、正しい実行を続けられる出口がある」
からこそです。その出口の実装が、デオプティマイズです。デオプティマイズが
なければ、JIT は保守的な最適化しかできず、動的言語はちっとも速くなりません。

> [!IMPORTANT]
> **デオプティマイズは、最適化を「あきらめる」仕組みではなく、最適化を
> 「攻めさせる」ための仕組みです。** 後戻りできる安全網があるからこそ、
> JIT は大胆な決め打ちができる。守りの技術に見えて、その実、攻めの最適化の
> 前提条件なのです。

## 何が難しいのか：状態の作り替え

戻ること自体は、口で言うほど簡単ではありません。問題は、**最適化された
コードと、戻り先（インタプリタ）とで、実行状態の持ち方がまったく違う**
ことです。

最適化コードでは、[なぜ JIT か](why-jit.md)で見たとおり、値は箱から
出され、CPU レジスタ上に生の整数として乗っています。ローカル変数の一部は、
[中間表現](ir.md)の最適化で消えてしまっている（定数畳み込みで定数に
なった、他の値から計算できるので保持をやめた）かもしれません。

一方インタプリタは、すべてのローカル変数をきちんと箱に入れて、スタック
フレームの決まった場所に持っています。デオプティマイズの瞬間、

> **「レジスタにばらけた生の値」を、「インタプリタが期待する、箱詰めされた
> 整然としたフレーム」へ組み立て直す**

必要があります。これがデオプティマイズの技術的な核心です。消えた変数は
他の値から復元し、生の整数は箱に詰め直し、インタプリタの `pc` を
「ガードに対応するバイトコードの位置」に合わせる。

このために JIT は、最適化コードのガードごとに**デオプティマイズ情報**
（deoptimization metadata、スコープ記述子とも）を一緒に生成しておきます。
「このガードで戻るときは、インタプリタのローカル変数 `x` には今レジスタ
`r3` の値を箱詰めして入れ、`pc` はバイトコード 42 にせよ」という**復元の
レシピ**です。

> [!NOTE]
> この技術は、SELF という動的オブジェクト指向言語の処理系で、
> 1992 年に確立されました[](#cite:holzle1992)。もとの動機は意外にも
> **デバッグ**でした。「最適化されたコードを、あたかも最適化していないか
> のようにデバッガに見せたい」。そのために「最適化コードの状態を、
> 最適化前の状態へ動的に復元する」技術が編み出され、それがそのまま
> 投機的 JIT の安全網になったのです。歴史の妙です。

## デオプティマイズを動かしてみる

概念を Ruby のおもちゃで体感しましょう。配列の各要素の二乗の和を求める
ループを考えます。汎用版（どんな数でも動く）と、整数決め打ちの「最適化版」
（生の整数で計算する想定）を用意し、最適化版が整数でない要素に出会ったら
**状態を抱えてデオプティマイズし、汎用版が続きを引き継ぐ**様子を作ります。

```ruby
# 汎用版インタプリタ（常に正しい。Float でも Integer でも動く）
def interp_sum_squares(arr)
  acc = 0; i = 0
  while i < arr.size
    acc = acc + arr[i] * arr[i]
    i += 1
  end
  acc
end

# 「最適化版」：全要素が Integer だと投機する。賭けが外れたら、
# 生きている状態（acc, i）を抱えてデオプティマイズする。
DEOPT = Object.new
def opt_sum_squares(arr, acc = 0, i = 0)
  while i < arr.size
    x = arr[i]
    # ガード：投機の前提は「x は Integer」
    return [DEOPT, acc, i] unless x.is_a?(Integer)   # 外れた → 状態を返して脱出
    acc = acc + x * x        # 本物の JIT はここを生の整数演算にする
    i += 1
  end
  acc
end

# 駆動部：最適化版を走らせ、デオプトしたら汎用版で「続きから」再開する
def run(arr)
  r = opt_sum_squares(arr)
  return r unless r.is_a?(Array) && r[0].equal?(DEOPT)

  _, acc, i = r                          # ★ 保存しておいた状態を取り出す
  puts "  deopt at i=#{i} (#{arr[i].inspect} は整数でない)。インタプリタで再開"
  while i < arr.size                     # ★ その状態から汎用版を続ける
    acc = acc + arr[i] * arr[i]
    i += 1
  end
  acc
end

p run([1, 2, 3, 4])        # 全部整数 → 最適化経路のまま => 30
p run([1, 2, 3.5, 4])      # 3.5 で i=2 デオプト → 続きを汎用版で => 33.25
```

実行するとこうなります。

```
30
  deopt at i=2 (3.5 は整数でない)。インタプリタで再開
33.25
```

ここで起きたことが、デオプティマイズの本質です。`opt_sum_squares` は
`i=2` で `3.5` に出会い、ガードが破れました。このとき、**そこまでに
計算した `acc`（=1+4=5）と、どこまで進んだか `i`（=2）を抱えて脱出**し、
汎用版がその状態から計算を続けて `33.25` を得ました。「途中まで最適化版で
進み、続きをインタプリタが引き継ぐ」—— `[DEOPT, acc, i]` という戻り値が、
本物の JIT における**デオプティマイズ情報＝状態の引き継ぎ**のミニチュアに
なっています。

> [!TIP]
> このおもちゃで `acc` と `i` を返しているのが、本物の「復元のレシピ」に
> あたります。本物では、レジスタにばらけた生の値をどう箱詰めし、
> インタプリタフレームのどこへ置き、`pc` をどこに合わせるか —— という、
> ずっと込み入った情報になります。けれど発想は同じで、**ガードの地点で、
> インタプリタが再開できる状態を作って渡す**だけです。

## ガードはどこに、どれだけ置くか

デオプティマイズはタダではありません。ガードのチェックそのものにコストが
あり、何より**デオプティマイズ情報を保持するメモリ**が要ります。ガードが
多いほど、最適化器が動かせる自由（[中間表現](ir.md)で見たガードの巻き上げや
統合）も制約されます。だから JIT は、ガードを**減らす**努力をします。

- **重複ガードの除去**：上流で「`x` は整数」と確かめたら、下流の同じ
  ガードは消す。[基本ブロックバージョニング](compilation-units.md)は、
  これをブロックの版分けで自然に達成する手法でした。
- **ガードの巻き上げ**：ループ内で毎回チェックされる不変なガードを、
  ループの外へ 1 回だけ出す。
- **デオプト頻度の学習**：あるガードが何度も破れるなら、そこは投機が
  外れがちと判断し、次のコンパイルでは投機をやめる。

## OSR：実行中のループを乗り換える

デオプティマイズの双子の兄弟が **OSR**（On-Stack Replacement）です。
直訳すれば「スタック上での置き換え」。**すでに実行中の関数（特に長い
ループ）の途中で、実行形式を別のものに切り替える**技術です。

なぜ要るのか。[階層的 JIT](tiered.md)で見たように、ふつう新しい
コンパイル済みコードは「次にその関数を呼んだとき」から使われます。
ところが、

```ruby
# 一度しか呼ばれないが、中で 1 億回回るループ
def main
  sum = 0
  100_000_000.times { |i| sum += i * i }   # ← ここに入ったまま出てこない
  sum
end
```

このループは `main` を一度呼んだだけで延々と回り続けます。「次に呼んだ
とき」を待っていては、いつまでも最適化版に乗り換えられません。**今まさに
回っているこのループを、途中から最適化版へ移したい。** これが OSR の
出番です。

OSR には二つの向きがあります。

- **昇格 OSR（OSR-in）**：実行中のインタプリタ／ベースラインのループを、
  途中から最適化版へ乗り換える。上の `main` のループを最適化版に移す
  ケース。
- **降格 OSR（OSR-out）**：これは実はデオプティマイズそのものです。
  実行中の最適化版から、途中でインタプリタへ落とす。

どちらも、やることは同じ —— **実行中のスタックフレームを、移り先の形式が
期待する状態へ作り替える**ことです。昇格 OSR なら「インタプリタのフレーム
（箱詰めの変数）」を読んで「最適化版のフレーム（レジスタ上の生の値）」を
組み立て、ループの対応する地点へジャンプする。デオプティマイズで使った
「状態の作り替え」の機械を、逆向きに使うわけです。

> [!NOTE]
> OSR の実装は、対象言語の構造に左右されます。ループが構文的にきれいな
> 言語では入口を見つけやすく、`goto` だらけで構造が崩れた言語では、
> OSR のための「乗り換え地点」を作るだけでひと苦労です。研究では、
> 構造化されていない言語での OSR をどう実現するか、といったテーマも
> 扱われています。

## デオプティマイズと正しさ

ここで一つ、深い問いが立ち上がります。**「途中まで最適化版で進み、
続きをインタプリタが引き継ぐ」とき、本当に元のプログラムと同じ答えに
なると、どうして保証できるのか。**

おもちゃの例では `acc` と `i` を渡すだけでしたが、本物では最適化器が
コードを大きく組み替えています。命令を並べ替え、変数を消し、計算を
前倒しし…。それでも「どのガードの地点で抜けても、インタプリタが正しく
再開できる」状態を**常に**復元できなければなりません。最適化のどんな
変形が、この性質を壊さずに許されるのか。

これは「気をつければ大丈夫」で済ませてよい問題ではなく、**形式的に
証明すべき**問題です。実際、投機的最適化とデオプティマイズの正しさを
数学的に定式化し、証明する研究があります[](#cite:fluckiger2018)。
この重要なテーマは、第 II 部の
[投機的最適化の正しさ](research-deopt-correctness.md)で改めて扱います。

## まとめ

- **デオプティマイズ**は、投機が外れたとき、最適化コードの実行状態を
  インタプリタが再開できる状態へ作り替えて戻る仕組み。投機を**攻めさせる**
  ための前提。
- 難しさは**状態の作り替え**にある。レジスタにばらけた生の値や、消えた
  変数を、箱詰めの整然としたインタプリタフレームへ復元する。そのための
  **復元のレシピ**（デオプティマイズ情報）をガードごとに用意する。
- 起源は 1992 年の SELF で、動機は**デバッグ**だった。
- ガードはコストなので、**重複除去・巻き上げ・頻度の学習**で減らす。
- **OSR** は実行中のループを途中で別形式へ乗り換える技術。昇格 OSR
  （インタプリタ→最適化）と降格 OSR（＝デオプティマイズ）がある。
- 「途中から正しく再開できる」ことの保証は、形式的に証明される研究テーマ。

これで第 I 部の技法はほぼ出そろいました。最後に、これらを一から実装せずに
済ませる道具立て —— JIT フレームワークを概観します。
