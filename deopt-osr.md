# 脱最適化と OSR

本書はここまで、何度も「投機」という言葉を使ってきました。[なぜ JIT か](why-jit.md)で
導入し、[トレーシング](tracing.md)のガード、[階層的 JIT](tiered.md)の
降格として現れてきた、JIT の中心戦略です。この章では、その投機の
**安全網**、つまり賭けに負けたときに、何事もなかったかのように元の正しい
実行へ戻る仕組みを扱います。

二つの技術が主役です。**脱最適化**（deoptimization）と、
その双子の兄弟である **OSR**（On-Stack Replacement、実行中スタックの差し替え）。
どちらも「すでに走り出した実行の状態を、別の実行形式の状態へ作り替える」
という、JIT ならではの離れ業です。

> [!NOTE]
> 「脱最適化」は英語 **deoptimization** の訳語です。「最適化された状態から
> 降りて、下位の（投機していない）実行形式へ戻る」という意味で、文献に
> よっては「最適化解除」とも呼ばれます。[HotSpot](case-jvm.md) に登場する
> **uncommon trap**（後述）は脱最適化の同義語ではなく、「めったに通らない
> はず」の経路に仕掛けておく、脱最適化の**引き金**を指す言葉です。本書では
> この訳語「脱最適化」に統一します。

## なぜ戻れなければならないのか

JIT が攻めの最適化をできるのは、**外れたら戻れる**という保証があるから
です。これは強調してもしすぎることはありません。

たとえば「この変数 `x` はいつも整数だ」と観測したとき、JIT は二つの
態度を取れます。

- **保守的**：念のため、整数、浮動小数点数、文字列…すべての場合に
  対応したコードを出す。安全だが、ちっとも速くならない。
- **投機的**：整数だと決め打ちし、生の整数演算で一直線のコードを出す。
  ただし入口に「本当に整数か」のガードを置き、外れたらここから抜ける。

投機が成り立つのは、「外れたときに、正しい実行を続けられる出口がある」
からこそです。その出口の実装が、脱最適化です。脱最適化が
なければ、JIT は保守的な最適化しかできず、動的言語はちっとも速くなりません。

> [!IMPORTANT]
> **脱最適化は、最適化を「あきらめる」仕組みではなく、最適化を
> 「攻めさせる」ための仕組みです。** 後戻りできる安全網があるからこそ、
> JIT は大胆な決め打ちができる。守りの技術に見えて、その実、攻めの最適化の
> 前提条件なのです。

## 状態の作り替えがなぜ難しいか

戻ること自体は、口で言うほど簡単ではありません。問題は、**最適化された
コードと、戻り先（インタプリタ）とで、実行状態の持ち方がまったく違う**
ことです。

最適化コードでは、[なぜ JIT か](why-jit.md)で見たとおり、値は箱から
出され、CPU レジスタ上に生の整数として乗っています。ローカル変数の一部は、
[中間表現](ir.md)の最適化で消えてしまっている（定数畳み込みで定数に
なった、他の値から計算できるので保持をやめた）かもしれません。

一方インタプリタは、すべてのローカル変数をきちんと箱に入れて、スタック
フレームの決まった場所に持っています。脱最適化の瞬間、

> 「レジスタにばらけた生の値」を、「インタプリタが期待する、箱詰めされた
> 整然としたフレーム」へ組み立て直す

必要があります。これが脱最適化の技術的な核心です。消えた変数は
他の値から復元し、生の整数は箱に詰め直し、インタプリタの `pc` を
「ガードに対応するバイトコードの位置」に合わせる。

このために JIT は、最適化コードのガードごとに**脱最適化情報**
（deoptimization metadata、スコープ記述子とも）を一緒に生成しておきます。
「このガードで戻るときは、インタプリタのローカル変数 `x` には今レジスタ
`r3` の値を箱詰めして入れ、`pc` はバイトコード 42 にせよ」という**復元の
レシピ**です。

難しさをもう一段引き上げるのが、[インライン化](why-jit.md)です。最適化
コードでは呼び出し先のメソッドが何段もインライン展開され、機械語の上では
フレーム 1 個に溶け合っています。ところが戻り先のインタプリタは「呼び出しの
たびにフレームを積む」世界です。だから展開の中ほどにあるガードで脱最適化
すると、機械語の 1 フレームを、インタプリタフレーム N 個の積み重なりへ
展開し直す必要があります。呼び出し元のフレーム、その中で呼ばれたメソッドの
フレーム、さらにその中…と、それぞれのローカル変数、スタック、`pc` を
別々に復元するのです。スコープ記述子が入れ子の構造を持つのは、まさに
このためです。インライン化のすえに「確保ごと消された」オブジェクトを
脱最適化の瞬間に作り直す話も含め、HotSpot での実際は
[JVM のケーススタディ](case-jvm.md#Java 特有の最適化であるエスケープ解析)で見ます。

> [!NOTE]
> この技術は、Self という動的オブジェクト指向言語の処理系で、
> 1992 年に確立されました[](#cite:holzle1992)。もとの動機は意外にも
> **デバッグ**でした。「最適化されたコードを、あたかも最適化していないか
> のようにデバッガに見せたい」。そのために「最適化コードの状態を、
> 最適化前の状態へ動的に復元する」技術が編み出され、それがそのまま
> 投機的 JIT の安全網になったのです。歴史の妙です。

## 脱最適化を動かしてみる

概念を Ruby のおもちゃで体感しましょう。配列の各要素の二乗の和を求める
ループを考えます。汎用版（どんな数でも動く）と、整数決め打ちの「最適化版」
（生の整数で計算する想定）を用意し、最適化版が整数でない要素に出会ったら
**状態を抱えて脱最適化し、汎用版が続きを引き継ぐ**様子を作ります。

```ruby
# 汎用版インタプリタ（常に正しい。Float でも Integer でも動く）
def interp_sum_squares(arr)
  acc = 0; i = 0
  while i < arr.size
    acc = acc + arr[i] * arr[i]
    i += 1
  end
  acc
end

# 「最適化版」：全要素が Integer だと投機する。賭けが外れたら、
# 生きている状態（acc, i）を抱えて脱最適化する。
DEOPT = Object.new
def opt_sum_squares(arr, acc = 0, i = 0)
  while i < arr.size
    x = arr[i]
    # ガード：投機の前提は「x は Integer」
    return [DEOPT, acc, i] unless x.is_a?(Integer)   # 外れた → 状態を返して脱出
    acc = acc + x * x        # 本物の JIT はここを生の整数演算にする
    i += 1
  end
  acc
end

# 駆動部：最適化版を走らせ、脱最適化したら汎用版で「続きから」再開する
def run(arr)
  r = opt_sum_squares(arr)
  return r unless r.is_a?(Array) && r[0].equal?(DEOPT)

  _, acc, i = r                          # ★ 保存しておいた状態を取り出す
  puts "  deopt at i=#{i} (#{arr[i].inspect} は整数でない)。インタプリタで再開"
  while i < arr.size                     # ★ その状態から汎用版を続ける
    acc = acc + arr[i] * arr[i]
    i += 1
  end
  acc
end

p run([1, 2, 3, 4])        # 全部整数 → 最適化経路のまま => 30
p run([1, 2, 3.5, 4])      # 3.5 で i=2 脱最適化 → 続きを汎用版で => 33.25
```

実行するとこうなります。

```
30
  deopt at i=2 (3.5 は整数でない)。インタプリタで再開
33.25
```

ここで起きたことが、脱最適化の本質です。`opt_sum_squares` は
`i=2` で `3.5` に出会い、ガードが破れました。このとき、そこまでに
計算した `acc`（=1+4=5）と、どこまで進んだか `i`（=2）を抱えて脱出し、
汎用版がその状態から計算を続けて `33.25` を得ました。「途中まで最適化版で
進み、続きをインタプリタが引き継ぐ」。この `[DEOPT, acc, i]` という戻り値が、
本物の JIT における**脱最適化情報＝状態の引き継ぎ**のミニチュアに
なっています。

> [!TIP]
> このおもちゃで `acc` と `i` を返しているのが、本物の「復元のレシピ」に
> あたります。本物では、レジスタにばらけた生の値をどう箱詰めし、
> インタプリタフレームのどこへ置き、`pc` をどこに合わせるか、という
> ずっと込み入った情報になります。けれど発想は同じで、**ガードの地点で、
> インタプリタが再開できる状態を作って渡す**だけです。

## 復元情報をどう持つか

おもちゃでは生きている状態が `acc` と `i` の二つだけで、しかも素直に変数として
存在していたので、「そのまま返す」だけで復元できました。ところが本物の最適化
コードは、そう甘くありません。[中間表現](ir.md)の最適化は、復元したいはずの値を
平気で消したり、動かしたり、別の形に変えたりします。

- ループカウンタ `i` は、レジスタに乗っているかもしれないし、別の値から計算
  できると分かって保持をやめられているかもしれない。
- 中間変数 `y = x * 2` は、一度も箱に入れられず、`x` から作れるからと
  捨てられているかもしれない。
- 短命なオブジェクトは[エスケープ解析](glossary.md)とスカラ置換（オブジェクトを
  確保せず、フィールドをばらばらの値として持つ最適化）でそもそも
  作られず、フィールドの値がレジスタにばらけているだけかもしれない。

それでも脱最適化の瞬間には、インタプリタが期待する「箱詰めされた整然とした
フレーム」を、ガードのあるどの地点からでも組み立て直せなければなりません。
ここに、JIT 設計者がいつも突き当たる二択があります。

1. **コンパイル結果に制約を課す**：最適化器の自由をあえて縛り、復元を簡単に
   保つ。たとえば「脱最適化に関わる値は、ガードを越えて消したり大きく動かしたり
   してはいけない」「生きている変数は決まった場所に置く」といった規則を最適化器に
   守らせる。復元情報は小さく単純になりますが、その分、最適化の伸びしろを削ります。
2. **巨大な復元マップを持つ**：最適化器には自由に攻めさせ、その代償として、
   ガードごとに「いまどの値がどのレジスタ／スタック位置／定数にあり、消えた
   変数はどう計算し直すか」を事細かに記述した脱最適化マップを吐く。最速の
   コードが得られますが、このマップは生成コードと同じか、それ以上の大きさに
   膨れることもあり、メモリと生成時間を食います。

現実の JIT は、ほぼ②に倒したうえで、マップを切り詰めて耐えます。本当に必要な
セーフポイント（次節）でだけ情報を記録する、近いガード間で差分だけを持つ、複数の
ガードで記述を共有する、そして極めつけが**再具体化**（rematerialization）です。

再具体化とは、「消えてしまった値を、保存しておくのではなく、戻るときにレシピから
作り直す」発想です。これは②のマップを①の制約へ歩み寄らせる折衷でもあります。
スカラ置換で消したオブジェクトなら、「脱最適化したら、いまレジスタにある
フィールド値からオブジェクトを作り直して詰めよ」というレシピを持つ。定数に
なった変数なら「その定数を入れよ」と書くだけでよい。値そのものを抱える代わりに、
作り方を抱えるのです。

```ruby
# 最適化版では中間変数 y(=x*2) を保持していない（x から作れるので消した）。
# 脱最適化時には「レシピ」で y を再具体化してインタプリタ状態を組み立てる。

# 各ガード地点の復元レシピ：生きている変数 → どう作るか
#   [:live] なら今ある値、[:recompute, ...] なら作り方
RECIPE = {
  x: [:live],                       # x はレジスタに生きている
  y: [:recompute, :mul, :x, 2],     # y は保持していない。x*2 で作り直す
  i: [:live],
  obj: [:realloc, :Point, :x, :y],  # obj はスカラ置換で消えている。
                                    # x, y のフィールド値からオブジェクトを作り直す
}

# スカラ置換で消えていたオブジェクトを表す、ごく単純な構造体
Point = Struct.new(:x, :y)

# 脱最適化：今生きている値の集合 regs とレシピから、インタプリタフレームを組む
def rebuild_frame(regs, recipe, frame = {})
  recipe.each do |var, how|
    frame[var] =
      case how[0]
      when :live      then regs.fetch(var)        # そのまま使う
      when :recompute                             # レシピで作り直す
        _, op, src, k = how
        base = regs.fetch(src)
        op == :mul ? base * k : base + k
      when :realloc                               # 消したオブジェクトを再アロケート
        _, klass, *fields = how
        # フィールド値は、すでに組んだ frame から拾う（x, y は復元済み）
        Object.const_get(klass).new(*fields.map { |f| frame.fetch(f) })
      end
  end
  frame
end

# 最適化版が x=21, i=7 のところでガードを破ったとする
# （y も obj も消えている）
live = { x: 21, i: 7 }
p rebuild_frame(live, RECIPE)
# => {x: 21, y: 42, i: 7, obj: #<struct Point x=21, y=42>}
#    y は x*2=42 として、obj は Point(x, y) として復元された
```

`y` はどこにも保存されていませんでしたが、レシピ `[:recompute, :mul, :x, 2]` の
おかげで、脱最適化の瞬間に `x * 2 = 42` として作り直せました。**保存量を増やさずに、
消した変数を取り戻せた**わけです。本物の JIT のスカラ置換（部分エスケープ解析で
オブジェクトの確保を消す最適化）[](#cite:stadler2014)もこれと同じで、消した
オブジェクトを「生きているフィールド値から再アロケートして詰め直す」レシピとして
持ちます。この「最適化が値を消すなら、消す代わりに復元レシピを残す」という規律が、
[投機的最適化の正しさ](research-deopt-correctness.md)の中心テーマにもなります。

> [!IMPORTANT]
> 脱最適化情報の設計は、つねに **「最適化の自由」対「復元情報の大きさ」** の
> 綱引きです。①最適化を縛って情報を小さく保つか、②自由に攻めて巨大なマップで
> 支払うか。多くの実装は②を選びつつ、セーフポイントの限定、差分圧縮、**再具体化**
> （値でなく作り方を持つ）でマップを切り詰めます。「脱最適化はタダではない。
> その値段は、たいていメモリで払われる」と覚えておくと、各処理系の設計判断が
> 読み解けます。

## ガードを機械語でどう置くか

ガードは、最適化コードの中ではごく数命令にすぎません。[最小の JIT](hello-jit.md)で
見た機械語の言葉でいえば、たとえば「`x` は整数か」のガードは、

```
   ; rax に x が入っている。最下位ビットが 1 なら整数（タグ付き整数）と約束
   test al, 1            ; 最下位ビットを見る
   jz   deopt_stub_7     ; 0 なら（整数でない）→ 7 番の脱最適化スタブへ
   ... 整数前提の高速コードが一直線に続く ...
```

の二命令で済みます。肝心なのは、**外れたときの重い処理（フレームの組み立て直し）を、
この場に書かない**ことです。`jz` の飛び先 `deopt_stub_7` は、高速経路から外して
ある**コールドスタブ**（cold stub。めったに通らない場所に置く小さなスタブ）で、
「7 番のガードの復元レシピを使って
インタプリタへ戻れ」とランタイムを呼ぶだけ。こうすると、当たり続けるかぎり CPU は
分岐せず、命令キャッシュも分岐予測も汚れないので、ガードのコストはほぼゼロに
近づきます。[なぜ JIT か](why-jit.md)で「ガードは数命令」と言えたのは、この
重い後始末をコールドスタブへ追い出す工夫があってこそです。

さらに攻めると、ガードの命令すら消せる場合があります。代表が**暗黙の
ヌルチェック**です。`obj.field` のアクセスで `obj` が `nil` かどうかを、明示的に
`cmp`/`jz` で確かめる代わりに、そのままメモリを読みに行きます。`nil` を
内部表現で 0 とする処理系では、`obj` が `nil` のときフィールドアクセスは
「アドレス 0 ＋フィールドオフセット」を読みにいきます。このアドレス帯は OS が
あらかじめ**保護ページ**（アクセスすると必ず例外を出すページ）にしてあるので、
読んだ瞬間に CPU が例外（ページフォルト）を出します。すると OS の
シグナルハンドラが「どの命令で落ちたか」から対応する脱最適化地点を引き当て、
そこへ戻す。当たっているかぎり一命令も増えません。ただしこの技が使える
のは、フィールドのオフセットが保護ページの範囲に収まる場合だけです。
オフセットが大きすぎると「アドレス 0 ＋オフセット」が保護されていない番地に
届いてしまうので、そこは明示的なチェックに戻します。また、実際に `nil` が
頻繁に来てフォルトが多発する地点は、シグナル処理のコストのほうが高くつく
ため、明示チェック版で作り直されます（HotSpot はこれを自動で行います）。
「めったに起きないことの
チェックは、ハードウェアの罠に肩代わりさせる」という、JIT の常套手段です。

ただし、こうした状態の作り替えはどこでもできるわけではありません。脱最適化が
起こせる地点は、**JIT が復元情報（前節のレシピ）をあらかじめ用意した地点に
限られます**。これを脱最適化地点（deopt point）と呼びます。よく似た概念に
GC のための**セーフポイント**（safepoint。スレッドを安全に止めて、生きている
ポインタの位置を GC へ正確に伝えられる地点）があります。両者は**別系統**の
仕組みですが、「いま機械語のどの値が、論理的にどの状態（変数の値、生きている
オブジェクト）に対応するか」を正確に言える地点だ、という点で**同類**です。
どちらも「機械語の状態を論理状態へ写像できる地点」でだけ発動でき、その地点でだけ
対応表を用意すればよい。この共通の発想が、前節の「巨大なマップ」を現実的な
大きさに抑える効き目になります（GC との関係は次章で再訪します）。

## ガードはどこに、どれだけ置くか

脱最適化はタダではありません。ガードのチェックそのものにコストが
あり、何より脱最適化情報を保持するメモリが要ります。ガードが
多いほど、最適化器が動かせる自由（[中間表現](ir.md)で見たガードの巻き上げや
統合）も制約されます。だから JIT は、ガードを**減らす**努力をします。

- **重複ガードの除去**：上流で「`x` は整数」と確かめたら、下流の同じ
  ガードは消す。[基本ブロックバージョニング](compilation-units.md)は、
  これをブロックの版分けで自然に達成する手法でした。
- **ガードの巻き上げ（hoisting）**：ループ内で毎周チェックされる不変な
  ガードを、ループの外へ 1 回だけ追い出す。たとえば `arr` の要素を毎周
  「整数か」確かめる代わりに、ループ前で「`arr` は整数だけの配列か」を
  一度だけ確かめ、中では裸の整数演算で回す。1 億周なら、ガードを 1 億回から
  1 回へ減らせます。
- **ガードの強化（strengthening）と統合**：近接する複数の弱いガードを、
  1 つの強いガードにまとめる。`x >= 0` と `x < n` を別々に置く代わりに、
  「`0 <= x < n` を一度に」確かめれば、命令も復元情報も 1 か所で済みます。
  [中間表現](ir.md)の範囲解析で「ここはもう範囲内」と分かれば、配列の
  境界チェックごと消せます。
- **チェック順の最適化**：多態なガード（来る型が数種類）は、来る確率の
  高い型を先に並べ、分岐予測を当てる。プロファイルで「9 割は `Integer`」と
  分かっていれば、`Integer` の枝を一番上に置きます。
- **めったに来ない枝は罠に逃がす（uncommon trap）**：コンパイル時に
  「ここはまず通らない」と分かる枝（型エラー処理、稀な分岐）は、機械語を
  出さずに脱最適化地点だけを置く。コードが小さくなり、その枝が消えたぶん
  上流の最適化も伸びます。本当に通ったら、そこで作り直す。[HotSpot](case-jvm.md)の
  uncommon trap がまさにこれです。「投機」と「コンパイルしない決断」は地続き
  なのです。
- **脱最適化頻度の学習と指数的後退**：あるガードが何度も破れるなら、そこは
  投機が外れがちと判断します。ただし一度の失敗で諦めると揺り戻し（また熱く
  なって再コンパイル）が激しいので、破れた回数に閾値を設け、何度も外れた
  地点だけ「ここは投機しない」より保守的な版で作り直します。このとき再投機を
  許すまでの猶予（外れの許容回数）を、失敗のたびに 2 倍、4 倍…と倍々に
  延ばしていくのが**指数的後退**（exponential backoff）です。この学習が甘いと、
  脱最適化→再コンパイル→また脱最適化…を繰り返して性能が安定しません。これが
  [ウォームアップ問題](research-warmup.md)で見る現象の、実装側の原因です。

## 即時脱最適化と遅延脱最適化

ガードが破れたとき、いつ、どうやって戻すか。これにも二つの流儀があります。

- **即時脱最適化（eager）**：ガードを踏んだその場で、いま走っているフレームを
  インタプリタの形へ組み立て直し、対応するバイトコード位置へ飛ぶ。これまで
  見てきたのはこれです。「自分のフレームを、自分で降りる」素直なやり方。
- **遅延脱最適化（lazy）**：コードを「もう信用できない」と**印だけ付けておき**、
  実際の組み立て直しは、そのフレームに制御が戻ってきたときまで先送りする。

なぜ遅延が要るのか。理由は二つあり、どちらも「自分が走っているフレームの
ガードを、自分で踏んで降りる」即時方式では立ち行かない場面です。

- **他フレームを外から組み替える危険**：脱最適化したいフレームが、いま自分が
  走っているフレームとは別の、スタックの奥で実行中のフレームであることが
  あります。実行中のフレームの内部状態を、外から横入りして組み替えるのは危険で、
  そもそも安全に止められる地点（前節のセーフポイント）に来ているとも限りません。
- **大域前提の崩壊による一括無効化**：[部分評価](research-partial-eval.md)や
  [JVM](case-jvm.md)で見る「このメソッドは上書きされていない」といった大域的な
  前提は、新しいクラスがロードされた瞬間に崩れ、それに依存して最適化された
  すべてのコードが一斉に無効になります。無効化する対象は一つではなく、いま
  スタックの奥で実行中のものも含めた広範囲にわたります。

二つは別の問題ですが、解き方は共通です。いずれも「無効化はすぐ印を付けるだけに
しておき、組み立て直しはそのフレームに制御が戻るまで先送りする」ことで解けます。
そこで lazy 方式は、無効化されたコードの戻り番地を書き換えます。奥のフレームが
仕事を終えて「戻ってくる」その瞬間に、書き換えられた番地が脱最適化ルーチンへ
誘導し、そこで初めてフレームを組み立て直す。「走っている列車は止めず、駅に
着いたら乗り換えさせる」イメージです。

ここでも前々節の二択が顔を出します。前提の崩れうる場所すべてにガードを置く
（チェックが散らばる）か、**ガードを置かず大域的な前提として登録し、崩れたら
依存コードを一括無効化する**（ふだんのコストはゼロだが、無効化は大がかり）か。
クラス階層に基づく脱仮想化のように「めったに覆らないが、覆ると広範囲に効く」
前提では、後者（大域的前提＋一括の lazy 脱最適化）が好まれます。

> [!NOTE]
> 「ガードをばらまく」のと「大域的な前提を一つ張る」のは、同じ投機の二つの
> 実装です。前者は外れた地点だけ局所的に戻ればよいぶん単純ですが、チェックが
> 散らばります。後者はふだんノーコストで攻められますが、崩れたときに「誰が
> その前提に依存していたか」を追って一掃する仕組み（依存関係の登録と lazy な
> 脱最適化）が要ります。どちらを選ぶかは、その前提が**どれだけ覆りにくいか**で
> 決まります。

## 実行中のループを乗り換える OSR

脱最適化の双子の兄弟が **OSR**（On-Stack Replacement）です。
直訳すれば「スタック上での置き換え」。**すでに実行中の関数（特に長い
ループ）の途中で、実行形式を別のものに切り替える**技術です。

なぜ要るのか。[階層的 JIT](tiered.md)で見たように、ふつう新しい
コンパイル済みコードは「次にその関数を呼んだとき」から使われます。
ところが、

```ruby
# 一度しか呼ばれないが、中で 1 億回回るループ
def main
  sum = 0
  100_000_000.times { |i| sum += i * i }   # ← ここに入ったまま出てこない
  sum
end
```

このループは `main` を一度呼んだだけで延々と回り続けます。「次に呼んだ
とき」を待っていては、いつまでも最適化版に乗り換えられません。今まさに
回っているこのループを、途中から最適化版へ移したい。これが OSR の
出番です。

OSR には二つの向きがあります。

- **昇格 OSR（OSR-in）**：実行中のインタプリタ／ベースラインのループを、
  途中から最適化版へ乗り換える。上の `main` のループを最適化版に移す
  ケース。
- **降格 OSR（OSR-out）**：これは実は脱最適化そのものです。
  実行中の最適化版から、途中でインタプリタへ落とす。

> [!NOTE]
> 「降格」と言ってきましたが、降格先は**インタプリタとは限りません**。
> HotSpot はインタプリタへ、V8 もインタプリタ（Ignition）へ戻すのに対し、
> WebKit の JavaScriptCore は、最上位の FTL で投機が外れると原則として
> ベースライン JIT へ OSR exit します[](#cite:pizlo2020)。降格先に
> 必要な資格は「投機なしでも必ず正しく実行できる形式」であることだけで、
> 最下段である必要はないのです。降格先が速いほど、外れたあとの落ち込みも
> 浅くて済みます。各処理系の選択は事例編で見ます。

どちらも、やることは同じです。実行中のスタックフレームを、移り先の形式が
期待する状態へ作り替えるわけです。昇格 OSR なら「インタプリタのフレーム
（箱詰めの変数）」を読んで「最適化版のフレーム（レジスタ上の生の値）」を
組み立て、ループの対応する地点へジャンプする。脱最適化で使った
「状態の作り替え」の機械を、逆向きに使うわけです。

具体的には、昇格 OSR はこう進みます。[プロファイリング](profiling.md)の
バックエッジカウンタが閾値を超え、「このループはもう十分熱い」と分かると、
JIT はそのループの **OSR 専用版**をコンパイルします。ふつうの最適化版が
「関数の入口から入る」前提なのに対し、OSR 版は「ループの途中から、いまの
インタプリタフレームの値を受け取って入る」専用の入口（OSR エントリ）を持ちます。
次にバックエッジを踏んだとき、ランタイムは現在の変数（箱詰め）を読み出して
OSR 版の期待するレジスタ配置へ詰め替え、その入口へジャンプする。以後は最適化版が
ループを回し切ります。

降格（脱最適化）と違って、昇格 OSR には**正しい型情報がまだ手元で確定して
いない**という難しさがあります。ループの途中で乗り換えるので、ループ変数が
いま実際にどんな型かを、その場のフレームから観測して特殊化しなければなりません。
また「乗り換え地点」をどう作るかも勘どころです。熱さの判定はバックエッジ
（ループ末尾から先頭へ戻る辺）を踏んだ回数で行い、乗り換え地点そのものは
ループ先頭に置くのがふつうで、そこに OSR エントリ用のコードを余分に
用意するコストもかかります。だから JIT は、OSR を「めったに使わないが、長いループを救う
切り札」として、限られた地点にだけ用意します。

> [!NOTE]
> OSR の実装は、対象言語の構造に左右されます。ループが構文的にきれいな
> 言語では入口を見つけやすく、`goto` だらけで構造が崩れた言語では、
> OSR のための「乗り換え地点」を作るだけでひと苦労です。研究では、
> 構造化されていない言語での OSR をどう実現するか、といったテーマも
> 扱われています[](#cite:delia2018)。

> [!TIP]
> OSR には、性能測定にまつわる有名な罠があります。`main` に長いループを
> 一つ書いて測るマイクロベンチマークでは、そのループは通常の入口からでは
> なく **OSR 版**としてコンパイルされ、測られます。OSR 版は「ループの
> 途中から飛び込める入口」を持つぶん最適化が制約されることがあり、同じ
> コードをメソッド呼び出しとして温めて測った結果と食い違うのです。
> Java のベンチマークハーネス JMH が「素朴なループで測るな」と口を
> 酸っぱくして言うのには、こうした理由もあります。

## 脱最適化と正しさ

ここで一つ、深い問いが立ち上がります。**「途中まで最適化版で進み、
続きをインタプリタが引き継ぐ」とき、本当に元のプログラムと同じ答えに
なると、どうして保証できるのか。**

おもちゃの例では `acc` と `i` を渡すだけでしたが、本物では最適化器が
コードを大きく組み替えています。命令を並べ替え、変数を消し、計算を
前倒しし…。それでも「どのガードの地点で抜けても、インタプリタが正しく
再開できる」状態を常に復元できなければなりません。最適化のどんな
変形が、この性質を壊さずに許されるのか。

これは「気をつければ大丈夫」で済ませてよい問題ではなく、**形式的に
証明すべき**問題です。実際、投機的最適化と脱最適化の正しさを
数学的に定式化し、証明する研究があります[](#cite:fluckiger2018)。
この重要なテーマは、第 II 部の
[投機的最適化の正しさ](research-deopt-correctness.md)で改めて扱います。

## 脱最適化を「簡単にする」工夫

ここまで見てきたとおり、脱最適化はそれ自体が面倒です。復元情報を持ち、レジスタに
ばらけた状態を箱詰めのフレームへ組み立て直し、最適化との綱引きにも気を配る。
そこで「そもそも脱最適化を簡単にできないか」という方向の工夫があります。発想は
どちらも、前に見た二択のうち「①コンパイル結果に制約を課して復元を容易にする」を
極端まで推し進めることです。

第一の道は、**戻り先をインタプリタそのものにしてしまう**ことです。
[メタトレーシング](meta-tracing.md)や[部分評価](research-partial-eval.md)の
フレームワーク（PyPy/RPython、Truffle/GraalVM）では、最適化コードはインタプリタから
自動で導出されます。だとすれば「最適化コードのこの地点は、インタプリタのどの状態に
対応するか」をフレームワークが知っている。脱最適化は「このノードの解釈に戻って、
再特殊化する」だけになり、実装者は復元のレシピを一行も書きません
[](#cite:wurthinger2017)。インタプリタが唯一の正典なので、戻り先の状態は定義から
自明です。これは「制約側に振り切れば、巨大なマップが消える」ことの好例です。

第二の道は、**機械語でなくソースを生成する**処理系で、脱最適化をソースの構造として
表すことです。SQL を実行時にコンパイルする処理系（[言語処理系の外の JIT](beyond-languages.md)で
ふれます）のうち、C のようなソースコードを生成する流儀では、
機械語のスタックを直接いじれません。けれど、いじる必要もありません。それを示したのが
Essertel らの「複製・最適化・トランポリン」です[](#cite:essertel2021)。
ループを複数の版に複製し、各版を独立に最適化し、反復と反復の境目でだけ別の版へ
飛び移る（トランポリン）。生きている変数は引数（ポインタ）として明示的に持ち回るので、
脱最適化は「生きた状態を抱えたまま、一般版のループへ飛び移る」ことに尽きます。
復元マップも再具体化も要らず、必要なのは構造化された制御フロー（ループと条件分岐）だけ。
OSR を中間表現の特別な命令として持ち込まず、早い段階でただのループと関数呼び出しへ
翻訳し去るので、下流のコンパイラ（gcc/clang）を一切改造せずに済みます。OSR を
「関連する複数の版のあいだの実行の移送」と捉え直すこの見方は、近年 VM の外でも
研究が進んでいます[](#cite:delia2018)。

> [!TIP]
> どちらの道も効くのは、最適化版と戻り先が状態の表現を共有しているからです。
> 値がレジスタに閉じ込められておらず、インタプリタの構造や明示的なループ変数として
> 見えていれば、脱最適化は「その状態を渡して別の実行へ移る」だけで済みます。逆に、
> 攻めたレジスタ割り当てで状態を機械語の隅々に散らした本格的なメソッド JIT では、
> やはり巨大なマップ（②）の世界に戻ります。**どこに状態を置くかが、脱最適化の難しさを
> 決める**のです。

> [!TIP]
> 脱最適化は、フラグ一つで観察できます。V8（Node.js）では `--trace-deopt` で、
> どの関数がどの理由で脱最適化したかが逐一表示されます。HotSpot では
> `-XX:+PrintCompilation` のログに、無効化されたコンパイル済みコードが
> **made not entrant**（もう入口から入れない印）として現れます。Ruby の
> YJIT なら `--yjit-stats` が、ガードから抜けた回数とその内訳
> （`side_exit_count` や各命令の exit reasons）を集計してくれます。自分の
> コードで投機がどれだけ当たっているか、ぜひ覗いてみてください。

## まとめ

- **脱最適化**は、投機が外れたとき、最適化コードの実行状態を
  インタプリタが再開できる状態へ作り替えて戻る仕組み。投機を攻めさせる
  ための前提。
- 難しさは**状態の作り替え**にある。レジスタにばらけた生の値や、消えた
  変数を、箱詰めの整然としたインタプリタフレームへ復元する。そのための
  復元のレシピ（脱最適化情報）をガードごとに用意する。
- 復元情報は **「最適化の自由」対「情報の大きさ」** の綱引き。①最適化を縛って
  情報を小さく保つか、②自由に攻めて巨大な脱最適化マップで払うか。多くは②を
  選び、セーフポイントの限定、差分圧縮、再具体化（値でなく作り方を持つ）で切り詰める。
- ガードは機械語では**数命令**（`cmp`／`test`＋分岐→コールドスタブ）。暗黙の
  ヌルチェックのようにハードウェアの罠へ追い出せば命令すら消せる。脱最適化と
  GC はセーフポイントでだけ起こせる。
- 起源は Self の 1992 年の論文で、動機はデバッグだった。
- ガードはコストなので、重複除去、巻き上げ、強化、順序づけ、uncommon trap、
  頻度の学習で減らす。
- 戻し方には**即時（eager）**と**遅延（lazy）**がある。大域的な前提の一括
  無効化には、依存関係の登録と lazy 脱最適化が要る。
- **OSR** は実行中のループを途中で別形式へ乗り換える技術。昇格 OSR
  （インタプリタ→最適化、専用の OSR エントリを用意）と降格 OSR（＝脱最適化）がある。
- 脱最適化を**簡単にする**には、戻り先をインタプリタにする（[部分評価](research-partial-eval.md)や
  [メタトレーシング](meta-tracing.md)で復元を自動化）か、[ソース生成](beyond-languages.md)で
  「複製・最適化・反復境界でのトランポリン」として表す（機械語のスタックを触らない）。
  鍵は最適化版と戻り先が状態表現を共有すること。
- 「途中から正しく再開できる」ことの保証は、形式的に証明される研究テーマ。

投機と脱最適化の安全網が出そろいました。けれど、生成した機械語は孤立して
動くのではなく、ガベージコレクタやスレッド、そして悪意ある相手とも隣り合って
動きます。次章ではまず、JIT が GC やランタイムと協調して安全に動くための
仕組み（セーフポイントやスタックマップ）を見ます。
