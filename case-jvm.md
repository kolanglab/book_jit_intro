# ケーススタディ：JVM（HotSpot と GraalVM）

ケーススタディの最初は、JIT 研究の本流 —— **Java 仮想マシン**（JVM）です。
とりわけ標準実装の **HotSpot** は、20 年以上にわたって磨かれ、本書で
見てきた技法のほとんどを実戦投入してきた、いわば JIT の博物館です。
ここを丁寧に見ておくと、他の処理系が「HotSpot のどの部分を真似て、
どこを変えたか」として理解できるようになります。

## 出自：Self から HotSpot へ

HotSpot のルーツは、本書で何度も登場した **Self** の処理系にあります。
[脱最適化](deopt-osr.md)（1992 年、Self のデバッグのために発明）も、
[多態インラインキャッシュ](profiling.md)[](#cite:holzle1991)も、
[型フィードバック](profiling.md)[](#cite:holzle1994)も、Self で生まれました
（[インラインキャッシュ](profiling.md)そのものの発明はさらに遡り、
Smalltalk-80 の処理系です[](#cite:deutsch1984)）。

Self から HotSpot への道筋には、もう一段の物語があります。Self プロジェクト
出身の Lars Bak や Urs Hölzle らは、Smalltalk 処理系 **Strongtalk** を開発する
小さな会社 Animorphic Systems（LongView Technologies）に集まり、Self の
VM 技術を磨き直しました。1997 年に Sun がこの会社を買収すると、チームは
その VM 技術を Java へ向け、HotSpot として結実させます（1999 年に公開され、
JDK 1.3 で既定の JVM になりました）。だから HotSpot は、最初から
「観測して投機する」JIT の思想を血肉としています。なお、このチームを率いた
一人の Lars Bak は、のちに Google で [V8](case-v8.md) の開発を率いることに
なります。Self の系譜は、JVM と JavaScript の両方へ流れ込んでいるのです。

名前の "HotSpot" 自体が、[なぜ JIT か](why-jit.md)で見た**ホットスポット**
（熱い場所だけを最適化する）に由来します。

## 二つのコンパイラ：C1 と C2

HotSpot の JIT は、伝統的に二つのコンパイラを持ちます
[](#cite:paleczny2001)。これは[階層的 JIT](tiered.md)の典型例です。

- **C1（クライアントコンパイラ）**：素早くコンパイルする[ベースライン](tiered.md)
  寄りのコンパイラ。最適化は控えめだが、起動が速い[](#cite:kotzmann2008)。
- **C2（サーバコンパイラ）**：時間をかけて徹底的に最適化する。[中間表現](ir.md)で
  見た **Sea of Nodes**[](#cite:click1995)を使い、高品質な機械語を出す。

現代の HotSpot は、この二つとインタプリタを組み合わせた**階層的コンパイル**
（tiered compilation）を標準とします。実行レベルは 0〜4 まであり、

```
レベル0: インタプリタ
レベル1: C1（通常の C1 最適化あり、プロファイルなし）
レベル2: C1（カウンタのみの限定プロファイル）
レベル3: C1（フル・プロファイル収集つき）
レベル4: C2（全力最適化）
```

レベル 1〜3 はどれも同じ C1 によるコンパイルで、違いは**埋め込む
プロファイル計装の量**です。

と段を上がっていきます。ふつうの流れは「0 → 3（C1 でプロファイルを
集めながら動かす）→ 4（集めたプロファイルで C2 が最適化）」です。
[プロファイリング](profiling.md)で見たカウンタとフィードバックが、この
段の昇格を駆動します。

ここで HotSpot 特有の設計を一段具体に見ましょう。鍵は **C1 と C2 が
プロファイルを共有する**ことです。レベル 3 の C1 コードは、ただ速い
機械語を出すだけでなく、**計装（profiling）コードを埋め込んだ**機械語を
出します。具体的には、メソッドごとに `MethodData`（HotSpot の内部では
**MDO**, MethodDataObject と呼ばれる領域）を確保し、

- 各分岐の **taken/not-taken の回数**
- 各仮想呼び出し地点での**受け取った型（receiver type）とその頻度**
- 各 `null` チェックや配列ストアで**例外が起きたか**

を実行のたびに数えて書き込みます。レベル 4 で C2 が起動するとき、C2 は
この MDO を読み、「この分岐はほぼ常に taken」「この呼び出し地点には
99% `Circle` が来る」といった**実測の偏り**を前提に投機します。つまり
**C1 は観測係、C2 は投機係**という分業です。

> [!NOTE]
> なぜレベル 1・2 を分けるのでしょうか。プロファイル収集はタダでは
> ありません。カウンタのインクリメントぶんだけレベル 3 のコードは
> レベル 1 より遅くなります（おおむね数十パーセント遅いと言われます）。
> そこで HotSpot は、「もう C2 が詰まっていて当面レベル 4 に上がれない」
> と判断したメソッドを、計装の無いレベル 1/2 に逃がして速度を確保する、
> といった調停を行います。C2 のコンパイルキューが長いときに「観測しても
> どうせすぐ使わない」なら、観測コストを払わないほうが得、という
> トレードオフです。

各段への昇格は、**メソッド呼び出し回数カウンタ**と**バックエッジ
（ループの戻り）カウンタ**の二本立てで駆動されます。後者がループ内で
先に閾値を超えると、メソッドの実行中であってもそのループだけを
コンパイル済みコードへ差し替える [On-Stack Replacement（OSR）](deopt-osr.md)が
起こります。閾値はおおむね数千〜一万回のオーダーで、`-XX:CompileThreshold`
などで調整できますが、階層的コンパイルでは各段ごとに別の閾値
（`Tier3InvocationThreshold` など）が使われ、しかも**コンパイラの混み
具合に応じて動的に補正**されます。「閾値は固定値ではなく、システムの
負荷に追従する」のが現代 HotSpot の流儀です（こうした適応的な最適化
駆動の考え方を体系化したのは、同時代の Java VM である Jikes RVM の
研究でした[](#cite:arnold2000)）。

## 投機の主役：仮想呼び出しの脱仮想化

Java で JIT がもっとも効くのは、**仮想呼び出し**（virtual call、実行時に
相手のメソッドが決まる呼び出し）の最適化です。Java では、ほとんどの
メソッド呼び出しが原理的には仮想呼び出しで、[なぜ JIT か](why-jit.md)で
見た「探して飛ぶ」間接呼び出しのコストを伴います。

HotSpot はこれを**クラス階層解析**（Class Hierarchy Analysis、CHA）と
[型フィードバック](profiling.md)で攻めます。たとえば次のコードを考えます。

```java
abstract class Shape { abstract double area(); }
class Circle extends Shape {
    double r;
    Circle(double r) { this.r = r; }
    double area() { return 3.14159 * r * r; }
}

double totalArea(Shape[] shapes) {
    double sum = 0;
    for (Shape s : shapes) {
        sum += s.area();   // 仮想呼び出し：s の実際の型しだい
    }
    return sum;
}
```

`s.area()` は、`s` が `Circle` なのか他の `Shape` なのか、原理的には
実行時まで分かりません。ところが、もしプログラム全体で `Shape` の具体
クラスが `Circle` しかロードされていなければ、CHA は「`area()` の実装は
`Circle#area` 一つしかない」と判断できます。すると HotSpot は、`s.area()` を
**`Circle#area` への直接呼び出しに脱仮想化**し、さらに本体が小さいので
[インライン化](why-jit.md)します[](#cite:detlefs1999)[](#cite:ishizaki2000)。

```java
// HotSpot が最適化したあとのイメージ（概念）
for (Shape s : shapes) {
    // ループ内に型チェックは置かない。「実装は Circle#area だけ」という
    // 前提はクラスローダ側で監視される（後述の assumption）
    sum += 3.14159 * ((Circle) s).r * ((Circle) s).r;  // インライン展開
}
```

ここがまさに投機です。「`Circle` しかない」という前提は、将来別の
`Shape` サブクラスがロードされると崩れます。そのとき HotSpot は
[脱最適化](deopt-osr.md)（HotSpot の用語では **uncommon trap**、
めったに起きない罠）して、仮想呼び出しのある安全なコードへ戻ります。
**クラスのロードという、Java 特有の動的なイベントが、脱最適化の引き金**に
なるのが面白いところです。

### Assumption による一括無効化

ここで注目したいのは、HotSpot が**この前提をどう記録し、どう壊すか**の
実装テクニックです。CHA で「`area()` の実装は一つだけ」と仮定して
脱仮想化したとき、C2 はそのコンパイル済みメソッドに **assumption
（前提）**を一つ貼り付けます。たとえば「`Shape#area` の単一実装は
`Circle#area` である」という形です。これは**そのメソッドのガード命令と
してコード中に埋め込むのではなく**、クラスローダ側の依存テーブルに
登録されます。

そして将来、誰かが `class Square extends Shape` を**ロードした瞬間**、
クラスローダは「`Shape#area` の実装が増えた」ことを検知し、この
assumption に依存していた**コンパイル済みメソッドをまとめて無効化
（invalidate）**します。無効化されたコードはもう新規には呼ばれなくなり、
すでに実行中のスレッドは次にセーフポイントへ来たところで安全なコードへ
落とされます。

> [!TIP]
> このやり方の妙味は、**ホットループの中に「実装は一つか？」のチェックを
> 一切入れずに済む**ことです。前提が成り立っている間はチェックのコストが
> ゼロで、前提が壊れる稀なイベント（クラスロード）の側で一括して後始末を
> する。「観測（チェック）のコストを、頻度の低いイベント側へ寄せる」と
> いうこの発想は、[部分評価](research-partial-eval.md)で見た「前提を
> 固定して特化し、崩れたら作り直す」のと同型です。コストの置き場所を
> ずらすのが投機の本質、というわけです。同じ発想は他の処理系にもあり、
> Ruby の YJIT は **invariants** という仕組みとして持っています
> （[Ruby のケーススタディ](case-ruby.md)で見ます）。

### 単態・二態・多態の段階的対応

CHA が単一実装を保証できないとき —— たとえば `Shape` のサブクラスが
複数ロードされている場合 —— は、CHA ではなく **型フィードバック**
（MDO に記録した受け取り型の頻度）に頼ります。

```java
// 型フィードバックで「99% は Circle」と分かったときの概念
for (Shape s : shapes) {
    if (s.getClass() == Circle.class) {       // ガード（速い同値比較）
        sum += 3.14159 * ((Circle) s).r * ((Circle) s).r;  // インライン
    } else {
        sum += s.area();   // フォールバック：本物の仮想呼び出し or uncommon trap
    }
}
```

このとき CHA 由来の脱仮想化と違い、ガードは**コードの中に残ります**
（クラスロードでは壊れず、実行中に別の型が来ることがあるため）。

> [!NOTE]
> 呼び出し先が 1 種類なら単態、2 種類なら**二態**（bimorphic）として
> インライン化し、それ以上に散らばると[多態インラインキャッシュ](profiling.md)
> [](#cite:holzle1994)や、あきらめて仮想呼び出しのまま、と段階的に対応
> します。「いくつまでなら投機的にインライン化するか」は HotSpot の
> 重要なチューニングどころです。HotSpot の既定では、観測された型が
> 二つまでならガード付きで両方インライン化し、三つ以上に散らばると
> インラインキャッシュ経由の呼び出しか、頻度のもっとも高い一つだけを
> インラインして残りは uncommon trap に逃がす、といった判断をします。

### uncommon trap：来ない分岐はコンパイルしない

uncommon trap は脱仮想化だけでなく、**およそ「めったに来ない」と
分かったあらゆる地点**で使われる、HotSpot のもっとも汎用的な投機の
道具です。具体例を挙げます。

- MDO で「この `else` 分岐は 1 万回中 0 回しか来ていない」と分かれば、
  C2 はその `else` 側の機械語を**そもそも生成しません**。代わりに
  uncommon trap への分岐を一つ置くだけにします。コードは小さくなり、
  来る側のホットパスだけがまっすぐ並びます。
- 配列の境界チェックや `null` チェックも、MDO 上「一度も失敗して
  いない」なら、例外を投げるコードを展開せず uncommon trap に逃がします。

そして万一その稀な分岐に**実際に到達**したら、uncommon trap が発火して
[脱最適化](deopt-osr.md)し、インタプリタへ戻ります。HotSpot はその地点を
「もう稀ではない」と学習し直し、次回コンパイル時には今度はちゃんと
その分岐のコードを生成します。**「来ないと思って手を抜き、来たら作り
直す」**——この一往復で、ホットパスの機械語を極限まで痩せさせるのが
uncommon trap の狙いです。

## インライン化の閾値という設計判断

脱仮想化までで「どこへ飛ぶか」が決まっても、本当に効くのはそのあと
**インライン化**して呼び出しそのものを消し、呼び出し境界をまたいだ
最適化（定数畳み込み、エスケープ解析、ループ最適化）を可能にする
ところです。[なぜ JIT か](why-jit.md)で見たとおりインライン化は最適化の
母ですが、無制限に展開するとコードが膨れ、命令キャッシュを汚し、
かえって遅くなります。だから HotSpot は**いくつもの閾値とヒューリ
スティクス**でインライン化を制御します。具体感をつかむために、代表的な
ものを挙げます。

- **小さいメソッドは無条件にインライン**：バイトコードがごく小さい
  （目安として `MaxInlineSize`、既定で 35 バイトコード程度）メソッドは、
  呼び出し回数にかかわらずインラインします。ゲッタ／セッタのような
  アクセサはこれでほぼ全部消えます。
- **ホットなら大きめでも許す**：頻繁に呼ばれる（hot な）メソッドは、
  もっと大きくても（目安として `FreqInlineSize`、既定で 325 バイト
  コード程度まで）インラインします。「熱い呼び出し境界ほど消す価値が
  高い」という判断です。
- **深さと総量の上限**：インライン化は再帰的に進むので、`MaxInlineLevel`
  でネストの深さを、コンパイル単位全体のサイズで総量を抑えます。
  ここを超えると、たとえホットでも展開を打ち切ります。

```java
class Vector { double x, y; double getX() { return x; } }

double sumX(Vector[] vs) {
    double s = 0;
    for (Vector v : vs) s += v.getX();  // getX は 4 バイトコード程度
    return s;                            // → 必ずインライン、s += v.x になる
}
```

> [!IMPORTANT]
> インライン化の閾値はバイトコードの**サイズ**で測られる点に注意して
> ください。ソース行数でも実行命令数でもなく、コンパイルしようとして
> いるメソッドのバイトコード長です。だから「同じ処理でもメソッドを
> 大きく書くとインラインされなくなる」ことが起こり得ます。ライブラリの
> ホットメソッドを意図的に小さく保つ、という最適化テクニックの背景が
> これです。

> [!NOTE]
> インライン化と並ぶ「呼び出しを消す」道具として、HotSpot は **intrinsics
> （組み込み特殊化）** も持っています。`System.arraycopy` や `Math.sqrt` の
> ような頻出メソッドは、Java で書かれた本体をコンパイルするのではなく、
> コンパイラがあらかじめ知っている特製の機械語（しばしば SIMD 命令まで
> 使った手書き同然のコード）を直接埋め込みます。[Ruby の章で見る組み込み
> メソッドの特殊化](case-ruby.md#組み込みメソッドの特殊化)と同型の発想です。

## Java 特有の最適化：エスケープ解析

もう一つ、Java の JIT で効果が大きいのが**エスケープ解析**（escape
analysis）です。Java はあらゆるオブジェクトをヒープに確保し、[ガベージ
コレクション](glossary.md)で回収します。短命なオブジェクトを大量に作ると、
確保と回収のコストがかさみます。

エスケープ解析は、「あるオブジェクトが、それを作ったメソッドの外へ
**漏れ出す（エスケープする）か**」を調べます。漏れ出さない —— そのメソッドの
中でしか使われず、外に渡らない —— と分かれば、ヒープに確保する必要が
ありません。

```java
double distance(double x1, double y1, double x2, double y2) {
    Point p1 = new Point(x1, y1);   // この Point は…
    Point p2 = new Point(x2, y2);   // …メソッドの外へ出ない
    return p1.distanceTo(p2);
}
```

`p1` と `p2` は `distance` の中で作られ、中で使われ、外へ返りも渡りも
しません。C2 はこれを見抜き、**オブジェクトを作らずに、そのフィールド
（`x`, `y`）を直接ローカル変数（できれば CPU レジスタ）に置きます**。
これを**スカラ置換**（scalar replacement）と呼びます。[トレーシング](tracing.md)で
ふれた「ループ内アロケーションの除去」と同じ発想を、メソッド JIT で
実現したものです。`new` が消え、GC の負担も消えます。

ここで大事なのは、**エスケープ解析はインライン化を前提にして初めて
効く**ということです。`distanceTo` が別メソッドのままなら、`p1`/`p2` は
そのメソッドへ「渡る」のでエスケープしてしまいます。HotSpot はまず
`distanceTo` を `distance` の中へインライン展開し、呼び出し境界を消した
**あとで**エスケープ解析をかけます。だから前節のインライン化と
本節のスカラ置換は連鎖して働く、と理解してください。

> [!NOTE]
> GraalVM はこの「多態インライン化 ＋ 部分エスケープ解析 ＋ スカラ置換」の
> 連鎖をさらに推し進めます。C2 のスカラ置換は「メソッド全体で漏れない」場合に
> 限られますが、Graal は「一部の分岐でだけ漏れる」オブジェクトも漏れる経路の
> 直前まで確保を遅延できる**部分エスケープ解析**へ一般化しました[](#cite:stadler2014)。
> これにより `map` や `foldLeft` のような**高水準のコレクション
> 操作**でさえ、中間オブジェクトやイテレータ・ボックス化の確保ごと消し去れる
> ことを実証しました[](#cite:prokopec2017)。関数型の高級な書き方が、手書きの
> 素朴なループに並ぶ速度まで最適化される —— インライン化・部分エスケープ
> 解析・投機が噛み合った好例です。

HotSpot はエスケープの度合いを 3 段階で扱います。

- **エスケープしない（NoEscape）**：メソッド内で完結。→ **スカラ置換**で
  `new` ごと消す。最強。
- **引数としてのみ漏れる（ArgEscape）**：呼び出し先には渡るが、ヒープや
  別スレッドへは出ない。→ スカラ置換はできないが、後述の**ロック省略**の
  対象にはなる。
- **完全に漏れる（GlobalEscape）**：フィールドへ格納される、戻り値で
  返る、別スレッドから見える、など。→ 何もできない。素直にヒープへ。

もう一つ、スレッド内に閉じていると分かったオブジェクトに対する
**ロック省略**（lock elimination、同期の除去）も重要です。

```java
StringBuffer sb = new StringBuffer();  // 外へ漏れない
sb.append("a"); sb.append("b");        // append は内部で synchronized
return sb.toString();
```

`StringBuffer#append` は内部で `synchronized` ですが、`sb` がこのメソッドの
外へ漏れない（＝他スレッドから絶対に触られない）と分かれば、その
ロック取得・解放は**意味を持たないので丸ごと削除**できます。これが
ロック省略です。「漏れないなら排他は要らない」という、エスケープ解析の
もう一つの果実です。

> [!WARNING]
> スカラ置換は強力ですが、**前提が崩れたときの後始末**が必要です。
> もしこのメソッドの途中で uncommon trap が発火して脱最適化したら、
> インタプリタへ戻る時点では「存在しないはずだったオブジェクト」を
> 実体化して渡し直さねばなりません。HotSpot はこれを **再具体化**
> （rematerialization）として行います。消したはずの `new` を、脱最適化の
> 瞬間にだけ実行してオブジェクトを復元するのです。投機的に消す最適化には、
> ほぼ必ずこの「崩れたら復元する」機構が裏で付いてきます。「最適化で
> 消した状態を脱最適化時に組み立て直す」という発想自体の源流は、Self の
> 脱最適化の研究にあります[](#cite:holzle1992)。

## セーフポイント：脱最適化と GC が「待たされる」事情

ここまで「クラスがロードされたら無効化する」「uncommon trap で
インタプリタへ戻る」と気軽に書いてきましたが、実装上は厄介な問題が
あります。**スレッドが最適化済みの機械語をまさに実行している最中に、
その足元のコードを差し替えることはできない**のです。レジスタの中身、
スタックフレームの形、どこまで実行したか —— こうした状態が「最適化後の
世界」でできているとき、いきなり「インタプリタの世界」の状態へ
読み替えることは、任意の地点ではできません。

そこで使われるのが、[JIT とランタイムの協調](gc-runtime.md)で見た
**セーフポイント**（safepoint）です。本節では、あの仕組みを HotSpot が
どう運用しているかを見ます。セーフポイントとは、**スレッドの状態
（全レジスタとスタックの中身）の意味が正確に分かっている地点**の
ことでした。HotSpot のコンパイラは機械語を出すとき、主に

- ループのバックエッジ（戻りの枝）
- メソッドからの復帰部

にセーフポイント（のポーリング）を仕込み、その地点ごとに「このとき
レジスタ `rax` は変数 `i` を、スタックのこの位置は変数 `sum` を持つ」と
いう対応を、[スタックマップ（OopMap）](gc-runtime.md)—— どのレジスタ／
スタック位置がオブジェクト参照かを示す表 —— などとして記録しておきます。

GC や脱最適化を行いたい VM は、全スレッドへ「次のセーフポイントで止まれ」と
要求します。各スレッドは自分のペースで実行を続け、**次にセーフポイントへ
来たところで初めて停止**します。全員が止まって初めて、

- **GC** は、各スレッドのスタックマップを見て「どのレジスタ／スタック
  位置がオブジェクト参照か」を正確に知り、安全にオブジェクトを移動できる。
- **脱最適化** は、そのセーフポイントの記録を使って「最適化後の状態」を
  「インタプリタの状態」へ翻訳し、スカラ置換したオブジェクトを再具体化
  して、安全にインタプリタへ落とせる。

```
スレッド: ──[最適化コード実行中]──●セーフポイント──[ここで初めて止まれる]
VM:        「止まれ」要求 ↑                  ↑ 全員揃ったら GC / 脱最適化
```

> [!IMPORTANT]
> ここに重要なトレードオフがあります。セーフポイントを細かく置けば「止まれ」に
> 素早く応答できますが、記録が増えて最適化の自由度が下がります
> （セーフポイントをまたいで値を動かしにくい）。逆に粗く置けば最適化は
> 自由になりますが、**セーフポイントに到達しないループ（例：セーフポイントの
> 無い長大なカウントループ）があると、全スレッドがそのループの終了を待たされ、
> GC や脱最適化がいつまでも始められない**という事故が起きます。
> 「最適化の自由度」と「VM が割り込める応答性」の綱引きが、セーフポイントの
> 配置設計です。

つまり[脱最適化](deopt-osr.md)は、概念としては「いつでも安全側へ戻れる」
ものですが、実装上は「**セーフポイントまで来て初めて**戻れる」。この
「待ち合わせ」の存在が、JIT を持つ VM の停止時間（pause）特性を
形づくっています。

## Graal：Java で書かれた JIT

HotSpot の物語は C2 で終わりません。**Graal** は、C2 を置き換えるべく
開発された新しい最適化コンパイラです。最大の特徴は、**Graal 自身が
Java で書かれている**ことです（C2 は C++ で書かれています）。「JIT
コンパイラを、その VM が動かす言語自身で書く」という、自己言及的な
設計です。

Graal は[中間表現](ir.md)に Sea of Nodes 系の表現を使い、[投機の前提と
脱最適化点を第一級のノード](deopt-osr.md)として扱う洗練された設計を
持ちます[](#cite:duboscq2013)。そしてこの Graal が、[部分評価と Futamura
射影](research-partial-eval.md)で見た **Truffle** の土台でもあります。
Truffle 言語（TruffleRuby・GraalPy など）は、Graal の部分評価機構の上に
載っているのです。

さらに **GraalVM** は、[ウォームアップ問題](research-warmup.md)で
ふれた **Native Image** —— Java アプリを AOT コンパイルして瞬時に起動
できるようにする機能 —— も提供します[](#cite:wimmer2019)。「JIT（実行時最適化）と AOT
（瞬時起動）を一つの基盤で両立する」という、ウォームアップ問題への
JVM 世界の回答が、GraalVM に集約されています。

> [!NOTE]
> ただし、Graal が C2 を置き換える未来は、本書執筆時点では実現していません。
> Graal は JDK 9 から実験的 JIT として JDK に同梱されましたが、利用が
> 広がらず、JDK 17 で JDK 本体からは削除されました[](#cite:jep410)。以後
> Graal を使うには別配布の GraalVM を使います。OpenJDK には Graal の
> 再統合を目指す Project Galahad も発足しましたが、2025 年に Oracle が
> GraalVM チームの重心を GraalPy・GraalJS など非 Java 言語へ移す方針を
> 発表し[](#cite:graalvmdetach)、Galahad は 2026 年に解散しました。
> HotSpot の既定の最適化コンパイラは、今も C2 です。

> [!TIP]
> 手元の JDK で JIT の挙動を覗くこともできます。`-XX:+PrintCompilation`
> を付けて Java プログラムを動かすと、どのメソッドが何レベルでコンパイル
> されたかが流れます。`-XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining`
> でインライン化の判断も見えます。本章で説明した「熱くなって段を上がり、
> インライン化され、ときに脱最適化する」様子を、実物で観察してみて
> ください。

## まとめ

- **HotSpot** は Self の系譜を継ぎ、[脱最適化](deopt-osr.md)・[型
  フィードバック](profiling.md)・[インラインキャッシュ](profiling.md)を
  実戦に持ち込んだ JIT の本流。
- **C1（速い）＋ C2（強い、[Sea of Nodes](ir.md)）** の[階層的コンパイル](tiered.md)。
  レベル 0〜4 を昇格する。**C1 が計装つきコード（MDO への記録）を出し、
  C2 がそれを読んで投機する**分業が肝。昇格の閾値は混み具合で動的に補正。
- 投機の主役は**仮想呼び出しの脱仮想化**。CHA で単一実装を **assumption**
  として仮定し、ホットループ内のチェックをゼロにする。新クラスのロードで
  assumption に依存するコードを**一括無効化**。型フィードバックでは
  単態・二態をガード付きでインライン、散れば多態 IC か **uncommon trap**。
- **uncommon trap** で「来ない分岐はそもそもコンパイルしない」。来たら
  脱最適化して作り直す。ホットパスの機械語を痩せさせる汎用手段。
- **インライン化の閾値**はバイトコード**サイズ**で判断。小メソッドは
  無条件、ホットなら大きめも、深さと総量に上限。エスケープ解析の前提でもある。
- **エスケープ解析＋スカラ置換**で、漏れ出さないオブジェクトの `new` を
  消す。漏れなければ**ロック省略**も。脱最適化時は **再具体化** で復元。
- **セーフポイント**：GC も脱最適化も、全スレッドがセーフポイントに来て
  初めて実行できる。最適化の自由度と VM の応答性のトレードオフ。
- **Graal** は Java で書かれた新世代コンパイラで、[Truffle](research-partial-eval.md)の
  土台。**GraalVM** は JIT と AOT（Native Image）を両立。

次は、JavaScript を四段で速くする、ヒドゥンクラスとインラインキャッシュの
王国 —— V8 を見ます。
