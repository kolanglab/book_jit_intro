# Copy-and-Patch：貼って埋めるだけの高速コード生成

[ウォームアップ問題](research-warmup.md)で見たとおり、JIT のコンパイル
それ自体が遅いと、起動の速さを損ない、短命なプロセスでは利得を取り逃します。
[フレームワーク](frameworks.md)の章でも、「LLVM は最適化は強いが生成が
遅い」という JIT 特有の悩みを見ました。

では、**最適化の品質をほどほどに諦める代わりに、コード生成を極限まで
速くする**と、どこまで行けるでしょうか。2021 年の **Copy-and-Patch**
[](#cite:xu2021)は、この方向を突き詰めました。論文の SQL クエリ
コンパイラの評価（TPC-H ベンチマーク）では、**LLVM `-O0` より 2 桁
（約 100 倍）、それより高い最適化レベルより 3 桁（約 1000 倍）速く**
コードを生成しながら、生成した機械語は**インタプリタより一桁（約 10 倍）
速く、しかも LLVM `-O0` 出力より 14% 速い**という結果を出しています
[](#cite:xu2021)。そして 2024 年、この技法は **CPython 3.13 の公式
JIT**[](#cite:cpython313jit)に採用され、研究から主流へ駆け上がりました。
この章では、その鮮やかなアイデアを、実際に動く機械語生成で体験します。

## アイデア：コンパイルを「貼り絵」にする

ふつうのコンパイラは、命令を**その場で組み立て**ます。レジスタを割り当て、
オペコードを選び、即値を符号化し…という作業を実行時に行う。これが
時間を食います。

Copy-and-Patch の発想は、その作業を**ほとんど前もって済ませておく**ことです。
バイトコードの各命令について、それを実行する機械語の断片を**あらかじめ
コンパイルしておき**、実行時には**コピーして、穴を埋めるだけ**にする。

この、あらかじめ用意した「穴あきの機械語断片」を**ステンシル**（stencil、
型抜き）と呼びます。ステンシルには、実行時にしか分からない値 —— 即値、
変数のアドレス、ジャンプ先 —— の場所が**穴**（hole）として空いています。
コード生成は、ステンシルをコピーして、その穴に実際の値を**パッチ**
（patch、埋める）するだけ。組み立てではなく**貼り絵**です。

> [!NOTE]
> 「コピーして穴を埋める」という操作には、聞き覚えがあるかもしれません。
> [リンカ](glossary.md)が複数のオブジェクトファイルをつなぐときに行う
> **再配置**（relocation）—— コードをコピーし、まだ決まっていなかった
> アドレスを後から埋める —— と、まったく同じ仕組みです。Copy-and-Patch は、
> リンカの再配置の技術を、JIT のコード生成へ持ち込んだものだと言えます。
> だから既存のコンパイラとリンカの仕組みをそのまま再利用でき、ステンシルの
> 生成を LLVM などに任せられるのが、実装上の大きな利点でした。

### 「穴」の正体は再配置スロットそのもの

「穴」をもう少し具体的に詰めておきましょう。穴とは、ステンシルの機械語
バイト列のうち、**実行時にしか決まらない値が入るべき場所**です。具体的には
次の 3 つが代表例です。

- **即値の穴**：`add eax, imm32` の `imm32` のように、コードに直接埋め込む
  4 バイトや 8 バイトの定数。私たちのおもちゃが扱ったのはこれです。
- **アドレスの穴**：変数の格納場所や、ヘルパー関数（ランタイム関数）の
  絶対アドレス。`call rel32` や `mov rax, imm64` の即値部に埋まります。
- **ジャンプ先の穴**：次に実行するステンシルや分岐先への相対オフセット。
  「継続の穴」とも呼びます（後述）。

ここで大事なのは、これらが ELF などのオブジェクトファイルが持つ
**再配置テーブル**（relocation table）のエントリと一対一で対応する、と
いう点です。コンパイラが `add eax, n` の `n` をまだ知らないまま出力すると、
そこを 0 で埋めたうえで「このバイト位置に、後で `n` という値を `R_X86_64_32`
形式で書き込め」という再配置エントリを残します。リンカやローダはそれを見て
穴を埋める。**Copy-and-Patch のステンシル生成は、まさにこの再配置エントリを
そのまま穴の一覧として流用します**。

```
# ELF オブジェクト中の add ステンシルが持つ再配置情報（概念図）
.text:  05 00 00 00 00      ; add eax, 0  ← 即値部を 0 で空けてある
.rela.text:
  offset=1  type=R_X86_64_32  symbol=PATCH_VALUE   ; ←「穴」1個
```

`offset=1` は「先頭から 1 バイト目（オペコード `05` の直後）から穴が
始まる」、`type` は「4 バイトをそのまま書く」、`symbol` は「埋めるべき値の
名前」を表します。Copy-and-Patch のビルド工程は、各ステンシルを 1 つの
小さな関数として clang でコンパイルし、できあがった `.o` から `.text`
（機械語本体）と `.rela.text`（穴の一覧）を抜き出して、**穴の位置・型・
何を埋めるかの表**に変換します。実行時の JIT は、その表を見ながらコピーと
穴埋めをするだけ。新しい再配置の仕組みを発明する必要はなく、何十年も
枯れたリンカの仕組みに**ただ乗り**できたことが、この技法を実装可能にした
最大の鍵でした。

> [!TIP]
> 「穴に何形式で書くか」は再配置タイプそのものです。`R_X86_64_32` なら
> 絶対 4 バイト、`R_X86_64_PC32` なら**現在位置からの相対** 4 バイト
> （`call`/`jmp` の `rel32` 向け）、`R_X86_64_64` なら絶対 8 バイト。
> 相対型の穴を埋めるときは「埋める値 − 穴のアドレス − 4」を書く、という
> リンカと同じ計算をします。おもちゃでは絶対の即値だけを扱いましたが、
> ジャンプの穴を足すなら、この相対計算が顔を出します。

## ステンシルとパッチを動かしてみる

[最小の JIT](hello-jit.md)で作った機械語生成の道具立てを使って、
Copy-and-Patch のミニチュアを作ります。題材は同じく「入力 `x` への
足す・引く・掛けるの列」です。各演算に対応する**ステンシル**を、即値の
部分を穴（`HOLE`）にして用意します。

```ruby
HOLE = :imm32    # 実行時に埋める 4 バイトの穴

# 各バイトコード命令に対応する、あらかじめ用意した機械語ステンシル
STENCILS = {
  prologue: [0x89, 0xF8],                          # mov  eax, edi（入力を eax へ）
  add:      [0x05, HOLE, HOLE, HOLE, HOLE],         # add  eax, imm32
  sub:      [0x2D, HOLE, HOLE, HOLE, HOLE],         # sub  eax, imm32
  mul:      [0x69, 0xC0, HOLE, HOLE, HOLE, HOLE],   # imul eax, eax, imm32
  epilogue: [0xC3],                                # ret
}

# ステンシルをコピーし、穴に即値を埋める（パッチ）
def patch(stencil, imm)
  immb = [imm].pack('l<').bytes   # 即値を 4 バイト（リトルエンディアン）に
  k = -1
  stencil.map { |b| b == HOLE ? immb[k += 1] : b }
end

# 命令列を「ステンシルを貼り、穴を埋める」だけでコンパイルする
def copy_and_patch(ops)
  code = STENCILS[:prologue].dup
  ops.each { |op, n| code.concat(patch(STENCILS[op], n)) }
  code.concat(STENCILS[:epilogue])
  code
end
```

`copy_and_patch` には、**オペコードを選ぶ判断も、命令を組み立てる作業も
ありません**。ただステンシルを連結し、`patch` で穴を埋めるだけです。
これを[最小の JIT](hello-jit.md)の `jit`（実行可能メモリへ置く関数）に
渡せば、本物の機械語として走ります。

```ruby
prog = [[:add, 1], [:mul, 3], [:sub, 2]]   # (x + 1) * 3 - 2
fn = Fiddle::Function.new(jit(copy_and_patch(prog)),
       [Fiddle::TYPE_INT], Fiddle::TYPE_INT)

p fn.call(10)   # => 31
p fn.call(5)    # => 16
```

ちゃんと `(10+1)*3-2 = 31`、`(5+1)*3-2 = 16` が出ます。[最小の JIT](hello-jit.md)の
`compile_expr` と結果は同じですが、**やっていることが違います**。
`compile_expr` は命令ごとに `case` で「どのバイトを出すか」を**判断して
組み立て**ていました。`copy_and_patch` は、判断済みのステンシルを
**コピーして穴を埋めるだけ**。この「判断を前倒しし、実行時はコピー＆
パッチに徹する」ことが、本物の Copy-and-Patch が桁違いに速くコードを
生成できる理由の縮図です。

> [!IMPORTANT]
> 速さの源泉は、**コンパイル時の仕事を、ステンシルを作る「事前」へ
> 移したこと**です。実行時に残るのは、メモリのコピーと数値の穴埋めだけ。
> これは CPU がいちばん得意とする単純作業です。Copy-and-Patch は
> 「コンパイルの賢さ」を捨てたのではなく、**賢い部分を実行時から追い出した**
> のです。

### 穴を「位置の表」として持つ：再配置テーブル風に書き直す

さきほどの `patch` は、ステンシルの中に `HOLE` という番兵を混ぜておき、
それを走査しながら埋めました。本物の Copy-and-Patch はこうではなく、
前節で見た**再配置テーブルそのもの**を持ちます。つまり機械語バイト列は
穴の場所を 0 で埋めた状態で確定させておき、「どのバイト位置に・何形式で・
何を埋めるか」を別表に分けるのです。おもちゃもこの形に寄せてみましょう。

```ruby
# ステンシルを「機械語本体」と「穴の一覧（再配置表風）」に分ける
Stencil = Struct.new(:code, :holes)
# holes の各要素 = [埋める位置(オフセット), 種類]
#   :imm  …… 4 バイト絶対値（即値）
#   :next …… 次のステンシルへの相対ジャンプ先（後述）

STENCILS = {
  prologue: Stencil.new([0x89, 0xF8], []),                 # mov eax, edi
  add: Stencil.new([0x05, 0,0,0,0],       [[1, :imm]]),     # add  eax, imm32
  sub: Stencil.new([0x2D, 0,0,0,0],       [[1, :imm]]),     # sub  eax, imm32
  mul: Stencil.new([0x69, 0xC0, 0,0,0,0], [[2, :imm]]),     # imul eax, eax, imm32
  epilogue: Stencil.new([0xC3], []),                        # ret
}

# 穴の位置に 4 バイト即値を書き込む（リトルエンディアン）
def patch32!(code, at, value)
  [value].pack('l<').bytes.each_with_index { |b, i| code[at + i] = b }
end

def copy_and_patch(ops)
  code = STENCILS[:prologue].code.dup
  ops.each do |op, n|
    st   = STENCILS[op]
    base = code.length          # このステンシルを貼る先頭位置
    code.concat(st.code.dup)    # 1) まるごとコピー
    st.holes.each do |at, kind| # 2) 穴を埋める
      patch32!(code, base + at, n) if kind == :imm
    end
  end
  code.concat(STENCILS[:epilogue].code)
  code
end
```

`base + at` が「貼った先での穴の絶対位置」です。番兵を探して回るのでなく、
**表に書かれた位置へ直接書き込む**ので、本物の実装に一歩近づきました。
結果は前と同じで、`copy_and_patch([[:add,1],[:mul,3],[:sub,2]])` を `jit`
へ渡せば `(x+1)*3-2` が走ります。やっていることは、[リンカ](glossary.md)が
`.o` を貼り合わせて再配置をかける作業と**完全に同型**だと分かります。

## ステンシルを数珠つなぎにする：継続の穴

おもちゃは演算を**直線**に並べただけでした。本物の Copy-and-Patch は、
分岐・ループ・関数呼び出しも扱えなければなりません。その鍵が、最初に
名前だけ挙げた**継続の穴**です。

各ステンシルの末尾に「次のステンシルの先頭へ飛ぶ」ジャンプを置き、その
飛び先を穴にします。すると、ステンシルを 1 個貼るたびに「直前のステンシルの
末尾ジャンプの穴」を、いま貼ったステンシルの先頭アドレスで埋められます。
こうしてステンシルが**数珠つなぎ**になります。これはちょうど、関数が処理を
終えたら戻らずに次の関数へ飛ぶ**末尾呼び出し**（tail call）の連鎖と同じ
形で、各バイトコードを「次のバイトコードへ末尾ジャンプする小関数」に
見立てる、いわば**継続渡し風**の構成です。

```ruby
# 末尾に「次へ飛ぶ」相対ジャンプ(E9 + rel32)を持たせたステンシル群。
# 穴 :imm  には即値を、穴 :next には「次のステンシル先頭への相対オフセット」を埋める
STENCILS_CHAINED = {
  add: Stencil.new(
    [0x05, 0,0,0,0,            # add eax, imm32   ← :imm の穴
     0xE9, 0,0,0,0],           # jmp rel32        ← :next の穴
    [[1, :imm], [6, :next]]),
  sub: Stencil.new(
    [0x2D, 0,0,0,0, 0xE9, 0,0,0,0],     # sub eax, imm32 ; jmp rel32
    [[1, :imm], [6, :next]]),
  mul: Stencil.new(
    [0x69, 0xC0, 0,0,0,0, 0xE9, 0,0,0,0],  # imul eax,eax,imm32 ; jmp rel32
    [[2, :imm], [7, :next]]),
}

# 相対ジャンプの穴を埋める：rel32 = 飛び先 - (穴の位置 + 4)
def patch_rel!(code, at, target)
  rel = target - (at + 4)     # PC 相対。リンカの R_X86_64_PC32 と同じ計算
  patch32!(code, at, rel)
end

def chain(ops)
  code, fixups = [0x89, 0xF8], []      # prologue（mov eax, edi）
  ops.each do |op, n|
    here = code.length                  # このステンシルの先頭
    fixups.each { |at| patch_rel!(code, at, here) }  # 直前の :next 穴を解決
    fixups = []
    st = STENCILS_CHAINED[op]
    code.concat(st.code.dup)
    st.holes.each do |at, kind|
      patch32!(code, here + at, n)        if kind == :imm
      fixups << (here + at)               if kind == :next  # まだ埋めない
    end
  end
  fixups.each { |at| patch_rel!(code, at, code.length) }  # 最後は ret へ
  code.concat([0xC3])
  code
end
```

末尾ジャンプの穴は「次が貼られて初めて飛び先が決まる」ので、`fixups` に
**未解決の穴の位置**を覚えておき、次のステンシルを貼った瞬間に埋めます。
これも、シンボルの定義が後から現れるのを待つリンカの**前方参照の解決**と
同じ段取りです。分岐は「真の継続」と「偽の継続」の 2 本の継続の穴を持つ
ステンシルを用意すれば表現でき、ループは継続の穴を**後ろ向き**に
（すでに貼ったステンシルの先頭へ）埋めれば作れます。

> [!NOTE]
> 「末尾ジャンプでつなぐ」構成には、もう 1 つ実装上のうまみがあります。
> 各ステンシルを**末尾呼び出しする独立した関数**として書けば、ステンシルの
> 機械語は clang が自動生成してくれます。Copy-and-Patch[](#cite:xu2021)も
> CPython 3.13[](#cite:cpython313jit)も、ステンシルを C で
> 「`return next_op(...)`」のように書き、`musttail`（末尾呼び出しの強制）を
> 付けて clang にコンパイルさせます。すると clang が末尾ジャンプ込みの
> 機械語を吐き、その `call`/`jmp` 先が自然に**継続の穴（再配置）**になる
> ——人手で機械語を書く部分がほぼ消えるのです。論文[](#cite:xu2021)では、
> 制御が親へ戻らず次の処理へ直接渡る**継続渡しスタイル**を採り、末尾呼び出しに
> なるその受け渡しを clang がジャンプ命令に落とします。さらに**全パラメータを
> レジスタで渡す GHC 呼び出し規約**（GHC＝Glasgow Haskell Compiler 由来で、
> LLVM が `ghccc` として備える規約）を使うことで、ステンシル間の呼び出し
> オーバーヘッドをほぼ消しています。

### レジスタ割り当てを「固定規約」で省く

ステンシルどうしを安全につなぐには、もう 1 つ約束が要ります。**あるステン
シルが値をどのレジスタに残し、次のステンシルがどのレジスタから受け取るか**
です。ふつうのコンパイラなら[レジスタ割り当て](ir.md)で解く問題ですが、
それは実行時にやるには重すぎます。

そこで Copy-and-Patch は**固定の呼び出し規約**を決めてしまいます。「スタック
トップは常に `rbx`、次は `r12`、ベースポインタは `r13`……」のように、論理
スタックの上位いくつかと特別な値を**特定のレジスタへ固定で割り当てる**。
すると各ステンシルは「自分はどのレジスタに何が入っている前提で動き、終わったら
何をどのレジスタに残すか」を**ビルド時に確定できる**ので、つなぐときに
辻褄合わせの命令を挟まずに済みます。私たちのおもちゃが `eax` 1 本だけを
使い、ステンシルをただ連結できたのは、まさにこの「固定規約」を極端に
単純化したものでした。値が `eax` に乗っている前提を全ステンシルが共有して
いるから、何も調整せずに貼れたのです。

> [!TIP]
> 固定規約は割り当てを省ける代わりに、レジスタの本数ぶんしか論理スタックの
> 上位を載せられません。あふれたぶんはメモリ（[スピル](ir.md)）になります。
> 本物では「スタックトップ 2 個までレジスタ、3 個目以降はメモリ」のような
> 浅い固定割り当てを使い、しかも**スタック深さごとに別のステンシルを
> 用意**することで、よくある浅い式は全部レジスタで回るよう調整します。
> これが「ステンシルを何種類も持つ」の正体の一つです。

## 本物のステンシルはもっと賢い

私たちのおもちゃのステンシルは、即値の穴しかありませんでした。本物の
Copy-and-Patch[](#cite:xu2021)のステンシルは、もっと多くの種類の穴を
持ちます。

- **値の穴**：即値や定数（私たちが見たもの）。
- **アドレスの穴**：変数やオブジェクトのメモリ上の位置。
- **継続の穴**：次に実行するステンシル（次の命令）への**ジャンプ先**。

特に「継続の穴」が重要です。これにより、ステンシルどうしを数珠つなぎに
でき、分岐や呼び出しも表現できます。さらに本物では、ステンシルを
**よく使う組み合わせ（変数がレジスタにある版、スタックにある版など）
ごとに何種類も用意**しておき、状況に合った版を選ぶことで、コピーするだけで
そこそこ質の良いコードになるよう工夫されています。ステンシルの一群は、
LLVM を使って**ビルド時に**生成されます。実行時に LLVM は動きません ——
だから速いのです。

具体的にどれくらい種類を増やすかを、数で感じてみましょう。元論文
[](#cite:xu2021)では、1 つの処理（たとえば「整数の加算」）に対し、

- オペランドが**即値か・レジスタか・スタックか**の組み合わせ、
- 結果を**どのレジスタ／スタック位置へ残すか**、
- よく使う**定数（0, 1 など）を即値の穴ではなくステンシルに焼き込んだ版**、

といった軸でステンシルを増やします。論文[](#cite:xu2021)では、各 AST ノード
について 1 つの**ステンシル生成器**を C++ で書きます。変種を表す
**テンプレートのメタ変数**（meta-variable）と、実行時に埋める穴を表す
特別なマクロを使い、**MetaVar** と呼ぶ仕組みがコンパイル時にメタ変数の
値を総当たりで回して、有効な組み合わせごとにテンプレートを実体化します。
こうして 1 つの処理から多数のステンシル変種がビルド時に自動生成されます。
「賢さ」はすべて、この事前生成されたステンシル表に蓄えられ、実行時の JIT
本体（コピーと穴埋めのループ）は対照的にごく小さく済みます。

ステンシルを選ぶ作業（どの版を貼るか）は、[中間表現](ir.md)で見た
[線形走査レジスタ割り当て](ir.md)[](#cite:poletto1999)よりもさらに軽い、
**ほぼ表引き 1 回**の判断です。「左オペランドはレジスタ、右は即値、結果は
スタックトップ」と決まれば、対応するステンシルが一意に引けます。本物の
コンパイラがレジスタ割り当てや命令選択で悩むところを、Copy-and-Patch は
**事前に列挙し尽くした表の中から選ぶ**ことで、選択そのものを定数時間に
してしまうのです。

## なぜ桁違いに速いのか：費用を概算する

「LLVM `-O0` の約 100 倍、高い最適化レベルの約 1000 倍速い」と最初に
書きました[](#cite:xu2021)。その差はどこから来るのか、ざっくりと費用を
見積もってみましょう。正確な数字は環境次第ですが、**桁の感覚**をつかむことが
目的です。

最適化コンパイラ（LLVM など[](#cite:lattner2004)）が 1 命令を機械語に
するまでには、おおまかに次の段階があります。

1. ソース／バイトコードを **IR** へ変換する。
2. IR の上で最適化パスを何十本も回す（[定数畳み込み](ir.md)、共通部分式
   削除、ループ最適化……）。
3. [レジスタ割り当て](ir.md)と命令選択をする。
4. 機械語へ**符号化**して出力する。

このうち時間を食うのは圧倒的に 2 と 3 です。LLVM が最適化込みでコードを
生成する速度は、ごく大まかに**毎秒数百キロ〜数メガバイト程度の機械語**
（言い換えると 1 命令あたり**数百ナノ秒〜マイクロ秒のオーダー**）です。
一方 Copy-and-Patch が実行時にやるのは、

1. バイトコードごとに**ステンシルを 1 つ選ぶ**（表引き、ほぼ定数時間）。
2. ステンシルの機械語を**コピーする**（`memcpy`、数十バイト）。
3. **穴を数個埋める**（4〜8 バイトの書き込みを 1〜数回）。

だけです。1 つのバイトコードあたり、せいぜい**数十バイトのコピーと数回の
書き込み**。現代の CPU にとって数十バイトの `memcpy` は**数ナノ秒**の
仕事です。穴埋めも 1 回数ナノ秒。合わせても 1 バイトコードあたり**十数〜
数十ナノ秒のオーダー**に収まります。

```
                     1 命令あたりのコード生成コスト（概算・桁の感覚）
  LLVM(最適化込み)   ≈ 数百 ns 〜 µs   ── IR・最適化・割り当て・符号化
  Copy-and-Patch     ≈ 十数〜数十 ns   ── 表引き + memcpy + 穴埋め
                       └─ LLVM -O0 比で約 100 倍、-O2 等の比で約 1000 倍
```

差を生む本質は、**事前に償却したか否か**です。最適化や符号化という重い
仕事を、Copy-and-Patch はステンシルを作る**ビルド時に一度だけ**払い、
出来上がりを表に貯めておく。実行時にはその成果を `memcpy` で貼るだけなので、
何度コンパイルしても重い仕事は二度と発生しません。これはちょうど、
[線形走査](ir.md)が「命令を一度なめるだけ」でグラフ彩色の重さを避けたのと
同じ精神を、**さらに極端に**推し進めたものです。線形走査は実行時に区間を
計算しますが、Copy-and-Patch は割り当ての結論まで**事前に焼き込んで**
しまうので、実行時に残るのは線形走査よりもなお軽い「コピーと書き込み」
だけになります。

> [!IMPORTANT]
> 上記の桁（2〜3 桁）は、あくまで**コード生成にかかる時間**の比です。
> 生成された機械語そのものの**実行速度**は別の話で、そちらは LLVM の
> 最適化コードに負けます（次節）。とはいえ論文[](#cite:xu2021)では、
> 生成コードは**インタプリタより約 10 倍速く、LLVM `-O0` 出力より 14% 速い**。
> WebAssembly 用コンパイラでは、Chrome のベースラインコンパイラ **Liftoff**
> より生成が **4.9〜6.5 倍速く**、生成コードの実行も Coremark・PolyBenchC で
> **39〜63% 速い**という結果も報告されています[](#cite:xu2021)。Copy-and-Patch
> が大きく勝っているのは「コードを作る速さ」、譲っているのは「最適化コンパイラの
> 磨き上げたコードの速さ」です。
> [ウォームアップ問題](research-warmup.md)が「作る速さ」を重視する場面で
> こそ、この技法が輝きます。

## トレードオフ：粒度が品質を縛る

ただより速いものはありません。Copy-and-Patch の速さには、はっきりした
代償があります。**生成されるコードの品質が、ステンシルの粒度に縛られる**
ことです。

ステンシルは「バイトコード 1 個ぶんの機械語断片」をあらかじめ固めたもの
です。だから最適化は**ステンシルの内側**でしか効きません。**命令と命令の
あいだ**にまたがる最適化 —— たとえば、

- 隣り合う 2 つの加算を 1 つの `lea` にまとめる、
- ある命令が `eax` に置いた値を、次の命令が**レジスタのまま**受け取って
  メモリを経由しない、
- 死んだ計算をまるごと消す、

といった**命令間の最適化**は、原理的にできません。ステンシルの境界が
最適化の壁になるのです。私たちのおもちゃで言えば、`(x+1)*3-2` を貼ると、
加算・乗算・減算のステンシルが**そのまま 3 個並ぶ**だけで、「`+1` のあと
`-2` だから実質 `-1` だ」のような畳み込みは起きません。本物の最適化
コンパイラなら一掃するはずの無駄が、ステンシルの継ぎ目に残ります。

固定の[呼び出し規約](hello-jit.md)も品質に響きます。レジスタの使い方を
全ステンシル共通に縛るぶん、その関数だけを見れば明らかに良い割り当てが
あっても、規約に従わざるを得ません。結果のコードは「悪くはないが、
最適コンパイラの磨き上げには遠い」あたりに落ち着きます。

> [!WARNING]
> 粒度を細かく（バイトコード 1 個ずつ）すれば組み合わせが減ってステンシル
> 表は小さく済みますが、命令間最適化の機会を多く逃します。逆に「2〜3 命令
> のよくある並びを 1 つの**スーパーステンシル**にまとめる」と質は上がり
> ますが、組み合わせ爆発でステンシル数が膨らみ、ビルド時間と配布サイズが
> 増えます。**粒度の選択は、コード品質とステンシル表の大きさのトレード
> オフ**です。Copy-and-Patch はこの曲線の「速くて軽い」端に寄せた設計だと
> 理解してください。

## なぜいま重要なのか：CPython 3.13 から

Copy-and-Patch が単なる面白い研究で終わらなかったのは、**CPython が
公式の JIT としてこれを採用した**からです[](#cite:cpython313jit)。

CPython（Python の標準実装）は長らく JIT を持ちませんでした。理由の一つは、
[フレームワーク](frameworks.md)で見たような重い JIT 基盤を、巨大で移植性を
重んじる CPython に組み込むのが難しかったことです。LLVM を実行時に
抱えるのは、依存も起動コストも大きすぎます。

Copy-and-Patch はこの制約に見事に噛み合いました。ステンシルは**ビルド時**に
LLVM で生成し、CPython の配布物には**生成済みのステンシル表**だけを
同梱する。実行時の JIT は、ステンシルをコピーして穴を埋めるごく小さな
コードで済みます。重い依存も、長いウォームアップもない。「[軽い JIT](research-warmup.md)が
欲しい」という、まさにウォームアップ問題が突きつけた要求に応える設計
でした。CPython の JIT の現状と位置づけは、[Python のケーススタディ](case-python.md)で
詳しく見ます。

具体的に、CPython 3.13 の JIT[](#cite:cpython313jit)はこうやってステンシルを
作っています。

1. CPython の**バイトコード（マイクロオペコード）1 つ 1 つを、C の小さな
   関数として**書く。これはインタプリタの命令実装をほぼそのまま流用した
   ものです。
2. ビルド時に、それらを `clang` で**末尾呼び出し付き**（`musttail`）に
   コンパイルし、各関数の機械語と再配置情報を取り出す。
3. 取り出した「機械語＋穴の一覧」を、`jit_stencils.h` という**C のヘッダ
   ファイル**として書き出し、CPython 本体へ埋め込む。

つまり配布されるバイナリには、ステンシル表が**ただの定数データ**として
入っているだけで、LLVM も clang も**実行時には一切要りません**。これが
「移植性を重んじる CPython に重い JIT 基盤は入れられない」という長年の
制約を、見事に回避した点です。ビルドに clang は要りますが、それは
CPython を**作る人**の手元の話で、Python を**使う人**には関係ありません。

この設計判断は、CPython がすでに持っていた仕掛けともよく噛み合いました。
CPython 3.11 以降は **PEP 659 の特殊化適応型インタプリタ**[](#cite:pep659)で、
実行時に「この `+` はいつも整数だ」と観測してバイトコードを特殊化版へ
差し替えています。その特殊化済み・型を絞り込んだバイトコード列は、
**ステンシルを貼る素材としてうってつけ**でした。JIT は新たに型推論を
やり直す必要がなく、特殊化が済んだ命令列を、対応するステンシルへ
ほぼ一対一で写すだけでよいのです。

> [!NOTE]
> CPython の JIT は「**コピー＆パッチ JIT**」とも呼ばれますが、当初は
> 性能向上そのものより、**JIT の足場を CPython に入れること**が主目的
> でした。Copy-and-Patch を選んだのは、生成コードが最速だからではなく、
> 「重い依存なし・短いウォームアップ・移植しやすい」という、まさに
> [ウォームアップ問題](research-warmup.md)が要求した条件を満たす
> [ベースライン段](tiered.md)を、最小の労力で用意できたからです。実際
> PEP 744[](#cite:cpython313jit)も「現状の JIT は既存の特殊化適応型
> インタプリタとほぼ同じ速さで、常時有効化が明確な勝ちとはまだ言えない」と
> 率直に認めています。だから 3.13 ではこの JIT は**既定でビルドに含まれず**、
> ビルド時の `--enable-experimental-jit` で**実験的に有効化**する位置づけで、
> 「実験段階のうちは本番では使うべきでなく、予告なく壊れたり削除され得る」と
> 明記されています[](#cite:cpython313jit)。Python 3.14 では macOS と Windows の
> 公式バイナリに実験的 JIT が含まれるようになりましたが、`PYTHON_JIT=1` で
> 明示的に有効化して試す段階で、本番利用は推奨されていません
> [](#cite:python314whatsnew)。

> [!TIP]
> Copy-and-Patch のコードは、徹底的に最適化された LLVM の出力には性能で
> 及びません。けれど、**インタプリタよりは確実に速く、生成は桁違いに
> 軽い**。[階層的 JIT](tiered.md)の言葉で言えば、これは理想的な
> **ベースライン段**です。「最高速度の一歩手前を、ほぼ無料のコンパイルで
> 手に入れる」—— この費用対効果が、Copy-and-Patch の真価です。

## まとめ

- **Copy-and-Patch**[](#cite:xu2021)は、命令ごとの機械語断片
  （**ステンシル**）をあらかじめ用意し、実行時は**コピーして穴を
  パッチするだけ**でコードを生成する技法。組み立てでなく貼り絵。
- **穴**＝即値・アドレス・ジャンプ先を後から埋める場所で、その正体は
  [リンカ](glossary.md)の**再配置スロット**そのもの。位置・型・埋める値を
  表に持ち、`R_X86_64_32`（絶対）や `R_X86_64_PC32`（相対）と同じ計算で
  埋める。既存のコンパイラ・リンカ技術を再利用でき、ステンシルは
  **ビルド時に** LLVM/clang で生成する（実行時に LLVM は動かない）。
- ステンシルどうしは末尾ジャンプの**継続の穴**で数珠つなぎにし、分岐・
  ループ・呼び出しを表現する。レジスタは**固定の呼び出し規約**で縛り、
  実行時のレジスタ割り当てを丸ごと省く。
- 速さの源泉は、**最適化・符号化という重い仕事を実行時から事前へ追い出した**
  こと。実行時に残るのは表引き＋`memcpy`＋穴埋めだけで、1 命令あたり
  十数〜数十ナノ秒のオーダー。論文[](#cite:xu2021)では LLVM `-O0` の約 100 倍、
  高い最適化レベルの約 1000 倍速く生成でき、[線形走査](ir.md)よりさらに軽い。
  生成コードもインタプリタの約 10 倍速く、LLVM `-O0` より 14% 速い。
- 代償は**命令間最適化ができない**こと。最適化はステンシルの内側に閉じ、
  品質は粒度に縛られる。粒度を粗くすれば質は上がるがステンシルが組み合わせ
  爆発する、というトレードオフ。最適化品質は LLVM に及ばないが、
  インタプリタより速く生成は桁違いに軽い理想的な**ベースライン段**。
- **CPython 3.13** が公式 JIT に採用[](#cite:cpython313jit)。バイトコードを
  C で書き `musttail` 付き clang でステンシル化し、`jit_stencils.h` として
  同梱。**PEP 659 の特殊化**[](#cite:pep659)済み命令列を素材にできるのも
  好都合だった。重い依存も長いウォームアップもない「軽い JIT」への要求に
  応えた。ただし現状は既存インタプリタとほぼ同速で、**既定では無効・実験的**
  （`--enable-experimental-jit` で有効化）な位置づけ[](#cite:cpython313jit)。
  Python 3.14 では公式 macOS/Windows バイナリに含まれるが、`PYTHON_JIT=1` で
  明示的に試す実験的機能のまま[](#cite:python314whatsnew)。

次は、投機的な JIT が「本当に元のプログラムと同じ意味を保っているか」を、
数学的に証明しようという、もっとも理論寄りの研究を見ます。
