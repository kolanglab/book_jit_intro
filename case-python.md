# ケーススタディ：Python（PyPy と CPython）

Python には、JIT への**二つの対照的な道**があります。一つは、別実装の
**PyPy** が選んだ[メタトレーシング](meta-tracing.md)。もう一つは、標準
実装の **CPython** が長年の慎重な歩みの末にたどり着いた、特殊化適応型
インタプリタと[Copy-and-Patch](research-copy-and-patch.md) JIT です。
同じ言語の二つの実装が、まったく違う戦略で速さを追う —— JIT の設計選択を
学ぶには絶好の対比です。

## PyPy：メタトレーシングの旗手

**PyPy** は、Python を Python（の制限された部分集合 RPython）で実装した
処理系です。[メタトレーシング](meta-tracing.md)の章で詳しく見たとおり、
PyPy の真価は、**ユーザの Python プログラムでなく、Python インタプリタ
自身をトレースする**ことにあります[](#cite:bolz2009)[](#cite:rigo2006)。

実装者は Python のインタプリタを RPython で書くだけ。RPython ツール
チェインが、それを C へ変換しつつ、**メタトレーシング JIT を自動で
組み込み**ます。ホットなループに来ると、インタプリタの 1 周をトレース
として記録し、[インタプリタのディスパッチや `pc` 管理を定数畳み込みで
消し去り](meta-tracing.md)、ユーザの計算だけが残った機械語を生成します。

結果として、PyPy は数値計算やループの重い処理で、CPython の何倍も
速く動きます。とりわけ「同じループをひたすら回す」たぐいのコードは、
[トレーシング JIT](tracing.md)の独壇場です。

### 例えばこうする：PyPy のインタプリタに置く具体的なヒント

「インタプリタを書くだけ」と言っても、良い JIT を引き出すには
[メタトレーシングの章](meta-tracing.md)で見た**ヒント**を、PyPy の
バイトコードインタプリタの要所に手で置く必要があります。具体的には、
こういう配置になります。

```python
# PyPy のバイトコードインタプリタの骨格（疑似コード、実 API ではない）
jitdriver = JitDriver(
    greens=['next_instr', 'pycode'],   # 命令位置とコードオブジェクト＝定数の鍵
    reds=['frame'],                    # フレーム（ローカル・スタック）＝流れる値
)

def dispatch(self, pycode, next_instr, frame):
    while True:
        jitdriver.jit_merge_point(            # ★ ディスパッチの先頭（合流点）
            next_instr=next_instr, pycode=pycode, frame=frame)
        opcode = ord(pycode.co_code[next_instr])
        if opcode == JUMP_ABSOLUTE:
            new_instr = self.decode_arg(pycode, next_instr)
            if new_instr < next_instr:        # 後ろ向き＝バックエッジ
                jitdriver.can_enter_jit(       # ★ ここでループの熱さを数える
                    next_instr=new_instr, pycode=pycode, frame=frame)
            next_instr = new_instr
        elif opcode == BINARY_ADD:
            w_y = frame.popvalue(); w_x = frame.popvalue()
            frame.pushvalue(self.add(w_x, w_y))
            next_instr += 1
        # ... 以下、各バイトコードの実装が続く
```

ここでの**設計判断のトレードオフ**を、PyPy 固有の事情で具体的に挙げます。

- **`pycode`（コードオブジェクト）を green にする**：同じ関数のバイト
  コードは実行中に書き換わらないので、定数の鍵にできます。これにより
  `co_code[next_instr]` のフェッチがトレース中に定数畳み込みされ、
  ディスパッチが消えます。もし green にし忘れると、トレースに巨大な
  `if/elif` のディスパッチがそのまま焼き付いてしまいます。
- **`frame` を red にする**：ローカル変数やスタックは毎周変わるので
  red です。そのうえで `frame` を `virtualizable` にしておくと、PyPy の
  トレーサはフレームオブジェクトの確保を消し、**ローカル変数を個別の
  値（できればレジスタ）としてばらして持ち**ます。Python の関数呼び出しは
  本来フレームをヒープに作る重い操作ですが、ホットなループ内ではこの
  確保がトレースから消え失せるのが PyPy の速さの大きな源です。
- **`w_x.class` を `promote` する**：`BINARY_ADD` の中身（上の `self.add`）
  では、被演算子の型（`W_IntObject` か `W_FloatObject` か…）を `promote`
  して「いまは整数」と決め打ちます。当たれば加算 1 命令に縮みますが、
  整数と浮動小数点数が交互に来る相手では[脱最適化](deopt-osr.md)が連発し、
  かえって遅くなる —— これが投機のトレードオフの実例です。

> [!TIP]
> PyPy は単一の「W_Object」を継ぎ足すのではなく、整数なら `W_IntObject`、
> 浮動小数点数なら `W_FloatObject` というように**型ごとに別クラスの
> ボックス**を持ちます。`promote(w_x.class)` でこの具体クラスが定数化
> されると、トレーサは仮想化（`virtualizable`）と組み合わせて
> **ボックスの確保すらトレースから消し**、生の機械整数だけが残った加算へ
> 縮められます。「型ごとに別クラス」というインタプリタの書き方が、
> そのまま JIT の質を決める好例です[](#cite:bolz2009)。

> [!NOTE]
> PyPy が示したのは、「[メタトレーシング](meta-tracing.md)は実用になる」
> という証明でした。同じ RPython 基盤から、Python だけでなく多数の言語の
> 高速処理系が派生しています。一方で、PyPy は CPython と C 拡張の互換性
> （NumPy など C で書かれたライブラリとの相性）に長く苦労してきました。
> 「速いが、エコシステムの重心を動かしきれない」というのは、別実装が
> 背負いがちな宿命でもあります。

## CPython：JIT なしで何十年も走ってきた

標準実装の **CPython** は、対照的に、長らく JIT を持たない素朴な
バイトコードインタプリタでした。理由はいくつもあります。巨大な C 拡張
エコシステム（CPython の内部構造に依存したライブラリ群）との互換性、
あらゆる OS・CPU への移植性、実装の単純さの価値。JIT という複雑で
重い機械を、この巨艦に組み込むのは容易ではありませんでした。

転機は、性能改善プロジェクト（しばしば "Faster CPython" と呼ばれます）の
始動です。CPython は、いきなり機械語 JIT へ飛ぶのでなく、**段階的に**
JIT へ近づく道を選びました。

## 第一歩：特殊化適応型インタプリタ（3.11）

CPython 3.11 が導入したのが、[プロファイリング](profiling.md)の章で
詳しく見た**特殊化適応型インタプリタ**（specializing adaptive
interpreter）[](#cite:pep659)です。機械語 JIT を持たずとも、

> バイトコード命令を、実行時に観測した型に特化した別命令へ**書き換える**

ことで、かなりの高速化を達成しました。汎用の `BINARY_OP` を、整数同士だと
観測したら `BINARY_OP_ADD_INT` へ差し替える、`LOAD_ATTR` を特定の
オブジェクトの形に特化する、といった具合です。これは[プロファイリングの
章で動かした quickening の玩具](profiling.md) —— 「命令が観測結果で自分
自身を書き換える」—— の、本物の実装です。

```python
# CPython 3.11+ の特殊化は dis で覗ける（adaptive 命令として現れる）
import dis
def add(a, b):
    return a + b

# 何度か実行してから逆アセンブルすると、BINARY_OP が特殊化された
# 命令（例: BINARY_OP_ADD_INT）に変わっている様子が観察できることがある
# （特殊化はしきい値や型の安定性に依存するため、常に見えるとは限らない）
for _ in range(100):
    add(1, 2)
dis.dis(add, adaptive=True)
```

機械語を一切吐かずに、**命令の差し替えだけ**でここまで速くする。これは
[コンパイルの単位](compilation-units.md)で「最も軽い単位」として挙げた
「命令単位の特殊化」の、堂々たる実例です。そして同時に、来たるべき
機械語 JIT のための**型フィードバックを集める下地**にもなりました。

### 例えばこうする：quickening の三段階

PEP 659[](#cite:pep659)の特殊化は、いきなり特化版へ飛ぶのではなく、
**ウォームアップを観測してから賭ける**という三段階を踏みます。具体的な
命令の遷移で追いましょう。`a + b` をコンパイルすると、まず汎用の
`BINARY_OP` が置かれます。

1. **適応的に観測する（quickening）**：`BINARY_OP` 自身が**適応的な命令**で、
   小さなカウンタを持ち、最初の数回は「来る型を観測して特化先を決める係」と
   して振る舞います（CPython には独立した `BINARY_OP_ADAPTIVE` 命令が
   あるわけではなく、汎用の `BINARY_OP` そのものが適応版を兼ねます）。
2. **特化する（specialize）**：適応的な `BINARY_OP` が数回実行され、被演算子が
   いつも整数だと観測できたら、その場所のバイトコードを `BINARY_OP_ADD_INT` へ
   書き換えます。`int` 同士の `float` への昇格やオーバーフローの分岐を
   省いた、整数専用の高速経路です。文字列なら `BINARY_OP_ADD_UNICODE`、
   浮動小数点数なら `BINARY_OP_ADD_FLOAT`、と特化先は型ごとに用意されて
   います。観測が安定せず特化先を決められない場所は、汎用の `BINARY_OP`
   のまま据え置かれます（賭けないのも一つの判断です）。
3. **脱最適化する（書き戻し）**：特化版 `BINARY_OP_ADD_INT` は、実行の
   冒頭で**ガード**（被演算子が本当に `int` か）を確かめます。外れたら、
   その場で汎用の `BINARY_OP` へ命令を書き戻して([プロファイリングの章で
   見た玩具](profiling.md)の `code[pc] = [:add]` そのものです)、汎用経路で
   処理を続けます。これが[脱最適化](deopt-osr.md)の、命令単位での最も軽い
   姿です。

> [!IMPORTANT]
> ここで効いているのが、すぐに脱最適化を最終判断にしない**カウンタ付きの
> 猶予**です。CPython は特化版の各場所に小さな「失敗カウンタ」を持たせ、
> ガードが**たまたま一度**外れただけでは汎用へ戻しません。何度も連続で
> 外れて初めて「この場所はもう整数では来ない」と見限り、書き戻します。
> 一度きりの型のゆらぎで特化を捨てると、すぐまた特化し直す**振動**
> （脱最適化と再特化の往復）でかえって遅くなる —— その回避策が、この
> 猶予つきの判定です。[ウォームアップ](research-warmup.md)で見た
> 「プロファイルの陳腐化」に対する、CPython の現実的な処方箋でもあります。

### 例えばこうする：インラインキャッシュをバイトコードに埋める

特化版が速いもう一つの理由は、[プロファイリングの章](profiling.md)で
見た**インラインキャッシュ**を、**バイトコード列そのものに埋め込んで**
いることです。CPython 3.11 のバイトコードは、各命令の直後に
**インラインキャッシュ用の空き語**（cache entries）を持つ可変長形式に
なりました。例えば `LOAD_ATTR` の特化版 `LOAD_ATTR_INSTANCE_VALUE` は、
直後のキャッシュ語に「前回観測したクラスの**バージョンタグ**」と
「属性が置かれているオブジェクト内の**オフセット**」を書き込みます。

```python
# LOAD_ATTR の特化（概念。実際は C で、命令直後のキャッシュ語を読む）
# point.x のような属性アクセスが、観測後にこう振る舞う：
def load_attr_instance_value(obj, cache):
    if obj.class_version == cache.version:   # ★ クラスの形が前回と同じか
        return obj.values[cache.offset]      #   ハッシュ探索なしで即取得
    else:
        deoptimize_to_generic_load_attr()    # 外れたら汎用の属性探索へ戻す
```

汎用の `LOAD_ATTR` は、オブジェクトの `__dict__` を辞書探索し、なければ
クラスの MRO（継承順）を辿る重い処理です。特化版は、**クラスの形が
前回と同じであるかぎり**、辞書探索をすべて省いて固定オフセットの 1 アクセスに
する。`LOAD_GLOBAL` も同様で、特化版はモジュール辞書とビルトイン辞書の
**バージョンが変わっていないか**だけを確かめ、変わっていなければキャッシュ
した値の位置を直接読みます。

> [!NOTE]
> 「クラスのバージョンタグ」は、属性やメソッドが書き換わるたびに増える
> 番号です。`point.__class__` にメソッドを追加したりすると版が上がり、
> 古い版を前提にした特化版のガードが外れて[脱最適化](deopt-osr.md)します。
> これは[メタトレーシングの `elidable` でメソッド探索を畳むとき](meta-tracing.md)に、
> 「クラス定義の変更で探索結果が変わるなら版を引数に含めよ」と注意した
> のと、**まったく同じ問題への同じ解**です。バージョンタグ 1 つで
> 「クラスの形が変わっていない」を安く確かめられるのが鍵です。

## 第二歩：Tier 2 とマイクロオペ（3.13、基盤は 3.12 の API）

次に CPython は、特殊化したバイトコードを、さらに細かい**マイクロ
オペレーション**（micro-ops、uops）の列へ展開する「第二ティア」の
仕組みを整えました。Python 3.12 では最適化器を差し込むための内部 API の
枠組みだけが用意され、実際にマイクロオペ列・Tier 2・機械語 JIT が動き出すのは
3.13 からです。一つのバイトコード命令を、より原始的な操作の列に
分解するのです。この細かい表現は、[中間表現](ir.md)の役割を果たし、
冗長なチェックの除去などの最適化や、機械語への変換の足場になります。

これは[階層的 JIT](tiered.md)の考え方そのものです。Tier 1（特殊化適応型
インタプリタ）で素早く動かしつつフィードバックを集め、熱い場所を Tier 2
（マイクロオペ列）に展開して最適化・機械語化する。CPython は、JVM や V8 が
歩んだ階層化の道を、自分のペースで歩み始めたのです。

### 例えばこうする：バイトコードをマイクロオペ列に割る

具体的に、特化済みのバイトコード 1 命令が、どんなマイクロオペの列に
割れるかを見ましょう。`BINARY_OP_ADD_INT` は、論理的には「両被演算子が
int であることを確かめ、加算し、結果を積む」という三つの原始操作です。
Tier 2 では、おおよそこう展開されます。

```
# BINARY_OP_ADD_INT（1 バイトコード）→ マイクロオペ列（uops）
_GUARD_BOTH_INT      # 左右が int か確かめる（外れたら脱最適化）
_BINARY_OP_ADD_INT   # 生の整数として加算
# （スタック操作は uop 列の中で暗黙に扱われる）
```

なぜわざわざ細かく割るのか。**冗長になったチェックを、列の上で消せる**
からです。例えばループ内で同じ変数を何度も足すコードでは、`_GUARD_BOTH_INT`
が連続して並びます。Tier 2 の最適化器は、列を眺めて「この変数は直前の
ガードで既に int だと確かめた。同じ変数への次のガードは要らない」と判断し、
**二度目以降のガードを削除**できます。バイトコードのままでは「1 命令の
内側」に閉じていたチェックが、マイクロオペに割ることで**命令の境界を
越えて**まとめて消せるようになる —— これが Tier 2 の中間表現としての
値打ちです。

> [!NOTE]
> CPython の Tier 2 は、熱いループの**実行経路を一直線につないだ列**
> （実行トレースに近いもの。CPython では「トレース」や「superblock」と
> 呼ばれます）として作られます。分岐は「片方の経路を本線にし、もう一方は
> ガードで抜ける」形になり、抜けた先は Tier 1 のバイトコードへ戻る ——
> これは[トレーシング JIT](tracing.md)で見た「一直線に焼いて、外れたら
> 脱最適化」と同じ構図です。CPython は機械語の出し方こそ Copy-and-Patch
> ですが、**何を機械語にするかの単位はトレース風**なのです。

## 第三歩：Copy-and-Patch JIT（3.13）

そしてついに、CPython 3.13 が**実験的な機械語 JIT** を載せました
[](#cite:cpython313jit)。その方式が、[研究の章で詳しく見た](research-copy-and-patch.md)
**Copy-and-Patch**[](#cite:xu2021)です。

なぜ Copy-and-Patch だったのか。それは、CPython の制約に完璧に噛み合った
からです。[Copy-and-Patch の章](research-copy-and-patch.md)で見たとおり、

- ステンシル（機械語の断片）は**ビルド時**に LLVM で生成され、配布物には
  生成済みのステンシル表だけが入る。**実行時に LLVM を抱えなくてよい。**
- 実行時の JIT は、Tier 2 のマイクロオペ列を、対応するステンシルを
  **コピーして穴を埋める**だけで機械語にする。コンパイルが極めて軽い。

巨大で移植性を重んじ、重い依存を嫌う CPython にとって、「重い基盤を
実行時に持たず、軽く機械語を出せる」Copy-and-Patch は理想的でした。
[ウォームアップ問題](research-warmup.md)が突きつけた「軽い JIT が欲しい」と
いう要求への、CPython なりの回答です。

### 例えばこうする：マイクロオペ 1 つに 1 ステンシル

CPython のステンシル方式の具体は、**マイクロオペと一対一**にした点に
あります。[Copy-and-Patch の章](research-copy-and-patch.md)で作った玩具では
バイトコード 1 命令にステンシル 1 つを対応させましたが、CPython は
Tier 2 の**マイクロオペ 1 つ**ごとにステンシルを 1 つ用意します。
`_GUARD_BOTH_INT` のステンシル、`_BINARY_OP_ADD_INT` のステンシル…という
具合に、ビルド時に LLVM（Clang）で各 uop の本体をコンパイルし、
**穴あきの機械語断片の表**として CPython のバイナリに同梱します。

実行時の JIT は、最適化済みのマイクロオペ列を順に見て、対応する
ステンシルを**コピーして連結し、穴を埋める**だけです。

```
# CPython 3.13 JIT のコード生成（概念）
最適化済み uop 列:  _GUARD_BOTH_INT, _BINARY_OP_ADD_INT, ...
                    │
   各 uop に対応するステンシルを表から引いて、
   ・即値の穴（定数・オペランド）を埋める
   ・継続の穴（次の uop のステンシルへの飛び先）を埋める
                    ↓
机上の機械語:        [GUARD のコード][ADD_INT のコード][...]  ← 連結するだけ
```

ここでの設計上の工夫が二つあります。**第一に、ステンシルの末尾を次の
uop へジャンプする「継続の穴」にしておく**こと。これでステンシルを数珠
つなぎにでき、列を上から貼っていくだけで一本の機械語になります。
**第二に、よく続く組み合わせをまとめた「super-instruction」のステンシルを
用意する**こと。`_GUARD_BOTH_INT` の直後にほぼ必ず `_BINARY_OP_ADD_INT`
が来るなら、その二つをくっつけた専用ステンシルを 1 つ持っておけば、
継続のジャンプも省けて、コピー量も減ります。

> [!TIP]
> なぜ「最高性能の LLVM-JIT」でなく、あえて性能で劣る Copy-and-Patch
> なのか。答えは[ウォームアップの章](research-warmup.md)にあります。
> Python は短命なスクリプトの実行も多く、起動とコンパイルが遅いと
> **JIT した利得を取り戻す前にプロセスが終わって**しまいます。
> Copy-and-Patch は生成が桁違いに軽いので、暖まるのが速く、短命な実行でも
> 損をしにくい。「ピーク性能を少し諦めて、ウォームアップの軽さを買う」
> という[階層的 JIT のベースライン段](tiered.md)としての割り切りが、
> CPython の事情にぴたりと合ったわけです[](#cite:cpython313jit)。

> [!WARNING]
> CPython 3.13 の JIT は**既定では無効の実験的機能**です。Python 3.14 では
> macOS と Windows の公式バイナリに実験的 JIT が含まれ、`PYTHON_JIT=1` で
> 試せるようになりましたが、公式ドキュメントでも本番利用は推奨されておらず、
> 典型的な性能影響はワークロードによって遅くも速くもなり得るとされています
> [](#cite:python314whatsnew)。性能は段階的に
> 改善中で、現時点では「JIT を有効にすれば常に速い」とは限りません。
> Tier 2 のマイクロオペ最適化器が育つほど効いてくる設計なので、真価を
> 発揮するのはこれからです。本書が描くのは到達点ではなく、
> **「巨艦に軽い JIT を載せる」という設計選択の現在地**です。

> [!IMPORTANT]
> CPython の歩みは、JIT 導入の**模範的な漸進**です。いきなり機械語 JIT へ
> 飛ぶのでなく、(1) 命令の[特殊化](profiling.md)で型フィードバックの
> 仕組みを整え、(2) [マイクロオペ列という中間表現](ir.md)を用意し、
> (3) その上に[軽量な機械語 JIT](research-copy-and-patch.md)を載せる。
> 各段が次の段の土台になっています。「巨大な既存処理系に、互換性を
> 壊さず JIT を入れる」ことの、教科書のような道のりです。

## 二つの道の対比

Python の二実装は、JIT 設計の二つの哲学を体現しています。

| | PyPy | CPython |
|---|------|---------|
| 出発点 | ゼロから別実装 | 既存の巨艦を改造 |
| JIT 方式 | [メタトレーシング](meta-tracing.md) | 特殊化＋[Copy-and-Patch](research-copy-and-patch.md) |
| 強み | ピーク性能（ループ） | 互換性・移植性・漸進性 |
| 弱み | C 拡張互換性 | ピーク性能は PyPy に及ばない |

どちらが「正しい」というものではありません。**ゼロから設計し直せば
メタトレーシングで高い性能が狙え、既存資産を守るなら漸進的に軽い JIT を
積み上げる。** 制約が設計を決める、という本書のくり返しのテーマが、
ここでも鮮やかに見えます。

## 余談：GIL と JIT

CPython には長らく **GIL**（Global Interpreter Lock、グローバル
インタプリタロック）があり、一度に一つのスレッドしか Python の
バイトコードを実行できません。JVM や V8 のような処理系は、コンパイラ
スレッドを別に走らせ、実行を止めずに機械語を用意して、できたら差し替える
という[階層的 JIT](tiered.md)の定石をとります。

一方 CPython 3.13 の Copy-and-Patch JIT は、そうした並行コンパイル用の
専用スレッドを持たず、**実行スレッド上で同期的に**機械語を生成します。
コンパイル自体が極めて軽いので、実行スレッドが片手間に貼り絵を済ませても
止まる時間がごくわずかで済む —— ここでも「軽い JIT」という選択が効いて
います。GIL 下では、命令の特殊化やインラインキャッシュの書き換えも一度に
一スレッドしか進めないため、書き換えの競合はそもそも起きません。

問題になるのは、GIL を撤廃する **free-threaded ビルド**（実験的）です。
複数スレッドが同じバイトコードを同時に実行しうるため、特化命令の
インラインキャッシュ書き換えを複数スレッドから安全に行う方法が、これからの
課題になります。

## まとめ

- Python には JIT への二つの道がある。
- **PyPy** は[メタトレーシング](meta-tracing.md)（RPython でインタプリタを
  書くと JIT が自動生成）。インタプリタの要所に `jit_merge_point`・
  `virtualizable`・`promote` を置くことで、フレーム確保やボックスの確保まで
  トレースから消える。ループの重い処理で CPython を大きく上回るが、
  C 拡張互換性に課題。
- **CPython** は漸進的に JIT へ近づいた：(1) 3.11 の[特殊化適応型
  インタプリタ](profiling.md)で `BINARY_OP`→`BINARY_OP_ADD_INT` のように
  命令を観測型へ書き換え（適応版→特化版→外れたら書き戻し）、インライン
  キャッシュをバイトコードに埋め、(2) [マイクロオペ列の中間表現](ir.md)に
  割って冗長なガードを消し、(3) 3.13 で[Copy-and-Patch](research-copy-and-patch.md)
  の機械語 JIT を実験導入（uop ごとに 1 ステンシルを貼る）。
- 「ゼロから設計（PyPy）」対「巨艦の漸進改造（CPython）」は、JIT 設計の
  哲学の対比そのもの。

見ます。
