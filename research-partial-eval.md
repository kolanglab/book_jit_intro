# 部分評価と Futamura 射影

[メタトレーシング](meta-tracing.md)の章で、「インタプリタを書くだけで
JIT が手に入る」という驚きの技法を見ました。その背後には、もっと古く、
もっと一般的な理論があります。**部分評価**（partial evaluation）と、
そこから導かれる **Futamura 射影**（Futamura projections）です。

この理論は 1971 年、日本の研究者・二村良彦によって提示されました
[](#cite:futamura1971)。そして 40 年以上を経て、**Truffle/GraalVM** という
形で実用の JIT として結実しました[](#cite:wurthinger2017)。この章では、
理論から実装までの長い橋を渡します。

## 部分評価とは何か

部分評価のアイデアは、拍子抜けするほど単純です。**二つの入力を取る
プログラムに、片方の入力だけを先に与えると、残りの入力だけを取る、より
速いプログラムが得られる。**

例で見ましょう。「`base` の `exp` 乗」を計算する関数です。

```ruby
def power(base, exp)
  r = 1
  exp.times { r *= base }
  r
end

p power(2, 5)   # => 32
```

この `power` は二つの入力 `base` と `exp` を取ります。ここで、`exp` だけが
**5 だと先に分かっている**としましょう。すると、ループを回す必要は
もうありません。「`base` を 5 回掛ける」と分かっているのだから、その
ぶんだけ展開した、`base` だけを取る関数を**生成**できます。

```ruby
# power を exp について部分評価する：exp を固定し、base だけ取る残余プログラムを作る
def specialize_power(exp)
  factors = ["1"] + Array.new(exp, "base")
  "->(base) { #{factors.join(' * ')} }"
end

src = specialize_power(5)
puts src                # => ->(base) { 1 * base * base * base * base * base }
pow5 = eval(src)
p pow5.(2)              # => 32
p pow5.(3)              # => 243
```

`specialize_power(5)` は、`exp=5` という既知の入力を焼き込んで、
`1 * base * base * base * base * base` という**ループのない残余プログラム**
（residual program）を生成しました。元の `power` にあった `exp.times` の
ループ管理が消え、`base` だけの一直線の式になっています。これが部分評価
です。「既知の入力で計算できるところは先に計算してしまい、未知の入力
だけが残る、より速いプログラムを吐く」。

> [!NOTE]
> 部分評価器（上の `specialize_power` のような、プログラムを特殊化する
> プログラム）を一般に **mix** と書く伝統があります。mix は「プログラムと、
> その入力の一部」を受け取り、「特殊化されたプログラム」を返します。
> ここでは `power` 専用の素朴な mix を手書きしましたが、理論上の mix は
> **任意のプログラム**を特殊化できる汎用の道具です。

## ここで、はっとする：インタプリタを部分評価すると？

さて、二村の閃きはここからです。**特殊化する対象を「インタプリタ」に
してみたら、何が起こるか。**

インタプリタは、二つの入力を取るプログラムだと見なせます。

```
interpreter(プログラムのソース, 実行時の入力) → 実行結果
```

第一の入力は「実行すべきプログラムのソース」、第二の入力は「そのプログラムに
与える実行時データ」です。ここで、**第一の入力（プログラムのソース）だけを
先に固定して**、インタプリタを部分評価したら、何が得られるでしょうか。

得られるのは、「**そのプログラム専用の、実行時データだけを取るプログラム**」
です。インタプリタのディスパッチ（`case insn[0]`）や `pc` の管理は、
プログラムのソースが固定なら**先に計算できる**ので、部分評価で消えます。
[メタトレーシング](meta-tracing.md)で「インタプリタの `pc` は定数畳み込みで
消える」と言ったのと、まったく同じことが起きるのです。残るのは、その
プログラムがやりたい計算だけ。

> [!IMPORTANT]
> これが衝撃の結論です。**インタプリタを、あるソースプログラムについて
> 部分評価すると、そのソースプログラムをコンパイルした結果が得られる。**
> インタプリタ ＋ 部分評価 ＝ コンパイラ。コンパイラを書かなくても、
> インタプリタさえあれば、部分評価がコンパイルをやってくれるのです。

## Futamura 射影：三段ロケット

この洞察を整理したものが、**Futamura 射影**です。記号で書くと
分かりやすいので、`mix` を部分評価器、`interp` をインタプリタ、`source` を
ソースプログラムとします。

**第一射影**：インタプリタをソースについて部分評価すると、コンパイル済み
プログラム（実行ファイルに相当）が得られる。

```
target = mix(interp, source)        # source をコンパイルした結果
```

**第二射影**：部分評価器を「インタプリタについて」部分評価すると、
**コンパイラ**そのものが得られる。

```
compiler = mix(mix, interp)         # source を入れると target を吐く道具
```

**第三射影**：部分評価器を「部分評価器について」部分評価すると、
**コンパイラ生成器**（インタプリタを入れるとコンパイラが出てくる道具）が
得られる。

```
cogen = mix(mix, mix)               # interp を入れると compiler が出る道具
```

第一射影が、いま私たちが見た「インタプリタを部分評価＝コンパイル」です。
第二・第三射影は理論的な美しさで知られますが、本書の実用上は**第一射影**が
すべての出発点です。「インタプリタを部分評価すれば、コンパイラを書かずに
コンパイル結果が手に入る」—— これを実際の JIT に落とし込めるか。それが
40 年越しの課題でした。

## 理論から実装へ：Truffle と Graal

長らく、Futamura 射影は「理論的には美しいが、実用の高性能処理系には
応用しにくい」と思われていました。素朴に部分評価すると、コードが爆発的に
膨らんだり、動的言語の型の動きをうまく畳めなかったりするのです。

これを実用の JIT として成立させたのが、Oracle Labs の **Truffle/GraalVM**
です[](#cite:wurthinger2013) [](#cite:wurthinger2017)。仕組みはこうです。

1. 言語実装者は、対象言語の **AST インタプリタ**（構文木を直接たどって
   実行するインタプリタ）を、Truffle のフレームワーク上で Java で書く。
2. 各 AST ノードは、[プロファイリング](profiling.md)で見た要領で、実行
   しながら**自分自身を観測した型に特殊化**する（自己最適化 AST
   [](#cite:wurthinger2012)）。たとえば加算ノードが「整数同士だった」と
   観測したら、整数加算版のノードに化ける。
3. AST が安定したら（特殊化が落ち着いたら）、Graal コンパイラがその AST を
   **部分評価**して、メソッド単位の最適化機械語を生成する。

ここが巧妙な点です。素朴に「全インタプリタを部分評価」するのでなく、
**実行時の観測で AST を特殊化してから**部分評価する。型が観測で確定して
いるので、部分評価がうまく畳めて、コードも膨らみません。観測（投機）と
部分評価を組み合わせたのが、Truffle の「実用的な部分評価」
[](#cite:wurthinger2017)の核心です。

> [!NOTE]
> [メタトレーシング](meta-tracing.md)（PyPy）と Truffle は、どちらも
> 「インタプリタを書くだけで JIT を得る」点で同じ目標を持ちますが、
> 手段が違います。PyPy は**インタプリタをトレースする**（トレーシング
> JIT を自動生成）。Truffle は**インタプリタを部分評価する**（メソッド
> JIT を自動生成）。[コンパイルの単位](compilation-units.md)で見た
> トレーシング対メソッドの対立が、フレームワークのレベルでも現れている
> のです。

## 投機もデオプトも、ちゃんとある

部分評価で生成した機械語も、観測した型を**投機**しています。だから、
[デオプティマイズ](deopt-osr.md)の安全網が当然必要です。Truffle/Graal は、
特殊化した AST ノードの前提が実行時に崩れたら（観測と違う型が来たら）、
デオプティマイズして AST インタプリタへ戻り、ノードをより一般的な版に
作り直してから、また部分評価し直します。[階層的 JIT](tiered.md)の昇格・
降格が、AST の特殊化・脱特殊化として現れているわけです。

このように、Truffle は本書の第 I 部で見た技法 —— 自己最適化（プロファイル
による特殊化）、投機、ガード、デオプティマイズ —— を、**部分評価という
一本の背骨**の上に統合した処理系です。実際に TruffleRuby（Ruby）・
GraalJS（JavaScript）・GraalPy（Python）など、多数の言語処理系がこの
基盤の上で高い性能を出しています。[ケーススタディ](case-others.md)で
再訪します。

## まとめ

- **部分評価**は、二入力プログラムの片方の入力を先に固定し、残りの入力
  だけを取る、より速い**残余プログラム**を生成する技法。
- **Futamura の第一射影**：インタプリタをソースについて部分評価すると、
  そのソースのコンパイル結果が得られる。**インタプリタ＋部分評価＝
  コンパイル**。
- 第二・第三射影は、コンパイラやコンパイラ生成器を生む理論的拡張。
- 40 年後、**Truffle/GraalVM** が実用化。**自己最適化 AST**（観測で型を
  特殊化）してから**部分評価**することで、コード爆発を抑えた高性能 JIT を
  実現した。
- 投機・ガード・デオプティマイズも備え、第 I 部の技法を部分評価の背骨で
  統合している。
- PyPy（メタトレーシング）と Truffle（部分評価）は同じ目標・別手段。

次は、JIT の「速くなるまでの時間」—— ウォームアップそのものを研究対象に
した、ベンチマークの常識を揺るがした仕事を見ます。
