# 部分評価と Futamura 射影

[メタトレーシング](meta-tracing.md)の章で、「インタプリタを書くだけで
JIT が手に入る」という驚きの技法を見ました。その背後には、もっと古く、
もっと一般的な理論があります。**部分評価**（partial evaluation）と、
そこから導かれる **Futamura 射影**（Futamura projections）です。

この理論は 1971 年、日本の研究者・二村良彦によって提示されました
[](#cite:futamura1971)。そして 40 年以上を経て、**Truffle/GraalVM** という
形で実用の JIT として結実しました[](#cite:wurthinger2017)。この章では、
理論から実装までの長い橋を渡します。

## 部分評価とは何か

部分評価のアイデアは、拍子抜けするほど単純です。**二つの入力を取る
プログラムに、片方の入力だけを先に与えると、残りの入力だけを取る、より
速いプログラムが得られる。**

例で見ましょう。「`base` の `exp` 乗」を計算する関数です。

```ruby
def power(base, exp)
  r = 1
  exp.times { r *= base }
  r
end

p power(2, 5)   # => 32
```

この `power` は二つの入力 `base` と `exp` を取ります。ここで、`exp` だけが
**5 だと先に分かっている**としましょう。すると、ループを回す必要は
もうありません。「`base` を 5 回掛ける」と分かっているのだから、その
ぶんだけ展開した、`base` だけを取る関数を**生成**できます。

```ruby
# power を exp について部分評価する：exp を固定し、base だけ取る残余プログラムを作る
def specialize_power(exp)
  factors = ["1"] + Array.new(exp, "base")
  "->(base) { #{factors.join(' * ')} }"
end

src = specialize_power(5)
puts src                # => ->(base) { 1 * base * base * base * base * base }
pow5 = eval(src)
p pow5.(2)              # => 32
p pow5.(3)              # => 243
```

`specialize_power(5)` は、`exp=5` という既知の入力を焼き込んで、
`1 * base * base * base * base * base` という**ループのない残余プログラム**
（residual program）を生成しました。元の `power` にあった `exp.times` の
ループ管理が消え、`base` だけの一直線の式になっています。これが部分評価
です。「既知の入力で計算できるところは先に計算してしまい、未知の入力
だけが残る、より速いプログラムを吐く」。

別の値で固定すれば、別の残余プログラムが出てきます。`exp=3` を固定
すれば、得られるのは `base * base * base`、つまり「`base` を 3 回掛ける」
だけの式です。

```ruby
pow3 = eval(specialize_power(3))   # ->(base) { 1 * base * base * base }
p pow3.(2)   # => 8
p pow3.(10)  # => 1000
```

ここで起きていることを、もう一段かみ砕きましょう。元の `power` には
**二種類の計算**が混ざっていました。

- `exp` だけで決まる計算 —— 「何回ループを回すか」「ループを続けるか
  終わるか」という**制御**。
- `base` が来ないと決まらない計算 —— 「実際に掛け算をする」という
  **データ処理**。

部分評価は、この二つを**分離**します。`exp` を固定した瞬間、前者
（ループ回数・継続判定・カウンタ操作）は全部「先に計算できる」側に
回り、コンパイル時に消えてなくなります。後者（掛け算そのもの）だけが
残余プログラムに残る。これを専門的には、入力を **static**（既知・先に
計算できる）と **dynamic**（未知・残す）の二色に塗り分ける、と言います。

> [!TIP]
> 部分評価器を自作するときの最初の作業は、この **binding-time analysis**
> （束縛時刻解析）—— 各変数・各式が static か dynamic かを判定する処理
> —— です。`exp` が static なら、`exp.times` の `times` も、ループ
> カウンタも、継続判定 `i < exp` も芋づる式に static になります。
> 逆に dynamic な値が一滴でも混じった式は dynamic に「伝染」します。
> 「どこまでを先に計算できるか」は、この伝染をどこで食い止めるかの
> 設計判断そのものです。

> [!NOTE]
> 部分評価器（上の `specialize_power` のような、プログラムを特殊化する
> プログラム）を一般に **mix** と書く伝統があります。mix は「プログラムと、
> その入力の一部」を受け取り、「特殊化されたプログラム」を返します。
> ここでは `power` 専用の素朴な mix を手書きしましたが、理論上の mix は
> **任意のプログラム**を特殊化できる汎用の道具です。

### 素朴な mix を「式を畳む」方向で書いてみる

文字列連結で残余プログラムを作るのは、説明には便利ですが、本物の
部分評価とは少し違います。本物の mix は、**プログラムの構造（AST や
バイトコード）をたどりながら、static な部分を実際に計算し、dynamic な
部分だけを残った式として書き出します**。おもちゃで雰囲気を見ましょう。
小さな式の木を、環境 `env`（static に分かっている変数の値）のもとで
部分評価する `peval` です。

```ruby
# 式は [演算子, 左, 右] か、[:var, 名前] か、リテラル整数
def peval(expr, env)
  case expr
  in Integer => n                       # リテラルは static
    n
  in [:var, name]
    env.key?(name) ? env[name] : expr   # 既知なら値に、未知なら式のまま残す
  in [:mul, a, b]
    pa, pb = peval(a, env), peval(b, env)
    if pa.is_a?(Integer) && pb.is_a?(Integer)
      pa * pb                           # 両辺 static → その場で掛けてしまう
    elsif pa == 1 then pb               # 1 * x → x の簡約（残余を小さく）
    elsif pb == 1 then pa
    else [:mul, pa, pb]                 # dynamic が残る → 掛け算を残す
    end
  end
end

# (2 * 3) * base を、base 未知のまま部分評価する
expr = [:mul, [:mul, 2, 3], [:var, :base]]
p peval(expr, {})        # => [:mul, 6, [:var, :base]]   2*3 は 6 に畳まれた
```

`2 * 3` は両辺が static なので、その場で `6` に**畳み込まれ**ました。
一方 `base` は env にないので dynamic と判定され、掛け算ごと残ります。
さらに `1 * x → x` の簡約も入れてあります。これが**定数畳み込み**と
**簡約**を部分評価器の内側で行うということです。残余プログラムが
小さくなるかどうかは、この簡約規則をどれだけ持つかに直結します。

> [!WARNING]
> 素朴な mix には**コード爆発**の罠があります。たとえば `exp` を
> static として `power` を展開すると、`exp=1000` のときは掛け算が
> 1000 個並んだ巨大な残余プログラムが出ます。「ループを展開すれば
> 速い」は、展開しすぎれば命令キャッシュを溢れさせ、かえって遅く
> なります。**どこまで static 扱いで展開し、どこからループのまま
> dynamic に残すか**が、部分評価を実用化するうえで最大のトレードオフ
> です。後で見る Truffle は、この線引きを「実行時の観測」で動的に
> 決めることで切り抜けます。

## ここで、はっとする：インタプリタを部分評価すると？

さて、二村の閃きはここからです。**特殊化する対象を「インタプリタ」に
してみたら、何が起こるか。**

インタプリタは、二つの入力を取るプログラムだと見なせます。

```
interpreter(プログラムのソース, 実行時の入力) → 実行結果
```

第一の入力は「実行すべきプログラムのソース」、第二の入力は「そのプログラムに
与える実行時データ」です。ここで、**第一の入力（プログラムのソース）だけを
先に固定して**、インタプリタを部分評価したら、何が得られるでしょうか。

得られるのは、「**そのプログラム専用の、実行時データだけを取るプログラム**」
です。インタプリタのディスパッチ（`case insn[0]`）や `pc` の管理は、
プログラムのソースが固定なら**先に計算できる**ので、部分評価で消えます。
[メタトレーシング](meta-tracing.md)で「インタプリタの `pc` は定数畳み込みで
消える」と言ったのと、まったく同じことが起きるのです。残るのは、その
プログラムがやりたい計算だけ。

具体で見ましょう。小さなスタックマシンのインタプリタを考えます。

```ruby
def interp(prog, input)        # prog=命令列(static), input=実行時データ(dynamic)
  stack = []
  pc = 0
  while pc < prog.size
    insn = prog[pc]
    case insn[0]               # ← ディスパッチ
    when :push then stack.push(insn[1])
    when :arg  then stack.push(input)
    when :add  then stack.push(stack.pop + stack.pop)
    when :mul  then b, a = stack.pop, stack.pop; stack.push(a * b)
    end
    pc += 1
  end
  stack.pop
end

# 「input に 1 を足して 2 倍する」プログラム
prog = [[:arg], [:push, 1], [:add], [:push, 2], [:mul]]
p interp(prog, 10)   # => 22
```

ここで `prog` を **static として固定**し、`input` だけ dynamic として
部分評価すると、何が起こるか。`while` の各周回で `pc` も `insn` も
`case` の分岐先も、すべて `prog` だけで決まる **static** な値です。
だからループは完全に展開され、`case` のディスパッチは「どの枝に行くか」が
コンパイル時に確定して消えます。手で部分評価をたどると、こうなります。

```ruby
# interp(prog, ・) を prog について部分評価した結果（手で畳んだもの）
->(input) { (input + 1) * 2 }
```

`while`・`pc`・`case`・`stack` といったインタプリタの足回りはすべて
static として畳み込まれ、残ったのは `prog` が表現していた計算
`(input + 1) * 2` そのものだけ。これが**インタプリタの部分評価＝
コンパイル**の実物です。

> [!NOTE]
> [メタトレーシング](meta-tracing.md)が同じ結果を出す経路と比べると
> 対照的です。メタトレーシングは、インタプリタを**実際に走らせて**
> ディスパッチループの軌跡を記録し、トレース上で `pc` や `case` を
> 定数畳み込みします。部分評価は、走らせずに**プログラムの構造を解析
> して**同じ畳み込みをします。「走らせて消す」のがトレース、
> 「解析して消す」のが部分評価。出てくる機械語の理想形は似ていますが、
> 単位がトレース（一本道）かメソッド（制御構造ごと）かで分かれます
> （[コンパイルの単位](compilation-units.md)）。

> [!IMPORTANT]
> これが衝撃の結論です。**インタプリタを、あるソースプログラムについて
> 部分評価すると、そのソースプログラムをコンパイルした結果が得られる。**
> インタプリタ ＋ 部分評価 ＝ コンパイラ。コンパイラを書かなくても、
> インタプリタさえあれば、部分評価がコンパイルをやってくれるのです。

## Futamura 射影：三段ロケット

この洞察を整理したものが、**Futamura 射影**です。記号で書くと
分かりやすいので、`mix` を部分評価器、`interp` をインタプリタ、`source` を
ソースプログラムとします。

**第一射影**：インタプリタをソースについて部分評価すると、コンパイル済み
プログラム（実行ファイルに相当）が得られる。

```
target = mix(interp, source)        # source をコンパイルした結果
```

**第二射影**：部分評価器を「インタプリタについて」部分評価すると、
**コンパイラ**そのものが得られる。

```
compiler = mix(mix, interp)         # source を入れると target を吐く道具
```

**第三射影**：部分評価器を「部分評価器について」部分評価すると、
**コンパイラ生成器**（インタプリタを入れるとコンパイラが出てくる道具）が
得られる。

```
cogen = mix(mix, mix)               # interp を入れると compiler が出る道具
```

第一射影が、いま私たちが見た「インタプリタを部分評価＝コンパイル」です。
第二・第三射影は理論的な美しさで知られますが、本書の実用上は**第一射影**が
すべての出発点です。「インタプリタを部分評価すれば、コンパイラを書かずに
コンパイル結果が手に入る」—— これを実際の JIT に落とし込めるか。それが
40 年越しの課題でした。

### 三段ロケットを、もう一段かみ砕く

記号だけだと飲み込みにくいので、各射影を「何を固定して何を入れると何が
出るか」で言い換えます。鍵は、**mix も二入力プログラムだ**（プログラムと
その入力の一部を取る）という点です。だから mix 自身を mix にかけられます。

第一射影は、`interp` という二入力プログラムの**第一引数 `source` を固定**
した残余を作る、ただの部分評価です。

```
target  = mix(interp, source)
# 「source 入りインタプリタ」を input だけ取る形に畳んだもの。
# これは source の機械語、つまりコンパイル結果。
```

第二射影は、一段ずらします。「`mix(interp, ・)` という処理」自体を、
**`interp` を固定して**畳むのです。`mix` の第一引数は「特殊化したい
プログラム」、第二引数は「焼き込む入力」。そこに `interp` を焼き込むと、
残るのは「`source` を入れると `target` を吐く処理」—— すなわち
**コンパイラ**です。

```
compiler = mix(mix, interp)
# 残った dynamic 入力は source。source を渡すと target が出る。
# 「インタプリタを焼き込んだ mix」＝そのインタプリタ専用コンパイラ。
p compiler.(source)   # ⇒ target （第一射影と同じ結果が、より速く出る）
```

なぜ第二射影が嬉しいか。第一射影は、ソースが来るたびに「重い汎用の
mix」を毎回呼びます。第二射影は、`interp` への特殊化を**一度だけ先に
済ませた専用コンパイラ**を手にするので、ソースごとのコンパイルが速く
なります。「毎回 mix を呼ぶ」と「専用コンパイラを一回作って使い回す」の
差です。

第三射影は、さらにもう一段。「`mix(mix, ・)` という処理」を、今度は
**`mix` を固定して**畳みます。残るのは「`interp` を入れると `compiler` を
吐く処理」—— **コンパイラ生成器**（cogen）です。

```
cogen = mix(mix, mix)
# 残った dynamic 入力は interp。インタプリタを渡すと、その言語専用の
# コンパイラが出てくる。
p cogen.(interp_ruby)   # 擬似。⇒ Ruby 用コンパイラ
p cogen.(interp_js)     # 擬似。⇒ JavaScript 用コンパイラ
```

つまり段が上がるほど「いつ仕事を済ませるか」が前倒しになります。
第一は実行のたび、第二はソースのたび、第三は言語ごとに一回。
**Truffle/GraalVM が実際にやっているのは第一射影**（AST を毎回部分評価
してメソッド機械語を出す）ですが、「インタプリタを書けば言語が手に
入る」というフレームワークの売り文句は、精神としては第二・第三射影の
「コンパイラを書かずに済む」を実現したものだと見ることができます。
第二・第三射影そのものが実装されないのは、`mix` を self-apply して
コンパイラやコンパイラ生成器を**生成**するより、汎用の部分評価器を
毎回動かす方が、工学的には保守しやすいからです。

## 理論から実装へ：Truffle と Graal

長らく、Futamura 射影は「理論的には美しいが、実用の高性能処理系には
応用しにくい」と思われていました。素朴に部分評価すると、コードが爆発的に
膨らんだり、動的言語の型の動きをうまく畳めなかったりするのです。

これを実用の JIT として成立させたのが、Oracle Labs の **Truffle/GraalVM**
です[](#cite:wurthinger2013) [](#cite:wurthinger2017)。その最初の全体像を
描いた "One VM to Rule Them All"[](#cite:wurthinger2013)は、**1 つの VM
基盤の上で多言語の高性能処理系を構築する**という構想を掲げ、その柱を
「インタプリタ実行中に**ノードが自分自身を書き換える**フレームワーク、
その構造を活用できる投機的最適化コンパイラ、そして投機が外れたときに
インタプリタ実行へ戻る脱最適化」の組み合わせだと述べています。狙いは
**「言語ごとの専用コンパイラを書かずに、インタプリタから高性能を引き出す」**
ことです[](#cite:wurthinger2013)。仕組みはこうです。

1. 言語実装者は、対象言語の **AST インタプリタ**（構文木を直接たどって
   実行するインタプリタ）を、Truffle のフレームワーク上で Java で書く。
2. 各 AST ノードは、[プロファイリング](profiling.md)で見た要領で、実行
   しながら**自分自身を観測した型に特殊化**する（自己最適化 AST
   [](#cite:wurthinger2012)）。たとえば加算ノードが「整数同士だった」と
   観測したら、整数加算版のノードに化ける。
3. AST が安定したら（特殊化が落ち着いたら）、Graal コンパイラがその AST を
   **部分評価**して、メソッド単位の最適化機械語を生成する。

ここが巧妙な点です。素朴に「全インタプリタを部分評価」するのでなく、
**実行時の観測で AST を特殊化してから**部分評価する。型が観測で確定して
いるので、部分評価がうまく畳めて、コードも膨らみません。観測（投機）と
部分評価を組み合わせたのが、Truffle の「実用的な部分評価」
[](#cite:wurthinger2017)の核心です。インタプリタは型情報やプロファイル
情報で被解釈プログラムを特殊化し、部分評価はその特殊化を**取り込んで**
コンパイル済みコードを導出する —— これにより「特定のプログラムにとって
重要な部分はすべてコンパイルしつつ、生成コードのサイズは小さく保つ」
ことが、動的言語の文脈で実用的になりました[](#cite:wurthinger2017)。原論文の
言葉を借りれば、コンパイル済みコードはインタプリタから**自動的に**導出され、
インタプリタが行った特殊化（型情報やプロファイル情報による被解釈プログラムの
特殊化）を**取り込んだうえで**部分評価される。これが「動的言語の文脈で部分評価を
実用的にする」鍵であり、「あるプログラムにとって重要な処理をすべてコンパイル
しながら、コンパイル済みコードのサイズを縮小する」と説明されています
[](#cite:wurthinger2017)。

> [!NOTE]
> 原論文[](#cite:wurthinger2017)は §5 でこの部分評価の仕組み（インタプリタの
> メソッド呼び出しグラフをたどってコンパイル済みコードを導く §5.1）と、
> 投機が外れたときの脱最適化（§5.2）を扱っています。部分評価を素朴に
> 行うと「インタプリタのすべての処理を取り込んでコンパイル済みコードが
> 爆発する」が、観測による特殊化を取り込むことでこれを避けられる、という
> のが本章で見たトレードオフの原論文での位置づけです。

> [!NOTE]
> 原論文[](#cite:wurthinger2017)は、最適化済みコードをインタプリタから
> 部分評価で導く手法を **「第一 Futamura 射影」（the first Futamura
> projection）** と明示的に呼び、その着想が 1971 年に二村によって述べられた
> ものであること[](#cite:futamura1971)を断ったうえで、「我々の知るかぎり、
> これを採用した高性能な言語実装は過去になかった」と述べています。理論として
> 知られていた第一射影を、実際に競争力のある処理系として成立させた点が
> この仕事の核心です。

### ノード特殊化（node rewriting）の具体

「AST ノードが自分を特殊化する」を、おもちゃで具体化しましょう。加算
ノードを考えます。Truffle では、ノードの実装メソッドに型ごとの版を並べ、
**実行時に来た型に合う版へノード自身を差し替えます**（Java では
`@Specialization` という注釈で各版を宣言します[](#cite:wurthinger2012)）。
擬似的に Ruby で書くとこうです。

```ruby
# まだ型が分からない「無特殊化」の加算ノード
class AddNode
  def initialize(left, right) = (@left, @right = left, right)

  def execute(frame)
    l = @left.execute(frame)
    r = @right.execute(frame)
    # 観測した型に応じて、自分をより特殊な版に書き換える（自己書き換え）
    if l.is_a?(Integer) && r.is_a?(Integer)
      replace(IntAddNode.new(@left, @right)).execute_int(frame)
    else
      replace(GenericAddNode.new(@left, @right)).execute_generic(frame)
    end
  end
end

# 整数専用に特殊化されたノード（@Specialization(Integer) 相当）
class IntAddNode
  def execute_int(frame)
    l = @left.execute(frame)
    r = @right.execute(frame)
    raise Deopt unless l.is_a?(Integer) && r.is_a?(Integer)  # ★ ガード
    l + r                          # 整数加算に直結。オーバヘッドなし
  end
end

# どんな型でも処理する一般版（オーバロード解決つき・遅い）
class GenericAddNode
  def execute_generic(frame)
    dynamic_dispatch_add(@left.execute(frame), @right.execute(frame))
  end
end
```

ポイントは `replace` です。`AddNode` は初回実行で整数を観測すると、
**自分自身を AST 上で `IntAddNode` に置き換えます**。次回からはディス
パッチも型判定もなく、いきなり整数版が走る。これが**ノード書き換え**
（node rewriting）です。

> [!NOTE]
> この「実行しながら観測した型に特殊化する」仕掛けは、
> [プロファイリング](profiling.md)で見た**適応型インタプリタ**
> （inline cache や型フィードバックで命令を特殊化版に差し替える）と
> **まったく同型**です。違いは置き換える対象が、バイトコード命令か
> AST ノードか、だけ。Truffle はその特殊化の仕組みをフレームワーク
> として一般化し、どの言語実装でも同じ枠組みで書けるようにしたものだ、
> と捉えると見通しがよくなります。

### 特殊化した AST を部分評価すると、ディスパッチが消える

特殊化が落ち着くと、AST は「整数版ノードの木」に固まります。ここで
Graal が AST を部分評価します。このとき効くのが、**ノードのフィールドや
ツリー構造が部分評価器から見て定数だ**、という宣言です（Truffle では
`@CompilationFinal`、本書では**コンパイル時定数**と呼びます）。

たとえば `IntAddNode#execute_int` を部分評価すると、`@left`・`@right`は
コンパイル時定数なので、`@left.execute(frame)` の呼び出しは**展開**
（インライン化）され、その先のノードもまた定数なので展開され…と、
AST をたどる `execute` の連鎖がまるごと畳まれます。残るのは木が表現して
いた計算そのもの。先ほどスタックマシンで見た「`while`・`pc`・`case` が
消えて `(input + 1) * 2` だけ残る」のと、寸分違わぬ現象が AST 版で
起きるわけです。

```
# 特殊化後の AST：  IntAdd( IntMul(arg, 2), 1 )  をコンパイル時定数として展開
#   ↓ 部分評価（execute の連鎖をインライン化、ガードだけ残す）
# 残余（疑似）:
->(frame) {
  a = frame.arg              # arg ノードは定数 → 直アクセスに
  guard a.is_a?(Integer)     # ★ 投機のガードだけ残る
  (a * 2) + 1                # execute の入れ子は全部消えた
}
```

> [!TIP]
> ループ本体の AST ノードがコンパイル時定数として畳めると、ループ
> 展開（unrolling）も部分評価器が自然に行えます。ただし前述のコード
> 爆発の罠があるため、Truffle は「ここから先はループのまま残す」と
> いう境界（explode するか否か）を、ループ回数が静的に分かるかどうかや
> ヒューリスティクスで制御します。**畳めるからといって全部畳むのが
> 正解ではない**、という設計判断がここでも顔を出します。

> [!NOTE]
> [メタトレーシング](meta-tracing.md)（PyPy）と Truffle は、どちらも
> 「インタプリタを書くだけで JIT を得る」点で同じ目標を持ちますが、
> 手段が違います。PyPy は**インタプリタをトレースする**（トレーシング
> JIT を自動生成）。Truffle は**インタプリタを部分評価する**（メソッド
> JIT を自動生成）。[コンパイルの単位](compilation-units.md)で見た
> トレーシング対メソッドの対立が、フレームワークのレベルでも現れている
> のです。

## 投機も脱最適化も、ちゃんとある

部分評価で生成した機械語も、観測した型を**投機**しています。だから、
[脱最適化](deopt-osr.md)の安全網が当然必要です。Truffle/Graal は、
特殊化した AST ノードの前提が実行時に崩れたら（観測と違う型が来たら）、
脱最適化して AST インタプリタへ戻り、ノードをより一般的な版に
作り直してから、また部分評価し直します。[階層的 JIT](tiered.md)の昇格・
降格が、AST の特殊化・脱特殊化として現れているわけです。これはまさに
原論文[](#cite:wurthinger2017)が描く往復そのもので、「投機が外れると実行は
インタプリタへ戻り、プログラムはインタプリタ内で再特殊化し、その後ふたたび
部分評価が新しいインタプリタの状態をコンパイル済みコードへ変換する」と
されています。**部分評価と脱最適化はワンセット**なのです。

### Assumption：大域的に投機し、一括で無効化する

ガードには、これまで見たような「**各所に置く**」やり方
（`guard a.is_a?(Integer)` を実行のたびに評価する）と、Truffle 特有の
もう一つのやり方があります。**Assumption**（前提）です。

たとえば「`Integer#+` は再定義されない」という、めったに破れないが
破れたら広範囲に効く前提を考えます。これを各加算ノードのガードとして
毎回チェックするのは無駄です。代わりに、**一個の `Assumption` オブジェクト
を作り、それが有効なあいだは前提が成り立つ**と約束します。

```ruby
class Assumption
  def initialize = @valid = true
  def valid? = @valid
  def invalidate!                    # 前提が破れた瞬間に一度だけ呼ぶ
    @valid = false
    # この前提に依存して生成された全コンパイル済みコードを一斉に捨てる
    Compiler.deoptimize_all_dependents_of(self)
  end
end

INTEGER_PLUS_STABLE = Assumption.new

# 部分評価時：「INTEGER_PLUS_STABLE が有効」を前提に、
# 演算子探索を丸ごと畳み込んで Integer#+ に直結させる
class IntAddNode
  def execute_int(frame)
    # ★ ここにガード命令は出ない。前提に依存している、という登録だけ
    register_dependency(INTEGER_PLUS_STABLE)
    @left.execute(frame) + @right.execute(frame)
  end
end

# どこかで Integer#+ が再定義されたら……
def redefine_integer_plus
  INTEGER_PLUS_STABLE.invalidate!    # 依存していた全コードが即・無効化
end
```

ここが Assumption の妙味です。**有効なあいだは、機械語の中に検査命令が
一切出ません。** 「`Integer#+` が安定」という前提のもとで、演算子の動的
探索ごと部分評価で消し去れます。コストはほぼゼロ。そして前提が破れる
瞬間（誰かが `Integer#+` を再定義した瞬間）に `invalidate!` を一度だけ
呼べば、その前提に依存して生成された機械語が**一斉に脱最適化**されます
（[脱最適化](deopt-osr.md)）。

> [!NOTE]
> これは原論文[](#cite:wurthinger2017)の `Assumption` 機構そのものです。
> 各 `Assumption` は妥当性を表すブール値を持ち、有効性を問い合わせる
> `isValid` メソッドを備えます。コンパイラはこの `isValid` を**何もしない命令
> （no-op）にたたみ込む**ため、「前提が有効かどうかをチェックするコードは
> コンパイラからいっさい出力されません」。代わりにコンパイラは、生成した
> コードを「その前提に依存している」として登録し、誰かが `invalidate` を
> 呼んだら**依存していたコンパイル済みコードがすべて無効化**されます
> [](#cite:wurthinger2017)。

> [!IMPORTANT]
> ガード方式と Assumption 方式のトレードオフは、こう整理できます。
>
> - **各所のガード**：チェックを*実行のたび*に払う。前提が破れても、
>   そのガードを通った実行だけが脱最適化されればよい（影響が局所的）。
>   破れやすい前提（型は来るたび変わりうる）に向く。
> - **Assumption**：*平常時のコストがゼロ*。ただし破れたとき、依存する
>   コードを*すべて*捨てる必要がある（影響が大域的・高コスト）。
>   「クラスは再定義されない」「メソッドは上書きされない」のような、
>   **めったに破れないが破れたら広範囲に効く前提**に向く。
>
> 言い換えると、**破れる頻度が低いほど Assumption が得**で、
> **高いほど各所ガードが安全**。投機の粒度をどう選ぶかという、
> [脱最適化](deopt-osr.md)で見た判断が、ここでは「ガードか
> Assumption か」という具体的な実装選択として現れます。

このように、Truffle は本書の第 I 部で見た技法 —— 自己最適化（プロファイル
による特殊化）、投機、ガード、脱最適化 —— を、**部分評価という
一本の背骨**の上に統合した処理系です。実際に TruffleRuby（Ruby）・
GraalJS（JavaScript）・GraalPy（Python）など、多数の言語処理系がこの
基盤の上で高い性能を出しています。[ケーススタディ](case-others.md)で
再訪します。

> [!NOTE]
> 原論文[](#cite:wurthinger2017)は、JavaScript・Ruby・R の各実装を、その
> 言語専用に作り込まれた最高水準の処理系と比較して評価しています。論文の
> ベンチマーク集合での速度比は、JavaScript で **V8 に対して 0.83 倍**（やや
> 及ばない）、Ruby で **JRuby に対して 3.8 倍**、R で **GNU R に対して 5 倍**と
> 報告されています。汎用の 1 つのコンパイル基盤でありながら、1 言語に
> 特化して重く最適化された処理系と**競争力がある**、というのがこの仕事の
> 主張です[](#cite:wurthinger2017)。

## まとめ

- **部分評価**は、二入力プログラムの片方の入力を先に固定し、残りの入力
  だけを取る、より速い**残余プログラム**を生成する技法。入力を
  **static / dynamic** に塗り分け（束縛時刻解析）、static な計算を畳み、
  dynamic だけを残す。
- 素朴な mix には**コード爆発**の罠があり、「どこまで展開しどこから
  ループのまま残すか」が最大のトレードオフ。
- **Futamura の第一射影**：インタプリタをソースについて部分評価すると、
  そのソースのコンパイル結果が得られる。**インタプリタ＋部分評価＝
  コンパイル**。`while`・`pc`・`case` が static として消え、計算だけが残る。
- 第二射影は**コンパイラ**、第三射影は**コンパイラ生成器**を生む。
  段が上がるほど「仕事を済ませる時刻」が前倒しになる。
- 40 年後、**Truffle/GraalVM** が実用化。**自己最適化 AST**（観測で型を
  特殊化＝ノード書き換え）してから**部分評価**することで、コード爆発を
  抑えた高性能 JIT を実現した。ノード書き換えは適応型インタプリタの
  特殊化と同型。
- コンパイル時定数（`@CompilationFinal`）により、特殊化済み AST を
  部分評価すると `execute` の連鎖がインライン化され、ディスパッチが消える。
- 投機・ガード・脱最適化も備える。**Assumption** は平常時コスト 0 で
  大域的に投機し、破れたら依存コードを一斉に脱最適化する。各所ガードと
  「破れる頻度」でトレードオフ。
- PyPy（メタトレーシング）と Truffle（部分評価）は同じ目標・別手段。
  「走らせて消す」のがトレース、「解析して消す」のが部分評価。

次は、JIT の「速くなるまでの時間」—— ウォームアップそのものを研究対象に
した、ベンチマークの常識を揺るがした仕事を見ます。
