# 中間表現：最適化のための作業台

[最小の JIT](hello-jit.md)では、式から機械語を 1 命令ずつ直接生成しました。
単純な式ならそれで十分ですが、本格的に最適化しようとすると、機械語を
直接いじるのは無理があります。「この変数はもう使われていない」「この
計算は前の計算と同じだ」といった分析をするには、機械語は低レベルすぎて、
かつ x86 や ARM といった**CPU ごとの違い**に縛られすぎています。

そこで JIT は、機械語に落とす前に、**中間表現**（intermediate
representation、IR）という作業台の上でプログラムを表現し、そこで最適化を
します。この章では、JIT でよく使われる中間表現と、その上での最適化、
そして最後に機械語へ落とす際の**レジスタ割り当て**を見ます。

## なぜ中間表現が要るのか

中間表現に求められるのは、おおむね次のことです。

- **分析しやすい**：データの流れ（どの値がどこで作られ、どこで使われるか）が
  見やすい形になっていること。
- **CPU 非依存**：x86 でも ARM でも共通に最適化できること。最後にだけ
  CPU 固有の機械語へ落とす。
- **変換しやすい**：最適化は「IR を読んで、より良い IR を書く」変換の
  連なりです。書き換えやすい構造であること。

多くの JIT は、複数の中間表現を**段階的に**使います。高レベルの IR で
言語に近い最適化をし、だんだん機械語に近い低レベルの IR へ落としていく。
この段階を **HIR（high-level IR）→ MIR（mid）→ LIR（low-level IR）** と
呼び分ける処理系もあります。もっとも本書では、この段階の区別には立ち入らず、
以降はまとめて単に **IR** と呼びます。

## 出発点：バイトコード

多くの JIT の入口は、すでにある**バイトコード**です。姉妹書
[言語処理系入門](https://kolanglab.github.io/book_langimpl_intro/)で作る
ようなスタックマシンのバイトコードは、それ自体が一種の中間表現です。

```
# x * x + y を計算するスタックバイトコード
load x
load x
mul
load y
add
```

しかしスタックバイトコードは、最適化には少し不便です。値が暗黙の
スタックに乗っているので、「この `mul` の結果がどこで使われるか」を
追うのに、スタックの動きを頭の中でシミュレートしなければなりません。
そこで JIT は、これを**データの流れが明示された形**へ作り直します。

## SSA 形式：すべての変数を一度だけ代入する

最適化用の中間表現として、いまや事実上の標準が **SSA 形式**（Static
Single Assignment、静的単一代入）です[](#cite:cytron1991)。SSA の規則は
たった一つ、

> **すべての変数は、プログラム中でちょうど一度だけ代入される。**

です。同じ変数に二度代入したいときは、`x`、`x₁`、`x₂` のように番号を
振った別の変数にします。

```
# 元のコード             # SSA 形式
x = a + b               x1 = a + b
x = x * 2               x2 = x1 * 2
y = x + 1               y1 = x2 + 1
```

なぜこれが嬉しいか。SSA では、**ある変数を見れば、それがどこで定義された
かが一意に決まる**のです。`x2` を使っている箇所は、必ず `x2 = x1 * 2` で
作られた値を指します。「この値はどこから来たか」を探す手間が消え、
データの流れがグラフとしてまっすぐ見えます。これが各種最適化を劇的に
簡単にします。

分岐が合流する場所では、「どちらの経路から来たか」で値が変わります。
SSA はこれを **φ（ファイ）関数**という特別な命令で表します。
`x3 = φ(x1, x2)` は「B1 から来たら `x1`、B2 から来たら `x2` を `x3` と
する」という意味です。φ 関数は、合流点で複数の定義を 1 つにまとめる
ための、SSA の縫い目です。

## SSA の上で最適化してみる

SSA がどれだけ最適化を楽にするか、Ruby で小さな最適化器を書いて
体感しましょう。命令を `[結果, 演算, 左, 右]` のタプルの列で表し、
**定数畳み込み**（コンパイル時に計算できる定数式を、その結果で
置き換える）と**コピー伝播・共通部分式の削除**を素朴に実装します。

```ruby
# SSA 命令の列に対する、定数畳み込み＋共通部分式削除
def optimize(insns)
  values = {}    # 変数名 → 確定した定数値（分かるもののみ）
  seen   = {}    # 計算式 → それを最初に計算した変数名
  out    = []

  insns.each do |dst, op, a, b|
    # 1) オペランドが定数と分かっていれば、その値に置き換える
    av = values.fetch(a, a)
    bv = values.fetch(b, b)

    # 2) 定数畳み込み：両辺が数なら、いま計算してしまう
    if av.is_a?(Integer) && bv.is_a?(Integer)
      result = case op
               when :add then av + bv
               when :sub then av - bv
               when :mul then av * bv
               end
      values[dst] = result          # dst は定数 result だと記録
      next                          # 命令そのものは消える
    end

    # 3) 共通部分式の削除：同じ式を前に計算済みなら使い回す
    key = [op, av, bv]
    if (prev = seen[key])
      values[dst] = prev            # dst は prev のコピー
      next
    end

    seen[key] = dst
    out << [dst, op, av, bv]
  end
  out
end

# (2 + 3) を使い、さらに同じ a*b を 2 回計算する例
program = [
  [:t1, :add, 2, 3],      # t1 = 2 + 3       → 定数 5 に畳める
  [:t2, :mul, :a, :t1],   # t2 = a * 5
  [:t3, :mul, :a, :b],    # t3 = a * b
  [:t4, :mul, :a, :b],    # t4 = a * b       → t3 と同じ。使い回せる
  [:t5, :add, :t3, :t4],  # t5 = t3 + t4
]

optimize(program).each { |i| p i }
# [:t2, :mul, :a, 5]
# [:t3, :mul, :a, :b]
# [:t5, :add, :t3, :t3]
```

`t1 = 2 + 3` はコンパイル時に `5` へ畳まれて消え、`t2` のオペランドに
直接 `5` が入りました。`t4 = a * b` は `t3` と同じ式なので削除され、
`t5` は `t3 + t3` になりました。**変数が一度しか定義されない**という SSA
の性質のおかげで、「この変数は本当にあの式か？」を疑う必要がなく、
こんなに素直に書けるのです。本物の JIT の最適化も、規模は違えど発想は
これと同じです。

> [!NOTE]
> ここで実装した最適化は、コンパイラ理論では**定数畳み込み**
> （constant folding）と**共通部分式の削除**（common subexpression
> elimination, CSE）と呼ばれる古典です。JIT 特有ではありませんが、
> [インライン化](why-jit.md)で呼び出し先の中身が見えるようになった
> あとに効くと効果が大きく、JIT でも主力の最適化です。

### デッドコード削除：使われない定義を消す

定数畳み込みや共通部分式の削除をすると、結果として**もう誰も使わない
定義**が残ります。さきほどの例でも `t4 = a * b` が `t3` に統合された
あと、もし `t4` を別の場所でだけ使っていたら、`t4` 自体は不要になります。
こうした**使われない値の計算**を消すのが**デッドコード削除**（dead code
elimination, DCE）です。

ここでも SSA が効きます。「変数は一度しか定義されない」ので、ある定義が
死んでいるかどうかは、**その変数を使っている辺が一つでもあるか**を数えれば
分かるからです。使用辺がゼロなら、その定義は消してよい。ただし消すと、
その命令が使っていたオペランドの使用辺も減るので、**消滅が連鎖**します。
そこで「使用回数が 0 になった定義を消し、それに伴って減った使用回数を
反映し、また 0 になったものを消す」を、変化がなくなるまで繰り返します。

```ruby
# SSA 命令列のデッドコード削除。
# insns: [dst, op, *args] の列。live_out: 外部から使われる変数（消せない）
def dce(insns, live_out = [])
  # 各変数の使用回数を数える
  uses = Hash.new(0)
  insns.each { |dst, op, *args| args.each { |a| uses[a] += 1 if a.is_a?(Symbol) } }
  live_out.each { |v| uses[v] += 1 }   # 戻り値などは「使われている」扱い

  # 純粋な演算（副作用なし）だけが削除候補。:call や :store は残す
  pure = %i[add sub mul phi copy]

  worklist = insns.map { |dst, _op, *| dst }.select { |d| uses[d].zero? }
  dead = {}
  until worklist.empty?
    dst = worklist.shift
    insn = insns.find { |d, _op, *| d == dst }
    next unless insn && pure.include?(insn[1]) && !dead[dst]
    dead[dst] = true
    # この命令が消えると、オペランドの使用が 1 ずつ減る
    insn[2..].each do |a|
      next unless a.is_a?(Symbol)
      uses[a] -= 1
      worklist << a if uses[a].zero?   # 0 になったら次の削除候補
    end
  end
  insns.reject { |dst, _op, *| dead[dst] }
end

program = [
  [:t1, :mul, :a, :b],   # t1 = a*b   … t2 でしか使われない
  [:t2, :add, :t1, 1],   # t2 = t1+1  … 誰も使わない（live_out にない）
  [:t3, :mul, :a, :a],   # t3 = a*a   … 戻り値
]
dce(program, [:t3]).each { |i| p i }
# [:t3, :mul, :a, :a]
```

`t2` が誰にも使われないので消え、すると `t1` の使用回数も 0 になって
**連鎖的に**消えました。`t3` だけが残ります。`pure` に載った副作用の
ない演算しか消さないのが要点です。`:call`（呼び出し）や `:store`
（メモリ書き込み）は、結果を使わなくても**副作用のために実行しなければ
ならない**ので、たとえ使用辺ゼロでも残します。

> [!WARNING]
> 「使われていないから消す」は、**副作用がないとき**だけ正しい変換です。
> JIT では、後で見るガードや脱最適化点[](#cite:duboscq2013)も「副作用を
> 持つ命令」の一種として扱わねばなりません。ガードは値を生まないのに、
> 「賭けが外れたら[脱最適化](deopt-osr.md)する」という観測可能な効果を
> 持つからです。安易に「使われない＝消せる」と一般化すると、投機の前提を
> 確かめる命令ごと消してしまいます。

### コピー伝播：余計な転送をたどって消す

CSE をすると `t5 = t3 + t3` のように、**ある変数が別の変数のただのコピー**
になることがあります（`values[dst] = prev` がそれです）。`x = y` という
コピーが残っていると、`x` を使う側はわざわざ `y` から `x` へ値を移す
無駄が生じます。**コピー伝播**（copy propagation）は、`x` の使用箇所を
すべて `y` に書き換えてしまい、コピーそのものをデッドコード削除に
任せて消す変換です。

```ruby
# x = copy(y) の形のコピーをたどり、使用側を元の変数に置き換える
def copy_propagation(insns)
  alias_of = {}                       # コピー先 → 真の出どころ
  resolve  = ->(v) {                  # コピーの連鎖を根までたどる
    v = alias_of[v] while alias_of.key?(v)
    v
  }
  insns.map do |dst, op, *args|
    args = args.map { |a| a.is_a?(Symbol) ? resolve.(a) : a }
    if op == :copy
      alias_of[dst] = args.first      # dst は args.first の別名と記録
    end
    [dst, op, *args]
  end
end

program = [
  [:t1, :add, :a, :b],
  [:t2, :copy, :t1],       # t2 は t1 のただのコピー
  [:t3, :mul, :t2, 2],     # t2 を使っている → t1 に直したい
]
copy_propagation(program).each { |i| p i }
# [:t1, :add, :a, :b]
# [:t2, :copy, :t1]       ← この行は後で DCE が消す
# [:t3, :mul, :t1, 2]     ← t2 が t1 に置き換わった
```

`t3` のオペランドが `t2` から `t1` に変わり、`t2` のコピーは誰も使わなく
なりました。あとは前項の `dce` を通せば `t2` の行が消えます。**コピー伝播
＋デッドコード削除**はセットで使い、最適化が生んだ「余りもの」を掃除する
基本コンビです。SSA だからこそ、`resolve` で根までたどっても**途中で
値が再代入される心配がない**点に注目してください。

### 大域値番号付け：式の同値を名前で管理する

さきほどの CSE は「直前までに**同じ演算と同じオペランド**で計算した式」を
使い回しました。これを**ブロックをまたいで**、しかも代数的な同値まで
扱えるよう一般化したのが**大域値番号付け**（global value numbering, GVN）
です。各値に「同じ計算結果なら同じになる**値番号**」を振り、番号が一致
すれば一方を捨てます。

GVN の肝は、**正規化**してから番号を引くことです。`a + b` と `b + a` は
可換なので同じ番号にしたい。`x * 1` は `x` と同じにしたい。こうした
正規化を噛ませると、見た目が違う式の同値が拾えます。

```ruby
# 値番号付け。式を正規化し、同じ式には同じ番号を割り当てる
def gvn(insns)
  num     = {}      # 変数 => 値番号
  expr2no = {}      # 正規化した式 => 値番号
  next_no = 0
  out     = []

  vn = ->(x) { x.is_a?(Symbol) ? (num[x] ||= (next_no += 1)) : [:const, x] }

  insns.each do |dst, op, a, b|
    na, nb = vn.(a), vn.(b)
    # 可換な演算はオペランドをソートして a+b と b+a を同一視
    na, nb = [na, nb].sort_by(&:hash) if %i[add mul].include?(op)
    key = [op, na, nb]

    if (existing = expr2no[key])
      num[dst] = existing            # 同じ計算 → 既存の番号を共有（命令は消す）
      next
    end
    no = (next_no += 1)
    expr2no[key] = no
    num[dst] = no
    out << [dst, op, a, b]
  end
  out
end

program = [
  [:t1, :add, :a, :b],
  [:t2, :add, :b, :a],     # a+b の可換 → t1 と同値
  [:t3, :mul, :t1, :t2],   # t1*t2 だが t1==t2 なので t1*t1
]
gvn(program).each { |i| p i }
# [:t1, :add, :a, :b]
# [:t3, :mul, :t1, :t2]
```

`t2 = b + a` が `t1 = a + b` と**同じ値番号**になり、`t2` の命令が消えま
した。CSE が「字面が同じ式」しか拾えなかったのに対し、GVN は正規化を
通して**意味が同じ式**まで拾えます。ここでも、SSA で各変数の定義が一意
だから「同じ番号なら本当に同じ値」と言い切れる —— もし変数が途中で再代入
されうるなら、番号が一致しても値が同じとは限らず、この手は使えません。

> [!TIP]
> 上の `sort_by(&:hash)` は「可換な式の正規化」を手短に見せるための便法です。
> Ruby の `Object#hash` は**プロセスごとに変わりうる**ので、これに依存すると
> 実行のたびに正規化の向きがぶれます。本物の実装は、ハッシュ値ではなく
> **値番号そのものを正規化キー**にして（たとえば番号の昇順に並べて）安定させます。

> [!TIP]
> ここまでの定数畳み込み・CSE・デッドコード削除・コピー伝播・GVN は、
> どれも「変数の定義が一意」という SSA の一点に寄りかかっています。
> 「SSA だから簡単になる」を実感するには、同じ最適化を**変数が再代入
> される普通の形**で書こうとしてみるのが一番です。たちまち「この `x` は
> どの代入か」を追う面倒が湧いてきて、SSA のありがたみが分かります。

### ループ不変式の巻き上げ：変わらない計算を外へ出す

最適化の効果がとりわけ大きいのが、**ループの中**です。ループは何度も
回るので、1 回あたりを少し速くするだけで全体に効きます。その代表が
**ループ不変式の巻き上げ**（loop-invariant code motion, LICM）です。
ループの中にあるが、**回るたびに値が変わらない計算**を、ループの前
（プリヘッダ、preheader）へ 1 回だけ追い出します。

「不変かどうか」の判定は、SSA だと素直です。**オペランドがすべて、
ループの外で定義されているか、それ自身も不変な命令か**なら、その命令は
不変です。これを不変なものが増えなくなるまで繰り返します。

```ruby
# ループ本体の命令のうち、不変なものを前へ巻き上げる。
# body: ループ内の [dst, op, *args]。outer: ループ外で定義済みの変数集合
def licm(body, outer)
  invariant = outer.dup               # 不変と分かっている変数の集合
  hoisted   = []
  changed   = true
  while changed                       # 不変集合が育たなくなるまで反復
    changed = false
    body.each do |insn|
      dst, _op, *args = insn
      next if invariant.include?(dst)
      # オペランドが全部「定数 or 既に不変」なら、この命令も不変
      if args.all? { |a| !a.is_a?(Symbol) || invariant.include?(a) }
        invariant << dst
        hoisted << insn
        changed = true
      end
    end
  end
  loop_body = body.reject { |insn| hoisted.include?(insn) }
  [hoisted, loop_body]                # [前へ出す命令, ループに残す命令]
end

# for i in ...:  t1 = n - 1;  t2 = a[t1];  s = s + i  のような本体。
# i はループ変数なので outer に入れない（毎回変わる）
outer = %i[n a s_in]
body = [
  [:t1, :sub, :n, 1],        # n も 1 も不変 → 不変！
  [:t2, :load, :a, :t1],     # a も t1 も不変 → 不変！（純粋な読みと仮定）
  [:s1, :add, :s_in, :i],    # i はループ変数 → 変動。残す
]
hoisted, kept = licm(body, outer)
puts "巻き上げ:"; hoisted.each { |i| p i }
puts "残す:";     kept.each   { |i| p i }
# 巻き上げ:
# [:t1, :sub, :n, 1]
# [:t2, :load, :a, :t1]
# 残す:
# [:s1, :add, :s_in, :i]
```

`t1 = n - 1` と、それに依存する `t2 = a[t1]` が、ともに不変と判定されて
プリヘッダへ巻き上がりました。ループが 100 万回回るなら、この 2 命令を
100 万回ではなく 1 回しか実行しません。

> [!WARNING]
> LICM には落とし穴があります。**ループが 0 回しか回らない**場合、
> 巻き上げた計算は「本来なら一度も実行されなかったはず」のものです。
> もしそれが例外を投げたりメモリを読みに行ったりすると、元のプログラムには
> なかった副作用やエラーがプリヘッダで起きてしまいます。だから巻き上げて
> よいのは原則**副作用がなく失敗もしない計算**だけで、`a[t1]` のような
> メモリ読み込みを巻き上げるには「ループが必ず 1 回以上回る」「その番地は
> 必ず読める」といった**追加の保証**が要ります。次節の範囲解析が、その
> 保証を与える道具になります。

### 範囲解析と境界チェック除去

配列アクセス `a[i]` は、安全のため「`0 <= i` かつ `i < a.length` か」を
毎回確かめる**境界チェック**を伴います。ループで配列をなめるコードでは、
このチェックが回るたびに走り、無視できない負担になります。けれど多くの
場合、`i` の動く範囲は静的に分かります。`for i in 0...n` で `i` を回し、
`a.length` が `n` 以上だと分かっていれば、**ループ内の境界チェックは
理論上すべて不要**です。

これを支えるのが**範囲解析**（range analysis）です。各値が取りうる範囲
`[下限, 上限]` を、命令をたどりながら伝播させます。たとえば `0...n` の
ループ変数 `i` は `[0, n-1]`、`i + 1` は `[1, n]`、といった具合です。
範囲が `0 <= i` と `i < n` を含むと示せれば、その配列アクセスの境界
チェックは消せます。

```ruby
# ごく素朴な範囲解析つき境界チェック除去。
# 各変数の範囲 [下限, 上限] を ranges に持ち、idx が 0 <= idx < len と
# 示せた bounds_check を「除去済み」の印へ書き換える
def remove_bounds_checks(insns, ranges)
  insns.map do |insn|
    dst, op, *args = insn
    next insn unless op == :bounds_check
    idx, len = args                 # idx が 0 <= idx < len か確かめる命令
    lo, hi = ranges[idx]
    if lo && lo >= 0 && hi && hi < ranges_upper(len, ranges)
      [dst, :check_removed, idx]    # 0 <= idx < len を証明 → チェック除去
    else
      insn                          # 範囲が足りない → チェックは残す
    end
  end
end

# len が「i の上限より大きい」と示せるなら、その上限値を返す。
# ここでは len == :n かつ i の上限が n-1 なら、:n より大の境界として扱う
def ranges_upper(len, ranges)
  len == :n ? ranges.fetch(:n_value, Float::INFINITY) : len
end

# i の範囲を [0, n-1] と解析できたとする。便宜上 n の値を 100 と置く
ranges = { i: [0, 99], n_value: 100 }

insns = [[:c1, :bounds_check, :i, :n], [:v, :load, :a, :i]]
remove_bounds_checks(insns, ranges).each { |i| p i }
# [:c1, :check_removed, :i]
# [:v, :load, :a, :i]
```

境界チェック `c1` が「証明済み」として除去され、`load` だけが残りました。
本物の範囲解析はもっと精密で、ループの単調増加や `min`/`max`、条件分岐で
絞り込まれた範囲なども追いますが、骨子は「**各値の取りうる範囲を伝播し、
ガードの条件がその範囲で恒真なら、ガードを消す**」です。

ここが JIT の最適化と[脱最適化](deopt-osr.md)が交わる勘所です。境界
チェックは、外れたら例外を投げる **ガードの一種**と見なせます。型ガードと
まったく同じ枠組みで、「上流の情報からガードの条件が恒真と示せたら消す」
わけです。

ここまでの「静的に証明できたら消す」は、実は AOT コンパイラもやる一般的な
最適化です。JIT の真骨頂は、**証明できないときにも投機で踏み込める**ことにあります。
添字が外部データ由来で範囲が読めない、解析が複雑すぎる —— そんなときでも
二つの手があります。一つは**ガードの巻き上げ**で、ループの前で「添字の最小・最大が
`0 <= lo` かつ `hi < len` か」を一度だけ確かめ、中の各反復はチェックなしの裸で
回す。前置きの確認が外れたらチェック付きの版へ落とす。もう一つは、
[プロファイル](profiling.md)から「この添字はいつも範囲内」と**投機**して境界
チェックを出さず、もし一度でも範囲外が来たら[脱最適化](deopt-osr.md)して、今度は
チェック付きで作り直す。**「消して、外れたら戻す」は静的コンパイラにはできない、
JIT だけの一手**です（[なぜ JIT か](why-jit.md)の投機が、配列の境界でも効いて
いるわけです）。

> [!NOTE]
> 境界チェックを消したり投機で飛ばしたりすることには、性能だけでなく
> **セキュリティ**の側面もあります。[JIT とセキュリティ](security.md)で見る
> 投機実行サイドチャネル（Spectre）は、まさに「境界チェックの先で CPU が
> 投機的に先走る」ことを突くものでした。攻めの境界チェック除去と、その安全性は
> 表裏一体です。

次に、このガードの削減を IR 変換として一般化します。

### ガードを IR 変換の対象にする

JIT 特有の事情として、中間表現は**投機**を表現できる必要があります。
[なぜ JIT か](why-jit.md)で見たガードと脱最適化 —— 「ここで型を確かめ、
外れたらインタプリタへ戻る」—— を、中間表現の中に**第一級の命令**として
持たせるのです。第一級にする最大の利点は、ガードを**ふつうの IR 変換の
対象**にできること。具体的には次の三つが定石です。

- **ガードの巻き上げ（hoisting）**：ループの中で毎回確かめている型ガードを、
  LICM と同じ理屈でループの外へ 1 回だけ追い出します。ガードの条件が
  ループ不変（`x` の型はループ中変わらない）なら成立します。
- **ガードの統合（merging）**：近い場所にある同じ・あるいは一方が他方を
  含意するガードを、強いほう 1 つにまとめます。`i < n` と `i < n-1` が
  並んでいれば、強い `i < n-1` だけ残せばよい。
- **ガードの除去（elimination）**：上流の情報からガードの条件が恒真と
  示せたら、丸ごと消します。前節の境界チェック除去がこれです。

下のおもちゃは、ループ内で繰り返される同一の型ガードを 1 つに統合する
様子です。

```ruby
# 同じ「x は Integer」ガードが複数あれば、最初の 1 つだけ残す
def dedup_guards(insns)
  established = {}                    # 既に確かめ済みの (変数, 条件)
  insns.reject do |dst, op, var, cond|
    if op == :guard
      key = [var, cond]
      if established[key]
        true                          # すでに確認済み → このガードは消す
      else
        established[key] = true
        false
      end
    else
      false
    end
  end
end

insns = [
  [:g1, :guard, :x, :is_int],   # 1 回目：残す
  [:t1, :add,   :x, 1],
  [:g2, :guard, :x, :is_int],   # 2 回目：x は変わっていない → 消す
  [:t2, :mul,   :x, 2],
]
dedup_guards(insns).each { |i| p i }
# [:g1, :guard, :x, :is_int]
# [:t1, :add, :x, 1]
# [:t2, :mul, :x, 2]
```

`g2` が消え、ループ中の型チェックが 1 回になりました。これは見た目には
ただのデッドコード削除や CSE と同じ「IR の書き換え」です。ガードを第一級に
した狙いどおり、**投機の安全装置が、ふつうの最適化の語彙で扱える**ように
なっています。

> [!IMPORTANT]
> ガードを消したり動かしたりするとき、**消した値には復元のレシピを残す
> 義務**があります。たとえば境界チェックを消した結果、もう `i < n` を
> 確かめる命令がコードに無くなったとしても、もし別のガードで[脱最適化](deopt-osr.md)
> してインタプリタへ戻るなら、その地点でのローカル変数の値は再現できなければ
> なりません。最適化器が中間値を消すたびに「消す代わりに、ガード地点で
> どう作り直すか（再具体化のレシピ）を記録する」という規律が要ります。
> この「最適化の自由」と「復元の正しさ」の綱引きは、
> [投機的最適化の正しさ](research-deopt-correctness.md)で正面から扱います。

### 本物の JIT でのガード：GraalVM の浮動ノード

いま Ruby で書いた `dedup_guards` の発想は、本物の JIT では中間表現の設計
そのものに組み込まれています。GraalVM の中間表現は、この「ガードと
脱最適化点」を明示的なノードとして設計しています[](#cite:duboscq2013)。
そこではガードは、制御フローの特定の点に固定されない**浮動ノード**
（floating node）として表され、IR の中を比較的自由に動かせます。しかも
他の浮動ノードと同じく**大域値番号付け**の対象になるので、同じ条件・同じ
アンカーを持つガードは自動的に一つへまとめられる（coalesce される）と
報告されています[](#cite:duboscq2013)。さきほどの `dedup_guards` が、
本物の JIT では GVN の副産物として実現されているわけです。投機の正しさを
保ったままこうした変形をしてよいのか、という問いは
[投機的最適化の正しさ](research-deopt-correctness.md)で扱います。

## Sea of Nodes：制御もデータも一つのグラフに

SSA をさらに推し進めた中間表現が、**Sea of Nodes**（ノードの海）です
[](#cite:click1995)。Cliff Click が考案し、JVM の C2 コンパイラ
[](#cite:paleczny2001)や、GraalVM の中核として使われています。

ふつうの中間表現は、命令を「上から順に並んだリスト」として持ち、その上に
制御フロー（どのブロックへ進むか）を重ねます。Sea of Nodes はこの「順番」を
**できるだけ捨てます**。各演算をグラフのノードにし、ノードからノードへ
「この値を使う」という**データ依存の辺**を張る。すると、互いに依存しない
計算には順序がなくなり、「依存関係さえ守れば、どの順で計算してもよい」
という自由度が露わになります。

この自由度がコンパイラに効きます。命令を動かす最適化（ループの外へ
追い出す、など）が、「辺を切らない範囲で自由に並べ替えてよい」という形に
なるからです。Click らの原論文も、評価順序への制約を従来モデルより緩める
ことで「解析・最適化アルゴリズムに、事実を見つけコードを組み替える自由が
より多く与えられる」と述べています[](#cite:click1995)。制御フローもまた
別のノードと辺で表し、データの部分グラフと制御の部分グラフが一つの「海」に
溶け合っていて、最適化はその区別をしません —— だから Sea of Nodes です。

実際の命令の**順序を最終的に決める**のは、最適化を一通り終えたあとの
スケジューリングです。原論文の割り当て器も、グラフをなめて命令を並べ
（直列化し）ながら、可能ならスピルを避けるようコードをスケジュールします
[](#cite:click1995)。「まず依存関係だけのグラフで最適化し、順序付けは後回し」
という分業が、Sea of Nodes の要点です。

> [!TIP]
> Sea of Nodes は強力ですが、人間には読みにくく、実装も難しい中間表現
> です。「順序を捨てて依存関係だけ残すと、並べ替えの自由が手に入る」と
> いう発想を押さえれば十分です。すべての JIT がこれを使うわけではなく、
> もっと素朴な「ブロックと命令リスト」の中間表現を選ぶ処理系も多くあります
> （[Ruby の YJIT/ZJIT](case-ruby.md) や [Cranelift](frameworks.md) など）。

## SSA から出る：φ 関数をコピーへ落とす

SSA は最適化には最高ですが、**機械には φ 関数という命令がありません**。
`x3 = φ(x1, x2)` は「どの経路から来たかで値が決まる」という抽象的な縫い目で、
実際の CPU 命令ではないからです。そこで機械語へ落とす前に、φ 関数を消して
ふつうのコードへ戻す **SSA から出る**（out-of-SSA）処理が要ります。

やり方の基本は単純です。`x3 = φ(x1, x2)` で「B1 から来たら `x1`、B2 から
来たら `x2`」なら、**B1 の末尾に `x3 = x1` を、B2 の末尾に `x3 = x2` を
置く**。各前駆ブロックの最後で、対応する値を φ の結果へコピーしておけば、
合流点に着いたときには `x3` に正しい値が入っています。

```ruby
# out-of-SSA：各ブロック末尾の φ を、前駆ブロックへのコピーに変換する。
# blocks: { ブロック名 => { insns:[...], phis:[[dst, {前駆=>値}]], preds:[...] } }
def out_of_ssa(blocks)
  copies = Hash.new { |h, k| h[k] = [] }   # 前駆ブロック => 末尾に足すコピー
  blocks.each do |name, b|
    b[:phis].each do |dst, sources|
      sources.each do |pred, val|
        copies[pred] << [dst, :copy, val]   # pred の末尾で dst = val
      end
    end
    b[:phis] = []                            # φ は消す
  end
  copies.each { |pred, cs| blocks[pred][:insns].concat(cs) }
  blocks
end

blocks = {
  b1: { insns: [], phis: [], preds: [] },
  b2: { insns: [], phis: [], preds: [] },
  merge: { insns: [[:y, :add, :x3, 1]],
           phis: [[:x3, { b1: :x1, b2: :x2 }]], preds: %i[b1 b2] },
}
out_of_ssa(blocks)
puts "b1 末尾: #{blocks[:b1][:insns].inspect}"
puts "b2 末尾: #{blocks[:b2][:insns].inspect}"
puts "merge: #{blocks[:merge][:insns].inspect}（φ は消えた）"
# b1 末尾: [[:x3, :copy, :x1]]
# b2 末尾: [[:x3, :copy, :x2]]
# merge: [[:y, :add, :x3, 1]]（φ は消えた）
```

φ が前駆ブロックのコピーへ散り、合流点からは φ が消えました。これらの
コピーの多くは、続けて**コピー伝播とデッドコード削除**を回せば消えるか、
レジスタ割り当てで「同じレジスタを割り当てて転送を省く」ことで実体を
持たなくできます。ただしこのおもちゃは、挿入したコピーを**並列コピーとして
逐次化する処理を省いています**。同じ前駆ブロックに複数の φ が同時に着地し、
しかも互いの値を入れ替える（swap）形になると、コピーを素朴に順番へ並べた
だけでは壊れます（次の WARNING）。

> [!WARNING]
> 素朴に「φ をコピーへ」だけやると、**並列コピー**の罠を踏みます。同じ
> 合流点に `a3 = φ(a1, b1)` と `b3 = φ(b1, a1)` のように φ が複数あると、
> 二つの φ は「同時に」起きるべきなのに、コピーを順番に実行すると先の
> コピーが後で使う値を壊します（`a3 = a1; b3 = a3` としてしまうと `b3` が
> 狂う）。これが **lost-copy 問題**・swap 問題として知られ、一時変数を
> 挟む、依存順に並べる、循環は退避用レジスタで断ち切る、といった**並列
> コピーの逐次化**が必要です。SSA から出る処理が「ただのコピー挿入」より
> 一段やっかいなのは、ここに理由があります。

## 機械語へ落とす：レジスタ割り当て

中間表現の上で最適化を終えたら、最後に機械語へ落とします。ここで避けて
通れないのが**レジスタ割り当て**（register allocation）です。

中間表現では、`t1`, `t2`, `t3`, … と好きなだけ変数（仮想レジスタ）を
使えます。ところが本物の CPU には、整数レジスタが十数本しかありません
（x86-64 で 16 本、うち自由に使えるのはさらに少ない）。無数の仮想
レジスタを、有限の物理レジスタへ**割り当て**なければなりません。同時に
「生きている」値が物理レジスタの数を超えたら、あふれた値を一時的に
メモリへ追い出します（**スピル**、spill）。

AOT コンパイラは、この問題を**グラフ彩色**という重い手法で高品質に
解きます。しかし JIT には時間がありません。**コンパイル自体が実行時間の
一部**なので、レジスタ割り当てに何ミリ秒もかけるわけにいかない。そこで
JIT がよく使うのが、**線形走査レジスタ割り当て**（linear scan register
allocation）です[](#cite:poletto1999)[](#cite:kotzmann2008)。

線形走査の発想は単純です。各変数が「いつ作られ、いつ最後に使われるか」の
区間（生存区間、live interval）を求め、命令を頭から一回なめながら、
区間が重なる変数どうしが同じレジスタを使わないように、空いているレジスタを
割り当てていく。区間が重なる変数が多すぎてレジスタが尽きたら、いちばん
遠くまで使わない変数をメモリへスピルする。生存区間を**開始点の昇順**に
一度なめるだけなので、レジスタ数によらず干渉グラフを作る必要がありません
[](#cite:poletto1999)。

```ruby
# 線形走査レジスタ割り当ての骨格
# intervals: { 変数 => [開始, 終了] }, regs: 使える物理レジスタの数
def linear_scan(intervals, regs)
  # 生存区間を開始点の順に並べる
  ordered = intervals.sort_by { |_, (start, _)| start }
  free    = (0...regs).to_a       # 空きレジスタ
  active  = []                    # いま割り当て中の [終了, 変数]
  alloc   = {}                    # 変数 => :reg(番号) または :spill

  ordered.each do |var, (start, finish)|
    # この変数の開始時点で、すでに死んだ区間のレジスタを回収する
    active.reject! do |end_pt, v|
      if end_pt < start
        free.push(alloc[v][1]) if alloc[v].first == :reg
        true
      end
    end

    if free.empty?
      alloc[var] = [:spill]       # 空きがなければメモリへ追い出す
    else
      alloc[var] = [:reg, free.shift]
      active.push([finish, var])
    end
  end
  alloc
end

# t1..t4 の生存区間。レジスタは 2 本しかないとする
intervals = { t1: [0, 2], t2: [1, 4], t3: [3, 5], t4: [2, 3] }
linear_scan(intervals, 2).each { |v, a| puts "#{v} => #{a.inspect}" }
# t1 => [:reg, 0]
# t2 => [:reg, 1]
# t4 => [:spill]
# t3 => [:reg, 0]
```

レジスタ 2 本に対し、開始時点で `t1`(reg0) と `t2`(reg1) が埋まり、
`t4` は両方生きているうちに始まるのでスピル。`t1` が死んだあと `t3` が
その reg0 を再利用しています。グラフ彩色ほど最適ではありませんが、
**ほぼ一直線になめるだけ**なので圧倒的に速く、「コンパイル時間が実行
時間に直結する」JIT の事情にぴったり合います。これが多くの JIT で
線形走査が好まれる理由です。実際、線形走査を提案した Poletto と Sarkar は、
高速なグラフ彩色器より割り当てが数倍速いうえ、生成コードの実行速度は
積極的なグラフ彩色のおおむね 10% 以内に収まったと報告しています
[](#cite:poletto1999)。速さと品質のトレードオフが、JIT に都合よく傾いて
いるわけです。

### スピルする値の選び方：いちばん遠い使用を追い出す

レジスタが尽きたとき、**どの値をメモリへ追い出すか**の選び方で、生成
コードの質が変わります。スピルした値は、使うたびにメモリから読み直す
（リロード）必要があるので、**近いうちに使う値をスピルすると損**です。

よく使われるのが、**いちばん遠くまで使わない値を追い出す**方針です。
これは仮想記憶のページ置換で知られる **Belady の最適置換**と同じ発想で、
「次に使うのが最も先の値」を犠牲にすれば、近い将来のリロードを最小化
できます。線形走査では、現在生きている区間（`active`）のうち**終了点が
最も遠いもの**を、いま割り当てたい区間と比べ、遠いほうをスピルします。
原論文も「現在地から最も遠くで終わる区間（the interval that ends last）」を
スピル対象に選んでおり、`active` を終了点順に保てばこれを素早く取り出せる、
としています[](#cite:poletto1999)。

```ruby
# レジスタが尽きたとき、active の中で最も遠くまで使う区間をスピルする
def alloc_or_spill(var, finish, active, alloc, free)
  if free.any?
    reg = free.shift
    alloc[var] = [:reg, reg]
    active << [finish, var]
  else
    # active を終了点の遠い順に並べ、先頭（最も遠い）を取り出す
    far_end, far_var = active.max_by { |end_pt, _| end_pt }
    if far_end > finish
      # 既存のほうが遠くまで生きる → そいつを追い出し、レジスタを var へ
      reg = alloc[far_var][1]
      alloc[far_var] = [:spill]
      alloc[var] = [:reg, reg]
      active.delete([far_end, far_var])
      active << [finish, var]
    else
      alloc[var] = [:spill]            # 自分のほうが遠い → 自分がスピル
    end
  end
  [active, alloc, free]
end

active, alloc, free = [[10, :t1], [3, :t2]], { t1: [:reg, 0], t2: [:reg, 1] }, []
# いま finish=5 の t3 を入れたい。レジスタは満杯
alloc_or_spill(:t3, 5, active, alloc, free)
alloc.each { |v, a| puts "#{v} => #{a.inspect}" }
# t1 => [:spill]          ← 終了 10 で最も遠い t1 を追い出した
# t2 => [:reg, 1]
# t3 => [:reg, 0]         ← t1 が空けた reg0 をもらう
```

終了点 `10` でいちばん遠い `t1` を追い出し、その reg0 を、より早く死ぬ
`t3` に与えました。「遠い値ほどレジスタに居座る価値が低い」という直感を
そのまま規則にしたものです。

### 生存区間の分割：途中で持ち場を替える

「ある値はレジスタかスピルか、二択で一生固定」だと無駄が出ます。長く生きる
値でも、**実際に使われていない長い空白**があれば、その間だけレジスタを
譲ればよいはずです。そこで**生存区間の分割**（live range splitting）を
します。一つの値の生存区間を複数に切り、ある区間ではレジスタに、別の区間では
スピルに、と**持ち場を替えられる**ようにします。

```ruby
# 生存区間を「使用がない長い空白」で 2 つに割る
def split_at_gap(uses, threshold)
  segments = []
  start = uses.first
  uses.each_cons(2) do |a, b|
    if b - a > threshold            # 空白が閾値より大きければそこで分割
      segments << [start, a]
      start = b
    end
  end
  segments << [start, uses.last]
  segments
end

# 0,1 で使われ、その後ずっと空いて 20,21 で再び使われる値
p split_at_gap([0, 1, 20, 21], 5)
# [[0, 1], [20, 21]]   ← 空白 1..20 の間はレジスタを手放せる
```

`1` から `20` までの長い空白でレジスタを手放し、その間ほかの値に使わせて
おいて、`20` でリロードして使う —— と割り当てが柔軟になります。分割の
切れ目では「レジスタ → メモリ」「メモリ → レジスタ」の転送命令が要るので、
**分割を増やすと転送が増える**トレードオフがあります。切りすぎず、空白の
大きいところだけ切るのが勘所です。

### SSA ベースのレジスタ割り当て

近年は、**SSA を保ったままレジスタ割り当てをする**手法が注目されています
[](#cite:wimmer2010)。
SSA 形式のプログラムは、生存区間の重なり方（干渉グラフ）が **chordal
graph**（弦グラフ。長さ 4 以上のサイクルには必ず「弦」=サイクルを横切る
辺がある、という素直な構造のグラフ）になることが知られ、これは「グラフ
彩色が線形時間で最適に解ける」族です。AOT で重かったグラフ彩色が、SSA の
おかげで**速く**解けるようになる —— JIT にとっては願ってもない性質です。
ただしこれが効くのは**彩色の段に限った**話で、φ 除去のためのコピー挿入や、
レジスタが尽きたときのスピル、特定の物理レジスタを強制する**プリカラー**
などは、また別の難しさを抱えた問題として残ります。

SSA ベースの割り当ては、おおむね「**φ 関数のところまでは SSA で割り当てを
決め、最後に SSA から出る処理でコピーを入れる**」流れになります。前節で見た
out-of-SSA のコピー挿入と、レジスタ割り当てが自然につながるわけです。

> [!TIP]
> 「割り当て → SSA から出る」の順にするか、「SSA から出る → 割り当て」の
> 順にするかで設計が変わります。先に SSA から出ると φ がコピーになって
> 命令が増え、割り当て対象が膨らむ。SSA のまま割り当てると構造の良さを
> 使える代わりに、実装は手が込みます。[Cranelift](frameworks.md) のような
> 実用 JIT バックエンドが、このあたりをどう割り切っているかは
> [フレームワーク](frameworks.md)の章も参照してください。

> [!NOTE]
> レジスタ割り当ては、AOT・JIT を問わずコンパイラの華のひとつです。
> 本書では「JIT は時間がないので、最適性より速さで線形走査を選ぶ」と
> いう設計判断の側面に注目しました。割り当てアルゴリズムそのものを
> 深く学びたい場合は、コンパイラの教科書[](#cite:aho2006)が詳しいです。

## 命令選択とピープホール：最後のひと削り

レジスタが決まったら、いよいよ IR の各命令を**実際の機械語**へ対応づけ
ます。これが**命令選択**（instruction selection）です。一つの IR 命令に
対し、機械語の選び方は一通りとは限りません。賢く選べば、命令数を減らし、
速い命令を使えます。

最も分かりやすいのが**強度低減**（strength reduction）です。`x * 2` は
乗算器を回すより**左シフト** `x << 1` のほうが速い。`x % 16` は `x & 15` に
化けます。乗除算は CPU の中でも重い命令なので、定数が 2 のべき乗ならシフトや
マスクへ落とすのが定番です。ただし**割り算は要注意**です。`x / 8` を算術右
シフト `x >> 3`（`sar`）にしてよいのは、`x` が**非負と分かっている**か、言語の
整数除算が**床関数**（floor、負の無限大方向。Ruby など）で定義されている場合
だけです。多くの言語の整数除算は**ゼロ方向への切り捨て**で、負数では
`(-1) / 8 == 0` に対し `(-1) >> 3 == -1` とずれます（**符号なし**整数なら論理
右シフト `shr` に落とせます）。

```ruby
# IR 命令を機械語風の命令へ落とす素朴な命令選択＋強度低減
def select(insn)
  dst, op, a, b = insn
  case [op, b]
  in [:mul, 1] then [dst, :mov, a]                 # x*1 → ただの転送
  in [:add, 0] then [dst, :mov, a]                 # x+0 → ただの転送
  in [:mul, Integer => k] if k > 1 && (k & (k - 1)).zero?   # k が 2 のべき乗
    shift = Math.log2(k).to_i
    [dst, :shl, a, shift]                          # x*k → x << log2(k)
  in [:div, Integer => k] if k > 1 && (k & (k - 1)).zero?
    [dst, :sar, a, Math.log2(k).to_i]              # x/k → x >> log2(k)（非負/床除算が前提）
  else insn
  end
end

[[:t1, :mul, :x, 8], [:t2, :div, :y, 4], [:t3, :mul, :z, 1]].each { |i| p select(i) }
# [:t1, :shl, :x, 3]    ← x*8 が左シフト 3 に
# [:t2, :sar, :y, 2]    ← y/4 が算術右シフト 2 に
# [:t3, :mov, :z]       ← z*1 がただの転送に
```

もう一つが**ピープホール最適化**（peephole optimization）です。生成
した機械語列を、小さな**のぞき窓**（数命令ぶん）でなめ、定型のもったいない
並びを良い並びへ書き換えます。連続する命令をまたいだ最適化なので、命令
選択のあと、隣り合う命令の組み合わせを見て効かせます。

```ruby
# のぞき窓 2 命令でパターンを潰すピープホール
def peephole(code)
  out = []
  i = 0
  while i < code.size
    a, b = code[i], code[i + 1]
    case [a, b]
    in [[dst1, :mov, src], [dst2, :mov, x]] if dst1 == x   # mov t,s; mov u,t
      out << [dst1, :mov, src] << [dst2, :mov, src]        #  → u も直接 s から
      i += 2
    in [[dst, :add, x, y], [dst2, :add, z, _]] if false then i += 1   # （FMA 等の枠）
    else
      out << a; i += 1
    end
  end
  out
end

p peephole([[:t, :mov, :s], [:u, :mov, :t]])
# [[:t, :mov, :s], [:u, :mov, :s]]   ← u を s から直接ロード（t 経由をやめる）
```

このほか、`a + b * c` を 1 命令の**積和（FMA 風）**へまとめる、よく使う
定数を毎回ロードする代わりに一度ロードして使い回す、ゼロ初期化を
`xor reg, reg`（同じレジスタの排他的論理和は速くて短い）で行う、といった
細工が、すべてこの「最後のひと削り」に属します。

> [!TIP]
> 命令選択とピープホールは、効果は小さく見えても**ホットループでは積もり
> ます**。一方で、IR の段階で済む最適化（定数畳み込みや GVN）と役割が
> かぶる部分もあります。原則は「**意味の最適化は IR で、機械語特有の事情
> （どの命令が速いか、どんな並びが短いか）はバックエンドで**」と切り分ける
> ことです。前者は CPU 非依存に書け、後者だけが x86 / ARM ごとに変わります。

## まとめ

- JIT は機械語へ落とす前に**中間表現**の上で最適化する。CPU 非依存で、
  データの流れが見やすい形が望まれる。
- **SSA 形式**（変数は一度だけ代入）は、データの流れを一意にし、
  定数畳み込み・共通部分式削除・**デッドコード削除**・**コピー伝播**・
  **大域値番号付け**・**ループ不変式の巻き上げ**を劇的に簡単にする。合流点は
  **φ 関数**で表し、機械語化の前に **SSA から出る**処理でコピーへ落とす
  （並列コピーの逐次化に注意）。
- **範囲解析**で `0 <= i < n` が示せれば**境界チェックを除去**できる。
  境界チェックも型ガードも「条件が恒真なら消せるガード」で、ガードを
  **第一級の命令**にすると巻き上げ・統合・除去がふつうの IR 変換になる。
  ただし**消した値には[脱最適化](deopt-osr.md)用の復元レシピを残す義務**がある。
- **Sea of Nodes** は順序を捨ててデータ依存だけ残し、命令の並べ替えの
  自由を得る。C2・Graal が採用。
- 機械語へ落とす際の**レジスタ割り当て**は、JIT では最適性より速さを
  取り、**線形走査**を使うことが多い。スピルは**最も遠くまで使わない値**を
  選び（Belady 風）、**生存区間の分割**で持ち場を替え、**SSA ベース**だと
  彩色が速く解ける。
- 最後に**命令選択**（`x*2`→シフト等の強度低減）と**ピープホール**で
  機械語特有のひと削りをする。意味の最適化は IR で、CPU 固有の細工は
  バックエンドで、と切り分ける。

中間表現が整ったところで、次の問題は「**そもそもどこをコンパイルするか**」
です。熱い場所を見つける技術 —— プロファイリングへ進みます。
