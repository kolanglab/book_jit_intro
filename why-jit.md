# なぜ JIT コンパイルするのか

JIT の技法に入る前に、いちばん大切な問いに答えておきます。
そもそも、なぜ実行時にコンパイルすると速くなるのか。そして
何から何へコンパイルするのか。この章の答えが、本書全体の「なぜ」に
なります。

## インタプリタは何にお金を払っているか

まず、JIT が倒そうとしている相手（インタプリタ）のコスト構造を
見ます。典型的なバイトコードインタプリタは、こんな形のループです。

```ruby
# スタックマシン型バイトコードインタプリタの心臓部
def run(insns, stack = [])
  pc = 0
  while (insn = insns[pc])         # ① 命令を取ってくる（フェッチ）
    case insn[0]                   # ② 種類で分岐する（ディスパッチ）
    when :push then stack.push(insn[1])
    when :add
      b = stack.pop; a = stack.pop # ③ オペランドを取り出す
      stack.push(a + b)            # ④ 実際の計算
    when :sub
      b = stack.pop; a = stack.pop
      stack.push(a - b)
    when :mul
      b = stack.pop; a = stack.pop
      stack.push(a * b)
    end
    pc += 1                        # ⑤ 次へ進む
  end
  stack.pop
end

# (1 + 2) * 3 を計算するバイトコード
prog = [[:push, 1], [:push, 2], [:add], [:push, 3], [:mul]]
p run(prog)   # => 9
```

`1 + 2` というたった 1 回の加算のために、インタプリタは①フェッチ、
②`case` による分岐、③スタックの出し入れ、⑤`pc` の更新と、
本来やりたい計算（④）の周りで大量の事務作業をこなしています。
本当にやりたいのは CPU の `add` 命令 1 つぶんの仕事なのに、その何倍もの
時間を「次は何をするんだっけ」の管理に使っているのです。この事務作業を
**インタプリタのオーバーヘッド**と呼びます。代表的なものを並べます。

- **ディスパッチのコスト**：命令ごとに `case`（実体は条件分岐の連鎖か
  ジャンプテーブル）を通ります。CPU の分岐予測が外れると特に高くつきます。
- **オペランドアクセスのコスト**：値を CPU レジスタではなく、メモリ上の
  スタック配列に出し入れします。
- **ボクシングのコスト**：動的言語では整数 `1` ですら、意味の上では
  「オブジェクト」です。素朴な実装ではメモリ上の「箱」に入れますが、
  実装表現は処理系で違い、CPython は実際に箱に入れる一方、CRuby は
  小さな整数をポインタのビットに埋め込む**タグ付き即値**で表します
  （後で詳しく見ます）。
- **動的ディスパッチのコスト**：`a + b` の `+` が何を意味するかは
  `a` の型しだいで、毎回メソッドを探索する必要があります。

ここで桁感を持っておくと、以降の議論がぐっと具体的になります。
CPU の `add` のような算術命令は、L1 キャッシュに乗ったデータに対して
おおむね **1 サイクル**で終わります（今どきの CPU なら 1 秒間に数十億
サイクル回ります）。ところがインタプリタでこの 1 命令を実行するまでの
事務作業は、ざっと次のように積み上がります。

- フェッチ＋デコード：`insns[pc]` の配列アクセスと `insn[0]` の取り出しで
  数命令、メモリアクセスを伴えば数サイクル。
- ディスパッチの分岐：`case` の比較が分岐予測ミスを起こすと、
  パイプラインのフラッシュで 10〜20 サイクル前後を失います。命令の
  種類はループのたびにバラバラなので、ここの予測は当たりにくいのです。
- スタックの出し入れ：`pop` 2 回と `push` 1 回で、メモリへの読み書きが
  数回。
- `pc += 1`：足し算とループ条件の判定。

つまり 1 サイクルで終わる仕事に、その十数倍から数十倍を払っている、
という勘定です。「インタプリタは遅い」を漠然と言うのではなく、
「`add` 1 サイクルに対し、ディスパッチの分岐ミスだけで 10〜20 サイクル」
と数で押さえておくと、あとで「ガードは数命令で済む」という話の損得が
ピンと来ます。

JIT の仕事を一言でいえば、これらのオーバーヘッドを、実行時に観測した
事実で消し去ることです。順に見ていきましょう。

## ディスパッチを消す

`run` のループは、命令を 1 つ実行するたびに「次の命令は何か」を `case`
で判定します。しかし `prog` を一度コンパイルしてしまえば、命令の並びは
もう分かっているのです。`[:push,1] [:push,2] [:add] ...` という順番は
固定です。ならば、その並びをそのまま展開した一本道のコードを作れば、
ループも `case` もいりません。

```ruby
# prog を Ruby のソースコードへ「コンパイル」する素朴な例
def compile(insns)
  lines = ["stack = []"]
  insns.each do |insn|
    case insn[0]
    when :push then lines << "stack.push(#{insn[1]})"
    when :add  then lines << "b = stack.pop; a = stack.pop; stack.push(a + b)"
    when :sub  then lines << "b = stack.pop; a = stack.pop; stack.push(a - b)"
    when :mul  then lines << "b = stack.pop; a = stack.pop; stack.push(a * b)"
    end
  end
  lines << "stack.pop"
  lines.join("\n")
end

prog = [[:push, 1], [:push, 2], [:add], [:push, 3], [:mul]]
puts compile(prog)
# stack = []
# stack.push(1)
# stack.push(2)
# b = stack.pop; a = stack.pop; stack.push(a + b)
# stack.push(3)
# b = stack.pop; a = stack.pop; stack.push(a * b)
# stack.pop

f = eval("lambda { #{compile(prog)} }")
p f.call   # => 9
```

これは「Ruby のバイトコードを Ruby のソースへコンパイルし、`eval` で
機械語ならぬ Ruby コードにした」ものです。本物の JIT はこの行き先が
機械語になるだけで、発想は同じです。ループとディスパッチが消え、
命令の並びが直線のコードになりました。この「実行する命令列が前もって
分かっているなら、分岐をたたんで一本道にできる」というのが、コンパイルが
速さを生む最初の理由です。

> [!NOTE]
> 上の `eval` を使った例は、JIT というより「ソース・トゥ・ソース変換」
> ですが、観測した命令列を別の実行形式へ翻訳して直接走らせるという
> JIT の骨格そのものです。次章[最小の JIT を作る](hello-jit.md)では、
> 行き先を本物の x86-64 機械語にします。

ディスパッチを軽くする手は、機械語まで行かなくても段階があります。
「AかBか」で並べると次のようになります。

- **(A) スレッデッドコード化する**：`case` の連鎖をやめ、命令ごとの処理
  関数のアドレスを並べた配列を、間接ジャンプでたどっていきます。比較の
  連鎖が 1 回の間接ジャンプになるので、命令数は減ります。ただし行き先が
  毎回変わる間接ジャンプは、それ自体が分岐予測ミスを起こしやすい。
- **(B) 命令の並びを直線コードに展開する**（上の `compile` の発想）：
  ディスパッチそのものが消えます。隣り合う命令の境界をまたいで最適化
  できる余地まで生まれます。代わりに、コードを生成する手間と、
  生成コードがメモリを食う問題が出てきます。

実装上のもう一つの定番が**スーパー命令**（superinstruction）です。
`push; push; add` のように頻出する並びを 1 つの命令に融合し、その分の
ディスパッチをまとめて省きます。

```ruby
# 頻出パターン push n; add を「定数加算」命令に融合する
when :add_imm                      # 1 命令で「スタックトップに n を足す」
  stack.push(stack.pop + insn[1])  # push と add のディスパッチを 1 回に圧縮
```

スーパー命令は「どの並びが頻出か」をプロファイルしてから作ると効きます。
何でも融合すると命令種が爆発し、ディスパッチ表が大きくなって逆効果に
なるためで、ここにも「種類を増やす手間」対「ディスパッチ削減の利得」の
損得があります。

> [!NOTE]
> ディスパッチを消すだけでも効きますが、本物の JIT は機械語へ降りるので、
> さらにレジスタ割り当てや後述のボクシング除去まで踏み込めます。どの
> 単位（命令／基本ブロック／メソッド／トレース）でコンパイルすると
> ディスパッチを最大限たためるかは[コンパイルの単位](compilation-units.md)で
> 扱います。

## 型による分岐を消す（投機と特殊化）

動的型付け言語で大きなコストになるのは、`a + b` の意味が実行するまで
決まらないことです。`a` と `b` が整数なら整数加算、文字列なら連結、
配列なら結合、独自クラスなら `+` メソッド呼び出し。インタプリタは毎回
この場合分けをしなければなりません。

```ruby
# 動的な + が毎回やっていること（概念図）
def generic_add(a, b)
  if a.is_a?(Integer) && b.is_a?(Integer)
    a + b                       # 整数加算
  elsif a.is_a?(String) && b.is_a?(String)
    a + b                       # 文字列連結
  else
    a.send(:+, b)              # 一般のメソッド探索＆呼び出し
  end
end
```

ところが実際のプログラムでは、ある場所の `a + b` は毎回同じ型で
あることがほとんどです。ループカウンタの加算なら、`a` も `b` も整数に
決まっています。JIT はこの観測を使います。「ここの `+` は 100 回連続で
整数同士だった。ならば整数だと決め打ちした高速なコードを出そう。
ただし、念のため実際に整数かどうかのチェック（ガード）を一行だけ
置いて、もし違ったら安全な経路へ逃がそう」。

```ruby
# JIT が出すコードのイメージ：型を決め打ちし、ガードで守る
def specialized_add(a, b)
  # ガード：賭けの前提が成り立つか、1 回だけ確かめる
  unless a.is_a?(Integer) && b.is_a?(Integer)
    return deoptimize(a, b)   # 外れたら遅い汎用版へ戻る（脱最適化）
  end
  a + b                        # 当たれば、場合分けなしの一直線
end
```

この「たぶんこうだろうと決め打ちし、念のためのチェックを置き、
外れたら戻る」という戦略を**投機的最適化**（speculative optimization）と
呼びます。決め打ちすること（型の前提を焼き込むこと）を**特殊化**
（specialization）、念のためのチェックを**ガード**（guard）、外れたときに
戻ることを**脱最適化**（deoptimization）と呼びます。
この三つ組は JIT のいたるところに現れる本書の主役です。この節の投機の
発想がすべての出発点で、ガードと戻りの具体は[脱最適化と OSR](deopt-osr.md)で、
観測のしかたは[プロファイリング](profiling.md)で詳しく扱います。

### ガードは機械語で何命令か

「ガードは安い」と言うとき、具体的に何命令なのかを見ておきましょう。
多くの処理系は小さな整数を**タグ付き整数**（次節の fixnum）で表します。
たとえば「ポインタの最下位ビットが 1 なら整数」という約束なら、
「整数かどうか」のガードは、機械語でおおむね 2 命令です。以下の例に出る
`rax` や `rcx` は x86-64 の汎用レジスタで、`deopt_stub` は脱最適化処理へ
飛ぶために置かれた小さな中継コード（**スタブ**、stub）です
（機械語の読み書きは次章で実際に手を動かします）。

```
    test  rax, 1        ; 最下位ビットを見る（整数タグが立っているか）
    jz    deopt_stub     ; 立っていなければ脱最適化スタブへ脱出
    ; ここから先は「rax は整数」と決め打ったコード
```

クラスを確かめるガードでも、`obj` の先頭にあるクラスポインタを読んで
期待値と比べるだけなので、せいぜい数命令です。

```
    mov   rcx, [rax]            ; obj のクラスポインタを読む
    cmp   rcx, [expected_class]  ; 期待するクラスか
    jne   deopt_stub             ; 違えば脱出
```

要は `cmp`（または `test`）＋ `jne`（または `jz`）の数命令です。しかも
ガードはほとんど当たる（＝分岐しない）ので、CPU の分岐予測はこの分岐を
「飛ばない」とほぼ確実に当て、実効コストはさらに小さくなります。

### 投機の損得を数で見る

この 2〜3 命令のガードが、何を消しているかを並べると損得が見えます。
ガードを置かない汎用版（前掲の `generic_add`）は、毎回こうなります。

- 型の場合分け：`is_a?` 相当のチェックを複数回（数命令×分岐の本数）。
- メソッド探索：`+` のメソッドをクラスのテーブルから引く。インライン
  キャッシュが効かなければ、ハッシュ表の探索で十数〜数十サイクル。
- 間接呼び出し：見つけた関数への間接ジャンプで、分岐予測ミスなら
  10〜20 サイクル。

ざっくり、汎用版が数十サイクルかかる一方、特殊化版は
ガード数命令（実効数サイクル）＋ `add` 1 サイクルで済みます。
当たる確率を `p`、特殊化版の費用を `c_fast`、汎用版を `c_slow`、外れた
ときの脱最適化を `c_deopt` とすると、期待費用は

```
特殊化:  c_fast + (1 - p) * c_deopt
汎用:    c_slow
```

`c_fast ≈ 5`、`c_slow ≈ 40`、`c_deopt ≈ 1000`（スタックの作り直しなどで
重い）と置くと、`p = 0.99` なら特殊化の期待費用は `5 + 0.01*1000 = 15`
で、汎用の `40` に対して十分得です。逆に `p = 0.9` まで落ちると
`5 + 0.1*1000 = 105` となり、汎用版より高くつきます。脱最適化が重い
ほど、投機は「ほとんど外れない」前提でしか割に合いません。この数式が、
本書で繰り返す「どれだけ当たるならコンパイルするか」の根っこです。

> [!TIP]
> 同じガードが連続する場面では、**ガードの巻き上げ**（hoisting）が効きます。
> ループ内の `a + b` が毎回同じ変数なら、整数ガードをループの外へ 1 回だけ
> 出し、ループ本体はガードなしの裸の `add` にできます。100 万回回るループの
> ガードを 1 回に減らせるなら、投機の損得は劇的に好転します。ただし
> 巻き上げてよいのは、ループ本体がその変数の型を変えない、と確かめられる
> 場合だけです。勝手な並べ替えは正しさを壊します。巻き上げが許される条件は
> [中間表現](ir.md)と[トレーシング JIT の仕組み](tracing.md)で扱います。

> [!NOTE]
> 「観測にもとづいて最適化する」だけなら、AOT コンパイラにも **PGO**
> （profile-guided optimization、プロファイルにもとづく最適化）があります。
> 決定的な違いは、賭けが外れたときです。AOT＋PGO は、プロファイルが
> 実態とずれても生成済みのコードで走り続けるしかないので、外れても遅く
> ならない控えめな最適化に寄ります。JIT は脱最適化でいつでも安全な実装へ
> 戻れるからこそ、「ほとんど当たる」程度の観測にも大胆に賭けられるのです。

### 脱最適化のために何を覚えておくか

ガードが外れて汎用版へ戻るとき、JIT は「最適化前のインタプリタが、いま
この瞬間どんな状態だったか」を**再現**しなければなりません。レジスタに
裸で持っていた整数を箱に詰め直し、消したローカル変数を復元し、正しい
バイトコード位置から実行を再開します。この状態復元のための情報を
どこまで持つかが、コンパイラの自由度と引き換えになります。「AかBか」で
言えば、

- **(A) コンパイル結果に制約を与えて、状態復元を容易にする**：ガード地点で
  「この変数はこのレジスタ／スタック位置にある」と決め打てる形にコードを
  生成しておく。復元は簡単になりますが、最適化の自由度（自由なレジスタ
  割り当てや命令の並べ替え）が縛られます。
- **(B) 膨大な脱最適化マップ（deopt map）を持つ**：各ガード地点ごとに、
  「最適化後のどのレジスタ／スタック位置が、最適化前のどの変数に対応するか」
  の対応表を全部記録する。最適化は自由にできますが、メタデータがコードと
  同じくらい、ときにそれ以上に膨らみます。

実際の JIT は両者の中間を取ります。ガードは**頻度の高いコードパス**に
集中するので、「外れたとき用」の復元情報は実行頻度の低い側に追いやって
（lazy に作る、圧縮して持つ）、ホットなコードの速度とメモリを両立させます。
脱最適化マップの作り方と OSR（実行中の関数を最適化版へ／から乗り換える）
の具体は[脱最適化と OSR](deopt-osr.md)で扱います。

> [!IMPORTANT]
> 投機が成立するのは、ガードのコストが、場合分けを消した利得より
> ずっと小さいからです。整数かどうかのチェックは機械語数命令ですが、
> それで型の場合分け、メソッド探索、分岐予測ミスをまとめて消せます。
> 「ほとんど当たる賭けなら、外れたときの保険を払ってでも決め打ちした
> ほうが得」、これが投機の損得勘定です。保険料にあたる脱最適化の費用
> `c_deopt` が大きいのは、この状態復元が重いからで、前掲の損得勘定で
> `c_deopt` を大きく見積もったのはこのためです。「めったに外れない賭けに
> だけ投機する」という方針は、裏を返せば「復元情報を安く済ませられる」
> という実装上の利得にもつながっています。

## ボクシングを消す

もう一つの大きなコストが**ボクシング**（boxing）です。Ruby や Python
では、整数 `1` も意味の上では「オブジェクト」です。ただし実装表現は
処理系で違います。素朴な実装や CPython の `PyLong` のような表現では、
値そのもの（数値のビット列）だけでなく、「これは整数オブジェクトですよ」
というクラス情報などを持った箱がメモリ上にあり、変数はその箱を指す
ポインタを持ちます。

```
変数 a ──→ ┌─────────────┐
           │ クラス: Integer │   ← 箱（ボックス）
           │ 値: 1          │
           └─────────────┘
```

この表現では、整数 1 つを足すだけでも、箱からポインタをたどって値を
取り出し（アンボックス）、計算し、結果をまた箱に詰める（ボックス）必要が
あります。短命な整数のために箱を作り続けると、ガベージコレクションの
負担にもなります。

JIT は、型を整数だと特殊化できたなら、その区間では箱を開けっぱなしの
生の整数（CPU レジスタ上の 64 ビット値）で計算を進められます。値が
オブジェクトとして外に見える必要が出たときだけ、箱に詰め直す。これを
**アンボクシング**（unboxing）と呼びます。数値の箱に限らず、オブジェクト
全体を割り当てずにフィールドごとの裸の値へばらしてしまう一般形は
**スカラ置換**（scalar replacement）と呼ばれ、後の章で再登場します。

この利得は、整数よりむしろ浮動小数点で大きく効きます。小さな整数は
次節のタグ付き即値でポインタの空きビットに埋め込んで箱を避けられますが、
64 ビットの浮動小数点（`Float`）はビット列をまるごと使うため、ポインタの
わずかな空きビットには入りません。つまり `Float` は素朴な実装ではほぼ
必ず箱が要り、数値ループでアンボクシングが効く度合いは整数以上です。
たとえば V8 は、要素がすべて `Float` だと分かった配列を、箱への
ポインタの配列ではなく、生の 64 ビット浮動小数点を並べた配列として
持ち、要素の読み書きから箱の出し入れを丸ごと消します。

### タグ付き整数（fixnum / Smi）の具体

一方で、CRuby や V8 など多くの処理系は、そもそも小さな整数を箱に入れません。
ポインタの最下位ビットを型タグに使うのが定番です。本物のオブジェクトはアドレスが
8 バイト境界に整列していて最下位 3 ビットが常に 0 になるので、その空き
ビットに「これは整数だ」という印を立てられます。たとえば

- **最下位ビットが 1 なら整数**、その値は `(v >> 1)`。
- 最下位ビットが 0 なら、`v` はそのまま本物のオブジェクトへのポインタ。

という約束です。これを Ruby では fixnum、V8 では **Smi**（small integer）
と呼びます。ただしタグの向きは処理系ごとの約束で、上の「最下位ビットが
1 なら整数」は CRuby の方式です。V8 の Smi は逆に、最下位ビットが 0 の
ほうが整数で、1 が立っていたらヒープオブジェクトへのポインタです。
以下では CRuby の向きで話を進めます。この表現だと、整数 `n` を作るのは
「左 1 ビットシフトして最下位に 1 を立てる」だけ、値を取り出すのは
「右 1 ビットシフト」だけです。

以下では、`rdi` に整数 `n`、`rsi` と `rdi` にタグ付きの `a` と `b`（それぞれ
`2a+1` と `2b+1`）が入っているものとします。

```
    ; n をタグ付き整数の表現に詰める（タグ付け）
    lea   rax, [rdi*2 + 1]   ; rax = n*2 + 1（1 命令で済むことも多い）

    ; タグ付き整数どうしの加算 a + b は、タグを外さずにこう書ける
    lea   rax, [rsi + rdi]   ; (2a+1)+(2b+1) = 2(a+b)+2
    dec   rax                ; 末尾を +1 に直して 2(a+b)+1 ＝ 正しいタグ付き整数
```

箱を作る、ポインタをたどる、GC で回収する、という重い処理がまるごと
消えて、タグの付け外しが 1〜2 命令にまで縮みます。これだけでも
大きいのですが、それでもタグの付け外しは残ります。

### JIT ならタグすら消せる

ループの中で整数だけを扱うと分かっている区間では、JIT はタグの付け外し
すら省き、生の 64 ビット値のまま回せます。

```ruby
# 観測：このループの sum と i は最初から最後まで整数だった
def sum_upto(n)
  sum = 0
  i = 1
  while i <= n
    sum += i      # タグ付き整数でも毎回シフト＆タグ調整が要る
    i += 1
  end
  sum
end
```

このループを「`sum` と `i` は整数」と特殊化できれば、本体は素の
`add rax, rbx` の連続になり、シフトやタグ調整が消えます。境界（ループに
入る前にタグを外し、結果を返す前にタグを付け直す）でだけタグ操作が残る、
という形に追い込めるわけです。ここでも「AかBか」の設計判断があります。

- **(A) 生の値で持つ**：本体は最速。ただし途中で `Integer` を超える
  巨大数（多倍長）に化けたり、整数以外が来る可能性があるなら、ガードか
  オーバーフロー検査が要ります。`add` のあとに `jo`（overflow なら脱出）を
  1 命令足す、というのが典型です。
- **(B) タグ付きのまま持つ**：表現を変えずに済み、オブジェクトとして
  外へ漏れても安全。代わりに毎回シフト 1〜2 命令を払い続けます。

JIT は前者を狙い、後者へ安全に戻れるよう脱最適化を用意する、という
組み合わせを取ります。

> [!NOTE]
> 「値をどう表現するか（箱に入れるか、ポインタに埋め込むか、生の値で
> 持つか）」は、それ自体が大きなテーマです。値表現とボクシングの詳細は
> 姉妹書『言語処理系のデータ構造入門』の値の表現の章が詳しいです。
> 本書では「JIT が型を特殊化すると、箱を外して生の値で計算できる」と
> いう利得の面だけ押さえておけば十分です。

## 間接呼び出しとインライン化

メソッド呼び出しもコストの巣窟です。`obj.foo` は、`obj` の型（クラス）を
見て、そのクラスのメソッドテーブルから `foo` を探し、見つかった関数へ
ジャンプします。この探して飛ぶ処理は、CPU にとって行き先が読みにくい
**間接呼び出し**で、しかも探索自体にも時間がかかります。

### 探索結果を覚えるインラインキャッシュ

メソッド探索をまるごと毎回やるのは無駄なので、前回ここで探した結果を
その場に覚えておくのが**インラインキャッシュ**（inline cache, IC）です。
呼び出し地点ごとに「直前に来たクラスと、そのとき見つかった関数」を 1 組
記録しておき、次に来たクラスがそれと同じなら探索を飛ばします。

```ruby
# 呼び出し地点 obj.foo に貼り付くインラインキャッシュ（概念）
cached_class = nil; cached_method = nil
->(obj) {
  klass = obj.class
  if klass.equal?(cached_class)        # ヒット：クラス比較 1 回だけ
    cached_method                      # 探索を飛ばして即決
  else
    m = klass.lookup(:foo)             # ミス：本格的に探索して…
    cached_class, cached_method = klass, m  # …結果を覚える
    m
  end
}
```

来るクラスが 1 種類だけの状態を**単態**（monomorphic。モノモーフィック
とも）と呼びます。これがいちばん速く、クラス比較 1 回でヒットします。
来るクラスが 2〜数種類に増えると**多態**（polymorphic）で、覚える組を
複数（PIC, polymorphic inline cache）にして、数回の比較で当てます
[](#cite:holzle1991)。
さらに散らかってクラスがころころ変わると**メガモーフィック**
（megamorphic）で、もはやキャッシュを諦めて毎回探索に戻ります。

| 状態 | 来るクラス数 | 取る手 | 概算コスト |
|------|------------|--------|-----------|
| 単態 | 1 | キャッシュ 1 組、比較 1 回 | 数命令 |
| 多態 | 2〜数種 | 複数組を線形に比較 | 比較の本数ぶん |
| メガモーフィック | 多数 | 諦めて毎回探索 | 十数〜数十サイクル |

JIT の投機は単態の観測と相性が抜群です。「ここは
99% 単態だった」なら、キャッシュ比較すら 1 本のクラスガードへ
焼き込めます。どの地点がどの状態かを測るのが[プロファイリング](profiling.md)の
仕事で、観測結果が次の脱仮想化とインライン化の前提になります。

### 脱仮想化（devirtualization）

JIT は「ここの `obj` はいつも `Point` クラスだった」と観測すれば、
探索を省いて `Point#foo` へ直接飛べます。間接呼び出しを直接呼び出しに
変えるこの操作を**脱仮想化**（devirtualization）と呼びます[](#cite:detlefs1999)[](#cite:ishizaki2000)。手順は単純です。

1. プロファイルで、その呼び出し地点が単一クラス（単態）だと
   確認する。
2. そのクラスを期待するクラスガードを 1 本置く（前述の `cmp`＋`jne`）。
3. ガードを通った先は、もう探索も間接ジャンプもいらない直接呼び出しに
   する。
4. ガードが外れたら脱最適化で安全な汎用呼び出しへ戻す。

```ruby
# obj.foo を脱仮想化したイメージ
return deopt(obj) unless obj.is_a?(Point)  # クラスガード（2〜3 命令）
Point.instance_method(:foo).bind(obj).call # Point#foo を直接（探索なし）
```

### インライン化（最適化の母）

さらに `Point#foo` の中身が小さければ、呼び出しすらやめて呼び出し元に
展開してしまえます。これが**インライン化**（inlining）です。脱仮想化で
行き先が 1 つに定まっているからこそ、その本体を丸ごと貼り込めるわけです。

```ruby
class Point
  def initialize(x, y) = (@x, @y = x, y)
  def x = @x            # とても小さいメソッド
end

# 元のコード
def distance_sq(p)
  p.x * p.x            # p.x はメソッド呼び出し
end

# JIT がインライン化したあとのイメージ（p はいつも Point だと観測した）
def distance_sq_jit(p)
  # ガード：p は本当に Point か？
  return deopt(p) unless p.is_a?(Point)
  # 呼び出しを消し、p.x を @x の読み出しに展開する
  p.instance_variable_get(:@x) * p.instance_variable_get(:@x)
end
```

インライン化は、それ自体の呼び出しコストを消すだけでなく、展開した
コードどうしをまたいだ最適化を可能にする点が重要です。呼び出しの
向こう側が見えるようになると、共通部分式の削除、定数畳み込み、不要な
ガードの除去…といった古典的な最適化が一気に効くようになります。
インライン化が「最適化の母」と呼ばれるゆえんです。

具体的に、最適化が連鎖していく様子を 1 例で追ってみます。先ほどの
`distance_sq(p)` を例に取り、`p.x` を 2 回呼んでいるとします。

```ruby
def distance_sq(p)
  p.x * p.x        # p.x を 2 回呼ぶ
end
```

(1) **脱仮想化＋インライン化**で `p.x` が `@x` の読み出しに展開されます。

ここで `load(p, :@x)` は「`p` のインスタンス変数 `@x` をフィールドとして
直接読み出す」ことを表す擬似記法です（Ruby に `p.@x` という構文はありません）。

```ruby
return deopt(p) unless p.is_a?(Point)   # クラスガード（1 本）
t1 = load(p, :@x)
return deopt(p) unless p.is_a?(Point)   # 展開で増えた 2 本目のガード
t2 = load(p, :@x)
t1 * t2
```

(2) **共通部分式の削除（CSE）**：`load(p, :@x)` を 2 回読んでいますが、間で
`p` の `@x` を書き換えていないので、1 回読めば足ります。

(3) **冗長ガードの除去**：同じ `p.is_a?(Point)` を 2 回確かめています。
1 本目を通れば 2 本目は必ず通るので、2 本目を消せます。

```ruby
return deopt(p) unless p.is_a?(Point)   # ガードは 1 本に
t = load(p, :@x)                         # 読み出しも 1 回に
t * t
```

(4) もし呼び出し側で `p` が定数オブジェクトだと分かっていれば、
**定数畳み込み**（定数だけからなる式を、実行前に計算して結果に置き換える
最適化）で `@x` の値そのものに置き換わり、ガードすら消えて、
最終的に**1 個の掛け算**になることさえあります。

このように、インライン化は「呼び出しを消す」直接の利得より、そのあとに
連鎖する CSE、冗長ガード除去、定数畳み込みの利得のほうが大きいことが
多いのです。だからこそ「最適化の母」と呼ばれます。これらの古典的最適化を
中間表現の上でどう実装するかは[中間表現](ir.md)で扱います。

> [!WARNING]
> インライン化はやりすぎると害になります。大きなメソッドを片端から展開
> すると、生成コードが膨れて命令キャッシュからあふれ、かえって遅くなる
> ことがあります。だから JIT は「呼び出し先が十分小さいか」「十分ホットか」
> を見て、インライン化するかどうかを賭けます。ここにも
> 「展開して得る最適化の利得」対「コードが膨れる損」の損得勘定があります。

## 何にコンパイルするのか

ここまで「機械語へコンパイルする」と言ってきましたが、行き先は一つでは
ありません。JIT の出力先には段階があります。

| 行き先 | 説明 | 速さ | 例 |
|--------|------|------|-----|
| 別のバイトコード | 命令を特殊化版に差し替える | わずかに速い | CPython 3.11 の特殊化命令[](#cite:pep659) |
| スレッデッドコード | 命令ごとの処理関数へのアドレス列 | 中 | 多くのインタプリタ |
| テンプレート機械語 | 命令ごとに用意した機械語片を並べる | 速い | V8 の Sparkplug、Wasm の Liftoff などのベースライン JIT |
| 最適化済み機械語 | 中間表現を経て本格的に最適化 | 最速 | JVM の C2、V8 の TurboFan |

それぞれ、実装的にはこういう一手です。

- **特殊化バイトコード**：観測した型に合わせて、汎用命令を専用命令へ
  その場で書き換えます。たとえば汎用 `BINARY_OP`（+）を、整数だと分かった
  時点で `BINARY_OP_ADD_INT` に差し替える、という具合です。命令の入れ替え
  だけなのでコード生成器がいらず、インタプリタの命令表に専用ハンドラを
  足すだけで実現できる手軽さが魅力です[](#cite:pep659)。外れたら汎用命令へ
  戻す「逆書き換え」がガード兼脱最適化になります。
- **スレッデッドコード**：`switch` の代わりに、各命令ハンドラの末尾で
  「次の命令のハンドラ」へ直接ジャンプします（ハンドラ末尾で次へ直接飛ぶ
  この方式を direct threading と呼びます）。ディスパッチを 1 か所のループに
  集めず分散させることで、CPU の分岐予測が命令ごとに別々に学習でき、
  予測精度が上がります。各ハンドラの番地を値として扱うコンパイラ拡張の
  ラベル値（GCC の `&&label`、ハンドラのアドレスを取れる）で書くのが定番です。

「コンパイル」というと最後の「最適化済み機械語」を思い浮かべがちですが、
JIT では軽い行き先も重要です。最適化にはそれ自体コストがかかるので、
「そこそこ速いコードを、ごく短時間で出す」段が要るのです。この段階分けが
後の章「[階層的 JIT](tiered.md)」の主題で、テンプレート方式の具体もそこで
扱います。本格的な最適化に使う表現は[中間表現](ir.md)で扱います。

## JIT が割に合う条件

JIT はただではありません。コンパイルそれ自体に時間とメモリを使います。
だから JIT は、コンパイルにかけたコストを生成コードの高速化で取り
返せる場所にだけ適用しなければ、かえって遅くなります。

元が取れるのは、同じコードが何度も実行される場所です。1 回しか
実行されない初期化コードを必死に最適化しても、コンパイル時間が無駄に
なるだけです。逆に、100 万回回るループの中身なら、コンパイル時間は
すぐに回収できます。この「何度も実行されて熱くなった場所」を
**ホットスポット**（hot spot）と呼び、それを見つけるのが
[プロファイリング](profiling.md)の役目です。

裏を返せば、熱くなるまでの最初の数千〜数万回はプロファイル収集と
コンパイルのぶんだけむしろ遅い、という負の側面があります。短時間で
終わる処理ほどこの初期コストを取り返せず、JIT がかえって損になりかねません。
この「温まるまで遅い」問題は[ウォームアップ問題](research-warmup.md)として
独立の研究テーマになっています。

> [!IMPORTANT]
> JIT の設計はすべて、この損得勘定の上に成り立っています。
> 「どれだけ熱ければコンパイルするか」「どれだけ手間をかけて最適化するか」
> 「投機の前提が外れたらどうするか」。本書に出てくる技法は、すべて
> 「コンパイルのコスト」対「高速化の利得」のバランスをどう取るかの
> 工夫だと思って読むと、見通しがよくなります。

## まとめ

- JIT が速いのは、実行時に観測した事実を前提に焼き込んだコードを
  生成できるから。
- 消せるオーバーヘッドは大きく 4 つ。ディスパッチ（分岐予測ミスで
  10〜20 サイクル）、型による分岐、ボクシング（タグ付き整数でも
  シフト 1〜2 命令）、間接呼び出し（探索＋間接ジャンプで数十サイクル）。
- 中心戦略は投機的最適化（決め打ち＋ガード＋脱最適化）と
  インライン化。ガードは `cmp`＋`jne` の数命令で、消す対象（場合分け、
  メソッド探索、分岐予測ミス）に比べてはるかに安い。
- インラインキャッシュの単態観測が、脱仮想化と
  インライン化の前提になり、そこから CSE、冗長ガード除去、定数畳み込みが
  連鎖する。
- 行き先は機械語だけでなく、特殊化バイトコードからテンプレート機械語、
  最適化機械語まで段階がある。
- JIT が割に合うのは、繰り返し実行されるホットスポットだけ。

次章では、この「コンパイルして直接実行する」を、Ruby から本物の
x86-64 機械語を生成して呼び出すことで、実際に体験します。
