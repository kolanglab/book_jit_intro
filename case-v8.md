# V8 のケーススタディ（JavaScript）

**V8** は、Google Chrome と Node.js を動かす JavaScript エンジンです。
[なぜ JIT か](why-jit.md)で見た動的型付けの難しさ（型による分岐、探して
飛ぶ呼び出し）を、もっとも極端な形で背負った処理系であり、それを四段の[階層的 JIT](tiered.md)と、
**ヒドゥンクラス**という巧妙なデータ構造で乗り越えています。本書の技法が
動的言語でどう働くかの、最良の見本です。

## 起動の速さと最高速度を両立する四段構成

V8 は、現在四つの実行ティアを持ちます。これは[階層的 JIT](tiered.md)で
「四段」の例として挙げたものです。

| ティア | 名前 | 形態 | 役割 |
|--------|------|------|------|
| 0 | Ignition | バイトコードインタプリタ | まず動かす。フィードバックを集める |
| 1 | Sparkplug | ベースライン JIT | バイトコードを一直線に機械語化 |
| 2 | Maglev | 中位最適化 JIT | 軽量な SSA 最適化 |
| 3 | TurboFan | 全力最適化 JIT | [Sea of Nodes](ir.md) で徹底最適化 |

なぜこれほど段が要るのか。JavaScript は**ウェブページを開くたびに
コンパイルし直される**ことが多く（V8 はコンパイル結果のコードキャッシュで
これを緩和していますが、限界があります）、[ウォームアップ](research-warmup.md)が
ユーザの体感に直結します。一方、ゲームや重い計算には最高速度も要る。
この相反する要求を、「素早く立ち上がる下の段」と「じっくり速くする上の段」を
並べることで両立しているのです。

- **Ignition**：JavaScript をバイトコードにして解釈実行。ここで
  [インラインキャッシュ](profiling.md)を通じて型のフィードバックを集めます。
- **Sparkplug**[](#cite:v8sparkplug)：最適化を一切せず、バイトコードを
  ほぼ一対一で機械語に変換する[ベースライン JIT](tiered.md)。中間表現すら
  作らず、バイトコードを一回なめるだけなので**コンパイルが極めて速い**。
  「速く雑に出す JIT」の典型です。
- **Maglev**[](#cite:v8maglev)：比較的新しい中位ティア。そこそこの SSA
  最適化を、TurboFan より速いコンパイルで行う。起動と最高速度の「谷間」を
  埋めるために追加されました。
- **TurboFan**：[JVM の C2](case-jvm.md)に相当する全力コンパイラ。
  [Sea of Nodes](ir.md)で、インライン化、型特殊化、エスケープ解析まで
  徹底的にやります。

では、各段はいつ、何を見て次へ移るのか。四段それぞれの移行条件を、
もう一段具体に見てみましょう。ここを押さえると、
[階層的 JIT](tiered.md)が V8 で実際にどう運用されているかが腑に落ちます。

- **Ignition → Sparkplug**：Ignition は関数ごとに **interrupt budget**
  （実行予算）を持ち、実行したバイトコードの量に応じて予算を減らして
  いきます。予算を使い切ると「そろそろ昇格か」の判断が走り、V8 は
  Sparkplug を起動します。Sparkplug は最適化判断のための重い解析を
  一切しないので、この予算はかなり小さく（呼び出し数回ぶん程度に）
  設定できます。「ちょっとでも繰り返し使われそうなら、とりあえず機械語に
  しておく」という気軽さです。
- **Sparkplug → Maglev / TurboFan**：ここからが本当の投機です。
  [プロファイリング](profiling.md)で見た呼び出しカウンタとバックエッジ
  カウンタ（ループ用）を見て、十分熱いと判断したら、その関数の
  フィードバックベクタが**十分に型情報で埋まっているか**も確認したうえで
  Maglev（または直接 TurboFan）へ昇格させます。フィードバックがまだ薄い
  関数を急いで最適化しても、特殊化の根拠が乏しく、すぐ[脱最適化](deopt-osr.md)
  する羽目になるからです。なお、長く回り続けるループの中で熱くなった
  場合は、関数の呼び直しを待たず、[OSR](deopt-osr.md) で実行中のループ
  ごと上の段へ乗り換えます。
- **Maglev → TurboFan**：Maglev で動いている関数がさらに長く回り続けると、
  TurboFan による全力最適化へ昇格します。TurboFan は Maglev の機械語を
  入力にするのではなく、バイトコードとフィードバックから組み立て直し
  ます。Maglev までの段で走らせつつ集めた観測を土台に、より攻めた投機
  （深いインライン化、エスケープ解析など）を重ねるのです。

Sparkplug が「バイトコードをほぼ一対一で機械語に落とす」とは、具体的には
こういうことです。Ignition のバイトコード命令を一つずつ見て、その命令の
処理に対応する機械語片を順に書き出していく。中間表現を作らず、レジスタ
割り付けの最適化もせず、命令の並べ替えもしない。Ignition が使っていた
バイトコードのレジスタファイルや[フィードバックベクタ](profiling.md)を
**そのまま共有**するので、インタプリタとデータ構造の互換性が保たれ、
両者を行き来しても状態がずれません。結果として、Sparkplug の出すコードは
インタプリタのディスパッチ（次の命令を読んで飛ぶオーバーヘッド）を消すだけの
素朴なものですが、それだけでインタプリタより目に見えて速くなります。これが
[ベースライン JIT](tiered.md)の真骨頂です。

> [!NOTE]
> Maglev が後から追加された経緯は、[ウォームアップ問題](research-warmup.md)の
> 実務的な現れとして示唆的です。「Sparkplug はコンパイルは速いが出すコードは
> さほど速くない、TurboFan は速いコードを出すがコンパイルが重い」。
> その中間が欲しい、という具体的な性能の谷を埋めるために、
> わざわざ一段増やした。段の数は理屈でなく、実測された性能曲線の形で
> 決まるのです。

> [!TIP]
> 四段を「コンパイル時間 対 実行速度」の軸で並べると、トレードオフが
> はっきりします。Ignition はコンパイル時間ゼロ（解釈実行）で実行は最遅。
> Sparkplug はコンパイルがバイトコード長にほぼ比例するだけの軽さで、
> 実行は中程度。Maglev は SSA を組み立てて軽い最適化を回すぶんコンパイルが
> 重くなり、そのぶん実行が速い。TurboFan は[Sea of Nodes](ir.md)上で
> 何度もパスを回すので最も重いが、最速のコードを出す。**同じ関数でも、
> 熱さに応じてこの四点のどこに置くのが得かが変わる**ので、段を分けて
> 動的に選ぶわけです。

## JavaScript 最大の難所、オブジェクトの形

JavaScript を速くするうえで、型の動的さと並ぶ難所が**オブジェクトの
柔軟さ**です。JavaScript のオブジェクトは、後からいくらでもプロパティを
足したり消したりできます。

```javascript
const p = {};
p.x = 1;     // 後から x を足す
p.y = 2;     // 後から y を足す
```

素朴に実装すると、各オブジェクトを「プロパティ名 → 値」の辞書（ハッシュ
マップ）として持つことになります。しかし `p.x` を読むたびに辞書を引くのは
遅い。さらに、同じ `{x, y}` という形のオブジェクトが何百万個あっても、
それぞれが別々に「`x` はここ、`y` はここ」という情報を重複して持つのは
メモリの無駄です。

## 形を共有するヒドゥンクラス

V8 の解決策が**ヒドゥンクラス**（hidden class、内部的には Map と呼ばれます。
他エンジンでは shape や structure とも）です。アイデアの直接の祖先は、
プロトタイプベース言語 Self の **maps**[](#cite:chambers1989self)（同じ形の
オブジェクトたちに「形の記述」を共有させる仕組み）で、内部名
Map もここに由来します。[JVM の章](case-jvm.md)で見たとおり、V8 を率いた
Lars Bak は Self から HotSpot へ続く系譜の人であり、これはその直系の
輸入品なのです。

> **同じ「形」（同じプロパティの集合と並び）のオブジェクトは、その形の
> 記述（どのプロパティがどのオフセットにあるか）を共有し、各オブジェクト
> 自身は値だけを並べて持つ。**

`p.x = 1; p.y = 2` を実行すると、V8 は内部でヒドゥンクラスを遷移させます。

```
{}          → ヒドゥンクラス C0（プロパティなし）
p.x = 1     → C0 から C1 へ遷移（x はオフセット 0）
p.y = 2     → C1 から C2 へ遷移（x はオフセット 0、y はオフセット 1）
```

同じ手順で作られたオブジェクトはすべて同じヒドゥンクラス `C2` を共有し、
各オブジェクトは `[1, 2]` のように値だけを持ちます。`p.x` の読み出しは
「ヒドゥンクラスから `x` のオフセット（0）を引いて、その位置を読む」だけに
なり、辞書引きが消えます。

ここで重要なのは、ヒドゥンクラスの遷移が**木（ツリー）として共有される**
ことです。`{}` の状態を表す `C0` から「`x` を足したら `C1` へ」という遷移は、
`C0` に遷移表として記録されます。だから別のオブジェクトが同じく `{}` から
始めて `x` を足すと、新しいクラスを作らず既存の `C1` を再利用します。

```
        C0 ({})
        │ "x" を足す
        ▼
        C1 ({x})
        │ "y" を足す
        ▼
        C2 ({x, y})
```

この遷移の木があるおかげで、何百万個のオブジェクトが同じ手順で作られても、
ヒドゥンクラスは木の一本の経路を共有するだけで済みます。逆に、後で見るように
**足す順序が違う**と木が枝分かれし、別々のクラスができてしまいます。

そしてここが JIT との接点です。ある場所で `p.x` を読むとき、[インライン
キャッシュ](profiling.md)は「前回ここに来たオブジェクトのヒドゥンクラスは
`C2` で、`x` はオフセット 0 だった」と覚えます。次回同じ `C2` のオブジェクトが
来れば、オフセット 0 を読むだけです。JIT（Maglev / TurboFan）は、この
フィードバックを使って `p.x` を「ヒドゥンクラスが `C2` であることをガード
してから、固定オフセットを読む」コードに[特殊化](why-jit.md)します。

### 単態から多態、メガモーフィックへ移る IC の状態

[プロファイリング](profiling.md)で見たインラインキャッシュの状態は、V8 では
プロパティアクセスの一点ごとに次のように遷移します。具体例で押さえましょう。

- **未初期化（uninitialized）**：まだ一度も実行していない。何も覚えていない。
- **単態（monomorphic）**：いつも同じヒドゥンクラス（例：`C2`）が来る。
  JIT はこれを「`C2` であることをガードして、オフセット 0 を即ロード」という
  最良のコードに[特殊化](why-jit.md)できます。
- **多態（polymorphic）**：来るヒドゥンクラスが 2〜4 種類に散らばる
  （V8 が IC に覚えさせる上限はちょうど 4 種類です）。IC はそれぞれの
  クラスとオフセットを並べて覚え[](#cite:holzle1991)、JIT は「`C2` なら 0、
  `C5` なら 1、…」という小さな分岐の連なりに展開します。クラスごとに
  オフセットは違っても、まだ十分速い。
- **メガモーフィック（megamorphic）**：来るクラスが多すぎて覚えきれない（V8 では
  5 種類目が来た時点でこの状態へ移ります）。IC は個別の記憶をあきらめ、
  グローバルなハッシュ表を引く
  遅い汎用経路に落ちます。JIT も固定オフセットへの特殊化を断念します。

擬似コードで、単態 IC が固定オフセットアクセスへ落ちる様子を示します。

```javascript
// 元のソース
function getX(p) { return p.x; }   // p.x の読み出し

// IC が単態（C2 ばかり来た）と観測したあとで JIT が出すコード（概念）
function getX_optimized(p) {
  if (p.hidden_class !== C2) deoptimize();  // 形をガード
  return p.slots[0];                        // オフセット 0 を直接ロード
}
```

`p.hidden_class` の比較（**Map のチェック**と呼ばれます）はポインタ 1 個の
等値比較で、続くロードはオフセット即値。辞書引きが、たった 2 命令程度に
畳まれます。これが JavaScript のプロパティアクセスを速くする核心です。

### フィードバックベクタの中身

[プロファイリング](profiling.md)でふれた**フィードバックベクタ**を、もう
少し具体的に見ます。V8 は関数ごとに 1 本のフィードバックベクタ（配列）を
持ち、その関数の中の「型を観測すべき場所」（プロパティアクセス、メソッド
呼び出し、二項演算など）に**スロット**を割り当てます。スロットの中身は
観測点の種類で異なります。

| 観測点 | スロットに溜まる情報 |
|--------|----------------------|
| `p.x`（プロパティ読み） | 見たヒドゥンクラス（Map）と、そのオフセット |
| `f(...)`（呼び出し） | 呼ばれた関数オブジェクト、呼び出し回数 |
| `a + b`（二項演算） | 観測した被演算子の型（小整数か、浮動小数点か、文字列かなど） |
| `arr[i]`（要素アクセス） | 配列の elements kind |

Sparkplug や Ignition が実行のたびにこのスロットを更新し、Maglev や
TurboFan は昇格時にこのベクタを読み込んで投機の根拠にします。たとえば
`a + b` のスロットに「ずっと小整数同士だった」と記録されていれば、JIT は
「整数加算として特殊化し、オーバーフローしたらそのときだけ[脱最適化](deopt-osr.md)」
というコードを出す。フィードバックベクタは、インタプリタの高速化装置で
ありながら、そのまま**JIT への観測記録**を兼ねているのです。

## 形を揃えることの実践的な含意

ヒドゥンクラスの仕組みは、JavaScript の性能に関する有名な助言を生みます。
**「同じ形のオブジェクトを作れ」**。

```javascript
// 良い例：すべて同じ手順で同じ形 → 同じヒドゥンクラスを共有
function makePoint(x, y) {
  const p = {};
  p.x = x;
  p.y = y;
  return p;
}

// 悪い例：プロパティを足す順序が違う → 別々のヒドゥンクラスになる
const a = {}; a.x = 1; a.y = 2;   // C0→C1(x)→C2(x,y)
const b = {}; b.y = 2; b.x = 1;   // C0→C1'(y)→C2'(y,x)  ← a とは別のクラス！
```

`a` と `b` は、できあがりは同じ `{x, y}` ですが、プロパティを足した順序が
違うため、別々のヒドゥンクラスをたどります。すると、両方を扱う `p.x` の
[インラインキャッシュ](profiling.md)は、2 種類のヒドゥンクラスを見ることに
なり、[単態から多態へ](profiling.md)劣化します。多態が進んでメガモーフィックな
状態になると、JIT は固定オフセットの特殊化をあきらめ、
遅い辞書引きに戻ってしまいます。

具体的な「形を乱す」落とし穴は、ほかにもいくつもあります。例えばこうします。

- **コンストラクタの外でプロパティを足す**：オブジェクトを作ったあとに
  条件分岐の中で `if (cond) p.z = 0;` のようにプロパティを足すと、`cond` が
  真のときと偽のときで別々のヒドゥンクラスが生まれます。同じ型のはずの
  オブジェクトが二つの形に分裂し、それを読む場所の IC が多態化します。
- **プロパティを `delete` する**：`delete p.x` は形を後戻りさせます。V8 は
  これを嫌い、`delete` されたオブジェクトを高速なヒドゥンクラス管理から外して
  **辞書モード（slow properties）**へ落とすことがあります。一度辞書モードに
  なると、そのオブジェクトのプロパティアクセスは固定オフセットに特殊化
  できません。
- **数値を添字にした「プロパティ」**：`p[0] = ...` のような数値キーは、
  名前付きプロパティとは別の「要素（elements）」として扱われ、次節の
  elements kind の対象になります。意図せず混ぜると挙動が読みにくくなります。

> [!IMPORTANT]
> ここから導かれる最も実践的な助言は「**オブジェクトはコンストラクタの中で、
> 常に同じ順序で、すべてのプロパティを初期化しきれ**」です。あとから足さない、
> 消さない、順序を変えない（使わないものは `null` で初期化しておくのが定石）。
> そうすればヒドゥンクラスが一本の経路に収束し、IC が単態に保たれ、JIT が固定
> オフセットアクセスへ特殊化できます。逆に形がばらつけば IC が乱れ、特殊化は
> 効きません。「形を揃えると JavaScript は速い」という経験則は魔法ではなく、
> ヒドゥンクラスとインラインキャッシュの仕組みから必然的に導かれるものです。
> 本書の[プロファイリング](profiling.md)と[投機](why-jit.md)の知識があれば、
> その「なぜ」が腑に落ちます。

## 配列にも形がある

同じ発想は配列にも及びます。V8 は配列の要素の種類を **elements kind**
として追跡します。「全部が小整数の配列」「全部が倍精度浮動小数点の配列」
「穴あき（一部が未定義）の配列」などを区別し、それぞれに特化した格納と
アクセスを使います。整数だけの配列に浮動小数点を一つ入れると、より一般的な
（遅い）表現へ遷移します。これもヒドゥンクラスと同じ「形を追跡して
特化する」思想の現れです。

elements kind の要点は、遷移が一方向にしか進まず、しかも**戻らない**ことです。
代表的な並びを具体的に挙げます。

```
PACKED_SMI_ELEMENTS        ← 隙間なし・全部が小整数
   │ 浮動小数点を入れた
   ▼
PACKED_DOUBLE_ELEMENTS     ← 隙間なし・全部が倍精度
   │ オブジェクトや文字列を入れた
   ▼
PACKED_ELEMENTS            ← 隙間なし・任意の値
```

それぞれに、添字に**穴（hole、未定義の隙間）**が空いた版があります。

```
PACKED_SMI  →  HOLEY_SMI
PACKED_DOUBLE  →  HOLEY_DOUBLE
PACKED_ELEMENTS  →  HOLEY_ELEMENTS
```

`PACKED`（隙間なし）から `HOLEY`（穴あき）への遷移も、やはり一方向です。
具体的なコードで、どの操作がどの遷移を引き起こすかを見ます。

```javascript
const a = [1, 2, 3];   // PACKED_SMI_ELEMENTS（一番速い）
a.push(4.5);           // → PACKED_DOUBLE_ELEMENTS（倍精度へ昇格）
a.push("x");           // → PACKED_ELEMENTS（任意の値へ昇格）
a[10] = 0;             // → HOLEY_ELEMENTS（添字 5〜9 が穴になった）
```

なぜ「戻らない」のか。それは**投機を単純に保つため**です。もし「全部また
小整数に戻ったら `PACKED_SMI` へ戻す」を許すと、JIT は配列を触るたびに
「今どの種類か」を疑い続けねばならず、特殊化の前提が安定しません。一方向と
割り切れば、JIT は「この配列は少なくとも `PACKED_DOUBLE` 以上だ」と一度
判断すれば、その仮定が緩む方向（より一般的な種類へ）にしか動かないと
信じられます。これは[投機](why-jit.md)の前提を単調に保つ設計判断です。

> [!IMPORTANT]
> elements kind が一番速いのは `PACKED_SMI` / `PACKED_DOUBLE` です。
> ここから導かれる実践的な助言は、「**配列に穴を空けるな、型を混ぜるな**」。
> 具体的には、`new Array(n)`（中身が穴だらけ）より要素を順に `push` する、
> 整数配列に不用意に浮動小数点を混ぜない、配列の末尾の先へ飛び飛びに
> 代入しない（`a[1000] = x` で大量の穴を作らない）、といった点です。これも
> ヒドゥンクラスの助言と同じく、elements kind の仕組みから必然的に
> 導かれます。

## 脱最適化の引き金

V8 の[脱最適化](deopt-osr.md)は、これまで見た投機の前提が崩れた
ときに起こります。JavaScript ならではの引き金が並びます。

- 想定と違うヒドゥンクラスのオブジェクトが来た（形が変わった）。
  Map のガードが破れる、最も基本的な引き金です。
- 整数だと思っていた値が、実は大きすぎて浮動小数点になった
  （JavaScript の数値は基本 64 ビット浮動小数点で、V8 は小整数（Smi）を
  投機的に整数として扱う）。整数加算がオーバーフローして Smi の範囲を
  超えた瞬間も、これに当たります。
- 配列の elements kind が遷移した（整数配列に浮動小数点を入れた、など）。
- 配列の要素を読もうとしたら、そこが穴（hole）だった。`PACKED` だと
  投機していたのに穴に当たると、`undefined` を返すための一般経路へ戻ります。
- 関数の引数の個数や型が、観測と違った。
- `arguments` オブジェクトを触る、`with` を使う、`delete` でプロパティを
  消すなど、最適化を諦めさせる構文に出くわした。ただしこれは
  **Crankshaft**（V8 の旧最適化コンパイラ。2010 年に登場し、のちに
  TurboFan へ置き換えられた）時代の通念で、`arguments` の素直な使い方などは TurboFan/Maglev
  世代では最適化できるようになりました。`with` のようにスコープの根を
  揺らす構文は今も忌避されますが、これらが一律に最適化を阻むわけでは
  ありません。

なお、V8 の脱最適化には、起こり方の違う二種類（**eager 脱最適化と
lazy 脱最適化**）があります。両者を区別できると、
[脱最適化](deopt-osr.md)の実装が一段はっきり見えます。

- **eager 脱最適化（即時）**：最適化コードを**実行している、まさにその場**で
  ガードが破れて戻るもの。`if (p.hidden_class !== C2) deoptimize();` のような
  ガードに引っかかった瞬間がこれです。今まさに走っているフレームを、その
  地点で Ignition のバイトコードへ作り直します（[脱最適化](deopt-osr.md)で
  見た「状態の復元」を、今動いている関数に対して行う）。
- **lazy 脱最適化（遅延）**：その最適化コードを**今は実行していない**けれど、
  前提が崩れたので次に入ったら無効にしたい、というもの。典型例は、
  ある関数 `f` が「オブジェクト `o` の形は `C2` のまま」と仮定して最適化
  されているとき、別の場所で `o` の形が変わったケースです。`f` は今
  スタックにいないかもしれませんが、もう前提は嘘になっている。そこで V8 は
  `f` の最適化コードを「無効」と印を付け、`f` のエントリを書き換えて、
  次に呼ばれたとき脱最適化経路を通るようにします。実行中のフレームは安全に
  走りきらせ、戻ってきた時点で差し替えます。

eager は「自分のガードで自分が落ちる」、lazy は「他人の都合で無効化される」
と捉えると分かりやすいでしょう。Map の遷移や要素種別の変化は、しばしば
**遠く離れた最適化コードを lazy に道連れにします**。

これらが起きると、TurboFan が生成した最適化コードは uncommon な経路として
[脱最適化](deopt-osr.md)し、Ignition のバイトコードへ戻って、再びフィード
バックを集め直します。同じ場所で脱最適化が何度も繰り返されると、V8 は
「ここは投機が外れがちだ」と学習し、その投機を控えたより保守的なコードで
作り直します（[脱最適化の頻度学習](deopt-osr.md)）。

## Sea of Nodes で全力最適化する TurboFan

四段の頂点、TurboFan が型特殊化を実装する基盤が、[中間表現](ir.md)で見た
**Sea of Nodes**（ノードの海）です[](#cite:click1995)。これは
[JVM の C2](case-jvm.md)と同じ系譜の設計で、命令を「制御フローの順番に
並んだリスト」ではなく、データ依存と制御依存だけでつながったグラフとして
表します。

### 命令の自由な移動と、第一級の脱最適化点

なぜ V8 がこれを選んだか。Sea of Nodes は順序の制約を最小限しか持たないので、
命令を自由に動かせるのが利点です。たとえば、ループの中で何度も読まれる
`p.x` が「ループ中で `p` の形は変わらない」と分かれば、Map のガードと
オフセットロードをループの外へ巻き上げる（loop-invariant code motion）。
あるいは、フィードバックベクタから「ここはいつも整数加算」と分かれば、
浮動小数点としての一般加算ノードを整数加算ノードへ型特殊化する。
こうした最適化が、グラフ上のノードの付け替えとして自然に表現できます。

そして[中間表現](ir.md)でふれたとおり、TurboFan は**脱最適化点を
グラフ上の第一級のノード**として扱います。「ここで Map が `C2` であることを
チェックし、外れたら脱最適化」というガードが、ただのコメントでなく
最適化対象のノードになっている。だからこそ、同じ Map チェックが複数あれば
一つに併合でき（重複ガードの除去）、ループ不変なら外へ追い出せる。
投機と脱最適化を中間表現の中心に据える設計が、V8 の攻めた最適化を
支えているのです。

### Sea of Nodes から離脱する Turboshaft

ただし、この物語には現在進行形の続きがあります。V8 は TurboFan の
パイプラインを、Sea of Nodes ではなく制御フローグラフ（CFG）ベースの
新しい中間表現 **Turboshaft** の上に再設計する移行を進めており、2025 年には
公式ブログ「Land ahoy: leaving the Sea of Nodes」で Sea of Nodes からの
離脱を表明しました[](#cite:v8turboshaft)。理由はまさに**実装と保守の
難しさ**です。順序の制約を最小限にする自由は、理論上の魅力ほどには実利を
生まず、グラフを目で追えない、デバッグしにくい、コンパイルが遅いという
負担のほうが勝った。これが V8 チームの総括です。[中間表現](ir.md)の
章で「Sea of Nodes は実装も難しい」とふれたことの、最前線からの実例と
言えます。

> [!WARNING]
> Sea of Nodes は強力ですが、実装と保守が難しい中間表現です。V8 では、
> 比較的新しい Maglev があえて Sea of Nodes を使わず、より素朴で
> 制御フローの順序を保った中間表現を採っています。これは「TurboFan ほど
> 攻めないかわりに、コンパイルを速く、実装を単純に」という Maglev の役割
> （起動と最高速度の谷を埋める中位ティア）に合わせた、明確な設計判断です。
> そして前述のとおり、TurboFan 自身も Turboshaft で CFG ベースへ向かって
> います。**同じエンジンの中でも、ティアの役割に応じて中間表現を選び分け、
> 経験に応じて選び直す**点は、[中間表現](ir.md)の議論の実例として示唆的です。

## まとめ

- **V8** は JavaScript を四段（**Ignition** → **Sparkplug** → **Maglev**
  → **TurboFan**）の[階層的 JIT](tiered.md)で速くする。起動と最高速度の
  両立が至上命題。段の移行は呼び出しカウンタとバックエッジカウンタ、それに
  フィードバックの充実度で決まる。
- **Sparkplug** はバイトコードをほぼ一対一で機械語化する（中間表現を
  作らない）ベースライン JIT、**TurboFan** は[Sea of Nodes](ir.md)で
  脱最適化点まで第一級ノード化する全力コンパイラ、**Maglev** はあえて
  Sea of Nodes を使わず谷を埋める中位ティア。TurboFan 自身も、CFG ベースの
  **Turboshaft** へのパイプライン再設計で Sea of Nodes から離脱しつつある
  [](#cite:v8turboshaft)。
- JavaScript の動的オブジェクトを、**ヒドゥンクラス**（形の記述を共有し、
  各オブジェクトは値だけ持つ）で速くする。プロパティ追加は遷移木をたどる
  **クラス遷移**。
- ヒドゥンクラス＋[インラインキャッシュ](profiling.md)で、`p.x` を
  「Map をガードして固定オフセットをロード」に[特殊化](why-jit.md)。IC は
  単態→多態→メガモーフィック、と劣化する。**フィードバックベクタ**が観測を集約し、
  Maglev / TurboFan が読む。「同じ形のオブジェクトを作れ」という助言の正体。
- 配列の **elements kind**（`PACKED_SMI`→`PACKED_DOUBLE`→`PACKED_ELEMENTS`→
  `HOLEY`…）は**一方向にしか遷移せず戻らない**。投機の前提を単調に保つ
  設計判断。
- 形の変化、数値の想定外、穴、`delete`、`arguments` などが
  [脱最適化](deopt-osr.md)の引き金。自分のガードで落ちる **eager** と、
  他人の都合で無効化される **lazy** の二種類がある。

YJIT を見ます。
