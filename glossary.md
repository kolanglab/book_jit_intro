# 用語集

本書に登場する用語を、五十音・アルファベット順に近い形でまとめます。
各項目から、詳しく扱った章へのリンクを張っています。知らない言葉に
出会ったら、ここに戻ってきてください。

## JIT の基本

**JIT コンパイル（Just-In-Time compilation、実行時コンパイル）**
: プログラムを実行しながら、その一部を機械語へコンパイルし、以後はその
  機械語を直接実行する技法。実行時に観測した事実を前提に焼き込める点が
  最大の武器。→ [なぜ JIT か](why-jit.md)

**AOT コンパイル（Ahead-Of-Time、事前コンパイル）**
: 実行より前に、あらかじめ機械語へコンパイルしておく方式。C の `gcc` が
  典型。起動は速いが、実行時の観測に基づく投機ができない。JIT の対義語。
  → [はじめに](introduction.md)

**ホットスポット（hot spot）**
: 繰り返し実行されて「熱く」なった、プログラムの一部。JIT はコンパイルの
  コストが元を取れるホットスポットだけを対象にする。→ [なぜ JIT か](why-jit.md)

**ウォームアップ（warmup、暖機）**
: JIT 処理系が、起動直後の遅い状態から、最適化が効いた速い状態へ移って
  いく過程。短命なプロセスでは暖まる前に終わってしまう問題がある。
  → [ウォームアップ問題](research-warmup.md)

## 投機まわり

**投機的最適化（speculative optimization）**
: 「たぶんこうだろう」と前提を決め打ちして高速なコードを出し、念のための
  チェック（ガード）を置き、外れたら戻る、という JIT の中心戦略。
  → [なぜ JIT か](why-jit.md)、[デオプティマイズと OSR](deopt-osr.md)

**特殊化（specialization）**
: 観測した型や値を前提として焼き込み、場合分けのない専用コードを作ること。
  → [なぜ JIT か](why-jit.md)

**ガード（guard）**
: 投機の前提が成り立っているかを実行時に確かめる、軽いチェック。成り立つ
  間は高速経路を一直線に進み、破れたら脱出する。→ [トレーシング](tracing.md)

**デオプティマイズ（deoptimization、最適化解除）**
: ガードが破れたとき、最適化コードの実行状態をインタプリタが再開できる
  状態へ作り替えて戻ること。投機を「攻めさせる」ための安全網。1992 年に
  SELF で発明。→ [デオプティマイズと OSR](deopt-osr.md)

**OSR（On-Stack Replacement、実行中スタックの差し替え）**
: 実行中の関数（特に長いループ）の途中で、実行形式を別のものへ乗り換える
  技術。インタプリタ→最適化（昇格）と、最適化→インタプリタ（＝デオプト、
  降格）の両向きがある。→ [デオプティマイズと OSR](deopt-osr.md)

## コンパイルの単位と方式

**メソッド JIT**
: メソッド（関数）の制御フローグラフ全体を、コンパイルの単位とする方式。
  境界が明快で、インライン化が生命線。HotSpot・V8・YJIT など主流派。
  → [コンパイルの単位](compilation-units.md)

**トレーシング JIT**
: 実際に通った 1 本の実行経路（トレース）を単位とする方式。分岐をガードに
  変えた一直線のコードを作る。LuaJIT・PyPy が代表。→ [トレーシング](tracing.md)

**トレース（trace）**
: ホットなループを 1 周する間に実行された命令を、分岐や呼び出しをまたいで
  一直線に記録したもの。→ [トレーシング](tracing.md)

**サイドイグジット（side exit）**
: トレース実行中にガードが破れて、トレースから抜けてインタプリタへ戻る
  こと。→ [トレーシング](tracing.md)

**基本ブロックバージョニング（BBV）**
: 基本ブロックを、入口で分かっている型情報ごとに別バージョンとして
  コンパイルし、型チェックを自然に除去する方式。Ruby の YJIT が採用。
  → [コンパイルの単位](compilation-units.md)、[Ruby](case-ruby.md)

**メタトレーシング（meta-tracing）**
: ユーザのプログラムでなく、それを実行しているインタプリタをトレースする
  ことで、言語ごとの JIT を書かずに済ませる技法。PyPy/RPython が実用化。
  → [メタトレーシング](meta-tracing.md)

**部分評価（partial evaluation）**
: 二入力のプログラムの片方を先に固定し、残りの入力だけを取る、より速い
  残余プログラムを生成する技法。インタプリタを部分評価するとコンパイル
  結果が得られる（Futamura の第一射影）。Truffle/Graal が実用化。
  → [部分評価と Futamura 射影](research-partial-eval.md)

**階層的 JIT（tiered JIT）**
: 起動の速い軽い段と、実行の速い重い段を並べ、コードを熱さに応じて段を
  上げ下げする運用。現代の主要処理系はほぼこの構成。→ [階層的 JIT](tiered.md)

**ベースライン JIT（baseline JIT）**
: 最適化をほとんどせず、ごく短時間で機械語を出す軽量なコンパイラ。「速く
  雑に出す JIT」。V8 の Sparkplug、JSC の Baseline など。→ [階層的 JIT](tiered.md)

## プロファイリングと観測

**プロファイリング（profiling）**
: 「どこが熱いか」「何を前提にできるか」を、通常実行に溶け込ませて軽く
  測る技術。→ [プロファイリング](profiling.md)

**インラインキャッシュ（inline cache、IC）**
: メソッド呼び出しやプロパティアクセスの箇所に、前回の探索結果（型と
  見つけた相手）を覚えておく仕掛け。探索を省くと同時に、JIT への型
  フィードバックを兼ねる。→ [プロファイリング](profiling.md)

**フィードバックベクタ（feedback vector）**
: 関数ごとに、各観測点で見た型などを集約した配列。V8 の用語。JIT が
  特殊化の判断に使う。→ [プロファイリング](profiling.md)

**quickening**
: バイトコード命令を、観測した型に特化した別命令へ実行時に書き換える
  こと。CPython の特殊化適応型インタプリタが実装。→ [プロファイリング](profiling.md)

## 中間表現とコード生成

**中間表現（IR、intermediate representation）**
: 機械語へ落とす前に、CPU 非依存でデータの流れが見やすい形でプログラムを
  表す作業台。→ [中間表現](ir.md)

**SSA 形式（Static Single Assignment、静的単一代入）**
: すべての変数がちょうど一度だけ代入される中間表現の形。データの流れが
  一意になり、最適化を劇的に簡単にする。合流点は φ 関数で表す。
  → [中間表現](ir.md)

**Sea of Nodes（ノードの海）**
: 命令の順序を捨て、データ依存と制御依存をグラフの辺として表す中間表現。
  命令の並べ替えの自由を得る。C2・Graal が採用。→ [中間表現](ir.md)

**レジスタ割り当て（register allocation）**
: 無数の仮想レジスタを、有限の物理レジスタへ割り当てる処理。あふれた値は
  メモリへ追い出す（スピル）。→ [中間表現](ir.md)

**線形走査（linear scan）**
: 生存区間を一度なめるだけでレジスタを割り当てる、高速なレジスタ割り当て
  手法。最適性より速さを取る JIT に向く。→ [中間表現](ir.md)

**ステンシル（stencil）**
: Copy-and-Patch で使う、穴（実行時に埋める値の場所）を空けた、あらかじめ
  コンパイル済みの機械語断片。→ [Copy-and-Patch](research-copy-and-patch.md)

**Copy-and-Patch**
: ステンシルをコピーして穴を埋めるだけでコードを生成する、極めて高速な
  コード生成技法。CPython 3.13 が採用。→ [Copy-and-Patch](research-copy-and-patch.md)

## 周辺の用語

**ボクシング（boxing）/ アンボクシング**
: 値（整数など）を、クラス情報などを持つヒープ上の「箱」に入れること／
  取り出すこと。JIT は型を特殊化すると箱を外して生の値で計算できる。
  → [なぜ JIT か](why-jit.md)

**NaN ボクシング**
: 浮動小数点数の NaN の空きビットパターンに、数値以外の値（型タグと
  ポインタ）を詰め込む値表現。LuaJIT が採用。→ [LuaJIT](case-luajit.md)

**インライン化（inlining）**
: メソッド呼び出しを、呼び出し先の中身で置き換えること。呼び出しコストを
  消すだけでなく、またいだ最適化を可能にする「最適化の母」。
  → [なぜ JIT か](why-jit.md)

**エスケープ解析（escape analysis）**
: オブジェクトが、それを作ったメソッドの外へ漏れ出すかを調べる解析。
  漏れなければヒープ確保を省ける（スカラ置換／アロケーションシンキング）。
  → [JVM](case-jvm.md)、[トレーシング](tracing.md)

**ヒドゥンクラス（hidden class）/ シェイプ（shape）**
: 同じ形（プロパティ集合と並び）のオブジェクトで、レイアウト記述を共有
  する仕組み。各オブジェクトは値だけを持ち、プロパティアクセスを固定
  オフセット化できる。V8 のヒドゥンクラス、Ruby の Object Shapes。
  → [V8](case-v8.md)、[Ruby](case-ruby.md)

**ガベージコレクション（garbage collection、GC）**
: 不要になったオブジェクトのメモリを自動で回収する仕組み。JIT が短命
  オブジェクトの確保を消すと、GC の負担も減る。

**リンカ（linker）/ 再配置（relocation）**
: 複数のオブジェクトファイルをつなぎ、まだ決まっていなかったアドレスを
  後から埋める（再配置）ツールと処理。Copy-and-Patch の「穴を埋める」は
  これと同じ仕組み。→ [Copy-and-Patch](research-copy-and-patch.md)

**メソッド / 仮想呼び出し（virtual call）**
: 実行時に相手の実装が決まる呼び出し。動的言語やオブジェクト指向で多用
  され、「探して飛ぶ」コストを伴う。JIT は脱仮想化とインライン化で攻める。
  → [なぜ JIT か](why-jit.md)、[JVM](case-jvm.md)

## さらに学ぶために

本書は JIT の地図を描くことを目的としました。各技法をさらに深く学ぶには、
本文で引用した原論文に当たるのが一番です。とりわけ、歴史を一望するには
Aycock の概説[](#cite:aycock2003)、投機と型フィードバックの源流には
SELF の一連の論文[](#cite:holzle1992) [](#cite:holzle1994)
[](#cite:chambers1989)、トレーシングには
Gal ら[](#cite:gal2009)と Bolz ら[](#cite:bolz2009)、部分評価には
Futamura[](#cite:futamura1971)と Würthinger ら[](#cite:wurthinger2017)が
出発点になります。コンパイラの一般的な基礎は、いわゆるドラゴンブック
[](#cite:aho2006)が定番です。

そして何より、本書で動かしたコードを土台に、自分の小さな処理系へ JIT を
足してみることが、いちばんの学びになります。[最小の JIT](hello-jit.md)で
機械語を吐き、[トレーシング](tracing.md)で経路を記録し、[デオプト](deopt-osr.md)で
戻る —— その小さな実装の中に、本書のすべてが詰まっています。
