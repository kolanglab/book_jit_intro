# ウォームアップ問題とコンパイルキャッシュ

[階層的 JIT](tiered.md)で、「JIT は起動直後は遅く、だんだん速くなる」と
いう**ウォームアップ**を見ました。インタプリタで動き始め、熱い場所が
順に上位ティアへ昇格し、やがて最高速度に達する —— 私たちはこれを
当然の前提として話してきました。

ところがこの章では、その「当然」を疑います。**JIT は、本当にいつも
ウォームアップして、安定した最高速度に落ち着くのか？** 2017 年の
ある研究[](#cite:barrett2017)が、この素朴な信念に正面から疑問を投げかけ、
JIT 研究の常識を揺さぶりました。そして後半では、ウォームアップの
コストそのものを減らす **コンパイルキャッシュ**の話へ進みます。

## 「暖まれば速くなる」という素朴な信念

JIT 処理系の性能を測るとき、研究者も実務家も、ある手順を踏みます。
同じ処理を**何度も繰り返し**（イテレーション）、最初の数回は「ウォーム
アップ中だから」と捨て、その後の「安定した」測定値を本当の性能とみなす。

この手順の裏には、暗黙の三つの仮定があります。

1. プログラムは、繰り返すうちに**安定した状態**（steady state）に達する。
2. その安定状態は、最初より**速い**（ウォームアップは性能を上げる）。
3. いったん安定したら、**そのまま安定し続ける**。

「JIT は暖まれば速くなり、その後は安定する」—— あまりに自然に思えるので、
ふつう誰も疑いません。

## 注意深く測ってみたら、信念は崩れた

Barrett らは、この仮定を**徹底的に注意深い実験**で検証しました
[](#cite:barrett2017)。彼らは Krun という測定基盤を作り、性能測定を狂わせる
外乱を片端から潰しました。CPU の周波数変動（ターボブースト）を止め、
ベンチマークを 1 つずつ隔離し、OS のノイズを抑え、統計的に意味のある
回数だけ反復する。そうして、複数の主要処理系（HotSpot、V8、PyPy、
LuaJIT など）で、何千回ものイテレーションの性能曲線を丹念に記録しました。

結果は、三つの仮定をことごとく裏切るものでした。

- **安定しない**：多くのベンチマークが、何千回繰り返しても安定した状態に
  落ち着かなかった。性能がふらつき続けるのです。
- **遅くなることがある**：ウォームアップどころか、繰り返すうちに**だんだん
  遅くなる**（slowdown）ケースが観測された。
- **いったん安定してから崩れる**：安定したように見えたあと、突然性能が
  変わる（別の安定状態へ移る）ケースもあった。

「JIT は暖まって速くなり、安定する」という素朴な物語は、**実測では
しばしば成り立たない**。これがこの研究の衝撃でした。

> [!IMPORTANT]
> この発見の意味は二重です。第一に、**JIT の性能測定そのものが難しい**。
> 「最初の数回を捨てて、あとは安定」という標準手法が、そもそも成り立って
> いない場合がある。第二に、**JIT のヒューリスティクスには改善の余地が
> 大きい**。性能が遅くなったりふらついたりするのは、コンパイルの判断
> （いつ・何を・どう最適化するか）が、ときに裏目に出ている証拠だからです。

## なぜそんなことが起こるのか

性能がふらついたり遅くなったりする原因は、本書の第 I 部で見た仕組みの
裏返しとして理解できます。

- **デオプティマイズの繰り返し**：[投機](deopt-osr.md)が外れてデオプト
  →再コンパイル→また外れる、を繰り返すと、性能が安定しません。
  プログラムの「相」が途中で変わる（最初は整数、後で浮動小数点）と
  起こりがちです。
- **再コンパイルの連鎖**：[プロファイル](profiling.md)が更新されるたびに
  上位ティアを作り直すと、そのコストが性能曲線にギザギザを刻みます。
- **ガベージコレクションや CPU キャッシュ**：JIT 単独でなく、GC の
  タイミングやメモリ配置の都合も絡んで、再現性のないふらつきが生じます。

つまりウォームアップ問題は、「JIT が観測に基づいて投機し、外れたら作り
直す」という適応的なふるまいの、影の側面です。適応するからこそ、適応が
裏目に出ることもある。

## ウォームアップのコストを減らす：コンパイルキャッシュ

ウォームアップにはもう一つ、実務的に切実な側面があります。**毎回の起動の
たびに、また一からウォームアップし直さねばならない**ことです。

サーバを再起動するたび、コマンドを実行するたびに、処理系はゼロからプロファイルを
集め、ホットスポットを見つけ、コンパイルし直します。長時間動くサーバなら
一度暖まればよいですが、**短命なプロセス**（コマンドラインツール、
サーバレス関数、CI のジョブ）では、暖まる前に仕事が終わってしまい、
JIT の恩恵をほとんど受けられません。これは[階層的 JIT](tiered.md)の
「起動 vs 最高速度」のトレードオフの、最も厳しい現れです。

この無駄を省く発想が、**コンパイルの成果を保存して再利用する**ことです。
大きく二系統あります。

- **プロファイルの保存（PGO 的アプローチ）**：前回の実行で集めたホット
  スポットや型のプロファイルを保存しておき、次回の起動時に**最初から
  それを使って**コンパイルする。ゼロから観測し直す手間を省きます。
- **生成コードの保存（AOT キャッシュ）**：コンパイル済みの機械語そのものを
  ディスクに保存し、次回はそれを読み込んで使う。

JVM の世界では、これらが具体的な機能として実装されてきました。クラスの
読み込みを速める **CDS**（Class Data Sharing）、商用 JVM の **ReadyNow**
（前回のプロファイルを保存して投機の前提を引き継ぐ）、そして近年の
**Project Leyden** や **CRaC**（実行中のプロセスのスナップショットを取り、
暖まった状態から再開する）などです。発想はどれも「**一度払ったウォーム
アップのコストを、二度払わない**」ことです。

> [!NOTE]
> コンパイルキャッシュには固有の難しさがあります。保存したプロファイルや
> 機械語は、「前回の入力」に最適化されています。今回の入力の傾向が違えば、
> 古いプロファイルがかえって不利に働く（外れた投機を最初から抱え込む）
> こともあります。また、保存したコードがソースの変更や CPU の違いと
> 食い違わないよう、整合性の管理も要ります。「キャッシュは効くが、
> 過信は禁物」という、計算機科学のおなじみの教訓がここにもあります。

## AOT と JIT の合流

ウォームアップ問題は、長らく対立すると思われてきた **AOT と JIT を
歩み寄らせ**ました。

純粋な AOT（事前コンパイル）は起動が速い代わりに、[なぜ JIT か](why-jit.md)で
見た「実行時の観測に基づく投機」ができません。純粋な JIT はその逆です。
そこで現代の処理系は、**両者を組み合わせ**ます。GraalVM の Native Image は
[Truffle](research-partial-eval.md)処理系を AOT コンパイルして瞬時に起動
できるようにしつつ、実行時には JIT で更に最適化する道を残します。CPython
3.13 の JIT（[次章](research-copy-and-patch.md)で扱う Copy-and-Patch）も、
「起動が軽い JIT」を志向した設計です。

「起動の速さ（AOT 的）」と「実行時適応の賢さ（JIT 的）」をどう両立させるか。
ウォームアップ問題は、この問いを研究の中心テーマに押し上げました。

## まとめ

- 「JIT は暖まれば速くなり、安定する」という素朴な信念は、注意深い
  実測[](#cite:barrett2017)では**しばしば成り立たない**。安定しない・
  遅くなる・安定後に崩れる、が現実に観測された。
- 原因は、デオプトの繰り返し・再コンパイルの連鎖・GC やキャッシュの
  干渉など、JIT の適応的ふるまいの影の側面。
- ウォームアップのコストは、特に**短命なプロセス**で切実。毎回ゼロから
  暖め直す無駄が大きい。
- **コンパイルキャッシュ**（プロファイルの保存、生成コードの保存）で、
  ウォームアップを二度払わない工夫が進む（CDS・ReadyNow・Leyden・CRaC）。
- ウォームアップ問題は **AOT と JIT を歩み寄らせ**、「速い起動と賢い適応の
  両立」を中心課題にした。

次章は、その「軽い JIT」を極限まで突き詰めた、最適化コンパイラの数百倍
速くコードを生成する技法 —— Copy-and-Patch を見ます。
