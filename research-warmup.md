# ウォームアップ問題とコンパイルキャッシュ

[階層的 JIT](tiered.md)で、「JIT は起動直後は遅く、だんだん速くなる」と
いう**ウォームアップ**を見ました。インタプリタで動き始め、熱い場所が
順に上位ティアへ昇格し、やがて最高速度に達する —— 私たちはこれを
当然の前提として話してきました。

ところがこの章では、その「当然」を疑います。**JIT は、本当にいつも
ウォームアップして、安定した最高速度に落ち着くのか？** 2017 年の
ある研究[](#cite:barrett2017)が、この素朴な信念に正面から疑問を投げかけ、
JIT 研究の常識を揺さぶりました。そして後半では、ウォームアップの
コストそのものを減らす **コンパイルキャッシュ**の話へ進みます。

## ウォームアップは本当に起こるのか

まずは「JIT は暖まれば速くなり、その後は安定する」という信念そのものを、
前提・実測・原因の三段で吟味します。素朴な信念を明示し（前提）、Barrett らの
注意深い実測がそれを裏切ったことを見て（反証）、なぜそうなるのかを第 I 部の
仕組みから解きほぐします（原因）。そのうえで、ウォームアップのコストを減らす
**コンパイルキャッシュ**（次節）、起動と適応を両取りする **AOT と JIT の合流**
へと話を進めます。

### 「暖まれば速くなる」という素朴な信念

JIT 処理系の性能を測るとき、研究者も実務家も、ある手順を踏みます。
同じ処理を**何度も繰り返し**（イテレーション）、最初の数回は「ウォーム
アップ中だから」と捨て、その後の「安定した」測定値を本当の性能とみなす。

この手順の裏には、暗黙の三つの仮定があります。

1. プログラムは、繰り返すうちに**安定した状態**（steady state）に達する。
2. その安定状態は、最初より**速い**（ウォームアップは性能を上げる）。
3. いったん安定したら、**そのまま安定し続ける**。

「JIT は暖まれば速くなり、その後は安定する」—— あまりに自然に思えるので、
ふつう誰も疑いません。

### 注意深く測ってみたら、信念は崩れた

Barrett らは、この仮定を**徹底的に注意深い実験**で検証しました
[](#cite:barrett2017)。彼らは Krun という測定基盤を作り、性能測定を狂わせる
外乱を片端から潰しました。CPU の周波数変動（ターボブースト）を止め、
ベンチマークを 1 つずつ隔離し、OS のノイズを抑え、統計的に意味のある
回数だけ反復しました。そもそも、これほど厳密に測る必要があったのは、
JIT 処理系の性能はばらつきが大きく、素朴な計測では結論が容易に揺らぐ
からです。「統計的に厳密な性能評価」の必要性そのものは Georges らが
早くから論じており[](#cite:georges2007)、Barrett らの徹底した測定は
その方法論の系譜に連なるものでした。そうして、
八つの処理系 —— HotSpot（Java）、Graal、JRuby+Truffle、V8（JavaScript）、
PyPy（Python）、LuaJIT、HHVM（PHP）、そして比較対照の GCC（C）—— で、
標準的なマイクロベンチマークの性能曲線を、各構成あたり 2000 回のイテレーションまで
丹念に記録しました[](#cite:barrett2017)。

そして、得られた性能曲線を変化点解析（changepoint analysis）で
**自動分類**しました。各〈処理系, ベンチマーク〉の組ごとに、性能が
どうふるまったかを次のように名づけます[](#cite:barrett2017)。

- **warmup**：素朴な信念どおり、暖まって速くなり、その後安定する。
- **flat**：最初から最後まで性能が変わらない（暖まる余地がない）。
- **slowdown**：繰り返すうちに**だんだん遅くなる**。
- **no steady state**：何度繰り返しても安定した状態に**落ち着かない**。
  性能がふらつき続ける。

結果は、三つの仮定をことごとく裏切るものでした。最も条件の良い計測機でさえ、
安定した最高性能に一貫して到達した組はわずか **43.5%** に過ぎず、過半数が
そこに届きませんでした[](#cite:barrett2017)。しかも、

- **安定しない**：多くのベンチマークが、何度繰り返しても安定した状態に
  落ち着かなかった（no steady state）。
- **遅くなることがある**：ウォームアップどころか、繰り返すうちに**だんだん
  遅くなる**（slowdown）ケースが観測された。
- **安定しても最速とは限らない**：安定状態（steady state）に達しても、それが
  最高性能とは限らず、遅い安定状態に落ち着く組もあった。同じ組でも実行
  （プロセス）ごとに分類が食い違う **inconsistent** なふるまいさえあった。

「JIT は暖まって速くなり、安定する」という素朴な物語は、**実測では
しばしば成り立たない**。これがこの研究の衝撃でした。

> [!IMPORTANT]
> この発見の意味は二重です。第一に、**JIT の性能測定そのものが難しい**。
> 「最初の数回を捨てて、あとは安定」という標準手法が、そもそも成り立って
> いない場合がある。第二に、**JIT のヒューリスティクスには改善の余地が
> 大きい**。性能が遅くなったりふらついたりするのは、コンパイルの判断
> （いつ・何を・どう最適化するか）が、ときに裏目に出ている証拠だからです。

### なぜそんなことが起こるのか

性能がふらついたり遅くなったりする原因は、本書の第 I 部で見た仕組みの
裏返しとして理解できます。

- **脱最適化の繰り返し**：[投機](deopt-osr.md)が外れて脱最適化
  →再コンパイル→また外れる、を繰り返すと、性能が安定しません。
  プログラムの「相」が途中で変わる（最初は整数、後で浮動小数点）と
  起こりがちです。
- **再コンパイルの連鎖**：[プロファイル](profiling.md)が更新されるたびに
  上位ティアを作り直すと、そのコストが性能曲線にギザギザを刻みます。
- **ガベージコレクションや CPU キャッシュ**：JIT 単独でなく、GC の
  タイミングやメモリ配置の都合も絡んで、再現性のないふらつきが生じます。
- **コードキャッシュの圧迫と退去**：生成コードの置き場は有限です。
  [階層的 JIT](tiered.md)でふれたように、いっぱいになると古い版を退去
  （eviction）させ、必要になれば作り直す。この出し入れが、安定したはずの
  性能に段差を刻みます。インライン化の判断が観測のたびに変わって、同じ
  関数を大きくしたり小さくしたりするのも同類です。

つまりウォームアップ問題は、「JIT が観測に基づいて投機し、外れたら作り
直す」という適応的なふるまいの、影の側面です。適応するからこそ、適応が
裏目に出ることもある。

たとえば「最初の 1000 周は整数、その後は浮動小数点」というループを考えて
みてください。前半で「ここは整数」と投機して特殊化したコードができ、相が
変わった瞬間にガードが破れて[脱最適化](deopt-osr.md)。再コンパイルして
今度は浮動小数点に特殊化…と、たった一度の相の変化が、脱最適化・再コンパイル・
コードキャッシュの出し入れを連鎖させ、性能曲線に深い谷を刻みます。
[脱最適化と OSR](deopt-osr.md)で見た「脱最適化頻度の学習と指数的後退」は、
まさにこの谷を浅くするための実装上の工夫でした。ウォームアップの研究と
個々の技法は、こうして裏表でつながっています。

## ウォームアップのコストを減らす：コンパイルキャッシュ

ウォームアップにはもう一つ、実務的に切実な側面があります。**毎回の起動の
たびに、また一からウォームアップし直さねばならない**ことです。

サーバを再起動するたび、コマンドを実行するたびに、処理系はゼロからプロファイルを
集め、ホットスポットを見つけ、コンパイルし直します。長時間動くサーバなら
一度暖まればよいですが、**短命なプロセス**（コマンドラインツール、
サーバレス関数、CI のジョブ）では、暖まる前に仕事が終わってしまい、
JIT の恩恵をほとんど受けられません。これは[階層的 JIT](tiered.md)の
「起動 vs 最高速度」のトレードオフの、最も厳しい現れです。

この無駄を省く発想が、**コンパイルの成果を保存して再利用する**ことです。
大きく三系統あります。

- **プロファイルの保存（PGO 的アプローチ）**：前回の実行で集めたホット
  スポットや型のプロファイルを保存しておき、次回の起動時に**最初から
  それを使って**コンパイルする。ゼロから観測し直す手間を省きます。
- **生成コードの保存（AOT キャッシュ）**：コンパイル済みの機械語そのものを
  ディスクに保存し、次回はそれを読み込んで使う。
- **暖まった状態ごと保存（スナップショット）**：プロファイルやコードだけでなく、
  ヒープを含む**実行中プロセスの状態まるごと**を保存し、次回はそこから再開する。
  「暖機の途中から始める」のではなく「暖機し終わった瞬間に戻る」発想です。

JVM の世界では、これらが具体的な機能として実装されてきました。クラスの
読み込みを速める **CDS**（Class Data Sharing。クラスのメタデータを共有
アーカイブにして読み込みを省く）、商用 JVM の **ReadyNow**（前回のプロファイルを
保存して投機の前提を引き継ぎ、起動直後から上位ティアで走らせる）、そして近年の
**Project Leyden**（AOT キャッシュの標準化）や **CRaC**（実行中のプロセスの
スナップショットを取り、暖まった状態から再開する）などです。発想はどれも
「**一度払ったウォームアップのコストを、二度払わない**」ことです。ブラウザの
[V8](case-v8.md) がスクリプトのコンパイル結果をキャッシュするのも、[Android の
ART](case-others.md) が端末のアイドル時に AOT を回して次回起動を速くするのも、
同じ系譜です。

ただし「保存したものをそのまま信じてよいか」の**検証**が要ります。AOT
キャッシュは、ソースが変わっていないか（クラスのフィンガープリント）、保存時と
同じ CPU 機能（SSE/AVX など）が使えるか、を起動時に照合し、食い違えば捨てて
ふつうに JIT へ戻す。投機の前提（[V8 のヒドゥンクラス](case-v8.md)や型
フィードバック）を引き継ぐ場合は、それが今回も成り立つかを最初の[ガード](deopt-osr.md)で
確かめ、外れていれば[脱最適化](deopt-osr.md)してやり直します。「キャッシュは
当たれば大勝ち、外れても安全に捨てられる」ように作るのが勘どころです。

> [!NOTE]
> コンパイルキャッシュには固有の難しさがあります。保存したプロファイルや
> 機械語は、「前回の入力」に最適化されています。今回の入力の傾向が違えば、
> 古いプロファイルがかえって不利に働く（外れた投機を最初から抱え込む）
> こともあります。また、保存したコードがソースの変更や CPU の違いと
> 食い違わないよう、整合性の管理も要ります。「キャッシュは効くが、
> 過信は禁物」という、計算機科学のおなじみの教訓がここにもあります。

## AOT と JIT の合流

ウォームアップ問題は、長らく対立すると思われてきた **AOT と JIT を
歩み寄らせ**ました。

純粋な AOT（事前コンパイル）は起動が速い代わりに、[なぜ JIT か](why-jit.md)で
見た「実行時の観測に基づく投機」ができません。純粋な JIT はその逆です。
そこで現代の処理系は、**両者の境界を探り直して**います。GraalVM の
Native Image は、アプリを**閉世界仮定の AOT** でネイティブ実行ファイルにし、
**初期化をビルド時に済ませてヒープごと焼き込む**ことで、起動を極限まで速くし
メモリも抑えます[](#cite:wimmer2019)。これは基本的に「実行時 JIT を捨てて、
起動時間とメモリ使用量を取る」側の解です。一方、通常の JVM/Graal JIT は
起動後の観測で攻めた最適化を行います。CPython 3.13 の JIT
（[次章](research-copy-and-patch.md)で扱う Copy-and-Patch）は、この対立の
JIT 側から「起動が軽い JIT」を志向した設計です。

「起動の速さ（AOT 的）」と「実行時適応の賢さ（JIT 的）」をどう両立させるか。
ウォームアップ問題は、この問いを研究の中心テーマに押し上げました。

## まとめ

- 「JIT は暖まれば速くなり、安定する」という素朴な信念は、注意深い
  実測[](#cite:barrett2017)では**しばしば成り立たない**。八処理系の計測で、
  安定した最高性能に一貫到達した組は最良でも 43.5% にとどまり、warmup の
  ほかに flat・slowdown・no steady state が現実に観測された。
- 原因は、脱最適化の繰り返し・再コンパイルの連鎖・GC やキャッシュの
  干渉など、JIT の適応的ふるまいの影の側面。
- ウォームアップのコストは、特に**短命なプロセス**で切実。毎回ゼロから
  暖め直す無駄が大きい。
- **コンパイルキャッシュ**（プロファイルの保存、生成コードの保存、暖まった
  状態のスナップショット）で、ウォームアップを二度払わない工夫が進む（CDS・
  ReadyNow・Leyden・CRaC）。保存物は起動時に**検証**し、外れたら安全に捨てる。
- ウォームアップ問題は **AOT と JIT を歩み寄らせ**、「速い起動と賢い適応の
  両立」を中心課題にした。

次章は、その「軽い JIT」を極限まで突き詰めた、最適化コンパイラの数百倍
速くコードを生成する技法 —— Copy-and-Patch を見ます。
