# トレーシング JIT の仕組み

[コンパイルの単位](compilation-units.md)で、トレーシング JIT を
「実際に通った 1 本の経路をコンパイルの単位にする方式」として紹介
しました。発想は美しいのですが、実装の細部は繊細です。「どこから記録を
始め、どこで止め、分岐をどう扱い、ループの途中で別の道へそれたらどうするか」。
この章では、トレーシングの中身に踏み込み、Ruby で**動くおもちゃの
トレーサ**を作って、その挙動を目で確かめます。

> [!NOTE]
> 本章のおもちゃの一部は集合（`Set`）を使います。Ruby 3.2 以降は自動で
> 読み込まれますが、それ以前で動かすなら各スニペットの冒頭に
> `require 'set'` を足してください。

## トレーシングの全体像

トレーシング JIT の一生は、おおむね次の流れです。

1. **ホットなループを見つける**：[プロファイリング](profiling.md)で見た
   バックエッジカウンタで、よく回るループを検出する。
2. **トレースを記録する**：そのループを 1 周する間に実行された演算を、
   分岐や呼び出しをまたいでも、**一直線の命令列**として記録する。
3. **ガードを置く**：記録中に通った分岐の地点に「今回と同じ方向か」を
   確かめるガードを差し込む。
4. **トレースを最適化してコンパイルする**：一直線なので最適化が簡単。
   機械語にする。
5. **トレースを実行する**：以後、そのループに来たら機械語のトレースを
   直接回す。ガードが破れたら**サイドイグジット**（side exit）で
   インタプリタへ戻る。

ひとつずつ、コードで見ていきます。題材は、おなじみの「0 から n-1 までの
和を求めるループ」です。これを小さなバイトコードで表します。

```ruby
def val(vars, x) = x.is_a?(Symbol) ? vars[x] : x

# sum = 0; i = 0; while i < n { sum += i; i += 1 }; halt
PROG = [
  [:mov,   :sum, 0],          # 0
  [:mov,   :i,   0],          # 1
  [:lt,    :t,   :i, :n],     # 2  ループ先頭（ヘッダ）
  [:gotof, :t,   7],          # 3  もし t が偽なら 7 へ（ループを抜ける）
  [:add,   :sum, :sum, :i],   # 4
  [:add,   :i,   :i,  1],     # 5
  [:goto,  2],                # 6  バックエッジ（ヘッダへ戻る）
  [:halt],                    # 7
]
```

まず、ふつうのインタプリタで動かしておきます。

```ruby
def interpret(prog, n)
  vars = { n: n }
  pc = 0
  while (insn = prog[pc])
    case insn[0]
    when :mov   then vars[insn[1]] = val(vars, insn[2]); pc += 1
    when :add   then vars[insn[1]] = val(vars, insn[2]) + val(vars, insn[3]); pc += 1
    when :lt    then vars[insn[1]] = val(vars, insn[2]) < val(vars, insn[3]); pc += 1
    when :gotof then pc = vars[insn[1]] ? pc + 1 : insn[2]
    when :goto  then pc = insn[1]
    when :halt  then return vars[:sum]
    end
  end
end

p interpret(PROG, 10)   # => 45
```

このインタプリタは命令ごとに `case` でディスパッチし、`pc` をたどります。
トレーシング JIT は、この**ディスパッチを消した一直線**を作りに行きます。

## トレースを記録する

ループが熱くなったとしましょう（ここではプロファイリングは省き、熱く
なったことにします）。記録モードでは、ヘッダから 1 周ぶん、実行した演算を
そのまま `trace` 配列へ書き出します。分岐 `gotof` に来たら、**今回進んだ
方向を前提とするガード**を記録します。バックエッジでヘッダに戻ってきたら、
1 周ぶん記録できたので記録を止めます。

```ruby
# ヘッダ header_pc から 1 周ぶんを一直線のトレースに記録する
def record_trace(prog, vars, header_pc)
  trace = []
  pc = header_pc
  loop do
    insn = prog[pc]
    case insn[0]
    when :mov   then trace << insn; vars[insn[1]] = val(vars, insn[2]); pc += 1
    when :add   then trace << insn; vars[insn[1]] = val(vars, insn[2]) + val(vars, insn[3]); pc += 1
    when :lt    then trace << insn; vars[insn[1]] = val(vars, insn[2]) < val(vars, insn[3]); pc += 1
    when :gotof
      cond = vars[insn[1]]
      trace << [:guard, insn[1]]          # 今回ループ内へ進んだ。それを前提に固定
      pc = cond ? pc + 1 : insn[2]
    when :goto
      return trace if insn[1] == header_pc  # ヘッダへ戻った＝1 周終わり
      pc = insn[1]
    end
  end
end

trace = record_trace(PROG, { n: 10, sum: 0, i: 0 }, 2)
trace.each { |t| p t }
# [:lt, :t, :i, :n]
# [:guard, :t]
# [:add, :sum, :sum, :i]
# [:add, :i, :i, 1]
```

できあがったトレースを見てください。`case` も `pc` のジャンプもありません。
**分岐 `gotof` は、`[:guard, :t]`（「`t` が真であり続けるかぎりトレースを
続ける」）という一行に化けました。** これがトレーシングの核心です。
制御フローのうち「今回通った道」だけを残し、それ以外を**ガードという
脱出口**に押し込めたのです。

> [!IMPORTANT]
> トレースには**条件分岐がありません**。あるのはガードだけです。ガードは
> 「分岐」ではなく「前提のチェック」です。前提が成り立つかぎり一直線に
> 流れ、成り立たなくなった瞬間だけ外へ出る。だからトレースの本体は
> 完全な直線コードになり、[中間表現](ir.md)で見た直線向きの最適化
> （定数畳み込み、共通部分式削除）がそのまま強力に効きます。

## トレースを実行する：ガードとサイドイグジット

記録したトレースは、ループとして繰り返し実行できます。各周回でガードを
チェックし、破れたら**サイドイグジット**してインタプリタへ戻ります。
ここでは戻り先の代わりに、ただ結果を返すことで脱出を表します。

```ruby
# 一直線のトレースを、ガードが破れるまで繰り返す
def run_trace(trace, vars)
  iters = 0
  loop do
    trace.each do |insn|
      case insn[0]
      when :mov   then vars[insn[1]] = val(vars, insn[2])
      when :add   then vars[insn[1]] = val(vars, insn[2]) + val(vars, insn[3])
      when :lt    then vars[insn[1]] = val(vars, insn[2]) < val(vars, insn[3])
      when :guard then return [vars[:sum], iters] unless vars[insn[1]]  # サイドイグジット
      end
    end
    iters += 1
  end
end

sum, iters = run_trace(trace, { n: 100, sum: 0, i: 0 })
p [sum, iters]   # => [4950, 100]
```

`n = 10` のループから記録したトレースを、`n = 100` の新しい状態で回しても、
ちゃんと `4950`（0+1+…+99）が出て、100 周したところでガード `i < n` が
破れてサイドイグジットしています。**トレースはループの構造に依存せず、
記録した演算の並びと前提（ガード）だけで動く**。ここに、トレーシングが
「実行経路を単位にする」と言うことの実体があります。

> [!NOTE]
> 本物のトレーシング JIT では、`run_trace` の中身が `case` ディスパッチ
> ではなく**機械語**になります。そうなれば `[:add, :sum, :sum, :i]` は
> CPU の `add` 命令そのものになり、ループはディスパッチゼロで回ります。
> ここではしくみを見るために Ruby で「トレースのインタプリタ」を
> 書きましたが、最後の一段を[最小の JIT](hello-jit.md)の機械語生成に
> 差し替えれば、本物に近づきます。

## 分岐が分かれるとき：トレースの木

ここまでは「ループが毎回同じ道を通る」前提でした。現実には、ループの中の
分岐が**ときどき違う方向**へ行くことがあります。

```ruby
# n 個の数のうち、偶数だけ合計するループ（分岐が交互に分かれる）
# while i < n { if even?(i) { sum += i }  i += 1 }
```

最初に記録したトレースは「`even?(i)` が真だった道」かもしれません。すると
奇数の `i` のとき、`even?` のガードが破れてサイドイグジットします。毎回
偶奇が交代すると、半分の周回でトレースから落ちることになり、台無しです。

トレーシング JIT はこれを**トレースの木**（trace tree）で解決します。
組み込み Java（JVM バイトコード）向けの HotpathVM[](#cite:gal2006) は、最初に
記録した本トレース（primary trace）を**1 本の道**としてまず作り、あるガードが
**何度も破れる**（その脱出口が熱くなる）と、「そこから先の道」を**子トレース**
（child trace、論文では secondary trace）として記録して、元のガードに
**つなぎ込み**ます。具体的には、脱出してインタプリタへ戻る代わりに、ガードの
脱出ジャンプ先をその子トレースの先頭へ書き換えます[](#cite:gal2006)。こうして
ガードからガードへ枝分かれする木が育ち、よく通る道はすべて機械語のトレースで
覆われていきます。

> [!NOTE]
> **「トレースの木（trace tree）」という呼び名そのものは、後年に定着した
> 用語**です。HotpathVM の論文[](#cite:gal2006)の本文には「trace tree」という
> 語はなく、本トレースの脱出口へ二次トレース（secondary trace）をつなぎ込んで
> いく**仕組み**として記述されています。この仕組みを「木」という言葉で前面に
> 押し出した整理が広まり、いまではこの方式全体を指して「トレース木」と
> 呼ぶのが一般的です。本書も以降この呼び名を使いますが、指している中身は
> あくまで gal2006 が示した「ガードへ二次トレースをつなぎ込む」機構です。

```
   ループヘッダ
        │
   [本トレース：偶数だった道]
        │
   guard even?(i) ─破れがち→ [サイドトレース：奇数だった道]
        │                          │
       戻る ←──────────────────────┘
```

この**枝分かれの管理**こそ、トレーシング JIT の実装が難しいと言われる
最大の理由です。木が大きくなりすぎないか、ガードの脱出が偏っていないか、
別のトレースとどこでつなぐか —— これらの判断が処理系ごとに工夫されて
います。

## どこからトレースを始めるか

もう一つの設計判断が、**トレースの開始点**です。素直なのは「ホットな
ループのヘッダから」ですが、それだけではありません。

- **ループヘッダ起点**：いちばん基本。LuaJIT や TraceMonkey の出発点。
- **関数の入口起点**：ループでない熱い関数も対象にしたい場合。
- **サイドイグジット起点**：上で見たように、熱くなった脱出口から新しい
  トレースを伸ばす。木を育てる原動力。

トレースをどこまで伸ばすか（停止条件）も重要です。無限に伸ばすと
コンパイル単位が巨大になりすぎるので、「一定の長さを超えたら打ち切る」
「呼び出しが深くなりすぎたら止める」といった上限を設けます。再帰呼び出しの
扱いも厄介で、トレーシング JIT の論文の多くがこの停止条件に紙幅を割いて
います。

## トレース上の最適化

トレースが一直線であることは、最適化に絶大な利点をもたらします。

- **ループ不変式の巻き上げ**：ループの中で値が変わらない計算
  （`length` の取得など）を、トレースの外（ループの前）へ 1 回だけ
  追い出せます。一直線なので「本当に不変か」の判定が簡単です。
- **ガードの巻き上げ・統合**：同じガードが毎周回チェックされているなら、
  ループの外へ 1 回だけ出せます。連続する重複ガードも 1 つにまとめられます。
- **アロケーションの除去**：ループ内で作って同じループ内で捨てるだけの
  短命オブジェクトは、そもそも作らずにレジスタ上の値で済ませられます
  （[エスケープ解析](glossary.md)とスカラ置換）。

これらは [中間表現](ir.md) で見た古典的最適化と同じものですが、対象が
一直線のトレースだと**ぐっと素直に書ける**のがポイントです。分岐の合流
（φ 関数）に悩まされず、上から下へなめるだけで多くが片付きます。
トレーシングが「最適化しやすい」と言われるのはこのためです。

「素直に書ける」とはどの程度のものか、実際に**記録済みのトレース配列を
変換するおもちゃ**を書いて確かめましょう。題材として、少し冗長なトレースを
手で用意します。`a` は配列、`a.length` はループ内で変わらない（不変式）と
します。

```ruby
# while i < a.length { x = a.length;  sum += a[i] * x;  i += 1 }
# を 1 周記録したような、わざと冗長なトレース
TR = [
  [:guard_type, :a, :Array],    # a は Array か（型ガード）
  [:len,        :L, :a],        # L = a.length      ← 毎周同じ＝不変
  [:lt,         :t, :i, :L],    # t = i < L
  [:guard,      :t],            # t が真であり続けるか
  [:guard_type, :a, :Array],    # a は Array か（重複ガード）
  [:len,        :x, :a],        # x = a.length      ← また同じ計算
  [:idx,        :e, :a, :i],    # e = a[i]
  [:mul,        :m, :e, :x],    # m = e * x
  [:add,        :sum, :sum, :m],
  [:add,        :i, :i, 1],
]
```

このトレースには無駄が二つあります。`[:len, ...]` がループ内で 2 回計算
されている（しかも結果は毎周同じ）こと、`[:guard_type, :a, :Array]` が
2 回現れていることです。一直線なので、これらは上から下へ一度なめるだけで
見つけられます。

### ループ不変式の巻き上げを実装する

ある命令が**ループ不変**とは、「その入力がすべてループ前から確定して
いて、ループ内で書き換えられない」ことです。一直線のトレースでは、
代入される変数の集合（ループ内で変化する変数 = **可変変数**）を上から
集めていき、「入力がどれも可変変数でない命令」を不変と判定できます。
不変と分かった命令は、トレース本体（`body`）からループ前（`pre`）へ
移します。

```ruby
WRITES = { mov: 0, add: 0, lt: 0, len: 0, idx: 0, mul: 0 }  # insn[1] へ書く命令
def writes_var(insn) = WRITES.key?(insn[0]) ? insn[1] : nil
# その命令が読む変数（Symbol）を列挙
def reads_vars(insn)
  case insn[0]
  when :mov  then [insn[2]]
  when :len  then [insn[2]]
  when :idx  then [insn[2], insn[3]]
  when :add, :mul, :lt then [insn[2], insn[3]]
  when :guard then [insn[1]]
  when :guard_type then [insn[1]]
  else []
  end.select { |v| v.is_a?(Symbol) }
end

def hoist_invariants(trace)
  # まず「ループ内で書き換えられる変数」を集める
  mutated = trace.filter_map { |insn| writes_var(insn) }.to_set
  pre, body = [], []
  hoisted = Set.new   # すでに巻き上げた変数（その値も不変扱いになる）
  trace.each do |insn|
    inputs = reads_vars(insn)
    dst    = writes_var(insn)
    # 入力がどれも可変変数でなく（または巻き上げ済み）、ガードでなく、書き先がある
    invariant = dst && inputs.all? { |v| !mutated.include?(v) || hoisted.include?(v) }
    if invariant
      pre << insn
      hoisted << dst        # この変数は今後「不変」として扱える
      mutated.delete(dst)   # 巻き上げた以上ループ内では書かれない
    else
      body << insn
    end
  end
  [pre, body]
end

pre, body = hoist_invariants(TR)
puts "--- pre (ループ前に 1 回) ---"; pre.each { |t| p t }
puts "--- body (毎周) ---";          body.each { |t| p t }
```

実行すると、`[:len, :L, :a]` と `[:len, :x, :a]` の両方が `pre` へ移り、
ループ本体からは消えます（`a` は可変変数ではないので `len` は不変です）。

```
--- pre (ループ前に 1 回) ---
[:len, :L, :a]
[:len, :x, :a]
--- body (毎周) ---
[:guard_type, :a, :Array]
[:lt, :t, :i, :L]
[:guard, :t]
[:guard_type, :a, :Array]
[:idx, :e, :a, :i]
[:mul, :m, :e, :x]
[:add, :sum, :sum, :m]
[:add, :i, :i, 1]
```

`length` の取得が毎周から消え、ループ前に 1 回だけ計算されるように
なりました。分岐の合流がないので、「入力が可変か」を上から舐めるだけで
済んでいる点に注目してください。本物のメソッド JIT なら、ループに複数の
入口（合流点）があり、φ 関数をたどって不変性を証明する手間がかかります。

> [!NOTE]
> ここで巻き上げた `[:len, :a]` のような計算をループ前に出せるのは、
> 「ループの間 `a.length` は変わらない」という**前提**を置いているからです。
> もし途中で `a` に要素が足されたら、この前提は崩れます。本物の処理系では、
> 巻き上げと同時に「`a` が変更されたら脱出する」ガードを 1 つ立て、
> その前提を守ります。最適化はいつでも**ガードで前提を保証して初めて
> 成立する**のです。

### 連続する重複ガードを統合する

次に、`[:guard_type, :a, :Array]` が 2 回現れている問題です。同じ前提を
2 回確かめるのは無駄なので、**すでに確かめ済みのガード**を覚えておき、
重複を消します。一直線なら、間で `a` が書き換えられていないかぎり、
2 回目のガードは 1 回目に**支配**（dominate）されていて必ず成り立ちます。

```ruby
def merge_guards(trace)
  seen = {}          # すでに成立が分かっているガード => true
  out  = []
  trace.each do |insn|
    if insn[0] == :guard_type
      key = [insn[1], insn[2]]
      next if seen[key]      # 同じ型ガードは 2 回目以降を捨てる
      seen[key] = true
    end
    # ガードの対象変数が書き換えられたら、その前提は無効化する
    if (dst = writes_var(insn))
      seen.reject! { |(var, _ty), _| var == dst }
    end
    out << insn
  end
  out
end

merged = merge_guards(body)
merged.each { |t| p t }
```

実行すると 2 つ目の `[:guard_type, :a, :Array]` が消えます。`reject!` の
一行が効いていて、もし途中に `[:mov, :a, ...]` のように `a` を書き換える
命令が入れば、その後の `a` の型ガードは**消されません**（前提が崩れたから）。
このように「支配されているか」を、一直線では「すでに `seen` に入っていて、
かつ間で書き換えられていないか」という単純な走査で判定できます。

> [!TIP]
> 巻き上げと統合を続けて適用すると（`merge_guards(hoist_invariants(TR)[1])`）、
> ループ本体は `lt → guard → idx → mul → add → add` という、無駄のない
> 一直線まで痩せます。これらは [中間表現](ir.md) の共通部分式削除・ループ
> 不変式移動とまったく同じ最適化ですが、**φ 関数も支配木の構築もいらず**、
> 数行で書けてしまうのがトレースの強みです。LuaJIT[](#cite:luajit) は
> この性質を突き詰め、トレースを SSA 風の中間表現に直して、定数畳み込み・
> ストア転送・ループ不変式移動を 1 パスの線形走査で次々に適用します。

### トレードオフ：一直線の代償

ただ、この素直さにはきっちり**代償**があります。

- **道ごとにコードが重複する**：分岐の両方向がどちらも熱いと、本来 1 つで
  済んだ命令列が、トレースごとに丸ごとコピーされます。コードサイズが
  膨らみ、命令キャッシュを汚します。メソッド JIT が 1 つの制御フローグラフに
  まとめるのと対照的です。
- **最適化が「今回の道」しか見ない**：トレース上の不変式移動は、記録した
  1 本の道だけを見ます。別の道では不変でないかもしれず、その判断は
  ガードに肩代わりさせます。前提が外れる頻度が高いと、巻き上げた計算が
  むしろ無駄打ちになります。
- **巻き上げた計算とガードはセット**：上の `[!NOTE]` のとおり、巻き上げる
  たびに前提を守るガードが要ります。ガードが増えれば[脱最適化](deopt-osr.md)
  情報も増え、後で見る**スナップショット**の量がふくらみます。

つまりトレーシングの最適化は「直線だから速く強く書けるが、その強さは
ガードという前提に全面的にもたれている」のです。次の節では、その前提が
外れたときに安全に戻るための仕掛け —— スナップショットを実装します。

## サイドイグジット時の状態復元：スナップショット

最初の `run_trace` では、ガードが破れたとき `vars[:sum]` をそのまま返す
だけで済みました。`vars` がインタプリタと同じ「素のハッシュ」だったから
です。ところが、前節の最適化を進めると話が変わります。巻き上げや重複
削除でトレース内の変数の一部が**消えたり、別名になったり、レジスタに
畳み込まれたり**します。すると、ガードが破れてインタプリタへ戻るとき、
「インタプリタが期待する形の変数の束」をその場で**作り直す**必要があります。

このとき各ガードに持たせる「そこで生きている変数 → どう復元するか」の
対応表を**スナップショット**（snapshot）と呼びます。考え方は
[脱最適化](deopt-osr.md)の**復元のレシピ**とまったく同じものです
[](#cite:gal2009)。違いは、トレーシングではガードが文字どおり山ほど並ぶので、
スナップショットを**前のガードからの差分**で持つのが定石、という点くらいです。

おもちゃで体感しましょう。スナップショットに**値そのものを焼き込んでは
いけません**。記録時の値（`n=10` のときの `sum=0` など）を固定してしまうと、
別の `n` で回したときに間違った状態へ戻してしまいます。正しくは「インタプリタ
の変数 `:sum` は、トレース内の値 `:sum` から作れ」という**出どころ（作り方）**
を記録し、復元時に**そのときの生きた値**を引きます。ここでは出どころが同名
なので恒等写像ですが、最適化で別名や再計算が絡むと、この写像が効いてきます。

```ruby
# ガードに「インタプリタ変数 => トレース内の出どころ」のスナップショットを添える
def record_with_snapshot(prog, vars, header_pc, live)
  trace = []
  pc = header_pc
  loop do
    insn = prog[pc]
    case insn[0]
    when :mov then trace << insn; vars[insn[1]] = val(vars, insn[2]); pc += 1
    when :add then trace << insn; vars[insn[1]] = val(vars, insn[2]) + val(vars, insn[3]); pc += 1
    when :lt  then trace << insn; vars[insn[1]] = val(vars, insn[2]) < val(vars, insn[3]); pc += 1
    when :gotof
      # 生きている変数の「出どころ」を写し取る（値ではなく作り方）。
      # ここでは恒等：インタプリタの v は、トレース内の v から作れる。
      snapshot = live.to_h { |v| [v, v] }
      trace << [:guard, insn[1], { restore_pc: insn[2], snapshot: snapshot }]
      pc = vars[insn[1]] ? pc + 1 : insn[2]
    when :goto
      return trace if insn[1] == header_pc
      pc = insn[1]
    end
  end
end

tr = record_with_snapshot(PROG, { n: 10, sum: 0, i: 0 }, 2, [:sum, :i, :n])
p tr[1]
# [:guard, :t, {:restore_pc=>7, :snapshot=>{:sum=>:sum, :i=>:i, :n=>:n}}]
```

ガードに、抜けた先の `pc`（`7` = `halt`）と、生きている各変数の**出どころ**が
添えられました。実行時、このガードが破れたら、スナップショットの**作り方
どおりに**そのときの生きた値からインタプリタの変数フレームを組み立て、
`restore_pc` からインタプリタを再開すればよいわけです。

```ruby
# ガードが破れたら、スナップショットからインタプリタ状態を復元して返す
def run_trace_snap(trace, vars)
  loop do
    trace.each do |insn|
      case insn[0]
      when :mov then vars[insn[1]] = val(vars, insn[2])
      when :add then vars[insn[1]] = val(vars, insn[2]) + val(vars, insn[3])
      when :lt  then vars[insn[1]] = val(vars, insn[2]) < val(vars, insn[3])
      when :guard
        next if vars[insn[1]]
        # サイドイグジット：いまの生きた値を、出どころに従って組み立てる
        restored = {}
        insn[2][:snapshot].each { |interp_var, src| restored[interp_var] = vars.fetch(src) }
        return { pc: insn[2][:restore_pc], frame: restored }
      end
    end
  end
end

p run_trace_snap(tr, { n: 5, sum: 0, i: 0 })
# {:pc=>7, :frame=>{:sum=>10, :i=>5, :n=>5}}  ← n=5 でも正しく sum=10 で戻れる
```

記録時は `n=10` だったのに、`n=5` で回しても `sum=10`（0+1+…+4）で正しく
戻れている点が肝心です。スナップショットが**値ではなく出どころ**を持つから、
どんな実行状態からでも正しいインタプリタ状態を組み立てられます。そして、
この `src`（出どころ）の部分こそが、
[脱最適化](deopt-osr.md)で見た**再具体化**（rematerialization）の差し込み口
です。もし最適化で消した中間変数があるなら、`snapshot` のエントリを
「いまある値」ではなく「`i * 2` を作れ」のような**作り方**にしておき、
復元時にそのレシピを評価して値をよみがえらせます。トレーシング JIT の
スナップショットは、まさにこの「値ではなく作り方を持つ」設計でガードあたりの
情報量を切り詰めています。深掘りは[脱最適化と OSR](deopt-osr.md)に譲ります。

> [!IMPORTANT]
> 巻き上げ・ガード統合・後で見るアロケーションシンキングは、どれも
> 「トレース内では値を消す／動かす」最適化です。消した値はサイドイグジット
> の瞬間に**インタプリタが期待する形へ戻せなければなりません**。だから
> トレーシングでは、最適化とスナップショットは**つねにセット**で設計します。
> 速くするほど戻し方は複雑になる —— これが[脱最適化](deopt-osr.md)の
> 「最適化の自由」対「復元情報の大きさ」の綱引きそのものです。

## トレースの連結とトレース木の育成

「[分岐が分かれるとき：トレースの木](#分岐が分かれるとき：トレースの木)」で、
熱くなった脱出口からサイドトレースを伸ばして木を育てる、と述べました。
ここではその**つなぎ込みの手順**を、もう少し実装に寄せて具体化します。

まず、ガードごとに「ここから何回脱出したか」を数えるカウンタを持ちます
（[プロファイリング](profiling.md)のカウンタと同じ発想です）。脱出回数が
しきい値を超えたガードを**熱い脱出口**とみなします。

```ruby
# 各ガードの脱出回数を数え、熱くなった脱出口を見つける
class TraceTree
  HOT = 50
  def initialize(root) = (@root = root; @counts = Hash.new(0); @sides = {})

  def on_side_exit(guard_id)
    @counts[guard_id] += 1
    @counts[guard_id] == HOT ? :record_side_trace : :interpret
  end

  # 記録したサイドトレースを、脱出元ガードへつなぎ込む
  def attach(guard_id, side_trace)
    @sides[guard_id] = side_trace
  end
end
```

つなぎ込みの一連の流れは、次のようになります。

1. ある周回で、ガード `g` が破れてサイドイグジットする。
2. `on_side_exit(g)` がカウンタを上げる。しきい値に達したら、
   インタプリタへ戻るその場で**記録モードに入る**。
3. **脱出元のスナップショットを初期状態として**、そこから先の道を
   1 本のサイドトレースとして記録する（記録のしかたは本トレースと同じ）。
4. サイドトレースの末尾が元のループヘッダ（や別の既存トレース）に
   合流したら、そこで記録を止め、`attach(g, side_trace)` で**ガード `g` の
   脱出先をサイドトレースへ差し替える**。

差し替えとは、機械語のレベルでは「ガード `g` の脱出ジャンプ先を、
インタプリタ復帰スタブから、新しいサイドトレースの先頭アドレスへ書き換える」
ことです。これを**バックパッチ**（backpatching）と呼びます。一度つなげば、
以後そのガードが破れても、インタプリタへは戻らずサイドトレースへ滑り込み、
機械語のまま走り続けられます。

```
   ループヘッダ ──→ [本トレース] ──→ guard g ──(成立)──→ ループ末尾へ
                                       │
                                  (50 回破れた)
                                       │ バックパッチで差し替え
                                       ▼
                              [サイドトレース] ──→ ループヘッダへ合流
```

ここで効くのが前節の**スナップショット**です。サイドトレースの記録は
「ガード `g` のスナップショットが復元したインタプリタ状態」から始まります。
スナップショットがあるからこそ、本トレースの途中から自然に枝を生やせる
わけです。スナップショットとトレース木は、こうして表裏一体になっています
[](#cite:gal2006)。

> [!NOTE]
> 合流先は「元のループヘッダ」とは限りません。別のサイドトレースの途中へ
> 合流させると木がますます複雑になるため、多くの実装は**合流先をループ
> ヘッダか既存トレースの先頭に限る**という制約を置きます。これも
> 「最適化の自由を縛って実装を単純に保つ」一例です。LuaJIT は、こうして
> 育った木全体を 1 つの領域として扱い、トレース間で値を受け渡します。

## アボートとブラックリスト

トレースの記録は、いつも成功するとは限りません。次のような場合、記録を
**途中で諦める**ことになります。これを**アボート**（abort）と呼びます。

- **長すぎる**：[どこからトレースを始めるか](#どこからトレースを始めるか)で
  ふれた停止条件に引っかかり、上限の命令数を超えてもループヘッダに戻って
  こない。
- **伸ばせない構造**：例外送出、トレース化していない C 関数呼び出し、
  あるいは記録器が扱えない命令に出くわす。
- **合流できない**：再帰が深くなりすぎる、末尾がどの既存トレースにも
  つながらない。

アボートしたトレースをただ捨てるだけだと、困ったことが起きます。その
ループは相変わらず熱いので、次にしきい値を超えるとまた記録を始め、また
同じ理由でアボートする。これを延々と繰り返し、**記録のコストばかり
かかって 1 つもトレースが生まれない**状態 —— これを**暴発**（thrashing）と
言います。

対策は単純で、**何度もアボートするループを「トレース禁止」に登録**します。
この登録簿を**ブラックリスト**（blacklist）と呼びます。

```ruby
class Recorder
  ABORT_LIMIT = 3
  def initialize = (@aborts = Hash.new(0); @blacklist = Set.new)

  # このループを記録してよいか？
  def may_record?(loop_pc) = !@blacklist.include?(loop_pc)

  # 記録に失敗したとき呼ぶ。規定回数を超えたらブラックリスト入り
  def on_abort(loop_pc, reason)
    @aborts[loop_pc] += 1
    if @aborts[loop_pc] >= ABORT_LIMIT
      @blacklist << loop_pc      # 以後このループは二度と記録しない
      :blacklisted
    else
      :will_retry
    end
  end
end

rec = Recorder.new
3.times { p rec.on_abort(2, :too_long) }
# :will_retry
# :will_retry
# :blacklisted
p rec.may_record?(2)   # => false（もう記録しに行かない）
```

ブラックリスト入りしたループは、以後ずっとインタプリタで実行されます。
「トレース化できないなら、潔く諦めてインタプリタに任せる」という割り切りが、
処理系全体の安定性を守ります。

> [!WARNING]
> しきい値の設定はとても繊細です。アボート上限が小さすぎると、たまたま
> 数回失敗しただけの「本当は熱くて素直なループ」までブラックリスト入りし、
> 永遠にインタプリタのままになります。逆に大きすぎると、暴発が長引いて
> ウォームアップが遅くなります。[ティアドコンパイル](tiered.md)同様、
> ここのチューニングは処理系の体感速度を左右する勘どころです。

## アロケーションシンキング：短命オブジェクトを消す

最後に、トレーシングがとりわけ得意とする最適化、**アロケーション
シンキング**（allocation sinking）を見ます[](#cite:stadler2014)。狙いは、「ループ内で作り、同じ
ループ内で使い切るだけの短命オブジェクト」を、トレースから**消し去る**
ことです。

題材として、`[x, y]` のような小さな配列（ペア）をループ内で作っては
すぐ要素を取り出すだけ、というありがちな処理を考えます。

```ruby
# while i < n { p = [i, i*2];  sum += p[0] + p[1];  i += 1 }
# ペア p は作ってすぐ分解するだけ＝ループの外へ漏れない
TR2 = [
  [:mul,    :d,   :i, 2],       # d = i*2
  [:alloc,  :p,   :d2, :i, :d], # p = new Pair(i, d)   ← 短命な確保
  [:field,  :a,   :p, 0],       # a = p[0]
  [:field,  :b,   :p, 1],       # b = p[1]
  [:add,    :s,   :a, :b],      # s = a + b
  [:add,    :sum, :sum, :s],
  [:add,    :i,   :i, 1],
]
```

`p` は作られた後、自分の要素を読まれるだけで、ループの外へは漏れません
（**エスケープしない**）。ならば `p` を実際に確保する必要はなく、`p[0]` は
`i` に、`p[1]` は `d` に置き換えてしまえます。これを行うのがアロケーション
シンキングです。一直線のトレースなら、「確保された変数のフィールドが、
どの変数から来たか」を上から追うだけで実現できます。

```ruby
def sink_allocations(trace)
  fields  = {}    # 生きている確保: p => { 0 => :i, 1 => :d }
  escaped = Set.new
  out = []
  trace.each do |insn|
    case insn[0]
    when :alloc
      # p = new(field0, field1) を、確保せず「中身の対応」だけ覚える
      obj, *vals = insn[1], *insn[3..]
      fields[obj] = vals.each_with_index.to_h { |v, k| [k, v] }
      # この alloc 命令はトレースから消す（出力に積まない）
    when :field
      dst, obj, k = insn[1], insn[2], insn[3]
      if fields.key?(obj)
        out << [:mov, dst, fields[obj][k]]   # p[0] → もとの変数の参照に
      else
        out << insn
      end
    else
      out << insn
    end
  end
  [out, fields]   # fields は復元レシピとして使う
end

sunk, fields = sink_allocations(TR2)
sunk.each { |t| p t }
p fields
```

実行すると `[:alloc, ...]` が消え、`p[0]`・`p[1]` の読み出しが
`[:mov, :a, :i]`・`[:mov, :b, :d]` に化けます。ループからオブジェクト確保が
丸ごと消え、ガベージコレクションの圧力もなくなりました。

ところが —— `p` が本当に外へ漏れないと**今回のトレースでは**言えても、
ガードが破れてサイドイグジットした先のインタプリタは、`p` という
オブジェクトが存在する前提で動いています。だから、消した `p` を
**サイドイグジット時にだけ実体化し直す**必要があります。これが
[脱最適化](deopt-osr.md)で見た**再具体化**そのものです。`sink_allocations`
が返した `fields`（「`p` は `i` と `d` から `new Pair` で作れる」という
レシピ）を、そのガードのスナップショットへ埋め込んでおきます。

```ruby
# サイドイグジットの瞬間に、消したオブジェクトをレシピから実体化する
def rematerialize(fields, vars)
  fields.transform_values { |slots| slots.values.map { |src| vars.fetch(src, src) } }
end

# 例：脱出時に i=7, d=14 だったら…
p rematerialize(fields, { i: 7, d: 14 })
# {:p=>[7, 14]}   ← 消していた p を、戻る瞬間だけ作り直す
```

つまりアロケーションシンキングは、「**速い道では確保を消し、外れた道に
出る瞬間だけ確保し直す**」という賭けです。ループが最後まで素直に回る
かぎり、オブジェクトは一度も作られません。これがトレーシング（や
[脱最適化](deopt-osr.md)を備えたメソッド JIT）の数値計算での強さの大きな
源泉です。

> [!TIP]
> アロケーションシンキングは、トレースが一直線だからこそ「このオブジェクトは
> 外へ漏れない」を上から下へなめるだけで証明できます。一般の制御フロー
> グラフ上でこれをやるのが[エスケープ解析](glossary.md)で、分岐の合流を
> またいで「どの道でも漏れないか」を示す必要があり、ずっと手間がかかります。
> トレーシングは、難しい解析を**直線という形で前払い**しているのです。

## トレーシングの歴史と到達点

トレーシングのルーツは、JIT ですらない動的最適化器 **Dynamo**
[](#cite:bala2000) にあります。Dynamo は機械語を実行しながら熱いトレースを
見つけて最適化し直しました。これを JIT に応用したのが、**資源制約デバイス**
（組み込み Java、JVM バイトコード）向けの **HotpathVM**[](#cite:gal2006) と、
JavaScript 向けの **TraceMonkey**[](#cite:gal2009) です。HotpathVM は、頻繁に
実行されるループヘッダを見つけてトレース記録を始め、本トレースの熱い脱出口へ
子トレースをつなぎ込んで木を育てる方式で、素直に回るプログラムでは高速な
インタプリタに対して**7〜11 倍**の高速化を報告しています[](#cite:gal2006)。
そして商用品質に磨き上げたのが
[LuaJIT](case-luajit.md) と、次章で扱う**メタトレーシング**の
[PyPy](case-python.md) です。

> [!TIP]
> トレーシングは万能ではありません。[コンパイルの単位](compilation-units.md)で
> ふれたとおり、分岐が毎回ばらつくコードではトレースの木が膨れ、かえって
> 不利になります。JavaScript の主要エンジンがトレーシングからメソッド
> JIT へ回帰したのも、ウェブのコードが「素直なループ」ばかりではない
> ことが大きな理由でした。一方、数値計算やインタプリタのように「同じ道を
> ひたすら回る」コードでは、トレーシングが圧倒的に強い。**ワークロード
> 次第**です。

## まとめ

- トレーシング JIT は、ホットなループを 1 周記録し、**条件分岐をガードに
  変えた一直線のトレース**を作って実行する。
- トレースには分岐がなく、あるのは**ガード**（前提のチェック）だけ。
  前提が破れたら**サイドイグジット**でインタプリタへ戻る。
- 分岐がばらつくと、熱い脱出口から枝を伸ばして**トレースの木**を育てる。
  脱出回数を数え、熱い脱出口からサイドトレースを記録し、**バックパッチ**で
  ガードへつなぎ込む。この管理がトレーシング実装の最難関。
- 一直線ゆえに**ループ不変式の巻き上げ・重複ガードの統合・アロケーション
  シンキング**が、φ 関数も支配木もなしに数行で書ける。ただし道ごとの
  コード重複や「今回の道しか見ない」代償も伴う。
- 最適化が値を消すなら、ガードごとの**スナップショット**で外れた瞬間に
  状態を復元する。消したオブジェクトは**再具体化**で作り直す。これは
  [脱最適化](deopt-osr.md)の復元レシピと同じ思想。
- 記録に何度も失敗するループは**アボート**を数えて**ブラックリスト**入り
  させ、**暴発**を防ぐ。
- ルーツは Dynamo、応用が HotpathVM と TraceMonkey、到達点が LuaJIT と PyPy。

次章では、このトレーシングの威力を**インタプリタを書くだけで**手に入れる、
驚くべき技法 —— メタトレーシングへ進みます。
