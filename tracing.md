# トレーシング JIT の仕組み

[コンパイルの単位](compilation-units.md)で、トレーシング JIT を
「実際に通った 1 本の経路をコンパイルの単位にする方式」として紹介
しました。発想は美しいのですが、実装の細部は繊細です。「どこから記録を
始め、どこで止め、分岐をどう扱い、ループの途中で別の道へそれたらどうするか」。
この章では、トレーシングの中身に踏み込み、Ruby で**動くおもちゃの
トレーサ**を作って、その挙動を目で確かめます。

## トレーシングの全体像

トレーシング JIT の一生は、おおむね次の流れです。

1. **ホットなループを見つける**：[プロファイリング](profiling.md)で見た
   バックエッジカウンタで、よく回るループを検出する。
2. **トレースを記録する**：そのループを 1 周する間に実行された演算を、
   分岐や呼び出しをまたいでも、**一直線の命令列**として記録する。
3. **ガードを置く**：記録中に通った分岐の地点に「今回と同じ方向か」を
   確かめるガードを差し込む。
4. **トレースを最適化してコンパイルする**：一直線なので最適化が簡単。
   機械語にする。
5. **トレースを実行する**：以後、そのループに来たら機械語のトレースを
   直接回す。ガードが破れたら**サイドイグジット**（side exit）で
   インタプリタへ戻る。

ひとつずつ、コードで見ていきます。題材は、おなじみの「0 から n-1 までの
和を求めるループ」です。これを小さなバイトコードで表します。

```ruby
def val(vars, x) = x.is_a?(Symbol) ? vars[x] : x

# sum = 0; i = 0; while i < n { sum += i; i += 1 }; halt
PROG = [
  [:mov,   :sum, 0],          # 0
  [:mov,   :i,   0],          # 1
  [:lt,    :t,   :i, :n],     # 2  ループ先頭（ヘッダ）
  [:gotof, :t,   7],          # 3  もし t が偽なら 7 へ（ループを抜ける）
  [:add,   :sum, :sum, :i],   # 4
  [:add,   :i,   :i,  1],     # 5
  [:goto,  2],                # 6  バックエッジ（ヘッダへ戻る）
  [:halt],                    # 7
]
```

まず、ふつうのインタプリタで動かしておきます。

```ruby
def interpret(prog, n)
  vars = { n: n }
  pc = 0
  while (insn = prog[pc])
    case insn[0]
    when :mov   then vars[insn[1]] = val(vars, insn[2]); pc += 1
    when :add   then vars[insn[1]] = val(vars, insn[2]) + val(vars, insn[3]); pc += 1
    when :lt    then vars[insn[1]] = val(vars, insn[2]) < val(vars, insn[3]); pc += 1
    when :gotof then pc = vars[insn[1]] ? pc + 1 : insn[2]
    when :goto  then pc = insn[1]
    when :halt  then return vars[:sum]
    end
  end
end

p interpret(PROG, 10)   # => 45
```

このインタプリタは命令ごとに `case` でディスパッチし、`pc` をたどります。
トレーシング JIT は、この**ディスパッチを消した一直線**を作りに行きます。

## トレースを記録する

ループが熱くなったとしましょう（ここではプロファイリングは省き、熱く
なったことにします）。記録モードでは、ヘッダから 1 周ぶん、実行した演算を
そのまま `trace` 配列へ書き出します。分岐 `gotof` に来たら、**今回進んだ
方向を前提とするガード**を記録します。バックエッジでヘッダに戻ってきたら、
1 周ぶん記録できたので記録を止めます。

```ruby
# ヘッダ header_pc から 1 周ぶんを一直線のトレースに記録する
def record_trace(prog, vars, header_pc)
  trace = []
  pc = header_pc
  loop do
    insn = prog[pc]
    case insn[0]
    when :mov   then trace << insn; vars[insn[1]] = val(vars, insn[2]); pc += 1
    when :add   then trace << insn; vars[insn[1]] = val(vars, insn[2]) + val(vars, insn[3]); pc += 1
    when :lt    then trace << insn; vars[insn[1]] = val(vars, insn[2]) < val(vars, insn[3]); pc += 1
    when :gotof
      cond = vars[insn[1]]
      trace << [:guard, insn[1]]          # 今回ループ内へ進んだ。それを前提に固定
      pc = cond ? pc + 1 : insn[2]
    when :goto
      return trace if insn[1] == header_pc  # ヘッダへ戻った＝1 周終わり
      pc = insn[1]
    end
  end
end

trace = record_trace(PROG, { n: 10, sum: 0, i: 0 }, 2)
trace.each { |t| p t }
# [:lt, :t, :i, :n]
# [:guard, :t]
# [:add, :sum, :sum, :i]
# [:add, :i, :i, 1]
```

できあがったトレースを見てください。`case` も `pc` のジャンプもありません。
**分岐 `gotof` は、`[:guard, :t]`（「`t` が真であり続けるかぎりトレースを
続ける」）という一行に化けました。** これがトレーシングの核心です。
制御フローのうち「今回通った道」だけを残し、それ以外を**ガードという
脱出口**に押し込めたのです。

> [!IMPORTANT]
> トレースには**条件分岐がありません**。あるのはガードだけです。ガードは
> 「分岐」ではなく「前提のチェック」です。前提が成り立つかぎり一直線に
> 流れ、成り立たなくなった瞬間だけ外へ出る。だからトレースの本体は
> 完全な直線コードになり、[中間表現](ir.md)で見た直線向きの最適化
> （定数畳み込み、共通部分式削除）がそのまま強力に効きます。

## トレースを実行する：ガードとサイドイグジット

記録したトレースは、ループとして繰り返し実行できます。各周回でガードを
チェックし、破れたら**サイドイグジット**してインタプリタへ戻ります。
ここでは戻り先の代わりに、ただ結果を返すことで脱出を表します。

```ruby
# 一直線のトレースを、ガードが破れるまで繰り返す
def run_trace(trace, vars)
  iters = 0
  loop do
    trace.each do |insn|
      case insn[0]
      when :mov   then vars[insn[1]] = val(vars, insn[2])
      when :add   then vars[insn[1]] = val(vars, insn[2]) + val(vars, insn[3])
      when :lt    then vars[insn[1]] = val(vars, insn[2]) < val(vars, insn[3])
      when :guard then return [vars[:sum], iters] unless vars[insn[1]]  # サイドイグジット
      end
    end
    iters += 1
  end
end

sum, iters = run_trace(trace, { n: 100, sum: 0, i: 0 })
p [sum, iters]   # => [4950, 100]
```

`n = 10` のループから記録したトレースを、`n = 100` の新しい状態で回しても、
ちゃんと `4950`（0+1+…+99）が出て、100 周したところでガード `i < n` が
破れてサイドイグジットしています。**トレースはループの構造に依存せず、
記録した演算の並びと前提（ガード）だけで動く**。ここに、トレーシングが
「実行経路を単位にする」と言うことの実体があります。

> [!NOTE]
> 本物のトレーシング JIT では、`run_trace` の中身が `case` ディスパッチ
> ではなく**機械語**になります。そうなれば `[:add, :sum, :sum, :i]` は
> CPU の `add` 命令そのものになり、ループはディスパッチゼロで回ります。
> ここではしくみを見るために Ruby で「トレースのインタプリタ」を
> 書きましたが、最後の一段を[最小の JIT](hello-jit.md)の機械語生成に
> 差し替えれば、本物に近づきます。

## 分岐が分かれるとき：トレースの木

ここまでは「ループが毎回同じ道を通る」前提でした。現実には、ループの中の
分岐が**ときどき違う方向**へ行くことがあります。

```ruby
# n 個の数のうち、偶数だけ合計するループ（分岐が交互に分かれる）
# while i < n { if even?(i) { sum += i }  i += 1 }
```

最初に記録したトレースは「`even?(i)` が真だった道」かもしれません。すると
奇数の `i` のとき、`even?` のガードが破れてサイドイグジットします。毎回
偶奇が交代すると、半分の周回でトレースから落ちることになり、台無しです。

トレーシング JIT はこれを**トレースの木**（trace tree）で解決します
[](#cite:gal2006)。あるガードが**何度も破れる**（その脱出口が熱くなる）と、
「そこから先の道」を新しいトレースとして記録し、元のガードに**つなぎ込み**
ます。こうしてガードからガードへ枝分かれする木が育ち、よく通る道はすべて
機械語のトレースで覆われていきます。

```
   ループヘッダ
        │
   [本トレース：偶数だった道]
        │
   guard even?(i) ─破れがち→ [サイドトレース：奇数だった道]
        │                          │
       戻る ←──────────────────────┘
```

この**枝分かれの管理**こそ、トレーシング JIT の実装が難しいと言われる
最大の理由です。木が大きくなりすぎないか、ガードの脱出が偏っていないか、
別のトレースとどこでつなぐか —— これらの判断が処理系ごとに工夫されて
います。

## どこからトレースを始めるか

もう一つの設計判断が、**トレースの開始点**です。素直なのは「ホットな
ループのヘッダから」ですが、それだけではありません。

- **ループヘッダ起点**：いちばん基本。LuaJIT や TraceMonkey の出発点。
- **関数の入口起点**：ループでない熱い関数も対象にしたい場合。
- **サイドイグジット起点**：上で見たように、熱くなった脱出口から新しい
  トレースを伸ばす。木を育てる原動力。

トレースをどこまで伸ばすか（停止条件）も重要です。無限に伸ばすと
コンパイル単位が巨大になりすぎるので、「一定の長さを超えたら打ち切る」
「呼び出しが深くなりすぎたら止める」といった上限を設けます。再帰呼び出しの
扱いも厄介で、トレーシング JIT の論文の多くがこの停止条件に紙幅を割いて
います。

## トレース上の最適化

トレースが一直線であることは、最適化に絶大な利点をもたらします。

- **ループ不変式の巻き上げ**：ループの中で値が変わらない計算
  （`length` の取得など）を、トレースの外（ループの前）へ 1 回だけ
  追い出せます。一直線なので「本当に不変か」の判定が簡単です。
- **ガードの巻き上げ・統合**：同じガードが毎周回チェックされているなら、
  ループの外へ 1 回だけ出せます。連続する重複ガードも 1 つにまとめられます。
- **アロケーションの除去**：ループ内で作って同じループ内で捨てるだけの
  短命オブジェクトは、そもそも作らずにレジスタ上の値で済ませられます
  （[エスケープ解析](glossary.md)とスカラ置換）。

これらは [中間表現](ir.md) で見た古典的最適化と同じものですが、対象が
一直線のトレースだと**ぐっと素直に書ける**のがポイントです。分岐の合流
（φ 関数）に悩まされず、上から下へなめるだけで多くが片付きます。
トレーシングが「最適化しやすい」と言われるのはこのためです。

## トレーシングの歴史と到達点

トレーシングのルーツは、JIT ですらない動的最適化器 **Dynamo**
[](#cite:bala2000) にあります。Dynamo は機械語を実行しながら熱いトレースを
見つけて最適化し直しました。これを JIT に応用したのが、組み込み Java 向けの
**HotpathVM**[](#cite:gal2006) と、JavaScript 向けの **TraceMonkey**
[](#cite:gal2009) です。そして商用品質に磨き上げたのが
[LuaJIT](case-luajit.md) と、次章で扱う**メタトレーシング**の
[PyPy](case-python.md) です。

> [!TIP]
> トレーシングは万能ではありません。[コンパイルの単位](compilation-units.md)で
> ふれたとおり、分岐が毎回ばらつくコードではトレースの木が膨れ、かえって
> 不利になります。JavaScript の主要エンジンがトレーシングからメソッド
> JIT へ回帰したのも、ウェブのコードが「素直なループ」ばかりではない
> ことが大きな理由でした。一方、数値計算やインタプリタのように「同じ道を
> ひたすら回る」コードでは、トレーシングが圧倒的に強い。**ワークロード
> 次第**です。

## まとめ

- トレーシング JIT は、ホットなループを 1 周記録し、**条件分岐をガードに
  変えた一直線のトレース**を作って実行する。
- トレースには分岐がなく、あるのは**ガード**（前提のチェック）だけ。
  前提が破れたら**サイドイグジット**でインタプリタへ戻る。
- 分岐がばらつくと、熱い脱出口から枝を伸ばして**トレースの木**を育てる。
  この管理がトレーシング実装の最難関。
- 一直線ゆえに**ループ不変式の巻き上げ・ガード統合・アロケーション除去**が
  素直に効く。
- ルーツは Dynamo、応用が HotpathVM と TraceMonkey、到達点が LuaJIT と PyPy。

次章では、このトレーシングの威力を**インタプリタを書くだけで**手に入れる、
驚くべき技法 —— メタトレーシングへ進みます。
