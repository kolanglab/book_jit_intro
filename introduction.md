# はじめに：この本の地図

「インタプリタ言語は遅い」という言い回しを、どこかで聞いたことがあると
思います。Ruby も Python も JavaScript も、ソースコードを機械語に
変換してから動かす C や Rust にくらべて遅い、という常識です。

ところが、いまの JavaScript はその「遅いインタプリタ言語」のはずなのに、
ブラウザの上でゲームや画像処理を動かしています。Java はサーバの世界で
C++ と速度を競っています。Ruby も、バージョン 3.1 以降は YJIT という
仕組みで目に見えて速くなりました。

この「インタプリタ言語なのに速い」を支えている中心技術が、本書のテーマ
**JIT コンパイル**です。Just-In-Time、つまり「ちょうど間に合うように」
「実行のその場で」コンパイルする、という意味です。

## JIT とは何か、一言でいうと

ひとことで言えば、JIT コンパイルとは

> **プログラムを実行しながら、その一部を機械語へコンパイルし、以後は
> その機械語を直接実行する**

という仕組みです。「コンパイル」と「実行」は、ふつう別の時間に起こります。
C 言語なら、開発者が手元で `gcc` を走らせて実行ファイルを作り（コンパイル
時）、それを後でユーザが動かします（実行時）。これを**AOT コンパイル**
（Ahead-Of-Time、事前コンパイル）と呼びます。JIT はこの 2 つの時間を
重ね合わせ、**動かしながらコンパイルする**ところに本質があります。

なぜそんな回りくどいことをするのか。コンパイルが実行中に起こるなら、
コンパイラは**プログラムが実際にどう動いているか**を見ることができます。
「この変数にはいつも整数しか入っていない」「この `if` はいつも真になる」
「このメソッド呼び出しの相手はいつも同じクラスだ」—— こうした、
ソースコードを眺めるだけでは分からない事実を観測し、それを**前提として
焼き込んだ**高速なコードを生成できる。これが JIT の最大の武器です。
詳しくは[なぜ JIT コンパイルするのか](why-jit.md)で見ます。

## なぜいま JIT を学ぶのか

JIT は新しい技術ではありません。アイデアの源流は 1960 年代の LISP や
1980 年代の Smalltalk[](#cite:deutsch1984) にさかのぼれます
[](#cite:aycock2003)。それでもいま学ぶ価値があるのは、第一に、
**現代のあらゆる主要処理系が JIT を持っている**からです。JVM、V8
（Chrome / Node.js）、.NET、PyPy、LuaJIT、そして CRuby（YJIT）と
CPython 3.13。あなたが日々使っている言語の「速さ」の正体を知ることは、
性能を読み解く目を与えてくれます。

第二に、JIT は**言語処理系という分野のおもしろさが凝縮した場所**だから
です。コンパイラ理論、計算機アーキテクチャ、データ構造、オペレーティング
システム（実行可能メモリの確保）、そして「賭けに負けたら戻る」という
独特の制御。これらが一点に集まっています。

## 本書の歩き方

本書は 3 部構成です。

第 I 部「[基本のテクニック](part-basics.md)」では、JIT を構成する道具を
ひとつずつ手に入れます。理屈だけでなく、[最小の JIT](hello-jit.md) を
実際に Ruby から機械語を吐いて動かすところから始めます。

第 II 部「[最新研究](part-research.md)」では、2010 年代以降の研究を
トピックごとに 5 章で追います。論文への入口として読んでください。

第 III 部「[ケーススタディ](part-cases.md)」では、実在の処理系を題材に、
学んだ技法が現実にどう使われているかを見ます。

巻末には[用語集](glossary.md)を付けました。知らない言葉が出てきたら
そちらも参照してください。

> [!TIP]
> 章は前から順に読むのがいちばんですが、興味のあるケーススタディ
> （第 III 部）から先に眺めて、分からない用語が出てきたら第 I 部に
> 戻る、という読み方もできます。各章は相互にリンクしているので、
> 気になった概念をたどっていけます。

## 前提知識

本書は言語処理系の実装を扱う本ですが、ゼロからは始めません。
次のことは前提とします。

- プログラムを 1 つ書いたことがある（言語は問いません）。
- 「コンパイラ」と「インタプリタ」がだいたい何をするものか、
  ふんわりとでもイメージがある。
- 16 進数とバイト、メモリのアドレス、という言葉に拒否反応がない。

逆に、次のことは前提としません。本書のなかで説明します。

- バイトコードとは何か、スタックマシンとは何か。
- アセンブリや機械語の細かい知識（必要な部分だけ、その場で説明します）。
- SSA・基本ブロック・データフロー解析などのコンパイラ用語。

バイトコードインタプリタの作り方そのものをもっと丁寧に知りたい場合は、
姉妹書の[言語処理系入門](https://kolanglab.github.io/book_langimpl_intro/)
が土台になります。本書はその「先」、つまり**できあがったインタプリタを
どう速くするか**を扱う副読本です。

それでは、まず「なぜ JIT コンパイルすると速くなるのか」という、
いちばん大事な問いから始めましょう。
