# JIT とランタイムの協調：GC・セーフポイント・コードキャッシュ

[最小の JIT](hello-jit.md)で見たとおり、JIT が吐くのは「機械語のバイト列」です。
けれどそのバイト列は、CPU の上でぽつんと孤立して動くわけではありません。
同じプロセスには、**ガベージコレクタ**（GC）がいて、**別のスレッド**が
走っていて、ときには**デバッガ**がのぞき込んでいる。生成した機械語は、こうした
ランタイムの住人たちと**協調**して初めて、安全に動けるのです。

前章の[脱最適化と OSR](deopt-osr.md)で、私たちは投機の安全網 ——
賭けに負けたとき正しい実行へ戻る仕組み —— を見ました。この章で扱うのは、
それと並ぶ**もう一つの安全網**です。生成コードが「いま自分はどういう状態か」を
ランタイムに正しく申告し、GC やスレッドの都合に合わせて立ち止まったり、自らの
コードを書き換えたりする —— その作法をひとつずつ見ていきます。

> [!NOTE]
> 脱最適化が「投機が外れたときの安全網」だとすれば、この章のテーマは
> 「**ランタイムの他の住人と足並みを揃えるための安全網**」です。どちらも、
> 「生成コードは勝手気ままに動いてよいわけではない」という同じ事実の、別の顔です。
> 実際、両者は地続きで、後述するセーフポイントは**脱最適化と GC の共通の足場**に
> なっています。

## なぜ協調が要るのか

たとえば、JIT が `a + b` を「`a` も `b` も整数」と投機して、生の整数加算
一命令に落としたとします。このコードが走っている最中に、別スレッドの GC が
「いまヒープを掃除したい」と言い出したら、どうなるでしょう。

GC は、生きているオブジェクトを探すために、各スレッドのスタックとレジスタを
走査します。「このスロットはオブジェクトへのポインタだから、その先は生きている」
と判断していく。ところが**生成コードの真っ最中**、レジスタには生の整数や、
計算途中の中間値や、まだ箱に詰めていない値がばらけて乗っています。GC から見れば、
それが「ポインタ」なのか「たまたまアドレスに見える整数」なのか、区別がつきません。

ここで協調が要ります。JIT は、

- GC が走査してよい**地点を限定**し（セーフポイント）、
- その地点で「どのレジスタ／スタック位置がポインタか」を**表として残し**（スタックマップ）、
- **ムービング GC**（断片化の解消や確保の高速化のため、オブジェクトをヒープ内の
  別の場所へ移動させる方式の GC）がオブジェクトを動かしたら、コードやスタックの
  中のポインタを**書き換えさせる**。

これらをすべて、コンパイル時にあらかじめ仕込んでおく必要があります。「速い機械語を
吐く」ことと「GC と協調できる機械語を吐く」ことは、別々の仕事ではなく、**同じ
コンパイルの中で同時にやらねばならない**仕事なのです。

## セーフポイント：止まってよい地点を決める

GC も[脱最適化](deopt-osr.md)も、**任意の機械語境界では行えません**。レジスタに
計算途中の値が乗っている最中に止められても、その状態を誰も解釈できないからです。
止まってよいのは、「いまのレジスタ／スタックが、論理的にどの状態に対応するか」を
JIT が正確に言える地点 —— **セーフポイント**（safepoint）—— だけです。

### どこにセーフポイントを置くか

セーフポイントは、多ければよいというものではありません。多すぎれば情報（後述の
スタックマップ）がかさみ、少なすぎれば「GC したいのにどのスレッドもなかなか
止まらない」事態になります。定石は、**どんな実行経路をたどっても、ほどよい間隔で
必ずセーフポイントを通る**ように、次の三種類の場所へ置くことです。

- **ループのバックエッジ**：ループは何億回も回りうる。ここにセーフポイントが
  なければ、ループを抜けるまで GC が待たされる。
- **メソッド呼び出し**：呼び出しの前後は、レジスタの状態が整理されていて記述しやすい。
- **オブジェクト確保点**：確保はそもそも GC を誘発しうる地点なので、ここで止まれると都合がよい。

### ポーリング方式：旗を数命令で確かめる

「止まれ」をスレッドに伝える素直なやり方が**ポーリング**です。各セーフポイントに、
「GC を待っている旗が立っていないか」を確かめる**数命令**を埋め込みます。旗が
立っていなければ素通り、立っていれば止まる。模型で見ましょう。

```ruby
# 機械語の各命令を「実行」しながら、セーフポイントでだけ GC を許す模型。
# ポーリング方式：命令列の所々に safepoint が埋まっていて、
# そこを通るときだけ「GC してよいか」を確認する。

program = [
  [:op, "acc = acc + x*x"],
  [:safepoint, 0],            # ループ先頭のセーフポイント
  [:op, "x = arr[i]"],
  [:op, "i += 1"],
  [:safepoint, 1],            # 確保点の手前のセーフポイント
  [:op, "obj = alloc(...)"],
]

gc_requested = true   # 別スレッドが「GC したい」と旗を立てた、とする

program.each do |kind, info|
  case kind
  when :op
    # 普通の命令はそのまま進む。ここでは GC できない（レジスタの途中状態）
  when :safepoint
    # ポーリング：旗が立っていれば、ここで全スレッドが揃うのを待って GC
    if gc_requested
      puts "safepoint ##{info}: 旗が立っている → ここで停止して GC を実行"
      gc_requested = false
    else
      puts "safepoint ##{info}: 旗なし → 数命令のチェックだけで素通り"
    end
  end
end
```

実行するとこうなります。

```
safepoint #0: 旗が立っている → ここで停止して GC を実行
safepoint #1: 旗なし → 数命令のチェックだけで素通り
```

肝心なのは、**当たり前の経路では数命令の素通りで済む**ことです。GC が起きるのは
たまにだけなので、ふだんのコストはほぼゼロ。これは前章で見たガードの哲学
（「めったに起きないことのチェックは安く」）とまったく同じ発想です。

> [!TIP]
> 実装はもっと攻めます。旗を「メモリの 1 ページ」にして、ふだんは読める／GC
> 時だけ読めなくする手があります。各セーフポイントには「そのページを読むだけ」の
> 一命令を置く。ふだんは何の問題もなく読めて素通りし、GC 時にはそのページを
> 保護してアクセス不能にしておく —— するとセーフポイントの読みが**ページフォルト**を
> 起こし、シグナルハンドラがスレッドを止める。当たっているかぎり分岐も比較も増えず、
> [脱最適化](deopt-osr.md)で見た「暗黙のヌルチェック」と同じく、**ハードウェアの罠に
> 肩代わりさせる**手口です。

### 全スレッドが揃うまで待つ：stop-the-world

ムービング GC やフルな掃除では、**全スレッドがセーフポイントに来て止まる**まで
待つ必要があります。これが**stop-the-world**（STW）です。一つでもセーフポイントの
ないループに入ったスレッドがいると、他の全員がそいつを待たされる。だから

> ループのバックエッジにセーフポイントを置く

のは、性能のためというより、**STW を成立させるための必須要件**です。セーフポイントの
間隔は、そのまま「最悪、GC 開始までどれだけ待つか」になります。

> [!IMPORTANT]
> セーフポイントの密度は、**「ふだんのコスト」対「止まるまでの遅れ」**の綱引きです。
> 密に置けば素早く全員を止められますが、ポーリング命令とスタックマップがかさむ。
> 粗く置けば情報は小さく済みますが、長いループに入ったスレッドを待つあいだ、
> GC が始められない。多くの実装は「ループ・呼び出し・確保点」に限って置き、
> ホットなループのバックエッジでは前述のページ保護方式で命令を一つに抑えます。

> [!NOTE]
> この綱引きが現実の障害になった例が、HotSpot の **counted loop** です。
> 最適化コンパイラ C2 は、「int のカウンタで回数の上限が分かるループ」からは
> セーフポイントを省く最適化をしていました。ループ内のポーリングコストをゼロに
> できる、攻めた判断です。ところが回数が分かっていても、10 億回回るループは
> 長い。全スレッドが止まり終わるまでの時間（time-to-safepoint）がそのループに
> 引きずられ、GC の停止が遅れる障害が実際に起きました。対症療法として
> counted loop にもセーフポイントを入れるフラグ `-XX:+UseCountedLoopSafepoints`
> が用意され、JDK 10 の C2 はループを外側・内側の二重ループに割り直して
> **外側にだけ**セーフポイントを置く **loop strip mining** で、「ふだんのコストは
> ほぼゼロ」と「止まるまでの遅れに上限を付ける」の両立にたどり着きました。

## スタックマップ（OopMap）：どこがポインタかを記す表

セーフポイントで止まれたとして、GC は次に「生きているオブジェクト」を探さねば
なりません。そのために各セーフポイントで、**どのレジスタ／スタック位置が
オブジェクト参照（ポインタ）で、どこが生の値か**を記録した表 —— **スタックマップ**
（stack map、JVM では **OopMap**、oop = ordinary object pointer）—— を、JIT は
コンパイル時に吐いておきます。

GC はこの表を使って、止まった各スレッドのフレームを走査し、「参照」と印の付いた
スロットだけを辿ります。そして**ムービング GC**（コピー GC やコンパクション）なら、
オブジェクトを別の場所へ動かしたうえで、スタックやレジスタの中のポインタを**新しい
アドレスへ書き換えます**。これを模型で見ましょう。

```ruby
# スタックマップ（OopMap）でフレームの参照を列挙し、
# ムービング GC（コピー GC）がアドレスを付け替える模型。

# ヒープ：アドレス(整数) => オブジェクト。GC でオブジェクトを別アドレスへコピーする。
heap = { 0x1000 => "Cat", 0x1008 => "Dog", 0x1010 => 99 }  # 99 は非参照(生の整数)

# あるセーフポイントでのフレーム。r0,r1 はレジスタ、s0,s1 はスタックスロット。
frame = { r0: 0x1000, r1: 99, s0: 0x1008, s1: 12345 }

# スタックマップ：このセーフポイントで「どのスロットが参照か」を記録した表。
# true なら参照（GC が追って・付け替える）、false なら生の値（触らない）。
oopmap = { r0: true, r1: false, s0: true, s1: false }

# --- コピー GC：参照スロットの指す先を新領域へコピーし、フレームを付け替える ---
forwarded = {}      # 旧アドレス => 新アドレス（一度コピーしたら使い回す）
next_addr = 0x2000

oopmap.each do |slot, is_ref|
  next unless is_ref                 # 参照でないスロットは一切触らない
  old = frame[slot]
  unless forwarded[old]              # まだコピーしていなければコピーする
    forwarded[old] = next_addr
    heap[next_addr] = heap[old]      # オブジェクトを新領域へ
    next_addr += 8
  end
  frame[slot] = forwarded[old]       # ★ フレーム中のポインタを新アドレスへ書き換える
end

frame.each do |slot, v|
  if oopmap[slot]
    puts "#{slot}: 0x#{v.to_s(16)}  (参照 → 付け替え済み)"
  else
    puts "#{slot}: #{v}  (生の値 → 触らない)"
  end
end
```

実行結果です。

```
r0: 0x2000  (参照 → 付け替え済み)
r1: 99  (生の値 → 触らない)
s0: 0x2008  (参照 → 付け替え済み)
s1: 12345  (生の値 → 触らない)
```

`r0` と `s0` はオブジェクト参照だったので、コピー先の新アドレス（`0x2000`,
`0x2008`）へ書き換わりました。一方 `r1`（生の整数 `99`）と `s1`（生の値）は、
スタックマップが「参照ではない」と言っているので**一切触られていません**。
もしここでスタックマップを間違えて「`r1` も参照だ」と記録していたら、GC は
`99` をアドレスとみなして辿り、メモリを破壊します。**スタックマップの正しさは、
GC の正しさそのもの**なのです。

### 正確 GC と保守的 GC

このように「どこがポインタか」を JIT が正確に申告できる GC を**正確 GC**
（precise / exact GC）と呼びます。これがムービング GC の前提です。逆に、
スタックマップを持たず「アドレスっぽい値はぜんぶポインタかもしれない」と
**疑ってかかる**のが**保守的 GC**（conservative GC）です。保守的 GC は実装が
楽（JIT がスタックマップを吐かなくてよい）ですが、生の整数を誤ってポインタと
みなしてしまうと、

- そのオブジェクトを「生きている」と誤判定して回収できない（偽の生存）、
- だから**動かせない**（書き換えると壊すので、ムービングにできない）。

という制約を負います。JIT と協調してスタックマップを吐く手間をかけるのは、
**ムービング GC の自由（断片化解消・確保の高速化）を手に入れるため**です。

> [!NOTE]
> 逆向きの割り切りが活きている実例が CRuby です。CRuby の GC は、ヒープ内の
> オブジェクト同士の参照は正確にたどる一方、**C のスタック（機械スタック）と
> レジスタは保守的に走査**します。C 拡張がオブジェクトへのポインタを生のまま
> C のローカル変数に置けるようにするための、古くからの設計です。この性質は
> JIT にとって思わぬ追い風になりました。[YJIT](case-ruby.md) や ZJIT の生成
> コードがレジスタやスタックに参照を置いたままでも、保守的走査が拾ってくれる
> ので、**JIT がセーフポイントごとのスタックマップを吐く必要がない**のです。
> 本章で見てきた GC 協調の難所を一つ丸ごと省けたことが、YJIT/ZJIT の実装を
> 大きく単純にしています（その代償として、保守的に拾われたオブジェクトは
> 動かせない、という前述の制約は引き受けたままです）。

### derived pointer：内部を指すポインタの難しさ

スタックマップには、もう一つやっかいな相手がいます。**derived pointer**
（派生ポインタ）です。最適化器は、配列 `arr` の先頭ではなく**途中の要素 `&arr[i]`**
を指すポインタをレジスタに作ることがあります（ループの強度低減で、添字計算を
ポインタの足し算に変えた結果など）。これはオブジェクトの**内部**を指していて、
オブジェクトの先頭アドレス（base pointer）ではありません。

ムービング GC が `arr` を動かしたとき、この内部ポインタも一緒にずらさねば
なりませんが、そのためには「これは `arr` の先頭から何バイト目を指しているか」を
知る必要があります。だからスタックマップは、derived pointer について
**「どの base pointer から派生したか」**もあわせて記録します。GC は base を
動かした差分（new_base − old_base）を derived にも足し込むのです。

> [!WARNING]
> derived pointer は、JIT と GC の協調が破綻しやすい急所です。base を保持せずに
> derived だけをレジスタに残すと、オブジェクトが動いたとき正しくずらせません。
> そのため最適化器には「derived を生かすあいだは、対応する base も生かしておけ
> （keepalive）」という制約が課されます。攻めの最適化（[中間表現](ir.md)の
> アドレス計算の巻き上げ）と、GC の協調が真正面からぶつかる場所です。

## ライトバリア：参照書き込みの後ろに数命令を足す

**世代別 GC**（generational GC）は、「オブジェクトの多くは生まれてすぐ死ぬ」
という経験則に基づく方式です。ヒープを、生まれたてのオブジェクトが入る
**若い世代**と、何度かの回収を生き延びたオブジェクトが移る**老世代**に分け、
ふだんは若い世代だけを対象にした小さな回収（**minor GC**）を高頻度に回します。
こうした世代別 GC や並行 GC は、「ヒープ全体を毎回走査する」のを避けるために、JIT に
**ある協力**を求めます。**参照を書き込んだら、その事実を記録せよ** —— これが
**ライトバリア**（write barrier）です。JIT は、`obj.field = value` のような
参照書き込みの**後ろに、バリアコードを自動で挿入**します。

### カードマーキングの数命令

代表的なバリアが**カードマーキング**です。ヒープを「カード」（たとえば 512
バイト）に区切り、参照を書き込んだらそのカードを **dirty** に印す。GC（特に若い
世代の minor GC）は、ヒープ全体ではなく **dirty なカードだけ**を走査すればよく
なります。「老世代のどこかが若い世代を指しているかも」を、カード単位で粗く覚えて
おくわけです。模型で見ます。

```ruby
# 世代別 GC のためのライトバリア（カードマーキング）の模型。
# 老世代のフィールドに若い世代への参照を書いたら、その場所のカードを dirty にする。

CARD_BITS = 9                       # 1 カード = 512 バイト（2^9）
$card_table = Hash.new(false)       # カード番号 => dirty か
def card_of(addr) = addr >> CARD_BITS

# obj.field = value に相当。第4引数は JIT が静的に判断する「バリアが要るか」。
def store_ref(obj_addr, field_offset, value_addr, need_barrier:)
  # ... ここで実際の store が起きる ...
  if need_barrier
    $card_table[card_of(obj_addr + field_offset)] = true   # カードマーキング（数命令）
    puts "  store @0x#{(obj_addr+field_offset).to_s(16)}: バリア実行 → カード##{card_of(obj_addr+field_offset)} を dirty"
  else
    puts "  store @0x#{(obj_addr+field_offset).to_s(16)}: バリア省略（JIT が不要と判断）"
  end
end

# (1) 老世代(0x4000)に若い世代(0x9000)を書く → バリアが要る
store_ref(0x4000, 16, 0x9000, need_barrier: true)
# (2) たった今 alloc した新規オブジェクト(0xA000)のフィールド初期化 →
#     まだ誰からも指されておらず若い世代同士 → JIT はバリアを消せる
store_ref(0xA000, 0, 0x9000, need_barrier: false)

puts "dirty なカード: #{$card_table.select { |_, d| d }.keys.map { |c| "##{c}" }.join(", ")}"
```

実行結果です。

```
  store @0x4010: バリア実行 → カード#32 を dirty
  store @0xa000: バリア省略（JIT が不要と判断）
dirty なカード: #32
```

カードマーキングの本体は、機械語では「アドレスを右シフトしてカード表のバイトに
`1` を書く」**数命令**にすぎません。実際のカード表は 1 バイト／カードのバイト配列で、
`card_table[addr >> 9] = 1` がほぼそのまま機械語になります。短いとはいえ、
**参照を書くたび**に走るので、ホットなコードでは塵も積もれば山になります。

### 不要なバリアを最適化で消す

そこで JIT は、「**このバリアは要らない**」と静的に分かる場合に、バリアコードを
**消します**。上の模型の `need_barrier: false` がそれです。代表例は、

- **たった今確保した新規オブジェクト**のフィールド初期化。まだ誰からも指されて
  いない若い世代のオブジェクトに若い世代を入れるなら、世代をまたぐ参照は生まれない。
- **同一世代への書き込み**と静的に分かる場合（書き込み先も値も若い世代だと型・
  確保情報から言える、など）。
- **同じ場所への連続書き込み**。同一カードへ続けて書くなら、印すのは一度でよい
  （二度目以降のマーキングは冗長）。
- **ループ不変なバリア**を、[中間表現](ir.md)の最適化でループ外へ巻き上げる。

これは前章のガード削減（重複除去・巻き上げ）とまったく同じ構図です。GC との
協調コードもまた、**最適化で削る対象**なのです。

> [!NOTE]
> 並行 GC（実行と並行してヒープをたどる GC）では、バリアの役割が変わります。
> 並行 GC の多くは、オブジェクトを「まだ見ていない（白）」「見ている途中（灰）」
> 「見終わった（黒）」の三色に塗り分けながらヒープをたどります。GC がたどって
> いる最中にプログラムが参照を書き換えると、GC が「もう見た（黒）」オブジェクトの
> 先を見落とすことがある。これを防ぐため、並行 GC は読み書きのバリアで
> 「黒が白を直接指さない」といった**色の不変条件**を保ちます。バリアの形は GC のアルゴリズムごとに違い、
> **JIT はその GC が要求するバリアを、知ったうえで挿入する**必要があります。
> 「JIT は GC を知らなくてよい」とはいかないのです。

## コードキャッシュ：コードのための、もう一つの GC

ここまではデータ（ヒープ上のオブジェクト）の GC の話でした。実は JIT には、
**コードのための置き場の管理**という、よく似た問題がもう一つあります。生成した
機械語を置く領域 —— **コードキャッシュ**（code cache）—— もまた有限で、長く
動かせば埋まり、断片化し、古い版の掃除が要ります。

この話は[階層的 JIT](tiered.md#コードキャッシュの管理：置き場は有限)で
退去つきキャッシュの模型として一度扱いました。ここでは、それが**「コードの GC」**
だという視点から、もう一歩踏み込みます。データの GC とコードキャッシュ管理は、
驚くほど似た問題を、別の対象について解いています。

| | データの GC | コードキャッシュ管理 |
|---|---|---|
| 対象 | ヒープ上のオブジェクト | 生成された機械語 |
| 生存判定 | 参照から到達可能か | まだ呼ばれうるか・最新版か |
| 回収 | 不要オブジェクトを解放 | 古い版・冷えた版を退去 |
| 断片化対策 | コンパクション | コードのコンパクション／領域分割 |
| 「動かす」難しさ | スタック中のポインタ書き換え | コードを指す**戻り番地**の書き換え |

最後の行が、コードならではの難所です。コードキャッシュをコンパクションして
コードを移動すると、**いままさにそのコードを実行中**のスレッドがいるかもしれない。
そのスレッドのスタックには、移動したコードへの**戻り番地**が積まれています。
データの GC がスタック中のデータポインタを書き換えたのと同じように、コードの
コンパクションでは**スタック中の戻り番地を書き換え**ねばなりません。ここでも、
セーフポイントとスタックマップ（こんどは「どのスロットが戻り番地か」）が要るのです。
もっとも、これはあまりに危険で複雑なため、実装では**コードは動かさない**選択が
主流です。固定サイズの区画に切って再利用し、断片化はコードの退去と区画管理で
抑え込む —— 「動かす自由」をあきらめて単純さを取る、データの GC とは逆向きの
割り切りがふつうです。

> [!WARNING]
> コードキャッシュの枯渇は、長時間動くサーバで実際に起きる障害です。
> JVM では code cache が満杯に近づくと、まず古い・冷えたコードの退去
> （既定で有効な `UseCodeCacheFlushing`）で空きを作ろうとし、それでも
> 足りなければ新規コンパイルが止まり、その分インタプリタで動く割合が
> 増えて性能がじわじわ落ちます。「最近 JIT が遅くなった」の原因が、メモリでも
> CPU でもなく**コード置き場の枯渇**だった、というのはありがちな落とし穴です。
> 退去戦略は「どの版を生かすか」という[脱最適化](deopt-osr.md)・降格の判断と
> 密接に絡みます。
>
> なお[HotSpot](case-jvm.md)系の処理系では、コードキャッシュを「非メソッド領域
> （ランタイムのスタブ）」「プロファイル付きコード（C1）」「最適化済みコード（C2）」
> のように**種別ごとに分ける**設計（セグメント化コードキャッシュ）が採られます。寿命や
> 退去のしやすさが違うものを混ぜないことで、断片化と相互の圧迫を抑える狙いです。

## コードのパッチと自己書き換え

JIT のいちばん危なっかしい、しかし強力な手口が、**実行中に自分の機械語を書き換える**
ことです。前章の[脱最適化](deopt-osr.md)で見た「ガードの無効化」や[階層的 JIT](tiered.md)の
「できあがった上位版へのこっそり差し替え」は、どれも実体としては**コードのパッチ**
です。代表的な用途を挙げます。

- **インラインキャッシュの差し替え**：呼び出し地点に「最後に見た型と、その型用の
  飛び先」を埋め込んでおき、型が変わったら埋め込みを書き換える。
- **呼び出し先の張り替え**：まだコンパイルされていない関数の呼び出しを、いったん
  「コンパイルしてから呼ぶ」スタブに向けておき、コンパイルが済んだら直接呼びに張り替える。
- **ガードの無効化**：大域的な前提（「このメソッドは上書きされていない」など）が
  崩れたとき、依存コードの先頭に「脱最適化へ飛べ」を上書きして無効化する。

### 模型で見る：インラインキャッシュの差し替え

まずインラインキャッシュの差し替えを、模型で見ましょう。

```ruby
# インラインキャッシュの差し替えを「実行中のコードを書き換える」模型で見る。
# 呼び出し地点には、最後に見た型と、その型用の飛び先が埋め込まれている（単態 IC）。

CallSite = Struct.new(:cached_klass, :target) do
  # cached_klass と target を「アトミックに一組」差し替えるイメージ。
  def patch!(klass, target)
    self.cached_klass = klass
    self.target = target
  end
end

# 受け取った型に対して IC で宛先を解決する。外れたら patch する。
def dispatch(site, klass, method_table)
  if site.cached_klass == klass
    "ICヒット → #{site.target} を直接呼ぶ（ガードだけ、表引きなし）"
  else
    resolved = method_table.fetch(klass)
    site.patch!(klass, resolved)          # ★ 実行中にコードを書き換える
    "ICミス → 表引きで #{resolved} を解決し、呼び出し地点を書き換えた"
  end
end

methods = { "Cat" => "Cat#speak", "Dog" => "Dog#speak" }
site = CallSite.new(nil, nil)

puts dispatch(site, "Cat", methods)   # 初回：ミス → patch
puts dispatch(site, "Cat", methods)   # 2回目：ヒット
puts dispatch(site, "Dog", methods)   # 型が変わった：ミス → 再 patch
```

実行結果です。

```
ICミス → 表引きで Cat#speak を解決し、呼び出し地点を書き換えた
ICヒット → Cat#speak を直接呼ぶ（ガードだけ、表引きなし）
ICミス → 表引きで Dog#speak を解決し、呼び出し地点を書き換えた
```

初回は表引きで解決し、呼び出し地点を「次は `Cat` なら直接 `Cat#speak`」と
**書き換え**ます。二回目以降、`Cat` が来るかぎりは表引きなしの一直線。型が
`Dog` に変わったら、また書き換える。インラインキャッシュ自体は[なぜ JIT か](why-jit.md)で
扱いましたが、その**書き換えが実行中の機械語に対して起きる**点が、ここでの主題です。

### 他スレッドが実行中のコードを安全に書き換える

模型では Ruby のオブジェクトを書き換えただけですが、本物の難所はここからです。
**書き換える当のコードを、別スレッドがちょうど実行しているかもしれない**。
ある CPU が `mov`/`jmp` を読み込んでいる最中に、別の CPU がそのバイト列を
書き換えたら、命令の途中で中身がすり替わり、何が実行されるか分からなくなります。

これを安全にやるための具体的な作法があります。

- **アトミックな 1 ワード書き換え**：書き換え対象を、**1 回のアトミックな書き込みで
  完結する大きさ・境界**に収める。安全な典型は、命令そのものではなく**データ**を
  差し替えることです。飛び先を「このアドレスセルに入っている番地へ飛べ」という
  **間接ジャンプ**にしておき、セル（8 バイト）を 8 バイト境界に揃えて置けば、
  飛び先の差し替えは 64 ビットの store 一発で済み、途中状態が他スレッドから
  観測されません。一方、**命令バイト列そのもの**の並行書き換えに対する ISA の
  保証は弱く、たとえば AArch64 が実行と並行した書き換えを許すのは
  `B`/`BL`/`BRK`/`NOP` などごく少数の命令の間に限られます。複数命令にまたがる
  書き換えが要るときは、まず先頭に「自分自身へのジャンプ（無限ループ）」や
  `trap` 命令を置いてスレッドをそこで足止めし、残りを書き換え終えてから、
  最後に先頭の足止めを外す —— というように**順序**で安全を作ります。
- **命令キャッシュの同期**：CPU の**データキャッシュ**に書いただけでは、**命令
  キャッシュ**（i-cache）には古いバイト列が残っていることがあります。書き換え後は、
  各 CPU に「命令キャッシュを捨てて読み直せ」と指示するキャッシュ同期が要ります
  （アーキテクチャによっては明示の命令や、全スレッドをセーフポイントで一度
  止める操作で代用）。これを怠ると、書き換えたはずなのに古いコードが走り続けます。
- **セーフポイントでまとめてやる**：いちばん確実なのは、**全スレッドをセーフ
  ポイントで止めてから書き換える**こと。ただし「止まっていれば自由に書き換え
  られる」わけではありません。止まったスレッドは、まさに**そのコードの内部**
  （ループ内のポーリング地点など）で止まっているかもしれず、再開したとき踏む
  命令を書き換えてしまえば壊れます。だから実際の手順は二本立てです。コードの
  **入口**に「ここへはもう入るな」というパッチを打って**新規の流入だけを止め**、
  すでにコードの中にいる**活性フレーム**は[脱最適化](deopt-osr.md)で安全に
  降ろす（HotSpot がコンパイル済みコードを無効化するときの
  `make_not_entrant`（「入れなくする」の意）が、まさにこの方式です）。コストは
  かかりますが、ガードの一括無効化のような「広範囲をまとめて」無効化する操作には
  これが向きます。

ここで[最小の JIT](hello-jit.md)で見た **W^X**（Write XOR eXecute、書き込みと
実行を同時に許さない）が効いてきます。コードページは普段 RX（読み・実行のみ）で
保護されているので、パッチするには一瞬 RW に切り替え、書いてから RX に戻す、という
二段構えが要ります。「実行中のコードを書き換える」のは、W^X の壁をくぐる操作でも
あるのです。

> [!IMPORTANT]
> 自己書き換えは、JIT の表現力の源であると同時に、**もっとも壊れやすい部分**です。
> 「1 ワードのアトミック書き換え」と「セーフポイントでの一括書き換え」を、書き換える
> 範囲の大きさと頻度で使い分ける —— これがコードパッチ設計の勘どころです。小さく
> 頻繁（IC の差し替え）はアトミック書き換え、大きく稀（前提崩壊の一括無効化）は
> セーフポイントでまとめて、と覚えておくとよいでしょう。

## コードレイアウトと CPU：速いコードはキャッシュに優しい

最後に、生成コードを**メモリ上のどこに置くか**という、地味ですが効くテーマです。
JIT は機械語を一から並べられる立場なので、CPU が気持ちよく走れるレイアウトを
**自分で設計できます**。

### ホット/コールド分割

前章で、ガードの飛び先（脱最適化スタブ）を高速経路から外して**コールドスタブ**へ
追い出す話をしました。これを一般化したのが**ホット/コールド分割**です。

- **ホット**：よく通る主経路。一直線に、密に並べる。
- **コールド**：めったに通らない経路 —— [脱最適化](deopt-osr.md)スタブ、例外
  処理、エラーパス、稀な分岐 —— を、関数の**末尾のほうへまとめて**追い出す。

なぜこれが効くのかを、命令キャッシュの「行」を数える模型で見ましょう。

```ruby
# ホット/コールド分割の効果を、命令キャッシュの「行」を数える模型で見る。
# 関数のコードを命令の配列とし、よく通る経路(hot)と稀な経路(cold)を含む。
# キャッシュ行は連続した数命令ぶん。hot だけが密に並べば、触る行が減る。

CACHE_LINE = 4   # 1 キャッシュ行 = 4 命令ぶん、とする

# 各命令: [:hot/:cold, ラベル]
def lines_touched(code)
  # 実行で実際に触れた命令の入る「行番号」の集合
  touched = code.each_index.select { |i| code[i][0] == :hot }
                .map { |i| i / CACHE_LINE }.uniq
  touched.size
end

# (A) 混在レイアウト：hot と cold が入り混じっている
mixed = []
8.times { mixed << [:hot, "h"] << [:cold, "deopt_stub"] }

# (B) 分割レイアウト：hot を先頭に固め、cold を末尾へ追い出す
split = Array.new(8) { [:hot, "h"] } + Array.new(8) { [:cold, "deopt_stub"] }

puts "混在レイアウト: hot が #{lines_touched(mixed)} 本のキャッシュ行に散らばる"
puts "分割レイアウト: hot が #{lines_touched(split)} 本のキャッシュ行に収まる"
```

実行結果です。

```
混在レイアウト: hot が 4 本のキャッシュ行に散らばる
分割レイアウト: hot が 2 本のキャッシュ行に収まる
```

同じ 8 個のホット命令でも、コールド命令と混ざっていると 4 本のキャッシュ行に
散らばり、分割して固めれば 2 本に収まりました。**触るキャッシュ行が半分**に
なるわけです。実機では、これが命令キャッシュのヒット率と **iTLB**（命令側の
アドレス変換キャッシュ）のヒット率に直結します。

### 分岐予測・iTLB・近接配置

レイアウトが効く先は、命令キャッシュだけではありません。

- **分岐予測**：稀な分岐（コールドへ飛ぶ枝）を「通常は飛ばない」向きに置けば、
  CPU の分岐予測が当たりやすい。脱最適化スタブを末尾に置き、`jz deopt_stub` を
  「めったに成立しない前方分岐」にするのは、予測器に「ここは飛ばない」と教える
  意味もあります。
- **iTLB**：ホットなコードを少数のページに密集させると、命令側のページ変換
  （iTLB）のミスが減る。コードがあちこちのページに散ると、変換だけで足を引っ張ります。
- **近接配置**：**よく一緒に実行されるコードを近くに置く**。ホットな関数 A が
  ホットな関数 B をいつも呼ぶなら、A と B を隣り合わせに置けば、A を実行中に
  B のコードも同じキャッシュ行・近いページに乗ってきます。[プロファイリング](profiling.md)で
  集めた「どの呼び出しが熱いか」を、コードの**配置**にまで使うわけです。

> [!TIP]
> 「速いコード」は、たいてい「キャッシュに優しいコード」でもあります。理由は単純で、
> 現代の CPU では**命令を CPU まで運んでくるコスト**（キャッシュ・TLB・分岐予測）が、
> 命令そのものの実行コストに匹敵するか、しばしば上回るからです。だからホットな
> 主経路を小さく密に保ち、稀な経路を遠くへ追い出すレイアウトは、命令数を一つも
> 減らしていなくても速くなる。JIT が「機械語をどこに置くか」まで設計できるのは、
> **静的コンパイラにはない、実行時プロファイルに基づくレイアウトの自由**です。

> [!NOTE]
> こうしたレイアウトやコード生成の判断は、古典的なコンパイラの知見[](#cite:aho2006)を
> 土台にしつつ、JIT では**実行時に観測したプロファイル**で味付けされます。
> 静的コンパイラが「だいたいこうだろう」と推測する分岐確率を、JIT は[プロファイリング](profiling.md)で
> 実測してから配置できる —— ここでも「動かしてから決める」という JIT の強みが効きます。

## デバッガとプロファイラ：生成コードの名前を教える

章の冒頭で、ランタイムの住人として「デバッガ」も挙げました。最後に、この
住人との協調を見ておきます。問題はこうです。`perf` や GDB のようなツールは、
実行中のアドレスを関数名に直すとき、実行ファイルのシンボル表を引きます。
ところが JIT が吐いたコードは**どの実行ファイルにも属していない**ので、
ツールから見ると「正体不明のアドレスで時間を食っている何か」にしかなりません。
プロファイルの一番熱い行が `[unknown]` では、せっかくの観測が台無しです。

そこで、JIT の側から「このアドレス範囲は、この名前のコードだ」と教える
約束事がいくつかあります。

- **perf のシンボルマップ**：`/tmp/perf-<pid>.map` というテキストファイルに
  「開始アドレス・サイズ・名前」を 1 行ずつ書き出しておくと、Linux の `perf` が
  表示時にそれを引いてくれる。生成のたびに 1 行追記するだけの、いちばん手軽な方式。
- **jitdump**：perf のより本格的な形式。コード本体や行番号情報まで記録でき、
  `perf inject --jit` で後処理すると、生成コードの中身まで命令単位で観察できる。
- **GDB JIT interface**：GDB が定めた登録口（`__jit_debug_register_code` 関数と
  `__jit_debug_descriptor` 大域変数）へ生成コードのシンボル情報を実行時に登録
  すると、GDB が JIT コードのフレームも普通の関数のように表示できる。

多くの処理系がこれらに対応しています（Ruby の `--yjit-perf`、V8 の
`--perf-basic-prof` など）。速くするための協調ではありませんが、JIT の挙動を
**観測できるようにする**協調であり、実務で性能問題を追うときは、しばしば
最初に必要になる足場です。

## まとめ

- 生成した機械語は孤立して動くのではなく、**GC・スレッド・デバッガといった
  ランタイムの住人と協調**して初めて安全に動く。これは[脱最適化](deopt-osr.md)に
  続く、もう一つの安全網。
- GC も脱最適化も、止まってよい**セーフポイント**でだけ起こせる。ループ・
  呼び出し・確保点に置き、**ポーリング**（旗を数命令で確かめる／ページ保護で
  一命令に）で実装する。全スレッドが揃うまで待つのが **stop-the-world**。
- 各セーフポイントで「どのスロットが参照か」を記す表が**スタックマップ（OopMap）**。
  GC は生きたフレームを走査し、ムービング GC ならスタック・コード中のポインタを
  **書き換える**。**正確 GC** の前提であり、その正しさは GC の正しさそのもの。
  **derived pointer** は base とともに記録して扱う。
- 世代別／並行 GC のため、JIT は参照書き込みの後ろに**ライトバリア**（カード
  マーキングの数命令）を挿入し、**不要なものは最適化で消す**（新規オブジェクト・
  同一世代・巻き上げ）。
- 生成コードの置き場 **コードキャッシュ**は有限。退去・断片化・古い版の破棄を
  管理する、いわば**「コードの GC」**。枯渇は実在の障害。コードを動かすには
  **戻り番地の書き換え**が要る。
- **コードのパッチ（自己書き換え）**は IC の差し替え・呼び出し先の張り替え・
  ガードの無効化を実行中に行う。他スレッドが実行中でも安全にやるため、
  **アトミックな 1 ワード書き換え**（データセルの差し替え）・**i-cache 同期**・
  **セーフポイントでの一括無効化**（入口パッチ＋活性フレームの脱最適化）を
  使い分ける。[W^X](hello-jit.md) の壁をくぐる操作でもある。
- **ホット/コールド分割**と近接配置で、命令キャッシュ・分岐予測・iTLB に効く
  レイアウトを作る。**速いコードはキャッシュに優しいコード**であり、JIT は
  実行時プロファイルでそれを設計できる。
- 生成コードは perf のシンボルマップ（`/tmp/perf-<pid>.map`）・jitdump・
  GDB JIT interface で、**デバッガやプロファイラに居場所と名前を教える**。

ここまで、生成コードがランタイムの住人たちと足並みを揃える作法を見てきました。
けれど協調すべき相手は、いつも善意の住人とはかぎりません。同じプロセスに**悪意を
持って入り込む相手**もいる。実行中にメモリへ機械語を書き込み、それを走らせる
JIT は、その性質ゆえに格好の攻撃対象でもあります。「協調」の最後に、もっとも
協調の難しい相手 —— **攻撃者との協調** —— を扱いましょう。次章は
[JIT とセキュリティ](security.md)です。
