# 最小の JIT を作る

前章で「コンパイルとは命令列を別の実行形式へ翻訳すること」と言いました。
この章では、その「別の実行形式」を本物の x86-64 機械語にします。
Ruby のプログラムから機械語のバイト列を作り、それをメモリに置いて、
CPU に直接実行させる。これが JIT の物理的な正体です。

理屈は短いので、さっさと動かしましょう。題材は Linux の x86-64 環境です
（Windows や ARM の Mac では細部が変わりますが、考え方は同じです）。

## 機械語をメモリに置いて実行する、とはどういうことか

ふだん私たちが書くデータ（配列や文字列）は、メモリの「読み書きできるが
実行はできない」領域に置かれます。一方、プログラムの命令は「読めて実行
できるが書き換えられない」領域に置かれます。この区別を**ページ保護**と
呼び、OS が管理しています。

JIT がやることは単純です。

1. 「実行できる」メモリ領域を OS にもらう。
2. そこに機械語のバイト列を書き込む。
3. その先頭アドレスを関数ポインタとみなして呼ぶ。

Linux でこの「実行できるメモリ」をもらうシステムコールが `mmap` です。
Ruby からは標準ライブラリ `fiddle` を使って、C の関数（`mmap` や、
作った機械語そのもの）を直接呼べます。

> [!NOTE]
> `fiddle` は Ruby から C の関数や生のメモリを触るための標準ライブラリ
> です。ここでは「C の世界への裏口」くらいの理解で十分です。本来は
> 危険な操作なので、ふつうのプログラムでは使いません。今回は JIT の
> 仕組みを覗くための特別な道具立てです。なお `fiddle` は Ruby 3.5 以降
> bundled gem になったため、素の `ruby` から `require 'fiddle'` するだけ
> なら追加の手当ては不要ですが、Bundler 配下で動かす場合は Gemfile に
> `gem "fiddle"` の 1 行が必要です。

## 実行可能メモリを確保する土台

まず、機械語のバイト列を受け取って実行可能メモリに置き、その先頭
アドレスを返す関数 `jit` を用意します。

### 読み書き実行すべて可で確保する手抜き版

```ruby
require 'fiddle'

# mmap に渡すフラグ（Linux x86-64）
PROT_READ   = 0x1
PROT_WRITE  = 0x2
PROT_EXEC   = 0x4         # この領域の命令を実行してよい
MAP_PRIVATE = 0x2
MAP_ANON    = 0x20        # ファイルでなく、ただのメモリが欲しい

libc = Fiddle.dlopen(nil)                # 自プロセスに見えている libc
Mmap = Fiddle::Function.new(libc['mmap'],
  [Fiddle::TYPE_VOIDP, Fiddle::TYPE_SIZE_T, Fiddle::TYPE_INT,
   Fiddle::TYPE_INT, Fiddle::TYPE_INT, Fiddle::TYPE_LONG],
  Fiddle::TYPE_VOIDP)

# 機械語のバイト配列を実行可能メモリへ置き、先頭アドレスを返す
def jit(bytes)
  size = 4096                            # 1 ページ
  mem = Mmap.call(nil, size,
                  PROT_READ | PROT_WRITE | PROT_EXEC,
                  MAP_PRIVATE | MAP_ANON, -1, 0)
  code = bytes.pack('C*')               # 整数配列 → バイト列
  Fiddle::Pointer.new(mem.to_i)[0, code.bytesize] = code  # 書き込む
  mem
end
```

`PROT_READ | PROT_WRITE | PROT_EXEC` で「読み書き実行すべて可」の領域を
もらい、そこへバイト列を書き込んでいます。これで土台は完成です。なお、
ここでは簡略化のため `mmap` の失敗チェックも、使い終えた領域の `munmap` も
省いています（確保した領域を貯めて管理するコードキャッシュの運用は、
後述の「[ここから先で足りないもの](hello-jit.md#ここから先で足りないもの)」で触れます）。

> [!WARNING]
> 「書き込めて実行もできる」メモリは、セキュリティ上避けられます
> （攻撃者が悪意あるコードを書き込んで実行する隙になるため）。多くの
> OS や処理系は **W^X**（Write XOR eXecute、書き込みと実行を同時に
> 許さない）を守り、書き込み後に実行権限へ切り替える二段構えにします。
> 本物の JIT もそうしています。ここでは学習のため簡略化しています。
> 環境によっては `PROT_EXEC` 付き `mmap` が拒否され、後述の実行で
> 落ちることがあります。

### 書いてから実行権限へ落とす W^X の二段構え

上の `jit` は手抜きで、最初から `PROT_EXEC` を付けてしまいました。本物の
JIT に近づけるなら、こうします。

1. まず **RW（読み書きのみ、実行不可）** でメモリをもらう。
2. そこへ機械語を書き込む。
3. 書き終わったら `mprotect` で **RX（読み実行のみ、書き込み不可）** へ
   落とす。

こうすれば「書き込み可能なあいだは実行できず、実行可能になったら書き込め
ない」が常に成り立ち、W^X を満たします。`mprotect` も `fiddle` で呼べます。

```ruby
PROT_NONE = 0x0

Mprotect = Fiddle::Function.new(libc['mprotect'],
  [Fiddle::TYPE_VOIDP, Fiddle::TYPE_SIZE_T, Fiddle::TYPE_INT],
  Fiddle::TYPE_INT)

# W^X 版：RW で確保 → 書き込み → RX へ落とす
def jit(bytes)
  size = 4096
  mem  = Mmap.call(nil, size,
                   PROT_READ | PROT_WRITE,            # ← EXEC を付けない
                   MAP_PRIVATE | MAP_ANON, -1, 0)
  code = bytes.pack('C*')
  Fiddle::Pointer.new(mem.to_i)[0, code.bytesize] = code
  Mprotect.call(mem, size, PROT_READ | PROT_EXEC)     # ← ここで実行可へ
  mem
end
```

`mprotect` は引数で渡したアドレスからの長さ `size` ぶんの保護属性を
切り替えます。`mmap` がページ境界（ここでは 4096 バイト境界）を返すので、
そのまま 1 ページぶんを切り替えられます。境界に揃っていないアドレスを
渡すと `mprotect` は `EINVAL` で失敗するので注意してください。

> [!NOTE]
> 現代の JIT がこの二段構えを徹底するのには、もう 1 つ現実的な理由が
> あります。Apple Silicon（ARM の Mac）や iOS では W^X が**強制**で、
> 同じページを RW と RX で同時に持てません。生成のたびに `mprotect`
> 相当（`pthread_jit_write_protect_np`）で切り替えるのが必須です。
> 1 ページごとに `mprotect` を呼ぶとシステムコールのコスト（数百
> ナノ秒〜マイクロ秒のオーダー）が無視できないので、本物の JIT は
> コードを大きなアリーナにまとめて書き、切り替えをまとめて 1 回
> 行う、といった工夫をします。「正しさ（W^X）」と「速さ（システム
> コール回数）」のトレードオフです。

## 42 を返す最初の機械語

いちばん簡単な機械語から。引数を取らず、整数 42 を返すだけの関数です。
x86-64 では、整数の戻り値はレジスタ `eax` に入れる約束（呼び出し規約）に
なっています。だから「`eax` に 42 を入れて、戻る」だけです。

```
B8 2A 00 00 00    mov eax, 42      ; eax ← 42（42 は 16 進で 0x2A）
C3                ret              ; 呼び出し元へ戻る
```

左の 16 進が、その命令を表す機械語のバイトです。`mov eax, 即値` は
`B8` のあとに 4 バイトの即値（リトルエンディアン）が続く、と決まって
います。これをそのまま `jit` に渡します。

```ruby
mem = jit([0xB8, 0x2A, 0x00, 0x00, 0x00, 0xC3])

# このアドレスを「引数なし・int を返す関数」とみなして呼ぶ
f = Fiddle::Function.new(mem, [], Fiddle::TYPE_INT)
p f.call   # => 42
```

`42` と表示されれば成功です。いま起きたことを噛みしめてください。
Ruby のプログラムが、**自分で組み立てた機械語を、実行中に CPU へ食わせて
答えを得た**のです。これが JIT のいちばん底の層です。あとはこの上に
「どんな機械語を、どうやって賢く組み立てるか」を積んでいくだけです。

## 誰がどこに値を置くかの約束、呼び出し規約

機械語を生成する側は、呼ぶ側（ここでは `fiddle`、ひいては C の世界）との
約束ごとを守らねばなりません。Linux x86-64 の標準である
**System V AMD64 ABI** の、最低限おさえるべき点はこれだけです。

- **整数・ポインタ引数**は、左から順に `rdi, rsi, rdx, rcx, r8, r9` の
  6 本に入って渡ってくる（7 個目からはスタック経由）。32 ビット整数なら
  それぞれの下位 32 ビット `edi, esi, ...` を見ればよい。
- **戻り値**は `rax`（32 ビットなら `eax`）に入れて返す。
- **呼ばれた側が壊してよい（caller-saved）**レジスタ：`rax, rcx, rdx,
  rsi, rdi, r8〜r11`。自由に使い捨てできる。
- **呼ばれた側が保存すべき（callee-saved）**レジスタ：`rbx, rbp,
  r12, r13, r14, r15`。使うなら退避し、戻る前に復元する。
- `call` の直後（関数の入り口）では、スタックポインタ `rsp` が
  16 バイト境界 − 8 に揃っている（`call` が戻りアドレス 8 バイトを
  積んだ直後だから）。逆にいえば、自分が `call` を出すときは、**`call` を
  出す直前に `rsp` を 16 バイト境界へ揃える**必要がある（そうすれば相手の
  入り口は規約どおり「16 バイト境界 − 8」になる）。

なお、同じ x86-64 でも呼び出し規約は OS ごとに違います。Windows x64 ABI では
整数引数は `rcx, rdx, r8, r9` の 4 本で、呼び出し側がスタックに 32 バイトの
shadow space を確保する義務もあります。本書の例は System V（Linux/macOS）を
前提にするので、移植するときは対象プラットフォームの ABI に読み替えてください。

前節の「42 を返す」も、この後つくる「足し算」も、`eax`/`edi`/`esi` という
caller-saved レジスタだけを使い、スタックにも触れないので、プロローグ・
エピローグ（入り口と出口の定型処理）は不要です。けれど、callee-saved レジスタを
使ったり、他の関数を `call` したりするなら、次の定型が要ります。

```
55          push rbp        ; 呼び出し元の rbp を退避
48 89 E5    mov  rbp, rsp   ; 自分のフレームの基準点を作る
...                         ; （本体。ここで rbx 等を使うなら別途 push/pop）
C9          leave           ; mov rsp, rbp; pop rbp と同じ（後始末）
C3          ret             ; 戻りアドレスへ戻る
```

`leave` は `mov rsp, rbp` と `pop rbp` をまとめた 1 バイト命令（`C9`）です。
プロローグで積んだものを綺麗に巻き戻してくれます。

> [!IMPORTANT]
> 16 バイト境界はうっかり破りやすい落とし穴です。プロローグの `push rbp`
> まで済んだ時点で `rsp` は 16 バイト境界に揃っています。そこから 8 バイトの
> `push` を**奇数回**したまま別の関数を `call` すると、`rsp` が 8 ずれたまま
> 呼んでしまい、呼ばれた先が SSE 命令（`movaps` など 16 バイト境界を要求する
> 命令）で落ちます。原因が「自分のバグ」でなく「相手が境界を仮定していた」と
> ころにあるので、追いにくい不具合の筆頭です。困ったら「`call` の前に
> ダミーで `push` を 1 つ足して偶奇を合わせる」を疑ってください。

## 引数を受け取って足し算する関数

次は、2 つの整数を受け取って足す関数です。System V の規約どおり、
整数の第 1 引数は `edi`、第 2 引数は `esi` に入って渡ってきます。
だから「`edi` を `eax` にコピーし、そこへ `esi` を足し、戻る」です。

```
89 F8    mov eax, edi    ; eax ← 第1引数
01 F0    add eax, esi    ; eax ← eax + 第2引数
C3       ret
```

```ruby
mem = jit([0x89, 0xF8, 0x01, 0xF0, 0xC3])
add = Fiddle::Function.new(mem,
        [Fiddle::TYPE_INT, Fiddle::TYPE_INT], Fiddle::TYPE_INT)
p add.call(20, 22)   # => 42
```

レジスタの番号と命令のバイト列の対応（`edi` が 7、`esi` が 6 で、それが
`F8` や `F0` というバイトのどこに埋まるか）は、x86-64 の命令フォーマットの
規則で決まっています。本書ではその規則の全体を覚える必要はありません。
大事なのは、**命令とバイト列のあいだに機械的な対応があり、表を引けば
組み立てられる**ということです。本物の JIT は、この「表引き」を
[フレームワーク](frameworks.md)の章で紹介するアセンブラライブラリに
任せます。

## 式をコンパイルする小さな JIT

固定の機械語を流し込むだけでは「JIT」と呼ぶには寂しい。最後に、
入力に応じて機械語を組み立てる小さなコンパイラを書きます。
これでようやく、前章の「`compile` で Ruby ソースを作った」例の、
行き先を機械語にしたものになります。

題材は、1 つの整数入力 `x` に対する、足す・引く・掛けるの列です。
たとえば `[[:add, 1], [:mul, 3], [:sub, 2]]` は `(x + 1) * 3 - 2` を
表します。これを直線の機械語へコンパイルします。使う命令は次の通りです。

```
89 F8                  mov  eax, edi              ; eax ← x（入力）
05 nn nn nn nn         add  eax, imm32            ; eax ← eax + n
2D nn nn nn nn         sub  eax, imm32            ; eax ← eax - n
69 C0 nn nn nn nn      imul eax, eax, imm32       ; eax ← eax * n
C3                     ret
```

`nn nn nn nn` の部分に、即値 `n` をリトルエンディアンの 4 バイトで
埋め込みます。これを Ruby で組み立てます。

```ruby
# 演算の列を x86-64 機械語へコンパイルする小さな JIT コンパイラ
def compile_expr(ops)
  code = [0x89, 0xF8]                      # mov eax, edi（入力を eax へ）
  ops.each do |op, n|
    imm = [n].pack('l<').bytes             # n を 4 バイト（リトルエンディアン）に
    case op
    when :add then code += [0x05] + imm            # add  eax, n
    when :sub then code += [0x2D] + imm            # sub  eax, n
    when :mul then code += [0x69, 0xC0] + imm      # imul eax, eax, n
    end
  end
  code << 0xC3                             # ret
  code
end

prog = [[:add, 1], [:mul, 3], [:sub, 2]]   # (x + 1) * 3 - 2
fn   = Fiddle::Function.new(jit(compile_expr(prog)),
         [Fiddle::TYPE_INT], Fiddle::TYPE_INT)

p fn.call(10)   # => 31   ((10 + 1) * 3 - 2)
p fn.call(0)    # => 1    ((0  + 1) * 3 - 2)
```

`compile_expr` は、れっきとした**コンパイラ**です。入力プログラム
（演算の列）を受け取り、それを実行する機械語を生成しています。生成した
機械語にはもうループも `case` もありません。`fn.call(10)` を呼ぶと、
CPU は私たちが組み立てた数命令を一直線に実行して 31 を返します。
前章で見た「ディスパッチを消す」が、機械語のレベルで実現したのです。

> [!TIP]
> `compile_expr` の出力を変えてみてください。`prog` を長くすれば
> 機械語も長くなり、それでも一直線です。これが「命令の並びが分かって
> いれば分岐をたためる」の正体です。インタプリタなら演算の数だけ
> ディスパッチを繰り返すところを、JIT は一度コンパイルしてしまえば
> 以後はディスパッチゼロで走り続けます。

ここまでで「ディスパッチを消す」までは機械語で体感できました。けれど
`compile_expr` は、**入力 `x` が常に整数である**ことを暗黙の前提にして
います。Ruby のような動的言語では、その前提が破れる（`x` が浮動小数や
文字列かもしれない）。次は、その「前提が当たっているか」を機械語で
確かめる仕掛け（**ガード**）を作ります。これが本物の JIT で型投機を
支える、いちばん小さな部品です。

## 賭けを機械語で守るタグ検査ガード

動的言語の JIT は「どうせ整数だろう」と**賭けて**、整数決め打ちの速い
コードを出します。けれど賭けが外れたとき（`x` が整数でなかったとき）に
備えて、コードの先頭に**ガード**（前提が成り立つか確かめる検査）を
置きます。当たれば検査を 1〜2 命令くぐって一直線、外れたら安全な汎用
処理へ逃がす。この「外れたときに別の実装へ逃がす」ことを本書では
**脱最適化**と呼びます（詳しくは[脱最適化と OSR](deopt-osr.md)）。

まず、動的言語が整数をどう表すかを思い出します。多くの処理系は、整数を
ポインタと区別するために**タグ**を使います。たとえば「値の**最下位
ビット（LSB）が 1 なら小さな整数（fixnum）**、0 ならヒープ上の
オブジェクトへのポインタ」という約束です。CRuby の fixnum がまさに
この方式で、`2*n+1` の形で整数を表します（かつてあった `Fixnum` クラスは
Ruby 2.4 で `Integer` に統合されましたが、内部表現としての fixnum は
健在です）。

すると整数かどうかの検査は「LSB が 1 か？」を見るだけで済みます。
x86-64 では `test` 命令でビットを調べられます。

```
40 F6 C7 01    test dil, 1      ; 第1引数 rdi の最下位8ビットと 1 の AND を取る
                                ; （結果は捨て、フラグだけ立てる）
```

`test dil, 1` は「`rdi` の下位 8 ビット `dil` と即値 `1` のビット AND を
計算し、結果が 0 かどうかをフラグ（ZF）に記録する（結果自体は捨てる）」
命令です。LSB が 1 なら AND は 1（≠0）で ZF=0、LSB が 0 なら AND は 0 で
ZF=1 になります。`40` は `dil` を指すための REX プレフィクスです。`dil`
（`rdi` の下位 8 ビット）のような新形式の 8 ビットレジスタは、たとえ
ビットを 1 つも立てない空の REX（`40`）であっても、これを 1 バイト前置
しないと指定できません（前置しないと従来の `bh` などを指してしまいます）。

このフラグを `jne`（jump if not equal、ZF=0 のとき飛ぶ）で見て
分岐します。ZF=0 になるのは AND が非ゼロ、すなわち LSB=1 のときです。
そこで本書のおもちゃは約束を「**LSB が 1 なら脱最適化（飛ぶ）、0 なら
速い整数経路（飛ばず一直線）**」と置きます（CRuby の `2*n+1` 方式とは
LSB の意味が逆ですが、既出の `test`＋`jne` の組み合わせをそのまま
再利用できるので、説明の都合でこの向きを採ります。`jne` を `jz`（ZF=1 で
飛ぶ、`0F 84 rel32`）に替えれば、命令長そのままで CRuby と同じ向きにも
できます）。

```
40 F6 C7 01    test dil, 1            ; rdi & 1 を計算（フラグだけ）
0F 85 nn nn nn nn  jne  deopt         ; 非ゼロ（LSB=1）なら脱最適化へ
```

`jne` には飛距離の短い 1 バイト版（`75 rel8`）と、遠くへ飛べる
`0F 85 rel32` 版があります。後でバックパッチ（次節）するときに飛び先が
遠いと困らないよう、本書では一貫して `0F 85` の rel32 版を使います。
`rel32` は**次の命令の先頭からの符号付き 32 ビット相対オフセット**です
（ここが要注意。「現在の命令の先頭から」ではありません）。

脱最適化先へは「汎用関数のアドレスへ飛ぶ」だけにします。アドレスは実行
時にしか分からない 64 ビット値なので、`movabs` でレジスタに即値ロード
してから間接ジャンプします。

```
48 B8 <8 bytes>  movabs rax, 汎用関数のアドレス   ; rax ← 絶対アドレス
FF E0            jmp    rax                       ; そこへ飛ぶ
```

全体を組み立てる動くおもちゃです。話を最短にするため、ここでは
「LSB が 0（＝速い経路に来てよい値）」のとき第 1・第 2 引数をそのまま
足して返し、「LSB が 1」のときは脱最適化スタブ（ここでは `-1` を返す
だけのダミー汎用関数）へ逃がします。ガードを置くのも第 1 引数だけで、
第 2 引数は無検査です（本物なら `rsi` 側にも同じ検査が要りますが、骨格を
見るために省きます）。なお、ここまでの足し算が 32 ビットの
`eax`/`edi` で済んでいたのに対し、ここから先は `48` 付きの 64 ビット
`rax`/`rdi`/`rsi` に切り替えます。タグ付き値は**ポインタと同じ幅**で扱う
必要があり（下位ビットにタグを載せたポインタ大の値だから）、32 ビットに
切り詰めると上位ビットを失ってしまうためです。

```ruby
# 脱最適化スタブ：本来は「型を見て汎用処理する関数」。
# ここでは動作確認のため、常に -1 を返すだけのダミー。
#   mov rax, -1   (48 C7 C0 FF FF FF FF)
#   ret           (C3)
stub      = jit([0x48, 0xC7, 0xC0, 0xFF, 0xFF, 0xFF, 0xFF, 0xC3])
stub_addr = stub.to_i

code  = []
code += [0x40, 0xF6, 0xC7, 0x01]            # test dil, 1
code += [0x0F, 0x85, 0, 0, 0, 0]            # jne rel32（飛び先は後で埋める）
patch_at = code.size - 4                     # 0 を置いた rel32 の位置を覚えておく

# --- 速い経路（賭けが当たったとき）：一直線 ---
code += [0x48, 0x89, 0xF8]                  # mov rax, rdi
code += [0x48, 0x01, 0xF0]                  # add rax, rsi
code += [0xC3]                              # ret

# --- 脱最適化経路（賭けが外れたとき） ---
deopt_off = code.size                        # この経路の先頭オフセット
code += [0x48, 0xB8] + [stub_addr].pack('Q<').bytes  # movabs rax, stub_addr
code += [0xFF, 0xE0]                         # jmp rax

# --- バックパッチ：rel32 = 飛び先 - 「jne の次の命令の先頭」 ---
rel = deopt_off - (patch_at + 4)
code[patch_at, 4] = [rel].pack('l<').bytes

f = Fiddle::Function.new(jit(code),
      [Fiddle::TYPE_LONG, Fiddle::TYPE_LONG], Fiddle::TYPE_LONG)

p f.call(20, 22)   # => 42   （20 は偶数＝LSB 0 → 速い経路、20+22）
p f.call(21, 22)   # => -1   （21 は奇数＝LSB 1 → 脱最適化スタブ）
```

`f.call(20, 22)` のとき、CPU は `test` と `jne` の 2 命令で「賭けが
当たった」と判断し、そのまま `mov`/`add`/`ret` を一直線に走ります。
分岐予測が効けば、当たり続けるかぎりガードはほぼ無視できるコスト
（数サイクル）で素通りされます。これが「投機が当たれば一直線」の
体感です。`f.call(21, 22)` のときだけ `jne` が成立し、脱最適化スタブへ
逃げます。

> [!NOTE]
> 本物の型投機はもっと込み入っています。整数をタグ付きで受け取った
> なら、足す前に**タグを外す**（`sar rax, 1` で 2 で割る、など）必要が
> あり、結果を返す前に**タグを付け直す**（`2*n+1` に戻す）必要も
> あります。さらに、足し算でのオーバーフローを `jo`（jump if overflow）
> で検査して、桁あふれしたら多倍長整数の経路へ脱最適化する、という
> ガードも要ります。ここでは仕組みの骨格だけを見せるため、それらを
> 省いています。詳しい型投機とガードの設計は
> [脱最適化と OSR](deopt-osr.md) で扱います。

## 飛び先がまだ分からない前方ジャンプとバックパッチ

前節でさらりと「飛び先は後で埋める」とやりました。これは JIT の
コード生成で毎度出てくる手筋なので、独立した節で取り出します。

問題はこうです。`jne deopt` を出力する時点では、脱最適化経路の機械語は
まだ生成していないので、飛び先のオフセットが分かりません。かといって
脱最適化経路を先に出すと、今度は速い経路の位置が分からなくなる。
**前方への参照（forward reference）**は、コードを 1 パスで素直に吐く
かぎり、必ずこのジレンマを生みます。

定石は **バックパッチ（後埋め）** です。

1. `jne` の rel32 のところに、とりあえず 0 を 4 バイト置いて先へ進む。
   その位置（バイトオフセット）を覚えておく。
2. 生成を続け、飛び先（脱最適化経路の先頭）の位置が確定する。
3. **オフセットを計算して、覚えておいた 4 バイトを埋め直す。**

計算は `rel32 = 飛び先の位置 − jne の次の命令の先頭の位置` です。前節の
コードでいえば、`jne` 本体は `0F 85` の 2 バイト＋ rel32 の 4 バイトで、
rel32 は `patch_at` から始まるので、「次の命令の先頭」は `patch_at + 4`
です。だから `rel = deopt_off - (patch_at + 4)`。仮に `deopt_off` が
速い経路（`mov`/`add`/`ret` の 7 バイト）の直後なら `rel = 7` で、
`jne` は 7 バイト先へ飛びます。

> [!TIP]
> 飛び先が後ろにある（後方ジャンプ、ループの戻りなど）場合は、
> パッチは要りません。飛び先の位置がすでに確定しているので、その場で
> `rel32 = 飛び先 − 次命令` を計算して埋められます。このとき `rel32` は
> 負の値（後ろ向き）になります。`[rel].pack('l<')` は負数も正しく
> 符号付き 4 バイトにしてくれます。

この「とりあえず 0 を置いて、後で正しい値で埋め直す」操作は、リンカが
オブジェクトファイルをつなぐときに行う**再配置（relocation）**と、まったく
同じ発想です[](#cite:levine2000)。アセンブラやリンカは、シンボルの最終
アドレスが決まるまで「ここを後で埋める」という印（再配置エントリ）を
残しておき、最後にまとめて穴を埋めます。JIT のバックパッチは、それを
実行時にメモリ上で行っているにすぎません。

本物の JIT では、ラベルごとに「まだ解決していない参照のリスト」を持ち、
ラベルが確定した瞬間にそのリストを総なめして全部埋める、という形に
一般化します。[フレームワーク](frameworks.md)の章で触れるアセンブラ
ライブラリ（DynASM など[](#cite:dynasm)）は、この面倒なラベル管理を
肩代わりしてくれます。さらに、生成済みの命令列をテンプレートとして
持っておき、定数部分だけを実行時に「穴埋め」する手法まで推し進めたものが
[Copy-and-Patch](research-copy-and-patch.md) で、これはバックパッチの
考え方をコンパイル戦略そのものに昇格させたものだと言えます。

> [!WARNING]
> 生成した機械語を後から書き換えた（自己書き換えをした）ときは、CPU の
> **命令キャッシュ（icache）とデータキャッシュ（dcache）の一貫性**に
> 注意が要ります。書き込みはデータとして dcache を通りますが、CPU が
> 命令を読むのは icache 経由なので、放っておくと古い命令が実行され
> 続けることがあります。x86-64 は比較的寛容で、同じコアでの書き換え
> なら多くの場合ハードウェアが整合性を取ってくれます（ただし他コアや
> 投機実行が絡むと話は別です）。一方 **ARM では明示的なキャッシュ
> フラッシュ**が必須で、書き込み後に `__builtin___clear_cache` 相当の
> 処理（dcache を書き戻し、icache を無効化する一連の命令）を呼ばないと
> 正しく動きません。本書のおもちゃは「書いてから初めて呼ぶ」だけで
> 自己書き換えはしないので問題になりませんが、本物の JIT が
> パッチ・再コンパイルをするときは、必ずこの同期を行っています。

## ここから先で足りないもの

この最小の JIT には、本物の JIT が持つものがまだ何もありません。
裏を返せば、本書の残りはこの「足りないもの」を埋めていく旅です。

> [!NOTE]
> 「無条件にコンパイルする」JIT は、実在しました。初代 V8（2008 年）は
> インタプリタを持たず、実行する関数をすべていきなり機械語へコンパイル
> していました。しかし、ベースラインコンパイラの生成コードだけで
> Chrome の JavaScript ヒープの 3 分の 1 近くを占めるまでに膨らみ、
> 2016 年、V8 はバイトコードインタプリタ Ignition を導入して「まず
> 解釈して、熱い所だけコンパイルする」方式へ回帰します。この方式の
> 行き来は[階層的 JIT](tiered.md)で扱います。

- **いつコンパイルするか**：ここでは無条件にコンパイルしました。本物は
  熱い場所だけをコンパイルします → [プロファイリング](profiling.md)。
- **何を単位にコンパイルするか**：式 1 本でなく、メソッドやループ、
  あるいは実行経路を単位にします → [コンパイルの単位](compilation-units.md)。
- **どう最適化するか**：ここでは素朴に 1 命令ずつ出しました。本物は
  中間表現を経て最適化します → [中間表現](ir.md)。
- **型をどう投機するか**：ここでは整数固定でした。動的言語では型を
  観測し、ガードで守ります → [脱最適化と OSR](deopt-osr.md)。
- **ガードが外れたらどうするか**：本章では脱最適化先を「`-1` を返す
  だけのダミー」にしました。本物は、外れた地点のレジスタ・スタックの
  値をインタプリタの状態へ復元し、続きをインタプリタで実行します。この
  状態復元の正しさが脱最適化の肝です → [脱最適化と OSR](deopt-osr.md)。
- **パッチをどう管理するか**：本章のバックパッチは 1 か所を手で埋め
  ました。本物はラベルと未解決参照のリストを管理し、再コンパイル時には
  既存コードの呼び出し口をパッチして差し替えます。生成済みコードへの
  パッチを突き詰めると [Copy-and-Patch](research-copy-and-patch.md) に
  至ります。
- **書き込みと実行をどう分けるか**：本章で `mmap`(RW)→`mprotect`(RX) の
  W^X を入れましたが、切り替え回数とアリーナ管理、ARM での icache
  同期まで含めると、コードメモリの運用は独立した設計課題です。攻撃者を
  相手取る話は [JIT とセキュリティ](security.md)で扱います。
- **ランタイムとどう共存するか**：生成したコードは GC・スレッドと折り合いを
  付けて生きていく必要があり、貯まったコードの置き場（コードキャッシュ）も
  管理が要ります → [JIT とランタイムの協調](gc-runtime.md)。
- **レジスタ割り当て**：ここでは `eax` 1 本で済みました。値が増えれば、
  限られたレジスタへどう割り当てるかが問題になります → [中間表現](ir.md)。

## まとめ

- JIT の物理的な正体は、**実行可能メモリに機械語を書き、関数として
  呼ぶ**こと。Linux では `mmap`(RW) → 書き込み → `mprotect`(RX) の
  順で、**W^X** を守りながら実行可能メモリを用意する。
- 生成コードは**呼び出し規約**（System V AMD64：引数 `rdi,rsi,...`、
  戻り `rax`、callee-saved の保存、`rsp` の 16 バイト境界）を守る必要が
  ある。
- 命令とバイト列には機械的な対応があり、表を引けば組み立てられる。
  `mov eax,42` = `B8 2A 00 00 00`、`test dil,1` = `40 F6 C7 01`、
  `jne rel32` = `0F 85 ……` のように、1 命令ずつ符号化できる。
- 入力に応じて機械語を組み立てれば、それはもう**コンパイラ**。
  生成コードにはディスパッチもループもない。
- 動的言語では「型は整数だろう」と**賭け**、先頭に**ガード**（`test`＋
  `jne`）を置く。当たれば一直線、外れれば**脱最適化**先へ逃がす。
- 飛び先が未定の前方ジャンプは、0 を置いて進み後で 4 バイトを埋め直す
  **バックパッチ**で解決する。リンカの再配置と同型。`rel32` は次命令
  からの相対である点に注意。
- 自己書き換え後は icache/dcache の同期が要る（x86 は緩いが ARM は明示
  フラッシュ必須）。
- 「いつ・何を・どう」コンパイルするかが、これ以降の章のテーマ。

機械語を吐く感触がつかめたところで、次は視点を上げます。「そもそも
**何を単位に**コンパイルするのか」。メソッドか、実行経路か。これが
JIT の設計を二分する最初の大きな分岐点です。
