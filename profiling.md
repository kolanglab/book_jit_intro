# プロファイリング：どこが熱いかを軽く知る

[なぜ JIT か](why-jit.md)で見たとおり、JIT が割に合うのは**繰り返し
実行されるホットスポット**だけです。だから JIT の最初の仕事は、
「どこが熱いか」と「そこは何を前提にできるか」を知ること —— すなわち
**プロファイリング**（profiling）です。

ここには本質的なジレンマがあります。プロファイリングは**インタプリタが
普通に実行している最中に**行われます。つまり、観測のコストがそのまま
実行を遅くします。**正確に測れば測るほど、測ること自体が重くなる。**
JIT のプロファイリングは、この「軽さ」と「正確さ」の綱引きです。
本章では、現場で使われる手法を軽い順に並べて見ていきます。

## いちばん素朴：カウンタで数える

最も基本的なのは、**数える**ことです。メソッドごと、ループごとに
カウンタを持ち、実行のたびに 1 増やす。閾値を超えたら「熱い」と判定して
コンパイルキューに積む。

```ruby
# カウンタによるホットスポット検出（概念）
class Interpreter
  THRESHOLD = 1000

  def initialize
    @counters = Hash.new(0)
  end

  def call_method(method)
    @counters[method] += 1
    if @counters[method] == THRESHOLD
      enqueue_for_compilation(method)   # 閾値ちょうどで一度だけ積む
    end
    interpret(method)
  end
end
```

数えるべき対象は二種類あります。

- **呼び出しカウンタ**：メソッドが呼ばれた回数。
- **バックエッジカウンタ**：ループの後ろ向きジャンプ（ループの先頭へ
  戻る辺）を踏んだ回数。

なぜループを別に数えるかというと、「**1 回しか呼ばれないが、中で 100 万回
回るループ**」を救うためです。呼び出しカウンタだけ見ていると、このメソッドは
「1 回しか呼ばれていない＝冷たい」と誤判定されます。バックエッジを数えれば、
ループの熱さを直接捉えられます。そして、すでに回り始めてしまった長い
ループを途中から機械語へ乗り換える技術が、[OSR](deopt-osr.md)（On-Stack
Replacement）です。

> [!NOTE]
> 閾値の値（上の例の 1000）は、JIT のチューニングの要です。低すぎると
> 「すぐ冷める場所」まで無駄にコンパイルし、高すぎるとコンパイルが
> 遅れて高速化の機会を逃します。HotSpot のような成熟した処理系では、
> 呼び出しカウンタとバックエッジカウンタを組み合わせた式で判定します。

## カウンタはタダではない

カウンタは軽そうに見えますが、**すべてのメソッド呼び出し・すべての
ループ周回でインクリメントが走る**ことを忘れてはいけません。一番熱い
ループの中で、毎周「カウンタを読んで、足して、書いて、閾値と比べる」のは、
それ自体が無視できないコストです。

そこで実装上の工夫がいくつもあります。

- **カウンタを減らす方向に使う**：閾値から始めてゼロまで**減らし**、
  ゼロになったら発火。比較が「ゼロか？」だけになり、わずかに速い。
- **コンパイル後はカウンタを外す**：一度コンパイルしたら、そのメソッドの
  カウンタ更新コードを取り除く（機械語版にはカウンタがない）。

## もっと軽く：サンプリング

カウンタは「全部数える」方式でした。逆の発想が**サンプリング**
（sampling）です。一定間隔（たとえば 10 ミリ秒ごと）にタイマー割り込みを
かけ、その瞬間に**どのメソッドを実行していたか**だけを記録する。
これを集計すれば、「実行時間の大半をどこで使っているか」の統計が、
**ごく低いコストで**得られます。

サンプリングの利点は、オーバーヘッドが**実行されるコード量に比例しない**
ことです。100 万回回るループでも、サンプリングのコストは「割り込みの
回数」ぶんだけ。だから熱いコードほど割安になります。Jalapeño/Jikes RVM の
適応的最適化システムは、この割り込みサンプリングでメソッドの熱さを
推定しました[](#cite:arnold2000)。

欠点は**不正確さ**です。サンプリングは確率的なので、たまにしか実行され
ないが毎回重い、といったコードを取りこぼすことがあります。多くの処理系は
カウンタとサンプリングを併用します。

## 何が熱いかだけでなく、何を前提にできるか

ホットスポットの場所が分かっても、それだけでは投機できません。
[なぜ JIT か](why-jit.md)で見たように、動的言語を速くする鍵は
**型の決め打ち**です。そのためには「この場所では、実際にどんな型が
来ていたか」を知る必要があります。これを担うのが**インラインキャッシュ**と
**フィードバック**です。

### インラインキャッシュ

`obj.foo` という呼び出しは、毎回 `obj` のクラスからメソッド `foo` を
探します。ところが現実には、**同じ場所の `obj` はいつも同じクラス**で
あることがほとんどです。ならば、「前回ここに来たときのクラスと、その
ときに見つけたメソッド」を、その**呼び出し場所に覚えておけば**、次回は
探索を省けます。これが**インラインキャッシュ**（inline cache、IC）です
[](#cite:deutsch1984)。

```ruby
# 1 つの呼び出し場所に付くインラインキャッシュ（単態：monomorphic）
class CallSite
  def initialize(name)
    @name = name
    @cached_class  = nil
    @cached_method = nil
  end

  def invoke(obj, *args)
    klass = obj.class
    if klass == @cached_class          # キャッシュ命中：探索を飛ばす
      @hits += 1
    else                               # キャッシュ外れ：探して覚え直す
      @cached_class  = klass
      @cached_method = klass.instance_method(@name)
      @misses += 1
    end
    @cached_method.bind(obj).call(*args)
  end

  attr_reader :hits, :misses
  def initialize_counters; @hits = 0; @misses = 0; end
end
```

呼び出し場所に「前回のクラス」を 1 つ覚えるものを**単態インライン
キャッシュ**（monomorphic IC）と呼びます。来るクラスが 2〜数種類なら、
複数覚える**多態インラインキャッシュ**（polymorphic IC、PIC）にします
[](#cite:holzle1994)。それより多くの種類が来る場所は「乱れた」
（megamorphic）と見なし、キャッシュをあきらめて普通の探索に戻します。

### キャッシュが、そのまま JIT への報告書になる

ここに JIT の鮮やかな再利用があります。インラインキャッシュは元々
「探索を省くインタプリタの高速化」ですが、**そこに溜まった情報は、
そのまま「この場所にはこの型が来る」という観測記録**です。JIT は
コンパイルするとき、このキャッシュを**型のヒント**として読みます。

V8 はこの観測記録を**フィードバックベクタ**（feedback vector）という
配列に集約します。関数ごとに、「この呼び出し場所のレシーバの形」
「この加算の被演算子の型」といった観測スロットを並べた配列です。JIT
（V8 なら [Maglev や TurboFan](case-v8.md)）は、これを読んで「ここは
整数同士の加算として特殊化してよい」と判断します。つまり、インライン
キャッシュは**インタプリタを速くしながら、同時に JIT のためのプロファイルを
貯めている**のです。一粒で二度おいしい。

> [!IMPORTANT]
> このアイデア —— 「実行を速くするために置いた仕掛けが、同時に最適化の
> ための観測を兼ねる」—— は JIT 設計の中心思想です。プロファイリングを
> 別建ての重い処理にするのではなく、**ふだんの実行に溶け込ませる**。
> インラインキャッシュはその最も成功した例です。

## 命令を自分で書き換える：特殊化適応型インタプリタ

機械語の JIT を持たなくても、観測した型でコードを差し替えるだけで、
かなりの高速化ができます。**バイトコード命令を、観測した型に特化した
別命令へ実行時に書き換える**のです。これを**特殊化適応型インタプリタ**
（specializing adaptive interpreter）と呼び、CPython 3.11 が採用しました
[](#cite:pep659)。汎用の `BINARY_OP` を、整数同士だと観測したら
`BINARY_OP_ADD_INT` へ差し替える、といった具合です。

この「命令が観測結果で自分自身を書き換える」仕掛けは、玩具で再現できます。

```ruby
# quickening：命令が観測した型で自分を特殊化版に書き換える
def execute(code, stack)
  pc = 0
  while (insn = code[pc])
    case insn[0]
    when :push then stack << insn[1]
    when :add                              # 汎用版：型を調べてから足す
      b, a = stack.pop, stack.pop
      code[pc] = [:add_int] if a.is_a?(Integer) && b.is_a?(Integer)
      stack << a + b                       # ↑ 観測した型で命令を書き換える
    when :add_int                          # 特殊化版：整数前提の高速経路
      b, a = stack.pop, stack.pop
      code[pc] = [:add] unless a.is_a?(Integer) && b.is_a?(Integer)
      stack << a + b                       # ↑ 賭けが外れたら汎用版へ戻す
    end
    pc += 1
  end
  stack.pop
end

code = [[:push, 1], [:push, 2], [:add]]
p execute(code, [])   # => 3
p code[2]             # => [:add_int]   命令が書き換わっている
```

実行後、`code[2]` が `[:add]` から `[:add_int]` に変わっています。
本物の特殊化版は、整数だと分かっているぶん型タグの検査や分岐を省き、
場合によってはアンボックスした生の整数で計算するので速くなります。
ここで見るべきは仕掛けのほう —— **命令列という配列が、プロファイルを
記録する可変のデータ構造を兼ねている**ことです。インラインキャッシュと
同じ思想が、命令そのものに宿っています。この「特殊化を取り消して汎用へ
戻す」（`code[pc] = [:add]`）動きは、機械語 JIT における
[デオプティマイズ](deopt-osr.md)のいちばん軽いミニチュアでもあります。

## プロファイルはいつまで信じられるか

最後に、プロファイリングの落とし穴を一つ。**プロファイルは過去の観測で
あって、未来の保証ではありません。** 「最初の 1000 回は整数だった」からと
いって、1001 回目も整数とは限りません。だからこそ JIT は観測を**ガード
付きの投機**として使うのです。観測を信じて特殊化し、しかし念のため
ガードを置き、外れたら[デオプティマイズ](deopt-osr.md)で戻る。

プログラムの「ふるまいの相」が途中で変わる（最初は整数を、途中から
浮動小数点数を扱い始める、など）と、JIT はせっかくコンパイルしたコードを
捨てて作り直すことになります。この**プロファイルの陳腐化**は、後で
[ウォームアップ問題](research-warmup.md)として研究テーマにもなります。

## まとめ

- プロファイリングは「**どこが熱いか**」と「**何を前提にできるか**」を、
  通常実行に溶け込ませて軽く測る技術。正確さと軽さの綱引き。
- 場所の検出：**呼び出しカウンタ**と**バックエッジカウンタ**（ループ用）。
  全数えのカウンタと、低コストな**サンプリング**を併用する。
- 型の観測：**インラインキャッシュ**が探索を省きつつ型を記録し、それが
  そのまま JIT への**フィードバック**（フィードバックベクタ）になる。
- 機械語 JIT がなくても、**命令を観測型で特殊化版へ書き換える**だけで
  速くできる（CPython の特殊化適応型インタプリタ）。
- プロファイルは過去の観測にすぎず、**ガード付きの投機**として使う。

熱い場所と、そこで来る型が分かりました。次は、その情報をもっとも積極的に
使う方式 —— 実行経路をまるごと記録してコンパイルする**トレーシング JIT**の、
記録と最適化の中身に踏み込みます。
