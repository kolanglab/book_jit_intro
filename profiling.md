# どこが熱いかを軽く知るプロファイリング

[なぜ JIT か](why-jit.md)で見たとおり、JIT が割に合うのは**繰り返し
実行されるホットスポット**だけです。だから JIT の最初の仕事は、
「どこが熱いか」と「そこは何を前提にできるか」を知ること、すなわち
**プロファイリング**（profiling）です。

ここには本質的なジレンマがあります。プロファイリングは、インタプリタが
普通に実行している最中に行われます。つまり、観測のコストがそのまま
実行を遅くします。正確に測れば測るほど、測ること自体が重くなる。
JIT のプロファイリングは、この「軽さ」と「正確さ」の綱引きです。
本章では、現場で使われる手法を軽い順に並べます。

## カウンタで数える

最も基本的なのは、**数える**ことです。メソッドごと、ループごとに
カウンタを持ち、実行のたびに 1 増やす。閾値を超えたら「熱い」と判定して
コンパイルキューに積む。

```ruby
# カウンタによるホットスポット検出（概念）
class Interpreter
  THRESHOLD = 1000

  def initialize
    @counters = Hash.new(0)
  end

  def call_method(method)
    @counters[method] += 1
    if @counters[method] == THRESHOLD
      enqueue_for_compilation(method)   # 閾値ちょうどで一度だけ積む
    end
    interpret(method)
  end
end
```

数えるべき対象は二種類あります。

- **呼び出しカウンタ**：メソッドが呼ばれた回数。
- **バックエッジカウンタ**：ループの後ろ向きジャンプ（ループの先頭へ
  戻る辺）を踏んだ回数。

なぜループを別に数えるかというと、「1 回しか呼ばれないが、中で 100 万回
回るループ」を救うためです。呼び出しカウンタだけ見ていると、このメソッドは
「1 回しか呼ばれていない＝冷たい」と誤判定されます。バックエッジを数えれば、
ループの熱さを直接捉えられます。そして、すでに回り始めてしまった長い
ループを途中から機械語へ乗り換える技術が、[OSR](deopt-osr.md)（On-Stack
Replacement）です。

> [!NOTE]
> 閾値の値（上の例の 1000）は、JIT のチューニングの要です。低すぎると
> 「すぐ冷める場所」まで無駄にコンパイルし、高すぎるとコンパイルが
> 遅れて高速化の機会を逃します。HotSpot のような成熟した処理系では、
> 呼び出しカウンタとバックエッジカウンタを組み合わせた式で判定します。

## カウンタはタダではない

カウンタは軽そうに見えますが、**すべてのメソッド呼び出し、すべての
ループ周回でインクリメントが走る**ことを忘れてはいけません。一番熱い
ループの中で、毎周「カウンタを読んで、足して、書いて、閾値と比べる」のは、
それ自体が無視できないコストです。

そこで実装上の工夫がいくつもあります。

- **カウンタを減らす方向に使う**：閾値から始めてゼロまで**減らし**、
  ゼロになったら発火。比較が「ゼロか？」だけになり、わずかに速い。
- **コンパイル後はカウンタを外す**：一度コンパイルしたら、そのメソッドの
  カウンタ更新コードを取り除く（機械語版にはカウンタがない）。

減算カウンタの実装を、少しだけ具体的に見ておきましょう。機械語に
落としたとき、「読んで・引いて・書いて・ゼロなら飛ぶ」は、メモリを直接
減算できる x86 なら `dec` と条件ジャンプの 2 命令、ロードとストアが分かれる
ISA でも 3〜4 命令で済みます。これを毎呼び出しの先頭に置くわけです。

```ruby
# 減算カウンタ：閾値から 0 まで引き、0 で一度だけ発火
class CountdownCounter
  def initialize(threshold) = @count = threshold

  def tick                       # 呼び出し/バックエッジのたびに 1 回
    @count -= 1                  #  ← 機械語なら dec + jz の 2 命令相当
    @count <= 0                  # ゼロ以下なら「熱い」
  end
end

c = CountdownCounter.new(3)
p [c.tick, c.tick, c.tick, c.tick]   # => [false, false, true, true]
```

### 古い熱を冷ますエイジング

単純に増やし続ける（あるいは引き続ける）カウンタには弱点があります。
**プログラムの相が変わっても、過去の蓄積が残り続ける**ことです。起動直後の
初期化フェーズでたまたま 800 回呼ばれたメソッドが、本処理に入ると二度と
呼ばれない。それでもカウンタは 800 のままで、あと 200 回ぶんの「下駄」を
履いています。これではいまの相を正しく追えません。

そこで使うのが**エイジング**（aging）／**減衰**（decay）です。一定間隔で
すべてのカウンタを**半分にする**（あるいは一定割合だけ引く）。こうすると、
最近よく実行されている場所のカウンタだけが高く保たれ、過去の熱は時間と
ともに忘れられます。指数移動平均と同じ発想です。

```ruby
# エイジング付きカウンタ：定期的に半減して「最近の熱」だけ残す
class AgingCounter
  HOT = 1000
  def initialize = @count = 0

  def tick
    @count += 1
    @count >= HOT
  end

  def age!                       # セーフポイントやタイマーの節目で全カウンタに適用
    @count >>= 1                 #  ← 右シフト 1 = 半減（割り算より速い）
  end
end

c = AgingCounter.new
500.times { c.tick }; p c.instance_variable_get(:@count)  # => 500
c.age!;                p c.instance_variable_get(:@count)  # => 250
c.age!;                p c.instance_variable_get(:@count)  # => 125
```

半減を右シフト 1 ビットで済ませているのがミソで、割り算を使わずに
減衰できます。HotSpot はセーフポイント（後述）の節目などでメソッドの起動カウンタを減衰させ、
「ずっと昔に少しだけ熱かった」コードを誤ってコンパイルしないようにして
います。

> [!TIP]
> エイジングの間隔と減衰率はトレードオフです。**速く減衰させすぎる**と、
> 本当はじわじわ熱いコードがいつまでも閾値に届かず、コンパイルが遅れます。
> **減衰が緩すぎる**と、相変化への追従が鈍くなります。目安として、HotSpot は
> 呼び出しカウンタの**半減期を既定で 30 秒**（`CounterHalfLifeTime`）とし、
> 減衰処理そのものは最短でも 500 ミリ秒の間隔を置いて走らせます。

### オーバーフロー飽和とカウンタの除去

カウンタは有限ビットの整数です。8 ビットや 16 ビットの狭い領域に詰め込む
実装では、加算し続けると**桁あふれ**（overflow）します。あふれて 0 に
戻ると「急に冷たく見える」ので、上限に達したらそこで**飽和**させる
（それ以上増やさない）か、あるいは飽和をトリガにコンパイルを起動します。
減算カウンタなら逆に、下限 0 で止めれば自然に飽和します。

そして最後に効くのが**コンパイル後のカウンタ除去**です。一度機械語へ
コンパイルしてしまえば、そのメソッドの「熱さ」はもう判定済みなので、
カウンタを更新し続ける理由がありません。コンパイル済みコードからは
インクリメント命令をまるごと外し、ホットパス（実行が集中する経路）の
命令数を削ります。
「観測のための仕掛けは、観測が終わったら撤去する」。これもプロファイ
リングのコストを下げる定石です。

## サンプリング

カウンタは「全部数える」方式でした。逆の発想が**サンプリング**
（sampling）です。一定間隔（たとえば 10 ミリ秒ごと）にタイマー割り込みを
かけ、その瞬間にどのメソッドを実行していたかだけを記録する。
これを集計すれば、「実行時間の大半をどこで使っているか」の統計が、
ごく低いコストで得られます。

サンプリングの利点は、オーバーヘッドが実行されるコード量に比例しない
ことです。100 万回回るループでも、サンプリングのコストは「割り込みの
回数」ぶんだけ。だから熱いコードほど割安になります。Jalapeño/Jikes RVM の
適応的最適化システムは、この割り込みサンプリングでメソッドの熱さを
推定しました[](#cite:arnold2000)。

欠点は不正確さです。サンプリングは確率的なので、たまにしか実行され
ないが毎回重い、といったコードを取りこぼすことがあります。多くの処理系は
カウンタとサンプリングを併用します。

サンプリングの中身は、**割り込み→スタックトップ→集計**という、おおむね
次の三段です。

1. **タイマー割り込みを仕掛ける**：OS のインターバルタイマー（たとえば
   `setitimer` で 10 ミリ秒ごと）を設定し、シグナルハンドラを登録する。
2. **割り込んだ瞬間の「実行中メソッド」を読む**：ハンドラの中で、いま
   走っているスレッドの**スタックトップ**（最上段のフレーム）を覗き、
   どのメソッドのどの位置にいたかを 1 件記録する。
3. **集計する**：記録を表に足し込む。あるメソッドのサンプル数が多ければ、
   そこに実行時間が集中している、と推定できる。

おもちゃで「集計」の部分だけ再現すると、こうなります。割り込みの代わりに、
あらかじめ用意した「割り込み時刻ごとのスタックトップ列」を流し込みます。

```ruby
# サンプリングプロファイラの集計部（割り込みは擬似的に与える）
def aggregate(samples)
  hist = Hash.new(0)
  samples.each { |top_method| hist[top_method] += 1 }   # スタックトップを数える
  total = samples.size
  hist.sort_by { |_, n| -n }.map { |m, n|
    [m, n, (100.0 * n / total).round(1)]                # メソッド, サンプル数, 割合%
  }
end

# 10ms ごとの割り込みで観測された「そのとき実行中だったメソッド」
samples = %i[parse parse eval parse eval gc parse eval parse eval]
aggregate(samples).each { |m, n, pct| puts "#{m}: #{n} (#{pct}%)" }
# parse: 5 (50.0%)
# eval: 4 (40.0%)
# gc: 1 (10.0%)
```

実装上の急所は、**セーフポイント**（safepoint）での取得です。割り込みは
本当にいつでも飛んできます。レジスタの値が中途半端だったり、オブジェクトの
レイアウトが書き換え途中だったりする瞬間にスタックを覗くと、整合しない
データを読みかねません。そこで多くの処理系は、スタックを「安全に」歩ける
地点をあらかじめ決めておき（メソッド境界やバックエッジなど）、割り込みを
受けたら次のセーフポイントまで実行を進めてからスタックを観測します。

> [!NOTE]
> ただし、セーフポイントでの観測はタダではありません。Mytkowicz らは、
> 広く使われていた 4 つの Java プロファイラに同じプログラムを測らせると、
> 互いに矛盾するホットスポットを報告することを示しました
> [](#cite:mytkowicz2010)。原因は、どのプロファイラもサンプル取得を
> セーフポイントまで遅らせるため、観測結果がセーフポイントの配置に
> 引きずられること（**セーフポイントバイアス**）です。「安全に測る工夫」が
> そのまま「測定の偏り」になる。軽さと正確さの綱引きの、苦い実例です。

> [!NOTE]
> カウンタとサンプリングは役割分担します。カウンタは「この特定の
> メソッドが閾値に達したか」という**個別の発火判定**に向き、サンプリングは
> 「いま全体としてどこが重いか」という**俯瞰の優先順位付け**に向きます。
> Jikes RVM は、サンプリングで重いと分かったメソッド群を、コスト／利益の
> 見積もりに基づいて最適化キューへ並べました[](#cite:arnold2000)。

## 何が熱いかだけでなく、何を前提にできるか

ホットスポットの場所が分かっても、それだけでは投機できません。
[なぜ JIT か](why-jit.md)で見たように、動的言語を速くする鍵は
型の決め打ちです。そのためには「この場所では、実際にどんな型が
来ていたか」を知る必要があります。これを担うのが**インラインキャッシュ**と
**フィードバック**です。

### インラインキャッシュ

`obj.foo` という呼び出しは、毎回 `obj` のクラスからメソッド `foo` を
探します。ところが現実には、**同じ場所の `obj` はいつも同じクラス**で
あることがほとんどです。そこで「前回ここに来たときのクラスと、そのとき
見つけたメソッド」を呼び出し場所に覚えておく。これが
[なぜ JIT か](why-jit.md)で導入した**インラインキャッシュ**（IC）でした
[](#cite:deutsch1984)。本章では、この IC を観測装置として読み解きます。
なお、IC の「鍵」は言語のクラスとは限りません。たとえば V8 はオブジェクトの
**ヒドゥンクラス**（シェイプ）を鍵にします（詳しくは
[V8 の事例](case-v8.md)で）。本章では話を簡単にするため、鍵をクラスとして
扱います。まず単態の IC を、観測の様子が見えるおもちゃとして
書き直しておきます。

```ruby
# 1 つの呼び出し場所に付くインラインキャッシュ（単態：monomorphic）
class CallSite
  def initialize(name)
    @name = name
    @cached_class  = nil
    @cached_method = nil
    @hits = @misses = 0
  end

  def invoke(obj, *args)
    klass = obj.class
    if klass == @cached_class          # キャッシュ命中：探索を飛ばす
      @hits += 1
    else                               # キャッシュ外れ：探して覚え直す
      @cached_class  = klass
      @cached_method = klass.instance_method(@name)
      @misses += 1
    end
    @cached_method.bind(obj).call(*args)
  end

  attr_reader :hits, :misses
end
```

呼び出し場所に「前回のクラス」を 1 つ覚えるものを**単態インライン
キャッシュ**（monomorphic IC）と呼びます。来るクラスが 2〜数種類なら、
複数覚える**多態インラインキャッシュ**（polymorphic IC、PIC）にします
[](#cite:holzle1994)[](#cite:holzle1991)。それより多くの種類が来る場所は
メガモーフィックと見なし、キャッシュをあきらめて普通の探索に戻します。

### 多態インラインキャッシュ（PIC）の動くおもちゃ

単態キャッシュは、`obj` のクラスが 2 種類以上で交互に来ると、毎回外れて
覚え直す**キャッシュのばたつき**（thrashing）を起こします。たとえば
`Integer` と `Float` が交互に来る加算では、単態キャッシュの命中率は
ほぼ 0% に落ちてしまいます。

PIC は、ひとつの呼び出し場所に複数の (クラス→メソッド) を配列で持ち、
線形に試します。Self 言語で導入されたこの工夫が、多態な呼び出しを
実用速度にしました[](#cite:holzle1991)。容量（ここでは 4）を超える種類が
来たら、これ以上覚えても割に合わないと判断して**メガモーフィック**
（megamorphic）状態へ落とし、キャッシュをあきらめてグローバル探索に戻します。

```ruby
# 多態インラインキャッシュ（PIC）：最大 CAPACITY 種類まで線形に保持
class PolymorphicCallSite
  CAPACITY = 4                      # これを超えたら megamorphic に転落

  def initialize(name)
    @name    = name
    @entries = []                   # [[klass, method], ...] を線形に持つ
    @state   = :uninitialized       # :mono / :poly / :mega と遷移
    @hits = @misses = @lookups = 0
  end

  def invoke(obj, *args)
    klass = obj.class

    if @state == :megamorphic       # メガモーフィックな場所：もうキャッシュしない
      @lookups += 1
      return klass.instance_method(@name).bind(obj).call(*args)
    end

    @entries.each do |k, m|         # キャッシュを線形に走査（命中を狙う）
      if k == klass
        @hits += 1
        return m.bind(obj).call(*args)
      end
    end

    @misses += 1                    # どのエントリにも無い → 探索して追加
    meth = klass.instance_method(@name)
    if @entries.size < CAPACITY
      @entries << [klass, meth]
      @state = @entries.size == 1 ? :mono : :poly
    else
      @state   = :megamorphic       # 容量超過：転落、以後は探索のみ
      @entries = []                 # 持っても無駄なので捨てる
    end
    meth.bind(obj).call(*args)
  end

  attr_reader :hits, :misses, :lookups, :state
end

# 試運転：2 クラスが交互（PIC が活きる） vs 多種類（megamorphic に転落）
class A; def foo = :a; end
class B; def foo = :b; end
class C; def foo = :c; end
class D; def foo = :d; end
class E; def foo = :e; end

cs = PolymorphicCallSite.new(:foo)
1000.times { cs.invoke(A.new); cs.invoke(B.new) }   # 2 クラスが交互
p [cs.state, cs.hits, cs.misses]
# => [:poly, 1998, 2]    最初の各 1 回だけ外れ、あとは全部命中

cs2 = PolymorphicCallSite.new(:foo)
[A, B, C, D, E].each { |k| 100.times { cs2.invoke(k.new) } }
p [cs2.state, cs2.hits, cs2.misses, cs2.lookups]
# => [:megamorphic, 396, 5, 99]   5 種類目で転落し、以後は探索に戻る
```

ここでは最初の 4 クラスが各 1 回だけ外れ、残りは 99 回ずつ命中します
（4 クラス × 99 回 = 396 命中。ミスは 5 クラスの初回ぶんで計 5）。
5 種類目（E）が現れた瞬間に `megamorphic` へ落ち、
それ以降の 99 回は線形走査すらやめて素直に探索（`lookups`）へ戻ります。
単態キャッシュなら最初の例は約 2000 回すべて外れていたところ、PIC では
わずか 2 回の外れで済み、残りは命中する、という差も見てとれます。

### 単態・多態・メガモーフィックの三状態

単態から多態、メガモーフィックへという三状態と、それぞれのキャッシュ側の
ふるまいは[なぜ JIT か](why-jit.md)の表で見たとおりです。本章の関心は
その先、**状態を JIT がどう読むか**にあります。

| 状態（見たクラス数） | JIT の打ち手 |
|------|------|
| 単態（1） | レシーバ型を 1 つに決め打ち、**インライン化**しやすい |
| 多態（2〜4 程度） | 数種類への**分岐付きインライン化**、または最頻クラスを投機 |
| メガモーフィック（5 以上程度） | 型を絞れない。**インライン化を断念**し汎用の呼び出しに任せる |

> [!IMPORTANT]
> 「何種類でキャッシュをあきらめるか」（上の例の 4）は、設計上の重要な
> ノブです。大きくしすぎると、滅多に命中しないエントリを線形に
> 舐め続けてかえって遅くなり、メモリも食います。小さくしすぎると、
> 本当は 3〜4 種類で安定する場所まで早々にメガモーフィック扱いして、せっかくの
> 投機機会を捨てます。実装では 4〜8 あたりが一つの相場です。メガモーフィックに
> 落ちた場所は「ここは型で語れない」という、それ自体が貴重な情報になります。

### キャッシュを JIT への報告書として読む

ここに JIT の鮮やかな再利用があります。インラインキャッシュは元々
「探索を省くインタプリタの高速化」ですが、**そこに溜まった情報は、
そのまま「この場所にはこの型が来る」という観測記録**です。JIT は
コンパイルするとき、このキャッシュを型のヒントとして読みます。

V8 はこの観測記録を**フィードバックベクタ**（feedback vector）という
配列に集約します。関数ごとに、「この呼び出し場所のレシーバの形」
「この加算の被演算子の型」といった観測スロットを並べた配列です。JIT
（V8 なら [Maglev や TurboFan](case-v8.md)）は、これを読んで「ここは
整数同士の加算として特殊化してよい」と判断します。つまり、インライン
キャッシュは**インタプリタを速くしながら、同時に JIT のためのプロファイルを
貯めている**のです。一つの仕掛けで二つの役を兼ねます。

### フィードバックベクタの具体構造

もう少し中身を覗いてみましょう。関数を解析（バイトコード化）するとき、
処理系は観測したい地点にあらかじめ番号（スロット番号）を振ります。
おもに次の三種類です。

- **呼び出しスロット**：`obj.foo(...)` のレシーバの形と、呼び出し先メソッド。
- **二項演算スロット**：`a + b` などの被演算子の型（整数だけ見たか、
  浮動小数点も混じったか、文字列が来たか）。
- **要素アクセススロット**：`a[i]` の配列の種類（要素が整数密だけか、
  穴あきか、オブジェクト混在か）や、添字が整数だったか。

関数ごとに、これらを 1 本の配列にまとめたものがフィードバックベクタです。
なお「どちらの枝を何回通ったか」という**分岐プロファイル**は、V8 の
フィードバックベクタではなく、HotSpot がメソッドごとに持つ **MDO**
（MethodData オブジェクト）のような構造が蓄える情報です。次のおもちゃでは
話を簡単にするため、これも同じベクタに分岐スロットとして相乗り
させます。

```ruby
# フィードバックベクタのおもちゃ：関数内の観測点ごとにスロットを持つ
class FeedbackVector
  Slot = Struct.new(:kind, :seen_types, :targets, :true_n, :false_n)

  def initialize(n_slots)
    @slots = Array.new(n_slots) { Slot.new(nil, [], Hash.new(0), 0, 0) }
  end

  # 呼び出し地点：レシーバのクラスと呼び出し先を記録
  def record_call(slot, recv, target)
    s = @slots[slot]; s.kind = :call
    s.seen_types |= [recv.class]        # 見た型の集合（重複しない）
    s.targets[target] += 1
  end

  # 二項演算地点：両オペランドの型を記録
  def record_binop(slot, a, b)
    s = @slots[slot]; s.kind = :binop
    s.seen_types |= [a.class, b.class]
  end

  # 分岐地点：真偽どちらを通ったか数える
  def record_branch(slot, taken)
    s = @slots[slot]; s.kind = :branch
    taken ? s.true_n += 1 : s.false_n += 1
  end

  def [](slot) = @slots[slot]
end

fv = FeedbackVector.new(3)
100.times { fv.record_binop(0, 1, 2) }              # 整数 + 整数ばかり
100.times { |i| fv.record_branch(1, i < 95) }       # ほぼ true に偏る
3.times { fv.record_call(2, "x", :String_upcase) }  # レシーバは String

p fv[0].seen_types               # => [Integer]            単態の加算
p [fv[1].true_n, fv[1].false_n]  # => [95, 5]              枝は強く偏る
p fv[2].seen_types               # => [String]             単態の呼び出し
```

JIT がこのベクタをどう読むかは単純明快です。スロット 0 は `Integer` しか
見ていないので「整数同士の加算」に特殊化し、型タグ検査を 1 回のガードに
まとめます。スロット 1 は 95% が `true` なので、`true` 側をホットパス
として直線に並べ、`false` 側は外へ追い出します。スロット 2 はレシーバが
`String` 一択なので `String#upcase` をインライン展開してもよい、と
判断します。

逆に、もしスロット 0 の `seen_types` が `[Integer, Float, String]` の
ように散らばっていれば、JIT は「ここは型で語れない」と見て特殊化を
あきらめ、汎用の加算ルーチンに委ねます。観測が**特殊化の許可証**として
機能しているわけです。

> [!IMPORTANT]
> このアイデア（「実行を速くするために置いた仕掛けが、同時に最適化の
> ための観測を兼ねる」）は JIT 設計の中心思想です。プロファイリングを
> 別建ての重い処理にするのではなく、**ふだんの実行に溶け込ませる**。
> インラインキャッシュはその最も成功した例です。

### 分岐確率と値プロファイル

型だけが投機の材料ではありません。**どちらの枝に何割流れたか**（分岐確率）と、
**特定の値が支配的に来ていないか**（値プロファイル）も、強力な手がかりです。

分岐確率は、`if` の真偽それぞれを通った回数を数えるだけです。たとえば
`if x.nil?` がほぼ毎回 `false` なら、JIT は `false` 側を**フォールスルー**
（条件分岐せず素直に次へ進む側）に置き、`true` 側を関数の末尾やコールド
領域へ追い出します。こうするとホットパスが直線的に並び、CPU の命令
フェッチと分岐予測に優しいコードになります。さらに「片方は一度も通って
いない」と分かれば、そちらをまるごと省いたコードを投機的に生成し、
もし通ったら[脱最適化](deopt-osr.md)で戻す、という強気な最適化もできます。

```ruby
# 分岐プロファイルから「ホットな枝」と「死んでいる枝」を見分ける
def classify_branch(true_n, false_n)
  total = true_n + false_n
  return :unknown if total == 0
  p_true = true_n.to_f / total
  case
  when true_n  == 0 then :always_false   # true 側は実行投機で削れる
  when false_n == 0 then :always_true    # false 側は削れる
  when p_true >= 0.95 then :hot_true      # true をフォールスルーに
  when p_true <= 0.05 then :hot_false     # false をフォールスルーに
  else :balanced                          # 偏りなし。投機しない
  end
end

p classify_branch(1000, 0)    # => :always_true
p classify_branch(970, 30)    # => :hot_true
p classify_branch(520, 480)   # => :balanced
```

値プロファイルは一歩踏み込んで、変数にある特定の値ばかり来ていないかを
見ます。典型は「ループ回数 `n` がいつも 3」「フラグ引数がいつも `true`」
「除数がいつも 8（＝シフトに置換できる）」といったケースです。支配的な値が
あれば、その値を**定数として焼き込んだ**特殊版を生成し、ガードで本物の値と
照合します。当たればループ展開や定数畳み込みが一気に効きます。

```ruby
# 値プロファイル：支配的な値があるか調べ、あれば投機の種にする
class ValueProfile
  def initialize = @hist = Hash.new(0)
  def record(v) = @hist[v] += 1

  def dominant(threshold = 0.9)        # 9 割以上を占める値があれば返す
    total = @hist.values.sum
    return nil if total == 0
    v, n = @hist.max_by { |_, c| c }
    n.to_f / total >= threshold ? v : nil
  end
end

vp = ValueProfile.new
100.times { vp.record(8) }; 5.times { vp.record(16) }
p vp.dominant   # => 8     除数 8 が支配的 → x/8 を x>>3 に投機できる
```

ただし値プロファイルは**当たり外れの差が大きい**手法です。当たれば破格に
効きますが、外れると毎回ガードに失敗して[脱最適化](deopt-osr.md)を繰り返し、
かえって遅くなります。だから処理系は、偏りが極端なときだけ（上の例の
9 割など）控えめに使います。

## 命令を書き換える特殊化適応型インタプリタ

機械語の JIT を持たなくても、観測した型でコードを差し替えるだけで、
かなりの高速化ができます。**バイトコード命令を、観測した型に特化した
別命令へ実行時に書き換える**のです。この「実行時に命令を高速版へ差し替える」
技法そのものは **quickening**（クイックニング）と呼ばれる古典的な手法で、
これを観測駆動の特殊化として徹底したものが**特殊化適応型インタプリタ**
（specializing adaptive interpreter）です。CPython 3.11 が採用しました
[](#cite:pep659)。汎用の `BINARY_OP` を、整数同士だと観測したら
`BINARY_OP_ADD_INT` へ差し替える、といった具合です。

この「命令が観測結果で自分自身を書き換える」仕掛けは、玩具で再現できます。

```ruby
# quickening：命令が観測した型で自分を特殊化版に書き換える
def execute(code, stack)
  pc = 0
  while (insn = code[pc])
    case insn[0]
    when :push then stack << insn[1]
    when :add                              # 汎用版：型を調べてから足す
      b, a = stack.pop, stack.pop
      code[pc] = [:add_int] if a.is_a?(Integer) && b.is_a?(Integer)
      stack << a + b                       # ↑ 観測した型で命令を書き換える
    when :add_int                          # 特殊化版：整数前提の高速経路
      b, a = stack.pop, stack.pop
      code[pc] = [:add] unless a.is_a?(Integer) && b.is_a?(Integer)
      stack << a + b                       # ↑ 賭けが外れたら汎用版へ戻す
    end
    pc += 1
  end
  stack.pop
end

code = [[:push, 1], [:push, 2], [:add]]
p execute(code, [])   # => 3
p code[2]             # => [:add_int]   命令が書き換わっている
```

実行後、`code[2]` が `[:add]` から `[:add_int]` に変わっています。
本物の特殊化版は、整数だと分かっているぶん型タグの検査や分岐を省き、
場合によってはアンボックスした生の整数で計算するので速くなります。
ここで見るべきは仕掛けのほうです。**命令列という配列が、プロファイルを
記録する可変のデータ構造を兼ねている**。インラインキャッシュと
同じ思想が、命令そのものに宿っています。この「特殊化を取り消して汎用へ
戻す」（`code[pc] = [:add]`）動きは、機械語 JIT における
[脱最適化](deopt-osr.md)のいちばん軽いミニチュアでもあります。

## プロファイルはいつまで信じられるか

最後に、プロファイリングの落とし穴を一つ。**プロファイルは過去の観測で
あって、未来の保証ではありません。** 「最初の 1000 回は整数だった」からと
いって、1001 回目も整数とは限りません。だからこそ JIT は観測をガード
付きの投機として使うのです。観測を信じて特殊化し、しかし念のため
ガードを置き、外れたら[脱最適化](deopt-osr.md)で戻る。

プログラムの「ふるまいの相」が途中で変わる（最初は整数を、途中から
浮動小数点数を扱い始める、など）と、JIT はせっかくコンパイルしたコードを
捨てて作り直すことになります。この**プロファイルの陳腐化**は、後で
[ウォームアップ問題](research-warmup.md)として研究テーマにもなります。

## まとめ

- プロファイリングは「**どこが熱いか**」と「**何を前提にできるか**」を、
  通常実行に溶け込ませて軽く測る技術。正確さと軽さの綱引き。
- 場所の検出：**呼び出しカウンタ**と**バックエッジカウンタ**（ループ用）。
  減算・**エイジング（半減）**・飽和・コンパイル後の除去で軽くする。
  全数えのカウンタと、セーフポイントでスタックトップを拾う**サンプリング**を、
  個別判定と俯瞰の役割で併用する。
- 型の観測：**インラインキャッシュ**が探索を省きつつ型を記録する。来る
  クラスが増えると**単態→多態（PIC）→メガモーフィック**と遷移し、
  4〜8 種類あたりで諦めてインライン化も断念する。
- 観測の集約：関数ごとの**フィードバックベクタ**に、呼び出し・二項演算・
  要素アクセス・分岐のスロットを並べ、JIT が読んで特殊化する。型に加え、
  **分岐確率**と**値プロファイル**も投機の材料になる。
- 機械語 JIT がなくても、**命令を観測型で特殊化版へ書き換える**だけで
  速くできる（CPython の特殊化適応型インタプリタ）。
- プロファイルは過去の観測にすぎず、**ガード付きの投機**として使う。

熱い場所と、そこで来る型が分かりました。次は、その情報をもっとも積極的に
使う方式、実行経路をまるごと記録してコンパイルする**トレーシング JIT**の、
記録と最適化の中身に踏み込みます。
