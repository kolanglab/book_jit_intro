# JIT フレームワーク：車輪を再発明しない

[最小の JIT](hello-jit.md)で、私たちは機械語のバイトを手で並べました。
`0x89, 0xF8` が「`mov eax, edi`」だと表を引きながら。あれは仕組みを
理解するには最高ですが、現実の処理系であれをやるのは正気の沙汰では
ありません。レジスタ割り当て、複数 CPU 対応、最適化…を全部自前で書くのは、
何年もかかる大事業です。

そこで、JIT の一部または全部を肩代わりしてくれる**フレームワーク**や
**ライブラリ**が存在します。この章では、それらを**抽象度の高さ**で
並べて地図にします。「自分の処理系には、どの高さの道具が合うか」を
判断できるようになるのが目標です。

## 道具を抽象度で並べる

JIT を支える道具は、低レベル（機械語に近い）から高レベル（言語に近い）
まで、おおよそ次の四層に分けられます。

| 層 | やってくれること | 自分で書くこと | 例 |
|----|------------------|----------------|-----|
| アセンブラ | 命令→バイト列の符号化 | 何の命令を出すか全部 | DynASM, AsmJit, GNU Lightning |
| 軽量コード生成器 | レジスタ割り当て・基本的最適化 | 中間表現の構築 | MIR, Cranelift, libgccjit |
| 最適化コンパイラ | SSA・本格最適化・多 CPU | IR への翻訳 | LLVM |
| 言語実装フレームワーク | JIT まるごと自動生成 | インタプリタを書くだけ | RPython, Truffle |

上に行くほど「楽」ですが、その分**重く**なり、生成の速さや制御の細かさが
犠牲になります。下に行くほど「自由」ですが、自分で書く量が増えます。
[階層的 JIT](tiered.md)の各段が違う道具を使うのは自然なことで、たとえば
ベースライン段はアセンブラ直叩き、最適化段は LLVM、という組み合わせも
あります。順に見ていきましょう。

## アセンブラ：符号化だけを任せる

いちばん低い層は、**命令を機械語のバイト列に符号化する**ところだけを
肩代わりするライブラリです。[最小の JIT](hello-jit.md)で手でやった
「`mov eax, edi` → `89 F8`」の表引きを、関数呼び出しでできるようにします。
レジスタ割り当ても最適化もしてくれません。**何の命令を出すかは、すべて
自分で決めます。**

代表が **DynASM**[](#cite:dynasm) です。LuaJIT の作者 Mike Pall が
LuaJIT のために作った、C のソースにアセンブリを埋め込めるプリプロセッサで、
[LuaJIT](case-luajit.md) のインタプリタと[ベースライン的なコード生成](case-luajit.md)を
支えています。考え方を Ruby の疑似コードで示すと、こんな雰囲気です。

```ruby
# アセンブラライブラリの「気分」（DynASM/AsmJit 風の疑似 API）
asm = Assembler.new
asm.mov(:eax, :edi)      # ライブラリが 0x89,0xF8 へ符号化してくれる
asm.add(:eax, :esi)      # 0x01,0xF0
asm.ret                  # 0xC3
code = asm.bytes         # 実行可能メモリに置けばそのまま動く
```

[最小の JIT](hello-jit.md)で書いた `compile_expr` を、バイト列リテラルでなく
こうしたメソッド呼び出しで書けるようにしたもの、と思えば近いです。
C++ 向けの **AsmJit** や、複数 CPU を一つの API で扱える
**GNU Lightning** も同じ層です。「投機もデオプトも全部自前で設計したい。
ただし符号化の退屈な作業だけは任せたい」という、細かい制御を求める
処理系に向きます。

## 軽量コード生成器：割り当てと軽い最適化まで

一段上がると、**中間表現を受け取り、レジスタ割り当てと基本的な最適化を
してくれる**コード生成器になります。あなたは「`t1 = a + b`」のような
[中間表現](ir.md)を組み立てて渡すだけ。物理レジスタへの割り当て
（[線形走査](ir.md)）や定数畳み込みは、ライブラリがやってくれます。
最適化は「ほどほど」ですが、その分**生成が速い**ので、JIT に向いています。

- **Cranelift**[](#cite:cranelift)：Rust 製。[WebAssembly](research-wasm.md)
  ランタイムの Wasmtime や、Rust 自身のデバッグビルド用バックエンドとして
  使われています。「速く生成すること」を最初から狙って設計されています。
- **MIR**[](#cite:makarov_mir)：C 製の小さな JIT エンジン。軽量・移植性・
  生成の速さを重視し、Ruby の MJIT 系の実験などで注目されました。
- **libgccjit**：GCC の最適化基盤を、ライブラリとして JIT から呼べるように
  したもの。GCC の枯れた最適化を借りられます。

この層は、「本格的な SSA 最適化までは要らないが、レジスタ割り当てを
手書きするのは嫌だ」という、ちょうど中間のニーズに応えます。生成の速さが
効くベースライン〜中位ティアに向きます。

## 最適化コンパイラ：LLVM

さらに上が、**本格的な最適化コンパイラ基盤**です。代表は **LLVM**
[](#cite:lattner2004)。LLVM は、SSA 形式の中間表現（LLVM IR）を受け取り、
数十種類の最適化パスをかけ、多数の CPU 向けに高品質な機械語を出します。
AOT コンパイラ（Clang など）の土台であると同時に、JIT としても使えます。

LLVM を JIT に使う魅力は、**最適化の品質**です。長年磨かれた最適化が
そのまま手に入ります。Julia や、初期の各種言語処理系が LLVM を JIT
バックエンドに選びました。

ただし弱点もあります。**LLVM は「速く生成すること」を主目的に設計されて
いない**ため、コンパイルに時間がかかります。[ウォームアップ](research-warmup.md)が
問題になるワークロードや、ベースライン段には重すぎます。だから LLVM を
使う処理系も、起動直後は軽い手段で動かし、本当に熱い場所だけを LLVM で
焼く、という[階層的](tiered.md)な使い方をします。

> [!NOTE]
> 「最適化が強い＝JIT に最適」ではない、というのが LLVM をめぐる教訓です。
> JIT では**コンパイル時間そのものが実行時間の一部**なので、最適化の
> 品質とコンパイルの速さのバランスが、AOT とは違う重みになります。
> この事情が、Cranelift や MIR のような「速く生成する」専用基盤が
> 生まれた背景です。

## 言語実装フレームワーク：JIT を書かずに手に入れる

最上層は、これまでとまったく発想が違います。中間表現すら組み立てません。
**対象言語のインタプリタを書くだけで、JIT がまるごと自動生成される**
フレームワークです。

- **RPython**：[メタトレーシング](meta-tracing.md)の章で見たとおり、
  RPython でインタプリタを書けば、トレーシング JIT が自動で付きます。
  [PyPy](case-python.md) の基盤。
- **Truffle/GraalVM**：[部分評価](research-partial-eval.md)により、AST
  インタプリタを書けばメソッド JIT が自動生成されます[](#cite:wurthinger2013)。
  Java・JavaScript・Ruby（TruffleRuby）・Python（GraalPy）など多言語の
  処理系がこの上に載っています。

この層の利点は圧倒的です。**JIT を一行も書かずに、最適化 JIT が手に入る。**
新しい言語を作る人にとって、これほど心強い道具はありません。代償は、
フレームワーク（RPython ツールチェインや GraalVM）という大きな土台に
乗ること、そして「フレームワークが得意とする書き方」に従う必要があること
です。この二つの道（メタトレーシングと部分評価）は第 II 部
[部分評価と Futamura 射影](research-partial-eval.md)で深掘りします。

## どの道具を選ぶか

選択は、処理系の目標と制約で決まります。判断の軸を整理します。

- **細かい制御が要るか**：投機やデオプトの設計を自分で握りたいなら、
  低い層（アセンブラ〜軽量生成器）。任せられるなら高い層。
- **生成の速さか、最適化の品質か**：起動の速さが命なら軽量生成器、
  長時間回るサーバで最高速度が命なら LLVM や Graal。多くは
  [階層的](tiered.md)に両方使う。
- **開発リソース**：少人数で新言語を作るなら、Truffle/RPython で JIT を
  丸ごと借りるのが現実的。大組織が既存処理系を磨くなら、自前のコード
  生成器（[YJIT](case-ruby.md) や [V8](case-v8.md) の各コンパイラ）を
  持つ。
- **既存資産との相性**：すでに C で書かれた処理系なら MIR や libgccjit が
  馴染みやすく、Rust なら Cranelift が自然。

> [!IMPORTANT]
> フレームワーク選びは、「どれが一番すごいか」ではなく「自分の処理系の
> **段ごとの要求**に、どの抽象度が合うか」で決まります。ベースライン段は
> アセンブラ直叩きで速く、最適化段は LLVM で高品質に、という**層をまたいだ
> 組み合わせ**こそ実戦的です。第 III 部のケーススタディで、実在の処理系が
> これらをどう組み合わせているかを観察してください。

## まとめ

- JIT の道具は抽象度で四層に並ぶ：**アセンブラ → 軽量コード生成器 →
  最適化コンパイラ → 言語実装フレームワーク**。
- **アセンブラ**（DynASM・AsmJit・GNU Lightning）は符号化だけ。細かい
  制御が要るベースライン段に。
- **軽量生成器**（Cranelift・MIR・libgccjit）はレジスタ割り当てと軽い
  最適化まで。生成の速さが武器。
- **LLVM** は最高品質の最適化。ただし重く、ウォームアップに不利。熱い
  場所専用や AOT 寄りの使い方に。
- **RPython・Truffle** はインタプリタを書くだけで JIT が自動生成される
  最上層。新言語に最適。
- 選択は「すごさ」でなく、各[ティア](tiered.md)の要求に合う抽象度で。

これで第 I 部は完結です。JIT を構成する技法と道具がそろいました。
ここから第 II 部では、これらの土台の上で進む**最新の研究**へと、
地図を広げていきます。
