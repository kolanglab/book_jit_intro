# JIT フレームワーク：車輪を再発明しない

[最小の JIT](hello-jit.md)で、私たちは機械語のバイトを手で並べました。
`0x89, 0xF8` が「`mov eax, edi`」だと表を引きながら。あれは仕組みを
理解するには最高ですが、現実の処理系であれをやるのは正気の沙汰では
ありません。レジスタ割り当て、複数 CPU 対応、最適化…を全部自前で書くのは、
何年もかかる大事業です。

そこで、JIT の一部または全部を肩代わりしてくれる**フレームワーク**や
**ライブラリ**が存在します。この章では、それらを**抽象度の高さ**で
並べて地図にします。「自分の処理系には、どの高さの道具が合うか」を
判断できるようになるのが目標です。

## 道具を抽象度で並べる

JIT を支える道具は、低レベル（機械語に近い）から高レベル（言語に近い）
まで、おおよそ次の四層に分けられます。

| 層 | やってくれること | 自分で書くこと | 例 |
|----|------------------|----------------|-----|
| アセンブラ | 命令→バイト列の符号化 | 何の命令を出すか全部 | DynASM, AsmJit, Xbyak, GNU Lightning |
| 軽量コード生成器 | レジスタ割り当て・基本的最適化 | 中間表現の構築 | MIR, Cranelift, libgccjit |
| 最適化コンパイラ | SSA・本格最適化・多 CPU | IR への翻訳 | LLVM |
| 言語実装フレームワーク | JIT まるごと自動生成 | インタプリタを書くだけ | RPython, Truffle |

上に行くほど「楽」ですが、その分**重く**なり、生成の速さや制御の細かさが
犠牲になります。下に行くほど「自由」ですが、自分で書く量が増えます。
[階層的 JIT](tiered.md)の各段が違う道具を使うのは自然なことで、たとえば
ベースライン段はアセンブラ直叩き、最適化段は LLVM、という組み合わせも
あります。順に見ていきましょう。

## アセンブラ：符号化だけを任せる

いちばん低い層は、**命令を機械語のバイト列に符号化する**ところだけを
肩代わりするライブラリです。[最小の JIT](hello-jit.md)で手でやった
「`mov eax, edi` → `89 F8`」の表引きを、関数呼び出しでできるようにします。
レジスタ割り当ても最適化もしてくれません。**何の命令を出すかは、すべて
自分で決めます。**

代表が **DynASM**[](#cite:dynasm) です。LuaJIT の作者 Mike Pall が
LuaJIT のために作った、C のソースにアセンブリを埋め込めるプリプロセッサで、
[LuaJIT](case-luajit.md) の**手書きアセンブリによるインタプリタ**の生成を
支えています（トレースを機械語に落とす JIT バックエンドのほうは DynASM では
ない、という役割分担は第 III 部の [LuaJIT の章](case-luajit.md)で見ます）。考え方を Ruby の疑似コードで示すと、こんな雰囲気です。

```ruby
# アセンブラライブラリの「気分」（DynASM/AsmJit 風の疑似 API）
asm = Assembler.new
asm.mov(:eax, :edi)      # ライブラリが 0x89,0xF8 へ符号化してくれる
asm.add(:eax, :esi)      # 0x01,0xF0
asm.ret                  # 0xC3
code = asm.bytes         # 実行可能メモリに置けばそのまま動く
```

[最小の JIT](hello-jit.md)で書いた `compile_expr` を、バイト列リテラルでなく
こうしたメソッド呼び出しで書けるようにしたもの、と思えば近いです。

### DynASM の書き味：C にアセンブリを埋め込む

DynASM の特徴は、API というより**記法**にあります。`.dasc` という拡張子の
ファイルに C を書き、その中に行頭が `|`（パイプ）で始まる行としてアセンブリを
直接書きます。ビルド前に Lua 製のプリプロセッサがその `.dasc` を通常の C に
変換します。雰囲気を疑似的に示すと、こうです。

```c
// add.dasc 風の疑似コード。| で始まる行がアセンブリ
| .arch x64                 // 対象アーキテクチャを宣言
| mov eax, edi              // 第 1 引数を eax へ
| add eax, esi              // 第 2 引数を足し込む
| ret
```

ここで効いてくるのが、**C の式をアセンブリに埋め込める**点です。たとえば
`| mov rax, [rbx + Dst->field_offset]` のように、オフセットを実行時の
C 変数で与えられます。つまり「どの命令を出すか」をコード生成時（実行時）に C の
ロジックで分岐させながら、命令そのものは読みやすいアセンブリで書ける、という
両取りができます。インタプリタのオペコードごとに対応する機械語片を吐く、
という[ベースライン](tiered.md)的なコード生成と、とても相性のよい書き味です。

### AsmJit の書き味：ビルダ API とラベル

C++ 向けの **AsmJit** は、プリプロセッサではなく**ビルダ API** でこの層を
提供します。アセンブラオブジェクトのメソッドを呼ぶたびに命令が積まれます。
DynASM が「埋め込み記法」なら、AsmJit は「普通の関数呼び出し」です。

```ruby
# AsmJit 風ビルダ API の疑似コード
a = Assembler.new
loop_top = a.new_label        # ラベルを宣言（まだ位置は未定）
done     = a.new_label

a.test(:ecx, :ecx)
a.jz(done)                    # ecx==0 なら done へ。飛び先はまだ未確定
a.bind(loop_top)              # ここが loop_top の位置だと確定させる
a.add(:eax, :edx)
a.dec(:ecx)
a.jnz(loop_top)               # 後方ジャンプ：飛び先はすでに確定済み
a.bind(done)
code = a.finalize             # ここで全ラベルの実アドレスへ解決
```

ここで現れるのが、アセンブラ層で必ず出くわす**ラベルとバックパッチ**の問題です。
`jz(done)` を出した時点では `done` がコードのどこに来るか分かりません。そこで
ライブラリはいったん飛び先を空けて命令を出し、`bind(done)` で位置が確定した
あと、**後から空けた箇所を正しいオフセットで埋め直します**。この「後から
埋め直す」処理がバックパッチです。[最小の JIT](hello-jit.md)で前方ジャンプを
手で書こうとすると、オフセットを自分で計算して該当バイトに書き戻す羽目に
なりますが、その面倒をラベル機構が肩代わりしてくれる、というのがこの層の
ありがたみの一段深いところです。

> [!TIP]
> ラベル解決には、後方ジャンプ（飛び先が既知）と前方ジャンプ（飛び先が未定）の
> 二種類があります。前者はその場で確定できますが、後者は**未解決参照のリスト**を
> ラベルごとに持っておき、`bind` 時にまとめて埋めます。さらに x86 では
> 「近距離なら 1 バイト変位、遠ければ 4 バイト」と命令長が変わるため、距離が
> 確定するまで命令サイズが決まらないという厄介さもあります。素朴な実装は
> 安全側に倒して常に 4 バイト変位を使います。

### Xbyak の書き味：C++ に完全に内蔵する

同じビルダ流でも、**Xbyak**[](#cite:xbyak) はさらに徹底しています。
光成滋生氏による C++ の**ヘッダオンリー**ライブラリで、ヘッダを
include するだけで、`mov(eax, edi)` のようにニーモニックをそのまま
メソッドとして呼べます。呼んだそばから実行時にバイト列へ符号化され、
できあがったコードはその場で関数として呼び出せます。プリプロセッサも
ビルド手順の追加も要らず、AVX-512 やその後の新しい命令拡張への対応が
早いことでも知られます。

面白いのは、Xbyak の主戦場が言語処理系では**ない**ことです。Intel の
深層学習ライブラリ **oneDNN** は、実行時に CPU の SIMD 拡張を判定して、
行列積や畳み込みのカーネルを Xbyak で JIT 生成します。富士通も
SVE 命令に完全対応した AArch64 版（Xbyak_aarch64）を開発しており、
oneDNN の AArch64 対応を支えています。「実行時に分かる情報（この CPU で
使える命令）に合わせてコードを焼く」という JIT の発想が、
[言語の外](beyond-languages.md)でも同じ道具で生きている好例です。

### GNU Lightning：複数 CPU を一つの API で

複数 CPU を一つの API で扱える **GNU Lightning** も同じ層です。Lightning は
実在の CPU より少し抽象化した「仮想的なレジスタマシン」を提供し、それを各
アーキテクチャの実命令へ落とします。同じコード生成コードを x86・ARM 両対応に
できる代わりに、CPU 固有の小技は使いにくくなる、というトレードオフです。

この層全体は、「投機も脱最適化も全部自前で設計したい。ただし符号化の退屈な
作業だけは任せたい」という、細かい制御を求める処理系に向きます。レジスタ
割り当てを自分で決められるので、たとえば[トレーシング](tracing.md)で
レジスタの中身を固定したい、といった攻めた最適化も思いどおりにできます。
反面、レジスタが枯渇したときのスピル（退避）も自分で書く必要があり、複雑な
コードを生成するほど負担が増えます。

## 軽量コード生成器：割り当てと軽い最適化まで

一段上がると、**中間表現を受け取り、レジスタ割り当てと基本的な最適化を
してくれる**コード生成器になります。あなたは「`t1 = a + b`」のような
[中間表現](ir.md)を組み立てて渡すだけ。物理レジスタへの割り当て
（[線形走査](ir.md)）や定数畳み込みは、ライブラリがやってくれます。
最適化は「ほどほど」ですが、その分**生成が速い**ので、JIT に向いています。

- **Cranelift**[](#cite:cranelift)：Rust 製。[WebAssembly](research-wasm.md)
  ランタイムの Wasmtime で本番採用されており、デバッグビルドを速くする Rust の
  代替バックエンドとしても使われています。「速く生成すること」を最初から
  狙って設計されています。
- **MIR**[](#cite:makarov_mir)：C 製の小さな JIT エンジン。軽量・移植性・
  生成の速さを重視し、Ruby 向けの JIT バックエンドの実験（Makarov による）
  などで注目されました。
- **libgccjit**：GCC の最適化基盤を、ライブラリとして JIT から呼べるように
  したもの。GCC の枯れた最適化を借りられます。

### Cranelift IR を組み立てて compile() するまで

この層の使い方を、Cranelift を例に「関数を一つ作って機械語にする」流れで
見てみましょう。Cranelift は **Cranelift IR（CLIF）** という SSA 形式の
[中間表現](ir.md)を持ちます。値には `v0`, `v1`… という番号が付き、
一度代入したら書き換えません。`a + b` を返す関数を組み立てる雰囲気は、
こんな疑似コードになります。

```ruby
# Cranelift 風：IR を組み立てて compile する疑似コード
sig = Signature.new(params: [I64, I64], returns: [I64])
fb  = FunctionBuilder.new(sig)

block = fb.create_block               # 基本ブロック
fb.append_block_params(block, sig.params)
fb.switch_to_block(block)

a = fb.block_param(block, 0)          # => v0
b = fb.block_param(block, 1)          # => v1
sum = fb.ins.iadd(a, b)               # => v2 = iadd v0, v1
fb.ins.return_(sum)

# 出来上がる IR の見た目（概念図）
#   function %add(i64, i64) -> i64 {
#   block0(v0: i64, v1: i64):
#       v2 = iadd v0, v1
#       return v2
#   }

code = fb.compile(target: :x86_64)    # ここでレジスタ割り当て＋符号化
```

ポイントは、あなたが書くのは `iadd v0, v1` という**抽象的な演算**までで、
「`v0` を `rdi`、`v1` を `rsi` に割り当てて `add` を出す」という物理レジスタの
話には立ち入らない点です。アセンブラ層では自分で握っていたレジスタ割り当てを、
`compile()` の内側がやってくれます。

### なぜ「速く生成」できるのか

ではなぜこの層は LLVM より速くコードを吐けるのでしょうか。答えは、**やらない
ことを最初から決めている**からです。具体的には、

- **レジスタ割り当てに軽量なアルゴリズムを使う**：たとえば[線形走査](ir.md)
  [](#cite:poletto1999)なら、変数の生存区間を一度なめるだけで割り当てを
  決められます。グラフ彩色系の重い割り当てより品質は落ちますが、桁違いに
  速いです（なお現在の Cranelift の割り当て器 regalloc2 は、線形走査ではなく
  もう少し手の込んだバックトラック式で、速度と品質のバランスを取っています）。
- **最適化パスを意図的に絞る**：定数畳み込みや単純な不要コード削除など、
  費用対効果の高い軽い最適化だけをかけ、ループ変換のような重い解析は
  行いません。
- **IR を一度しか走査しない設計**：パスを何十回も回す代わりに、構築から
  符号化までをほぼ一直線に流します。

これは[コンパイル単位](compilation-units.md)を「そこそこの速さで何度も焼く」
JIT の現場に効きます。生成の速さが効く[ベースライン〜中位ティア](tiered.md)に
うってつけ、というわけです。

### MIR と libgccjit の使い所

**MIR** はこの思想をさらに小さくまとめた C 製エンジンで、IR の構築から
コード生成までを軽い依存関係で完結させます。**処理系自身が C で書かれていて、
外部に巨大な依存を持ち込みたくない**ときに馴染みます。

**libgccjit** はやや毛色が違い、GCC 本体の最適化基盤を借りる窓口です。
生成は MIR や Cranelift ほど軽くありませんが、**GCC の枯れた最適化を
そのまま使える**のが強みです。「自前の SSA を書く気はないが、ある程度
品質のよいコードがほしい」という中位ティア向きの選択肢になります。
言語 VM 以外での採用例もあり、GNU Emacs はバージョン 28 から、Emacs Lisp の
バイトコードを libgccjit でネイティブコードへコンパイルする機能
（native compilation）を備えています。コンパイルはバックグラウンドで
逐次進み、結果は `.eln` ファイルとして保存されて次回以降も再利用される ——
エディタの拡張言語にまで「GCC の最適化を借りる」道具が届いている実例です。

> [!NOTE]
> 同じ「軽量コード生成器」でも、Cranelift は速度寄り、libgccjit は品質寄り、と
> 立ち位置が少しずつ違います。選ぶときは「自分の処理系がどの言語で書かれて
> いるか」（C なら MIR/libgccjit、Rust なら Cranelift）と「[ティア](tiered.md)の
> どの段に置くか」の二点で当たりを付けると、候補がすぐ絞れます。

この層は全体として、「本格的な SSA 最適化までは要らないが、レジスタ割り当てを
手書きするのは嫌だ」という、ちょうど中間のニーズに応えます。

## 最適化コンパイラ：LLVM

さらに上が、**本格的な最適化コンパイラ基盤**です。代表は **LLVM**
[](#cite:lattner2004)。LLVM は、SSA 形式の中間表現（LLVM IR）を受け取り、
数十種類の最適化パスをかけ、多数の CPU 向けに高品質な機械語を出します。
AOT コンパイラ（Clang など）の土台であると同時に、JIT としても使えます。

LLVM を JIT に使う魅力は、**最適化の品質**です。長年磨かれた最適化が
そのまま手に入ります。Julia や、初期の各種言語処理系が LLVM を JIT
バックエンドに選びました。

### LLVM IR を組み立て、パスをかけ、機械語にする

LLVM を JIT として使う流れは、おおむね「IR を組み立てる → 最適化パスを
かける → 機械語へ落とす → 実行可能メモリに載せて呼ぶ」の四段です。IR の
組み立ては、軽量生成器とよく似た**ビルダ API** で行います。

```ruby
# LLVM 風：IR ビルダで a + b を返す関数を組み立てる疑似コード
mod = Module.new("jit")
fn  = mod.add_function("add", ret: I64, args: [I64, I64])
bb  = fn.append_block("entry")
b   = Builder.new(at: bb)

sum = b.add(fn.arg(0), fn.arg(1))     # %sum = add i64 %0, %1
b.ret(sum)

# 出来上がる LLVM IR の見た目（概念図）
#   define i64 @add(i64 %0, i64 %1) {
#   entry:
#     %sum = add i64 %0, %1
#     ret i64 %sum
#   }

run_optimization_passes(mod, level: :O2)   # ここで数十のパスが走る
engine = ORCJIT.new(mod)                   # 機械語へコンパイルして載せる
ptr    = engine.address_of("add")          # 関数ポインタを取得して呼ぶ
```

ここで使うエンジンが **ORC JIT**（および旧来の **MCJIT**）です。MCJIT は
モジュール単位でまとめてコンパイルする素朴な仕組みで、ORC JIT はそれを
**遅延・並行・段階的**に拡張したものです。ORC では「シンボルを初めて呼んだ
ときに初めてコンパイルする」遅延コンパイルや、コンパイルを別スレッドに
逃がす、といった JIT 向けの制御がしやすくなっています。

### なぜコンパイルが重いのか

`run_optimization_passes` の一行が曲者です。`-O2` 相当では、定数伝播・
共通部分式除去・ループ不変式の巻き上げ・インライン展開・命令スケジューリング…と
**数十のパスが IR を何度も走査**します。AOT なら一度払えば済むこの費用が、
JIT では[実行時間の一部](research-warmup.md)としてのしかかります。
軽量生成器が[線形走査](ir.md)で一回なめて終わらせるのと対照的に、LLVM は
品質のために何度も回す設計だ、というのが重さの正体です。これは構造的な
性質で、**LLVM は「速く生成すること」を主目的に設計されていません**。
[ウォームアップ](research-warmup.md)が問題になるワークロードや、ベースライン段には
重すぎます。だから LLVM を使う処理系も、起動直後は軽い手段で動かし、本当に
熱い場所だけを LLVM で焼く、という[階層的](tiered.md)な使い方をします。

### JIT 特有の壁：脱最適化の状態をどう取り出すか

LLVM を JIT に使うとき、もう一つ避けて通れない壁があります。JIT は
[投機](why-jit.md)をします。「この変数はいつも整数だろう」と賭けて速い
コードを焼き、賭けが外れたら[脱最適化](deopt-osr.md)してインタプリタへ
戻ります。このとき問題になるのが、**最適化済みコードの途中の状態を、
インタプリタが理解できる形で取り出せるか**です。

ところが LLVM は最適化の過程で、変数をレジスタに散らし、不要な計算を消し、
命令を並べ替えます。すると「いま `x` の値はどのレジスタにあるのか」が
コンパイル後には自明でなくなります。素朴に書くと、最適化が[脱最適化](deopt-osr.md)に
必要な情報を消してしまうのです。

これを解くのが **スタックマップ**と **patchpoint** という仕掛けです。
考え方はこうです。脱最適化があり得る地点に印を置き、「この地点では、
インタプリタ側の変数 `x` は機械語のこのレジスタ（またはスタックのこの位置）に
ある」という対応表を LLVM に出力させます。これがスタックマップです。

```ruby
# 脱最適化地点に印を置く疑似コード（patchpoint/stackmap 風）
guard = b.icmp_eq(tag_of(x), INT_TAG)        # x は整数か？
b.cond_br(guard, fast_path, deopt)

b.at(deopt)
# 「ここで戻るなら、論理変数 x の値はこの SSA 値だ」と登録する。
# LLVM はこの x がどのレジスタ/スタック位置にあるかをスタックマップに記録し、
# その情報を最適化で消さないよう生かし続ける。
b.stackmap(id: 42, live_values: [x, y, frame_state])
b.deoptimize_to_interpreter(id: 42)
```

実行時に賭けが外れたら、ランタイムは id 42 のスタックマップを引き、機械語の
レジスタやスタックから `x`・`y` の現在値を吸い出して、インタプリタのフレームを
組み直します。**patchpoint** は、この印のために機械語へ「あとから書き換える
ための空き地（パッチ用の領域）」を確保しておく仕組みで、ガードのチェックや
[脱最適化](deopt-osr.md)への分岐を後から差し込めるようにします。
なお LLVM には、同じ発想を [GC との協調](gc-runtime.md)へ向けた
`gc.statepoint` という機構もあります。セーフポイントで「どのレジスタ／
スタック位置にオブジェクト参照が生きているか」を記録させるもので、
脱最適化用のスタックマップと対をなす仕掛けです。
LLVM が JIT 用にこうした機構を持っているからこそ、最適化の品質を保ったまま
投機と[脱最適化](deopt-osr.md)を両立できる、というわけです。

> [!IMPORTANT]
> 投機を行う JIT で LLVM を採用するとき、最大の設計課題は最適化の品質では
> なく、**脱最適化の状態をどう保全するか**です。スタックマップや patchpoint を
> 使わずに素朴に最適化をかけると、賭けが外れたときにインタプリタへ正しく
> 戻れません。「強い最適化器を借りる」ことと「[脱最適化](deopt-osr.md)を
> 安全に保つ」ことは別の仕事だ、と意識してください。

この「品質は高いが生成が重い」というトレードオフを実地で天秤にかけ、
LLVM を**捨てる**判断を下した実例もあります。第 III 部で見る
[JSC の最上位ティア FTL](case-others.md#FTL：B3 と、LLVM を捨てた判断) は、
当初 LLVM をバックエンドに使い、のちに生成速度のため自前の B3 へ差し替えました。

> [!NOTE]
> 「最適化が強い＝JIT に最適」ではない、というのが LLVM をめぐる教訓です。
> JIT では**コンパイル時間そのものが実行時間の一部**なので、最適化の
> 品質とコンパイルの速さのバランスが、AOT とは違う重みになります。
> この事情が、Cranelift や MIR のような「速く生成する」専用基盤が
> 生まれた背景です。

## 言語実装フレームワーク：JIT を書かずに手に入れる

最上層は、これまでとまったく発想が違います。中間表現すら組み立てません。
**対象言語のインタプリタを書くだけで、JIT がまるごと自動生成される**
フレームワークです。

- **RPython**：[メタトレーシング](meta-tracing.md)の章で見たとおり、
  RPython でインタプリタを書けば、トレーシング JIT が自動で付きます。
  [PyPy](case-python.md) の基盤。
- **Truffle/GraalVM**：[部分評価](research-partial-eval.md)により、AST
  インタプリタを書けばメソッド JIT が自動生成されます[](#cite:wurthinger2013)。
  Java・JavaScript・Ruby（TruffleRuby）・Python（GraalPy）など多言語の
  処理系がこの上に載っています。

ただし「インタプリタを書くだけ」と言っても、JIT に効く情報をフレームワークへ
渡す**ヒント**を、要所に置く必要があります。ここがこの層の実装の勘所です。

### RPython のヒント配置

RPython では、インタプリタの命令ディスパッチのループに「ここがトレーシングの
起点だ」と教えるヒントを置きます。雰囲気はこうです。

```ruby
# RPython 風：メタトレーシングのためのヒント配置（疑似コード）
jitdriver = JitDriver.new(
  greens: [:pc, :bytecode],   # この組が同じなら「同じループ」と見なす鍵
  reds:   [:frame, :stack],   # 実行ごとに変わる値
)

loop do
  jitdriver.jit_merge_point(pc:, bytecode:, frame:, stack:)
  op = bytecode[pc]
  # ... 各命令の処理 ...
  if back_edge?(op)           # ループの戻り辺なら
    jitdriver.can_enter_jit(pc:, bytecode:, frame:, stack:)
  end
end
```

`greens` と `reds` の区別が肝です。**`greens`（緑）は「同じならコードを
共有してよい」定数的な値**（プログラムカウンタやバイトコード本体）、
**`reds`（赤）は実行のたびに変わる値**です。フレームワークはこのヒントを
頼りに、どこをループの起点と見なし、何を定数化してトレースを焼くかを決めます。
インタプリタの本体は普通に書きつつ、要所にこの数行を置くだけで、
[メタトレーシング](meta-tracing.md)が回り始めるわけです。

### Truffle の @Specialization と Assumption

Truffle は AST インタプリタの各ノードに**注釈**を付けて JIT に情報を渡します。
代表が `@Specialization` です。「この加算ノードは、両辺が整数のときはこの
実装、文字列のときはこの実装」と型ごとの分岐を宣言で書くと、フレームワークが
実行時に観測した型に**特殊化**したノードへ自動で置き換えます。

```ruby
# Truffle 風：型ごとの特殊化を宣言する疑似コード
class AddNode
  specialization(when: ->(l, r) { l.int? && r.int? }) do |l, r|
    l + r                       # 整数同士の速いパス
  end
  specialization(when: ->(l, r) { l.string? && r.string? }) do |l, r|
    l.concat(r)                 # 文字列連結のパス
  end
  # どれにも当たらなければ汎用パスへ落ちる（＝脱最適化に相当）
end
```

もう一つの要が `Assumption` です。これは「この前提が成り立つ限り、この速い
コードは有効」という**取り消し可能な賭け**を表します。たとえば「このメソッドは
再定義されていない」を `Assumption` にしておき、再定義が起きたら
`assumption.invalidate` で無効化します。すると、その前提に依存して焼いた
コードが一斉に[脱最適化](deopt-osr.md)されます。`@Specialization` が型に
基づく投機、`Assumption` が大域的な前提に基づく投機、と捉えると、両者の
役割分担が見えます。背後では[部分評価](research-partial-eval.md)が、
特殊化されたノード木を一段の機械語へ畳み込んでいます[](#cite:wurthinger2017)。

> [!TIP]
> 注釈やヒントは「魔法の呪文」ではありません。`@Specialization` は
> [インラインキャッシュ](why-jit.md)を宣言的に書いたもの、`Assumption` は
> ガードの大域版、RPython の `jit_merge_point` はトレース起点の指定、と
> 既習の概念に対応づけて読むと、各フレームワークの作法が一気に見通せます。

この層の利点は圧倒的です。**JIT を一行も書かずに、最適化 JIT が手に入る。**
新しい言語を作る人にとって、これほど心強い道具はありません。代償は、
フレームワーク（RPython ツールチェインや GraalVM）という大きな土台に
乗ること、そして「フレームワークが得意とする書き方」に従う必要があること
です。この二つの道のうち、メタトレーシングの仕組みは
[メタトレーシング](meta-tracing.md)で見たとおりです。部分評価の側は
第 II 部の[部分評価と Futamura 射影](research-partial-eval.md)で深掘りします。

## どの道具を選ぶか

選択は、処理系の目標と制約で決まります。判断の軸を整理します。

- **細かい制御が要るか**：投機や脱最適化の設計を自分で握りたいなら、
  低い層（アセンブラ〜軽量生成器）。任せられるなら高い層。
- **生成の速さか、最適化の品質か**：起動の速さが命なら軽量生成器、
  長時間回るサーバで最高速度が命なら LLVM や Graal。多くは
  [階層的](tiered.md)に両方使う。
- **開発リソース**：少人数で新言語を作るなら、Truffle/RPython で JIT を
  丸ごと借りるのが現実的。大組織が既存処理系を磨くなら、自前のコード
  生成器（[YJIT](case-ruby.md) や [V8](case-v8.md) の各コンパイラ）を
  持つ。
- **既存資産との相性**：すでに C で書かれた処理系なら MIR や libgccjit が
  馴染みやすく、Rust なら Cranelift が自然。

> [!IMPORTANT]
> フレームワーク選びは、「どれが一番すごいか」ではなく「自分の処理系の
> **段ごとの要求**に、どの抽象度が合うか」で決まります。ベースライン段は
> アセンブラ直叩きで速く、最適化段は LLVM で高品質に、という**層をまたいだ
> 組み合わせ**こそ実戦的です。第 III 部のケーススタディで、実在の処理系が
> これらをどう組み合わせているかを観察してください。

実在の処理系が主にどの道具を選んでいるかを、早見表にしておきます。

| 処理系 | 主に使う道具 | 抽象度の層 |
|---|---|---|
| [YJIT](case-ruby.md)（Ruby） | 自前アセンブラ | アセンブラ |
| [ZJIT](case-ruby.md)（Ruby 4.0） | 自前の SSA IR とコード生成（Rust 製） | 自前の最適化コンパイラ |
| [TruffleRuby](case-ruby.md) | Truffle/Graal | 言語実装フレームワーク |
| [PyPy](case-python.md) | RPython | 言語実装フレームワーク |
| Julia | LLVM（ORC JIT） | 最適化コンパイラ |
| Wasmtime | Cranelift | 軽量コード生成器 |

同じ言語でも実装が違えば選ぶ道具も変わる（Ruby だけでも YJIT・ZJIT・
TruffleRuby の三様）ことに注目してください。詳しくは第 III 部の各ケース
スタディで扱います。

抽象的に「組み合わせる」と言われてもピンと来ないので、**層をまたぐ
組み合わせの具体例**を一つ。架空の処理系「MyLang」を例に、二つの段で
別の道具を使う設計を具体化してみます。

```ruby
# MyLang の二段構成（疑似コード）
def on_function_hot(fn, hotness)
  if hotness == :warm
    # ベースライン段：アセンブラ層を直叩きして「速く」焼く
    asm = Assembler.new
    fn.each_bytecode { |op| emit_baseline(asm, op) }  # 命令ごとに機械語片を吐く
    install(fn, asm.finalize)                          # 即座に差し替え
  elsif hotness == :hot
    # 最適化段：自前 SSA を組み立てて品質重視で焼く
    ir = build_ssa(fn)                # 中間表現へ
    optimize(ir)                      # 投機点にガードとスタックマップを仕込む
    code = lower_to_machine(ir)       # 機械語へ
    install_with_osr(fn, code)        # 実行中のフレームを乗り換える（OSR）
  end
end
```

ここで**ベースライン段は[アセンブラ層](hello-jit.md)を直叩き**して、命令ごとに
定型の機械語片を吐くだけにします。最適化はほぼせず、とにかく速く install して
インタプリタから卒業させます。一方**最適化段は LLVM や自前 SSA** を使い、
時間をかけて品質の高いコードを焼き、賭けが外れたら[脱最適化](deopt-osr.md)で
ベースラインへ降ります。同じ関数でも、温度が上がるにつれ使う道具を切り替えて
いくわけです。この段ごとの道具の使い分けが、[階層的 JIT](tiered.md)の実装
そのものです。[YJIT](case-ruby.md) や [V8](case-v8.md) の各段は、まさに
こうした「段ごとに違う抽象度」を採用しています。

## どの層でも必要になる横断的な実装事項

ここまで層ごとの違いを見てきましたが、**どの層を選んでも避けて通れない**
共通の実装事項があります。生成した機械語を「実際に CPU に実行させる」ための
泥臭い作業で、フレームワークが肩代わりしてくれる範囲もまちまちです。
[最小の JIT](hello-jit.md)では深入りしなかったここを、まとめて押さえます。

### W^X：誰が面倒を見るか

生成した機械語は、いったんメモリに**書いてから実行**します。前章の
[JIT とセキュリティ](security.md#wx書けるなら実行できない実行できるなら書けない)で見た
**W^X** の作法 —— RW で書いてから RX へ切り替える、Apple Silicon では
スレッド単位の切り替え API を使う、ARM 系では書き込み後に命令キャッシュと
データキャッシュの整合を取る —— は、どの層の道具を選んでも**誰かが必ず**
やらねばなりません。層によって違うのは、**それを誰がやるか**です。
アセンブラ層や Cranelift は機械語のバイト列（と後述の再配置情報）を返す
だけで、実行可能メモリの確保・権限の切り替え・キャッシュ同期は**呼び出し側
（あなた）の仕事**です（LLVM の ORC JIT には既定のメモリ管理が付属しますが、
本番の処理系では自前の管理に差し替えるのが普通です）。一方、Truffle や
RPython のような言語実装フレームワークは、コードの設置から保護までを
**まるごと隠して**くれます。低い層の自由は、こうした安全の作法まで自分の
責任範囲になることと引き換えなのです。

### 再配置：絶対アドレスを後から埋める

生成した機械語は、ランタイム関数の呼び出しや、別の[コンパイル単位](compilation-units.md)への
ジャンプを含みます。ところが「呼び先のアドレス」は、コードを置くメモリ位置が
決まるまで確定しません。そこで、アドレスを空けて命令を出しておき、**配置先が
決まってから正しいアドレスを埋める**作業が要ります。これが再配置（relocation）です。

```ruby
# 再配置の疑似フロー
asm.call(placeholder: :gc_alloc)    # 呼び先を空けて call を出す
# ... コードを実行可能メモリへ配置し、最終アドレスが決まる ...
code.relocations.each do |r|
  patch(code, r.offset, resolve(r.target))  # 空けた箇所に実アドレスを書く
end
```

これはアセンブラ層で見た**ラベルとバックパッチ**（同一コード内の前方ジャンプ）の、
**コードをまたぐ版**だと捉えると分かりやすいです。LLVM や Cranelift は
この再配置情報を生成して返してくれるので、ランタイムはそれを見て埋めるだけで
済みます。アセンブラ層を直叩きするなら、自分で管理する必要があります。

### call が届く距離：コードを「近く」に置く

再配置で埋めるアドレスには、もう一つ落とし穴があります。**呼び出し命令の
射程**です。[最小の JIT](hello-jit.md)で見たとおり、x86-64 の `call`/`jmp` の
標準形は飛び先を **rel32**（±2 GiB の相対距離）で持つ、わずか 5 バイトの
命令です。ところが 64 ビットの広大なアドレス空間では、`mmap` が返すコード
領域と、ホストバイナリ（ランタイムの C 関数たち）が 2 GiB を超えて離れる
ことが普通に起きます。すると `call rel32` 一発では届かず、

```
movabs rax, 0x7f3a12345678   ; 64 ビット絶対アドレスをロード（10 バイト）
call   rax                   ; レジスタ経由で呼ぶ（2 バイト）
```

のように**一回り長く、レジスタを一本潰す**形になります。JIT コードはガードの
脱出やオブジェクトの確保のたびにランタイムを呼ぶので、この差は積もります。

だから実装は、コード領域を**意図的に「近く」へ確保**します。`mmap` に
ヒントアドレスを渡して自分のバイナリの近傍を狙う、LuaJIT のように下位
2 GiB に確保してしまう、HotSpot のようにコードキャッシュを**一つの連続領域
として予約**し、少なくとも JIT コード同士は常に rel32 で呼び合えるように
する、という具合です。V8 には、よく呼ぶ組み込み関数のコードを JIT コード
領域のすぐ隣へコピーして近距離 call で呼べるようにする最適化
（short builtin calls）まであります。それでも届かない呼び先には、射程内に
置いた**踏み台（トランポリン）**を経由させます——リンカが遠距離分岐に挟む
veneer と同じ発想です。AArch64 は `bl` の射程が **±128 MiB** とさらに
短いので、この設計はいっそう切実になります。

### コードキャッシュ管理：焼いたコードを置く場所

焼いた機械語をどこに置き、いつ捨てるか、という管理も横断的な課題です。
JIT は実行中ずっとコードを生成し続けるので、置き場所（コードキャッシュ）は
**有限の資源**です（これを「コードの GC」として捉える見方は
[JIT とランタイムの協調](gc-runtime.md)章で扱いました）。

- **断片化への対処**：[脱最適化](deopt-osr.md)で不要になったコードや、
  再コンパイルで置き換えられた古いコードが穴を空けます。詰め直す（コンパクション）か、
  あらかじめ固定サイズの区画に切って管理します。
- **いっぱいになったとき**：これ以上焼けないなら、古いコードを捨てて
  インタプリタへ戻すか、新規コンパイルを止めます。捨てる際は、そのコードを
  実行中のスレッドがいないことを保証してから解放しなければなりません。
- **メタデータの随伴**：各コードには、再配置情報・スタックマップ・
  [脱最適化](deopt-osr.md)用の対応表が付き従います。コードを捨てるときは、
  これらも一緒に管理・解放します。

> [!NOTE]
> W^X・再配置・call の射程・キャッシュ管理は、どの抽象度の道具を選んでも顔を出します。
> 高い層（LLVM や Truffle）ほどフレームワークが肩代わりしてくれますが、
> [アセンブラ層](hello-jit.md)を直叩きするほど自分の責任範囲が広がります。
> 「どこまで任せ、どこから自分でやるか」を見極めることも、道具選びの一部です。

## まとめ

- JIT の道具は抽象度で四層に並ぶ：**アセンブラ → 軽量コード生成器 →
  最適化コンパイラ → 言語実装フレームワーク**。
- **アセンブラ**（DynASM・AsmJit・GNU Lightning）は符号化だけ。DynASM は
  C への埋め込み記法、AsmJit はビルダ API。**ラベルとバックパッチ**で
  ジャンプ先を後から埋めてくれる。細かい制御が要るベースライン段に。
- **軽量生成器**（Cranelift・MIR・libgccjit）は IR（`v2 = iadd v0, v1` 風）を
  受け取り、[線形走査](ir.md)などの軽量なレジスタ割り当てとパスを絞った
  軽い最適化で**速く生成**する。
- **LLVM** は最高品質の最適化（ORC JIT・MCJIT）。ただし数十のパスで重く、
  ウォームアップに不利。投機を伴うなら**スタックマップ/patchpoint で
  [脱最適化](deopt-osr.md)の状態を保全**する設計が要る。
- **RPython・Truffle** はインタプリタを書くだけで JIT が自動生成される
  最上層。RPython の `jit_merge_point`、Truffle の `@Specialization`/
  `Assumption` のように、要所へヒントを置くのが作法。新言語に最適。
- **W^X・再配置・call の射程・コードキャッシュ管理**は、どの層でも顔を出す横断的な実装事項。
- 選択は「すごさ」でなく、各[ティア](tiered.md)の要求に合う抽象度で。
  ベースラインはアセンブラ直叩き、最適化段は LLVM や自前 SSA、という
  層をまたいだ組み合わせが実戦的。
