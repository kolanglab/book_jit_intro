# ケーススタディ：Ruby（YJIT と仲間たち）

本書のサンプルでずっと使ってきた Ruby は、JIT の歴史としては後発ですが、
近年もっとも動きの激しい処理系の一つです。特に **YJIT** は、
[コンパイルの単位](compilation-units.md)で紹介した**基本ブロック
バージョニング**（BBV）を実用化し、Ruby を目に見えて速くしました。
この章では、Ruby の JIT のいくつもの試みと、Ruby 固有のオブジェクト
モデルへの寄り添い方を見ます。

## いくつもの JIT：MJIT・YJIT・RJIT・ZJIT

Ruby（CRuby）の JIT には、いくつかの世代があります。整理しておきましょう。

| 名前 | 登場 | 方式 | 実装言語 | 特徴 |
|------|------|------|----------|------|
| MJIT | Ruby 2.6 (2018) | メソッド JIT | C | C コードを吐いて gcc/clang でコンパイル |
| YJIT | Ruby 3.1 (2021) | BBV メソッド JIT | Rust | [基本ブロックバージョニング](compilation-units.md) |
| RJIT | Ruby 3.3 (2023) | メソッド JIT | Ruby | MJIT の後継・実験的 |
| ZJIT | 開発中 | SSA メソッド JIT | Rust | [中間表現](ir.md)を持つ新世代 |

**MJIT** は最初の公式 JIT で、ユニークな方式でした。Ruby のバイトコードを
**C のソースコード**に変換し、それを**システムの C コンパイラ（gcc や
clang）で実行時にコンパイル**して共有ライブラリにし、読み込んで使う。
[フレームワーク](frameworks.md)で見た「重い最適化コンパイラを借りる」
発想を、外部の C コンパイラで実現したものです。最適化品質は高い反面、
外部プロセスを起動するため[ウォームアップ](research-warmup.md)が重く、
Web アプリのような短命・多様なワークロードでは効果が出にくいのが課題でした。

そして登場したのが **YJIT** です。

## YJIT：基本ブロックバージョニング

**YJIT**[](#cite:chevalier2021)は、Shopify が開発し Ruby 3.1 で導入された
JIT です。[コンパイルの単位](compilation-units.md)で紹介した**基本ブロック
バージョニング**（BBV）[](#cite:chevalier2015)を中核に据えています。
復習すると、BBV の考え方はこうでした。

> 基本ブロックを、**そのブロックに入ってくる時点で分かっている型情報
> ごとに、別バージョンの機械語**としてコンパイルする。型が分かっている
> 版では、その型のガードを置かなくてよい。

Ruby のバイトコード（YARV 命令列）を、ブロック単位で、**遅延的に**
（lazy、実際にそのブロックに到達して初めて）コンパイルします。実行が
たどり着いていないブロックはコンパイルしないので、無駄がありません。
そして到達したとき、そこまでに判明している型（レシーバが整数か、配列か、
など）に応じた版を生成します。

具体例で考えましょう。

```ruby
def add_one(x)
  x + 1
end
```

`x + 1` のコンパイルでは、`x` の型が問題です。YJIT は、`add_one` が
最初に呼ばれたときの `x` の型を観測します。もし整数なら、

1. 「`x` は整数か？」のガードを置く。
2. その先のブロックを「`x` は整数版」としてコンパイルする。整数加算の
   高速経路を、型チェックなしで出す。

そして「`x` は整数版」のブロックは、以後 `x` が整数で来るかぎり、追加の
型チェックなしで使い回されます。型情報がブロックの版として固定されるので、
[トレーシング](tracing.md)のような経路の記録をせずとも、「実際に来た型に
特化したコード」が得られる —— これが BBV の妙味でした。

### スタブで「来た版だけ」を遅延生成する

BBV の遅延性を支えているのが、**スタブ**（stub）という仕掛けです。例えば
こうします。あるブロックの末尾に、まだコンパイルしていない後続ブロックへの
分岐があるとき、YJIT はその分岐先に**本物のコードではなく、小さな
スタブへのジャンプ**を埋めておきます。スタブの中身は「YJIT 本体を呼び出して
後続ブロックを今コンパイルさせ、できあがったコードのアドレスで、自分への
ジャンプを上書きする」という処理です。

```
[ブロック B0 のコード]
   ...
   jmp  stub_for_B1     ; まだ B1 は無い。とりあえずスタブへ飛ぶ

stub_for_B1:
   ; YJIT を呼ぶ。「B1 を、いまの型コンテキストでコンパイルして」
   ; 戻ってきたら jmp 命令を本物の B1 アドレスに書き換える(パッチ)
   ; 以後この経路は二度とスタブを通らない
```

これは[フレームワーク](frameworks.md)で見た**自己書き換えコード**そのもので、
[ダイレクトスレッディングの先祖返り](why-jit.md)とも言える素朴さがあります。
利点は二つです。第一に、**実際に到達した分岐先しかコンパイルしない**こと。
`if` 文の片側にしか実行が来なければ、もう片側は永久にスタブのままで、機械語を
1 バイトも生成しません。第二に、**到達した時点の型コンテキストを使える**こと。
スタブを踏んだ瞬間にレジスタやスタックの型が判明しているので、その型に特化した
版を作れます。先に全部コンパイルしてしまう方式では、この「来てから決める」が
できません。

> [!NOTE]
> スタブを踏んでパッチするまでの一瞬は、YJIT 本体（Rust 側）へ制御が移ります。
> ここは「コンパイル時間」そのものなので、[ウォームアップ](research-warmup.md)の
> コストになります。YJIT は中間表現を介さず YARV 命令から直接機械語を吐く
> 設計なので、このスタブ解決が**非常に軽い**のが効いています。重い最適化
> コンパイラなら、ここで数ミリ秒待たされかねません。

### 版が増えすぎないようにする

型ごとに版を分ける、という発想には落とし穴があります。**多態なコードでは版が
爆発しうる**のです。例えばこうです。

```ruby
def render(obj)
  obj.to_s   # obj に Integer, String, Array, ... と何でも来うる
end
```

`obj` に来る型が 1 種類なら版は 1 つで済みますが、10 種類来れば、素朴には
`to_s` 周辺のブロックが 10 版に膨らみます。機械語の量（コードサイズ）が
増え、命令キャッシュを汚し、かえって遅くなりかねません。

そこで YJIT は、**1 つのブロックが持てる版の数に上限**を設けています。例えば
こうします。上限（既定では数個程度）に達したら、それ以上は型を区別しない
**総称版**（型を一つに固定せず、実行時に型を見て分岐する汎用的な版）を 1 つ
だけ作り、以後の型はすべてそこへ流します。「よく来る数種類は特化版で速く、
残りは総称版でそこそこ」という割り切りです。これは[コンパイルの単位](compilation-units.md)で
論じた「特化と汎用のトレードオフ」を、版の個数という具体的なつまみで
調整している例だと言えます。

> [!TIP]
> `--yjit-stats` の出力には、各命令がどれだけ特化に成功し、どれだけ総称版に
> 落ちたか（多くは `..._megamorphic` のような名前で集計されます）が出ます。
> 自分のコードで「版が爆発している多態なホットスポット」を探すのに使えます。

> [!NOTE]
> YJIT が **Rust** で書かれているのも特筆点です（CRuby 本体は C）。コード生成は、
> [フレームワーク](frameworks.md)の章で見た**アセンブラ層**を自前で持ち、x86-64 と
> ARM64 の両方に対応します。「なぜ Rust の小さなコンパイラなのか」という設計判断は、
> 本章後半で改めて扱います。

## Ruby 固有の難所：オブジェクトの形（Object Shapes）

[V8 のヒドゥンクラス](case-v8.md)を覚えているでしょうか。Ruby も、同じ
問題を抱えていました。Ruby のオブジェクトはインスタンス変数を後から
自由に足せます。

```ruby
class Point
  def initialize(x, y)
    @x = x      # インスタンス変数を足す
    @y = y
  end
end
```

従来の CRuby は、各オブジェクトのインスタンス変数を、内部的にやや
込み入った方法で管理しており、`@x` の読み出しが JIT にとって特殊化
しにくいものでした。そこで Ruby 3.2 で導入されたのが **Object Shapes
（オブジェクトの形）** です。これは[V8 のヒドゥンクラス](case-v8.md)と
同じ発想を Ruby に持ち込んだものです。

> オブジェクトが持つインスタンス変数の集合と並びを「シェイプ」として
> 管理し、同じ手順で作られたオブジェクトはシェイプを共有する。各
> インスタンス変数は、シェイプ上の固定オフセットに割り当てられる。

`@x = x; @y = y` を実行すると、オブジェクトのシェイプが「空 → `@x` あり →
`@x`,`@y` あり」と[遷移](case-v8.md)します。同じ手順の `Point` はすべて
同じシェイプを共有し、`@x` の読み出しは固定オフセットアクセスになります。

そして YJIT は、このシェイプを[投機](why-jit.md)に使います。`point.x` の
ような[メソッド](glossary.md)経由のインスタンス変数アクセスを、「シェイプが
期待どおりかガードして、固定オフセットを読む」コードに特殊化します。
[V8 で見た](case-v8.md)「同じ形のオブジェクトを作ると速い」という話が、
Ruby でもそのまま成り立つようになったのです。Object Shapes は、JIT を
効かせるための**データ構造側の地ならし**でした。

具体的に、`@x` を読む `attr_reader :x` 経由のアクセスがどう機械語に落ちるかを
見てみましょう。脱最適化を伴わない素朴な版では、概念的にはこうなります。

```
; レシーバ self の @x を読む（特殊化前のイメージ）
mov  rax, [self + shape_id_offset]   ; self のシェイプ ID を取る
cmp  rax, EXPECTED_SHAPE_ID          ; 期待したシェイプか？ (ガード)
jne  side_exit                       ; 違えばサイドイグジット
mov  rax, [self + IVAR_X_OFFSET]     ; 固定オフセットから @x を直読み
```

ポイントは、**ガード 1 回と直読み 1 回**に縮んでいることです。シェイプが
合うと分かっていれば、「`@x` がそもそも定義されているか」「何番目の
インスタンス変数か」「インスタンス変数表が拡張されてヒープ外に追い出されて
いないか」といった本来必要な分岐が、すべて消えます。`IVAR_X_OFFSET` は
コンパイル時に確定した定数なので、メモリアクセスは 1 命令です。これは
[V8 のヒドゥンクラス](case-v8.md)で見た特殊化と、機械語のレベルでそっくりです。

> [!WARNING]
> ここでトレードオフが一つあります。**インスタンス変数を加える順序が
> ばらばらだと、シェイプが分岐して特殊化が効きにくくなります**。例えば
> ある経路では `@x` を先に、別の経路では `@y` を先に代入すると、別々の
> シェイプができ、同じ `point.x` でも複数のシェイプ ID をガードしなければ
> なりません。版が増える話と同じで、ここでもシェイプが多態化すると、最後は
> ガードをあきらめて総称的なアクセス（実行時にシェイプ表を引く遅い経路）に
> 落ちます。「インスタンス変数は `initialize` で、いつも同じ順に初期化する」
> という古典的な作法が、Object Shapes 時代には**性能上の意味**を持つように
> なったわけです。[V8 で「同じ形を作れ」と言った](case-v8.md)のと同じ教訓です。

> [!IMPORTANT]
> ここに、本書全体を貫くテーマが現れています。**JIT を速くするには、
> JIT 単体でなく、オブジェクトモデルや[値の表現](why-jit.md)といった
> データ構造の側を「特殊化しやすい形」に整える**必要がある。V8 の
> ヒドゥンクラスも、Ruby の Object Shapes も、「JIT が固定オフセットへ
> 特殊化できるように、オブジェクトの形を共有・追跡する」という同じ
> 設計判断です。JIT とデータ構造は、両輪なのです。

## 即値：整数と小さな浮動小数点

Ruby はもう一つ、JIT に有利な値表現の工夫を持っています。**即値**
（immediate value）です。[なぜ JIT か](why-jit.md)でふれた[ボクシング](why-jit.md)の
コストを避けるため、小さな整数（Fixnum）や一部の浮動小数点数（Flonum）、
`nil`・`true`・`false`・シンボルは、**ヒープのオブジェクトを作らず、
ポインタのビットに値を埋め込んで**表現されます。

```ruby
p 42.object_id        # 整数は値から計算される即値（85 = 42*2+1 のような形）
p nil.object_id       # nil も即値
```

整数のポインタは、最下位ビットを 1 にして「これは箱でなく即値の整数だ」と
区別します。YJIT は `x + 1` を特殊化するとき、この即値表現を直接扱う
機械語を出せます。箱を開ける必要がないので、整数演算が軽くなります。
[V8 が小整数（Smi）を投機的に扱う](case-v8.md)のと同じ工夫です。

具体的に、`x + 1` で `x` が整数だと投機できているとき、加算はこう落ちます。
即値整数は「実値を 1 ビット左シフトして最下位を 1 にした」表現（`Fixnum`
タグ付け）なので、`+ 1` は「タグを外して足して付け直す」と素朴に書くと無駄が
出ます。例えばこうして縮めます。

```
; x + 1 （x は即値整数とガード済み）。VALUE 表現は (実値<<1)|1
; 1 を足すとは、即値表現では 2 (= 1<<1) を足すこと。ただしタグの 1 が
; 二重に乗らないよう調整する
mov  rax, rx
add  rax, 2          ; 即値表現のまま +1 相当
jo   side_exit       ; 桁あふれ(Bignum 化が必要)なら脱最適化
```

`jo`（オーバーフロー時ジャンプ）が効いています。Ruby の整数は本来いくらでも
大きくなれる（`Bignum` へ自動昇格する）ので、JIT は「**この回は `Fixnum`
の範囲に収まる**」と投機し、収まらなかったらサイドイグジットで安全な経路へ
逃がします。これは次節のサイドイグジットの典型例です。

## サイドイグジット：外れたら素直にインタプリタへ

YJIT の設計でとりわけ特徴的なのが、**サイドイグジット**（side exit）の
扱い方です。ガードが破れたとき、YJIT は[脱最適化](deopt-osr.md)を行いますが、
その戻り先は**多くの場合インタプリタ**です。つまり「最適化版から、もっと
最適化された別の版へ」ではなく、「最適化版から、素朴なインタプリタ実行へ」
戻します。例えばこうです。

```
guard_integer:
   test rx, 1
   jz   side_exit       ; x が即値整数でない（箱だった）→ サイドイグジット
   ; ... 整数版の高速経路 ...

side_exit:
   ; いまのレジスタ/スタックの値を YARV のスタックフレームへ書き戻し、
   ; この YARV 命令の先頭から CRuby のインタプリタに実行を委ねる
```

ここがポイントです。サイドイグジットは、レジスタに散っている値や、計算途中の
スタックを、**インタプリタが期待する形（YARV のスタックフレーム）に作り直して**
から制御を渡します。これは[脱最適化](deopt-osr.md)の章で「状態の作り替えが
難しい」と述べたことの、Ruby における実装です。YJIT はこのために、各
サイドイグジット地点で「どのレジスタが、どの YARV スタックスロットに対応するか」
というマップを保持しています。

なぜ YJIT は「外れたら素直にインタプリタへ」を多用するのでしょうか。これは
**意図的な設計判断**で、トレードオフがあります。

- **利点（単純さ）**：脱最適化先のコードを別途生成・管理しなくてよい。常に
  正しく動くインタプリタが「最後の受け皿」として既にあるので、JIT は
  「うまくいく場合だけ」を生成すればよい。実装が劇的に単純になります。
- **利点（安全側）**：知らない型・珍しい経路に出会っても、クラッシュせず
  インタプリタへ戻るだけ。[フレームワーク](frameworks.md)で論じた「正しさを
  保つ最後の砦」を、別経路の最適化コードでなくインタプリタが担います。
- **欠点（再特殊化しにくさ）**：一度サイドイグジットした地点は、その後も
  インタプリタに落ちがちです。V8 のような多段[ティアリング](tiered.md)なら
  「外れたら別の投機でもう一段最適化」もできますが、素朴なサイドイグジットは
  「外れたら、その回はインタプリタ」で終わります。

> [!IMPORTANT]
> この「素直にインタプリタへ戻る」設計は、YJIT の哲学を象徴しています。
> **派手な最適化より、生成の速さ・実装の単純さ・安全性を優先する**。Web
> アプリのように[ウォームアップ](research-warmup.md)が許されず、コードの
> 形が多様なワークロードでは、「重い最適化を完璧にやる」より「軽い特殊化を
> 素早く広く効かせ、外れたら静かにインタプリタへ」のほうが、現実的に速い
> ——これが MJIT の反省から得た YJIT の答えでした。

## メソッド呼び出しの特殊化

Ruby のプログラムは、煎じ詰めればほとんどがメソッド呼び出し（`send`）です。
ここを速くできるかが JIT の成否を分けます。YJIT はいくつかの手で `send` を
特殊化します。

### インラインキャッシュ：宛先を覚える

`obj.foo` を呼ぶたびに「`obj` のクラスの[メソッド](glossary.md)表を引いて
`foo` の実体を探す」のは高くつきます。そこで CRuby は**インラインキャッシュ**
（呼び出し地点ごとに、前回引いた結果を覚えておく仕組み）を持っており、YJIT は
これを機械語のガードに焼き直します。例えばこうです。

```
; obj.foo の呼び出し（インラインキャッシュ + ガード）
mov  rax, [obj + class_offset]
cmp  rax, CACHED_CLASS         ; 前回と同じクラスか？ (ガード)
jne  side_exit                 ; 違えばインタプリタへ（再解決させる）
call CACHED_METHOD_BODY        ; 解決済みのメソッド本体へ直接 call
```

クラスが前回と同じなら、メソッド探索を丸ごと飛ばして本体へ直行できます。これは
[V8 のヒドゥンクラスを使った呼び出し](case-v8.md)や、[インラインキャッシュの
古典](#cite:deutsch1984)と同じ発想です。なお、レシーバのクラスが毎回違う多態な
呼び出し地点では、BBV の版上限と同様にキャッシュが効かず、総称的な探索に
落ちます。

### 組み込みメソッドの特殊化

さらに YJIT は、`Integer#+`、`Array#[]`、`Array#length`、`Hash#[]` といった
**C で書かれた組み込みメソッド**を、特別扱いします。これらは Ruby のソースが
無く C 関数として存在するので、素朴には「C 関数を `call` する」しかありません。
しかし頻出するものについては、例えばこうします。

```ruby
arr[i]   # Array#[] の呼び出し
```

これを「`call Array#[]` の C 関数」ではなく、**配列要素読み出しの機械語を
直接埋め込む**よう特殊化します。レシーバが本当に配列か、インデックスが範囲内か
をガードし、外れたらサイドイグジット、当たれば数命令で要素を読む、という形です。
C 関数呼び出しの段取り（引数のレジスタ詰め、フレーム作り）をまるごと省けるので、
ループ内の配列・ハッシュアクセスが大きく速くなります。これは LuaJIT などが
[組み込み関数を IR に埋め込む](case-luajit.md)のと同じ狙いで、YJIT は中間表現を
持たないぶん、**特定の組み込みメソッドに対する機械語テンプレート**として
持っているイメージです。

> [!TIP]
> どの組み込みメソッドが特殊化されるかは Ruby のバージョンで増えていきます。
> `--yjit-stats` で `send_` 系のカウンタを見ると、特殊化に成功した呼び出しと、
> 総称的な `call` に落ちた呼び出しの内訳が分かります。

## なぜ YJIT は「小さなコンパイラ」を選んだのか

ここまでの手法を貫く設計思想を、一度まとめておきます。YJIT は、**中間表現を
ほとんど持たず、YARV 命令から直接機械語を吐く小さなコンパイラ**です。
[中間表現](ir.md)の章で見たような SSA 形式の作業台や、大域的な命令並べ替えは、
基本的にやりません。これは「最適化品質」だけ見れば不利な選択です。なぜ
あえてそうしたのでしょうか。

理由は[フレームワーク](frameworks.md)の章の議論にそのままつながります。

- **生成が速い**：中間表現の構築・解析・最適化パスを省くぶん、ブロックを
  機械語にする時間が短い。[ウォームアップ](research-warmup.md)が効かない
  Web ワークロードでは、これが効きます。MJIT が外部 C コンパイラを起動して
  重くなった反省が、ここに生きています。
- **実装が単純**：BBV のブロック単位生成、スタブ、サイドイグジットという
  少数の道具で全体が組み上がる。中間表現を介した最適化器に比べ、コード量も
  バグの面積も小さい。
- **メモリ安全な言語で書ける**：機械語生成は間違えやすい領域ですが、Rust の
  型・所有権で多くの誤りを防げます。小さく単純な設計だからこそ、新規言語での
  実装が現実的になりました。

つまり YJIT は、[フレームワーク](frameworks.md)で並べた「抽象度の階段」の
うち、**重い最適化コンパイラ（最高速だが遅い・重い）ではなく、軽量な
コード生成器寄りの位置**を、意図的に選んだ処理系です。BBV という単位の取り方が、
この「軽く・速く・素直に」という方針と噛み合っていたのです。

## そして ZJIT へ

（Ruby の JIT 事情は版ごとに動きが速いので、以下は本書執筆時点の理解です。
最新の状況は各 Ruby リリースのリリースノートで確認してください。）

YJIT は BBV で大きな成果を上げましたが、BBV には「[中間表現](ir.md)を
経た本格的な最適化（命令の大域的な並べ替えや、メソッドをまたいだ最適化）が
やりにくい」という性質もあります。基本ブロック単位で素早く出すことに
特化しているぶん、メソッド全体を見渡した最適化は不得手なのです。

そこで開発が進んでいるのが **ZJIT** です。ZJIT は比較的新しめの取り組みで、
YJIT の経験を踏まえ、**[SSA 形式](ir.md)の中間表現を持つメソッド単位の
JIT** を目指す方向だと説明されています。[本書で見た](ir.md)ような中間表現の
上で、定数畳み込み・型推論・インライン化といった大域的な最適化を行い、その
うえで機械語を出す、という発想です。

両者の違いを、本書の言葉で対比すると分かりやすくなります。

| 観点 | YJIT | ZJIT が目指す方向 |
|------|------|-----------|
| 単位 | [基本ブロック](compilation-units.md)（BBV） | [メソッド](compilation-units.md)（SSA 上で） |
| [中間表現](ir.md) | ほぼ持たない | [SSA 形式](ir.md)を持つ |
| 最適化の範囲 | ブロック内・局所的 | メソッド全体・大域的 |
| 生成の速さ | 非常に速い | 相対的に重くなりうる |
| 得意な最適化 | 型特殊化・即値化 | 命令の並べ替え・[インライン化](compilation-units.md)・大域的な定数畳み込み |

YJIT が「ブロックを見て素早く特化する」のに対し、ZJIT は「メソッドを見渡して
じっくり最適化する」。前者が消しにくかった「メソッドをまたぐ最適化」や
「ループ全体を見た不変式の巻き上げ」を、後者は中間表現の上で狙えます。
代わりに、生成は相対的に重くなりうるので、どこで[ティアリング](tiered.md)的に
使い分けるか、といった設計が課題になります。YJIT の素早さと、中間表現による
最適化の深さを、どう両立させるか——詳細は流動的なので断定は避けますが、
Ruby の JIT が「[BBV の軽量さ](compilation-units.md)」から
「[中間表現による本格最適化](ir.md)」へと、本書でたどってきた地図の上を
進もうとしている、その方向性はたしかです。

> [!TIP]
> 手元の Ruby（3.1 以降）で YJIT を試せます。`ruby --yjit your_script.rb` で
> 有効になり、`RUBY_YJIT_ENABLE=1` 環境変数でも有効化できます。
> `--yjit-stats` を付けると、どれだけのブロックがコンパイルされ、どの
> 操作が特殊化されたか（あるいはされなかったか）の統計が見えます。本章で
> 説明した BBV や Object Shapes の効きぐあいを、実物で観察できます。

## まとめ

- Ruby の JIT は世代を重ねた：**MJIT**（C を吐いて外部コンパイラへ）→
  **YJIT**（BBV、Rust 製）→ **RJIT**（Ruby 製）→ 開発中の **ZJIT**
  （[SSA 中間表現](ir.md)）。
- **YJIT**[](#cite:chevalier2021)は[基本ブロックバージョニング](compilation-units.md)を
  実用化。ブロックを入口の型ごとに版分けし、遅延コンパイルで型チェックを
  自然に除去する。**スタブ**で「実際に来た版だけ」を遅延生成し、版が増えすぎ
  たら総称版に落として版の爆発を防ぐ。
- Ruby 3.2 の **Object Shapes** は[V8 のヒドゥンクラス](case-v8.md)を
  Ruby に持ち込み、インスタンス変数を固定オフセット化。アクセスは
  「シェイプをガード→固定オフセット直読み」に縮む。[特殊化](why-jit.md)
  できる土台を作った。
- **即値**（Fixnum・Flonum・nil など）で[ボクシング](why-jit.md)を避ける
  値表現も、JIT を助ける。整数加算は即値表現のまま `add` し、桁あふれは
  サイドイグジットで安全な経路へ逃がす。
- **サイドイグジット**は[脱最適化](deopt-osr.md)先を多くインタプリタに取る。
  別経路の最適化コードを作らず、正しいインタプリタを最後の受け皿にする
  ——生成の速さ・単純さ・安全性を優先する設計判断。
- メソッド呼び出しは**インラインキャッシュ**[](#cite:deutsch1984)でクラスを
  ガードし宛先を直結。`Array#[]` などの**組み込みメソッド**は機械語を直接
  埋め込んで特殊化する。
- YJIT が[中間表現](ir.md)をほぼ持たない**小さなコンパイラ**を選んだのは、
  生成の速さ・実装の単純さ・Rust による安全性のため
  （[フレームワーク](frameworks.md)の階段で軽量側を意図的に選んだ）。
- **ZJIT** は比較的新しめの取り組みで、BBV の軽さから[SSA 中間表現](ir.md)に
  よるメソッド全体の本格最適化へ向かう Ruby JIT の次の方向。
