# ケーススタディ：Ruby（YJIT と仲間たち）

本書のサンプルでずっと使ってきた Ruby は、JIT の歴史としては後発ですが、
近年もっとも動きの激しい処理系の一つです。特に **YJIT** は、
[コンパイルの単位](compilation-units.md)で紹介した**基本ブロック
バージョニング**（BBV）を実用化し、Ruby を目に見えて速くしました。
この章では、Ruby の JIT のいくつもの試みと、Ruby 固有のオブジェクト
モデルへの寄り添い方を見ます。

## いくつもの JIT：MJIT・YJIT・RJIT・ZJIT

Ruby（CRuby）の JIT には、いくつかの世代があります。整理しておきましょう。

| 名前 | 登場 | 方式 | 実装言語 | 特徴 |
|------|------|------|----------|------|
| MJIT | Ruby 2.6 (2018) | メソッド JIT | C | C コードを吐いて gcc/clang でコンパイル |
| YJIT | Ruby 3.1 (2021) | BBV メソッド JIT | Rust | [基本ブロックバージョニング](compilation-units.md) |
| RJIT | Ruby 3.3 (2023) | メソッド JIT | Ruby | MJIT の後継・実験的 |
| ZJIT | 開発中 | SSA メソッド JIT | Rust | [中間表現](ir.md)を持つ新世代 |

**MJIT** は最初の公式 JIT で、ユニークな方式でした。Ruby のバイトコードを
**C のソースコード**に変換し、それを**システムの C コンパイラ（gcc や
clang）で実行時にコンパイル**して共有ライブラリにし、読み込んで使う。
[フレームワーク](frameworks.md)で見た「重い最適化コンパイラを借りる」
発想を、外部の C コンパイラで実現したものです。最適化品質は高い反面、
外部プロセスを起動するため[ウォームアップ](research-warmup.md)が重く、
Web アプリのような短命・多様なワークロードでは効果が出にくいのが課題でした。

そして登場したのが **YJIT** です。

## YJIT：基本ブロックバージョニング

**YJIT**[](#cite:chevalier2021)は、Shopify が開発し Ruby 3.1 で導入された
JIT です。[コンパイルの単位](compilation-units.md)で紹介した**基本ブロック
バージョニング**（BBV）[](#cite:chevalier2015)を中核に据えています。
復習すると、BBV の考え方はこうでした。

> 基本ブロックを、**そのブロックに入ってくる時点で分かっている型情報
> ごとに、別バージョンの機械語**としてコンパイルする。型が分かっている
> 版では、その型のガードを置かなくてよい。

Ruby のバイトコード（YARV 命令列）を、ブロック単位で、**遅延的に**
（lazy、実際にそのブロックに到達して初めて）コンパイルします。実行が
たどり着いていないブロックはコンパイルしないので、無駄がありません。
そして到達したとき、そこまでに判明している型（レシーバが整数か、配列か、
など）に応じた版を生成します。

具体例で考えましょう。

```ruby
def add_one(x)
  x + 1
end
```

`x + 1` のコンパイルでは、`x` の型が問題です。YJIT は、`add_one` が
最初に呼ばれたときの `x` の型を観測します。もし整数なら、

1. 「`x` は整数か？」のガードを置く。
2. その先のブロックを「`x` は整数版」としてコンパイルする。整数加算の
   高速経路を、型チェックなしで出す。

そして「`x` は整数版」のブロックは、以後 `x` が整数で来るかぎり、追加の
型チェックなしで使い回されます。型情報がブロックの版として固定されるので、
[トレーシング](tracing.md)のような経路の記録をせずとも、「実際に来た型に
特化したコード」が得られる —— これが BBV の妙味でした。

> [!NOTE]
> YJIT が Rust で書かれているのも特筆点です。CRuby は C で書かれて
> いますが、YJIT は新規コンポーネントとして Rust で実装され、CRuby に
> 組み込まれています。メモリ安全な言語で JIT を書くことで、機械語生成と
> いう間違えやすい領域の安全性を高める狙いです。コード生成自体は、
> [フレームワーク](frameworks.md)の章で見た**アセンブラ層**を自前で持ち、
> x86-64 と ARM64 の両方に対応しています。

## Ruby 固有の難所：オブジェクトの形（Object Shapes）

[V8 のヒドゥンクラス](case-v8.md)を覚えているでしょうか。Ruby も、同じ
問題を抱えていました。Ruby のオブジェクトはインスタンス変数を後から
自由に足せます。

```ruby
class Point
  def initialize(x, y)
    @x = x      # インスタンス変数を足す
    @y = y
  end
end
```

従来の CRuby は、各オブジェクトのインスタンス変数を、内部的にやや
込み入った方法で管理しており、`@x` の読み出しが JIT にとって特殊化
しにくいものでした。そこで Ruby 3.2 で導入されたのが **Object Shapes
（オブジェクトの形）** です。これは[V8 のヒドゥンクラス](case-v8.md)と
同じ発想を Ruby に持ち込んだものです。

> オブジェクトが持つインスタンス変数の集合と並びを「シェイプ」として
> 管理し、同じ手順で作られたオブジェクトはシェイプを共有する。各
> インスタンス変数は、シェイプ上の固定オフセットに割り当てられる。

`@x = x; @y = y` を実行すると、オブジェクトのシェイプが「空 → `@x` あり →
`@x`,`@y` あり」と[遷移](case-v8.md)します。同じ手順の `Point` はすべて
同じシェイプを共有し、`@x` の読み出しは固定オフセットアクセスになります。

そして YJIT は、このシェイプを[投機](why-jit.md)に使います。`point.x` の
ような[メソッド](glossary.md)経由のインスタンス変数アクセスを、「シェイプが
期待どおりかガードして、固定オフセットを読む」コードに特殊化します。
[V8 で見た](case-v8.md)「同じ形のオブジェクトを作ると速い」という話が、
Ruby でもそのまま成り立つようになったのです。Object Shapes は、JIT を
効かせるための**データ構造側の地ならし**でした。

> [!IMPORTANT]
> ここに、本書全体を貫くテーマが現れています。**JIT を速くするには、
> JIT 単体でなく、オブジェクトモデルや[値の表現](why-jit.md)といった
> データ構造の側を「特殊化しやすい形」に整える**必要がある。V8 の
> ヒドゥンクラスも、Ruby の Object Shapes も、「JIT が固定オフセットへ
> 特殊化できるように、オブジェクトの形を共有・追跡する」という同じ
> 設計判断です。JIT とデータ構造は、両輪なのです。

## 即値：整数と小さな浮動小数点

Ruby はもう一つ、JIT に有利な値表現の工夫を持っています。**即値**
（immediate value）です。[なぜ JIT か](why-jit.md)でふれた[ボクシング](why-jit.md)の
コストを避けるため、小さな整数（Fixnum）や一部の浮動小数点数（Flonum）、
`nil`・`true`・`false`・シンボルは、**ヒープのオブジェクトを作らず、
ポインタのビットに値を埋め込んで**表現されます。

```ruby
p 42.object_id        # 整数は値から計算される即値（85 = 42*2+1 のような形）
p nil.object_id       # nil も即値
```

整数のポインタは、最下位ビットを 1 にして「これは箱でなく即値の整数だ」と
区別します。YJIT は `x + 1` を特殊化するとき、この即値表現を直接扱う
機械語を出せます。箱を開ける必要がないので、整数演算が軽くなります。
[V8 が小整数（Smi）を投機的に扱う](case-v8.md)のと同じ工夫です。

## そして ZJIT へ

YJIT は BBV で大きな成果を上げましたが、BBV には「[中間表現](ir.md)を
経た本格的な最適化（命令の大域的な並べ替えや、メソッドをまたいだ最適化）が
やりにくい」という性質もあります。基本ブロック単位で素早く出すことに
特化しているぶん、メソッド全体を見渡した最適化は不得手なのです。

そこで開発が進んでいるのが **ZJIT** です。ZJIT は、YJIT の経験を踏まえ、
**[SSA 形式](ir.md)の中間表現を持つメソッド単位の JIT** を目指しています。
[本書で見た](ir.md)ような中間表現の上で、定数畳み込み・型推論・インライン化
といった大域的な最適化を行い、そのうえで機械語を出す。YJIT の素早さと、
中間表現による最適化の深さを、両立しようという試みです。Ruby の JIT は、
「[BBV の軽量さ](compilation-units.md)」から「[中間表現による本格最適化](ir.md)」へと、
本書でたどってきた地図の上を、いままさに進んでいるところです。

> [!TIP]
> 手元の Ruby（3.1 以降）で YJIT を試せます。`ruby --yjit your_script.rb` で
> 有効になり、`RUBY_YJIT_ENABLE=1` 環境変数でも有効化できます。
> `--yjit-stats` を付けると、どれだけのブロックがコンパイルされ、どの
> 操作が特殊化されたか（あるいはされなかったか）の統計が見えます。本章で
> 説明した BBV や Object Shapes の効きぐあいを、実物で観察できます。

## まとめ

- Ruby の JIT は世代を重ねた：**MJIT**（C を吐いて外部コンパイラへ）→
  **YJIT**（BBV、Rust 製）→ **RJIT**（Ruby 製）→ 開発中の **ZJIT**
  （[SSA 中間表現](ir.md)）。
- **YJIT**[](#cite:chevalier2021)は[基本ブロックバージョニング](compilation-units.md)を
  実用化。ブロックを入口の型ごとに版分けし、遅延コンパイルで型チェックを
  自然に除去する。
- Ruby 3.2 の **Object Shapes** は[V8 のヒドゥンクラス](case-v8.md)を
  Ruby に持ち込み、インスタンス変数を固定オフセット化。JIT が[特殊化](why-jit.md)
  できる土台を作った。
- **即値**（Fixnum・Flonum・nil など）で[ボクシング](why-jit.md)を避ける
  値表現も、JIT を助ける。
- **ZJIT** は、BBV の軽さから[中間表現](ir.md)による本格最適化へ向かう
  Ruby JIT の次の一歩。

二つの道を見ます。
