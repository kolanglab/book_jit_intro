# 投機的最適化の正しさ

[脱最適化と OSR](deopt-osr.md)の章の最後に、深い問いを残しました。
**「途中まで最適化版で進み、続きをインタプリタが引き継ぐ」とき、本当に
元のプログラムと同じ答えになると、どうして保証できるのか。**

これは「気をつければ大丈夫」で済む問題ではありません。JIT は投機の上に
投機を重ね、ガードを動かし、変数を消し、命令を並べ替えます。そのどれか
一つでも、脱最適化の正しさをひそかに壊していたら —— ある特定の
入力でだけ、最適化版とインタプリタ版の答えが食い違う、という悪夢の
バグになります。しかも投機が外れる経路は稀にしか通らないので、テストでも
見つかりにくい。

2018 年、Flückiger らはこの問題に**形式手法**で正面から取り組みました
[](#cite:fluckiger2018)。投機的最適化と脱最適化の正しさを数学的に
定義し、「どんな最適化なら正しさを保つか」を証明したのです。この章では、
論文の数式には立ち入らず、その**考え方**を追います。

## 何が「正しい」のか

まず、何を証明したいのかをはっきりさせます。投機的 JIT には、同じ
プログラムに対して二つの実行があります。

- **インタプリタ版**：投機なしで、常に正しく動く基準（リファレンス）。
- **最適化版**：投機して特殊化したコード。ガードが破れたら
  [脱最適化](deopt-osr.md)してインタプリタ版へ戻る。

正しさとは、ひとことで言えば、

> **最適化版は、いつ・どこで脱最適化しても、インタプリタ版と
> まったく同じ観測可能なふるまい（同じ出力、同じ副作用）をしなければ
> ならない。**

ということです。脱最適化は**透明**でなければならない —— 外から
見て、それが起きたかどうか分からない。途中で最適化版からインタプリタ版へ
乗り換えても、結果が 1 ビットも変わってはいけない。この「透明性」こそが、
証明すべき性質です。Flückiger らはこれを **弱双模倣（weak bisimulation）**
として厳密に定義しました[](#cite:fluckiger2018)。最適化版とインタプリタ版を
構成（configuration）どうしの対応関係で結び、対応する構成は同じ観測可能な
ふるまいを持ち、一歩進めても再び対応した構成になる —— この双模倣が
成り立つことが、彼らの正しさの主張そのものです。

## 難しさの正体：最適化と脱最適化の綱引き

なぜこれが難しいのか。最適化は、コードを**変形**して速くします。一方
脱最適化は、その変形したコードのどこからでも、**インタプリタが
期待する元の状態へ戻れ**と要求します。この二つは引っ張り合います。

具体例で見ましょう。あるコードで、変数 `y` が次のように使われているとします。

```ruby
x = read()          # 何か読む
y = x + 1           # y を計算する
... (y はこの先脱最適化後にだけ必要)
assume x == 42      # 投機：x は 42 だと賭ける（ガード）
... (高速な経路)
```

最適化器が「`x` は 42 と賭けたのだから、`y = x + 1` は `y = 43` だ」と
**定数畳み込み**して、`y = x + 1` の計算自体を消したくなったとします。
高速経路では `y` を使わないので、消しても速い経路は正しく動きます。

ところが、です。**ガード `assume x == 42` が破れて脱最適化した
とき**、インタプリタは `y` の値を必要とします。もし最適化器が `y` の
計算を消してしまっていたら、戻り先のインタプリタに渡すべき `y` の値が
ありません。「`y = 43`」と決め打ちで渡してよいでしょうか？ いいえ ——
ガードが破れたということは `x` は 42 では**なかった**のだから、本当の
`y` は `x + 1`（≠ 43）であるべきです。決め打ちの 43 を渡せば、
脱最適化後の実行が間違います。

> [!IMPORTANT]
> これが投機的最適化の正しさの核心的な罠です。**「投機の前提を使って
> よいのは、投機が当たっている経路だけ」**。脱最適化で戻るのは
> 投機が**外れた**瞬間なので、戻り先へ渡す状態（脱最適化情報）の
> 計算に、投機の前提を使ってはいけない。速い経路の最適化と、戻り先の
> 状態の正しさは、注意深く分けて扱わねばならないのです。

いまの `y` の例は「**デッドコード削除**が、脱最適化後にだけ必要な値を
消してしまう」壊れ方でした。これは典型例の一つにすぎません。投機的最適化が
脱最適化経路を壊す**パターンを、三つ並べて**みます。どれも「速い経路だけ
見れば確かに正しいのに、外れた瞬間にだけ破綻する」という共通の顔をしています。

**(a) デッドコード削除が、脱最適化後にだけ必要な値を消す。**
先ほどの例をもう一段くわしく見ます。最適化前の中間表現はこうでした。

```
x = read()
y = x + 1                 # (*) 高速経路では誰も y を読まない
assume x == 42 else L_deopt[ y, x ]   # 外れたら L_deopt へ。y と x を持って戻る
... 高速経路（y を使わない）...

L_deopt:                  # インタプリタ再開地点
    # ここで y, x が「生きて」いる前提でインタプリタが続きを実行する
```

最適化器は「高速経路で `y` は読まれない」と見て、`(*)` の `y = x + 1` を
**死んだ命令**だと判定したくなります。ところが `assume` の脱最適化情報
`L_deopt[ y, x ]` が `y` を要求しています。ここを見落として `y = x + 1` を
消すと、脱最適化時に渡す `y` が消滅します。**`y` は死んでいません ——
脱最適化経路という、もう一つの「使い道」を通じて生きているのです。**

```
x = read()
                          # y = x + 1 を消してしまった！
assume x == 42 else L_deopt[ y, x ]   # y がもう存在しない → 破綻
```

正しい削除判定は「高速経路で読まれない」ではなく、「高速経路でも、
**どの `assume` の脱最適化情報でも**読まれない」でなければなりません。

**(b) 命令の巻き上げ・並べ替えが、ガード破れ時に未計算の値を参照させる。**
Sea of Nodes（[中間表現](ir.md)）のように順序の自由度が高い表現では、
コンパイラは依存関係さえ守れば命令を自由に動かせます。ところがこの自由が
脱最適化と衝突します。次を考えます。

```
assume cond else L_deopt[ a, b ]   # 外れたら a, b を持って L_deopt へ
b = heavy(a)              # 高速経路で使う重い計算
... b を使う高速経路 ...
```

`b = heavy(a)` は `assume` の**後**にあります。コンパイラが「`b` は
`a` にしか依存しないから、`assume` より前へ巻き上げても速い経路は同じだ」
と判断して動かしたとします。

```
b = heavy(a)              # assume の前へ巻き上げた
assume cond else L_deopt[ a, b ]
... b を使う高速経路 ...
```

速い経路はこれでも正しく動きます。しかし脱最適化情報 `L_deopt[ a, b ]` が
`b` を要求していることに注意してください。**`b` の値が `assume` の地点で
すでに確定している前提**に、いつの間にか変わってしまいました。逆に、もし
`b = heavy(a)` を `assume` の**後ろ**へ下げる（sink する）最適化をかけると、
今度はガードが破れた瞬間 `b` がまだ計算されておらず、`L_deopt[ a, b ]` が
**未計算の値を参照**します。命令を動かすときは、脱最適化情報が要求する
値がその地点で確定しているかを必ず確認しなければなりません。

**(c) スカラ置換でオブジェクトを消したのに、脱最適化先がそれを要求する。**
**スカラ置換**（scalar replacement）は、エスケープしない（その関数の外へ
参照が漏れない、と[エスケープ解析](glossary.md)で分かった）オブジェクトを
丸ごと消し、そのフィールドを個別の局所変数（スカラ）に分解する最適化です。

```
p = new Point(x, y)       # 高速経路ではフィールドにしか触らない
... p.x, p.y を読むだけ ...
assume cond else L_deopt[ p ]   # 外れたら p（オブジェクト）を渡して戻る
```

高速経路では `p.x`, `p.y` を読むだけなので、コンパイラは `p` という
オブジェクトの生成自体を消し、`px = x`, `py = y` という二つのスカラに
置き換えたくなります。実際、速い経路だけならこれは完全に正しく、確保
（アロケーション）を一つ丸ごと省けます。

```
px = x                    # p を消してフィールドをスカラ化
py = y
... px, py を読むだけ ...
assume cond else L_deopt[ p ]   # しかし L_deopt は「p というオブジェクト」を要求
```

ところがインタプリタは `p` を**オブジェクトとして**期待しています。脱最適化
した先で `p.method()` を呼ぶかもしれないし、`p` を別の関数へ渡すかもしれない。
消してしまった `p` を、戻る瞬間に**もう一度組み立て直す**必要があります。
ここで効くのが、次節で述べる「復元レシピ」です。スカラ置換は `p` を消す
代わりに、脱最適化情報に「`p` は `Point(px, py)` を**再確保して**作れ」という
レシピを残します。こうすればオブジェクトの確保を高速経路から消しつつ、
脱最適化時にだけ `p` を実体化（materialize）できます。

> [!WARNING]
> (a)(b)(c) はどれも、最適化を個別に見れば「正しい変形」です。壊れるのは
> **脱最適化情報という第二の使い道を勘定に入れ忘れたとき**だけ。だから
> バグは高速経路のテストをすべて通過し、稀にしか踏まれない脱最適化経路で
> ある特定の入力のときにだけ牙をむきます。場当たりのコードレビューで全部を
> 捕まえるのは、現実には困難です。

## assume を第一級の命令にする

Flückiger らの工夫は、投機（ガードと脱最適化）を、IR の中の
**第一級の命令**として明示的に表したことです。彼らは sourir という小さな
中間言語を定義し、その中に `assume` という命令を置きました。これは
[中間表現](ir.md)の章でふれた「ガードと脱最適化点を IR に持たせる」設計
[](#cite:duboscq2013)を、形式的に突き詰めたものです。

`assume` 命令は、二つのものを束ねて持ちます。

1. **投機の条件**（例：`x == 42`）。これが成り立つ間は高速経路を進む。
   論文では条件は**述語のリスト**として持たれ、すべてが成り立てば次の命令へ
   進みます[](#cite:fluckiger2018)。
2. **脱最適化の行き先と、そこで復元すべき状態のレシピ**
   （どのインタプリタ位置へ、どの変数にどの値を入れて戻るか）。論文では
   この脱最適化メタデータは `F.V.L[x1=e1, .., xn=en]` という形をとり、
   戻り先（関数 `F`・版 `V`・命令ラベル `L`）と、各変数 `xi` に式 `ei` を
   束ねる **varmap** からなります[](#cite:fluckiger2018)。脱最適化時には
   各 `ei` を「古い（最適化版の）環境」で評価し、新しい環境の `xi` に
   束ねる —— レシピとはまさにこの varmap のことです。

ポイントは、この「復元のレシピ」が**命令の一部としてIRに書かれている**
ことです。だから最適化器は、`y = x + 1` を消そうとするとき、「`y` は
どこかの `assume` の復元レシピで使われているか？」を IR を見て確認
できます。使われているなら、レシピが必要とする計算は勝手に消せない
（あるいは、レシピのほうに `y` の正しい計算を含めて保存する）。
**正しさの制約が、IR の上で明示的にチェックできる形になった**のです。

具体的に、`assume` 命令の中身をもう少し開いて書いてみます。さきほどの
スカラ置換 (c) の後の状態は、こう表せます。

```
assume cond else L_deopt {
    # 復元レシピ：脱最適化したらインタプリタの状態をこう組み立てよ
    var p  := Point(px, py)   # 消した p を px, py から再確保せよ
    var x  := px              # 局所変数 x にはこの値を入れよ
    pc     := L_deopt          # この位置からインタプリタを再開せよ
}
```

レシピは「どのインタプリタ位置(`pc`)へ戻り、各局所変数に何を入れ、消した
オブジェクトをどう作り直すか」を、一つずつ書き下した手順書です。
最適化器がスカラ置換で `p = new Point(x, y)` を消すとき、同時にこの
`p := Point(px, py)` という一行をレシピへ書き足す —— これが「**消すなら、
消した分を復元できるレシピを残す**」という義務の正体です。Flückiger らの
枠組みでは、最適化はこのレシピを更新する責任を負い、レシピを正しく保つ
かぎり透明性が保たれることを証明します。

> [!TIP]
> `assume` のレシピは、[脱最適化と OSR](deopt-osr.md)で出てきた
> 「脱最適化情報（フレームをどう組み直すかの記述）」を、IR の第一級要素に
> 格上げしたものだと考えると見通しがよいです。実装上はメタデータとして
> 脇に置かれがちなこの情報を、**最適化が踏みつけてはいけない命令そのもの**
> として中央に据えたのが、この研究の手さばきです。

## 状態を保つ二つの方式：レシピか、巨大な脱最適化マップか

先に見取り図を述べておくと、**本章で扱う `assume` は、これから示す二方式の
うち「方式 1（復元レシピ）」側を IR の第一級要素にまで引き上げたもの**です。
この対比を念頭に読むと位置づけがはっきりします。

なぜ「復元レシピ」という形が効くのか。脱最適化のために元の状態を保つには、
大きく二つのやり方があり、それぞれにトレードオフがあるからです。

**方式 1：最適化に制約を課し、復元を「レシピ」で表す。**
消してよいのは「高速経路でもレシピでも使われない値」だけ、と最適化を
縛ります。消した値も、レシピに**再計算の手順**を書いておけば復元できます。
さきほどの `p := Point(px, py)` がまさにこれで、`p` の実体は高速経路には
存在しないのに、脱最適化時にだけ手順書から組み立て直せます。この方式の
利点は、**高速経路は身軽なまま**で、復元コストを「外れたとき」へ後回しに
できること。欠点は、最適化が常にレシピの更新義務を背負い、その義務を
満たしているかを検証しなければならないことです。

**方式 2：各ガード地点に、フレーム状態を丸ごと記述した脱最適化マップを持つ。**
ガードごとに「この地点では、各局所変数・各スタックスロットがどの
レジスタ／スタック位置にあるか」を**そのつど全部書き出した表**を用意します。
復元は表を引くだけなので素直です。しかし投機的 JIT はガードを大量に置く
ため、**ガードの数だけこの表が増えます**。表はコード生成時に作る必要があり、
実行コードと並んでメモリに居座ります。

> [!NOTE]
> 脱最適化マップは、けっして小さくありません。実用 JIT では、生成した
> 機械語と同じくらい、あるいはそれ以上の容量を脱最適化メタデータが占める
> こともあります。ガードは「ほとんど踏まれない安全網」なのに、その安全網の
> ための記述がメモリと生成時間を食う —— ここに方式 2 の重さがあります。
> 方式 1 の「レシピ＋再計算」は、この重さを軽くする方向の工夫だと位置づけ
> られます。

二つは排他ではなく、実際の処理系は**混ぜて**使います。よく使われる値は
マップにそのまま記録し、再計算で安く戻せる値（定数や単純な算術の結果）は
レシピに手順として残してマップから省く、といった按配です。Flückiger らの
`assume` は、この「レシピ」側を IR の第一級要素として正面から扱い、
**どの最適化がレシピをどう書き換えてよいか**を証明可能な形に整えたものだ、
と読むことができます。

## どんな変形なら許されるかを証明する

この枠組みの上で、Flückiger らは主要な最適化が**脱最適化の
透明性を保つこと**を証明しました。論文が証明したのは、大きく二群に
分かれます[](#cite:fluckiger2018)。一つは**従来からあるコンパイラ最適化**を
投機の存在下でも正しいと示したもの、もう一つは**脱最適化に固有の変形**です。

- **定数畳み込み（定数伝播）・unreachable コードの削除・関数のインライン化**：
  これら伝統的な最適化が、`assume` のある世界でも透明性を壊さないことを
  証明しました。投機の前提を使うのは高速経路に限り、`assume` の復元レシピ
  （varmap）には元の計算を保つ、という条件のもとで正しいというものです。
- **脱最適化に固有の三変形**：論文は脱最適化そのものを操作する変形として、
  **無制限の脱最適化（unrestricted deoptimization）**・
  **述語の巻き上げ（predicate hoisting）**・**`assume` の併合（assume
  composition）**の正しさを確立しました[](#cite:fluckiger2018)。ガードを
  増やしたり、巻き上げたり、まとめたりする変形も、復元レシピを正しく
  引き継げば透明性を保ちます。

これらは[中間表現](ir.md)や[脱最適化](deopt-osr.md)で「実用上
こうする」と紹介してきた最適化そのものです。本章の研究は、それらが
**確かに正しい**ことを、場当たり的な議論でなく**数学的な証明**で
裏づけました。「ガードを巻き上げてよい」「重複ガードを消してよい」と
JIT 実装者が経験的に行ってきた変形に、形式的な根拠を与えたのです。

> [!NOTE]
> 「正しさを証明する」と聞くと、実務から遠い話に思えるかもしれません。
> しかし投機的 JIT のバグは、**特定の入力でだけ、稀な脱最適化経路で
> 結果が狂う**という、最も発見しにくい種類のバグです。テストでは
> ほぼ見つかりません。だからこそ、「この最適化はどんな場合でも透明性を
> 壊さない」という保証には、実用上の大きな価値があります。形式手法は、
> JIT のような「賭けを安全網で支える」システムと、とても相性がよいのです。

## 「観測している経路だけ正しければよい」の落とし穴

透明性の証明を支えるのが、**「ガードの前提が成り立つ経路の意味だけ
保てばよい」**という原則です。`assume cond` の下流は、`cond` が真である
世界だけを考えればよい。コンパイラはこの前提を使って、その下流をいくらでも
特殊化・単純化してかまいません。前提が外れた瞬間は、高速経路の意味など
どうでもよく、**復元レシピで元の状態へ戻る**だけだからです。

この原則は強力ですが、**レシピの計算そのものに投機の前提を使ってしまうと
壊れます**。具体例で見ましょう。冒頭の `y` を、こう「改善」したくなったと
します。

```
x = read()
y = x + 1
assume x == 42 else L_deopt[ y, x ]
```

「`assume x == 42` の下流では `x` は 42 だから、レシピの `y` も `42 + 1 = 43`
で十分だ」と考え、レシピ側の `y` を定数 43 に畳み込んでしまったとします。

```
x = read()
                          # y = x + 1 を消し
assume x == 42 else L_deopt[ y:=43, x ]   # レシピで y は定数 43、として戻す
```

下流（`cond` が真の世界）では確かに `y` は 43 です。ところが**レシピが
使われるのは `cond` が外れた瞬間、すなわち `x` が 42 でなかったとき**です。
そこで `y` に 43 を渡せば、本来 `x + 1`（≠ 43）であるべき `y` が狂います。
これがまさに最初に見た罠の再来であり、「観測している経路だけ正しければよい」
を**レシピにまで適用してしまった**誤りです。

> [!IMPORTANT]
> 原則を正確に言い直すと、**高速経路の最適化には投機の前提を使ってよいが、
> 復元レシピの計算には使ってはならない**。レシピは「前提が外れた世界」へ
> 戻すためのものだからです。Flückiger らの証明は、最適化ごとに「どちらの
> 側に対する変形か」を区別し、前提を使ってよいのは高速経路側だけ、という
> 線引きを形式的に固定することで、この落とし穴を原理的に塞ぎます。

## 検証手法：guard/assume ノードから機械検証へ

この正しさを、実際にはどう検証するのでしょうか。出発点は、ここまで見てきた
**`assume`（ガード）を IR の第一級ノードにする**ことそのものです。
[中間表現](ir.md)の Sea of Nodes[](#cite:click1995)では、各演算がノードで
表され、依存関係が辺で表されました。ガードや脱最適化点も、この海の中の
**専用ノード**として置かれます[](#cite:duboscq2013)。すると「`y` という
ノードを消してよいか」は、**`y` から `assume` ノードの復元レシピへ向かう辺が
残っていないか**というグラフ上の問いに翻訳できます。辺が残っていれば、
そのノードは脱最適化経路を通じて生きており、消せない —— 透明性の制約が、
データ依存のグラフを辿るだけの**機械的なチェック**になるのです。

ここから検証は二段階で深まります。

- **第一段階：最適化を「レシピを保つ変形」として記述・チェックする。**
  各最適化に「この変形は `assume` の復元レシピをこう更新する」という規則を
  添え、変形後も全レシピが元の意味を復元できることを、IR 上の不変条件として
  確認します。Flückiger らの sourir はこの段階を形式的に定式化したもので、
  彼ら自身が R 言語向けに開発していた中間表現 RIR から着想を得て設計されて
  います[](#cite:fluckiger2018)。論文自体は特定の処理系に縛られない一般的な
  IR として sourir を据え、証明は紙の上の弱双模倣で与えています。
- **第二段階：機械検証（証明支援系での証明）へ広げる。**
  Flückiger らの証明そのものは紙と鉛筆による弱双模倣の議論ですが、続く研究は
  「この最適化はどんな入力・どんなガード破れの順でも透明性を壊さない」を、
  人手の議論でなく証明支援系で機械的に確かめる方向へ進みます。ガード／
  脱最適化を第一級ノードにしておいたおかげで、検証器は「最適化前後でレシピが
  復元する状態が一致するか」を形式的な対象として扱えます。

> [!TIP]
> 要点は、**正しさのチェックを「人が気をつける」から「グラフ／論理の上で
> 機械が確かめる」へ移したこと**です。ガードと復元レシピを IR の中央に
> 据えたからこそ、削除・巻き上げ・スカラ置換のいずれもが「レシピへの辺を
> 壊していないか」という一様な問いに帰着し、最終的に機械検証の射程に
> 入ってきます。

## 研究の広がり

この仕事は、投機的最適化の形式化という分野を切り開きました。続く研究は、
実際に機械検証へ踏み込みます。Barrière らは、投機と脱最適化を備えた
モデル JIT コンパイラを構成し、投機を活かす主要な最適化の正しさを、
証明支援系 Coq で**機械検証**しました[](#cite:barriere2021)。紙の上の
弱双模倣の議論だった透明性が、機械がチェックする証明になったのです。
また実在の処理系を対象に、[Graal](case-jvm.md) の IR の意味論と最適化を
Isabelle/HOL で形式化する研究（Webb・Utting・Hayes ら）も進んでいます
[](#cite:webb2021)。「投機して速くし、
外れたら戻る」という JIT の根本戦略が、性能だけでなく**正しさの理論**の
対象としても成熟してきたのです。実際、本章の `assume` が IR に据えた
「投機の前提と復元レシピ」に相当するものを、[Graal](case-jvm.md) は
投機の前提と脱最適化点を**第一級の IR ノード**として持っています
[](#cite:duboscq2013)。理論の枠組みと実在の処理系の設計が、同じ発想で
出会っているわけです。

実装面との接点も見逃せません。sourir は、著者らが R 言語向けに開発していた
中間表現 RIR から着想を得て設計されており[](#cite:fluckiger2018)、そこで
得た知見をどの言語にも縛られない一般的な IR として抽象化したものです。
理論が実在の処理系の設計から生まれ、また設計へ環流していく —— その往復が
この研究の価値です。

> [!NOTE]
> 実用 JIT の検証は、言語処理系にとどまりません。Linux カーネル内で動く
> eBPF の JIT コンパイラを対象にした **Jitterbug**（OSDI 2020）は、JIT の
> 正しさを形式仕様として定めて検証し、運用中の複数アーキテクチャの JIT
> から 16 件の未知のバグを発見・修正して、その変更をカーネル本体に取り
> 込ませました[](#cite:nelson2020)。形式手法が、現役で配備されている JIT
> の品質保証として機能した実例です。

## まとめ

- 投機的 JIT の正しさとは、**いつ脱最適化してもインタプリタ版と
  同じふるまいになる**こと（脱最適化の**透明性**）。
- 最適化が脱最適化経路を壊す典型は三つ：(a)デッドコード削除が脱最適化後に
  だけ要る値を消す、(b)巻き上げ・並べ替えがガード破れ時に未計算の値を
  参照させる、(c)スカラ置換で消したオブジェクトを脱最適化先が要求する。
  どれも**高速経路だけ見れば正しい**ので発見しにくい。
- 難しさの核心：**投機の前提を使ってよいのは投機が当たる高速経路だけ**。
  脱最適化で戻る先の状態（復元レシピ）の計算に前提を使うと、外れた
  ときに答えが狂う。
- 状態保持には二方式：(1)最適化に制約を課しレシピで復元（消した値も再計算
  手順を残す。高速経路は身軽）か、(2)各ガードに巨大な脱最適化マップを持つ
  （素直だがメモリ・生成コスト大）。実装は両者を混ぜる。
- Flückiger ら[](#cite:fluckiger2018)は、投機を `assume` という**第一級
  命令**として IR に明示し、復元レシピを命令の一部にした。最適化はレシピの
  更新義務を負い、正しさの制約が IR 上でチェックできる。
- 定数畳み込み・unreachable コード削除・関数インライン化に加え、脱最適化
  固有の変形（無制限の脱最適化・述語の巻き上げ・`assume` の併合）が透明性を
  保つことを、**弱双模倣による形式的証明**で示した[](#cite:fluckiger2018)。
  実務の経験則に理論的根拠を与えた。
- 検証は guard/assume ノード（[中間表現](ir.md)）の辺を辿るグラフ上の
  チェックから、証明支援系による**機械検証**（Barrière らの Coq 検証
  [](#cite:barriere2021)、Linux カーネルの eBPF JIT を検証して実バグを
  見つけた Jitterbug[](#cite:nelson2020)）へと広がっている。JIT が
  正しさの理論の対象に。

これで理論寄りの研究を見終えました。第 II 部の最後は、いま最も新しい
実行基盤 —— WebAssembly の JIT へ目を向けます。
