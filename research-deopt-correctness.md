# 投機的最適化の正しさ

[デオプティマイズと OSR](deopt-osr.md)の章の最後に、深い問いを残しました。
**「途中まで最適化版で進み、続きをインタプリタが引き継ぐ」とき、本当に
元のプログラムと同じ答えになると、どうして保証できるのか。**

これは「気をつければ大丈夫」で済む問題ではありません。JIT は投機の上に
投機を重ね、ガードを動かし、変数を消し、命令を並べ替えます。そのどれか
一つでも、デオプティマイズの正しさをひそかに壊していたら —— ある特定の
入力でだけ、最適化版とインタプリタ版の答えが食い違う、という悪夢の
バグになります。しかも投機が外れる経路は稀にしか通らないので、テストでも
見つかりにくい。

2018 年、Flückiger らはこの問題に**形式手法**で正面から取り組みました
[](#cite:fluckiger2018)。投機的最適化とデオプティマイズの正しさを数学的に
定義し、「どんな最適化なら正しさを保つか」を証明したのです。この章では、
論文の数式には立ち入らず、その**考え方**を追います。

## 何が「正しい」のか

まず、何を証明したいのかをはっきりさせます。投機的 JIT には、同じ
プログラムに対して二つの実行があります。

- **インタプリタ版**：投機なしで、常に正しく動く基準（リファレンス）。
- **最適化版**：投機して特殊化したコード。ガードが破れたら
  [デオプティマイズ](deopt-osr.md)してインタプリタ版へ戻る。

正しさとは、ひとことで言えば、

> **最適化版は、いつ・どこでデオプティマイズしても、インタプリタ版と
> まったく同じ観測可能なふるまい（同じ出力、同じ副作用）をしなければ
> ならない。**

ということです。デオプティマイズは**透明**でなければならない —— 外から
見て、それが起きたかどうか分からない。途中で最適化版からインタプリタ版へ
乗り換えても、結果が 1 ビットも変わってはいけない。この「透明性」こそが、
証明すべき性質です。

## 難しさの正体：最適化とデオプトの綱引き

なぜこれが難しいのか。最適化は、コードを**変形**して速くします。一方
デオプティマイズは、その変形したコードのどこからでも、**インタプリタが
期待する元の状態へ戻れ**と要求します。この二つは引っ張り合います。

具体例で見ましょう。あるコードで、変数 `y` が次のように使われているとします。

```ruby
x = read()          # 何か読む
y = x + 1           # y を計算する
... (y はこの先デオプト後にだけ必要)
assume x == 42      # 投機：x は 42 だと賭ける（ガード）
... (高速な経路)
```

最適化器が「`x` は 42 と賭けたのだから、`y = x + 1` は `y = 43` だ」と
**定数畳み込み**して、`y = x + 1` の計算自体を消したくなったとします。
高速経路では `y` を使わないので、消しても速い経路は正しく動きます。

ところが、です。**ガード `assume x == 42` が破れてデオプティマイズした
とき**、インタプリタは `y` の値を必要とします。もし最適化器が `y` の
計算を消してしまっていたら、戻り先のインタプリタに渡すべき `y` の値が
ありません。「`y = 43`」と決め打ちで渡してよいでしょうか？ いいえ ——
ガードが破れたということは `x` は 42 では**なかった**のだから、本当の
`y` は `x + 1`（≠ 43）であるべきです。決め打ちの 43 を渡せば、
デオプト後の実行が間違います。

> [!IMPORTANT]
> これが投機的最適化の正しさの核心的な罠です。**「投機の前提を使って
> よいのは、投機が当たっている経路だけ」**。デオプティマイズで戻るのは
> 投機が**外れた**瞬間なので、戻り先へ渡す状態（デオプティマイズ情報）の
> 計算に、投機の前提を使ってはいけない。速い経路の最適化と、戻り先の
> 状態の正しさは、注意深く分けて扱わねばならないのです。

## assume を第一級の命令にする

Flückiger らの工夫は、投機（ガードとデオプティマイズ）を、IR の中の
**第一級の命令**として明示的に表したことです。彼らは sourir という小さな
中間言語を定義し、その中に `assume` という命令を置きました。これは
[中間表現](ir.md)の章でふれた「ガードとデオプト点を IR に持たせる」設計
[](#cite:duboscq2013)を、形式的に突き詰めたものです。

`assume` 命令は、二つのものを束ねて持ちます。

1. **投機の条件**（例：`x == 42`）。これが成り立つ間は高速経路を進む。
2. **デオプティマイズの行き先と、そこで復元すべき状態のレシピ**
   （どのインタプリタ位置へ、どの変数にどの値を入れて戻るか）。

ポイントは、この「復元のレシピ」が**命令の一部としてIRに書かれている**
ことです。だから最適化器は、`y = x + 1` を消そうとするとき、「`y` は
どこかの `assume` の復元レシピで使われているか？」を IR を見て確認
できます。使われているなら、レシピが必要とする計算は勝手に消せない
（あるいは、レシピのほうに `y` の正しい計算を含めて保存する）。
**正しさの制約が、IR の上で明示的にチェックできる形になった**のです。

## どんな変形なら許されるかを証明する

この枠組みの上で、Flückiger らは主要な最適化が**デオプティマイズの
透明性を保つこと**を証明しました。証明したのは、たとえば次のような変形です。

- **定数畳み込み・コピー伝播**：投機の前提を使うのは高速経路に限り、
  `assume` の復元レシピには元の計算を保つ、という条件のもとで正しい。
- **不要コードの削除**：その値が高速経路でも**復元レシピでも**使われて
  いないことを確認したうえでなら、消してよい。
- **`assume` の挿入・移動・併合**：ガードを増やしたり、巻き上げたり、
  まとめたりする変形も、復元レシピを正しく引き継げば透明性を保つ。

これらは[中間表現](ir.md)や[デオプティマイズ](deopt-osr.md)で「実用上
こうする」と紹介してきた最適化そのものです。本章の研究は、それらが
**確かに正しい**ことを、場当たり的な議論でなく**数学的な証明**で
裏づけました。「ガードを巻き上げてよい」「重複ガードを消してよい」と
JIT 実装者が経験的に行ってきた変形に、形式的な根拠を与えたのです。

> [!NOTE]
> 「正しさを証明する」と聞くと、実務から遠い話に思えるかもしれません。
> しかし投機的 JIT のバグは、**特定の入力でだけ、稀なデオプト経路で
> 結果が狂う**という、最も発見しにくい種類のバグです。テストでは
> ほぼ見つかりません。だからこそ、「この最適化はどんな場合でも透明性を
> 壊さない」という保証には、実用上の大きな価値があります。形式手法は、
> JIT のような「賭けを安全網で支える」システムと、とても相性がよいのです。

## 研究の広がり

この仕事は、投機的最適化の形式化という分野を切り開きました。続く研究では、
さらに踏み込んで、**実際の JIT コンパイラのデオプティマイズを機械検証
（証明支援系で証明）する**試みや、特定の処理系（[Graal](case-jvm.md) など）の
デオプティマイズを対象にした検証へと広がっています。「投機して速くし、
外れたら戻る」という JIT の根本戦略が、性能だけでなく**正しさの理論**の
対象としても成熟してきたのです。

実装面でも、この枠組みは Ř（R 言語の研究用 JIT）などで実証されました。
理論が実在の処理系の設計に環流している点も、この研究の価値です。

## まとめ

- 投機的 JIT の正しさとは、**いつデオプティマイズしてもインタプリタ版と
  同じふるまいになる**こと（デオプティマイズの**透明性**）。
- 難しさの核心：**投機の前提を使ってよいのは投機が当たる高速経路だけ**。
  デオプトで戻る先の状態（復元レシピ）の計算に前提を使うと、外れた
  ときに答えが狂う。
- Flückiger ら[](#cite:fluckiger2018)は、投機を `assume` という**第一級
  命令**として IR に明示し、復元レシピを命令の一部にした。これにより
  正しさの制約が IR 上でチェックできる。
- 定数畳み込み・不要コード削除・ガードの移動などの最適化が、透明性を
  保つことを**形式的に証明**。実務の経験則に理論的根拠を与えた。
- 投機的最適化の形式化・機械検証へと発展中。JIT が正しさの理論の対象に。

これで理論寄りの研究を見終えました。第 II 部の最後は、いま最も新しい
実行基盤 —— WebAssembly の JIT へ目を向けます。
