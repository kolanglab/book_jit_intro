# ケーススタディ：その他の処理系

最後の章では、独立した章を立てるには紙幅が足りないけれど、地図に
欠かせない処理系たちを巡ります。それぞれが、本書の技法に独自のひねりを
加えています。ここまでの章で身につけた語彙で、手早く読み解いていきましょう。

## JavaScriptCore：四段の別解

Apple の **JavaScriptCore**（JSC、Safari の JavaScript エンジン）は、
[V8](case-v8.md)と同じ JavaScript を対象にしながら、独自の四段構成を
持ちます。

| ティア | 名前 | 役割 |
|--------|------|------|
| 0 | LLInt | 低レベルインタプリタ（アセンブリ記述） |
| 1 | Baseline JIT | 素朴な[ベースライン](tiered.md)機械語 |
| 2 | DFG | データフローグラフ最適化 JIT |
| 3 | FTL | 全力最適化 JIT |

JSC の用語は独特ですが、構造は[V8](case-v8.md)や[JVM](case-jvm.md)と
同じ[階層的 JIT](tiered.md)です。**DFG**（Data Flow Graph）が中位最適化、
**FTL**（Faster Than Light）が最上位の全力最適化を担います。ここからは、
四段のそれぞれが「**例えばこうする**」という具体の積み重ねでできていることを、
段ごとに見ていきましょう。

### LLInt：機械語を生成しないインタプリタ

最下段の **LLInt**（Low-Level Interpreter）は、JIT ではなくインタプリタ
です。しかし、ただの C で書かれたインタプリタではありません。LLInt は
**offlineasm** という JSC 専用の移植可能アセンブリで記述されています。
なぜそんな手間をかけるのか。理由は二つあります。

- **呼び出し規約を JIT と一致させるため**。LLInt はレジスタの使い方や
  スタックフレームの形を Baseline JIT 以降と揃えてあり、インタプリタから
  JIT コードへ、あるいはその逆へ、**フレームを作り直さずに**飛び移れます。
  C で書いた素朴なインタプリタだと、ここで高価な変換が要ります。
- **インラインキャッシュをインタプリタ段から仕込むため**。LLInt は各
  バイトコードのオペランドに**インラインキャッシュ用の空き地**を持っており、
  まだ一度も JIT していない段階から、`obj.foo` の探索結果をその場に
  覚えます。[プロファイリング](profiling.md)で見たインラインキャッシュ
  [](#cite:deutsch1984)を、最下段から効かせる設計です。

> [!NOTE]
> LLInt が機械語を生成しないことは、製品上の重要な利点でもあります。
> iOS では W^X（書き込みと実行を同時に許さない）制約のため、実行可能
> メモリの確保が制限される場面があります。機械語を一切生成しない LLInt は
> その制約下でも動けるので、JIT が禁じられた環境のフォールバックとして
> 機能します。「最下段をインタプリタにしておく」判断は、性能だけでなく
> 移植性・安全性の保険でもあるのです。

### Baseline JIT：プロファイルを集めるための機械語

第一段の **Baseline JIT** は、最適化をほとんどしません。各バイトコードを
そのまま機械語のテンプレートに展開するだけの、素朴な[ベースライン](tiered.md)
です。生成は速く、コードは遅い。狙いは二つです。

- LLInt よりは速い機械語で、まだ熱くなりきらないコードを少し底上げする。
- そして何より、**上位段が必要とするプロファイルを稼ぐ**。Baseline が
  吐く機械語には、各オペランドが実際にどんな型の値を見たか（整数か、
  浮動小数点か、オブジェクトか）を記録する**値プロファイル**の埋め込みが
  あります。例えば `a + b` という加算なら、左右が常に Int32 だったのか、
  一度でも文字列が来たのか、を Baseline が観測して残します。

ここが JSC の設計思想の核です。**Baseline JIT の本当の仕事は高速化では
なく、上位段への投機材料を集めること**なのです。[プロファイリング](profiling.md)で
見た「観測の軽さと正確さの綱引き」を、JSC は「下位段ほど多く観測し、
上位段ほど投機する」という形で段に割り振っています。

### DFG：投機と OSR exit のかたまり

第二段の **DFG** は、Baseline が集めたプロファイルを根拠に、**大胆な投機**
を行います。例えば値プロファイルが「ここはずっと Int32 だった」と言って
いれば、DFG は `a + b` を**箱なしの 32 ビット整数加算**として生成します。
これは速い反面、もし実行時に Int32 でない値が来たら破綻します。そこで
**ガード**を置き、外れたら[脱最適化](deopt-osr.md)（deoptimization）で
下位段へ戻します。これが **OSR exit**（On-Stack Replacement による脱出）
です。

DFG の OSR exit には、具体的なトレードオフを伴う設計判断がいくつもあります。

- **OSR exit を「滅多に起きない」前提で重く作る**。DFG は exit 地点の
  コードをホットパスに置かず、まれにしか踏まない側へ追い出します。
  脱出時には、箱を外していた値を箱に戻し、レジスタに散っていた値を
  バイトコードのスタック位置へ書き戻す、という**状態の再構築**が必要
  ですが、その手順を**メタデータとして別表に持ち**、ホットなコードを
  汚しません。[脱最適化の章](deopt-osr.md)で見た「最適化コードと脱出
  経路の分離」の実例です。
- **watchpoint（監視点）で投機の前提を守る**。例えば「この関数は
  実行中に再定義されない」と仮定して最適化したとき、もし誰かが本当に
  再定義したら、その瞬間に登録しておいた watchpoint が発火し、依存して
  いた DFG コードを**まとめて無効化**します。ガードを毎回チェックする
  代わりに、「前提が崩れた瞬間だけ通知してもらう」設計です。

### FTL：B3 と、LLVM を捨てた判断

最上段の **FTL**（Faster Than Light）は、DFG の中間表現をさらに低い
[IR](ir.md)へ落とし、SSA 上で本格的な最適化（ループ不変式の巻き上げ、
冗長なガードの除去、レジスタ割り付け）をかけます。

FTL は当初 [LLVM](frameworks.md)をバックエンドに使っていました。最適化
品質は申し分なかったのですが、[ウォームアップ](research-warmup.md)で見た
「LLVM はコンパイルが遅い」問題が製品では致命的でした。そこで JSC チームは
**B3**（Bare Bones Backend）という JSC 専用の高速バックエンドを自作し、
LLVM を置き換えました。B3 は LLVM ほど凝った最適化はしないかわりに、
**生成が桁違いに速く、メモリも食わない**。「最適化品質と生成速度の
トレードオフで LLVM を捨てる」—— [フレームワークの章](frameworks.md)で
見た教訓が、実際の製品判断として現れた好例です。

> [!IMPORTANT]
> FTL でもう一つ注目すべきは、**OSR exit を上位段でも保持し続ける**設計
> です。最上段だからといって投機を諦めるのではなく、FTL もまた前提が
> 外れたら DFG や Baseline へ戻れます。さらに JSC は、ホットなループに
> 入る途中で下位段から FTL へ**乗り換える**ための OSR entry も持ちます。
> [OSR](deopt-osr.md)は脱出だけでなく**侵入**にも使う、という両刃の道具
> なのです。

### 製品判断としての「段の足し引き」

JSC の四段は、最初からこの形だったわけではありません。歴史的には、
LLInt → Baseline → DFG → LLVM(FTL) と段を**足しながら**進化し、その後
LLVM を B3 へ**差し替え**ました。これは[階層的 JIT](tiered.md)が教える
重要な現実 ——「段の数や役割は固定ではなく、起動速度・ピーク性能・
メモリ・移植性のバランスを見ながら**製品都合で足し引きされる**」—— を
体現しています。同じ JavaScript を対象にしても、[V8](case-v8.md)の段構成
（Ignition → Sparkplug → Maglev → TurboFan）と JSC の四段は異なります。
**正解の段構成は一つではない**のです。

JSC は[脱最適化](deopt-osr.md)や[OSR](deopt-osr.md)の実装でも知られ、
その公開された設計文書は JIT を学ぶ良い資料です。

## .NET：RyuJIT と AOT のあいだ

Microsoft の **.NET** は、C# などを共通中間言語（CIL）にコンパイルし、
それを実行時に **RyuJIT** という JIT で機械語にします。.NET の JIT には、
他の動的言語処理系とは少し違う事情があります。CIL は**静的型を持つ**ので、
[なぜ JIT か](why-jit.md)で見た「型が分からない」苦労が、JavaScript や
Ruby ほどは深刻ではありません。この点は[WebAssembly](research-wasm.md)に
近い立場です。

.NET も近年は[階層的コンパイル](tiered.md)を採り入れ、起動を速くする
素早い JIT と、熱い場所を最適化する JIT を使い分けます。具体的には
**Tier 0**（最適化を切った素早い JIT）でまず動かし、呼び出し回数が閾値を
超えたメソッドを **Tier 1**（フル最適化）へ昇格させます。ここで .NET ならではの
具体的な工夫がいくつもあります。

- **On-Stack Replacement で長いループを救う**。Tier 0 で動き始めた
  メソッドが、中で長大なループを回し続けると、呼び出しカウンタだけでは
  昇格できません（[プロファイリング](profiling.md)で見た「1 回しか
  呼ばれないが中で 100 万回回る」問題）。.NET は[OSR](deopt-osr.md)を
  導入し、回り続けているループを途中から Tier 1 コードへ乗り換えられる
  ようにしました。これがないと、Tier 0 を素早くするほど長いループが
  遅いままになる、というジレンマに陥ります。
- **Tier 0 でも軽量プロファイルを仕込む**。Tier 1 へ昇格するとき、ただ
  最適化し直すだけでなく、Tier 0 が集めた**呼び出し先の分布**（どの仮想
  メソッドが実際に呼ばれたか）を使い、熱い呼び出し先を投機的にインライン化
  します。静的型を持つ CIL でも、`interface` や `virtual` の先は実行時
  まで分かりません。そこに型フィードバックを効かせるわけです。
- **PGO（Profile-Guided Optimization）の動的版**。上記を一般化し、.NET は
  実行時プロファイルに基づく動的 PGO を備えます。「静的型言語だから投機は
  要らない」のではなく、**動的ディスパッチが残る限り投機は効く**、という
  ことです。

さらに、[ウォームアップ問題](research-warmup.md)への対策として
**ReadyToRun**（事前に部分 AOT コンパイルして同梱する）や **Native AOT**
（完全に AOT コンパイルして JIT なしで動かす）といった選択肢を持ちます。
ここにも具体的なトレードオフがあります。

- **ReadyToRun は「AOT した上で JIT で上書きできる」ハイブリッド**です。
  起動時は同梱の R2R コードで[ウォームアップ](research-warmup.md)なしに
  すぐ動き、熱くなった場所だけ Tier 1 JIT が後から差し替えます。AOT の
  起動の速さと、JIT のピーク性能を両取りする発想です。
- **Native AOT は JIT を完全に捨てる**かわりに、リフレクションや実行時
  コード生成といった「実行時に型を作る」機能に制限がかかります。起動の
  速さ・小さい配布サイズ・JIT 不可環境への対応と引き換えに、言語の動的さ
  を一部手放す、という判断です。

「JIT と AOT を状況で選べる」設計は、[GraalVM](case-jvm.md)とも通じる
現代的な潮流です。

> [!TIP]
> .NET の例が示すのは、「静的型だから JIT が単純になる」は**半分しか
> 正しくない**ことです。確かに値の型を当てるガードは減ります。しかし
> 仮想呼び出し・インターフェース・ジェネリクスの具体化など、**実行時まで
> 確定しない分岐**は静的型言語にも残り、そこには[V8](case-v8.md)や
> [Self](case-others.md)と同じ型フィードバックと投機的インライン化が
> 効くのです。技法は言語をまたいで再利用されます。

## Julia：型に特化した、科学計算のための JIT

**Julia** は、科学技術計算のために設計された言語で、JIT の使い方が独特
です。Julia は[LLVM](frameworks.md)をバックエンドに使いますが、その駆動の
仕方が、Julia の言語設計 —— **多重ディスパッチ**（multiple dispatch）と
**型特殊化** —— と深く結びついています。

Julia では、関数は引数の型の組み合わせごとに**別々の機械語へ特殊化**
されます。同じ関数 `f(x)` でも、`x` が整数のときと浮動小数点のときで、
それぞれに最適化された別の機械語が、**最初に呼ばれたときに**生成されます。

```julia
function add(a, b)
    return a + b
end

add(1, 2)        # Int64 同士の版を特殊化・コンパイル
add(1.0, 2.0)    # Float64 同士の版を別に特殊化・コンパイル

# 生成された機械語は @code_llvm / @code_native で覗ける
@code_native add(1, 2)     # 整数加算に特化したネイティブコードが見える
```

ここが Julia の鮮やかな点です。動的言語のように柔軟に書けるのに、
**呼び出しの型が決まった瞬間、その型に完全特化した静的なコードを生成**
するので、C 並みの速度が出ます。[なぜ JIT か](why-jit.md)で見た「型の
決め打ち」を、[投機とガード](deopt-osr.md)ではなく、**多重ディスパッチに
よる型ごとのコンパイル**で実現しているのです。投機が要らないので、
[V8](case-v8.md)や[Ruby](case-ruby.md)のような脱最適化の機械をあまり
必要としません。「言語設計で JIT を楽にする」という[WebAssembly](research-wasm.md)とも
通じる発想を、動的に書ける言語で実現したのが Julia です。

### 型推論と「型安定性」という設計契約

Julia の速さの鍵は、コンパイル直前に走る**型推論**です。引数の具体型が
決まると、Julia は関数本体をたどって**中間の変数の型まで伝播**させます。
推論がすべての変数の型を一意に確定できれば、加算も配列アクセスも箱を
外した機械語に落ち、C 並みになります。

ところが、もし推論が「この変数は `Int` か `Float64` のどちらか
（`Union{Int,Float64}`）」としか言えなかったら、Julia は実行時に型を
振り分ける**ボックス化された遅いコード**を生成せざるをえません。これが
**型不安定**（type instability）です。例えば次のような書き方が typical
な落とし穴です。

```julia
function sumto(n)
    s = 0            # ここで s は Int
    for i in 1:n
        s += i / 2   # i/2 は Float64。s が Int↔Float64 で揺れる
    end
    return s
end
# s = 0.0 と初期化すれば s は終始 Float64 で型安定になる
```

ここに Julia 固有の手法と文化があります。

- **`@code_warntype` で型の揺れを赤く可視化する**。プログラマは
  この道具で「どの変数が `Union` や `Any` に膨らんだか」を確認し、
  初期値や注釈を直して型を安定させます。**性能チューニングが「ガードを
  減らす」ではなく「推論を通す」作業になる**のが Julia 流です。
- **関数バリアで型不安定を閉じ込める**。型が定まらない部分を小さな
  関数に切り出すと、その内側は引数の型が決まった時点で再び特殊化され、
  不安定の伝播を断ち切れます。これは設計上の定石です。

> [!WARNING]
> 型特殊化は無料ではありません。引数の型の**組み合わせの数だけ**機械語が
> 生成されるので、多様な型で呼ばれる汎用関数は**コンパイル爆発**を起こし
> えます。さらに、プログラム起動後に初めて呼ばれた型の組では、その場で
> コンパイルが走る —— これが Julia 名物の「**最初の実行だけ異常に遅い**」
> （time-to-first-plot）問題で、[ウォームアップ](research-warmup.md)の
> Julia 版です。対策として、よく使う型の組をあらかじめコンパイルして
> イメージに焼き込む**プリコンパイル**が用意されています。投機を避けた
> 代償が、コンパイル量とウォームアップに回ってくるわけです。

> [!NOTE]
> Julia の「型ごとに特化したコードを生成する」は、[SELF のカスタマイズ](why-jit.md)
> [](#cite:chambers1989) —— オブジェクトのクラスごとにメソッドを特化する
> 1989 年の技術 —— の精神的な子孫です。古い JIT のアイデアが、新しい
> 言語の中心設計として生き返っているのが見えます。

## Dart と Android ART：開発時は JIT、出荷時は AOT

Google の **Dart**（Flutter の言語）と、Android の **ART**（Android
Runtime）は、**開発と出荷で実行方式を切り替える**戦略で知られます。

- **開発中**は **JIT** を使う。コードを書き換えてすぐ反映できる（Flutter の
  ホットリロード）柔軟さが、開発の速さに直結するからです。
- **出荷時**は **AOT** でコンパイルする。エンドユーザの端末では、起動の
  速さ・予測可能な性能・バッテリ消費が重要で、[ウォームアップ](research-warmup.md)の
  ない AOT が向くからです。

Android の ART はさらに巧妙で、**プロファイル誘導の AOT**（profile-guided
AOT）を行います。具体的な流れを追うと、ART の設計判断の細やかさが見えて
きます。

- **インストール直後は AOT しない**。かつての ART はインストール時に
  アプリ全体を AOT していましたが、これはインストールを遅くし、ストレージを
  食い、しかも**実際には使われないコードまでコンパイル**していました。
  そこで現在は、まずインタプリタと JIT で動かします。
- **JIT が「どこが熱いか」をプロファイルとして書き出す**。アプリを使う
  うちに、JIT は[プロファイル](profiling.md)（どのメソッドが熱いか、
  どのクラスが実際にロードされたか）をファイルに蓄積します。
- **端末がアイドル・充電中のときに `dex2oat` でバックグラウンド AOT**。
  集めたプロファイルに基づき、**熱い場所だけ**を選んで AOT コンパイルし、
  ディスクに保存します。コンパイルのコストをユーザが画面を見ていない時間に
  追い出す、という判断です。
- **次回起動からは AOT コードで素早く立ち上げ、未コンパイルの新しい
  熱所はまた JIT で拾う**。

これは[ウォームアップ問題](research-warmup.md)で見た
「[コンパイルキャッシュ](research-warmup.md)」を、**端末のアイドル時間と
電力事情を使って**実現した実例です。さらに、初回起動から AOT を効かせる
仕組みも二系統あります。一つは **Cloud Profiles**（Android 9 以降）で、
Google Play が**早期ユーザの ART プロファイルを集約**し、**匿名化した
「コアプロファイル」**（多くのセッションで頻繁に実行されたコードだけを残し、
外れ値を除いたもの）を後続のインストールへ**配信**します。もう一つは
**Baseline Profiles**で、こちらは**開発者が**プロファイルを用意してアプリ
（AAB）に**同梱**し、集約を待たずに初回から効かせます。「誰かが踏んだ熱所は、
皆にとっても熱いだろう」という賭けです。集めるのは「**どのコードが熱いか**」という
匿名・集約された統計だけで、ユーザの入力やデータは含まれない —— だからこそ
許される設計になっています。「JIT か AOT か」でなく「**JIT も AOT も、適材
適所で**」という現代的な答えが、ここに集約されています。

> [!NOTE]
> Dart の二刀流もまた、具体の積み重ねです。開発時の JIT は[ウォーム
> アップ](research-warmup.md)を厭わずホットリロードの柔軟さを取り、
> 出荷時の AOT は予測可能な性能とバッテリ持ちを取る。**同じ言語・同じ
> 意味論のまま、実行方式だけを開発フェーズで切り替える**ことで、開発体験と
> エンドユーザ体験という相反する要求を、それぞれの局面に最適な道具で
> 満たしているのです。

## Smalltalk と Self：すべての源流

最後に、歴史の源へ戻ります。本書の主要技法は、ごく少数の処理系が
1980〜90 年代に生み出したものの**組み合わせと変奏**です。その源流を、
発明の動機と具体的な手法までさかのぼって押さえておきましょう。

### Smalltalk-80：インラインキャッシュの誕生

時代をさかのぼると、まず **Smalltalk-80** の Deutsch–Schiffman 処理系
（1984 年）があります[](#cite:deutsch1984)。彼らが直面した問題は、
動的言語の宿命そのものでした —— `obj foo` というメッセージ送信のたびに、
`obj` のクラスからメソッド `foo` を**毎回探索**するのが遅い。

彼らの答えが二つあります。一つは、バイトコードを実行時に機械語へ翻訳する
**JIT そのもの**（当時は dynamic translation と呼ばれました）。もう一つが、
[プロファイリング](profiling.md)で詳しく見た**インラインキャッシュ**
[](#cite:deutsch1984)です。「同じ呼び出し場所に来る `obj` は、たいてい
同じクラス」という経験則を突き、前回見つけたクラスとメソッドを呼び出し
場所に覚えておく。**JIT と IC という、現代 JIT の二本柱がここで同時に
立った**のです。

### Self：カスタマイズという過激な賭け

**Self** は、1980 年代後半から 90 年代にかけて開発された、プロトタイプ
ベースのオブジェクト指向言語の処理系です。商業的には広まりませんでしたが、
その処理系は**現代の JIT のほとんどの技術を生み出した**、まさに源流です
[](#cite:ungar1987)。
Self は「すべてがメッセージ送信」という、Smalltalk より**さらに動的**な
言語でした。変数アクセスすら本質はメソッド呼び出しなので、素朴に作ると
遅くてどうにもなりません。この極端な設計が、過激な最適化を必要としたのです。

最初の一手が **カスタマイズ**（customization）です[](#cite:chambers1989)。
一つのメソッドを、**それが定義された型（クラス）ごとに別々の機械語へ
コンパイル**する。すると、メソッド本体に現れる `self` の型がコンパイル時に
確定するので、`self` に対する内部のメッセージ送信を**静的に解決**でき、
さらに[インライン化](compilation-units.md)まで踏み込めます。

> [!TIP]
> Julia の[多重ディスパッチによる型ごとの特殊化](case-others.md)を
> 思い出してください。あれは Self のカスタマイズ[](#cite:chambers1989)
> ——「型ごとに別のコードを出す」——の精神的な子孫です。Self は投機と
> ガードでこれを行い、Julia は型推論で前もって行う、という違いはあれど、
> **核のアイデアは同じ**です。少数のアイデアの変奏、という本章の主張が
> ここにも現れます。

ただしカスタマイズには明確なトレードオフがあります。型ごとにコードを
複製するので、**コードが膨れる**。Self の研究はこの肥大と戦う歴史でもあり、
「どこまで特殊化し、どこで共有するか」の線引きが、後年のすべての型特殊化
JIT に受け継がれる問いとなりました。

### 多態インラインキャッシュと型フィードバック

単態のインラインキャッシュは、呼び出し場所のクラスが一つに定まっている
ときは速い。しかし、複数のクラスが交互に来る場所では、毎回キャッシュが
外れて探索に落ちます。Hölzle らはここに **多態インラインキャッシュ**
（Polymorphic Inline Cache、PIC）を導入しました[](#cite:holzle1991)。
**一つの呼び出し場所に、観測したクラスとメソッドの組を複数並べて覚える**
という具体策です。

PIC の本当の価値は、速さだけではありませんでした。**PIC は、その場所に
実際にどんな型が・どんな割合で来たかという履歴そのもの**です。これを
コンパイラが読み取れば、「ここはほぼ整数だ」「この呼び出し先が支配的だ」と
分かる。こうして PIC は**型フィードバック**（type feedback）の情報源と
なり[](#cite:holzle1994)[](#cite:holzle1991)、[プロファイリング](profiling.md)で見た
「観測してから投機的に最適化する」というループの原型が完成しました。
[V8](case-v8.md)や[Ruby](case-ruby.md)が今やっている型フィードバックは、
まさにこの直系の子孫です。

### 脱最適化：デバッグのために生まれた

そして、本書が繰り返し扱ってきた **脱最適化**（deoptimization）です。
意外なことに、その発明動機は性能ではなく**デバッグ**でした
[](#cite:holzle1992)。

Self は最適化のために激しくインライン化します。すると、最適化された
機械語のスタックは、ソースコード上の呼び出し構造とまったく食い違って
しまう。ユーザがデバッガでスタックを覗いても、インライン化で消えた
フレームは見えず、デバッグになりません。Hölzle らの答えが大胆でした
——「**最適化コードを実行中に、いつでも最適化されていない状態へ巻き戻せる**
ようにする」。これが脱最適化です[](#cite:holzle1992)。

この「最適化コードを安全に下位段へ戻す」機構が、のちに**投機を支える
土台**へと転用されます。デバッグのために作った仕組みが、「型を投機して、
外れたら戻す」という現代 JIT の心臓部になったのです。[脱最適化と OSR の章](deopt-osr.md)で
見た技法は、ここに源を持ちます。

### 形の共有：マップ

最後に、Self の **マップ**（map）。同じ構造を持つオブジェクトたちが、
スロットの名前や位置といった**構造情報を一つのマップに共有**し、各
オブジェクトは値とマップへの参照だけを持つ、という仕組みです。これが
[V8](case-v8.md)の**ヒドゥンクラス**、[Ruby](case-ruby.md)の**シェイプ**の
直接の祖先です。動的にプロパティを足せる言語で、なお構造を静的に扱う
ための工夫の原型が、ここにあります。

### 主流への伝播

本書でここまで見てきた技法の、驚くほど多くが Self と Smalltalk に起源を
持ちます。改めて並べておきましょう。

- [インラインキャッシュ](profiling.md)（Smalltalk-80）[](#cite:deutsch1984)
- [多態インラインキャッシュと型フィードバック](profiling.md)
  [](#cite:holzle1994)[](#cite:holzle1991)
- [脱最適化](deopt-osr.md)（1992 年、デバッグのために）
  [](#cite:holzle1992)
- [クラスごとのカスタマイズ](why-jit.md)[](#cite:chambers1989)
- [ヒドゥンクラス/シェイプ](case-v8.md)の原型（Self の「マップ」）

Self の研究者たちは、その後 [HotSpot](case-jvm.md)（Java）や [V8](case-v8.md)
（JavaScript）の開発に関わり、Self のアイデアを主流の処理系へ運びました。
「[なぜ JIT か](why-jit.md)で導入した投機の発想は、いつ・どこで生まれたのか」
と問えば、その多くの答えが Self なのです。

> [!IMPORTANT]
> ケーススタディを巡ってきて見えるのは、**JIT は少数の強力なアイデアの
> 組み合わせと変奏でできている**ことです。投機・ガード・脱最適化・
> インライン化・型特殊化・階層化・形の共有 —— これらが Self で生まれ、
> 各言語の事情に合わせて組み替えられ、磨かれてきた。本書の第 I 部で
> 学んだ技法の地図さえ持っていれば、新しい処理系の JIT に出会っても、
> 「あの技法の、この変奏だ」と読み解けるはずです。

## まとめ

- **JavaScriptCore**：LLInt → Baseline → DFG → FTL の四段。LLInt は機械語を
  生成しないインタプリタで JIT 不可環境の保険、Baseline は投機材料を稼ぐ
  ためのプロファイラ、DFG は投機と OSR exit のかたまり、FTL は LLVM を
  自前の **B3** に置き換えた（[生成速度のための判断](frameworks.md)）。段は
  製品都合で**足し引き**される。
- **.NET**：静的型の CIL を **RyuJIT** で。Tier 0 → Tier 1 の[階層的](tiered.md)
  ＋[OSR](deopt-osr.md)＋動的 PGO。静的型でも仮想呼び出しには型フィード
  バックが効く。**ReadyToRun**/**Native AOT** で JIT と AOT を選べる。
- **Julia**：[多重ディスパッチ](case-others.md)で**型ごとに特化した
  コードを生成**。投機でなく型推論で型を確定させ、C 並みの速度を出すが、
  **型不安定**とコンパイル爆発・ウォームアップという代償を払う。
- **Dart / ART**：**開発は JIT、出荷は AOT**。ART はインストール時 AOT を
  やめ、JIT のプロファイルを基に端末のアイドル・充電時間で
  [プロファイル誘導 AOT](research-warmup.md)を行う。
- **Smalltalk / Self**：[インラインキャッシュ](profiling.md)（Deutsch–
  Schiffman）・[多態 IC と型フィードバック](profiling.md)・[脱最適化](deopt-osr.md)
  （デバッグ動機）・[カスタマイズ](why-jit.md)・[マップ＝シェイプ](case-v8.md)を
  生んだ**すべての源流**。

ここまでは「言語処理系」の JIT を見てきました。しかし JIT は、言語処理系
だけのものではありません。次章では、正規表現エンジンからデータベース、
カーネル、GPU、そして動的バイナリ変換まで、**言語の外で働く JIT** を巡り、
「実行時情報による特化」という一つの発想がどこまで広がっているかを見届けて、
第 III 部を締めくくります。
